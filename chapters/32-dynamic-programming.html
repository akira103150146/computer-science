動態規劃

初學者最害怕的四個字,其實只是一句樸實的忠告:同樣的功,別做第二遍。學會把「記住答案」變成一門手藝,你就握住了演算法工具箱裡威力最大、也最優雅的一把刀。

回到那個算到天荒地老的費氏數列

還記得第 3 章那個讓電腦當機的程式嗎?我們寫了一個美得像數學定義的費氏數列(Fibonacci)函式, 然後發現它有個致命傷:算 fib(50) 要等到地老天荒。我們也當場診斷出了病因—— 它把同樣的子問題算了一遍又一遍。這一章,我們要為這個病開出處方,而這帖藥的名字, 就是讓無數初學者聞之色變的四個字:動態規劃(dynamic programming,常簡稱 DP)。

先別怕。等你讀完這一章你會發現,動態規劃這個唬人的名字背後,其實只是一句再樸實不過的忠告: 「同樣的功,不要做第二遍。把算過的答案記下來。」就這樣。難的不是這個想法, 而是「認出哪些問題能這樣解」與「把它寫對」——而這兩件事,都有可以照著走的步驟。我們慢慢來。

先把第 3 章那個樸素遞迴版請回現場,當作我們的病人: 

def fib(n):
    if n < 2:                       # base case:fib(0)=0, fib(1)=1
        return n
    return fib(n - 1) + fib(n - 2)  # recursive case:呼叫自己「兩次」

print(fib(35))                      # 答得出來,但你會感覺到電腦在喘

問題出在最後那行呼叫了自己兩次。把 fib(5) 的呼叫過程畫成一棵樹 (第 3 章的圖 3-3),你會看到 fib(3) 整棵子樹被算了 2 次、fib(2) 被算了 3 次。 n 越大,這種重複越誇張:這棵樹的節點數量大約是 $O(2^n)$——指數成長。 fib(50) 要呼叫超過兩百億次。你的電腦不是壞了,是真的在做兩百億件重複的傻事。

🔑
DP 的核心直覺,一句話

當一個遞迴會反覆碰到同樣的子問題時,我們只要把每個子問題的答案算一次、記起來, 往後再遇到就直接查表。重複的功被砍光,指數爆炸就塌縮成線性。 這就是動態規劃的全部精神——其餘都是技巧與包裝。

記憶化:給遞迴配一本「查得到的筆記」

最直接的修法,是幾乎不改原本那支漂亮的遞迴,只多給它一本筆記本:每算出一個 fib(k),就把 k → 答案 記進筆記;下次再被要求算 fib(k),先翻筆記, 翻得到就直接抄,翻不到才真的動手算。這個技巧叫記憶化(memoization)。

(順帶一提,memoization 不是 memorization 拼錯——它源自拉丁文 memorandum「備忘錄」, 指的正是「把結果寫進備忘錄」這件事。)

我們用第 3 章教過的「None 當預設值」安全寫法,避免可變預設參數的陷阱: 

def fib(n, memo=None):
    if memo is None:                # 第一次呼叫才建立筆記本(避開可變預設值陷阱)
        memo = {}
    if n < 2:                       # base case
        return n
    if n not in memo:               # 筆記裡沒有,才真的動手算
        memo[n] = fib(n - 1, memo) + fib(n - 2, memo)
    return memo[n]                  # 直接抄筆記

print(fib(100))                     # 354224848179261915075,瞬間算完

關鍵只多了兩件事:算之前先查 memo、算完把結果寫進 memo。 效果立竿見影:第一次算 fib(3) 把它記下來後,第二次需要 fib(3) 時, 整棵子樹被一句「查表」取代,不必再展開。把 fib(5) 的記憶化過程畫出來:

因為現在每個 fib(k) 只會被真正計算一次(總共 $n+1$ 個不同的子問題), 記憶化版的時間複雜度從 $O(2^n)$ 一口氣降到 $O(n)$,代價是用一個 dict 存下 $O(n)$ 個答案。 用空間換時間,正是 DP 永恆的交易。

更省事的寫法:functools.lru_cache

手動管理 memo 字典雖然清楚,但 Python 標準函式庫已經幫你把這件事做成一個裝飾器(decorator)。 只要在函式上方加一行 @lru_cache,Python 就會自動幫你快取每組參數對應的回傳值: 

from functools import lru_cache

@lru_cache(maxsize=None)            # maxsize=None 表示快取無上限
def fib(n):
    if n < 2:
        return n
    return fib(n - 1) + fib(n - 2)  # 跟最原始的版本一模一樣!

print(fib(100))                     # 一樣瞬間算完
print(fib.cache_info())             # 還能看快取命中幾次

注意函式本體跟最樸素的版本一字不差——快取完全由裝飾器代勞。這就是記憶化最迷人的地方: 它幾乎不破壞你原本「貼近數學定義」的漂亮遞迴,只是悄悄替它配了一本筆記本。 Python 3.9 之後甚至有更簡潔的 @functools.cache(等同 lru_cache(maxsize=None))。

ℹ️
記憶化又叫「由上而下(top-down)」

記憶化是從你真正想要的那個大問題(fib(n))出發,需要哪個子問題才往下遞迴去算—— 像從樹頂往下伸手,缺什麼摘什麼,摘到的就放進口袋(快取)。 因為「從大往小、需要才算」,它被歸類為 top-down 的 DP。 下一節的表格法則是反過來,從最小的子問題開始往上堆。

表格法:乾脆從最小的開始,由小到大填一張表

記憶化是「被動」的:大問題來敲門,缺哪塊才回頭去補。另一種風格是「主動」的—— 我事先就知道要算 fib(n) 一定會用到 fib(0), fib(1), …, fib(n-1), 那何不乾脆從 fib(0) 開始,由小到大、一格一格把答案填進一張表? 這種風格叫表格法(tabulation),也叫由下而上(bottom-up)。 

def fib(n):
    if n < 2:
        return n
    dp = [0] * (n + 1)              # dp[k] 將存放 fib(k),這就是我們的「表」
    dp[0], dp[1] = 0, 1            # base case 先填好
    for i in range(2, n + 1):      # 由小到大,一格一格填
        dp[i] = dp[i - 1] + dp[i - 2]   # 用「前面已經填好的格子」算出當前格子
    return dp[n]                   # 表的最後一格就是答案

print(fib(100))                    # 354224848179261915075

沒有遞迴、沒有堆疊、沒有快取查詢的開銷,就是一個老老實實的迴圈,把一張表從左填到右。 每格只填一次,所以一樣是 $O(n)$ 時間。把這張表畫出來,你會看到一條清楚的「資料流」: 每一格都站在它前兩格的肩膀上。

空間優化:其實只需要記得「前兩格」

看圖 32-2 你會發現一件事:要算 dp[i],我們只用到 dp[i-1] 和 dp[i-2], 前面那一長串早就用不到了。既然如此,何必存整張表?用兩個變數滾動著走就好: 

def fib(n):
    a, b = 0, 1                # a = fib(i-2),b = fib(i-1)
    for _ in range(n):
        a, b = b, a + b        # 往前滾一步
    return a

print(fib(100))               # 一樣的答案,但空間從 O(n) 降到 O(1)

這正是第 3 章那個 fib_iter!現在你看懂它的真面目了:它就是做了空間優化的 bottom-up DP。 當你發現「當前狀態只依賴最近的幾個狀態」時,就能把整張表丟掉,只留一個滑動視窗—— 從 $O(n)$ 空間壓到 $O(1)$。這個「只留需要的前幾項」的把戲,在 DP 裡會反覆出現。

💡
記憶化 vs 表格法,怎麼選?

兩者算的東西完全一樣,只是方向相反。記憶化(top-down)幾乎不改原本的遞迴、 只算「真正用得到」的子問題,當狀態空間很大但實際只會碰到一小部分時特別划算,寫起來也最直覺。 表格法(bottom-up)沒有遞迴的堆疊開銷與深度上限,而且因為「填表順序」明確, 最容易做空間優化。新手建議:先用記憶化想通,再視需要翻成表格法。

面向記憶化 top-down表格法 bottom-up
方向從大問題往小遞迴從小子問題往大迭代
實作遞迴 + 快取(dict / lru_cache)迴圈 + 陣列(表)
只算用得到的子問題?是(惰性)否,通常全填
堆疊風險有(遞迴深度上限)無
空間優化較難容易(滾動陣列)
上手難度低(像原本的遞迴)需想清楚填表順序

什麼樣的問題「可以」用 DP?兩個前提

記憶化很神,但它不是萬靈丹。一個問題能用動態規劃高效解決,背後必須同時滿足兩個條件。 看懂這兩個條件,你就能在面對新題目時,一眼判斷「這題能不能 DP」。

前提一:最佳子結構(optimal substructure)

一個問題的最佳解,可以由它的子問題的最佳解組合出來。 以費氏為例:fib(n) 的答案,就是 fib(n-1) 與 fib(n-2) 這兩個子問題答案的和—— 子問題的解直接拼出大問題的解。再以「從台北到高雄的最短路線」為例:如果最短路線經過台中, 那麼「台北到台中」這一段,本身一定也是台北到台中的最短路線(否則你換掉它就能讓全程更短,矛盾)。 這種「大最佳解內含小最佳解」的性質,就叫最佳子結構。

並非所有問題都有這個性質。例如「最長簡單路徑(不能重複經過點)」就沒有最佳子結構—— 把兩段各自最長的路徑接起來,可能會重複經過某個點而違規。沒有最佳子結構,DP 就使不上力。

前提二:重疊子問題(overlapping subproblems)

同一個子問題會被反覆需要很多次。這正是費氏的痛點:fib(3) 在樸素遞迴裡被算了好幾遍。 因為會重疊,「把答案記下來」才有意義——記一次,省下無數次重算。 如果每個子問題都只會被用到一次(不重疊),那記下來也沒得省,記憶化就毫無價值。

🔑
DP 成立的兩個前提

① 最佳子結構:大問題的最佳解,能用子問題的最佳解拼出來(這保證「用遞推式求解」是對的)。
② 重疊子問題:同樣的子問題一再出現(這保證「把答案記起來」是划算的)。
兩者缺一不可:沒有①,遞推式根本不正確;沒有②,記憶化省不到任何東西。 看到一個問題同時具備這兩點,你心裡就該亮起燈號:這題,DP 有戲。

和分治、貪婪比一比

重疊子問題這個前提,正好把 DP 和它的兩個近親區分開來。第 27 章的分治法(divide and conquer) 也是把大問題拆成小問題遞迴去解,但分治拆出來的子問題互不重疊 (例如合併排序把陣列切成左右兩半,左半跟右半毫無交集),所以分治不需要記憶化。 第 31 章的貪婪法(greedy)則更激進:它每一步都只挑「當下看起來最好」的選擇, 選了就不回頭,不像 DP 會把所有可能的選擇都試過、記下、再挑最佳。 貪婪快,但只在「貪婪選擇性質」成立時才正確;不成立時,它會給出錯的答案(這一章稍後的硬幣找零就是活生生的反例)。

範式子問題每一步的決策正確性靠什麼代表問題
分治法(第 27 章)拆成不重疊的子問題遞迴解子問題,再合併問題可乾淨切分合併排序、快速冪
貪婪法(第 31 章)每步縮成一個子問題選當下最好,不回頭貪婪選擇性質 + 最佳子結構Dijkstra、霍夫曼編碼、最小生成樹
動態規劃(本章)大量重疊的子問題試遍所有選擇,記住結果取最佳最佳子結構 + 重疊子問題背包、LCS、編輯距離

一句話總結三者的關係:分治是「拆了各自解,反正不重複」;貪婪是「賭一條路走到底,不回頭」; DP是「每條路都看看,但把走過的路記下來,絕不重走」。DP 可以看成「會記筆記、肯回頭比較的窮舉」—— 它聰明地窮舉,而不是盲目地窮舉。

📜
「動態規劃」這名字其實是個煙霧彈

這個技巧由數學家 Richard Bellman 於 1950 年代在 RAND 公司提出。 名字裡的「programming」指的是「規劃、做決策表」的舊義(跟「寫程式」無關), 而「dynamic」則據 Bellman 自己回憶,是刻意挑來「聽起來很厲害、又不像在做數學」的詞—— 因為當時他的上司、國防部長 Wilson 極度厭惡「研究(research)」和「數學」這類字眼, Bellman 需要一個能矇混過關、聽起來無害的名字來幫他的研究爭取經費。 所以,如果你覺得「動態規劃」這名字取得莫名其妙、跟它在做的事兜不太起來—— 恭喜你,它本來就是故意取得讓人摸不著頭緒的。

設計 DP 的五步框架

DP 之所以讓初學者害怕,常常不是因為某一題特別難,而是因為「不知道從哪裡下手」。 好消息是:解 DP 幾乎都能套同一套流程。把下面這五個步驟當成你的解題清單, 遇到新題目就一條一條問自己。這一章後面每一道經典題,我們都會老老實實走一遍這五步。

  1. 定義狀態(state):這是最關鍵也最難的一步。你要決定「dp 的每一格代表哪一個子問題的答案」。 狀態定義錯了,後面全錯。例如費氏:「dp[i] = 第 i 個費氏數」。一個好狀態通常能用幾個整數參數(索引、容量、位置……)唯一描述一個子問題。
  2. 寫出狀態轉移方程(recurrence):用「更小的狀態」表達「當前狀態」。 這是 DP 的心臟。例如 $dp[i] = dp[i-1] + dp[i-2]$。通常的句型是:「當前這格的最佳解 = 對所有可能的『最後一步選擇』取最佳」。
  3. 確定 base case:那些小到不必再拆、可以直接寫死答案的狀態。例如 $dp[0]=0,\ dp[1]=1$。漏掉或寫錯 base case 是 DP 最常見的 bug。
  4. 決定計算順序:選 top-down(記憶化)或 bottom-up(填表)。若用填表,要確保「算每一格時,它依賴的格子都已經填好了」——這決定了迴圈該往哪個方向跑。
  5. 從表回推答案(reconstruct):有時你要的不只是「最佳值是多少」,還要「最佳解長什麼樣」(選了哪些物品、是哪個子序列)。這時要從填好的表「倒著走回去」,還原出實際的選擇。
💡
卡住時,先攻第①步

九成的 DP 難題,難就難在「定義狀態」。一個實用的起手式:先寫出一個會動但很慢的暴力遞迴, 觀察「這個遞迴函式靠哪幾個參數就能決定回傳值」——那幾個參數,往往就是你的狀態。 換句話說,暴力遞迴的函式簽名,常常直接告訴你 dp 的維度。這條捷徑,我們會在最後一節正式收束成通用流程。

經典題一:爬樓梯(熱身)

題目:一段樓梯有 $n$ 階,你每次可以跨 1 階或 2 階。問:從地面爬到第 $n$ 階,總共有幾種不同的走法?

套五步框架:

  1. 狀態:dp[i] = 爬到第 i 階的走法數。
  2. 轉移:你到達第 i 階的「最後一步」,要嘛從第 i-1 階跨 1 階上來,要嘛從第 i-2 階跨 2 階上來。這兩種情形互斥且涵蓋所有可能,所以 $dp[i] = dp[i-1] + dp[i-2]$。
  3. base case:dp[0] = 1(待在地面,一種「什麼都不做」的走法),dp[1] = 1。
  4. 順序:bottom-up,i 從小到大。
  5. 回推:本題只要個數,不需還原路徑。
def climb_stairs(n):
    if n <= 1:
        return 1
    a, b = 1, 1                # dp[0], dp[1]
    for _ in range(2, n + 1):
        a, b = b, a + b        # dp[i] = dp[i-1] + dp[i-2]
    return b

print(climb_stairs(5))        # 8

看出來了嗎?爬樓梯就是費氏數列換了一件衣服。這正是 DP 學習的重要一課: 非常多「看起來不一樣」的問題,骨子裡是同一條遞推式。當你能一眼認出「啊,這又是 $dp[i]=dp[i-1]+dp[i-2]$」, 你就開始有 DP 的「手感」了。(練習裡有它的變形:每次可跨 1、2 或 3 階。)

經典題二:硬幣找零(貪婪會在這裡翻車)

題目:給你面額為 coins 的硬幣(每種數量無限),要湊出金額 amount, 問最少需要幾枚硬幣?湊不出來則回傳 -1。

你的第一直覺也許是「貪婪」:每次都拿能用的最大面額。對台幣或美元這種「漂亮」的幣制,貪婪確實對。 但換一組幣制就會出事。假設 coins = [1, 3, 4],要湊 amount = 6:

  • 貪婪:先拿最大的 4,剩 2;只能拿 1+1。總共 4+1+1 = 3 枚。
  • 最佳解:3+3 = 2 枚。貪婪輸了!

貪婪在這裡翻車,正是因為它「選了 4 就不回頭」,沒看到「退一步用兩個 3」更省。 這就是第 31 章會反覆強調的:貪婪只在特定條件下才正確,幣制不對它就會給出次佳解。 而 DP 不會被騙——因為它會把「每種湊法」的代價都記下來再挑最小。

套五步框架:

  1. 狀態:dp[a] = 湊出金額 a 所需的最少硬幣數。
  2. 轉移:湊出 a 的「最後一枚硬幣」是某個面額 c(且 $c \le a$),那麼剩下的 a-c 就用它自己的最佳解。對所有可選的 c 取最小:$dp[a] = \min_{c \in coins,\ c \le a} \big(dp[a-c] + 1\big)$。
  3. base case:dp[0] = 0(湊 0 元不需要任何硬幣)。其餘先設成「不可能的大數」代表尚未湊出。
  4. 順序:bottom-up,a 從 1 填到 amount。
  5. 回推:本題只求數量;若要知道「用了哪些硬幣」,可額外記錄每格選了哪枚(留作思考)。
def coin_change(coins, amount):
    INF = amount + 1                    # 比任何可能答案都大,當「不可能」的標記
    dp = [0] + [INF] * amount           # dp[0]=0,其餘先設成 INF
    for a in range(1, amount + 1):
        for c in coins:
            if c <= a:                  # 這枚硬幣放得進金額 a
                dp[a] = min(dp[a], dp[a - c] + 1)
    return dp[amount] if dp[amount] != INF else -1

print(coin_change([1, 3, 4], 6))       # 2   (= 3 + 3,DP 找到了最佳解)
print(coin_change([2], 3))             # -1  (湊不出奇數)

經典題三:0/1 背包(DP 的鎮店之寶)

題目:你是個小偷,背包最多裝 W 公斤。眼前有 n 件寶物,第 i 件重 w[i] 公斤、 值 v[i] 元。每件寶物要嘛整件拿走、要嘛不拿(不能只拿半件)。 問:在不超重的前提下,能偷走的最大總價值是多少?

這個「要嘛全拿、要嘛不拿」就是名字裡 0/1 的由來。它跟第 31 章的分數背包(fractional knapsack) 只差一個字,命運卻天差地遠:分數背包可以切一塊寶物的零頭塞進去,於是「按 CP 值(價值/重量)從高到低貪婪地裝」就保證最佳—— 貪婪在那裡是對的。但 0/1 背包不能切,貪婪會卡在「裝不下整件、又不能塞半件」的尷尬,給出次佳解。 所以 0/1 背包必須用 DP。

套五步框架:

  1. 狀態:dp[i][w] = 「只考慮前 i 件寶物、背包容量為 w」時能拿到的最大價值。 (注意這裡的狀態需要兩個參數:看到第幾件、還剩多少容量。)
  2. 轉移:對第 i 件寶物,只有兩種選擇—— 不拿:價值就是 dp[i-1][w]; 拿(前提是裝得下,$w_i \le w$):價值是 dp[i-1][w - w_i] + v_i。取兩者較大: $$dp[i][w] = \max\big(\,dp[i-1][w],\ \ dp[i-1][w - w_i] + v_i\,\big)$$
  3. base case:dp[0][w] = 0(一件都不考慮,價值是 0)。
  4. 順序:bottom-up,外層跑寶物 i、內層跑容量 w。
  5. 回推:從 dp[n][W] 倒著走,就能還原「到底拿了哪幾件」(見練習)。
def knapsack(weights, values, W):
    n = len(weights)
    # dp[i][w]:前 i 件、容量 w 的最大價值;多開一行一列放 base case
    dp = [[0] * (W + 1) for _ in range(n + 1)]
    for i in range(1, n + 1):
        wi, vi = weights[i - 1], values[i - 1]   # 第 i 件(注意 0-based)
        for w in range(W + 1):
            dp[i][w] = dp[i - 1][w]              # 先假設「不拿第 i 件」
            if wi <= w:                          # 裝得下,才考慮「拿」
                dp[i][w] = max(dp[i][w], dp[i - 1][w - wi] + vi)
    return dp[n][W]

weights = [1, 3, 4, 5]
values  = [1, 4, 5, 7]
print(knapsack(weights, values, 7))   # 9  (拿第 2、3 件:重 3+4=7,值 4+5=9)

空間優化:二維壓成一維

觀察轉移式:dp[i][...] 只依賴上一列 dp[i-1][...]。所以其實不必存整張二維表, 只要一條一維陣列、就地更新即可——但有個陷阱:內層的容量迴圈必須從大到小跑, 否則同一件寶物會被重複拿(那就變成另一種「無限背包」了)。 

def knapsack_1d(weights, values, W):
    dp = [0] * (W + 1)
    for wi, vi in zip(weights, values):
        for w in range(W, wi - 1, -1):       # 注意:容量「從大到小」!
            dp[w] = max(dp[w], dp[w - wi] + vi)
    return dp[W]

print(knapsack_1d([1, 3, 4, 5], [1, 4, 5, 7], 7))   # 9
⚠️
一維背包:迴圈方向錯了,答案就錯

0/1 背包壓成一維時,容量迴圈一定要由大到小。若由小到大,當你更新 dp[w] 時, 它依賴的 dp[w - wi] 已經是「這一輪剛被同一件寶物更新過」的值,等於允許同一件寶物拿好幾次。 這個方向的差別,正好區分了「0/1 背包(由大到小)」與「完全背包/無限背包(由小到大)」—— 一個減號的方向,藏著兩種完全不同的問題。

經典題四:最長共同子序列 LCS(這就是 diff 的心臟)

題目:給兩個字串 X 與 Y,找出它們最長共同子序列(longest common subsequence, LCS)的長度。 所謂「子序列」是指可以跳著選、但不可以改順序取出的字元序列。例如 "ABCBDAB" 與 "BDCAB" 的一個 LCS 是 "BCAB"(長度 4)。

為什麼值得學?因為這正是你天天在用的 git diff(第 18 章)背後的演算法。 把兩個版本的檔案各看成「一串行(line)」,它們的 LCS 就是「沒被改動、共同保留的那些行」; 不在 LCS 裡的行,就是新增或刪除——這就是 diff 的本質。拼字檢查、DNA 序列比對、抄襲偵測,也都建立在類似的子序列比對上。

套五步框架:

  1. 狀態:dp[i][j] = 「X 的前 i 個字元」與「Y 的前 j 個字元」的 LCS 長度。
  2. 轉移:看兩個字串的「最後一個字元」X[i-1] 與 Y[j-1]—— 若相同:這個字元一定能加進 LCS,$dp[i][j] = dp[i-1][j-1] + 1$; 若不同:至少得捨棄其中一個字串的最後一個字元,取較好的那邊:$dp[i][j] = \max\big(dp[i-1][j],\ dp[i][j-1]\big)$。 $$dp[i][j] = \begin{cases} dp[i-1][j-1] + 1, & X[i-1] = Y[j-1] \\[4pt] \max\big(dp[i-1][j],\ dp[i][j-1]\big), & \text{否則} \end{cases}$$
  3. base case:任一字串為空時 LCS 為 0,即 dp[0][j] = dp[i][0] = 0。
  4. 順序:bottom-up,i、j 由小到大(每格依賴它的上、左、左上,都比它早算)。
  5. 回推:從 dp[m][n] 倒著走:遇到「字元相同(走左上對角)」就把該字元記下來,即可還原出實際的 LCS 字串。
def lcs_length(X, Y):
    m, n = len(X), len(Y)
    dp = [[0] * (n + 1) for _ in range(m + 1)]
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if X[i - 1] == Y[j - 1]:
                dp[i][j] = dp[i - 1][j - 1] + 1      # 字元相同,延長
            else:
                dp[i][j] = max(dp[i - 1][j], dp[i][j - 1])  # 不同,取較好的一邊
    return dp[m][n]

print(lcs_length("ABCD", "ACBD"))   # 3  (LCS 為 "ABD" 或 "ACD")

LCS 與 diff、與編輯距離

把 LCS 反過來看:兩個檔案不屬於 LCS 的行,就是要刪掉(舊檔獨有)或加上(新檔獨有)的行—— 這正是 git diff 印出那些 +/- 的依據。下一題的「編輯距離」是同一家族的近親: LCS 問「最多能保留多少」,編輯距離問「最少要改幾刀」。兩者都是經典的二維字串 DP,填表的長相幾乎一個模子。

經典題五:最長遞增子序列 LIS

題目:給一個數字陣列,找出最長的嚴格遞增子序列(longest increasing subsequence, LIS)的長度。 (一樣是「子序列」:可跳著選,但要保持原本的前後順序。)例如 [10, 9, 2, 5, 3, 7, 101, 18] 的一個 LIS 是 [2, 3, 7, 101],長度 4。

套五步框架(這題的狀態定義很有代表性):

  1. 狀態:dp[i] = 「以第 i 個元素結尾」的最長遞增子序列長度。 (注意這個「以某元素結尾」的定法——很多序列型 DP 都這樣定狀態,值得記住。)
  2. 轉移:要讓 a[i] 接在某個更早、且比它小的 a[j] 後面。對所有這樣的 j 取最好: $$dp[i] = 1 + \max\big(\{\,dp[j] : j \lt i,\ a[j] \lt a[i]\,\}\ \cup\ \{0\}\big)$$ (那個 $\{0\}$ 是為了處理「前面沒有任何比它小的元素」時,a[i] 自己單獨成一段,長度為 1。)
  3. base case:每個 dp[i] 至少是 1(元素自己就是一個長度 1 的遞增序列)。
  4. 順序:i 由小到大;內層掃 j 從 0 到 i-1。
  5. 回推:最終答案是整個 dp 陣列的最大值(而非 dp[n-1]!因為 LIS 可能結尾在任何位置)。
def lis_length(a):
    if not a:
        return 0
    n = len(a)
    dp = [1] * n                       # 每個元素自成長度 1
    for i in range(n):
        for j in range(i):
            if a[j] < a[i]:            # a[i] 可以接在 a[j] 之後
                dp[i] = max(dp[i], dp[j] + 1)
    return max(dp)                      # 答案是「結尾在任何位置」中最長的

print(lis_length([10, 9, 2, 5, 3, 7, 101, 18]))   # 4

這個版本是兩層迴圈,時間複雜度 $O(n^2)$。

ℹ️
LIS 還能更快

LIS 有一個漂亮的 $O(n \log n)$ 解法:維護一個「目前各長度遞增子序列的最小可能結尾值」陣列, 對每個新元素用二分搜尋(第 28 章)找到它該替換的位置。它已經不太像傳統的填表 DP,而更接近貪婪 + 二分搜尋的混血。 這裡先知道「存在更快的做法」即可;真正想挑戰的讀者,可在學完二分搜尋後回來推敲它。

經典題六:編輯距離(拼字檢查與 DNA 比對的底層)

題目:給兩個字串 a 與 b,每次操作可以插入一個字元、刪除一個字元、或替換一個字元。 問:把 a 變成 b,最少要幾次操作?這個最少次數叫編輯距離(edit distance), 也叫 Levenshtein 距離。

你每天都在享受它的成果:拼字檢查把你打錯的 "recieve" 跟字典裡的字算編輯距離, 找出距離最近的 "receive" 建議給你;搜尋引擎的「你是不是要找……」、生物資訊學裡比對兩段 DNA / 蛋白質序列 相似度,核心都是編輯距離或它的變形。

套五步框架:

  1. 狀態:dp[i][j] = 把「a 的前 i 個字元」變成「b 的前 j 個字元」的最少操作數。
  2. 轉移:看最後一個字元 a[i-1] 與 b[j-1]—— 若相同:這個字不用動,$dp[i][j] = dp[i-1][j-1]$; 若不同:取「刪除、插入、替換」三條路裡最省的,再加一次操作: $$dp[i][j] = 1 + \min\big(\underbrace{dp[i-1][j]}_{\text{刪除 }a[i-1]},\ \underbrace{dp[i][j-1]}_{\text{插入 }b[j-1]},\ \underbrace{dp[i-1][j-1]}_{\text{替換}}\big)$$
  3. base case:dp[i][0] = i(把長度 i 的字串砍成空字串,要刪 i 次); dp[0][j] = j(從空字串補出長度 j,要插 j 次)。
  4. 順序:bottom-up,i、j 由小到大。
  5. 回推:從 dp[m][n] 倒推每一步走的是刪/插/替/不動,即可還原「具體怎麼改」。
def edit_distance(a, b):
    m, n = len(a), len(b)
    dp = [[0] * (n + 1) for _ in range(m + 1)]
    for i in range(m + 1):
        dp[i][0] = i                    # a 砍成空字串
    for j in range(n + 1):
        dp[0][j] = j                    # 空字串補成 b
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if a[i - 1] == b[j - 1]:
                dp[i][j] = dp[i - 1][j - 1]         # 字元相同,免費
            else:
                dp[i][j] = 1 + min(dp[i - 1][j],    # 刪除
                                   dp[i][j - 1],    # 插入
                                   dp[i - 1][j - 1])  # 替換
    return dp[m][n]

print(edit_distance("horse", "ros"))    # 3
print(edit_distance("kitten", "sitting"))  # 3(經典範例)

ℹ️
還有一整個「區間 DP」家族

本章的題目狀態都靠「索引/容量」描述。還有一類 DP 的狀態是「一段區間 [i, j]」, 轉移時枚舉「在哪裡把區間切開」,稱為區間 DP—— 經典代表是矩陣鏈相乘(matrix-chain multiplication):決定一串矩陣相乘時加括號的順序, 讓乘法次數最少。它的轉移式 $dp[i][j] = \min_{i \le k \lt j}\big(dp[i][k] + dp[k+1][j] + \text{合併成本}\big)$ 就是「枚舉切點」的招牌長相。本章先點到為止,知道有這一支就好。

DP 不只在陣列上:圖上的 DP

到目前為止,我們的「表」都是一維或二維陣列。但 DP 的思想其實更廣—— 第 30 章你會學到兩個重量級的最短路徑演算法,它們骨子裡都是動態規劃,只是換了個舞台。

Bellman-Ford:對「邊數」做 DP

Bellman-Ford 求單一起點到所有點的最短路徑(且容許負權邊)。它的 DP 觀點是: 定義 dist[v] 為「最多用 k 條邊」時從起點到 v 的最短距離。 每多放寬一條邊(k 加 1),就把所有邊「鬆弛(relax)」一次:若 dist[u] + w(u,v) 比現有的 dist[v] 小就更新。 因為任何最短路徑最多 $V-1$ 條邊,放寬 $V-1$ 輪就收斂。這就是一個「對邊數遞增」的 DP: 

def bellman_ford(n, edges, src):
    INF = float('inf')
    dist = [INF] * n
    dist[src] = 0
    for _ in range(n - 1):              # 「允許的邊數」由 1 增到 n-1(DP 的層)
        for u, v, w in edges:
            if dist[u] != INF and dist[u] + w < dist[v]:
                dist[v] = dist[u] + w   # 鬆弛 = 一次狀態轉移
    return dist

Floyd-Warshall:對「允許的中繼點」做 DP

Floyd-Warshall 一口氣求出所有點對之間的最短路徑,它的 DP 狀態定義堪稱教科書級的優雅: dp[k][i][j] = 「只准用編號 $\le k$ 的點當中繼站」時,i 到 j 的最短距離。 轉移時問一個二選一的問題:這條最短路要不要經過第 k 個點?

$$dp[k][i][j] = \min\big(\underbrace{dp[k-1][i][j]}_{\text{不經過 }k},\ \ \underbrace{dp[k-1][i][k] + dp[k-1][k][j]}_{\text{經過 }k}\big)$$ 因為每一層只依賴上一層,實作時可以把 k 那一維省掉,就地更新成三層迴圈: 

def floyd_warshall(dist):
    # dist[i][j]:i→j 的邊權;無邊為 INF,對角線為 0。函式直接就地更新
    n = len(dist)
    for k in range(n):                       # 最外層:逐一「開放」第 k 個中繼點
        for i in range(n):
            for j in range(n):
                if dist[i][k] + dist[k][j] < dist[i][j]:
                    dist[i][j] = dist[i][k] + dist[k][j]
    return dist

注意那個 k 必須在最外層的迴圈順序——它正對應 DP 的「層」:你必須先把「用到第 k 點」的所有結果算完, 才能往第 k+1 點推進。這跟費氏「先填小的格子」是同一個道理,只是維度更高。第 30 章會把這兩個演算法講透; 這裡你只需要帶走一個震撼的認知:圖上的最短路徑,本質上也是在填一張(看不見的)DP 表。

🔑
DP 是一種「世界觀」,不是一種資料結構

別把動態規劃跟「二維陣列」綁死。DP 的本質是:把問題拆成有重疊的子問題、把子問題答案記下來、按依賴順序求解。 至於那張「表」長什麼樣——一維陣列、二維矩陣、圖的距離矩陣、甚至是一個 dict——都只是載體。 一旦你習慣用「狀態 + 轉移」的眼光看世界,你會在最意想不到的地方認出 DP 的身影。

通用流程:從一支暴力遞迴,長成一個 DP

講了這麼多題,最後把它收束成一條人人可學、照著走就行的升級流水線。面對一個新的最佳化/計數問題, 不要一開始就硬想 DP——按這四步走,DP 會自己浮現出來:

  1. 先寫一支「對的」暴力遞迴。別管效率,先確保它答案正確。 通常就是「枚舉最後一步的所有選擇,遞迴解剩下的子問題,取最佳/加總」。寫對它,你就同時得到了狀態(函式的參數)與轉移(函式的本體)——這是最難的一步,而你已經在不知不覺中完成了。
  2. 加上記憶化。看那支遞迴「靠哪幾個參數決定回傳值」,用那幾個參數當 key 加一本快取(或直接 @lru_cache)。 一行裝飾器,$O(2^n)$ 常常就變 $O(n)$ 或 $O(n^2)$。九成情況,到這裡就夠快、可以收工了。
  3. (需要時)翻成表格法。若你想擺脫遞迴堆疊、或要做空間優化,就把記憶化「反過來」: 找出子問題的依賴順序,用迴圈由小到大填表。
  4. (需要時)優化空間。觀察「每一格只依賴最近的哪幾格」,把整張表壓成滾動陣列,從 $O(n)$、$O(nW)$ 降到 $O(1)$、$O(W)$。
🔑
DP 其實「人人可學」

動態規劃會嚇人,是因為很多教材一上來就丟你一個填好的二維表,要你「看出」轉移式——那當然像變魔術。 但真正的解題現場不是這樣。你從一支老實的暴力遞迴開始,它的參數就是狀態、它的本體就是轉移; 加一行快取,它就成了 DP。你不需要天才般的靈光一閃,你只需要會寫遞迴,然後不重複做功。 這,就是 DP 之所以人人可學的祕密。

動手練習

DP 是徹底的「做中學」。以下八題難度遞增,請務必先寫一支會動的暴力遞迴,再優化成記憶化或表格法。 每一題,先逼自己回答五步框架的前三步:狀態是什麼?轉移怎麼寫?base case 在哪?卡住了再展開解答。

✏️ 練習 32.1 實作

把第 3 章那支樸素遞迴費氏 fib,分別用(a) 手動 dict 記憶化與(b) @lru_cache兩種方式加速, 並說明為什麼複雜度從 $O(2^n)$ 變成 $O(n)$。

看解答 
from functools import lru_cache

# (a) 手動 dict
def fib_memo(n, memo=None):
    if memo is None:
        memo = {}
    if n < 2:
        return n
    if n not in memo:
        memo[n] = fib_memo(n - 1, memo) + fib_memo(n - 2, memo)
    return memo[n]

# (b) 裝飾器
@lru_cache(maxsize=None)
def fib_lru(n):
    if n < 2:
        return n
    return fib_lru(n - 1) + fib_lru(n - 2)

print(fib_memo(50), fib_lru(50))   # 12586269025 12586269025

原本每個 fib(k) 會被重算無數次(遞迴樹指數膨脹);加了快取後,每個不同的 k 只被真正計算一次,之後都是 $O(1)$ 查表。一共只有 $n+1$ 個不同的子問題,所以總時間 $O(n)$。

✏️ 練習 32.2 實作

爬樓梯變形:現在每次可以跨 1、2 或 3 階。寫 climb3(n) 算出爬到第 n 階的走法數。 先寫暴力遞迴,再優化。

看解答

狀態 dp[i] = 到第 i 階的走法數;最後一步可從 i-1、i-2、i-3 上來, 所以 $dp[i] = dp[i-1] + dp[i-2] + dp[i-3]$。base case:dp[0]=1,負數階記為 0。

def climb3(n):
    dp = [0] * (n + 1)
    dp[0] = 1
    for i in range(1, n + 1):
        dp[i] = dp[i - 1]
        if i >= 2:
            dp[i] += dp[i - 2]
        if i >= 3:
            dp[i] += dp[i - 3]
    return dp[n]

print(climb3(4))   # 7

這串數列叫「Tribonacci」——費氏的三項版。空間可同樣優化成只留三個變數。

✏️ 練習 32.3 實作

硬幣找零最少數:實作 coin_change(coins, amount),回傳湊出 amount 的最少硬幣數,湊不出回 -1。 並舉一個貪婪會給出錯誤答案的幣制與金額,驗證你的 DP 比貪婪好。

看解答 
def coin_change(coins, amount):
    INF = amount + 1
    dp = [0] + [INF] * amount
    for a in range(1, amount + 1):
        for c in coins:
            if c <= a:
                dp[a] = min(dp[a], dp[a - c] + 1)
    return dp[amount] if dp[amount] != INF else -1

print(coin_change([1, 3, 4], 6))   # 2  (= 3+3)
# 貪婪反例:coins=[1,3,4], amount=6
# 貪婪先拿 4,再拿兩個 1 → 3 枚;DP 找到 3+3 → 2 枚,貪婪輸。

進階:再加一個 parent[] 記錄每格「最後一枚拿了哪種面額」,即可回推「實際用了哪些硬幣」。

✏️ 練習 32.4 實作

最大不相鄰子序列和(打家劫舍):給一排房子各有金額 nums[i], 你不能搶相鄰的兩間(會觸發警報)。求能搶到的最大總金額。例如 [2, 7, 9, 3, 1] 答案是 12(搶第 1、3、5 間:2+9+1)。

看解答

狀態 dp[i] = 考慮前 i 間房子的最大金額。對第 i 間,要嘛不搶(沿用 dp[i-1])、 要嘛搶(只能再加 dp[i-2],因為 i-1 不能碰):$dp[i] = \max(dp[i-1],\ dp[i-2] + nums[i])$。

def rob(nums):
    prev, cur = 0, 0           # prev = dp[i-2], cur = dp[i-1]
    for x in nums:
        prev, cur = cur, max(cur, prev + x)
    return cur

print(rob([2, 7, 9, 3, 1]))   # 12

注意它又是「只依賴前兩格」,所以直接用兩個變數滾動,空間 $O(1)$。這題和爬樓梯是同一個骨架的近親。

✏️ 練習 32.5 挑戰

0/1 背包 + 回推:在本章的 knapsack 上,除了回傳最大價值,還要回傳實際選了哪些物品(物品索引清單)。

看解答

先照常填二維表,再從 dp[n][W] 倒著走:若 dp[i][w] != dp[i-1][w],代表第 i 件被拿了, 記下它並把容量扣掉 w_i;否則沒拿,直接往上一列。

def knapsack_items(weights, values, W):
    n = len(weights)
    dp = [[0] * (W + 1) for _ in range(n + 1)]
    for i in range(1, n + 1):
        wi, vi = weights[i - 1], values[i - 1]
        for w in range(W + 1):
            dp[i][w] = dp[i - 1][w]
            if wi <= w:
                dp[i][w] = max(dp[i][w], dp[i - 1][w - wi] + vi)

    # 回推:從右下角倒走
    chosen, w = [], W
    for i in range(n, 0, -1):
        if dp[i][w] != dp[i - 1][w]:    # 值有變 → 第 i 件被拿了
            chosen.append(i - 1)        # 記下 0-based 索引
            w -= weights[i - 1]
    chosen.reverse()
    return dp[n][W], chosen

print(knapsack_items([1, 3, 4, 5], [1, 4, 5, 7], 7))   # (9, [1, 2])

結果 (9, [1, 2]):選了第 2、3 件(0-based 的 1、2),正是圖 32-5 回推出的 {②, ③}。

✏️ 練習 32.6 挑戰

LCS + 還原字串:在本章的 lcs_length 上,除了長度,還要回傳一個實際的最長共同子序列字串。 (想想 git diff 是怎麼靠它找出共同保留的行的。)

看解答

填完表後,從 dp[m][n] 倒推:字元相同就把它記下並往左上走;否則往「值較大的那一邊」(上或左)走。 最後把收集到的字元反轉。

def lcs(X, Y):
    m, n = len(X), len(Y)
    dp = [[0] * (n + 1) for _ in range(m + 1)]
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if X[i - 1] == Y[j - 1]:
                dp[i][j] = dp[i - 1][j - 1] + 1
            else:
                dp[i][j] = max(dp[i - 1][j], dp[i][j - 1])

    # 回推出實際的 LCS
    i, j, out = m, n, []
    while i > 0 and j > 0:
        if X[i - 1] == Y[j - 1]:
            out.append(X[i - 1])
            i, j = i - 1, j - 1
        elif dp[i - 1][j] >= dp[i][j - 1]:
            i -= 1
        else:
            j -= 1
    return dp[m][n], "".join(reversed(out))

print(lcs("ABCBDAB", "BDCAB"))   # (4, 'BCAB')

不同的「平手時往哪走」策略會還原出不同(但等長)的 LCS,這沒有對錯,都是合法答案。

✏️ 練習 32.7 挑戰

編輯距離:實作 edit_distance(a, b)(插入 / 刪除 / 替換各算一次操作), 並驗證 edit_distance("kitten", "sitting") == 3。先寫暴力遞迴,再加記憶化或改填表。

看解答

先寫貼近定義的暴力遞迴(它的兩個參數 i, j 直接就是狀態):

from functools import lru_cache

def edit_distance(a, b):
    @lru_cache(maxsize=None)
    def go(i, j):
        if i == 0:               # a 用完 → 把 b 剩下的全插入
            return j
        if j == 0:               # b 用完 → 把 a 剩下的全刪除
            return i
        if a[i - 1] == b[j - 1]:
            return go(i - 1, j - 1)
        return 1 + min(go(i - 1, j),      # 刪除
                       go(i, j - 1),      # 插入
                       go(i - 1, j - 1))  # 替換
    return go(len(a), len(b))

print(edit_distance("kitten", "sitting"))   # 3

這就是「先寫對的暴力遞迴 → 加一行 @lru_cache → 立刻變 $O(mn)$ DP」的完美示範。 暴力遞迴的參數 (i, j) 就是表的兩個維度,你幾乎沒額外動腦就得到了 DP。

✏️ 練習 32.8 挑戰

分割等和子集:給一個正整數陣列,判斷能否把它分成兩個總和相等的子集。 例如 [1, 5, 11, 5] 可以({1,5,5} 與 {11} 都是 11),[1, 2, 3, 5] 不行。 (提示:這其實是一個偽裝的子集和 / 0-1 背包問題。)

看解答

總和為奇數時直接不可能。否則令 target = 總和 / 2,問題化為「能否挑出一個子集,和恰為 target」—— 這是布林版的 0/1 背包(物品就是數字,容量就是 target)。 用一維 dp[s] 表示「能否湊出和 s」,內層容量一樣要從大到小跑:

def can_partition(nums):
    total = sum(nums)
    if total % 2 != 0:
        return False
    target = total // 2
    dp = [False] * (target + 1)
    dp[0] = True                          # 和為 0 永遠湊得出(什麼都不選)
    for x in nums:
        for s in range(target, x - 1, -1):   # 由大到小,避免重複用同一個數
            dp[s] = dp[s] or dp[s - x]
    return dp[target]

print(can_partition([1, 5, 11, 5]))   # True
print(can_partition([1, 2, 3, 5]))    # False

看出 0/1 背包的影子了嗎?「能不能裝滿容量 target」就是把背包的 max(價值) 換成 or(可行性)。 認出「這題其實是背包」,正是 DP 練到後期最有價值的能力。

把這一章串起來

我們從第 3 章那個算到天荒地老的費氏數列出發,一路把它治好,並沿途把動態規劃的全貌攤了開來。回顧幾個關鍵路標:

  • 樸素遞迴費氏之所以 $O(2^n)$,病根是重複計算重疊的子問題。DP 的處方就一句話:算過的記下來,別做第二遍。
  • 記憶化(top-down):幾乎不改原遞迴,加一本快取(dict 或 @lru_cache),指數塌縮成線性。
  • 表格法(bottom-up):從最小子問題由小到大填表,沒有堆疊風險、最容易做空間優化(滾動陣列)。
  • DP 成立需要兩個前提:最佳子結構(遞推式才正確)與重疊子問題(記憶化才划算)。它和分治(子問題不重疊)、貪婪(不回頭)各有分工。
  • 五步框架:定義狀態 → 寫轉移方程 → 確定 base case → 決定計算順序 → 必要時回推解。每題都這樣走一遍。
  • 經典題演練:爬樓梯(費氏換衣服)、硬幣找零(貪婪翻車)、0/1 背包(分治/分數背包對照)、LCS(diff 的心臟)、LIS、編輯距離(拼字檢查/DNA)。
  • DP 不限於陣列:Bellman-Ford 對邊數做 DP、Floyd-Warshall 對中繼點做 DP——圖上的最短路其實也在填表。
  • 通用流程:先寫對的暴力遞迴 → 加記憶化 → 必要時改表格 → 優化空間。暴力遞迴的參數就是狀態,它的本體就是轉移——DP 因此人人可學。

如果這一章你只帶走一句話,讓它是這句:不要害怕 DP,先去寫一支會動的暴力遞迴,再叫它別重複做功。 你會發現,那個曾經像魔法的「動態規劃」,不過是「會記筆記、肯回頭比較」的窮舉而已。 下一章,我們要走向演算法世界的邊界——有些問題,連 DP 這樣聰明的窮舉也救不了, 它們是傳說中的「NP 完備」難題。系好安全帶。
