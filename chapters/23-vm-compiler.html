最後一段樓梯

我們已經爬得很高了。第 19、20 章用一堆 NAND 閘親手造出了 ALU 與記憶體;第 21 章把它們組成 CPU,看懂了機器碼; 第 22 章寫了一個組譯器(assembler),讓 D=D+A 自動變成 1110000010010000。 從組合語言到機器碼這一哩,已經通了。

但說真的——你願意用組合語言寫一個貪食蛇嗎?光是「如果分數超過最高分就更新」這種小事, 用 @、D=M、0;JMP 拼起來就要寫一大串,還得自己記住每個變數借住在 RAM 的哪一格。 人類真正想寫的,是這樣的東西: 

if (score > high) {
    high = score;
}

從這種高階語言(high-level language)的「人話」,到 CPU 真正吃的那串 0 與 1, 中間還隔著好幾層樓梯。這一章,我們要把這段樓梯一階一階鋪好。而第一個問題,出乎意料地不是「怎麼翻譯」, 而是——「翻譯成什麼?」

為什麼要多一層:M×N 的惡夢

最直覺的想法是:寫一個編譯器,把高階語言直接翻成某顆 CPU 的機器碼。聽起來很合理,直到你開始數數量。

假設世界上有 M 種高階語言(C、Java、Python、Go、Rust……)和 N 種CPU 平台 (x86、ARM、RISC-V、我們的 Hack……)。如果每個語言都直接翻到每個平台,你需要幾個編譯器? 答案是 M×N 個——每多一種語言,就得為「所有」平台各寫一個後端;每出一顆新晶片,就得為「所有」語言各補一個編譯器。 這是一場組合爆炸的惡夢。

聰明的解法,是電腦科學最愛用的一招:多加一層中間層(intermediate layer)。我們發明一種人造的、 不對應任何真實 CPU 的虛擬機語言(VM language,也叫位元組碼 bytecode),讓它夾在中間:

  • 前端(front end):每種高階語言只要寫一個編譯器,把自己翻成這套通用的 VM 語言。
  • 後端(back end):每個平台只要寫一個 VM 翻譯器,把 VM 語言翻成自己的機器碼。

於是總數從 M×N 驟降為 M+N。多一種語言?寫一個前端就好,它立刻能在所有平台上跑。 出一顆新晶片?寫一個後端就好,所有語言立刻能用它。

🔑
兩階段編譯(two-tier compilation)

整個翻譯過程被切成兩段,中間以一套虛擬機語言為樞紐: ① 編譯器(compiler):高階語言 → VM 語言(與硬體無關)。 ② VM 翻譯器(VM translator):VM 語言 → 某平台的機器碼(與語言無關)。 這條界線把「語言的事」和「硬體的事」乾淨地切開——這正是關注點分離(separation of concerns)的經典範例, 也是 Java、C# 等現代語言能「跨平台」的根本原因。

這個想法不是教科書上的玩具。Java 的編譯器把 .java 翻成 .class(Java 位元組碼), 再交給各平台的 JVM(Java Virtual Machine)去執行;微軟 .NET 把 C#、F# 等語言翻成 CIL 位元組碼,交給 CLR(Common Language Runtime)。這就是 Java 那句著名口號 「Write Once, Run Anywhere(一次編譯,到處執行)」的祕密:你編譯出來的位元組碼不認得任何特定 CPU, 到處執行的是那台虛擬的機器,而每個平台都備了一台它的「替身」。

那麼問題來了:這台虛擬機,該長什麼樣?它要簡單到「每個平台都好實作」,又要通用到「什麼語言都翻得進來」。 Nand2Tetris 給的答案,跟絕大多數真實 VM(包括 JVM)一樣,是一個優雅得驚人的設計——堆疊機。

堆疊:一疊盤子的魔法

在設計虛擬機之前,我們得先認識它的心臟:堆疊(stack)。 你其實天天在用它的實體版——一疊餐盤。新洗好的盤子疊到最上面;要用盤子時,也從最上面拿。 你永遠不會從中間抽一個。這種「最後放進去的、最先被拿出來」的規則,叫做 後進先出(LIFO,Last In, First Out)。

堆疊只開放兩個動作(這也是它美妙的地方——簡單到不能再簡單):

  • push(壓入):把一個值放到堆疊頂端。
  • pop(彈出):把頂端的值拿走,並回傳它。

就這兩招。沒有「讀第三個」、沒有「插到中間」。正因為這麼受限,它反而出奇地好用—— 尤其是拿來做算術運算。

為什麼堆疊適合做算術

想算 7 + 8。在堆疊機(stack machine)上,這件事被拆成三個極簡的動作: push 7(把 7 壓進去)、push 8(把 8 壓進去)、add (把頂端兩個值彈出來、相加、把結果壓回去)。我們把這三步畫出來:

注意一件關鍵的事:add 這個命令不帶任何參數。它不需要說「把誰跟誰相加」, 因為遊戲規則就是「永遠拿頂端那兩個」。這種「運算元位置固定在頂端」的特性,讓堆疊機的指令短到不能再短, 也讓翻譯變得超級規律——這正是我們選它當虛擬機的原因。

堆疊機怎麼吃下複雜算式

厲害的還在後頭。複雜算式 (7 + 8) * 3 怎麼辦?一樣只用 push 與運算: 

push 7      // 堆疊:[7]
push 8      // 堆疊:[7, 8]
add         // 彈 8、彈 7,壓回 15      → 堆疊:[15]
push 3      // 堆疊:[15, 3]
mul         // 彈 3、彈 15,壓回 45     → 堆疊:[45]
            // 結束時堆疊頂端 = 答案 45

每個運算子都「就地」消化掉它下方剛算好的中間結果,完全不需要暫存變數、不需要管「先算哪個」—— 因為我們在產生這串命令時,就已經把順序安排好了。這種寫法,其實有個古老的名字。

📜
逆波蘭表示法(Reverse Polish Notation)

把運算子寫在運算元後面的寫法(7 8 + 而非 7 + 8),叫後序(postfix)或 逆波蘭表示法(RPN),1920 年代由波蘭邏輯學家 Jan Łukasiewicz 的前序記法演變而來。 它最迷人的地方是:完全不需要括號,運算順序由位置唯一決定。1970 年代的 HP 工程計算機就用 RPN 輸入, 至今仍有死忠擁護者。而堆疊機執行的,本質上就是一串 RPN——本章最後一個練習,你會親手寫一台 RPN 計算機。

Hack VM 語言:兩類命令搞定一切

現在來看 Nand2Tetris 實際定義的虛擬機語言。它跟 Hack 機器語言一樣走極簡路線,整套只有兩大類命令: 算術/邏輯命令(操作堆疊頂端)與記憶體存取命令(push/pop)。先看第一類。

算術與邏輯命令

這九個命令全都不帶參數,清一色是「從堆疊頂端拿值、算、把結果放回去」:

命令拿幾個做什麼等價算式(x 在下、y 在上)
add2相加x + y
sub2相減x − y
neg1取負−y
eq2是否相等(真壓 −1,假壓 0)x == y
gt2是否大於x > y
lt2是否小於x < y
and2逐位元 ANDx & y
or2逐位元 ORx | y
not1逐位元 NOT~y

留意一個常見的坑:sub 算的是「下面那個減上面那個」(x − y),不是反過來。順序很重要, 待會的練習會專門考你這點。另外注意比較命令(eq/gt/lt)的「真」是 −1(十六個位元全 1), 「假」是 0——這跟第 20 章布林邏輯的慣例一致。

記憶體存取:push / pop 與 segment

光有算術還不夠——數字得從某個地方來、算完得存到某個地方去。這就是另一類命令: push 與 pop。但這裡有個關鍵設計:它們的格式是 

push segment index
pop  segment index

segment(段)指出「從哪一塊記憶體區域」,index 是「那塊區域裡的第幾格」。 為什麼要分段?因為高階語言裡的變數,天生就分好幾種角色:有區域變數、有函式引數、有物件欄位…… VM 用段把它們分門別類,各住各的區,翻譯時才不會打架。Hack VM 有八個段:

段(segment)裝的是對應的高階概念
constant常數本身(虛擬的段,只能 push)字面值,如 push constant 7 就是把 7 壓進去
local目前函式的區域變數var 宣告的變數
argument呼叫時傳進來的引數函式的參數
this / that目前物件 / 陣列指到的記憶體物件欄位、陣列元素
static整個檔案共用的靜態變數類別層級的 static 變數
temp8 格通用暫存空間編譯器產生的臨時值
pointer2 格,用來設定 this/that 指向哪物件/陣列的位址操作

實際用起來,直覺很單純。push local 2 是「把第 2 號區域變數的值,壓到堆疊頂端」; pop local 2 是「把堆疊頂端彈出,存進第 2 號區域變數」。來看一個完整小例子: 把區域變數 local 0 與引數 argument 1 相加,再存回 local 0 (相當於高階語言的 x = x + n): 

push local 0      // 把 x 壓上來        堆疊:[x]
push argument 1   // 把 n 壓上來        堆疊:[x, n]
add               // 相加              堆疊:[x+n]
pop local 0       // 彈出存回 x         堆疊:[]
🔑
堆疊是「動詞」,段是「名詞」

看懂 VM 程式,只要記住兩件事:算術命令在堆疊頂端「做事」(動詞), push/pop 在堆疊與各個記憶體段之間「搬東西」(把名詞搬上台、搬下台)。 任何高階運算式,最後都被攤平成一串「push 一些值上來 → 用算術命令把它們消化掉」。 一旦抓到這個節奏,VM 程式就再也不神祕了。

函式呼叫協定:堆疊真正的殺手級應用

算術只是開胃菜。堆疊真正不可取代的地方,是函式呼叫。 回想第 3 章我們講遞迴時提到的呼叫堆疊(call stack)——當時是個比喻,現在我們要看見它的本體。

問題是這樣的:函式 A 呼叫函式 B,B 又呼叫 C……每一層都有自己的引數、自己的區域變數, 而且做完之後要能準確回到呼叫它的地方繼續跑。這些「每一層自己的東西」,該放哪? 答案優雅得不得了:放在堆疊上。每呼叫一次函式,就在堆疊頂端疊一塊新的 呼叫框(stack frame,也叫 activation record);函式回傳時,就把這塊框拆掉。

Hack VM 用三個命令完成這套協定:

  • function f k:宣告函式 f,它有 k 個區域變數(進入時會把這 k 格初始化為 0)。
  • call f n:呼叫 f,此時堆疊頂端已經放好 n 個引數。
  • return:把堆疊頂端的值當回傳值,拆掉呼叫框,跳回呼叫處。

call 背後做的事,概念上是:把「回返位址」與現在的 LCL/ARG/THIS/THAT 全部 push 到堆疊上保存, 然後重新設定 ARG(指向第一個引數)與 LCL(指向第一個區域變數),最後跳進函式。 return 則是反向操作:把回傳值蓋到 argument 0 的位置(呼叫者預期答案出現在那)、還原四個被保存的指標、 跳回返位址。你不需要背這些細節,但要抓到那個對稱的美感:呼叫時疊上去的,回傳時剛好一層層撤下來。

為什麼遞迴「天生」就能跑

這套協定最漂亮的回報,是遞迴自動成立。第 3 章我們算階乘 factorial(n) 時, 它會呼叫 factorial(n-1)、再呼叫 factorial(n-2)……每一層都在「同一個函式」裡跑, 卻各自有不同的 n。怎麼做到的?

因為每一次呼叫都會疊一塊全新、獨立的呼叫框。factorial(3) 的 n=3 住在它那塊框, factorial(2) 的 n=2 住在上面新疊的那塊,兩者井水不犯河水。 當最內層 factorial(1) 算完回傳,它的框被拆掉,控制權落回 factorial(2) 的框—— 那個 n=2 還好端端地等在那裡。堆疊的後進先出,恰好對應「最後呼叫的最先返回」。

🔑
遞迴不是語言的恩賜,是堆疊的副產品

很多初學者覺得「遞迴能跑」是一件神奇、要特別設計的事。其實不然:只要你的呼叫機制是 用堆疊保存每一層的現場,遞迴就自動能跑——因為「同一個函式的不同次呼叫」不過是堆疊上幾塊不同的框而已, 機器根本分不出、也不需要分出它們是不是「同一個」函式。這也解釋了第 3 章那個警告: 遞迴太深會堆疊溢位(stack overflow)——因為框疊太多,把堆疊空間用光了。 那個著名網站的名字,就是這麼來的。

VM 翻譯器:把 VM 命令落地成組合語言

VM 語言再怎麼漂亮,Hack CPU 也聽不懂——它只認得第 21 章那兩種機器指令。所以我們需要 VM 翻譯器(VM translator):讀進一行行 VM 命令,把每一條翻成幾條第 22 章的 Hack 組合語言。 (注意:跟組譯器「一行組語 = 一條機器碼」的一對一不同,VM 翻譯是一對多——一條 VM 命令通常展開成好幾條組語。)

關鍵前提:在 Hack 的記憶體裡,有一個約定俗成的堆疊指標 SP(存在 RAM[0]), 它永遠指向「堆疊頂端的下一個空格」。所有 push/pop 都圍繞著它打轉。先看 push constant 7 怎麼翻: 

// VM:  push constant 7
@7          // A = 7
D=A         // D = 7              (把常數放進 D)
@SP         // A = 0(SP 的位址)
A=M         // A = RAM[SP] = 堆疊頂端那一格的位址
M=D         // *SP = 7            (把 7 寫進堆疊頂端)
@SP
M=M+1       // SP++              (堆疊長高一格)

讀懂它:就是「把 7 放到 SP 指的那格,然後 SP 加一」。 每個 push 的最後兩步永遠是 @SP / M=M+1,規律到你可以閉著眼睛寫。再看算術命令 add: 

// VM:  add
@SP
AM=M-1      // SP--,同時 A = 新的 SP(指向頂端的 y)
D=M         // D = y             (頂端那個值)
A=A-1       // A 往下一格 → 指向 x
M=D+M       // x = x + y         (結果留在這格,它現在是新的頂端)

add 的精髓:把堆疊「縮短一格」(兩個運算元變一個結果),並把和留在原本 x 的位置。 sub 只要把最後一行改成 M=M-D(注意是 x − y,印證前面說的順序)。 其他算術命令都是這個模板的小變奏。

看出門道了嗎?VM 翻譯器的工作,大半是查表 + 套模板:每種 VM 命令對應一個固定的組語樣板, 把參數填進去就好。它不「思考」,只是機械地展開——這跟第 22 章的組譯器一脈相承,也跟第 21 章那句 「控制單元其實不太聰明,只是佈線」遙相呼應。整條翻譯鏈,都是這種「規律的笨」堆出來的奇蹟。

ℹ️
VM 翻譯 vs 直譯

我們這裡講的是提前翻譯(ahead-of-time):VM 命令先被翻成機器碼,再執行。 但 VM 也可以直譯(interpret):寫一個程式,一條一條讀 VM 命令、當場「假裝執行」它(就像本章開頭的 堆疊操作那樣維護一個堆疊陣列)。早期的 JVM 是直譯的,所以 Java 曾以「慢」聞名; 後來引入 JIT(Just-In-Time)即時編譯——執行時才把熱點位元組碼翻成原生機器碼——速度才大幅追上。 翻譯與直譯的取捨,我們留到第十部編譯器那章再細談。

往上一層:高階語言長什麼樣(Jack)

VM 語言已經比組合語言友善多了,但它仍然在「攤平」的層次——你還是得自己想清楚 push/pop 的順序。 真正讓人舒服的,是高階語言。Nand2Tetris 配了一個叫 Jack 的小語言, 它刻意設計得像一個簡化版的 Java,讓你一眼就有熟悉感。看一段 Jack: 

class Main {
    function void main() {
        var int sum, i;       // 宣告區域變數
        let sum = 0;
        let i = 1;
        while (i < 101) {      // 1 加到 100
            let sum = sum + i;
            let i = i + 1;
        }
        do Output.printInt(sum);   // 呼叫函式(印出結果)
        return;
    }
}

即使你沒學過 Jack,大概也猜得到它在幹嘛——這就是高階語言的價值:它用人類熟悉的結構 (類別 class、函式 function、變數宣告 var、流程控制 while/if、賦值 let)說話, 而不是逼你管堆疊指標。Jack 也支援物件導向(第 8 章的概念),例如定義一個「點」: 

class Point {
    field int x, y;                    // 物件的欄位

    constructor Point new(int ax, int ay) {  // 建構子
        let x = ax;
        let y = ay;
        return this;
    }

    method int getX() { return x; }    // 方法
}

重點不是學會 Jack 的每個語法,而是體會:這層東西,終究會被編譯成上面那種 VM 命令。 let sum = sum + i; 會變成 push local 0 / push local 1 / add / pop local 0; while 會變成「比較 + 條件跳躍」(就像第 21 章看到的); do Output.printInt(sum) 會變成 push 引數再 call Output.printInt 1。 把這層「人話」翻成 VM 命令的那個程式,就是編譯器。它到底怎麼做到的?

編譯器在做什麼:一條三段式的管線

① 詞法分析:把字元串切成 token

編譯器拿到的原始碼,本質上只是一長串字元:l、e、t、空格、x…… 它眼中沒有「單字」,得自己切。詞法分析(lexical analysis,俗稱 tokenizing 或 lexing) 就是把這串字元切成一個個有意義的最小單位——token(詞元),並標上類別。常見類別有:

  • 關鍵字(keyword):語言保留的字,如 class、if、while、let。
  • 符號(symbol):+、*、(、;、= 等。
  • 識別字(identifier):程式設計師自取的名字,如變數名 x、函式名 main。
  • 常數(constant):數字 42、字串 "hello"。

這一步其實不難——掃過字元、遇到空白就跳過、遇到數字就一路吃到不是數字為止、遇到符號就單獨切出來。 我們用 Python 替算術運算式寫一個迷你 tokenizer: 

def tokenize(text):
    """把算術運算式的字元串,切成 (類別, 值) 的 token 串列。"""
    tokens = []
    i = 0
    while i < len(text):
        c = text[i]
        if c.isspace():               # 空白:跳過
            i += 1
        elif c.isdigit():             # 數字:一路吃到不是數字
            num = ""
            while i < len(text) and text[i].isdigit():
                num += text[i]
                i += 1
            tokens.append(("NUM", int(num)))
        elif c.isalpha():             # 字母開頭:識別字(變數名)
            name = ""
            while i < len(text) and text[i].isalnum():
                name += text[i]
                i += 1
            tokens.append(("ID", name))
        elif c in "+-*/()":           # 符號
            tokens.append(("OP", c))
            i += 1
        else:
            raise ValueError(f"非法字元:{c!r}")
    return tokens

print(tokenize("x + y * 2"))
# [('ID', 'x'), ('OP', '+'), ('ID', 'y'), ('OP', '*'), ('NUM', 2)]

看那行輸出:原本一串沒結構的字元,現在成了一排乾淨、標好類別的 token。 注意 tokenizer 不在乎文法對不對——你餵它 + + ) 它照樣切得出 token。 「這些 token 拼起來合不合語法」是下一站的事。詞法分析只負責「斷詞」,就像把一句沒有空格的英文 切成一個個單字,但還沒讀懂句子。

② 語法分析:把 token 組成一棵樹

有了一排 token,接下來要搞懂它們的結構:誰是主、誰是從、先算誰。這叫 語法分析(syntactic analysis,俗稱 parsing),它的產物是一棵 剖析樹(parse tree),或經過精簡的抽象語法樹(AST,Abstract Syntax Tree)。

為什麼需要樹?因為運算式天生有階層。x + y * 2 不是「從左到右」算的—— 乘法優先,所以要先算 y * 2,再把結果跟 x 相加。這個「先後」的關係,用一棵樹表達最自然:

那 parser 怎麼從一排扁平的 token,長出這棵有階層的樹?最直覺、也最常被手寫編譯器採用的方法,叫 遞迴下降(recursive descent)。它的點子非常符合直覺:為文法裡的每一條規則,寫一個函式; 規則之間互相引用,函式就互相呼叫(包括遞迴呼叫)。先看我們這個小語言的文法(grammar): 

expr   → term (('+' | '-') term)*      // 運算式 = 一串用 +/- 連起來的 term
term   → factor (('*' | '/') factor)*  // term = 一串用 */÷ 連起來的 factor
factor → NUM | ID | '(' expr ')'       // factor = 數字、變數、或括號裡的運算式

這三條規則藏著大玄機:把 *、/ 放在比 +、- 更「裡層」的 term, 就自動讓乘除先算——優先序被編碼進文法的層次裡了。而 factor 又能回頭引用 expr (括號的情況),這個「繞回去」正是「遞迴」下降的由來,也是括號能任意巢狀的原因。

把這三條規則一對一翻成 Python 函式,一個會建出 AST 的 parser 就成形了。我們用巢狀的 tuple 表示樹節點 (運算子, 左子樹, 右子樹): 

class Parser:
    def __init__(self, tokens):
        self.tokens = tokens
        self.pos = 0

    def peek(self):                         # 看目前這個 token(不前進)
        return self.tokens[self.pos] if self.pos < len(self.tokens) else (None, None)

    def eat(self):                          # 吃掉目前這個 token,前進
        tok = self.tokens[self.pos]
        self.pos += 1
        return tok

    def parse_expr(self):                   # expr → term (('+'|'-') term)*
        node = self.parse_term()
        while self.peek() == ("OP", "+") or self.peek() == ("OP", "-"):
            op = self.eat()[1]
            right = self.parse_term()
            node = (op, node, right)        # 把目前的樹,變成新運算子的左子樹
        return node

    def parse_term(self):                   # term → factor (('*'|'/') factor)*
        node = self.parse_factor()
        while self.peek() == ("OP", "*") or self.peek() == ("OP", "/"):
            op = self.eat()[1]
            right = self.parse_factor()
            node = (op, node, right)
        return node

    def parse_factor(self):                 # factor → NUM | ID | '(' expr ')'
        kind, value = self.eat()
        if kind in ("NUM", "ID"):
            return value                    # 葉節點:直接是數字或變數名
        if (kind, value) == ("OP", "("):
            node = self.parse_expr()        # 遞迴!括號裡又是一個完整 expr
            self.eat()                      # 吃掉對應的 ')'
            return node
        raise ValueError(f"語法錯誤,非預期的 token:{(kind, value)}")


tokens = tokenize("x + y * 2")
tree = Parser(tokens).parse_expr()
print(tree)        # ('+', 'x', ('*', 'y', 2))

看最後那行輸出 ('+', 'x', ('*', 'y', 2))——它正是圖 23-5 那棵樹的文字版! + 在最外層(最後算),('*', 'y', 2) 巢在裡面(先算)。 我們什麼優先序規則都沒「特別處理」,優先序卻自己浮現了——因為它早就藏在文法的分層裡。這就是遞迴下降的優雅。

💡
文法不只用來「檢查」,更用來「生成程式」

注意我們做的事:文法的每條規則 → 一個函式;規則的選項(|)→ if 分支; 規則的重複(*)→ while 迴圈;規則引用別的規則 → 函式呼叫。 這種「文法直接變程式碼」的對應關係如此機械,以至於有工具(如 ANTLR、yacc/bison)能讀進文法、 自動幫你生出 parser。你現在手寫的這支,就是它們自動產生的東西的縮小版。

③ 程式碼產生:走訪樹,吐出 VM 命令

最後一站。我們手上有一棵 AST,目標是吐出堆疊機的 VM 命令。這裡有個美到不行的對應: 把運算式樹用「後序」走訪(先左子樹、再右子樹、最後自己),產生的命令順序剛好就是正確的堆疊機程式!

為什麼?因為堆疊機要求「運算子動作前,運算元得先在堆疊上」。後序走訪恰恰保證:處理一個運算子之前, 它的兩個子樹(運算元)都已經先被處理、值已經 push 上堆疊了。於是程式碼產生器短得驚人: 

def generate(node):
    """後序走訪 AST,印出堆疊機 VM 命令。"""
    if isinstance(node, int):             # 數字葉節點
        print(f"push constant {node}")
    elif isinstance(node, str):           # 變數葉節點
        print(f"push {node}")             # (簡化:假裝變數名能直接 push)
    else:                                 # (運算子, 左, 右)
        op, left, right = node
        generate(left)                    # 先把左子樹的值 push 上去
        generate(right)                   # 再把右子樹的值 push 上去
        ops = {"+": "add", "-": "sub", "*": "mul", "/": "div"}
        print(ops[op])                    # 最後輸出運算命令


generate(("+", "x", ("*", "y", 2)))
# push x
# push y
# push constant 2
# mul
# add

盯著那串輸出,跟本章開頭手算的堆疊機程式一模一樣!x + y * 2 被翻成 「push x、push y、push 2、mul、add」——先把 y * 2 算好留在堆疊上,再跟 x 相加。 我們從沒寫過「先算乘法」這條規則,但因為優先序已經長進了樹的形狀、後序走訪又忠實地照樹的形狀產生命令, 正確的順序就自動掉出來了。

真實的 Jack 編譯器只是「規模」更大

✏️ 練習 23.3 思考

用你自己的話解釋:為什麼 Java 能做到「一次編譯,到處執行(Write Once, Run Anywhere)」? 它跟「把 C 程式直接編成 x86 機器碼」這種傳統做法,本質差別在哪?這個設計付出了什麼代價?

看解答

核心機制:中間層。Java 編譯器不把 .java 直接編成某顆 CPU 的機器碼, 而是編成一種與硬體無關的位元組碼(bytecode)(.class)。 這份位元組碼不認得任何真實 CPU——它認得的是一台虛擬的機器:JVM。 每個平台(Windows/Mac/Linux/手機)各自裝一台 JVM 當「替身」,負責把位元組碼翻成(或直譯成)自己的機器碼。

與傳統做法的差別:傳統 C 編譯是「一次編譯,綁死一個平台」——你為 x86 編出的執行檔, 拿到 ARM 上不能跑,得重編。Java 把編譯切成兩段(原始碼→位元組碼、位元組碼→機器碼), 讓「跨平台」這件事,從「重編 M×N 次」變成「各平台寫一台 JVM」。同一份 .class 檔, 到處都能跑,因為到處都有它的虛擬機。

代價:多一層就多一層開銷。位元組碼在執行時還要被翻譯/直譯, 所以原始的 JVM 比原生機器碼慢(這是早年 Java「慢」的惡名來源)。後來靠 JIT 即時編譯(執行時把熱點程式碼即時編成原生碼)大幅縮小了差距, 但啟動時間與記憶體占用仍是它換來「可攜性」付出的學費。.NET 的 CIL/CLR 是完全相同的思路。

✏️ 練習 23.4 實作

擴充本章的 tokenize 函式,讓它能辨認比較運算子 >、<、 以及由兩個字元組成的 >=、<=、==。 (提示:遇到 >、<、= 時,要偷看下一個字元是不是 =, 來決定切出一個字元還是兩個字元的 token。)讓 tokenize("x >= 10") 能正確切出三個 token。

看解答

處理「多字元符號」的通用技巧叫前瞻(lookahead):先看下一個字元,再決定吃幾個。 我們在原本的符號分支前,加一段專門處理可能成對的運算子:

def tokenize(text):
    tokens = []
    i = 0
    while i < len(text):
        c = text[i]
        if c.isspace():
            i += 1
        elif c.isdigit():
            num = ""
            while i < len(text) and text[i].isdigit():
                num += text[i]; i += 1
            tokens.append(("NUM", int(num)))
        elif c.isalpha():
            name = ""
            while i < len(text) and text[i].isalnum():
                name += text[i]; i += 1
            tokens.append(("ID", name))
        elif c in "<>=":                       # 可能是 1 或 2 個字元的運算子
            nxt = text[i+1] if i+1 < len(text) else ""
            if nxt == "=":                       # 偷看到 '=',吃兩個字元
                tokens.append(("OP", c + "="))   # 例如 '>='
                i += 2
            else:
                tokens.append(("OP", c))         # 單獨的 '>' 或 '<'
                i += 1
        elif c in "+-*/()":
            tokens.append(("OP", c)); i += 1
        else:
            raise ValueError(f"非法字元:{c!r}")
    return tokens

print(tokenize("x >= 10"))
# [('ID', 'x'), ('OP', '>='), ('NUM', 10)]
print(tokenize("a < b"))
# [('ID', 'a'), ('OP', '<'), ('ID', 'b')]

這正是真實 tokenizer 每天在做的事:像 ++、-->、<=、!= 這種多字元符號,全靠前瞻來分辨。「貪心地盡量吃長」(maximal munch)是詞法分析的標準原則。

✏️ 練習 23.5 實作

當一回 VM 翻譯器。把下面這條 VM 命令,翻成 Hack 組合語言(可參考本章 push constant 的模板)。 假設常數段。

push constant 5

進階:接著翻 sub,並說明它跟本章 add 模板差在哪一行、為什麼。

看解答

push constant 5 就是把模板裡的 7 換成 5:把 5 放到 SP 指的格子,再讓 SP 加一。

// push constant 5
@5
D=A         // D = 5
@SP
A=M         // A = 堆疊頂端的位址
M=D         // *SP = 5
@SP
M=M+1       // SP++

進階 — sub:它跟 add 模板只差最後一行。 add 的最後一行是 M=D+M(x + y),sub 改成 M=M-D:

// sub
@SP
AM=M-1      // SP--,A 指向頂端的 y
D=M         // D = y
A=A-1       // A 指向下面的 x
M=M-D       // x = x - y   ← 跟 add 唯一的差別

為什麼是 M-D 不是 D-M?因為 D 裝的是頂端的 y、M 是下面的 x, 而 sub 規定算「下面減上面」=x−y。順序寫反就是經典 bug。 你會發現所有二元算術命令(add/sub/and/or)都是這個模板,只換最後那一行的運算—— VM 翻譯器的工作,大半就是這種「查表、套模板」。

✏️ 練習 23.6 挑戰

把本章學到的東西全部串起來,用 Python 寫一台極簡的「算式計算機」,支援 + - * 與括號,能正確處理運算子優先序。例如 calc("2 + 3 * (4 - 1)") 要回傳 11。

(建議走本章那條管線:tokenize → 轉成 RPN(後序)→ 用堆疊求值。 這恰好把「程式碼產生」和「堆疊機執行」兩件事都做了一遍。)

看解答

我們用三個小步驟,正好對應一條迷你編譯+執行管線。中間用經典的 調度場演算法(shunting-yard algorithm,Dijkstra 發明)把中綴轉成 RPN—— 它用一個「運算子堆疊」來處理優先序與括號,本身就是堆疊的漂亮應用。

def tokenize(text):
    """① 詞法分析:切成數字與符號。"""
    tokens, i = [], 0
    while i < len(text):
        c = text[i]
        if c.isspace():
            i += 1
        elif c.isdigit():
            num = ""
            while i < len(text) and text[i].isdigit():
                num += text[i]; i += 1
            tokens.append(int(num))
        elif c in "+-*()":
            tokens.append(c); i += 1
        else:
            raise ValueError(f"非法字元:{c!r}")
    return tokens


def to_rpn(tokens):
    """② 轉後序(調度場演算法),把優先序「攤平」成順序。"""
    prec = {"+": 1, "-": 1, "*": 2}      # * 優先序比 +/- 高
    output, ops = [], []
    for t in tokens:
        if isinstance(t, int):
            output.append(t)
        elif t in prec:
            # 把堆疊上「優先序 >= 自己」的運算子先倒出來
            while ops and ops[-1] != "(" and prec[ops[-1]] >= prec[t]:
                output.append(ops.pop())
            ops.append(t)
        elif t == "(":
            ops.append(t)
        elif t == ")":
            while ops[-1] != "(":
                output.append(ops.pop())
            ops.pop()                    # 丟掉那個 "("
    while ops:
        output.append(ops.pop())
    return output


def eval_rpn(rpn):
    """③ 堆疊機執行:這就是一台 VM!"""
    stack = []
    for t in rpn:
        if isinstance(t, int):
            stack.append(t)              # push 常數
        else:
            y = stack.pop()              # 注意:先彈的是上面(y)
            x = stack.pop()
            if t == "+": stack.append(x + y)
            elif t == "-": stack.append(x - y)
            elif t == "*": stack.append(x * y)
    return stack.pop()


def calc(text):
    return eval_rpn(to_rpn(tokenize(text)))


print(calc("2 + 3 * (4 - 1)"))   # 11
print(calc("(3 + 4) * 5"))       # 35
print(calc("3 + 4 * 5"))         # 23

體會這三段的對應:tokenize 是詞法分析;to_rpn 兼了 語法分析 + 程式碼產生(它直接輸出後序的「堆疊機程式」,優先序處理就藏在那個 while 條件裡); eval_rpn 則是一台堆疊式虛擬機,逐一執行 RPN——跟本章開頭手算 7 + 8 的過程一字不差。

你剛剛,用三十幾行 Python,造了一條「從原始碼到執行結果」的完整迷你工具鏈。 真實的編譯器與 VM,規模大上千倍,骨架卻就是這個。如果你想挑戰更純粹的做法, 也可以改用本章的遞迴下降 parser 直接建 AST、再走訪它求值——殊途同歸。

把這一章串起來

這一章,我們補完了抽象高塔裡「從人話到機器」最長、也最迷人的那段樓梯。回顧路標:

  • 直接把每種語言翻到每種 CPU 是 M×N 的惡夢。插一層虛擬機語言當中介,就降成 M+N——這是 JVM、CLR 跨平台的根本。
  • 堆疊是後進先出的結構,只開放 push/pop。它天生適合做算術(運算元固定在頂端)與函式呼叫(每次呼叫疊一塊呼叫框)。
  • Hack VM 語言只有兩類命令:操作堆疊頂端的算術/邏輯命令,與在堆疊和各段之間搬值的 push/pop。
  • 函式呼叫協定用呼叫框把每層的引數、區域變數、回返位址打包在堆疊上。正因如此,遞迴自動成立——它只是堆疊上幾塊不同的框。
  • VM 翻譯器把每條 VM 命令查表、套模板,展開成數條第 22 章的組合語言。
  • 編譯器是一條三段式管線:詞法分析(字元 → token)→ 語法分析(token → 剖析樹,用遞迴下降,優先序藏進文法分層)→ 程式碼產生(後序走訪樹 → VM 命令)。
  • 把第 19–23 章疊起來:Jack → VM 命令 → Hack 組語 → 機器碼 → 在你親手用 NAND 造的 CPU 上跑。每道箭頭都是一個不「思考」、只「規律地翻譯」的程式。

我們花了五個章節,從一顆 NAND 閘出發,蓋出了一整台能跑高階程式的電腦,還親手寫了讓它讀懂人話的整條翻譯鏈。 這是計算機科學裡最有成就感的一段旅程之一——你不再把電腦當黑盒子,因為這個盒子,是你自己蓋的。

但故事還沒完。我們蓋的這台機器,一次只能跑一個程式、還得自己管記憶體裡每個變數住哪。 真實的電腦能同時跑幾十個程式、管理檔案、分配記憶體、應付當機——這背後有一個更大的指揮者: 作業系統(operating system)。那,就是第五部接下來的故事。
