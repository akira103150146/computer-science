漸進複雜度與分析

同一台電腦、同一段程式,今天跑得飛快,明天換個輸入卻卡到天荒地老。我們究竟在「測量」演算法的什麼?這一章,我們把第 5 章那個「資料變大,時間變幾倍」的直覺,鍛造成一把精密、可證明的數學量尺。

一個無法重現的「實驗」

想像兩位工程師在吵架。小美說她的搜尋程式「只花了 0.3 秒」,小強說他的「只花了 0.1 秒」, 於是大家認定小強的演算法比較好。但等一下——小強用的是公司最新的工作站,小美用的是五年前的筆電; 小強寫的是 C,小美寫的是 Python;小強測的資料剛好讓答案出現在第一筆,小美那次資料則是最壞情況。 這場比較,根本沒有比到「演算法」本身。

這正是第 5 章埋下的伏筆。當時我們用「資料變大 10 倍,時間變幾倍」建立了 Big-O 的直覺, 並答應你:那些 $\exists\, c, n_0$ 的嚴謹數學,留到第 26 章。現在,我們把帳結清。

為什麼「測秒數」這件事這麼不可靠?因為跑出來的秒數,同時被三個跟演算法無關的因素污染:

  • 硬體:CPU 時脈、快取大小、記憶體速度——換台機器,數字全變。
  • 語言與實作:Python 直譯比 C 編譯慢上幾十倍;同樣的演算法,寫法不同也有常數差。
  • 輸入:同一個演算法,餵不同的資料,快慢可以天差地別(待會的「最壞 / 最佳情況」會細談)。

我們想要的,是一把能跨越機器、跨越語言、跨越單次運氣的量尺——一把只衡量 「演算法本身,隨著輸入規模 $n$ 變大,工作量會怎麼成長」的尺。 而要打造這把尺,我們得先學會一種看事情的角度:漸進分析。

🔑
我們衡量的是「成長率」,不是「秒數」

複雜度分析從不回答「這要跑幾秒」,它回答的是「當輸入規模翻倍 / 變十倍,工作量會怎麼跟著漲」。 這個「成長的形狀」是演算法與生俱來的性格,不會因為你換了電腦或換了語言而改變——這正是它珍貴的地方。

漸進分析:用望遠鏡看趨勢

假設你細細數過兩個演算法,精確算出它們在輸入規模 $n$ 下的操作次數:

$$T_A(n) = 100n + 5000 \qquad\qquad T_B(n) = 2n^2 + 3n + 10$$

當 $n$ 很小時,A 反而比較「貴」:$n = 1$ 時 A 要 5100 次、B 只要 15 次。 如果你只看小資料,可能會選 B。但這是個陷阱。讓 $n$ 慢慢長大:

過了某個門檻(這裡大約是 $n = 100$),B 就再也追不上 A,而且差距越拉越大。 這告訴我們一件關鍵的事:真正決定大規模表現的,是那個成長最快的項—— B 的 $2n^2$、A 的 $100n$。至於 $+5000$、$+3n+10$ 這些低階項,還有 $100$、$2$ 這些常數倍率, 在 $n$ 夠大時通通變成雜訊。

這就是漸進分析(asymptotic analysis)的精神。「Asymptotic(漸進)」這個字, 意思是「當 $n$ 趨向無限大時的行為」。我們刻意不看小資料、不看常數、不看低階項, 只盯住「當 $n$ 大到不能再大時,誰主宰一切」。

🔭
把它想成一支望遠鏡

漸進分析就像拿望遠鏡眺望遠方的地平線。近處的小石子、草叢的紋理(常數與低階項)全都模糊掉了, 你只看得見遠方那座山的輪廓(主導項)。我們不在乎眼前三公尺,我們在乎「往無限遠的方向,這條路是平的、是斜的、還是直直陡上天」。 這個「輪廓」,就是演算法的命運。

於是,我們要做的「化簡」其實只有兩條規則,接下來整章都會反覆用到:

  • 丟掉常數倍率:$100n$ 和 $n$ 是同一個成長形狀,都記作 $O(n)$。
  • 只留主導項(丟掉低階項):$2n^2 + 3n + 10$ 的命運由 $n^2$ 決定,記作 $O(n^2)$。

但「主導」「夠大」「可以忽略」這些詞,聽起來都還很模糊。要讓它們變成能證明的數學, 我們需要一個精確的記號——這就是大 O。

大 O:成長的上界

我們先用人話說一遍大 O 想表達什麼:「$f(n)$ 的成長,不會比 $g(n)$ 快(頂多差一個固定的常數倍)。」 換句話說,$g(n)$ 是 $f(n)$ 成長速度的一個上限。把這句話翻成數學,就是:

🔑
大 O 的正式定義

我們說 $f(n) = O(g(n))$,若存在兩個正常數 $c$ 與 $n_0$,使得: $$\forall\, n \ge n_0:\quad 0 \le f(n) \le c \cdot g(n)$$ 白話翻譯:「只要 $n$ 大過某個門檻 $n_0$,我就能找到一個固定的放大倍率 $c$, 讓 $c \cdot g(n)$ 從此永遠壓在 $f(n)$ 頭上。」

這個定義裡每個字都在工作,我們逐一拆解:

  • 「存在 $c$」:這就是「丟掉常數倍率」的來源。你被允許自由挑一個放大倍率, 所以 $5n$ 也好、$100n$ 也好,都能被某個 $c \cdot n$ 蓋過,它們都是 $O(n)$。
  • 「存在 $n_0$,且只要求 $n \ge n_0$」:這就是「只看夠大的 $n$」。 小資料上誰快誰慢無所謂,我們只在乎門檻之後的長期趨勢。這也是為什麼前面那個「$n$ 小的時候 B 比較快」完全不影響結論。
  • 「$f(n) \le c\cdot g(n)$」:這是上界。它說 $f$ 最多就長這麼快,不會更快。

把這個定義畫出來,你會立刻「看見」它在說什麼:

來做一個具體的小證明,把定義用熟。我們宣稱 $3n + 5 = O(n)$。要證明它,我們得找出一組可行的 $c$ 與 $n_0$:

取 $c = 4$、$n_0 = 5$。當 $n \ge 5$ 時,$5 \le n$,所以 $3n + 5 \le 3n + n = 4n = c\cdot n$。 定義被滿足,故 $3n + 5 = O(n)$。$\blacksquare$

注意:$c$ 與 $n_0$ 的選擇不唯一。取 $c = 8, n_0 = 1$ 也成立(當 $n \ge 1$,$3n + 5 \le 3n + 5n = 8n$)。 定義只要求「存在一組」,你不必找最緊的,能找到任何一組就贏了。

⚖️
一個惱人但重要的記號習慣:那個「=」其實是「∈」

我們寫 $f(n) = O(g(n))$,但這個等號其實是誤用。$O(g(n))$ 嚴格說是「所有成長不超過 $g$ 的函數所成的集合」, 正確該寫 $f(n) \in O(g(n))$。但歷史上大家寫順了「=」,於是它變成一種單向的等號: 你可以說「$3n+5$ 是 $O(n)$」,但不能反過來說「$O(n)$ 是 $3n+5$」。 看到 $O(\cdot)$ 旁邊的等號,心裡請自動翻譯成「是一種」。

Ω 與 Θ:下界,與真正的「緊界」

大 O 給了我們上界(成長頂多這麼快)。但只有上界,有時會說謊。 舉個極端例子:「線性搜尋是 $O(n^2)$」——這句話在技術上完全正確! 因為線性搜尋的 $n$ 次操作,確實不超過 $c\cdot n^2$。但這個上界鬆得離譜, 就像你說「我身高不超過 100 公尺」——沒說錯,但毫無資訊量。

要把話說精準,我們需要另外兩個記號,跟大 O 湊成一套完整的工具。

大 Ω(Omega):成長的下界

大 Ω 是大 O 的鏡像。它說「$f(n)$ 的成長至少有 $g(n)$ 這麼快」——$g$ 是個下限:

🔑
大 Ω 的正式定義

$f(n) = \Omega(g(n))$,若存在正常數 $c, n_0$,使得 $$\forall\, n \ge n_0:\quad 0 \le c \cdot g(n) \le f(n)$$ 把大 O 定義裡的不等號反過來就是它:現在 $c\cdot g(n)$ 是地板,$f(n)$ 從此踩在它上面。

大 Θ(Theta):上下界合一,真正的緊界

如果一個函數同時被 $g$ 從上面壓住、又被 $g$ 從下面頂住,那 $g$ 就剛剛好刻畫了它的成長。 這才是我們真正想要的「緊界(tight bound)」:

🔑
大 Θ 的正式定義

$f(n) = \Theta(g(n))$,若 $f(n) = O(g(n))$ 且 $f(n) = \Omega(g(n))$。 等價地:存在正常數 $c_1, c_2, n_0$,使得 $$\forall\, n \ge n_0:\quad c_1\cdot g(n) \le f(n) \le c_2 \cdot g(n)$$ 也就是 $f$ 被夾在 $g$ 的兩個常數倍之間——成長率完全相同。

⚠️
最常見的誤解:把 O 當成「就是」緊界

口語上我們常說「合併排序是 $O(n\log n)$」,聽起來像在說它的精確成長。但嚴格講,$O$ 只是上界。 說合併排序「是 $O(n^2)$」也沒有錯(只是很鬆),因為 $n\log n \le c\cdot n^2$。 真正精準的說法是「合併排序是 $\Theta(n\log n)$」。 實務上,當大家說某演算法「是 $O(f)$」時,心裡通常指的是 $\Theta$(緊界)—— 這是一種約定俗成的口語簡化。你自己分析時,要清楚分辨「我證的是上界,還是緊界」。

✏️ 練習 26.1 挑戰

用大 O 的定義證明:$3n^2 + 10n + 7 = O(n^2)$。也就是,請找出一組具體的 $c$ 與 $n_0$, 並寫出讓不等式成立的推導。

看解答

策略:把每個低階項都「放大成」$n^2$,再把係數加起來當作 $c$。

取 $n_0 = 10$。當 $n \ge 10$ 時:

  • $10n \le n \cdot n = n^2$(因為 $n \ge 10 \ge 10$,其實只要 $n \ge 10$ 就有 $10n \le n^2$)
  • $7 \le n^2$(因為 $n \ge 10 \Rightarrow n^2 \ge 100 \ge 7$)

所以當 $n \ge 10$: $$3n^2 + 10n + 7 \;\le\; 3n^2 + n^2 + n^2 \;=\; 5n^2.$$ 取 $c = 5$、$n_0 = 10$,定義被滿足,故 $3n^2 + 10n + 7 = O(n^2)$。$\blacksquare$

同樣的手法可以證任何多項式 $a_k n^k + \dots + a_1 n + a_0 = O(n^k)$:把每個低階項都放大成 $n^k$, $c$ 取所有係數絕對值之和即可。這就是「多項式的 Big-O 只看最高次項」的嚴格根據。

怎麼「讀」出一段程式的複雜度

理論講完,該動手了。分析程式碼的複雜度,聽起來玄,其實只要掌握幾條會反覆出現的「文法」, 你就能像讀句子一樣讀出複雜度。我們一條一條看。

規則一:循序的程式碼用「加法」,然後取主導項

一段接著一段執行的程式碼,複雜度是各段相加,加完再丟掉低階項,只留最大的那個: 

def process(arr):
    total = 0
    for x in arr:          # 段 A:掃過一遍 → O(n)
        total += x
    arr.sort()             # 段 B:Python 內建排序 → O(n log n)
    return total           # 段 C:回傳 → O(1)

# 總計:O(n) + O(n log n) + O(1) = O(n log n)
# 因為當 n 夠大,n log n 完全主宰另外兩項

$O(n) + O(n\log n) + O(1)$ 不是 $O(n + n\log n + 1)$ 慢慢算,而是直接看「誰最大」:$n\log n$。 所以整個函式是 $O(n\log n)$。

規則二:巢狀迴圈用「乘法」

一個迴圈套在另一個迴圈裡,複雜度是兩者相乘。經典例子是「檢查有沒有重複元素」: 

def has_duplicate(arr):
    n = len(arr)
    for i in range(n):              # 外層:跑 n 次
        for j in range(i + 1, n):  # 內層:跑約 n 次
            if arr[i] == arr[j]:
                return True
    return False

# 外層 n 次 × 內層約 n 次 = O(n²)

你可能會抗議:內層其實不是每次都跑 $n$ 次,而是 $n-1, n-2, \dots, 1$ 次,越來越少。沒錯, 把它們加總是 $(n-1)+(n-2)+\dots+1 = \frac{n(n-1)}{2}$。但別忘了漸進分析的兩條規則: 丟掉常數($\frac{1}{2}$)、丟掉低階項($-\frac{n}{2}$),$\frac{n(n-1)}{2}$ 的命運由 $n^2$ 決定。 所以仍是 $\Theta(n^2)$。「三角形」的迴圈和「正方形」的迴圈,漸進上是一樣的。

規則三:每次「砍半」就生出一個 $\log$

當迴圈每跑一次就把問題規模除以一個常數(最常見是除以 2), 迴圈的執行次數就是 $\log n$: 

def count_steps(n):
    steps = 0
    while n > 1:
        n = n // 2     # 每一輪,規模砍半
        steps += 1
    return steps

# n 要被砍幾次才會降到 1?答案是 log₂(n) 次 → O(log n)
# count_steps(1024) 回傳 10,因為 2¹⁰ = 1024

為什麼是 $\log$?因為「$n$ 連續除以 2,幾次後變成 1」這個問題,等價於問「$2$ 的幾次方等於 $n$」, 而那個次方,定義上就是 $\log_2 n$。這也正是第 5 章二分搜尋為什麼是 $O(\log n)$ 的根本原因—— 它每比一次,搜尋範圍就砍半。

把規則組合起來

真實的程式碼,是這些規則的組合。比方說「對每個元素,各做一次二分搜尋」: 

def count_present(queries, sorted_data):
    found = 0
    for q in queries:                  # 外層:len(queries) = n 次
        if binary_search(sorted_data, q) != -1:  # 每次 O(log m)
            found += 1
    return found

# n 次外層 × 每次 O(log m) = O(n log m)

化簡的速查表

分析到一半,把式子套這幾條規則收尾:

  • $O(c \cdot f(n)) = O(f(n))$ —— 丟掉常數倍率。$O(500) = O(1)$,$O(3n) = O(n)$。
  • $O(f(n) + g(n)) = O(\max(f, g))$ —— 相加取主導。$O(n^2 + n) = O(n^2)$。
  • $O(f(n)\cdot g(n))$ 不能再化簡(這是巢狀)。$O(n \cdot \log n) = O(n\log n)$。
  • 對數的底數無所謂:$O(\log_2 n) = O(\log_{10} n) = O(\ln n)$(換底只差一個常數倍)。
✏️ 練習 26.2 思考

判斷以下三段程式碼各自的時間複雜度(用 $n = $ len(arr)),並說明理由。

# (a)
def f(arr):
    s = 0
    for x in arr:
        s += x
    for x in arr:
        s += x * x
    return s

# (b)
def g(arr):
    n = len(arr)
    for i in range(n):
        for j in range(n):
            print(arr[i], arr[j])

# (c)
def h(arr):
    n = len(arr)
    i = n
    while i >= 1:
        print(arr[0])
        i = i // 2
看解答

(a) $O(n)$。兩個迴圈是循序(一個接一個,不是巢狀), 各為 $O(n)$,相加 $O(n) + O(n) = O(2n) = O(n)$。關鍵:兩個並排的迴圈是「加法」,不是「乘法」。 這是最常見的陷阱——看到兩個迴圈別反射性喊 $O(n^2)$,先看它們是「並排」還是「巢狀」。

(b) $O(n^2)$。內層迴圈巢狀在外層裡,兩層各跑 $n$ 次,相乘 $n \times n = O(n^2)$。

(c) $O(\log n)$。每一輪 i 砍半,從 $n$ 降到 1 需要 $\log_2 n$ 輪。 迴圈內部只是 $O(1)$ 的 print,所以總共 $O(\log n)$。

✏️ 練習 26.3 思考

化簡下列複雜度表達式(丟掉常數與低階項,寫成最簡的 $O(\cdot)$):

  1. $5n^3 + 200n^2 + 1000n + 9999$
  2. $n + n\log n$
  3. $2^n + n^{100}$
  4. $\log n + 100$
  5. $3n^2 + 2^n$
看解答
  1. $O(n^3)$。最高次項是 $n^3$,常數 5 與其餘低階項全丟。
  2. $O(n\log n)$。$n\log n$ 成長快過 $n$(多了一個 $\log n$ 因子),主導。
  3. $O(2^n)$。這題是陷阱:$n^{100}$ 雖然指數高達 100,但它是多項式; $2^n$ 是指數。任何指數最終都會輾壓任何多項式,所以 $2^n$ 主導。
  4. $O(\log n)$。常數 100 丟掉。
  5. $O(2^n)$。指數遠快過平方。

記住成長率的排序鏈,化簡就是「挑出最右邊那個」: $$1 \prec \log n \prec n \prec n\log n \prec n^2 \prec n^3 \prec 2^n \prec n!$$

✏️ 練習 26.4 思考

分析下面這個巢狀迴圈總共會印出幾次,並給出緊界 $\Theta(\cdot)$。

def triangle(n):
    for i in range(n):
        for j in range(i):    # 注意內層上限是 i,不是 n
            print(i, j)
看解答

當外層的 i 是 $0, 1, 2, \dots, n-1$ 時,內層 range(i) 分別跑 $0, 1, 2, \dots, n-1$ 次。 總次數是: $$0 + 1 + 2 + \dots + (n-1) = \sum_{i=0}^{n-1} i = \frac{n(n-1)}{2}.$$

展開是 $\frac{n^2 - n}{2}$。丟掉常數 $\frac{1}{2}$、丟掉低階項 $-\frac{n}{2}$,主導項是 $n^2$。 所以是 $\Theta(n^2)$。

重點教訓:即使內層迴圈次數會變動(這裡是個「三角形」而非「正方形」), 只要它平均下來與 $n$ 成正比,整體仍是平方級。別被「內層只跑 $i$ 次,比 $n$ 少」騙了—— 漸進上,$\frac{n^2}{2}$ 和 $n^2$ 是同一回事。

最壞、平均、最佳:同一個演算法的三張臉

前面我們小心地說「線性搜尋是 $O(n)$」,但其實它有三種命運。回想線性搜尋:從頭找一個目標。

  • 最佳情況(best case):目標剛好在第一格。只看一次就找到 → $O(1)$。
  • 最壞情況(worst case):目標在最後一格,或根本不存在。得看完全部 $n$ 個 → $O(n)$。
  • 平均情況(average case):假設目標等機率出現在任一格,平均看 $\frac{n}{2}$ 個 → 仍是 $O(n)$。

所以同一個演算法,可以同時是「最佳 $O(1)$」「最壞 $O(n)$」。 這也是為什麼前面強調:講複雜度時要說清楚你指的是哪種情況。

那為什麼業界預設都關注最壞情況?三個現實的理由:

  • 它是保證:最壞情況給你一個「絕不會更慢」的承諾。對於要排進航班、控制煞車、處理金流的系統, 你需要的不是「平均很快」,而是「最糟也撐得住」。
  • 它通常容易算:平均情況需要對「輸入的機率分布」做假設,而現實中的輸入往往不是均勻隨機的; 最壞情況不需要任何機率假設,直接看「最慘會怎樣」即可。
  • 壞輸入常常真的會出現:資料未必友善,有時甚至有人故意餵你最壞輸入來癱瘓系統 (這是一種真實的資安攻擊,稱為演算法複雜度攻擊)。
📚
快速排序:一個「平均 vs 最壞」差很大的名例

快速排序(quicksort)平均是 $\Theta(n\log n)$,跑起來飛快、實務上極受歡迎; 但它的最壞情況是 $\Theta(n^2)$(例如每次都挑到最差的樞紐 pivot)。 這就是「最壞」與「平均」分歧的經典案例——也是為什麼有些場合寧可選最壞也保證 $O(n\log n)$ 的合併排序或堆積排序。 這些演算法的細節,是本部後面幾章的主菜。

另一個維度:空間複雜度

到目前為止我們只談「時間」(操作次數)。但演算法還會吃掉另一種資源:記憶體。 空間複雜度(space complexity)用同一套 Big-O 語言,衡量「演算法額外需要的記憶體, 隨輸入規模 $n$ 怎麼成長」。

注意「額外」二字:我們通常算的是輔助空間(auxiliary space)—— 除了輸入本身佔的空間之外,演算法自己又多開了多少。看兩個求和的例子: 

# 版本一:空間 O(1)
def sum_inplace(arr):
    total = 0           # 不論 arr 多長,只用了一個變數
    for x in arr:
        total += x
    return total

# 版本二:空間 O(n)
def running_sums(arr):
    result = []         # 額外開了一個跟 arr 一樣長的陣列
    total = 0
    for x in arr:
        total += x
        result.append(total)
    return result       # 輔助空間隨 n 線性成長 → O(n)

遞迴也會悄悄吃空間:每一層遞迴呼叫都會在呼叫堆疊(call stack)上佔一格。 所以一個遞迴深度為 $n$ 的函式,即使每層只用幾個變數,空間複雜度也是 $O(n)$。 (這也是為什麼太深的遞迴會 RecursionError / stack overflow——回憶第 3 章。)

時間與空間的取捨

很多時候,你可以用空間換時間,或反過來。最乾淨的例子,還是「檢查重複」: 

# 方案 A:省空間,費時間。時間 O(n²),空間 O(1)
def has_dup_slow(arr):
    n = len(arr)
    for i in range(n):
        for j in range(i + 1, n):
            if arr[i] == arr[j]:
                return True
    return False

# 方案 B:費空間,省時間。時間 O(n),空間 O(n)
def has_dup_fast(arr):
    seen = set()              # 用一個雜湊集合記住看過的元素
    for x in arr:
        if x in seen:        # 查雜湊集合是平均 O(1)
            return True
        seen.add(x)
    return False

方案 B 多花了一個 $O(n)$ 大小的集合,換來把時間從 $O(n^2)$ 砍到 $O(n)$。 當資料量大時,這筆交易通常非常划算——記憶體很便宜,時間很貴。 這種「拿記憶體換速度」的思想,你會在很多地方再遇到它:

  • 快取(caching):把算過的結果存起來,下次直接拿,不重算。
  • 動態規劃(dynamic programming):用一張表記住子問題的解,把指數時間打成多項式時間 (這是本部後面的重頭戲)。
  • 雜湊表 / 索引:多存一份對照結構,把查詢從 $O(n)$ 變 $O(1)$。
🔑
沒有最好的演算法,只有最適合的權衡

時間與空間常常是蹺蹺板:壓低一邊,另一邊就翹起來。哪個重要,取決於你的處境—— 記憶體吃緊的嵌入式裝置會想省空間;追求極速的伺服器則樂意砸記憶體換時間。 工程的藝術,就在於認得出這個取捨,並為當下的限制選對那一端。

攤銷分析:動態陣列的 append 為什麼是 $O(1)$?

現在來解開一個你天天在用、卻可能從沒想過的謎。回憶第 4 章:Python 的 list 是一個 動態陣列(dynamic array)。底層的陣列在記憶體裡其實是固定大小的一塊連續空間。 那麼,當這塊空間被填滿了,你又呼叫 append,會發生什麼事?

系統得去要一塊更大的新空間,把舊資料整個搬過去,再放入新元素。 這個「搬家」動作要複製所有 $n$ 個既有元素,是 $O(n)$!那……我們怎麼還敢說 append 是 $O(1)$?

答案藏在「多大才擴一次」這個聰明的設計裡。動態陣列不是「滿了就加一格」, 而是「滿了就容量翻倍」。我們把這個過程畫出來:

來算一筆總帳。假設從容量 1 開始,連續做 $n$ 次 append。容量會在 $1, 2, 4, 8, \dots$ 時各觸發一次擴容, 每次擴容要複製當時的所有元素。這些複製的總成本是: $$1 + 2 + 4 + 8 + \dots + \text{(不超過 $n$ 的最大 2 次方)} \;\lt\; 2n.$$ 這是個幾何級數:它的總和不會超過最後一項的兩倍,也就是不到 $2n$。 再加上 $n$ 次 append 本身每次的 $O(1)$,做 $n$ 次 append 的總成本不到 $3n$, 也就是 $O(n)$。把總成本平攤到 $n$ 次操作上: $$\frac{\text{總成本}}{n} \;\lt\; \frac{3n}{n} = 3 = O(1).$$

這就是攤銷分析(amortized analysis):當一連串操作裡,昂貴的操作很罕見、 足以被大量便宜操作「平攤」掉,我們就用「平均每次操作」的成本來描述它。 所以雖然某一次 append 可能是 $O(n)$,但攤銷後每次 append 是 $O(1)$——這是個誠實、嚴謹的說法, 不是含糊的「大概啦」。

⚠️
攤銷 ≠ 平均(別搞混)

「攤銷 $O(1)$」和「平均情況 $O(1)$」是兩件不同的事。平均情況是對「隨機輸入」取期望, 可能某次運氣差就退化。攤銷則是對「一連串操作」做保證: 不管輸入怎樣,任何連續 $n$ 次 append 的總成本鐵定是 $O(n)$,沒有「運氣差」這回事。 攤銷是更強的承諾。

💡
為什麼是「翻倍」,不是「每次加 10」?

若改成「滿了就加固定 $k$ 格」,擴容會發生 $\frac{n}{k}$ 次,每次複製越來越多, 總複製成本變成 $k + 2k + 3k + \dots \approx \frac{n^2}{2k}$,是 $O(n^2)$——攤銷後每次 append 變 $O(n)$,糟透了! 正是「翻倍」讓擴容間隔呈幾何成長,才把總成本壓在 $O(n)$、攤銷成 $O(1)$。 這個「乘法擴張」的設計哲學,值得記一輩子。

✏️ 練習 26.5 挑戰

寫一段 Python,實際數出「容量翻倍」策略下,連續 append $n$ 個元素總共複製了幾個元素, 並驗證它確實小於 $2n$(亦即攤銷成本是常數)。模擬一個「自己管理容量」的動態陣列即可,不要直接用 list.append 的內建行為。

看解答 
def count_copies(n):
    capacity = 1          # 初始容量
    size = 0              # 目前元素數
    total_copies = 0      # 累計複製了幾個元素
    for _ in range(n):
        if size == capacity:          # 滿了 → 擴容
            total_copies += size      # 把現有 size 個元素複製到新空間
            capacity *= 2             # 容量翻倍
        size += 1                     # 放入新元素
    return total_copies

for n in [1, 10, 100, 1000, 10000, 100000]:
    c = count_copies(n)
    print(f"n={n:>7}  總複製={c:>7}  複製/n={c/n:.3f}  < 2 ? {c < 2*n}")

用實驗檢驗理論:讓數字自己說話

漸進分析是理論。但理論最迷人的地方,是它能被驗證。 我們前面說「測秒數不可靠」,那是指「拿秒數比較不同演算法」不可靠; 但如果是「對同一個演算法,在同一台機器上,測它隨 $n$ 變大的成長趨勢」, 那秒數反而是檢驗理論的好幫手——因為硬體、語言這些常數因素被固定住了,只剩 $n$ 在變。

Python 內建的 time 模組就夠用。注意要用 time.perf_counter()(高解析度計時器), 而不是 time.time()(精度較差): 

import time

def linear_work(n):
    """O(n):掃過一遍。"""
    total = 0
    for i in range(n):
        total += i
    return total

def quadratic_work(n):
    """O(n²):巢狀掃兩遍。"""
    count = 0
    for i in range(n):
        for j in range(n):
            count += 1
    return count

def measure(func, n):
    start = time.perf_counter()
    func(n)
    return time.perf_counter() - start   # 回傳耗時(秒)

print("=== 線性 O(n):n 翻倍,時間應該約 ×2 ===")
prev = None
for n in [1_000_000, 2_000_000, 4_000_000, 8_000_000]:
    t = measure(linear_work, n)
    ratio = f"(×{t/prev:.1f})" if prev else ""
    print(f"n={n:>9}  {t*1000:8.2f} ms  {ratio}")
    prev = t

print("\n=== 平方 O(n²):n 翻倍,時間應該約 ×4 ===")
prev = None
for n in [1000, 2000, 4000, 8000]:
    t = measure(quadratic_work, n)
    ratio = f"(×{t/prev:.1f})" if prev else ""
    print(f"n={n:>9}  {t*1000:8.2f} ms  {ratio}")
    prev = t

💡
「倍率測試」:你口袋裡最好用的複雜度偵測法

想知道一段不熟的程式碼大概是什麼複雜度?別硬讀,直接量:把 $n$ 翻倍,看時間變幾倍。 時間幾乎不變 → $O(1)$ 或 $O(\log n)$;約 $\times 2$ → $O(n)$;約 $\times 4$ → $O(n^2)$;約 $\times 8$ → $O(n^3)$。 這招在面對別人寫的、或自己忘了怎麼分析的程式時,特別好用。

✏️ 練習 26.6 實作

把上面的「倍率測試」套用到第 5 章的二分搜尋上。寫一段程式,對越來越大的(已排序)資料, 測量二分搜尋的耗時,並觀察「$n$ 翻倍時,時間幾乎沒變」這個 $O(\log n)$ 的招牌特徵。 (提示:單次二分搜尋太快,計時器量不準——請對同一個 $n$ 重複搜尋很多次再取平均。)

看解答 
import time

def binary_search(arr, target):
    low, high = 0, len(arr) - 1
    while low <= high:
        mid = (low + high) // 2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            low = mid + 1
        else:
            high = mid - 1
    return -1

REPEAT = 100_000   # 重複多次,讓總時間長到量得準
prev = None
for n in [100_000, 1_000_000, 10_000_000, 100_000_000]:
    data = list(range(n))            # 已排序
    target = n - 1                   # 找最後一個(接近最壞情況)
    start = time.perf_counter()
    for _ in range(REPEAT):
        binary_search(data, target)
    t = (time.perf_counter() - start) / REPEAT   # 單次平均(秒)
    ratio = f"(×{t/prev:.2f})" if prev else ""
    print(f"n={n:>11}  每次約 {t*1e6:7.3f} µs  {ratio}")
    prev = t

n=   10000000  每次約   0.870 µs  (×1.18)
n=  100000000  每次約   1.010 µs  (×1.16)

對照之下,如果你把 binary_search 換成線性搜尋,$n$ 每翻 10 倍,時間就會翻約 10 倍—— 這個強烈反差,就是 $O(\log n)$ 與 $O(n)$ 的差別看得見的證明。

✏️ 練習 26.7 思考

演算法 A 的執行時間是 $T_A(n) = 100n$,演算法 B 的是 $T_B(n) = n^2$。

  1. 當 $n$ 很小時,哪個比較快?
  2. 當 $n$ 很大時,哪個比較快?用 Big-O 說明。
  3. 它們的「交叉點」($T_A = T_B$)在哪個 $n$?這個交叉點告訴你什麼實務上的啟示?
看解答

1. 小 $n$:B 比較快。例如 $n = 10$,$T_A = 1000$,$T_B = 100$,B 勝。 因為 A 有個大常數 100 拖累它。

2. 大 $n$:A 比較快(而且差距無限拉大)。$A$ 是 $O(n)$、$B$ 是 $O(n^2)$, 平方終究遠快過線性地成長。無論 A 的常數多大,只要 $n$ 夠大,$100n$ 必定遠小於 $n^2$。

3. 交叉點:$100n = n^2 \Rightarrow n = 100$。當 $n < 100$,B 較快;$n > 100$,A 較快。

實務啟示:漸進複雜度是「夠大的 $n$」才主宰。如果你確定輸入永遠很小($n < 100$), 那個「理論上較差」的 B 反而是對的選擇——常數有時很重要。這也是為什麼工業級排序(如 Timsort) 會在小區段上改用看似較慢、但常數很小的插入排序。Big-O 告訴你終局,但別忘了現實中 $n$ 到底多大。

為什麼對數無所不在?

你可能已經注意到,$\log n$ 像個幽靈,在這本書裡一再現身:二分搜尋、平衡樹、堆積、合併排序裡的層數…… 為什麼?因為它們背後都藏著同一個動作:反覆砍半。

這是 $\log$ 的本質。問「$n$ 連續除以 2,幾次後降到 1」,答案就是 $\log_2 n$。反過來, 問「從 1 開始連續乘以 2,幾次後超過 $n$」,答案也是 $\log_2 n$。任何「每一步把規模乘上或除以一個固定倍數」 的過程,步數天生就是對數級:

  • 二分搜尋(第 5 章):每次比較砍掉一半搜尋範圍 → $O(\log n)$ 次比較。
  • 平衡二元搜尋樹 / 堆積:樹高約 $\log n$,因為每往下一層,節點數翻倍。
  • 分治演算法的遞迴深度:合併排序把陣列一切為二、再切為二……要切 $\log n$ 層才到底。
  • 快速冪、輾轉相除法(第 15 章):每步讓問題規模成倍縮小。

還記得我們說「對數的底數不影響 Big-O」嗎?$\log_2 n$ 和 $\log_{10} n$ 只差一個固定常數 ($\log_2 n = \frac{\log_{10} n}{\log_{10} 2}$),而 Big-O 本來就忽略常數,所以一律寫 $O(\log n)$,不標底數。

🔗
通往主定理的橋

當「砍半」遇上「遞迴」,事情會更精彩。一個把規模 $n$ 切成數個子問題、各自遞迴再合併的演算法, 它的執行時間會寫成一條遞迴關係(recurrence),例如合併排序的 $T(n) = 2T(n/2) + O(n)$。 解這條關係會解出 $T(n) = O(n\log n)$——那個 $\log$ 正是「切到底要切 $\log n$ 層」來的。 第 15 章已經預告過的主定理(Master Theorem),就是一張「查表卡」, 讓你一眼解出這類遞迴的複雜度。它的完整威力,留待本部第 27 章「分治法」盡情展開。

把這一章串起來

我們從「測秒數為什麼不準」出發,打造出一把能跨越機器與語言的量尺;再從直覺一路走到能證明的數學。 回顧這趟旅程的路標:

  • 複雜度衡量的是成長率,不是秒數。漸進分析只看「夠大的 $n$」,只留主導項,丟掉常數與低階項——像拿望遠鏡眺望趨勢。
  • 大 O 是上界($\le c\cdot g$)、大 Ω 是下界($\ge c\cdot g$)、大 Θ 是上下界合一的緊界。 記住:「O 是上界,不是緊界」——口語常把 O 當 Θ 用,自己分析時要分清楚。
  • 複雜度動物園從 $O(1)$ 排到 $O(n!)$;指數與階乘成長之恐怖,意味著「只有指數解」幾乎等於「大輸入無解」。
  • 讀程式三規則:循序相加、巢狀相乘、砍半生 $\log$;讀完再丟常數與低階項。
  • 同一演算法有最佳 / 平均 / 最壞三張臉;我們通常盯最壞,因為它是保證、好算、且壞輸入真的會出現。
  • 空間複雜度是另一個維度;時間與空間常可互換(快取、DP、雜湊都是拿空間換時間)。
  • 攤銷分析解釋了動態陣列 append 為何是 $O(1)$:昂貴的擴容很罕見,被大量便宜操作平攤掉。攤銷是「對一連串操作」的保證,比平均情況更強。
  • 理論可以用 Python 計時實驗驗證:$n$ 翻倍看時間變幾倍,就能反推複雜度。
  • 對數源於「反覆砍半」,因此在搜尋、樹、分治裡無所不在;底數不影響 Big-O。

這一章是整個第六部的數學地基。從下一章開始,我們會用這把量尺去丈量一個又一個漂亮的演算法與資料結構—— 分治法、排序、樹、圖、雜湊、動態規劃……每當你問「這個方法到底好不好」,答案都會用你今天學到的語言寫成。 歡迎來到演算法的世界。
