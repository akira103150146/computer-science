排序與搜尋

把一疊東西排好——這件你每天都在做、看似平凡到不行的事,藏著整個演算法世界最深刻的幾個祕密:一個關於「分治」的鋒利刀刃、一道用決策樹證明的數學下界,還有一群「不靠比較」就偷偷繞過它的排序法。

為什麼一本演算法書,總是從排序講起?

翻開任何一本演算法教科書,你幾乎都會在前幾章撞見「排序」。這不是巧合,也不是因為排序特別難。 恰恰相反——排序是那種「人人都直覺地知道答案長什麼樣」的問題:把一串東西,由小到大(或由大到小)排好。 正因為目標如此清晰,它成了演算法世界最理想的練習場與共同語言: 我們可以把各種解題思想(暴力、分治、隨機化、空間換時間……)全丟進這個乾淨的問題裡, 看它們各自表現如何。它就是演算法的「Hello, World」。

但排序遠不只是練習。它無所不在,而且常常是別的演算法能跑得快的前置步驟。 最直接的例子你上一部就見過了:二分搜尋(binary search)快得驚人, 但它有個鐵打的前提——資料必須先排好序。沒有排序,就沒有二分搜尋。 再往外看:找出重複的元素、求中位數、合併兩份名單、依時間軸處理事件、 資料庫的 ORDER BY 與索引建立、電腦繪圖裡決定誰擋住誰…… 背後幾乎都站著一個排序步驟。學會排序,等於拿到一把能打開無數扇門的萬用鑰匙。

🔑
排序是「賦能型」演算法

很多時候,排序本身不是你的最終目的,而是讓後續步驟變快的手段。 一旦資料有序,搜尋、去重、找極值、找鄰近元素、合併……全都能從 $O(n)$ 甚至 $O(n^2)$ 降到 $O(\log n)$ 或 $O(n)$。先花一次 $O(n\log n)$ 把資料排好, 常常是「一本萬利」的投資——這正是上一章「分治」思想的延伸與發揚。

這一章我們會從你在第 5 章熱身過的 $O(n^2)$ 排序快速複習起步,接著迎來重頭戲—— 快速排序(quicksort),把上一章的分治思想磨成一把鋒利的刀; 然後爬上一座理論高峰:證明「基於比較的排序」不可能快過 $\Omega(n\log n)$; 再看一群「不講道理」的排序如何繞過這道牆;最後回到搜尋,把二分搜尋的每個邊界都釘死, 並學會只找「第 $k$ 小」而不必整串排好的聰明技巧。出發。

複習:三個 $O(n^2)$ 排序,各有各的個性

第 5 章你已經摸過氣泡與插入排序的手感。這裡我們把三個經典的平方級排序放在一起對照, 因為它們雖然都是 $O(n^2)$,個性卻完全不同——而理解這些差異,正是後面欣賞快速排序與合併排序的基礎。

選擇排序(selection sort):每次挑出最小的

最符合「人類直覺」的排法:掃過整個未排序區,找出最小的那個,把它換到最前面; 然後對剩下的部分重複。就像反覆從一堆牌裡「選」出最小的一張放好——名字就是這麼來的。 

def selection_sort(arr):
    n = len(arr)
    for i in range(n):                 # i 是「已排序區」的右界
        min_idx = i                    # 先假設未排序區的第一個最小
        for j in range(i + 1, n):      # 在未排序區裡找真正的最小
            if arr[j] < arr[min_idx]:
                min_idx = j
        arr[i], arr[min_idx] = arr[min_idx], arr[i]   # 把最小的換到位
    return arr

print(selection_sort([5, 2, 9, 1, 7]))   # [1, 2, 5, 7, 9]

插入排序(insertion sort):像整理手上的撲克牌

把陣列分成「左邊已排序」與「右邊待處理」。每次拿右區的第一張牌,往左插到它該在的位置—— 比它大的元素統統往右挪一格,讓出空位。理牌的人都做過這件事。 

def insertion_sort(arr):
    for i in range(1, len(arr)):       # arr[0] 自成已排序區
        key = arr[i]                   # 手上要插入的牌
        j = i - 1
        while j >= 0 and arr[j] > key: # 比 key 大的,通通往右挪
            arr[j + 1] = arr[j]
            j -= 1
        arr[j + 1] = key               # 空出來的位置就是 key 的家
    return arr

print(insertion_sort([5, 2, 4, 6, 1]))   # [1, 2, 4, 5, 6]

氣泡排序(bubble sort):相鄰交換,大的浮上去

反覆掃過陣列,比較相鄰兩個,左大於右就交換。每掃一輪,當前最大的數會像氣泡一樣「浮」到最右端。 加一個「這輪都沒交換就提早收工」的旗標,它在已排序資料上能達到最佳的 $O(n)$。 

def bubble_sort(arr):
    n = len(arr)
    for i in range(n):
        swapped = False
        for j in range(n - 1 - i):     # 已浮上去的不必再比
            if arr[j] > arr[j + 1]:
                arr[j], arr[j + 1] = arr[j + 1], arr[j]
                swapped = True
        if not swapped:                # 一整輪沒交換 → 已排好,提早結束
            break
    return arr

三個一起看:它們到底差在哪?

三者最壞都是 $O(n^2)$,但魔鬼在細節。下表把關鍵差異攤開——尤其注意「最佳情況」 與「穩定性」這兩欄,它們之後會反覆出現:

排序最佳平均最壞穩定?一句話特色
選擇排序$O(n^2)$$O(n^2)$$O(n^2)$否*交換次數最少;但對任何輸入都一樣慢
插入排序$O(n)$$O(n^2)$$O(n^2)$是近乎有序、小資料時超快,實務愛用
氣泡排序$O(n)$$O(n^2)$$O(n^2)$是教學經典,實務幾乎用不到

穩定(stable)是個我們稍後會講透的關鍵詞,先記住一句直覺:「值相等的元素,排完後仍維持原本的先後順序」。 (*選擇排序的典型實作會因為「長距離交換」而打亂相等元素的相對位置,所以不穩定。)

💡
別瞧不起插入排序——它在對的場合是冠軍

當資料幾乎已經排好(每個元素離正確位置都不遠),插入排序的內層幾乎不用挪動, 整體逼近 $O(n)$,把所有 $O(n\log n)$ 的「高級」排序都打趴。當資料很小(比如少於 16 個)時, 它沒有遞迴、沒有額外配置,常數極小,也常常最快。正因如此,工業級排序(像 Python 的 Timsort、 C++ 的 introsort)都會在「小區段」上偷偷切換成插入排序。「慢」演算法在它的主場,可以很快。

快速排序:把分治磨成一把刀

上一章你學到了分治(divide and conquer):把大問題切成同類的小問題,各自解完再合併。 合併排序是分治的標準範例——對半切、各自排、再合併。現在登場的快速排序 (quicksort,1959 年由 Tony Hoare 發明)用了一個更狡猾的分治策略,它把力氣花在「切」的時候, 切完幾乎不用「合」。

快速排序的核心動作只有一個,叫分割(partition):挑一個元素當樞紐(pivot), 然後把整個陣列「攪」一下,讓——

  • 所有比 pivot 小的元素,跑到 pivot 的左邊
  • 所有比 pivot 大的元素,跑到 pivot 的右邊

分割做完後,有一件神奇的事發生了:pivot 已經在它最終正確的位置上了—— 它左邊全比它小、右邊全比它大,排序完成後它哪兒也不會去。剩下的,只要對左半和右半 遞迴地再做同樣的事。當每一段都縮到剩 0 或 1 個元素,整串就排好了——而且完全不需要合併步驟。

Lomuto 分割法:一個指標掃過去

分割有好幾種寫法,最容易理解的是 Lomuto 分割。它選最右邊的元素當 pivot, 用一個指標 i 標記「已知 ≤ pivot 的區域」的右界,再用 j 從左掃到右: 每遇到一個 ≤ pivot 的元素,就把 i 往右挪一格,並把這個元素換進「小於區」。 掃完後,把 pivot 換到「小於區」的正後方,它就歸位了。

把這個過程寫成原地(in-place)的 Python——它只在原陣列上搬動,不額外開新陣列:

def quicksort_inplace(arr, low=0, high=None):
    if high is None:
        high = len(arr) - 1
    if low >= high:                      # 區段剩 0 或 1 個,天生有序
        return
    p = partition(arr, low, high)        # 分割,回傳 pivot 歸位後的索引
    quicksort_inplace(arr, low, p - 1)   # 遞迴排左半
    quicksort_inplace(arr, p + 1, high)  # 遞迴排右半

def partition(arr, low, high):
    pivot = arr[high]                    # 選最右邊當 pivot(Lomuto)
    i = low - 1                          # i 是「≤ pivot 區」的右界
    for j in range(low, high):
        if arr[j] <= pivot:
            i += 1
            arr[i], arr[j] = arr[j], arr[i]   # 把這個元素收進小於區
    arr[i + 1], arr[high] = arr[high], arr[i + 1]   # pivot 歸位
    return i + 1

data = [7, 2, 1, 8, 6, 3, 5]
quicksort_inplace(data)
print(data)   # [1, 2, 3, 5, 6, 7, 8]

如果你覺得指標來指標去有點繞,這裡有一個不原地、但極好懂的版本,用串列生成式把分割寫得一目了然。 它清楚地展現了快速排序「小於 pivot / 等於 pivot / 大於 pivot」三分天下的本質: 

def quicksort(arr):
    if len(arr) <= 1:
        return arr
    pivot = arr[len(arr) // 2]                    # 隨手挑中間當 pivot
    less    = [x for x in arr if x < pivot]
    equal   = [x for x in arr if x == pivot]
    greater = [x for x in arr if x > pivot]
    return quicksort(less) + equal + quicksort(greater)

print(quicksort([7, 2, 1, 8, 6, 3, 5]))   # [1, 2, 3, 5, 6, 7, 8]

這個版本漂亮歸漂亮,但它每層都開了三個新串列,額外用掉 $O(n)$ 空間,失去了快速排序最大的賣點之一——原地。 實務上的快速排序幾乎都用上面那種原地版本。

它有多快?平均 $O(n\log n)$,最壞卻是 $O(n^2)$

分割本身要掃過整段,是 $O(n)$。關鍵在於遞迴會分多少層。 理想情況:每次 pivot 都剛好落在中位數,把陣列切成兩個等大的一半。 那麼遞迴深度是 $\log_2 n$,每層的分割工作加總起來都是 $O(n)$, 總成本 $O(n) \times \log_2 n = O(n\log n)$。這就是快速排序平均的好表現。

但最壞情況呢?想像 pivot 每次都挑到最大或最小的元素—— 比如陣列已經排好序、而你又永遠選最右邊當 pivot。那麼每次分割, 一邊是空的、另一邊是 $n-1$ 個元素。遞迴深度退化成 $n$,總成本變成 $n + (n-1) + (n-2) + \dots + 1 = \frac{n(n+1)}{2} = O(n^2)$。 諷刺的是,快速排序在「已經排好的資料」上,反而跑出最差的成績。

隨機化 pivot:用骰子打敗最壞情況

$O(n^2)$ 的最壞情況聽起來嚇人,但它有個致命弱點:它只在 pivot 恰好連續挑到極端值時才發生。 而「永遠選最右邊」這種固定規則,正好讓「已排序的輸入」變成它的剋星—— 在真實世界裡,「已排序或接近排序的資料」其實非常常見(這很要命)。

解法優雅得令人會心一笑:別讓對手預測你的 pivot。每次分割前, 從區段裡隨機挑一個元素當 pivot(把它換到最右邊,其餘程式碼照舊)。 這麼一來,沒有任何「特定輸入」能持續逼出最壞情況——因為 pivot 的選擇與輸入的排列無關了。 可以證明,隨機化快速排序的期望時間是 $O(n\log n)$,而且與輸入長相無關。 這正是我們在隨機演算法那一章談過的思想:用隨機性,把「最壞情況」變成「機率低到可以忽略的衰事」。 

import random

def partition(arr, low, high):
    r = random.randint(low, high)              # 隨機挑一個位置
    arr[r], arr[high] = arr[high], arr[r]      # 把它換到最右邊當 pivot
    pivot = arr[high]
    i = low - 1
    for j in range(low, high):
        if arr[j] <= pivot:
            i += 1
            arr[i], arr[j] = arr[j], arr[i]
    arr[i + 1], arr[high] = arr[high], arr[i + 1]
    return i + 1
# quicksort_inplace 完全不用改,只要換掉 partition 即可
⚠️
隨機化降低機率,不是消除可能

隨機化 pivot 讓最壞情況的機率低到天荒地老都遇不到一次,但它沒有把 $O(n^2)$ 的可能性從數學上抹除——理論上你還是可能連續擲出爛骰子。 對於需要「鐵打的最壞保證」的場合(例如即時系統、防惡意輸入的服務), 就得改用最壞也 $O(n\log n)$ 的合併排序或堆積排序,或對 pivot 用後面會提的「中位數的中位數」。 這也是 C++ std::sort 採用 introsort 的原因——它平時用快速排序, 一旦偵測到遞迴太深(快變 $O(n^2)$),就切換成堆積排序保底。

合併排序 vs 快速排序:一場精彩的取捨

你在上一章「分治」見過合併排序(merge sort):對半切、左右各自遞迴排好、 再把兩條有序串「拉鍊式」合併起來。它和快速排序都是 $O(n\log n)$ 的明星,卻是兩種性格。 我們先快速複習合併的核心——那個把兩條有序串併成一條的步驟: 

def merge(left, right):
    result, i, j = [], 0, 0
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:        # 用 <= 而非 <,是「穩定」的關鍵!
            result.append(left[i]); i += 1
        else:
            result.append(right[j]); j += 1
    result.extend(left[i:])           # 把剩下的尾巴接上
    result.extend(right[j:])
    return result

def merge_sort(arr):
    if len(arr) <= 1:
        return arr
    mid = len(arr) // 2
    return merge(merge_sort(arr[:mid]), merge_sort(arr[mid:]))

那麼,什麼時候該用哪個?三個維度決定一切:

維度快速排序合併排序
平均時間$O(n\log n)$$O(n\log n)$
最壞時間$O(n^2)$(隨機化後機率極低)$O(n\log n)$(鐵打保證)
額外空間$O(\log n)$(只有遞迴堆疊)$O(n)$(要開新陣列合併)
原地(in-place)是 ✅否 ❌
穩定(stable)否 ❌(分割會長距離交換)是 ✅
快取友善 / 常數很好,常數小(實務常勝)稍差(到處搬資料)

簡單地說:快速排序通常「實際跑起來」更快(常數小、原地、對快取友善), 是排「一堆數字」這種不在乎穩定性場景的首選;合併排序勝在「可預測」與「穩定」—— 最壞也保證 $O(n\log n)$,而且能保持相等元素的原始順序,還天生適合「資料大到塞不進記憶體」 的外部排序(把一塊塊排好的檔案合併)。

ℹ️
標準庫的答案:混合策略

實務上沒有人只用單一排序。聰明的標準庫會「看菜下飯」: C++ 的 std::sort 用 introsort(快速排序為主 + 遞迴過深時退守堆積排序 + 小區段用插入排序); Python 與 Java 的物件排序用 Timsort(合併排序的變體,專門偵測資料中「已經有序的片段」並善加利用,小區段同樣切換插入排序)。 它們都把本章學到的幾種排序組合起來,各取所長。

一道優美的牆:比較排序的 $\Omega(n\log n)$ 下界

到這裡你可能會想:既然快速排序、合併排序都是 $O(n\log n)$,有沒有人能想出更快的、 比如 $O(n)$ 的通用排序?這是個了不起的問題,而答案是電腦科學裡最漂亮的結果之一: 只要你的排序是「靠比較元素大小」來決定順序的,你就不可能快過 $\Omega(n\log n)$。 這不是「目前還沒人想到」,而是被嚴格證明的——任何聰明才智都突破不了的牆。

證明的工具是決策樹(decision tree)。先想清楚一件事:一個「基於比較 (comparison-based)」的排序演算法,它對輸入唯一能做的事,就是問問題: 「$a_i$ 和 $a_j$,誰比較大?」每問一次,得到「≤」或「>」兩種答案之一,程式就根據答案走向不同分支。 於是,整個演算法的執行流程,可以畫成一棵二元樹:每個內部節點是一次比較, 兩條分支是兩種答案,而每片葉子對應到一種最終的輸出排列。

現在,證明只需要兩個觀察,加一點點算術:

  1. 葉子至少要有 $n!$ 片。 $n$ 個相異元素共有 $n!$ 種可能的排列,而正確的排序必須能輸出其中任何一種 (端看輸入怎麼亂)。每種排列至少要對應到一片葉子,所以這棵樹的葉子數 $\ge n!$。
  2. 樹高決定最壞比較次數。 演算法在最壞情況下的比較次數,等於從根走到最深葉子的路徑長度, 也就是樹的高度 $h$。而一棵高度為 $h$ 的二元樹,葉子最多 $2^h$ 片。

把兩件事兜起來:葉子數既 $\ge n!$,又 $\le 2^h$,於是

$$ 2^h \ge n! \quad\Longrightarrow\quad h \ge \log_2(n!). $$

剩下的就是估計 $\log_2(n!)$ 有多大。用一個不需要史特靈公式的初等技巧:

$$ \log_2(n!) = \sum_{i=1}^{n}\log_2 i \;\ge\; \sum_{i=n/2}^{n}\log_2 i \;\ge\; \frac{n}{2}\log_2\frac{n}{2} = \Omega(n\log n). $$

(中間那步:後半段那 $n/2$ 個項,每一項都至少是 $\log_2(n/2)$。)更精確地,史特靈公式給出 $\log_2(n!) \approx n\log_2 n - n\log_2 e$,主項正是 $n\log_2 n$。 所以樹高 $h = \Omega(n\log n)$——任何比較排序,在最壞情況下都至少要做這麼多次比較。牆,就在這裡。

🔑
這道下界在說什麼

它說的不是「某個演算法很慢」,而是「所有靠比較的排序都逃不過 $\Omega(n\log n)$」。 因此合併排序、堆積排序這些 $O(n\log n)$ 的演算法,在「比較排序」這個賽道上已經是漸進最佳(asymptotically optimal)—— 再怎麼聰明也只能改善常數,不可能改變那個 $n\log n$。 要更快,唯一的出路是跳出賽道:不要靠比較。這就引出了下一節。

跳出牆外:不靠比較的排序

上面那道牆,只困得住「靠比較」的排序。但比較並不是排序的唯一手段。 如果我們能直接從元素的「值」算出它該放哪裡——而不是反覆問「誰比誰大」—— 就能完全繞過 $\Omega(n\log n)$。代價是:這類方法對資料有額外要求(通常得是範圍有限的整數或可分桶的鍵)。

計數排序(counting sort):用「值」當地址

假設你要排的是一堆 0 到 $k$ 之間的整數(比如考試分數 0–100)。計數排序的點子簡單到近乎作弊: 開一個大小為 $k+1$ 的計數陣列,數一數每個值各出現幾次,再依序「倒」出來。 它從頭到尾沒有比較過任何兩個元素。 

def counting_sort(arr):
    if not arr:
        return []
    k = max(arr)                       # 最大值,決定計數陣列大小(假設值 ≥ 0)
    count = [0] * (k + 1)              # count[v] = 值 v 出現的次數
    for x in arr:                      # ① 掃一遍,數次數:O(n)
        count[x] += 1
    out = []
    for v in range(k + 1):             # ② 依序倒出來:O(n + k)
        out.extend([v] * count[v])
    return out

print(counting_sort([2, 5, 2, 0, 3, 0]))   # [0, 0, 2, 2, 3, 5]

它的複雜度是 $O(n + k)$——當值域 $k$ 和元素個數 $n$ 同級時,這就是不折不扣的線性排序, 把 $\Omega(n\log n)$ 遠遠甩在後頭。但魔鬼藏在那個 $k$ 裡:如果你要排的整數可以大到上億(比如排序所有人的身分證號), $k$ 就會大到計數陣列塞爆記憶體。計數排序只在「值域不大」時划算。

⚠️
天下沒有白吃的午餐:它沒有違反下界

計數排序 $O(n)$ 不代表 $\Omega(n\log n)$ 的證明錯了。下界的前提是「基於比較」, 而計數排序根本不比較——它用「值」當記憶體地址直接定位。它用對輸入的額外假設 (值是範圍有限的整數)和額外的空間 $O(k)$,換來了時間上的突破。 這正是演算法世界反覆出現的交易:時間、空間、通用性,三者難以兼得。

基數排序(radix sort)與桶排序(bucket sort):計數排序的兩種放大術

計數排序怕「值域太大」。有兩個漂亮的點子能繞過這個限制:

  • 基數排序:不要把整個數字當一個值,而是一位數一位數地排。 先依個位數排(用穩定的計數排序),再依十位數排,再百位數……排完最高位,整體就有序了。 這裡穩定性是命脈:每一輪都必須保持前幾輪建立的順序,否則前功盡棄。 對 $d$ 位數、每位值域 $b$(例如十進位 $b=10$)的資料,複雜度是 $O(d\,(n+b))$—— 當位數 $d$ 視為常數時,等於線性。這就是排「固定長度整數或字串」的利器。
  • 桶排序:假設資料大致均勻分布在某個區間(例如 $[0,1)$ 的小數)。 把區間切成 $n$ 個「桶」,依值把每個元素丟進對應的桶,再對每個桶內部各自排序(桶內元素少,用插入排序就好), 最後依序串起來。在均勻分布的假設下,平均是 $O(n)$;但若資料全擠在同一個桶,就退化回桶內排序的複雜度。

三者共通的精神都是:用結構性的假設(整數、有限位數、均勻分布)取代「比較」, 把元素直接安置到位。它們不是萬用排序,而是「知道資料長相時」的特化武器。

回到搜尋:把二分搜尋的邊界釘死

排序鋪好了路,現在回到搜尋。線性搜尋($O(n)$,從頭找到尾)在第 5 章講過, 它的好處是不挑資料、隨時能用。但只要資料已排序,二分搜尋($O(\log n)$) 就能降維打擊:每次看中間,砍掉一半。概念你已經懂了,這一節我們要做的是更難、也更重要的事—— 把它的邊界與正確性寫到滴水不漏。二分搜尋是出了名「人人都覺得會、人人都寫錯」的演算法。

寫對二分搜尋的關鍵,是時時抓著一個迴圈不變量(loop invariant): 「答案(若存在)永遠落在區間 [low, high] 之內」。只要每一步都維持這個不變量、 而且區間嚴格縮小,演算法就保證正確且會停。注意三個最常出錯的細節: (1) 區間是閉區間 [low, high] 還是半開 [low, high)? (2) 迴圈條件是 < 還是 <=? (3) 更新時是 mid 還是 mid ± 1?三者必須首尾一致。 

def binary_search(arr, target):
    """在已排序的 arr 中找 target,回傳索引;找不到回傳 -1。"""
    low, high = 0, len(arr) - 1        # 閉區間 [low, high]
    while low <= high:                 # 區間非空時才繼續(注意是 <=)
        mid = low + (high - low) // 2  # 比 (low+high)//2 更防溢位(C/Java 重要)
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            low = mid + 1             # target 在右半,丟掉含 mid 的左半
        else:
            high = mid - 1           # target 在左半,丟掉含 mid 的右半
    return -1                         # 區間縮空了還沒找到

nums = [2, 5, 8, 12, 16, 23, 38, 45, 56, 72]
print(binary_search(nums, 23))   # 5
print(binary_search(nums, 100))  # -1
💡
為什麼是 low + (high - low) // 2?

在 Python 裡 (low + high) // 2 沒問題,因為整數無上限。但在 C、Java 這類有固定位寬整數的語言,當 low 與 high 都很大時,low + high 可能溢位(overflow)變成負數,釀成隱晦的 bug。寫成 low + (high - low) // 2 就永遠不會溢位。這個 bug 真實地潛伏在 Java 標準庫的二分搜尋裡長達九年,才被發現修正—— 二分搜尋的邊界,真的不簡單。

變體:找「第一個」「最後一個」與 lower_bound

標準二分搜尋找到「某一個」等於 target 的位置就回傳。但實務上你常想問更精細的問題: 當有重複值時,第一個等於 target 的位置在哪?或者更通用——第一個 ≥ target 的位置 在哪(這正是 C++ std::lower_bound 做的事)?它能回答「target 該插在哪」「有幾個元素小於 target」這類查詢。

訣竅是改用半開區間 [low, high),而且找到候選時不要急著回傳—— 記住它,繼續往左壓縮,看看左邊有沒有更靠前的合格者: 

def lower_bound(arr, x):
    """回傳第一個 >= x 的索引;若全部都 < x,回傳 len(arr)。"""
    low, high = 0, len(arr)            # 半開區間 [low, high),注意 high = len(arr)
    while low < high:                  # 半開區間,用 <(不是 <=)
        mid = low + (high - low) // 2
        if arr[mid] < x:
            low = mid + 1             # mid 太小,捨棄;答案在 (mid, high)
        else:
            high = mid               # arr[mid] >= x,mid 可能就是答案 → 保留它
    return low                         # low == high,即第一個 >= x 的位置

a = [1, 2, 2, 2, 5, 8]
print(lower_bound(a, 2))   # 1  ← 第一個 2 的位置
print(lower_bound(a, 3))   # 4  ← 3 該插在索引 4(第一個 >= 3 的是 5)
print(lower_bound(a, 9))   # 6  ← 全都比 9 小,回傳 len(a)

有了 lower_bound,很多查詢都變得輕鬆:「有幾個元素 < x」就是 lower_bound(a, x); 「第一個 > x」就是 lower_bound(a, x+1)(整數情形); 「x 出現幾次」就是 lower_bound(a, x+1) - lower_bound(a, x)。 Python 標準庫的 bisect 模組(bisect_left / bisect_right) 正是把這些寫好、調校到滴水不漏的版本,實務上直接用它即可。

ℹ️
二分搜尋不只用在「找數字」

它真正威力來自一個抽象:只要存在一個單調的判定條件(某個分界點之前都是「否」、之後都是「是」), 你就能用二分法在 $O(\log n)$ 內找到那個分界。這招叫「對答案二分」, 能解「最小可行容量」「最大化最小值」這類乍看不像搜尋的最佳化問題——是競賽與面試的常客。

選擇問題:只想要第 $k$ 小,何必整串排好?

最後一個漂亮的問題。假設你只想知道「第 $k$ 小的元素是誰」——比如中位數($k = n/2$)、 或考試的第 90 百分位。最直覺的做法是先排序($O(n\log n)$)再取第 $k$ 個。但這太浪費了: 我們把全部都排好,只為了拿一個。能不能更快?

能。關鍵的領悟是:快速排序的分割步驟,其實已經幫我們做了大半的事。 每次分割後,pivot 落在某個最終位置 p——這代表「pivot 是第 p 小的元素」(0 起算), 而且左邊全比它小、右邊全比它大。於是:

  • 若 p == k:賓果,pivot 就是答案。
  • 若 p < k:第 $k$ 小在右半,只要遞迴右半就好——左半完全不用碰。
  • 若 p > k:第 $k$ 小在左半,只遞迴左半。

這就是 quickselect(快速選擇)。和快速排序不同的是,它每次只追一邊, 不像排序要兩邊都遞迴。平均下來,工作量是 $n + \frac{n}{2} + \frac{n}{4} + \dots \approx 2n = O(n)$—— 線性時間就能找出第 $k$ 小,不必整串排好!(最壞情況和快速排序一樣是 $O(n^2)$,同樣用隨機 pivot 化解。) 

import random

def quickselect(arr, k):
    """回傳 arr 中第 k 小的元素(k 從 0 起算,k=0 即最小值)。"""
    if not 0 <= k < len(arr):
        raise IndexError("k 超出範圍")
    a = list(arr)                      # 複製一份,避免改動呼叫者的資料
    low, high = 0, len(a) - 1
    while True:
        p = partition(a, low, high)    # 借用前面隨機化的 Lomuto partition
        if p == k:
            return a[p]
        elif p < k:
            low = p + 1               # 第 k 小在右半,只追右邊
        else:
            high = p - 1              # 在左半,只追左邊

data = [7, 2, 1, 8, 6, 3, 5]
print(quickselect(data, 0))   # 1  (最小)
print(quickselect(data, 3))   # 5  (第 4 小,即中位數)
print(quickselect(data, 6))   # 8  (最大)
ℹ️
最壞也 $O(n)$:中位數的中位數

隨機化讓 quickselect 期望 $O(n)$,但理論上仍可能很慢。1973 年,Blum、Floyd、Pratt、Rivest、Tarjan 提出「中位數的中位數(median of medians)」:把元素每 5 個一組、各取組內中位數, 再遞迴地求這些中位數的中位數當 pivot。這個精心挑選的 pivot 保證每次至少切掉固定比例, 讓選擇問題的最壞情況也是 $O(n)$。它是理論上的里程碑(常數較大,實務多半還是用隨機化版本), 證明了「找第 $k$ 小」本質上不比「掃一遍」更難。

實務:站在巨人的標準庫上

講了這麼多手刻排序,最後要說一句最重要的實話:在真實專案裡,請用標準庫的排序。 它們由頂尖工程師打磨多年,比你我手寫的快、穩、又正確。手刻排序的價值在於「理解」,而非「上線」。

Python 提供兩個工具,背後都是高度優化的 Timsort(穩定、最壞 $O(n\log n)$):

# sorted():回傳「新的」排好序的串列,不動原本的
nums = [5, 2, 9, 1, 7]
print(sorted(nums))            # [1, 2, 5, 7, 9]
print(sorted(nums, reverse=True))   # [9, 7, 5, 2, 1]
print(nums)                    # [5, 2, 9, 1, 7]  ← 原串列沒變

# list.sort():原地排序,改動原串列本身,回傳 None
nums.sort()
print(nums)                    # [1, 2, 5, 7, 9]

最強大的是 key 參數:你給它一個函式,Python 會依「每個元素經過該函式算出的鍵」來排序。 這讓你能排序任何複雜物件: 

words = ["banana", "Kiwi", "apple", "fig"]
print(sorted(words, key=len))            # 依長度:['fig', 'Kiwi', 'apple', 'banana']
print(sorted(words, key=str.lower))      # 不分大小寫的字典序

students = [
    {"name": "小明", "score": 88, "age": 19},
    {"name": "小華", "score": 88, "age": 17},
    {"name": "小美", "score": 95, "age": 18},
]
# 多重排序鍵:先依分數「由高到低」,分數相同時年齡「由小到大」
ranked = sorted(students, key=lambda s: (-s["score"], s["age"]))
for s in ranked:
    print(s["name"], s["score"], s["age"])
# 小美 95 18 / 小華 88 17 / 小明 88 19

注意上面用 tuple (-score, age) 當鍵,就能一次表達「主鍵、次鍵」的排序規則—— tuple 是按元素逐一比較的。而因為 Timsort 穩定,你也可以用另一招: 「先排次要鍵,再排主要鍵」,穩定性會保證次要鍵的順序在主鍵相同時被保留下來。 這正是穩定排序在實務上最常見的用途。

至於 C,標準庫提供 qsort(名字源自 quicksort,但實作不保證、通常不穩定), 你得自己寫一個「比較函式」傳給它——這是 C 用函式指標達成泛型的經典手法,我們在系統那幾章已經見識過。

🔑
穩定性,到底何時重要?

當你的元素「不只是一個數字」、而是有多個欄位的紀錄,且你想分階段、多鍵排序時,穩定性就攸關正確。 例:先把員工名單依「姓名」排好,再依「部門」排——若排序穩定,同部門內的人會維持姓名順序; 若不穩定,第二次排序會把第一次的成果打亂。對「純數字、不在乎相等元素誰先誰後」的排序,穩定性則無關緊要。 記住這個判準:你會不會在乎「值相等的元素,排完後的相對先後」?會,就需要穩定排序。

✏️ 練習 28.1 實作

實作一個原地、隨機化的快速排序 quicksort(arr)(用 Lomuto 分割)。 要求:(1) 隨機選 pivot 以避開最壞情況;(2) 直接在原陣列上排序。 並說明:為什麼隨機化能讓「已排序的輸入」不再是它的剋星?

看解答 
import random

def quicksort(arr, low=0, high=None):
    if high is None:
        high = len(arr) - 1
    if low >= high:                  # 剩 0 或 1 個,已有序
        return
    p = partition(arr, low, high)
    quicksort(arr, low, p - 1)
    quicksort(arr, p + 1, high)

def partition(arr, low, high):
    r = random.randint(low, high)              # 隨機挑 pivot
    arr[r], arr[high] = arr[high], arr[r]      # 換到最右
    pivot = arr[high]
    i = low - 1
    for j in range(low, high):
        if arr[j] <= pivot:
            i += 1
            arr[i], arr[j] = arr[j], arr[i]
    arr[i + 1], arr[high] = arr[high], arr[i + 1]
    return i + 1

# 測試,包含「已排序」這個對固定-pivot 版本最毒的輸入
data = list(range(20))
random.shuffle(data)
quicksort(data)
assert data == list(range(20))
print("通過 ✅")

為什麼隨機化有效?「永遠選最右邊」的固定規則,讓「已排序輸入」每次都把 pivot 選到最大值, 切出 $0$ 與 $n-1$ 的極端分割,退化成 $O(n^2)$。一旦改成隨機選 pivot, pivot 的好壞就與「輸入的排列方式」脫鉤了——沒有任何特定輸入能持續逼出爛 pivot。 期望分割相當均勻,期望時間 $O(n\log n)$,與輸入長相無關。

✏️ 練習 28.2 實作

實作計數排序 counting_sort(arr),支援含負數的整數(例如 [-3, 1, -3, 0, 2])。 提示:用 min 求出偏移量,把每個值平移成非負的索引。

看解答

關鍵是用 lo = min(arr) 當「零點」,值 x 對應到計數陣列的索引 x - lo。

def counting_sort(arr):
    if not arr:
        return []
    lo, hi = min(arr), max(arr)
    count = [0] * (hi - lo + 1)        # 索引 0 對應最小值 lo
    for x in arr:
        count[x - lo] += 1            # 平移後當索引
    out = []
    for i, c in enumerate(count):
        out.extend([i + lo] * c)      # 還原:索引 i 對應值 i + lo
    return out

print(counting_sort([-3, 1, -3, 0, 2]))   # [-3, -3, 0, 1, 2]
print(counting_sort([]))                   # []

複雜度 $O(n + k)$,其中 $k = \text{hi} - \text{lo} + 1$ 是值域大小。 當值域遠大於元素個數(例如 [0, 1000000] 只有兩個元素)時,它會浪費大量空間,此時不該用計數排序。

✏️ 練習 28.3 實作

在一個已排序、可能有重複的陣列裡,實作 lower_bound(arr, x): 回傳第一個 ≥ x 的索引;若不存在(全部都 < x),回傳 len(arr)。 請寫出迴圈不變量,並用它論證程式為何正確。

看解答 
def lower_bound(arr, x):
    low, high = 0, len(arr)           # 半開區間 [low, high)
    while low < high:
        mid = low + (high - low) // 2
        if arr[mid] < x:
            low = mid + 1            # 答案不可能 <= mid,推進 low
        else:
            high = mid              # arr[mid] >= x,mid 可能是答案,保留
    return low

a = [1, 2, 2, 2, 5, 8]
assert lower_bound(a, 2) == 1
assert lower_bound(a, 3) == 4
assert lower_bound(a, 0) == 0
assert lower_bound(a, 9) == 6
print("通過 ✅")

不變量:「答案一定落在 [low, high] 之內」——即 arr[low-1] < x (low 左邊的都太小)且 arr[high] >= x 或 high == len(arr)(high 及其右邊都合格或越界)。 每輪檢查 mid:若 arr[mid] < x,則 mid 與其左側都不合格,安全地令 low = mid + 1; 否則 mid 是個合格候選,但左邊可能還有更早的,故令 high = mid(保留 mid)。 區間每輪嚴格縮小,最終 low == high,正是第一個 ≥ x 的位置。 用 high = mid(不減一)又能停機,正是因為配對的是「半開區間 + low < high」。

✏️ 練習 28.4 思考

用自己的話解釋:為什麼任何「基於比較」的排序,最壞情況都至少要 $\Omega(n\log n)$ 次比較? 並回答:計數排序是 $O(n)$,這有沒有推翻這個下界?為什麼?

看解答

下界論證(決策樹):一個比較排序對輸入唯一的操作就是「比較兩個元素」,每次比較有「≤/>」兩種結果。 因此整個演算法可畫成一棵二元決策樹:內部節點是比較,分支是結果,每片葉子是一種輸出排列。 因為 $n$ 個相異元素有 $n!$ 種可能排列,而正確排序必須能產生其中任何一種, 所以樹至少有 $n!$ 片葉子。高度 $h$ 的二元樹最多 $2^h$ 片葉子,於是 $2^h \ge n!$,即 $h \ge \log_2(n!)$。再用 $\log_2(n!) \ge \frac{n}{2}\log_2\frac{n}{2} = \Omega(n\log n)$ (或史特靈公式 $\log_2(n!)\approx n\log_2 n$)。樹高就是最壞比較次數,故任何比較排序都 $\ge \Omega(n\log n)$。

計數排序沒有推翻它。下界的前提是「基於比較」。計數排序不比較任何一對元素—— 它用元素的值直接當計數陣列的索引來定位。它是靠「對輸入的額外假設(範圍有限的整數)」與「額外空間 $O(k)$」 跳出了比較模型,因此不受這條下界約束。下界依然對所有比較排序成立。

✏️ 練習 28.5 實作

給定一串書籍紀錄,用 sorted 的 key 參數把它們排成: 先依「年份」由新到舊;同年份時,依「書名」字典序由 A 到 Z(不分大小寫)。

books = [

看解答

用 tuple 當複合鍵:第一鍵 -year(負號達成「由新到舊」),第二鍵 title.lower()(不分大小寫的字典序)。

ranked = sorted(books, key=lambda b: (-b["year"], b["title"].lower()))

    print(b["year"], b["title"])

✏️ 練習 28.6 實作

實作 quickselect(arr, k),在平均 $O(n)$ 時間內找出第 $k$ 小的元素(k 從 0 起算)。 不可以先把整串排好。再回答:為什麼平均是 $O(n)$ 而不是 $O(n\log n)$?

看解答 
import random

def quickselect(arr, k):
    if not 0 <= k < len(arr):
        raise IndexError("k 超出範圍")
    a = list(arr)                     # 複製,避免動到原陣列
    low, high = 0, len(a) - 1
    while True:
        p = partition(a, low, high)  # 用練習 28.1 的隨機化 Lomuto partition
        if p == k:
            return a[p]
        elif p < k:
            low = p + 1             # 只追右半
        else:
            high = p - 1           # 只追左半

data = [7, 2, 1, 8, 6, 3, 5]
assert quickselect(data, 0) == 1
assert quickselect(data, 3) == 5
assert quickselect(data, 6) == 8
print("通過 ✅")

為什麼是 $O(n)$?關鍵差別:快速排序分割後要遞迴兩邊,所以是 $T(n) = 2T(n/2) + O(n) = O(n\log n)$;而 quickselect 只需要追含答案的那一邊。 在均勻分割的期望下,工作量是 $n + \frac{n}{2} + \frac{n}{4} + \dots$,這個等比級數收斂到約 $2n$,即 $O(n)$。 「少走一半的路」省掉了那個 $\log n$ 因子。(最壞仍 $O(n^2)$,隨機化使其機率極低; 要鐵打的最壞 $O(n)$ 則需中位數的中位數選 pivot。)

✏️ 練習 28.7 挑戰

寫一個函式 is_stable(sort_fn),它接受一個「依 key 排序」的排序函式, 並實測該排序是否穩定(回傳 True/False)。 用它驗證你練習 28.1 的快速排序與 Python 內建 sorted。最後,舉一個具體情境 說明穩定性何時攸關正確。

看解答

測法:造一組記錄,每筆是 (key, 進場序號)。只依 key 排序後, 對每個相同 key 的群組,檢查其「進場序號」是否仍維持遞增——若被打亂,即不穩定。

def is_stable(sort_fn):
    # key 故意大量重複,序號 0..n-1 記錄原始順序
    data = [(2, 0), (1, 1), (2, 2), (1, 3), (2, 4), (1, 5)]
    out = sort_fn(data, key=lambda p: p[0])
    # 蒐集每個 key 對應的「序號出現順序」
    seen = {}
    for key, idx in out:
        seen.setdefault(key, []).append(idx)
    # 穩定 ⇔ 每個 key 的序號清單都是遞增的
    return all(lst == sorted(lst) for lst in seen.values())

# Python 內建 sorted 是穩定的(Timsort)
print(is_stable(sorted))                       # True

# 包裝一下快速排序,讓它符合 (data, key) 介面後測試
def quicksort_keyed(data, key):
    if len(data) <= 1:
        return list(data)
    pivot = key(data[len(data) // 2])
    less    = [x for x in data if key(x) < pivot]
    equal   = [x for x in data if key(x) == pivot]
    greater = [x for x in data if key(x) > pivot]
    return quicksort_keyed(less, key) + equal + quicksort_keyed(greater, key)

print(is_stable(quicksort_keyed))   # True 或 False?見下方說明

有趣的是,上面這個用串列生成式寫的快速排序其實是穩定的——因為 less/equal/greater 都按原順序逐一收集,沒有打亂相對位置。 但原地 Lomuto 版本不穩定:它的「長距離交換」會把相等元素的相對順序弄亂。 你可以把 is_stable 套到原地版本(需先包裝介面)親眼驗證它回傳 False。 這也點出一個重點:「快速排序穩不穩定」取決於實作方式,而不是演算法名稱。

穩定性何時攸關正確?一個具體情境:電商後台的訂單表,使用者先點「依下單時間排序」, 看到一個順序;接著再點「依金額排序」。如果排序穩定,金額相同的訂單會保留剛才的時間順序, 符合使用者「在金額相同時,沿用上一個排序」的心理預期;若不穩定,相同金額的訂單順序會莫名跳動, 體驗很差。更廣義地說,任何「多階段、分鍵」的排序(如基數排序、先次鍵後主鍵的技巧)都依賴穩定性才會正確。

把這一章串起來

從「把一疊東西排好」這個樸素的問題出發,我們走了一趟濃縮的演算法之旅:

  • 排序無所不在,而且常是其他演算法(二分搜尋、去重、找中位數……)能變快的前置步驟——它是「賦能型」演算法。
  • 三個 $O(n^2)$ 排序各有個性:選擇排序交換最少、插入排序在「近乎有序/小資料」時是冠軍、氣泡排序主要用於教學。
  • 快速排序把分治磨成刀:核心是分割,讓 pivot 一次歸位;平均 $O(n\log n)$、最壞 $O(n^2)$,隨機化 pivot 用骰子化解最壞情況,而且原地。
  • 快速排序 vs 合併排序是一場取捨:前者快、原地、不穩定;後者最壞有保證、穩定、但要 $O(n)$ 空間。標準庫用 introsort / Timsort 等混合策略各取所長。
  • $\Omega(n\log n)$ 下界:用決策樹證明,任何比較排序都逃不過——$n!$ 片葉子,樹高至少 $\log_2(n!)\approx n\log n$。這是一道優美而堅固的牆。
  • 非比較排序(計數 $O(n+k)$、基數、桶)不靠比較,用「值即地址」的假設與額外空間跳出牆外,但只在資料長相配合時划算。
  • 二分搜尋的威力在於把邊界釘死:抓住迴圈不變量、區間與條件首尾一致,就能寫出 lower_bound 等變體。
  • 選擇問題不必整串排好:quickselect 只追一邊,平均 $O(n)$;中位數的中位數把最壞也壓到 $O(n)$。
  • 實務上請用標準庫:Python 的 sorted/list.sort 是穩定的 Timsort,善用 key 與多重鍵,並理解穩定性何時攸關正確。

這一章你看到了演算法世界最迷人的幾種風景:用分治把問題切碎、用隨機性馴服最壞情況、 用一棵樹證明「沒有更好的方法存在」、再用額外的假設聰明地繞過它。 這些思想——分治、隨機化、下界證明、空間換時間——會在接下來的章節裡反覆出現。 排序只是它們第一次同台演出的舞台。
