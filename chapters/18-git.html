沒有版本控制的世界:一場惡夢

先講個你大概親身經歷過的故事。你在寫一份重要文件(或一段程式),為了「保險」,你開始這樣存檔: 

報告.docx
報告_修改版.docx
報告_修改版2.docx
報告_老闆改過.docx
報告_final.docx
報告_final_v2.docx
報告_final_v2_真的final.docx
報告_final_v2_真的final_這次真的.docx

於是惡夢開始了。三天後你完全不記得 真的final 跟 這次真的 差在哪;你想把某一版的第三段、跟另一版的結論合在一起,只能兩個檔開著肉眼比對、手動複製; 同事傳來他改的版本,你打開一看——他是改在 v2 上還是 老闆改過 上?沒人知道。 最後你硬著頭皮全選貼上,蓋掉了某段其實很重要的內容,而且再也回不去了。

這就是「土法煉鋼版本控制」的下場。它有四個根本問題:你看不出兩個版本差在哪、你無法安全地回到過去任一版、你沒辦法把不同人的修改乾淨地合併、而且沒有任何紀錄告訴你「這一行為什麼會變成這樣、是誰、什麼時候、為了什麼改的」。

版本控制系統(version control system, VCS)就是專門來終結這場惡夢的工具。它幫你做到四件事:

  • 時光機:完整記錄每一次變更,你能隨時回到任何一個歷史版本,毫髮無傷。
  • 協作:多人能同時改同一個專案,系統幫你把大家的修改安全地合併起來。
  • 實驗分支:想試一個大膽的點子?開一條「平行宇宙」去搞,失敗就丟掉,完全不影響主線。
  • 責任追蹤:每一行程式都查得到「最後是哪一次提交、由誰、何時、為什麼改的」(這叫 git blame,雖然名字聽起來很兇)。
📜
Git 的身世:一場分手催生的工具

Git 是 Linux 之父 Linus Torvalds 在 2005 年寫出來的。當時 Linux 核心開發團隊用的商業工具 BitKeeper 撤回了免費授權, Linus 一氣之下,花了大約十天親手刻出一個全新的版本控制系統,目標是快、分散式、而且資料絕對不會壞。 至於名字,他半開玩笑地說 git 是英式英語裡「討厭的傢伙」的意思——「我寫的所有東西都以我自己命名,先是 Linux,現在是 git。」如今它幾乎統治了整個軟體世界。

Git 的心智模型(這一節最重要)

絕大多數人學 Git 卡關,不是因為指令難背,而是因為腦中沒有正確的模型。 他們把 add、commit、push 當成三句沒道理的咒語死記,結果一遇到非典型狀況就完全不知道自己在幹嘛。 所以我們先不碰任何指令,花點時間把 Git 的世界觀建立起來。建好了,後面的指令會像呼吸一樣自然。

關鍵一:Git 存的是「快照」,不是「差異」

很多其他版本控制工具(例如老一輩的 SVN)是這樣運作的:它記錄「每個檔案隨時間累積了哪些差異(diff)」。 要還原某個版本,它得從最初的檔案開始,把一筆筆差異疊加回去。

Git 的想法根本不同。每當你提交(commit)一次,Git 做的事情比較像幫你整個專案的當下狀態拍一張照片——一張快照(snapshot)。 它記住「在這個時間點,所有檔案分別長什麼樣子」。如果某個檔案這次沒改,Git 不會重存一份,它聰明地用一個指標指向上一版那份一模一樣的內容(這就是為什麼 Git 又快又省空間)。 但你心智上應該把每個 commit 想成一張完整的快照,而不是一堆差異的疊加。

🔑
commit = 一張快照

每一次 commit,都是替整個專案在那個瞬間拍下的一張完整照片,並蓋上一個獨一無二的指紋(一串叫 hash 的英數字,例如 a3f9c2e)。 「回到過去」對 Git 來說,只是「把工作目錄換成某張舊照片的內容」——既快又安全。記住這句話,Git 的一切行為都會變得合理。

關鍵二:三個區域

Git 把你的專案分成三個區域,你下的每一個指令,本質上都是在「把東西從一個區域搬到另一個區域」。搞懂這張圖,你就懂了一大半的 Git:

  • 工作目錄(working directory):你眼睛看得到、編輯器正在改的那些檔案。這是你的「工作桌面」。
  • 暫存區(staging area,又叫 index):一個「下次要拍進快照的草稿」。你用 git add 把改好的東西放進來,等於跟 Git 說「這些我準備好了,下次提交算它一份」。
  • 儲存庫(repository):藏在 .git 資料夾裡的歷史檔案庫,存放所有已經提交的快照。一旦 commit,東西就被永久收進這裡。

為什麼要有「暫存區」這個中間站,不能改完直接提交就好?因為它讓你能精挑細選: 假設你這次同時修了一個 bug、又順手改了排版。你可以只 add 修 bug 的檔案、提交成一個 commit(訊息寫「修正登入崩潰」), 再 add 排版的改動、提交成另一個(訊息寫「調整首頁間距」)。歷史因此乾淨、每個 commit 只做一件事。暫存區就是你「組裝下一張快照」的工作台。

關鍵三:commit 是 DAG 上的節點

現在把每張快照(commit)串起來看。每個 commit 除了存著快照,還記著一個關鍵資訊:它的父節點(parent)是誰——也就是「我是從哪張快照接著做下來的」。 於是所有 commit 連成一條鏈,每個節點指向它的前一個:

當有了分支,這條鏈會分岔、再合流,變成一張有向無環圖(Directed Acyclic Graph, DAG):「有向」是因為每條邊都指向父節點;「無環」是因為你不可能成為自己的祖先(時間不會倒流)。 Git 的整個歷史,本質上就是這張 DAG。幾乎所有進階操作(分支、合併、rebase、reset),都只是在這張圖上移動指標、或新增節點而已。 當你心裡能畫出這張圖,Git 就再也不神秘了。

起步:第一個 repo,前三個 commit

理論講夠了,動手。所有指令都在終端機裡輸入(如果你還不熟終端機,可以先翻翻 Shell 那一章)。

一次性的全域設定

裝好 Git 後,先告訴它你是誰——這樣每個 commit 才知道作者署名。--global 代表「對這台機器上所有 repo 都生效」,只需設定一次: 

# 設定你的署名(會記在每個 commit 上)
git config --global user.name "Lin Xiaoming"
git config --global user.email "you@example.com"

# 讓 Git 輸出有顏色,好讀很多
git config --global color.ui auto

# 把預設分支名設成 main(現代慣例,取代舊的 master)
git config --global init.defaultBranch main

# 檢查設定
git config --list

建立 repo 並查看狀態

進到你的專案資料夾,用 git init 把它變成一個 Git 倉庫——這會偷偷建立一個 .git 隱藏資料夾,你的整段歷史都會住在那裡: 

mkdir my-site && cd my-site     # 建立並進入專案資料夾
git init                          # 初始化 repo,出現 .git/
git status                        # 隨時都該下的指令:現在是什麼狀況?

git status 是你最該養成反射動作的指令——隨時不確定就打它。剛 init 完它會說: 

On branch main

No commits yet

nothing to commit (create/copy files to get started)

現在來建立第一個檔案,再看一次狀態:

echo "# 我的個人網站" > README.md
git status
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        README.md

nothing added to commit but untracked files present

Git 看到了 README.md,但說它是「未追蹤(untracked)」——意思是「這檔案我注意到了,但你還沒叫我管它」。 對照圖 18-1:它現在在工作目錄,還沒進暫存區。

add 與 commit

git add README.md          # 放進暫存區(也可用 git add . 加入全部)
git status                 # 現在 README.md 變成「Changes to be committed」(綠色)
git commit -m "Add project README"   # 拍成快照,存進儲存庫

恭喜,你的第一張快照誕生了!-m 後面是commit 訊息,用來說明這次改了什麼。我們繼續做第二、第三個 commit: 

# 第二個 commit
echo "<h1>Hello</h1>" > index.html
git add index.html
git commit -m "Add home page"

# 第三個 commit
echo "body { font-family: sans-serif; }" > style.css
git add style.css
git commit -m "Add basic stylesheet"

用 log 回顧歷史

git log 列出所有 commit。加上幾個旗標會好讀非常多:

git log                      # 完整資訊(hash、作者、日期、訊息)
git log --oneline            # 每個 commit 濃縮成一行
git log --oneline --graph    # 再畫出分支的圖形,協作時超好用
$ git log --oneline
c91d8af (HEAD -> main) Add basic stylesheet
f02ab57 Add home page
a3f9c2e Add project README

看到 (HEAD -> main) 了嗎?這正是圖 18-2 在現實中的樣子:main 指向最新的 commit,HEAD 指向 main。 左邊那串 c91d8af 就是每個 commit 的短 hash 指紋,之後很多指令都會用它來指名某個 commit。

🔑
寫好 commit 訊息:給未來的自己一封信

commit 訊息不是給 Git 看的,是給三個月後完全忘記這段在幹嘛的你自己、以及你的隊友看的。業界通行的慣例:

  • 標題用祈使句、現在式:寫「Fix login crash」(修正登入崩潰),不要寫「fixed」或「修了一些東西」。一個訣竅:它應該能接在「If applied, this commit will ___」後面通順。
  • 標題簡短(約 50 字內),需要細節就空一行,在下面寫正文,解釋「為什麼」而不只是「做了什麼」。
  • 一個 commit 只做一件事。不要把「修 bug + 改排版 + 升級套件」塞進同一個 commit。

壞訊息長這樣:update、asdf、改好了、fix bug。好訊息長這樣:Fix crash when shopping cart is empty。

ℹ️
Conventional Commits:給訊息加個前綴

很多團隊採用一種叫 Conventional Commits 的格式,在標題前加上類型前綴,讓歷史一眼可分類、甚至能自動產生更新日誌: feat:(新功能)、fix:(修 bug)、docs:(文件)、refactor:(重構)、test:(測試)、chore:(雜務)。 例如 feat: add dark mode toggle。不強制,但是個好習慣。

看變化:diff 與 show

版本控制最基本的價值,就是「看出到底哪裡變了」。Git 用 git diff 回答這件事,但你得先搞清楚它在比較哪兩個區域——這又回到三區模型了。 

git diff              # 工作目錄 vs 暫存區:我改了、但還沒 add 的部分
git diff --staged     # 暫存區 vs 上次 commit:我已 add、即將提交的部分
                      #   (--staged 與 --cached 同義)
git diff HEAD         # 工作目錄 vs 上次 commit:全部尚未提交的改動

假設我們把 README.md 改了一行還沒 add,git diff 會輸出:

diff --git a/README.md b/README.md
index 1f3a2b0..9c4e1d7 100644
--- a/README.md
+++ b/README.md
@@ -1 +1,2 @@
 # 我的個人網站
+一個用來練習 Git 的小專案。

讀法很直覺:- 開頭的紅行是移除的內容(這裡沒有),+ 開頭的綠行是新增的內容。 @@ -1 +1,2 @@ 是「位置座標」,告訴你改動發生在第幾行附近。

想看某個已經提交的 commit 到底改了什麼,用 git show: 

git show              # 顯示 HEAD(最新 commit)的完整內容與 diff
git show f02ab57      # 用 hash 指名,顯示某個特定 commit
git show HEAD~1       # HEAD~1 = 上一個 commit;HEAD~2 = 再上一個

這裡出現一個超好用的記號:HEAD~n 代表「從 HEAD 往回數 n 個父節點」。HEAD~1 是上一個 commit,HEAD~2 是上上個。 它讓你能在 DAG 上輕鬆地「往回指」,後面復原時會不斷用到。

✏️ 練習 18.1 實作

從零開始:建立一個新資料夾並初始化 Git,設定好你的姓名與信箱,然後製造出前三個 commit。最後用一行指令印出乾淨的歷史,確認真的有三個。

看解答 
mkdir git-practice && cd git-practice
git init

# (如果還沒設定過全域身分)
git config user.name "Your Name"
git config user.email "you@example.com"

# commit 1
echo "# Git 練習" > README.md
git add README.md
git commit -m "Add README"

# commit 2
echo "print('hello')" > main.py
git add main.py
git commit -m "Add hello script"

# commit 3:修改既有檔案也算一次變更
echo "print('world')" >> main.py
git add main.py
git commit -m "Print world too"

git log --oneline

你會看到三行,每行一個 commit。注意第三個 commit 改的是「既有檔案」——版本控制不只追蹤新檔,也追蹤每一次內容變動。

復原:三種後悔藥,別吃錯

「啊我改壞了」「這個 commit 根本不該做」——這是日常。Git 提供好幾種復原手段,但它們後果差很多,吃錯藥會出事。我們一個一個來。

git restore:丟掉還沒提交的改動

最常見的情境:你把某個檔案改爛了,還沒 commit,只想回到上次提交時的樣子。 

# 丟棄工作目錄裡對某檔案的修改(回到上次 commit 的內容)
git restore index.html

# 把已經 add 進暫存區的檔案「退回」工作目錄(取消 add,改動還在)
git restore --staged index.html

對照圖 18-1:git restore --staged 是那條從暫存區退回工作目錄的箭頭(取消 add);git restore <file> 則是把上次快照的內容覆蓋回工作目錄(丟棄修改)。 (你可能也看過用 git checkout -- <file> 做同樣的事——那是舊寫法,restore 是 Git 2.23 後分出來、語意更清楚的新指令。)

git reset:移動分支指標(三種力道)

git reset 是把分支指標往回搬——讓「目前的 commit」退回到某個較早的 commit。它有三種模式,差別在於「對暫存區和工作目錄下手多重」。這是 Git 最容易搞混、卻最該搞懂的一組指令:

指令分支指標暫存區工作目錄白話
reset --soft後退不動不動「取消提交,但改動全留在暫存區,準備重新 commit」
reset --mixed(預設)後退跟著重設不動「取消提交、也取消 add,但檔案內容保留」
reset --hard後退跟著重設跟著重設「全部抹掉,當作那些 commit 與改動從沒發生過」⚠️
⛔
git reset --hard 會真的刪掉你的工作

git reset --hard 會用目標 commit 的內容覆蓋掉你工作目錄裡尚未提交的所有改動,而且不會問你「確定嗎」。 那些還沒 commit 的修改一旦被它沖掉,基本上救不回來(沒 commit 過的東西連 reflog 都救不了)。 下這個指令前,先停一秒問自己:「我工作目錄裡有沒有還沒提交、又不想丟的東西?」如果有,先 commit 或 stash 起來。

git revert:用「反向 commit」安全地撤銷

reset 是「改寫歷史、假裝那些 commit 沒發生過」。但如果那個爛 commit 已經推送給別人了呢?你一改寫歷史,別人的歷史就跟你對不上,天下大亂。 這時要用 git revert:它不刪除任何歷史,而是產生一個新的 commit,內容剛好是把某個舊 commit 的改動「反過來做一遍」。 

git revert HEAD          # 新增一個 commit,抵銷掉最新那個 commit 的改動
git revert f02ab57       # 抵銷掉指定的某個 commit
🔑
reset vs revert:私有 vs 公開的分水嶺

reset:把指標往回搬、改寫歷史。乾淨俐落,但只適合用在還沒分享出去的本地 commit。
revert:新增一個反向 commit、保留所有歷史。雖然多一筆紀錄,卻是撤銷已經推送、別人也有的 commit 的唯一安全做法。
一句口訣:「歷史還是我自己的,用 reset;歷史已經是大家的,用 revert。」

.gitignore:有些東西就是不該被追蹤

不是所有檔案都該進版本控制。編譯產物、相依套件資料夾、log 檔、還有絕對不能外流的密碼與金鑰,都應該被忽略。 在專案根目錄建一個 .gitignore 檔,把要忽略的模式寫進去,Git 就會對它們視而不見: 

# .gitignore 範例
__pycache__/        # Python 編譯快取
*.pyc               # 所有 .pyc 檔
node_modules/       # 龐大的相依套件,該由 npm install 重建,不該入庫
.env                # 環境變數檔(裡面常有密碼/API key!)
*.log               # 各種 log
.DS_Store           # macOS 自動產生的垃圾檔
dist/               # 打包輸出
⚠️
.gitignore 只擋「還沒被追蹤」的檔案

如果一個檔案已經被 commit 過了,事後才把它加進 .gitignore 是沒用的——Git 已經在追蹤它了。 你得先用 git rm --cached <file> 把它從追蹤名單移除(但保留本機檔案),再 commit。 這對「不小心把密碼提交進去」特別重要——不過要注意,光這樣做,那個密碼還留在歷史紀錄裡,真正外洩過的金鑰一定要當作已洩漏、立刻更換。這部分第七部「資訊安全」會深入談。

分支:Git 的殺手鐧

終於來到 Git 最強大、也最讓它與眾不同的功能:分支(branch)。 分支讓你能開一條「平行宇宙」去開發新功能或試驗瘋狂點子,完全不干擾主線;成功了就合併回來,失敗了就整條丟掉,主線毫髮無傷。

而 Git 的分支之所以快到幾乎免費(其他系統開分支可能要複製整個專案),關鍵在於一個極簡的真相:

🔑
分支只是一個指標

一個分支,本質上只是一個指向某個 commit 的標籤(指標),小到只是一個存著 40 字 hash 的檔案。 建立分支不會複製任何檔案,只是新增一個指標。所以在 Git 開分支、切分支,幾乎是瞬間完成的。 而 HEAD 是一個特殊指標,它指向「你目前所在的分支」——也就是「你現在站在哪裡」。

git branch                 # 列出所有分支,* 標記你目前在哪條
git branch feature-login   # 建立一條叫 feature-login 的新分支(但還沒切過去)
git switch feature-login   # 切換到該分支(HEAD 移過去)
git switch -c feature-login   # 一步到位:建立並立刻切換(-c = create)

# 你也會常看到舊寫法(checkout 身兼切換與其他功能):
git checkout feature-login
git checkout -b feature-login

切到新分支後,你做的每個 commit 都會長在這條分支上,主分支 main 完全不受影響。來看一張分岔的圖:

這就是「平行宇宙」:main 上的 C3 跟 feature 上的 F1、F2 互不影響。你可以在這條 feature 上盡情實驗,隨時用 git switch main 跳回主線——你工作目錄裡的檔案會瞬間變回 C3 的樣子,像沒事一樣。

合併:把分岔的故事線匯流

功能在分支上開發完了,該把它合併(merge)回主線。git merge 有兩種典型情況,值得分清楚。

情況一:快轉合併(fast-forward)

如果你開分支之後,main 完全沒有新的 commit(它還停在你當初分岔的那一點),那合併就簡單到不行:Git 只要把 main 這個指標往前滑到 feature 的最新 commit 就好,根本不用真的「合」。這叫快轉(fast-forward)。

情況二:三方合併(three-way merge)

但更常見的是:你在 feature 上忙的時候,main 也有別人(或你自己)推進了新 commit。兩條線都各自往前走了。 這時無法單純快轉,Git 會找出兩條分支的共同祖先,把「祖先→main」與「祖先→feature」兩邊的改動湊在一起,產生一個特殊的合併提交(merge commit)——它有兩個父節點,正是兩條故事線匯流的證明。因為牽涉三個點(共同祖先 + 兩個分支頂端),所以叫「三方合併」。

實際操作很簡單:先切回要被合併進去的分支(通常是 main),再 merge 對方:

git switch main                 # 先站到主線上
git merge feature-login         # 把 feature-login 合併進來

# 合併完、確認沒問題後,順手刪掉用完的分支
git branch -d feature-login

合併衝突(merge conflict):兩個人改了同一行

如果兩條分支改到了同一個檔案的同一段,Git 沒辦法自作主張決定該採用誰的——這就是合併衝突。 別怕,它不是錯誤,只是 Git 在說「這裡我不敢替你決定,你來」。它會在衝突的檔案裡插入特殊標記: 

<<<<<<< HEAD
<h1>歡迎來到我的首頁</h1>
=======
<h1>Welcome to my homepage</h1>
>>>>>>> feature-login

讀法:

  • <<<<<<< HEAD 到 ======= 之間,是你目前分支(HEAD,這裡是 main)的版本。
  • ======= 到 >>>>>>> feature-login 之間,是要合併進來那條分支的版本。

你的任務:手動編輯這個檔案,把它改成你真正想要的最終樣子,並且把那三行標記(<<<、===、>>>)全部刪掉。 可能是留 A、留 B、留兩者、或寫出全新的第三版。改好後: 

git status              # Git 會列出哪些檔案還在衝突中(Unmerged paths)
# ... 用編輯器手動解決每個衝突檔,刪掉標記 ...
git add index.html      # 解完一個檔就 add,等於告訴 Git「這個我搞定了」
git commit              # 全部解完後提交,完成這次合併
                        # (合併衝突的 commit 通常直接用 Git 給的預設訊息即可)
💡
解衝突不可怕,但要「真的讀懂」兩邊

新手最常犯的錯,是慌張地隨便留一邊、或忘了刪標記,結果把別人的修改默默弄丟、或讓 <<<<<<< 跑進正式程式碼裡。 冷靜下來,逐個衝突問自己:「這兩個版本各自想做什麼?最終正確的樣子是什麼?」現代編輯器(如 VS Code)會把衝突高亮、附上「採用目前/採用傳入/兩者都留」的按鈕,善用它。 真的解到一半想放棄?git merge --abort 能讓你回到合併前的乾淨狀態,重新來過。

rebase:另一種匯流的姿勢

除了 merge,還有第二種把分支整合進來的方式:rebase(變基)。它的想法很不一樣。 merge 是「把兩條線匯流成一個合併提交」;rebase 則是把你分支上的那串 commit「拔起來」,重新接到目標分支的最新 commit 後面,假裝你是從那裡才開始開發的。 

# 站在 feature 分支上,把它重新接到最新的 main 後面
git switch feature-login
git rebase main

rebase 的好處是歷史乾淨——沒有一堆合併提交,看起來像每個人都乖乖地一條線接著做。缺點是它改寫了歷史:F1 變成了一個全新的 commit F1′(hash 不同),原本的 F1 被丟棄了。這個「改寫」的特性,帶來一條鐵律:

⛔
黃金法則:絕不 rebase 已經推送的公開歷史

rebase 會用「新 commit」取代「舊 commit」。如果那些 commit 已經 push 出去、別人也已經基於它們在工作,你一 rebase,大家的歷史就跟你對不上, 別人 pull 下來會看到一團混亂、甚至重複的 commit,合併地獄就此展開。
口訣:「只 rebase 還躺在你自己電腦裡、沒分享給任何人的 commit。」 想整理已推送的歷史?改用 merge,或先跟團隊講好。

ℹ️
那到底該用 merge 還是 rebase?

沒有絕對答案,是團隊風格之爭。常見的折衷做法:在自己的功能分支上,用 rebase 把主線的最新進度同步過來、保持歷史整齊; 真正併回主線時則用 merge(或平台的 PR 合併),留下一個清楚的整合點。記住那條黃金法則,你怎麼選都不會出大錯。

遠端協作:把 repo 放到雲端

到目前為止,一切都發生在你自己的電腦裡。要跟別人協作、或單純想備份,你需要一個遠端(remote)——通常是放在 GitHub、GitLab 之類平台上的一份 repo 副本。

最常見的起手式是複製(clone)一個現成的遠端 repo 到本機: 

git clone https://github.com/someone/cool-project.git
cd cool-project
git remote -v       # 看看連到了哪些遠端
origin  https://github.com/someone/cool-project.git (fetch)
origin  https://github.com/someone/cool-project.git (push)

注意那個 origin:它只是這個遠端的預設暱稱(你 clone 來的那個來源)。你大可叫它別的名字,但全世界的慣例就是叫 origin,看到它別緊張。

如果是把本機既有的 repo連上一個空的遠端,則是:

git remote add origin https://github.com/you/my-site.git
git push -u origin main    # 第一次推送,-u 把本地 main 與遠端 main「綁定」

fetch、pull、push 的分工

這三個指令是協作的核心,務必分清楚誰把東西搬去哪:

指令方向它做什麼
git fetch遠端 → 本機(只下載)把遠端的新 commit 抓下來,但不動你的工作目錄。讓你先「看看別人做了什麼」
git pull遠端 → 本機(下載 + 合併)等於 fetch + merge:抓下來並立刻合併進你目前的分支
git push本機 → 遠端把你本機的 commit 上傳到遠端,分享給大家

關鍵差別在 fetch vs pull:fetch 是「只把信收進信箱,還沒拆」——安全,讓你能先 git log origin/main 看看遠端有什麼變化,再決定要不要合。 pull 則是「收信而且立刻把內容套用到你身上」——方便,但偶爾會在你毫無心理準備時跳出合併衝突。新手可以先養成「fetch 看一下,再 merge」的習慣,比較不會嚇到。 

# 典型的每日協作循環
git pull                       # 開工前先同步別人的進度
# ... 寫程式、git add、git commit ...
git push                       # 收工前把自己的成果推上去
ℹ️
追蹤分支:本地 main 與 origin/main 的關係

當你 push -u 或 clone 時,本地的 main 會和遠端的 origin/main 建立追蹤(tracking)關係。 有了它,你之後只要打 git push / git pull,Git 就知道「要跟誰同步」,不必每次都寫一長串。 git status 也會貼心地告訴你「你比遠端領先 2 個 commit」或「落後 3 個 commit」。origin/main 是一個唯讀的「遠端追蹤分支」,代表「上次同步時,遠端 main 的樣子」。

GitHub 與 Pull Request 工作流程

Git 是工具,GitHub 是把這個工具社交化的平台。它最重要的發明,是讓「貢獻程式碼」變成一套有禮貌、可審查的流程——核心就是 Pull Request(PR)。

Fork + Pull Request:對「別人的」專案做貢獻

假設你想替一個你沒有寫入權限的開源專案修個錯字,流程是這樣:

  1. Fork:在 GitHub 網頁上按「Fork」,把那個專案複製一份到你自己的帳號底下。現在你對這份副本有完整權限。
  2. Clone:把你 fork 出來的副本 clone 到本機。
  3. 開分支改:git switch -c fix-typo,做修改、commit。
  4. Push:把這條分支 push 回你自己的 fork。
  5. 開 Pull Request:在 GitHub 上發起一個 PR,等於正式跟原專案的維護者說:「我在我的分支上做了這些改動,請你拉(pull)過去看看,願不願意合併進你的專案?」
  6. Code Review:維護者會在 PR 頁面上逐行審查程式碼(code review)、留言、要求修改。你只要在同一條分支上繼續 commit、push,PR 會自動更新。
  7. Merge:雙方滿意後,維護者按下「Merge」,你的貢獻就正式進入原專案了。

Feature Branch Workflow:團隊內部的日常

如果你是團隊的一員、對主 repo 就有寫入權限,通常用更直接的功能分支工作流(feature branch workflow):

  1. 絕不直接在 main 上開發。main 永遠保持「隨時可上線」的乾淨狀態。
  2. 每個新功能/修正,都從最新的 main 開一條描述性命名的分支,例如 feat/search-bar、fix/login-redirect。
  3. 在分支上開發、commit、push。
  4. 開 PR,請隊友 review,跑自動化測試(CI)。
  5. 通過後合併回 main,刪掉分支。

這套流程的精髓在於:每個改動進入主線前,都至少被另一雙眼睛看過、被測試驗證過。 這也是為什麼許多團隊會在 GitHub 上設定「保護分支(protected branch)」,規定 main 只能透過通過審查的 PR 合併,不准任何人直接 push。

救命錦囊:當事情出錯時

最後這節是你的「急救包」。先記住一句話安心:只要你曾經 commit 過,東西幾乎不可能真的不見。Git 對已提交的資料保護得非常好。下面是幾個關鍵時刻能救你一命的工具。

git stash:臨時把工作收進口袋

情境:你正改到一半(還沒 commit),老闆突然說「線上有 bug,馬上修!」你需要切到別的分支,但又不想把這團半成品 commit 進去。git stash 幫你把目前的改動暫時收起來,讓工作目錄回到乾淨狀態: 

git stash            # 把目前未提交的改動收進「暫存堆疊」,工作目錄變乾淨
# ... 切去別的分支救火、commit、切回來 ...
git stash list       # 看看收了哪些(可以收很多份)
git stash pop        # 把最近收起來的那份拿回來,並從堆疊移除
git stash apply      # 拿回來但「不」移除(想套用到多個分支時好用)

git reflog:時光機的時光機

這是整章最該記住的救命指令。reflog 記錄了「你的 HEAD 曾經去過哪些地方」——每一次 commit、切換分支、merge、reset,都被偷偷記了一筆,即使是那些已經「不在任何分支上」、看似消失的 commit。

所以,當你手滑 git reset --hard 把幾個 commit「弄丟」時——別慌: 

git reflog
c91d8af (HEAD -> main) HEAD@{0}: reset: moving to HEAD~2
f02ab57 HEAD@{1}: commit: Add basic stylesheet
7b1e0d4 HEAD@{2}: commit: Add home page
a3f9c2e HEAD@{3}: commit: Add project README

看到沒?那個被你 reset 掉的 f02ab57 還好端端地記在 reflog 裡。只要把指標搬回去就救回來了: 

git reset --hard f02ab57     # 回到那個 commit(用 hash)
# 或者:
git reset --hard HEAD@{1}    # 用 reflog 的編號指名
🔑
reflog 是你的安全網

幾乎任何「我把 commit 弄丟了!」的恐慌,答案都是 git reflog。它證明了那句話:只要 commit 過,東西就還在某個角落,只是暫時沒有分支指向它而已。 (唯一的例外是從沒 commit 過的工作目錄改動——那種被 reset --hard 沖掉的,reflog 也救不回。所以才說:常 commit,是你最好的保險。)

git cherry-pick:單獨摘一個 commit 過來

有時你只想要某一條分支上的某一個 commit,而不是整條合併過來。例如某個 bug 修正在 feature 分支上,但你想立刻把它套到 main。cherry-pick(字面意思:精挑細選)就是「摘」那一個 commit 重做到目前分支: 

git switch main
git cherry-pick f02ab57    # 把 f02ab57 這個 commit 的改動,複製一份到 main

git commit --amend:修改最後一次 commit

剛 commit 完就發現訊息打錯字、或漏 add 了一個檔案?不必再多一個「修正上一個 commit」的 commit,直接修改最後那一次: 

git commit --amend -m "Add home page (正確的訊息)"   # 只改訊息
# 或:先把漏掉的檔案 add 進來,再 amend 把它併入上一個 commit
git add forgotten.css
git commit --amend --no-edit                          # 內容併入,訊息不變
⚠️
amend 也是改寫歷史:別碰已推送的

--amend 不是「編輯」原 commit,而是用一個新 commit 取代它(hash 會變)。所以它跟 rebase 一樣, 只能用在還沒 push 出去的 commit。如果你 amend 了一個已經推送的 commit,再 push 時會被遠端擋下來, 硬推(--force)則會弄亂別人的歷史。原則同前:改寫歷史的操作,只對「還沒分享」的 commit 動手。

好習慣:讓 Git 替你加分而不是扣分

工具學會了,最後談「品味」。下面這幾條習慣,是區分「會用 Git」和「用得好」的關鍵:

  • 小而頻繁地 commit:每完成一個有意義的小步驟就 commit,而不是攢一整天才提交一坨。小 commit 容易理解、容易回溯、出事容易定位。它也是你的安全網(還記得嗎?沒 commit 的東西救不回)。
  • 每個 commit 只做一件事:善用暫存區挑選改動,別把不相干的修改混在一起。
  • 寫有意義的訊息:祈使句、講「為什麼」。未來的你會感激現在的你。
  • 分支命名要有描述性:feat/user-profile、fix/payment-timeout,而不是 test、新分支、aaa。
  • main 永遠保持可運作:實驗都在分支上做,別把壞掉的東西推上主線。
  • 絕不提交祕密與大檔:密碼、API 金鑰、憑證一旦進了歷史就很難根除;node_modules 之類能重建的大東西也別入庫。善用 .gitignore。這點極其重要,第七部「資訊安全」會告訴你外洩一把金鑰的代價有多慘。
  • push 前先 pull:養成「同步再上傳」的節奏,減少衝突。

練習

✏️ 練習 18.2 實作

建立一條分支做修改,再合併回主線。要求:在 main 上先有一個含 app.txt 的 commit;然後開一條 feature 分支,在上面對 app.txt 新增一行並 commit;最後切回 main 把 feature 合併進來,並刪除該分支。用 git log --oneline --graph 觀察結果。

看解答 
git init merge-demo && cd merge-demo
echo "line 1" > app.txt
git add app.txt && git commit -m "Add app.txt"

git switch -c feature
echo "line 2 (from feature)" >> app.txt
git add app.txt && git commit -m "Add line 2 on feature"

git switch main
git merge feature          # main 沒有新進度 → 這會是一次 fast-forward
git branch -d feature      # 用完刪掉分支

git log --oneline --graph

因為 main 從分岔後沒有任何新 commit,這次是快轉合併:不會產生合併提交,main 直接滑到 feature 的頂端。想看到真正的「三方合併 + 合併提交」,可以在合併前先到 main 上額外做一個 commit,製造出兩邊都有進度的情況。

✏️ 練習 18.3 挑戰

故意製造、並解決一個合併衝突。讓 main 和一條分支改到同一行,合併時引發衝突,然後手動解決它、完成合併。

看解答 
git init conflict-demo && cd conflict-demo
echo "title: 首頁" > page.txt
git add page.txt && git commit -m "Add page title"

# 在分支上改第一行
git switch -c rename-title
echo "title: 主頁" > page.txt
git add page.txt && git commit -m "Rename title to 主頁"

# 回 main,也改同一行(製造分歧)
git switch main
echo "title: 歡迎頁" > page.txt
git add page.txt && git commit -m "Rename title to 歡迎頁"

# 合併 → 衝突!
git merge rename-title

此時 page.txt 裡會出現衝突標記:

<<<<<<< HEAD
title: 歡迎頁
=======
title: 主頁
>>>>>>> rename-title

用編輯器把它改成你要的最終版(並刪掉三行標記),例如決定就叫「歡迎頁」,於是檔案只留 title: 歡迎頁。然後:

git add page.txt      # 告訴 Git:這個衝突我解決了
git commit            # 完成合併(用預設訊息即可)
git log --oneline --graph

若中途想反悔,git merge --abort 會把一切回復到合併前。

✏️ 練習 18.4 實作

你做了一個錯誤的 commit。請分別示範用 reset 與 revert 復原它,並用一句話說明:什麼時候該用哪一個?

看解答

情況一:這個爛 commit 還沒 push(只在本機)→ 用 reset。

echo "oops 寫錯了" > bug.txt
git add bug.txt && git commit -m "Bad commit"

# 撤銷這個 commit,但把改動留在工作目錄讓你重改(預設 --mixed)
git reset HEAD~1
# 若連改動都不要了(確定沒有想保留的東西):
git reset --hard HEAD~1

情況二:這個爛 commit 已經 push、別人也有了 → 用 revert。

# 產生一個「反向 commit」抵銷它,歷史完整保留
git revert HEAD
git push        # 安全地把撤銷也分享出去

一句話:歷史還是我私有的,用 reset(直接抹掉);歷史已經公開分享,用 revert(新增反向 commit,不改寫過去)。

✏️ 練習 18.5 挑戰

進階急救:用 git reset --hard 故意「弄丟」兩個 commit,然後用 reflog 把它們完整救回來。

看解答 
git init rescue-demo && cd rescue-demo
echo a > f.txt && git add . && git commit -m "C1"
echo b >> f.txt && git add . && git commit -m "C2 (重要!)"
echo c >> f.txt && git add . && git commit -m "C3 (也很重要!)"

# 災難:手滑把 C2、C3 砍了
git reset --hard HEAD~2
git log --oneline          # 只剩 C1,C2/C3 看似消失……

# 救援:看 HEAD 去過哪
git reflog

reflog 會列出類似:

a1b2c3d (HEAD -> main) HEAD@{0}: reset: moving to HEAD~2
9f8e7d6 HEAD@{1}: commit: C3 (也很重要!)
4c5b6a7 HEAD@{2}: commit: C2 (重要!)
a1b2c3d HEAD@{3}: commit: C1
# 把指標搬回 C3,兩個 commit 完整復活
git reset --hard 9f8e7d6      # 或 git reset --hard HEAD@{1}
git log --oneline             # C1、C2、C3 都回來了 ✅

這就是為什麼老手對 reset --hard 不太害怕——只要 commit 過,reflog 幾乎都救得回。但別忘了那個唯一的例外:從沒 commit 過的工作目錄改動,被 --hard 沖掉就真的沒了。

✏️ 練習 18.6 思考

不用打指令,用你自己的話,完整描述一遍「替一個開源專案修錯字」的 Fork → PR 流程。要能講清楚:為什麼第一步要 fork?PR 到底是在「請求」什麼?code review 發生在哪個環節?

看解答

參考敘述:

  1. 因為我對原專案沒有寫入權限,不能直接 push,所以先 fork——在 GitHub 上把它複製一份到我自己的帳號下,這份副本我有完整控制權。
  2. 把我的 fork clone 到本機,開一條描述性分支(如 fix/typo-in-readme),改掉錯字、commit。
  3. 把這條分支 push 回我自己的 fork。
  4. 在 GitHub 上對原專案發起一個 Pull Request。PR 的本質,是正式請求原專案維護者:「請拉(pull)我這條分支的改動過去,考慮合併進你的專案。」它附上完整的 diff 與說明。
  5. Code review 發生在 PR 頁面上:維護者(或其他貢獻者)逐行看我的改動、留言提問、要求修正。我只要在同一條分支繼續 commit、push,PR 會自動帶入新的提交。也常會自動跑 CI 測試。
  6. 雙方滿意、測試通過後,維護者按下 Merge,我的修正正式進入原專案。任務完成。

關鍵心法:fork 解決「我沒權限」,branch 隔離「我的改動」,PR 把「合併」變成一個可討論、可審查、有禮貌的請求。這正是開源協作能夠規模化、又不失品質的祕密。

把這一章串起來

我們從 真的final.docx 的惡夢出發,抵達了一套讓全世界軟體得以協作的精密工具。回顧這趟旅程最重要的幾個路標:

  • 版本控制給你時光機、協作、實驗分支、責任追蹤——徹底取代「複製一個新資料夾」的土法煉鋼。
  • Git 的心智模型是一切的鑰匙:它存的是快照不是差異;有工作目錄、暫存區、儲存庫三個區域;commit 串成一張指向父節點的 DAG。指令都只是在這張圖上搬指標、加節點。
  • 日常循環:status 看狀況 → add 進暫存 → commit 拍快照 → diff/log 檢視。寫好訊息是給未來的禮物。
  • 復原三劍客:restore(丟未提交的改動)、reset(改寫私有歷史,--hard 要當心)、revert(安全撤銷公開歷史)。
  • 分支只是一個指標,所以開分支幾乎免費。merge 保留分岔的真實歷史;rebase 換來乾淨直線,但絕不能 rebase 已推送的公開歷史。
  • 遠端協作靠 clone/fetch/pull/push 與 origin;GitHub 的 Fork + PR 與功能分支工作流,讓「每個改動都被審查」成為可能。
  • 出事別慌:stash 收拾半成品、reflog 撈回弄丟的 commit、cherry-pick 單摘一筆、--amend 修補上一次提交。只要 commit 過,東西幾乎都還在。

Git 是那種「用一輩子、卻永遠有新東西可學」的工具。但你現在已經掌握了它的骨架與靈魂—— 只要心裡能畫出那張三區圖與 DAG,任何沒見過的指令,你都能推理出它在做什麼。接下來最好的學法,就是真的拿一個專案來用它:今天就 git init,開始替你的下一段程式拍快照吧。
