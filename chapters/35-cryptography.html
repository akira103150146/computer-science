一封在大庭廣眾下傳遞的情書

想像 Alice 要寄一張紙條給遠方的 Bob,但這張紙條必須經過一整條街的人手手相傳才會到——每個經手的人,都可能偷看、 甚至偷改內容。這聽起來很荒謬,但它正是網際網路的真實樣貌。

還記得第 25 章嗎?我們親眼看到,你的資料被切成一個個封包(packet),一跳一跳穿越幾十台陌生的路由器才抵達對方。 路上的任何一台機器、同一個 Wi-Fi 上的任何一個人,原則上都能把你的封包攔下來看。第 25 章我們賣了個關子, 說 HTTPS 那把「鎖」留到後面揭曉——這一章,就是揭曉的時刻。

在第 34 章,我們認識了資訊安全的三大支柱 CIA:機密性(Confidentiality)、完整性(Integrity)、可用性(Availability)。 密碼學(cryptography)正是達成前兩者、外加一個「真實性(authenticity)」的核心工具。它要回答一個看似不可能的問題:

在一條「每個字都會被陌生人看到、甚至被竄改」的通道上,我和你要怎麼祕密、可靠地溝通?

密碼學要對付的「壞人」,傳統上有兩種,搞清楚它們的差別很重要:

  • 被動攻擊者(Eve,Eavesdropper):只偷看,不動手。她想破壞的是機密性。
  • 主動攻擊者(Mallory,Malicious):不只偷看,還會竄改、偽造、重送訊息。她想破壞的是完整性與真實性。

一個只防得了 Eve、卻防不了 Mallory 的系統,在真實世界往往不堪一擊。記住這點,我們稍後就會看到: 為什麼「光是加密」常常不夠,還需要雜湊、簽章與憑證來補上完整性與真實性。先從密碼學的童年——古典密碼——開始。

古典密碼:聰明,但注定失敗

凱撒密碼:把字母往後推幾格

兩千年前,凱撒(Julius Caesar)寫軍事密令時,用了一個簡單的把戲:把每個字母往後推固定的格數。 例如往後推 3 格,A 變 D、B 變 E、… X 變 A(繞回來)。「HELLO」就變成「KHOOR」。收信的人只要往前推 3 格, 就能還原。這就是凱撒密碼(Caesar cipher),大概是史上最有名的密碼。

這裡出現兩個貫穿全章的詞:明文(plaintext)是原始訊息,密文(ciphertext)是加密後的亂碼; 而那個「往後推幾格」的數字,就是這套系統的金鑰(key)。我們用 Python 把它寫出來:

def caesar_encrypt(text, shift):
    result = ""
    for ch in text:
        if ch.isupper():                                  # 大寫字母
            result += chr((ord(ch) - ord('A') + shift) % 26 + ord('A'))
        elif ch.islower():                                # 小寫字母
            result += chr((ord(ch) - ord('a') + shift) % 26 + ord('a'))
        else:
            result += ch                                  # 非字母(空白、標點)原樣保留
    return result

def caesar_decrypt(text, shift):
    return caesar_encrypt(text, -shift)                   # 解密 = 反方向位移

print(caesar_encrypt("Hello, World!", 3))   # Khoor, Zruog!
print(caesar_decrypt("Khoor, Zruog!", 3))   # Hello, World!

注意那個 % 26——又是模運算(modular arithmetic)!還記得第 15 章的「時鐘算術」嗎? 字母表就是一個 26 格的時鐘,推過 Z 就繞回 A。密碼學從第一天起,就和模運算綁在一起。

它有多脆弱?暴力破解只要試 25 次

凱撒密碼的致命傷顯而易見:金鑰只有 25 種可能(推 1 到 25 格;推 0 格等於沒加密)。攻擊者根本不用聰明, 把 25 種全試一遍,哪個結果讀得通就是它。這叫暴力破解(brute-force attack): 

def caesar_bruteforce(cipher):
    for shift in range(1, 26):
        print(f"位移 {shift:2}: {caesar_decrypt(cipher, shift)}")

caesar_bruteforce("Khoor, Zruog!")
# 其中「位移  3」那一行會跳出可讀的 "Hello, World!" —— 破解完成

一個密碼系統,如果它的金鑰空間(key space,所有可能金鑰的數量)小到能被電腦在合理時間內全部試完,它就形同虛設。 現代密碼的金鑰動輒有 $2^{128}$ 種可能——就算每秒試一兆把,把宇宙年齡乘上一兆倍也試不完。夠大的金鑰空間,是安全的最低門檻。

替換密碼與頻率分析:聰明也救不了你

「金鑰太少」很容易補救:乾脆讓每個字母「隨機」對應到另一個字母,而不是整齊地推固定格數。這叫 替換密碼(substitution cipher)。它的金鑰是一張 26 個字母的對照表,可能的排列數高達 $26! \approx 4 \times 10^{26}$——暴力破解徹底沒戲。看起來無懈可擊?

九世紀的阿拉伯學者金迪(Al-Kindi)看穿了它的破綻:每個字母換了張臉,但它出現的「頻率」沒變。 英文裡 E 最常出現(約 12.7%),其次是 T、A、O……。所以只要密文夠長,數一數哪個符號最常出現,它八成就是 E; 次常出現的就是 T……。再配合「字母組合習慣」(例如 TH、單字母字通常是 a 或 I),整張對照表就能一塊塊拼回來。 這就是頻率分析(frequency analysis),人類史上第一個系統性的密碼破解術。 

from collections import Counter

cipher = "WKLV LV D VHFUHW PHVVDJH BRX FDQQRW UHDG"   # 一段密文
freq = Counter(c for c in cipher.lower() if c.isalpha())
print(freq.most_common(5))
# 找出最常出現的密文字母,對照英文字母頻率表(E、T、A、O…)去猜對應關係
# (這段密文其實就是凱撒位移 3;最常出現的 'h' 對到明文 'e')
🔑
Kerckhoffs 原則:安全只能靠金鑰,不能靠保密演算法

古典密碼最深的教訓,被 19 世紀的 Kerckhoffs 寫成了一條鐵律: 一個密碼系統,即使「演算法本身完全公開」,只要金鑰保密,就應該仍然安全。 換句話說,你不能把安全寄望在「沒人知道我怎麼加密的」上面——因為演算法總會外洩(離職員工、逆向工程、論文發表)。 這也叫「避免靠隱晦來安全(security through obscurity)」。今天所有正派的密碼演算法(AES、RSA、SHA-256) 都是完全公開、全世界公開檢驗的;它們的安全,純粹建立在「你不知道那把金鑰」之上。 一個不敢公開的密碼演算法,通常是因為它經不起公開。

✏️ 練習 35.1 實作

實作凱撒密碼的 caesar_encrypt(text, shift) 與 caesar_decrypt(text, shift), 要能正確處理大小寫並保留非字母字元。接著寫一個 caesar_bruteforce(cipher),把 25 種可能解全部印出來, 示範金鑰空間太小的後果。最後想一想:如果攻擊者「不想用肉眼」逐行看,有沒有辦法讓程式自動挑出最可能的那一行?

def caesar_encrypt(text, shift):
    out = ""
    for ch in text:
        if ch.isupper():
            out += chr((ord(ch) - 65 + shift) % 26 + 65)
        elif ch.islower():
            out += chr((ord(ch) - 97 + shift) % 26 + 97)
        else:
            out += ch
    return out

def caesar_decrypt(text, shift):
    return caesar_encrypt(text, -shift)

def caesar_bruteforce(cipher):
    for shift in range(1, 26):
        print(f"位移 {shift:2}: {caesar_decrypt(cipher, shift)}")

# 自動化:用「字母頻率」幫每個候選打分,挑分數最高的
ENGLISH_FREQ = {  # 各字母在英文中的大約出現比例(%)
    'e':12.7,'t':9.1,'a':8.2,'o':7.5,'i':7.0,'n':6.7,'s':6.3,'h':6.1,
    'r':6.0,'d':4.3,'l':4.0,'u':2.8,'c':2.8,'m':2.4,'w':2.4,'f':2.2}

def score(text):
    return sum(ENGLISH_FREQ.get(c, 0) for c in text.lower())

def best_guess(cipher):
    candidates = [(score(caesar_decrypt(cipher, s)), s) for s in range(26)]
    _, shift = max(candidates)            # 分數最高的那個位移最可能正確
    return shift, caesar_decrypt(cipher, shift)

print(best_guess("Khoor, Zruog!"))   # (3, 'Hello, World!')

關鍵洞見:破解不需要「試到對」的運氣,而是把人類的直覺(「哪一行讀起來像英文」) 變成一個可計算的評分函數。這正是現代密碼分析的精神——把破解變成最佳化問題。

對稱式加密:同一把鑰匙鎖門開門

古典密碼雖然失敗,但它示範了現代密碼學的第一大分支:對稱式加密(symmetric encryption)。 「對稱」的意思是——加密和解密用的是同一把金鑰。就像家門的鑰匙:同一把,鎖門開門都靠它。 凱撒密碼那個「位移量」就是把對稱金鑰,只是它弱得可笑。我們需要更強的。

XOR:對稱加密的最小原子

現代對稱加密的最底層,藏著一個你在第 8 章學過的位元運算:互斥或(XOR,記作 ^)。 XOR 有個美妙的性質:同一個數連續 XOR 兩次,會變回自己。用式子說就是 $a \oplus k \oplus k = a$。 這簡直是為加密量身打造的:用金鑰 $k$ 把明文 XOR 一次得到密文,收信方再 XOR 同一個 $k$,就還原成明文。 

def xor_bytes(data, key):
    # 把 data 的每個位元組,與金鑰循環對應的位元組做 XOR
    return bytes(b ^ key[i % len(key)] for i, b in enumerate(data))

key     = b"SECRET-KEY"
message = b"attack at dawn"

cipher = xor_bytes(message, key)
print(cipher.hex())               # 一串看不懂的十六進位亂碼

recovered = xor_bytes(cipher, key)   # 再 XOR 同一把金鑰
print(recovered)                      # b'attack at dawn' —— 完美還原
⚠️
上面這段「XOR 加密」其實很不安全

當金鑰比訊息短而被「循環重複」使用時,它就退化成一種多表替換密碼,可以用統計方法破解 (這正是攻破二戰恩尼格瑪、以及許多自製「加密」的途徑)。千萬別把這段程式當成真的加密工具—— 它只是用來理解 XOR 這顆原子。那麼,什麼時候 XOR 加密真的無法被破解?答案很驚人——

一次性密本:理論上完美,實務上沒用

如果金鑰滿足三個嚴格條件:(1) 真正隨機、(2) 和訊息一樣長、(3) 只用一次、用完即丟, 那麼這個 XOR 加密就升級成一次性密本(One-Time Pad, OTP)。而 OTP 被數學嚴格證明擁有 完美保密性(perfect secrecy):攻擊者就算有無限的算力,看著密文也完全得不到關於明文的任何資訊—— 因為任何一段明文,都存在一把金鑰能把它加密成這段密文,所以密文等於什麼都沒透露。這是 Claude Shannon 在 1949 年證明的。

既然完美,為什麼不全用它?因為那三個條件在現實裡幾乎做不到:

  • 金鑰要和訊息一樣長:你要傳 1GB 的影片,就得先安全地送一把 1GB 的金鑰過去——可是「安全地送金鑰」正是我們一開始要解決的問題,雞生蛋蛋生雞。
  • 金鑰只能用一次:重複用同一把(「two-time pad」)會立刻洩密,把兩段密文 XOR 起來,金鑰就消掉了,露出明文的關聯。
  • 必須真正隨機:用程式的偽隨機數會留下可預測的規律,完美保密蕩然無存。

OTP 告訴我們:完美保密存在,但代價高到只用在極端場合(如冷戰時期的紅色電話)。實務上,我們要的是「夠安全又夠實用」的折衷——這就是 AES 登場的舞台。

AES:現代對稱加密的標準

AES(Advanced Encryption Standard,進階加密標準)是今天全世界的對稱加密主力—— 你的 HTTPS 連線、加密硬碟、WhatsApp 訊息、Wi-Fi(WPA),底層幾乎都靠它。它在 2001 年被美國標準局選定, 經過二十多年全世界密碼學家的猛攻,至今沒有實用的破解法。你不需要懂它的內部細節(那是一連串精心設計的 位元混合與替換),只需要記住幾個關鍵事實:

  • 它是區塊密碼(block cipher):一次處理固定大小的一「塊」資料,AES 的區塊是 128 位元(16 位元組)。
  • 金鑰長度可選 128、192 或 256 位元。AES-256 是目前的高安全標準,連量子電腦在可見的未來也撼動不了它。
  • 它是公開的、被全世界檢驗過的——完全符合 Kerckhoffs 原則。

和區塊密碼相對的是串流密碼(stream cipher)(如 ChaCha20):它把金鑰展開成一條「金鑰流」, 像 OTP 那樣逐位元和明文 XOR,適合長度不定的資料流。兩者各有舞台,但概念上,區塊密碼是現代主流。

加密模式:那隻洩密的企鵝

這裡有個關鍵又常被忽略的坑。AES 一次只能加密 16 位元組,但你的檔案有幾百萬位元組。怎麼把一整個檔案餵進去? 最天真的做法叫 ECB(Electronic Codebook):把資料切成一塊塊,每塊各自獨立加密。聽起來合理, 但它有個致命缺陷——相同的明文區塊,會產生完全相同的密文區塊。

這意味著什麼?如果原圖有一大片同色的背景(很多相同的區塊),加密後那片區域也會是「同一種亂碼」—— 圖案的輪廓被原封不動地保留了下來。密碼學界有一張著名的示範圖:把 Linux 吉祥物企鵝 Tux 用 ECB 模式加密,結果你還是一眼就能認出那是一隻企鵝。

修正之道,是讓每個區塊的加密「彼此牽連」或「混入變化」,常見的安全模式有:

  • CBC(Cipher Block Chaining):每塊明文先和「前一塊的密文」XOR 再加密,於是相同明文也會產生不同密文。
  • CTR(Counter):把一個不斷遞增的計數器加密成金鑰流,再和明文 XOR——把區塊密碼變成串流密碼,可平行運算。
  • GCM(Galois/Counter Mode):在 CTR 之上再加一個「認證標籤」,同時保證機密性與完整性——這是今天 TLS 的首選。
🔑
加密 ≠ 完整性

ECB 企鵝給我們一個更深的教訓:「有加密」不等於「安全」。同樣的演算法(AES), 用錯模式(ECB)就洩密。而且光加密通常只擋得住 Eve(偷看),擋不住 Mallory(竄改)——攻擊者就算讀不懂密文, 也可能盲目地翻轉幾個位元去破壞它。所以現代做法是用 AEAD(帶認證的加密,如 AES-GCM): 一次同時提供機密性與完整性。這也是為什麼「自己拼裝加密」這麼危險——每個細節都是陷阱。

對稱加密的死穴:金鑰怎麼送過去?

現在對稱加密看起來很美:AES 又快又強。但它有一個從凱撒時代就存在、始終繞不開的根本問題—— 金鑰分配(key distribution)。加密和解密用同一把金鑰,所以 Alice 和 Bob 必須事先 共享這把金鑰。可是他們之間只有一條不安全的通道(那條 Eve 正在偷聽的線)!

你想用加密來保護通道,但你得先安全地把金鑰送過通道——這不就是你一開始要解決的問題嗎?

對只有兩個人來說,也許還能「見面時偷偷講好」。但網際網路上,你要和從沒見過面的淘寶、銀行、Google 建立安全連線,根本沒機會「先見面交換金鑰」。幾千年來,這道牆擋住了所有人。直到 1970 年代,幾位數學家想出一個近乎魔法的點子, 徹底繞過了它——這就是密碼學史上最美的突破,也是這一章的重頭戲:公開金鑰。

用 XOR 實作一個對稱加解密函式 xor_cipher(data, key)(加密與解密是同一個函式,為什麼?)。 接著請說明並用程式示範:當同一把金鑰被用來加密「兩段」訊息時,攻擊者把兩段密文 XOR 起來會發生什麼事? 這就是為什麼一次性密本的金鑰「只能用一次」。

看解答 
def xor_cipher(data, key):
    return bytes(b ^ key[i % len(key)] for i, b in enumerate(data))
# 加解密同一個函式,因為 (a ^ k) ^ k == a:XOR 自己是自己的反運算

key = b"reused-key-reused-key"     # 故意重複使用同一把金鑰(錯誤示範!)
m1 = b"HELLO ALICE THIS IS BOB"
m2 = b"HELLO BOB THIS IS ALICE"

c1 = xor_cipher(m1, key)
c2 = xor_cipher(m2, key)

# 攻擊者就算不知道 key,把兩段密文 XOR 起來,金鑰就「消掉」了:
#   c1 ^ c2 = (m1 ^ k) ^ (m2 ^ k) = m1 ^ m2
leak = bytes(a ^ b for a, b in zip(c1, c2))
print(leak)
# 結果只和 m1、m2 有關,完全和金鑰無關!
# 兩段明文相同的位置會 XOR 成 0(b'\x00'),洩漏出明文的結構。
# 在相同位置:H^H=0, E^E=0, L^L=0, L^L=0, O^O=0, ' '^' '=0 ...
print([i for i, x in enumerate(leak) if x == 0])  # 找出兩段明文相同的位置

這就是「two-time pad」災難:重複使用金鑰,等於把 m1 ^ m2 直接交給攻擊者, 再配合語言統計,兩段明文都能被還原。一次性密本的「一次性」三個字,是用血換來的鐵律。

密碼學雜湊:資料的單向指紋

在跳進公開金鑰之前,我們需要另一塊基石。先問一個問題:你下載一個 5GB 的安裝檔,怎麼確認它在傳輸途中 「一個位元都沒被改過」(完整性)?難道要把原檔再下載一次來逐位元比對?有個聰明得多的工具:密碼學雜湊函數 (cryptographic hash function)。

它把任意長度的資料,壓縮成一個固定長度的「指紋」(例如 SHA-256 永遠輸出 256 位元 / 64 個十六進位字元)。 你在第 13 章的雜湊表用過「雜湊」,但密碼學雜湊要求嚴格得多,它必須同時滿足:

  • 單向性(one-way / pre-image resistance):從輸入算出指紋很容易,但從指紋反推回輸入,在計算上不可行。
  • 抗碰撞性(collision resistance):極難找到「兩個不同的輸入,卻有相同指紋」的情況。
  • 雪崩效應(avalanche effect):輸入只改動一個位元,輸出也會面目全非(大約一半的位元翻轉)。

用 Python 內建的 hashlib 親眼看看雪崩效應:

import hashlib

def sha256(text):
    return hashlib.sha256(text.encode()).hexdigest()

print(sha256("hello"))
# 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824
print(sha256("hellp"))   # 只把最後一個字母 o 改成 p
# 977a8d9d2e1f... 完全不同!輸出看不出任何「只差一個字母」的痕跡

# 應用:驗證檔案完整性(下載網站會公布官方的 SHA-256 指紋)
def file_sha256(path):
    h = hashlib.sha256()
    with open(path, "rb") as f:
        for chunk in iter(lambda: f.read(8192), b""):
            h.update(chunk)            # 分塊餵入,不必一次讀進整個大檔
    return h.hexdigest()

雜湊的用途無所不在:完整性校驗(下載檔、Git 的每個 commit 其實就是內容的 SHA 指紋)、 數位簽章(待會就會用到:簽的不是整份文件,而是它的指紋)、 以及一個你天天間接用到、卻最常被做錯的場景——密碼儲存。

密碼儲存:為什麼絕不能存明文,甚至不能只存雜湊

網站該怎麼保存你的密碼?第一條鐵律:絕對不能存明文。資料庫一旦外洩,所有人的密碼當場全裸。 那存「密碼的雜湊」呢?登入時把你輸入的密碼算一次雜湊,和資料庫裡的比對就好,資料庫永遠不知道原始密碼——聽起來很安全?

可惜還不夠。有兩個攻擊在等著:

  • 彩虹表(rainbow table)攻擊:攻擊者事先把「幾億個常見密碼 → 它們的雜湊」算好存成一張巨表。 拿到你外洩的雜湊,他直接查表,瞬間反查出原始密碼。單向性在這裡幫不了你,因為他根本不是「反推」,而是「預先算好正著查」。
  • 速度:SHA-256 設計得很快(每秒幾十億次),這對校驗檔案是優點,對存密碼卻是災難——它讓攻擊者每秒能暴力猜上億個密碼。

正確做法有兩招,缺一不可:

  • 加鹽(salt):給每個使用者配一段隨機的「鹽」,把它和密碼拼在一起再雜湊,鹽則和雜湊一起存。 這樣同樣的密碼,每個人的雜湊都不同,彩虹表整個失效(攻擊者得為每一份鹽重算一張表)。
  • 慢雜湊(slow hashing):刻意用故意很慢、又吃記憶體的演算法,如 bcrypt、scrypt、Argon2 (或退而求其次的 PBKDF2)。它們把單次雜湊拖慢到幾十毫秒——你登入時無感,但攻擊者暴力破解的速度被砍掉好幾個數量級。
⛔
永遠不要用 MD5 或 SHA-1 來存密碼(其實也別拿來做任何安全用途)

MD5 與 SHA-1 都已經被攻破:研究者能實際製造出「碰撞」(兩份不同檔案、相同指紋), Google 在 2017 年就示範了 SHA-1 的碰撞(SHAttered)。更別說它們又快又無鹽,拿來存密碼等於把使用者送給駭客。 請用 bcrypt / scrypt / Argon2 存密碼;需要一般雜湊時用 SHA-256 或 SHA-3。 看到任何系統還在用 MD5「加密」密碼(連用詞都錯了,雜湊不是加密),那是嚴重的紅旗。

✏️ 練習 35.3 實作

用 hashlib 寫一個 sha256(text),然後設計一個小實驗量化雪崩效應: 對「hello」和只差一個字母的「hello!」各算 SHA-256,把兩個指紋轉成二進位,數數看總共 256 個位元裡有幾個不一樣。 猜猜看會接近多少?(提示:理想的雜湊,改動任何輸入都應讓「大約一半」的輸出位元翻轉。)

看解答 
import hashlib

def sha256_bits(text):
    digest = hashlib.sha256(text.encode()).digest()      # 32 位元組 = 256 位元
    return "".join(f"{byte:08b}" for byte in digest)     # 轉成 256 個 '0'/'1'

a = sha256_bits("hello")
b = sha256_bits("hello!")     # 只多了一個驚嘆號

# 漢明距離:兩個位元字串中不同位元的數量
diff = sum(x != y for x, y in zip(a, b))
print(f"256 位元中有 {diff} 個不同 (約 {diff/256:.0%})")
# 典型結果:約 128 個位元不同,也就是大約一半 —— 這就是雪崩效應

結果通常落在 128 上下,接近 50%。這代表輸出和輸入之間「沒有任何可見的關聯」—— 你無法從「指紋只變了一點點」推測「輸入只變了一點點」,因為事實正好相反。 這個性質正是雜湊能當「指紋」的根本原因。

公開金鑰:密碼學最美的魔法

現在回到那道千年高牆:Alice 和 Bob 素未謀面,只有一條 Eve 正在偷聽的通道,他們要怎麼開始祕密通訊? 1976 年,Diffie、Hellman(以及稍後的 RSA 三人組)提出一個顛覆性的想法——用兩把不同的金鑰。

🔑
郵筒比喻:公鑰加密、私鑰解密

想像一個有投信口的郵筒:任何人都能把信投進去(投信口是公開的),但只有持鑰匙的人能打開取信。 公開金鑰加密就是這樣運作:每個人有一對金鑰——公鑰(public key)像投信口,可以印在名片上、貼上網路、廣發給全世界; 私鑰(private key)像那把開鎖鑰匙,只有你自己保管。任何人用你的公鑰加密的訊息,只有你的私鑰能解開。 於是 Alice 要寄密信給 Bob,只要去抓 Bob 公開的公鑰來加密就行——金鑰分配問題,迎刃而解!

RSA:大數分解難,所以你的密文安全

我們在第 15 章已經親手拼過一個玩具 RSA,這裡快速複習它的靈魂。RSA 的安全性押注在一個「單向」的數學事實上: 把兩個大質數相乘很容易,但把乘積分解回那兩個質數,極其困難。 $61 \times 53 = 3233$ 你幾秒算得出來; 但給你一個 600 位數,要找出它是哪兩個質數的乘積,目前全世界最強的電腦也得算上比宇宙年齡還久。這個「易於正向、難於反向」的不對稱, 就是公開金鑰的命根子。

RSA 的步驟,全建立在第 15 章的工具上(模冪 pow(a, b, m)、擴展歐幾里得求模反元素):

def ext_gcd(a, b):
    if b == 0:
        return a, 1, 0
    g, x1, y1 = ext_gcd(b, a % b)
    return g, y1, x1 - (a // b) * y1

def mod_inverse(e, phi):
    g, x, _ = ext_gcd(e, phi)        # 求 d 使得 e·d ≡ 1 (mod phi)
    return x % phi

# 1. 造鑰匙:挑兩個質數(真實世界用上百位數;這裡沿用第 15 章的小例子)
p, q = 61, 53
n   = p * q                  # 3233(公開的模數)
phi = (p - 1) * (q - 1)      # 3120
e   = 17                     # 公鑰指數,需與 phi 互質
d   = mod_inverse(e, phi)    # 2753(私鑰指數)

print("公鑰 (e, n) =", (e, n))    # 人人可拿,用來加密
print("私鑰 (d, n) =", (d, n))    # 只有你有,用來解密

# 2. 加密 / 解密
m = 65                       # 明文(必須 < n)
c = pow(m, e, n)             # 加密:c = m^e mod n
print("密文:", c)            # 2790
print("解回:", pow(c, d, n)) # 65 —— 用私鑰解密,完美還原

攻擊者就算手握公鑰 $(e, n)$ 和密文 $c$,只要算不出私鑰 $d$ 就一籌莫展;而要算 $d$,他必須先把 $n$ 分解回 $p, q$—— 這正是那道翻不過的牆。「分解大數很難」這個信念,撐起了你每天的網路銀行。 (補一句現實:RSA 因金鑰較長、速度較慢,現代正逐漸被下面要講的橢圓曲線取代,但它仍是理解公開金鑰最好的範例。)

Diffie-Hellman:在大庭廣眾下調出一杯只有兩人知道的顏色

RSA 解決了「加密」,但還有另一條更優雅的路,專門解決「金鑰交換」——Diffie-Hellman 金鑰交換(DH)。 它讓 Alice 和 Bob 透過一條完全公開的通道,各自喊出幾個數字,最後憑空算出一把只有他們兩人知道的共同金鑰, 而旁邊全程偷聽的 Eve 卻算不出來。聽起來像魔術?用調顏料的比喻就懂了。

把「混顏料」換成數學,那個「易混難分」的單向操作就是模冪:算 $g^a \bmod p$ 很容易,但反過來從結果求出指數 $a$ (這叫離散對數問題)極其困難。用第 15 章的 pow 就能跑: 

# 公開參數:一個大質數 p 與生成元 g(這裡用小數字示範)
p, g = 23, 5

a = 6                  # Alice 的私密數(絕不公開)
A = pow(g, a, p)       # Alice 公開送出 A = g^a mod p  = 8

b = 15                 # Bob 的私密數(絕不公開)
B = pow(g, b, p)       # Bob 公開送出   B = g^b mod p  = 19

# 各自用「對方的公開值」再做一次自己的私密次方
alice_key = pow(B, a, p)   # (g^b)^a = g^(ab) mod p
bob_key   = pow(A, b, p)   # (g^a)^b = g^(ab) mod p
print(alice_key, bob_key)  # 2 2 —— 兩人算出同一把金鑰!

# Eve 在通道上看得到 p, g, A=8, B=19,卻無法算出 a 或 b,
# 因此也算不出共同金鑰 g^(ab)。這就是離散對數的「單向牆」。

妙就妙在 $g^{ab} = (g^a)^b = (g^b)^a$——指數相乘的順序無所謂,所以兩人殊途同歸。實務上, Alice 和 Bob 用 DH 在公開通道上「無中生有」協商出一把金鑰後,就拿它當對稱金鑰(餵給 AES), 往後的大量資料就用又快又強的對稱加密來傳。公開金鑰負責「交換鑰匙」,對稱金鑰負責「實際幹活」—— 這個分工,正是 TLS 的核心,馬上就會看到。

ℹ️
橢圓曲線(ECC):更小的金鑰,一樣的安全

今天更主流的做法是把 RSA 與 DH 的數論,搬到橢圓曲線(Elliptic Curve Cryptography, ECC)上。 它基於「橢圓曲線上的離散對數」這個更難的問題,於是能用短得多的金鑰達到同等安全 (256 位元的 ECC 約等於 3072 位元的 RSA),速度更快、更省電——所以手機、IoT、現代 TLS 大量採用 ECDH(橢圓曲線版的 Diffie-Hellman)。原理一句話帶過:換了個更難的數學舞台,但「易正向、難反向」的精神完全一樣。

✏️ 練習 35.4 實作

沿用第 15 章的工具,完成一個玩具 RSA:用 p, q = 61, 53、e = 17 造出金鑰, 對訊息 m = 65 做加密與解密,驗證能還原。接著做數位簽章: 注意簽章的方向和加密相反——用私鑰簽、用公鑰驗。請示範並解釋為什麼這樣就能證明「這訊息確實是私鑰持有者發的」。

看解答 
def ext_gcd(a, b):
    if b == 0:
        return a, 1, 0
    g, x1, y1 = ext_gcd(b, a % b)
    return g, y1, x1 - (a // b) * y1

def mod_inverse(e, phi):
    g, x, _ = ext_gcd(e, phi)
    return x % phi

p, q = 61, 53
n   = p * q                  # 3233
phi = (p - 1) * (q - 1)      # 3120
e   = 17
d   = mod_inverse(e, phi)    # 2753

m = 65

# --- 加密 / 解密:公鑰加密,私鑰解密 ---
c = pow(m, e, n)             # 加密
assert pow(c, d, n) == m     # 解密還原 ✔
print("加解密 OK,密文 =", c) # 2790

# --- 數位簽章:私鑰簽,公鑰驗(方向相反!) ---
signature = pow(m, d, n)            # 用「私鑰 d」對訊息簽名
verified  = pow(signature, e, n)    # 任何人用「公鑰 e」驗證
print("簽章 =", signature)
print("驗證得到:", verified, "==原訊息?", verified == m)  # True

# 偽造測試:沒有私鑰 d 的攻擊者,生不出能通過驗證的簽章
fake = 1234
print("偽造簽章通過驗證?", pow(fake, e, n) == m)   # False

為什麼可行?因為 $e$ 和 $d$ 互為模反元素,pow(pow(m, d, n), e, n) == m 恆成立。 只有握有私鑰 $d$ 的人才能產出那個「用公鑰一驗就變回 $m$」的數字。所以驗證通過,就證明了訊息來自私鑰持有者 (真實性),且訊息沒被改過(完整性)。這就是下一節數位簽章的數學核心。 (真實系統會先對訊息算雜湊再簽,而不是直接簽原文——這裡為了對應第 15 章而簡化。)

✏️ 練習 35.5 思考|實作

實作 Diffie-Hellman 金鑰交換:給定公開的 p, g,讓 Alice 與 Bob 各自選一個私密數, 交換公開值後算出共同金鑰,驗證兩邊一致。然後用「自己的話」解釋:Eve 在通道上看到了 p, g, A, B 四個數字, 為什麼她仍然算不出共同金鑰?她缺的到底是什麼?

看解答 
def diffie_hellman(p, g, a, b):
    A = pow(g, a, p)            # Alice 公開值
    B = pow(g, b, p)            # Bob 公開值
    alice_key = pow(B, a, p)    # Alice 算:B^a = g^(ab)
    bob_key   = pow(A, b, p)    # Bob 算:  A^b = g^(ab)
    return A, B, alice_key, bob_key

p, g = 23, 5
A, B, k1, k2 = diffie_hellman(p, g, a=6, b=15)
print("公開傳輸:A =", A, " B =", B)   # A = 8, B = 19
print("共同金鑰:", k1, k2)            # 2 2
assert k1 == k2

Eve 缺的是「指數」。她看得到 $A = g^a \bmod p$,但要從 $A$ 反推出 $a$, 等於要解離散對數問題——在夠大的 $p$ 下,這在計算上不可行。共同金鑰是 $g^{ab}$, 而 Eve 手上只有 $g^a$ 和 $g^b$;沒有 $a$ 或 $b$ 任一個,她無法把 $g^a$ 和 $g^b$「組合」成 $g^{ab}$ (這叫 Diffie-Hellman 難題)。關鍵不對稱:正向算模冪是 $O(\log)$ 的瞬間運算,反向求指數卻是天文數字的工作量。

但請注意:純 DH 只防得了被動的 Eve,防不了主動的 Mallory——她可以分別冒充雙方,和兩邊各做一次 DH, 坐在中間轉譯(中間人攻擊,man-in-the-middle)。要堵住它,雙方必須驗證對方身分, 這就需要數位簽章與憑證——也就是接下來兩節。

數位簽章:用私鑰蓋一個無法偽造的印章

加密保護的是機密性(別人看不到內容)。但很多時候,內容公不公開無所謂,你真正想確認的是: 這真的是「他」發的嗎?有沒有被人在路上改過? 這是真實性與完整性的問題, 而解法是公開金鑰的另一個威力——數位簽章(digital signature)。

它的精妙之處在於:把公開金鑰反過來用。 加密是「公鑰鎖、私鑰開」;簽章則是「私鑰簽、公鑰驗」:

  • 簽名:發訊者先對訊息算一個雜湊(指紋),再用自己的私鑰對這個指紋運算,產生一段「簽章」附在訊息後。
  • 驗證:任何人用發訊者公開的公鑰檢查簽章,若通過,就同時證明了三件事。

一個通過驗證的數位簽章,一次提供了三種保證:

  • 完整性(integrity):訊息只要被改一個位元,雜湊就變,簽章驗證立刻失敗。
  • 真實性(authenticity):只有私鑰持有者簽得出能通過驗證的簽章,所以確實是「他」發的。
  • 不可否認性(non-repudiation):他事後不能抵賴「這不是我簽的」——因為除了他,沒人有那把私鑰。

你在練習 35.4 已經用玩具 RSA 實作過它了。現實中,軟體更新、區塊鏈交易、電子公文、HTTPS 憑證,全靠數位簽章背書。 但這裡藏著一個尚未回答的問題:驗證簽章要用「發訊者的公鑰」——可是你怎麼知道那把公鑰真的屬於他,而不是 Mallory 偷天換日塞給你的假公鑰? 這就把我們帶到最後一塊拼圖。

PKI 與憑證:那把鎖頭背後的信任鏈

公開金鑰解決了「加密」與「簽章」,卻冒出一個新問題:公鑰的真偽。你連上「銀行網站」, 它傳給你一把公鑰說「這是我的」——但你怎麼確定送來的不是冒牌貨的公鑰?如果 Mallory 把她自己的公鑰冒充成銀行的, 你就會用她的公鑰加密、和她建立連線,還以為自己很安全。這就是中間人攻擊的核心。

解法是引入「可信任的第三方」來擔保。整套制度叫公鑰基礎建設(Public Key Infrastructure, PKI), 核心角色是憑證頒發機構(Certificate Authority, CA)——一些全世界公認可信的機構(如 Let's Encrypt、DigiCert)。

  • 銀行向 CA 證明「我真的擁有 bank.com 這個網域」,CA 查證後,用CA 自己的私鑰對「bank.com 的公鑰 + 網域名 + 有效期」打包簽章,做成一張數位憑證(certificate)。
  • 你連上銀行時,它把這張憑證給你。你的瀏覽器/作業系統裡預先內建了一份「受信任 CA 的公鑰清單」(信任根, root of trust)。
  • 瀏覽器用 CA 的公鑰驗證憑證上的簽章:通過,就代表「一個你信任的 CA 擔保:這把公鑰確實屬於 bank.com」。

實務上不是一張憑證,而是一條信任鏈(chain of trust):根 CA 簽中介 CA,中介 CA 再簽網站憑證, 像一條擔保的鎖鏈,一路追溯到你天生信任的根。只要鏈上每一環的簽章都驗得過,你就能信任最末端那把公鑰。

🔒
那把鎖頭到底在保證什麼?

網址列的 🔒 只保證兩件事:(1) 連線是加密的(沿途沒人偷看得到內容), (2) 你連到的網域,身分經過 CA 擔保(你確實連到了 bank.com,而不是冒牌站)。 它不保證這個網站本身是善良的——釣魚網站也能申請到合法憑證、也會顯示鎖頭。 所以「有鎖頭」代表「通道安全」,不代表「對方可信」。這個分寸,很多人一輩子搞錯。

TLS:把一切串成一條安全連線

我們手上的零件齊了:對稱加密(快)、公開金鑰(解決金鑰交換)、雜湊(完整性)、簽章與憑證(身分)。 TLS(Transport Layer Security)就是把它們全部編織在一起的協定,也就是 HTTP 變成 HTTPS 時多披的那層外殼 (第 25 章我們答應要揭曉的那把鎖,就是它)。它最聰明的設計,是一個分工:

TLS 的核心智慧:用慢的(非對稱)去換快的(對稱)

非對稱加密安全又能解決金鑰分配,但慢;對稱加密快,但有金鑰分配難題。 TLS 各取所長:握手階段用「非對稱」安全地協商出一把對稱金鑰,之後的海量資料全用「對稱」加密來傳。 公開金鑰只在開頭辛苦一下下,負責「把鑰匙安全交給對方」;真正的搬運苦力,交給又快又猛的 AES。

ℹ️
完美前向保密(Perfect Forward Secrecy)

現代 TLS 用 ECDHE(每次連線都產生臨時的金鑰交換),帶來一個重要性質: 完美前向保密——就算伺服器的長期私鑰未來被偷,攻擊者也無法解開他過去側錄下來的舊連線, 因為每次連線用的那把臨時金鑰早就用完即丟了。「今天錄起來、等以後破解」的攻擊,被這招擋了下來。

鐵律:永遠別自己造密碼系統

讀到這裡,你可能躍躍欲試,想自己寫個加密函式庫。請務必先讀完這一節。整章我們反覆看到一件事: 密碼學的演算法本身(AES、RSA)往往不是被攻破的點,真正出事的幾乎都是「怎麼用」的細節—— 用了 ECB 模式、重複用了金鑰、用了可預測的亂數、沒做完整性檢查、比對時洩漏了時間差(旁路攻擊,side-channel)…… 每一個都是深不見底的坑,而且程式跑得出正確結果,不代表它安全——這正是密碼學 bug 最可怕的地方:它默默地、無聲地不安全。

⛔
Don't roll your own crypto(別自己造密碼系統)

這是整個資安領域最重要的一句忠告。不要自己設計密碼演算法,也不要自己拼裝密碼協定。 請使用經過全世界專家公開審查、久經實戰的成熟函式庫——例如 Python 的 cryptography、 libsodium / PyNaCl,或直接用作業系統/語言內建、設定好的 TLS。它們替你處理了那成千上萬個你想不到的細節。 本章所有玩具程式碼,只為「教學、建立直覺」而存在,絕不可用於任何真實系統。 記住:在密碼學裡,自信地造輪子,通常等於自信地把使用者送進火坑。把聰明用在「正確地使用」成熟工具上,而不是重新發明它們。

✏️ 練習 35.6 挑戰

(任選一題)(a) 用程式具體示範「為什麼 ECB 模式會洩漏圖案」:寫一個玩具區塊加密, 把一段「含有重複區塊」的資料用 ECB 方式加密,指出相同明文區塊產生了相同密文區塊。 (b) 實作一個加鹽 + 慢雜湊的密碼儲存與驗證流程,並說明它如何同時擋掉彩虹表與暴力破解。

看解答

(a) ECB 洩漏圖案的最小示範:

# 玩具「區塊密碼」:每個位元組加上金鑰再取模。極不安全,只為示範 ECB 的結構問題。
def toy_block_encrypt(block, key):
    return bytes((b + key) % 256 for b in block)

def ecb_encrypt(data, key, block_size=4):
    blocks = [data[i:i+block_size] for i in range(0, len(data), block_size)]
    return [toy_block_encrypt(b, key).hex() for b in blocks]

data = b"SAMESAMESAMEDIFF"   # 前三個 4-byte 區塊都是 b"SAME"
for ct in ecb_encrypt(data, key=42):
    print(ct)
# 你會看到前三行的密文「完全相同」—— 因為明文區塊相同。
# 圖片裡大片同色背景 = 大量相同區塊 = 大量相同密文 → 輪廓被保留(ECB 企鵝)。
# 修正:用 CBC/CTR/GCM,讓每塊加密都混入前文或計數器,相同明文也得到不同密文。

(b) 加鹽 + 慢雜湊的密碼儲存:

import hashlib, os, hmac

def hash_password(password, salt=None):
    if salt is None:
        salt = os.urandom(16)                 # 每個使用者一份隨機鹽(16 位元組)
    # PBKDF2:故意迭代很多次,把單次雜湊拖慢
    digest = hashlib.pbkdf2_hmac("sha256", password.encode(), salt, 200_000)
    return salt, digest

def verify(password, salt, expected_digest):
    _, digest = hash_password(password, salt)
    # 用 compare_digest 做「定時比較」,避免靠比對時間差的旁路攻擊
    return hmac.compare_digest(digest, expected_digest)

# 註冊:存下 (salt, digest),絕不存明文密碼
salt, digest = hash_password("correct horse battery staple")
# 登入驗證:
print(verify("correct horse battery staple", salt, digest))  # True
print(verify("wrong password", salt, digest))                # False

為什麼這樣安全?

  • 鹽擋彩虹表:每個使用者鹽不同,同樣的密碼也算出不同雜湊,攻擊者預算好的彩虹表全部作廢,只能針對「每一份鹽」重新硬算。
  • 慢雜湊擋暴力:PBKDF2 迭代 20 萬次,把每次嘗試拖慢到數十毫秒。你登入時無感,但攻擊者每秒能猜的數量被砍掉好幾個數量級。
  • 定時比較擋旁路:hmac.compare_digest 不會「比到第一個不同的位元組就提早回傳」,避免攻擊者用回應時間反推出正確前綴。

(再次提醒:真實產品請直接用 bcrypt / argon2-cffi 等成熟函式庫,別自己組裝——呼應本章鐵律。)

把這一章串起來

我們從一張「在大庭廣眾下傳遞的情書」出發,一路走到了守護全球網路的 TLS。回顧這趟旅程的路標:

  • 密碼學的任務:在不安全的通道上,達成機密性、完整性與真實性(呼應第 34 章的 CIA、第 25 章的封包可被竊聽)。
  • 古典密碼(凱撒、替換)敗給暴力破解與頻率分析,留下 Kerckhoffs 原則:安全只能靠金鑰,不能靠保密演算法。
  • 對稱加密(AES)又快又強,但卡在金鑰分配;XOR/一次性密本理論完美卻不實用;模式很重要(ECB 企鵝的教訓)。
  • 密碼學雜湊(SHA-256)是單向指紋,提供完整性;存密碼要加鹽 + 慢雜湊,且絕不用 MD5/SHA-1。
  • 公開金鑰(RSA、Diffie-Hellman、ECC)用「易正向、難反向」的數論,繞過金鑰分配難題——這是全章的魔法核心。
  • 數位簽章把公開金鑰反過來用(私鑰簽、公鑰驗),一次給齊完整性、真實性與不可否認性。
  • PKI 與憑證用信任鏈解決「公鑰真偽」;TLS 把全部零件串起來:用非對稱換對稱,再用對稱高速傳輸。
  • 最後,那條用血換來的鐵律:永遠別自己造密碼系統。

密碼學最迷人的地方在於:它把「不可能」變成「可能」——讓素未謀面的兩人在眾目睽睽下交換祕密,讓你能證明身分卻不洩漏密碼, 讓一段亂碼承載你最重要的信任。而這一切,竟然只建立在幾個關於質數與餘數的古老定理之上。下一章,我們會把視角從「數學」 拉回「人與系統」,看看真實世界的攻擊者如何繞過密碼學、直接攻擊它周圍的軟體與人——畢竟,最堅固的鎖,也擋不住一扇沒關的窗。
