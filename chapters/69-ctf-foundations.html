一個不是 PNG 的 PNG

比賽開始三分鐘,你下載了第一題的附件:一個叫 cat.png 的檔案。雙擊它—— 圖片打不開,系統說「檔案毀損」。換成別人可能就放棄了:「啊,壞檔。」但你打開終端機, 敲了一行 file cat.png,螢幕回你:cat.png: ASCII text。

有意思。這根本不是圖片,是純文字偽裝的。你再敲 cat cat.png,跳出一行 RkxBR3tuMHRfYV9yM2FsX2NhdH0=——結尾那個 = 號、清一色的英數字, 是 Base64 的長相。你把它丟給 base64 -d,畫面浮出:FLAG{n0t_a_r3al_cat}。 第一題,解出。

這就是 CTF 的日常:一連串「東西不是它看起來的樣子」的小謎題,而你手裡握著一把把工具, 一層層把真相剝出來。沒有人被駭、沒有真實系統受害——所有題目都是出題者故意藏好、等你來找的。 這一章,我們就從零把你帶進這個世界:先弄懂遊戲規則,搭好工作台,然後玩第一場尋寶、寫第一支解碼工具。

從「奪旗」說起:CTF 是什麼

CTF(Capture The Flag)直譯是「搶旗子」,本來是一種戶外遊戲:兩隊各守一面旗, 想辦法攻進對方陣地把旗搶回來。資安界借用了這個名字,辦成一種競賽兼遊戲: 主辦方設計一堆刻意藏有漏洞或謎題的題目,你解開之後,會得到一小串特殊字串——那就是「旗(flag)」—— 把旗貼回計分板,就得分。誰分數高誰贏。

CTF 是全世界資安人「練功」與「較量」的共通語言。從高中生的入門賽(像 picoCTF), 到頂尖戰隊的世界大賽(像 DEF CON CTF),用的都是同一套精神。它最迷人的地方在於: 它把「攻擊」這件危險的事,圈進一個完全合法、可重來、不傷害任何人的沙盒裡, 讓你能放手嘗試、失敗、再嘗試。

flag 長什麼樣子

旗通常是一段一眼就認得出來的字串,最常見的格式是: 

FLAG{some_text_here}
picoCTF{y0u_g0t_1t_4b2c1d}
flag{th1s_1s_th3_pr1z3}

前面的 FLAG(或比賽自訂的前綴)像個「招牌」,大括號裡是內容。 這個格式不是隨便定的,它有兩個實用目的:第一,讓你一眼認出「找到了!」—— 在一堆亂碼裡看到 FLAG{ 開頭,就知道挖到寶了;第二,方便機器自動比對。 後面寫工具時你會看到,我們常常就是寫程式去檢查「解出來的東西是不是以 FLAG{ 開頭」。

🚩
為什麼旗裡常是「火星文」

你會發現 flag 內容常把 e 寫成 3、i 寫成 1、o 寫成 0 (這叫 leetspeak)。一部分是駭客文化的趣味,一部分是避免旗剛好等於某個正常單字, 以免在文章或程式裡誤觸發比對。看到 l1k3_th1s 別緊張,它就是「like this」。

學「攻擊」,是為了更會「防守」

你可能會問:一本立場是「防禦」的書,為什麼要教我解 CTF、教我「攻擊」的思維? 答案在第 34 章那句話裡——你無法防禦你不認識的東西。 一個從沒看過 SQL injection 怎麼打的工程師,寫出來的查詢就是會有洞; 一個親手用 strings 從執行檔裡撈出過硬寫密碼的人,以後絕不會把密碼寫死在程式裡。

CTF 是把「攻擊者視角」訓練到肌肉記憶最有效的方法。當你解過幾十題「開發者相信了不該相信的輸入」, 你寫程式時就會自動對所有輸入起疑心。這正是第 36 章「安全程式設計」反覆強調的習慣—— 而 CTF,是養成這個習慣的健身房。每一道你親手攻破的題,日後都變成你親手堵上的洞。

建立你的駭客工作台

玩 CTF 的第一件事,不是學什麼神祕指令,而是搭一個能跑工具的環境。 幾乎所有 CTF 工具都生長在 Linux 的土壤上;而對絕大多數用 Windows 的同學來說, 最省事、最不會把主系統搞壞的方法,是裝 WSL2(Windows Subsystem for Linux)—— 它讓你在 Windows 裡跑一個真正的 Ubuntu Linux,共用你的檔案,卻互不干擾。 

# 在 Windows 的 PowerShell(用系統管理員身分開啟)輸入這一行,然後重開機:
wsl --install

# 重開機後它會自動裝好 Ubuntu,要你設定一組 Linux 的帳號與密碼。
# 之後在開始選單打 "Ubuntu" 就能進入你的 Linux 終端機。

# 進去後,先把套件清單更新、裝幾個 CTF 基本工具:
sudo apt update
sudo apt install -y xxd binutils coreutils file grep findutils

(用 macOS 或本來就用 Linux 的同學更輕鬆,你的終端機已經就位,多數工具內建,缺的用 brew 或 apt 補即可。)裝好之後,你就有了一個「駭客工作台」—— 說穿了就是一個裝好工具的 Linux 終端機。沒有駭客電影裡滿螢幕滾動的綠色字, 就是一個樸素的黑底白字視窗。但這個視窗,是你接下來所有冒險的起點。

為什麼終端機是駭客的主場

第 16 章已經告訴你終端機的威力:可組合、可自動化、可重現。對 CTF 來說, 還要再加一個關鍵理由——圖形介面只給你「開發者想讓你看到」的東西,而 CTF 的旗, 偏偏都藏在「沒人想讓你看到」的地方:隱藏檔、檔案的尾端、二進位的縫隙、被編碼過的字串。 滑鼠雙擊一個「壞掉」的圖片,你只會看到「無法開啟」;但一行 file 指令,就揭穿了它的偽裝。

終端機讓你直視位元組的真相,而不是作業系統幫你「美化」過的表象。這就是為什麼, 在 CTF 圈,鍵盤永遠比滑鼠快、比滑鼠強。下面這幾個指令,你會用到天荒地老:

記不得 grep 怎麼遞迴搜尋?打 man grep,裡面有完整說明,按 q 離開。 學會「自己查手冊」是駭客最重要的軟實力之一——CTF 沒有標準答案,只有「你願不願意自己挖」。 真的卡死再問人,但先說清楚你已經試了什麼,這是這個圈子的禮貌。

玩一場「終端機尋寶」

光看指令表記不住。最好的學法,是把它當尋寶遊戲玩。想像有一個資料夾,裡面藏了 8 面旗, 每一面都用一個不同的指令才挖得出來,而且解開一面旗,它會提示你下一面藏在哪。 這就是「終端機尋寶」——CTF 社團公認最有效的 Linux 入門法。它的關卡設計大概像這樣: 

# 第 1 關:最基本的「看」。先看有什麼,再讀第一個檔。
$ ls
welcome.txt
$ cat welcome.txt
歡迎來到尋寶場!你的第一面旗是 FLAG{...}
下一關提示:有些檔案是「隱藏」的,試試 ls 的某個旗標……

# 第 2 關:隱藏檔。檔名以 . 開頭的,要 -a 才看得到。
$ ls -a
.   ..   .hidden_flag   welcome.txt
$ cat .hidden_flag

# 第 3 關:大海撈針。旗藏在 logs/ 裡上百個檔的某一行。
$ grep -r FLAG logs/        # -r:遞迴鑽進每個子目錄找含 FLAG 的行

# 第 4 關:深層迷宮。旗檔藏在很深的資料夾,但你知道它的檔名。
$ find . -name 'flag*'      # 從目前目錄往下,找檔名以 flag 開頭的

# 第 5 關:偽裝的二進位。data.bin 看似亂碼,其實藏了可讀的字。
$ file data.bin             # 先問它「你到底是什麼」
$ strings data.bin | grep FLAG    # 把可讀文字撈出來,再篩出旗

# 第 6 關:十六進位。secret.hex 是一串 hex,還原它。
$ xxd -r -p secret.hex      # -r 還原、-p 用純 hex 格式

# 第 7 關:Base64。message.b64 結尾有 = 號……
$ base64 -d message.b64     # 解開後你會懂「編碼不是加密」(下一節主角!)

# 第 8 關:大檔尾端。旗在一個超大 log 的最後幾行。
$ tail bigfile.log          # 看尾端;或直接 grep FLAG bigfile.log

看出門道了嗎?每一關都不是「猜」,而是「用對工具」。第 7 關尤其關鍵—— 那個被 Base64 包起來的旗,任何人只要會 base64 -d 就能解開,完全不需要密碼。 這個「咦,不用鑰匙就解得開?」的疑惑,正是我們下一大節要展開的核心觀念。 本章末的練習 69.1,就讓你親手玩一遍這種尋寶。

C 到 Python 速成:資安腳本的母語

指令能解掉很多入門題,但稍微複雜一點——要批次嘗試、要做運算、要跟一個遠端服務來回對話—— 你就需要寫程式。而在 CTF 與資安圈,程式語言幾乎只有一個答案:Python。

為什麼?因為 Python 寫得快(一個想法十分鐘變成可跑的腳本)、 函式庫多(處理位元、網路、密碼學的工具一應俱全:pwntools、requests、 pycryptodome……),而且幾乎所有公開的解題範例(writeup)都用 Python 寫。 在這個圈子,Python 就是通用語。

🔗

看到最後兩列了嗎?十六進位寫法與位元運算符號,C 和 Python 一字不差。 這正是學過 C 的人玩 CTF 的最大紅利:那些讓初學者頭痛的 ^(XOR)、&(AND)、 <<(左移),你早就熟得很。同一段邏輯,兩種語言對照一下: 

// C:把每個位元組與金鑰 0x5A 做 XOR
#include <stdio.h>
int main(void) {
    unsigned char data[] = {0x12, 0x34, 0x56};
    for (int i = 0; i < 3; i++) {
        printf("%02x ", data[i] ^ 0x5A);   // ^ 是 XOR
    }
    return 0;
}
# Python:一模一樣的邏輯,但短得多
data = [0x12, 0x34, 0x56]
for b in data:
    print(f"{b ^ 0x5A:02x}", end=" ")   # ^ 一樣是 XOR,f-string 補零成兩位 hex

Python 的頭號大坑:bytes ≠ str

如果說有一個觀念能讓 C 出身的人在 Python 玩 CTF 時少踩一半的坑,那就是這個: Python 嚴格區分「文字(str)」和「位元組(bytes)」,它們是兩種不同的東西。

  • str:給人看的文字,例如 "Hello"。
  • bytes:給機器處理的原始位元組,前面加個 b,例如 b"Hello"。

在 C 裡,字串本來就是一串 char(其實就是位元組),兩者沒分。但在 Python, 所有加密、編碼、雜湊、網路傳輸的函式,吃的、吐的都是 bytes——因為位元組才是世界的真相。 兩者之間用 .encode() 和 .decode() 來回轉換: 

s = "FLAG"            # str,文字
b = s.encode()        # str -> bytes,得到 b'FLAG'
print(b)              # b'FLAG'
print(list(b))        # [70, 76, 65, 71]  ← 每個字元的 ASCII 數值!

b2 = b'FLAG'
s2 = b2.decode()      # bytes -> str,變回 'FLAG'

# 常見錯誤:把 str 丟給只吃 bytes 的函式
import base64
# base64.b64encode("FLAG")    # ✗ TypeError!它只吃 bytes
base64.b64encode(b"FLAG")     # ✓ 正確,得到 b'RkxBRw=='
⚠️
看到 TypeError: ... requires a bytes-like object 別慌

這是初學者最常撞的錯誤,意思永遠是同一個:「你給了我文字,但我只收位元組。」 解法也永遠一樣:在你傳進去的那個值後面加 .encode()。反過來,如果結果印出來是 b'...' 帶個 b 開頭、你想要乾淨的文字,就加 .decode()。 把這兩個動作練成反射,Python 玩 CTF 就順了一大半。

本章主菜:編碼,不是加密

現在來到全章最重要的觀念。回想終端機尋寶的第 7 關:那面旗只是被 Base64 包了一下, 你不需要任何密碼、任何鑰匙,一行 base64 -d 就解開了。如果那是「加密」, 怎麼可能沒鑰匙就解得開?答案是:它根本不是加密,是「編碼」。

🔑
編碼 ≠ 加密(這是 Crypto 入門的第一課)

編碼(encoding)是為了「方便傳輸或表示」而換一種寫法,規則完全公開, 任何人都能還原——它不提供任何機密性。Base64、hex、URL 編碼都屬於這類。
加密(encryption)則是為了「保密」,沒有金鑰就還原不了——它提供機密性。
一句話分辨:需要鑰匙才能解開的,才是加密;人人都能解開的,只是編碼。 把編碼當成保護,是 CTF 裡(也是真實世界裡)最常見的安全錯誤之一。

為什麼要先搞懂這個?因為 CTF 的 Crypto 題,有一大半根本不是「破解加密」,而是 「認出這是哪種編碼,然後反向還原」。能不能一眼認出 = 結尾是 Base64、 全是 0–9a–f 是 hex、滿是 %20 是 URL 編碼——這就是基本功。接下來我們一個一個拆。

萬物的起點:ASCII

電腦只懂數字,那它怎麼存「文字」?靠一張對照表:ASCII。它規定每個英文字母、 數字、符號各對應一個 0–127 的數字。例如大寫 A 是 65、F 是 70、 a 是 97、空格是 32。所以當你存下字串 "FLAG",記憶體裡其實是四個數字: 

# 用 Python 看穿一個字串「真正的樣子」
for ch in "FLAG":
    print(ch, "->", ord(ch))      # ord:字元轉 ASCII 數值
# F -> 70    L -> 76    A -> 65    G -> 71

print(chr(70))     # chr:反過來,數值轉字元 -> 'F'

記住這件事:文字的底層就是一串數字(位元組)。所有編碼,玩的都是 「同一串位元組,換個方式寫出來」的把戲。理解了這點,後面全都通了。

十六進位(hex):位元組的素顏照

一個位元組是 8 個位元,能表示 0–255。每次寫 8 個 0/1 太累,所以我們用 十六進位(hexadecimal,簡稱 hex):用 0–9 和 a–f 這 16 個符號,剛好兩個 hex 字元 = 一個位元組(因為 16×16 = 256)。 於是任何資料都能寫成一串乾淨的 hex: 

data = b"FLAG"
print(data.hex())                 # bytes -> hex 字串:'464c4147'
#   F=0x46  L=0x4c  A=0x41  G=0x47   ← 每兩個 hex 一個位元組

print(bytes.fromhex("464c4147"))  # hex 字串 -> bytes:b'FLAG'

在終端機裡,xxd 做的就是這件事(看 hex);xxd -r -p 則是反向(把 hex 還原成原始位元組)。 看到一長串只有 0-9a-f 的字串,你的第一個念頭就該是:「這是 hex,先還原看看。」

Base64:為什麼 3 個位元組變成 4 個字元?

Base64 是 CTF 裡出現頻率最高的編碼,沒有之一。它的任務是:把「任意位元組」(可能含有 無法顯示的控制字元)安全地塞進「只能傳純文字」的管道——例如 email、網址、JSON。 它的做法很巧妙,而且理解它,你就真正懂了「編碼」在玩什麼。

核心訣竅是「換一種切法」:原始資料是一個位元組 8 個位元地排;Base64 不管位元組邊界, 把所有位元接成一長條,然後每 6 個位元切一刀。為什麼是 6?因為 $2^6 = 64$, 剛好可以用一張 64 個字元的表(A–Z、a–z、0–9、+、/)來表示每一段。 於是:每 3 個位元組(24 位元)= 4 段 6 位元 = 4 個字元。這就是「3 變 4」的祕密:

那如果位元組數不是 3 的倍數呢?Base64 會用 = 補位。這就是為什麼 Base64 字串常以一兩個 = 結尾——它是你認出 Base64 的最佳線索: 

import base64
print(base64.b64encode(b"Man"))   # b'TWFu'   3 bytes -> 4 字元,剛好,無 =
print(base64.b64encode(b"Hi"))    # b'SGk='   2 bytes -> 補 1 個 =
print(base64.b64encode(b"M"))     # b'TQ=='   1 byte  -> 補 2 個 =

print(base64.b64decode("TWFu"))   # b'Man'    反向還原,完全不需要金鑰

URL 編碼:網址裡的 %20

你一定看過網址裡出現 %20、%3D 這種東西。因為網址不能直接包含空格、 &、= 這些有特殊意義的字元,於是 URL 編碼(percent-encoding) 把它們換成 % 加上該字元的 hex 值。空格是 ASCII 32 = 0x20,所以變成 %20: 

import urllib.parse
print(urllib.parse.quote("a b&c=1"))    # 'a%20b%26c%3D1'
print(urllib.parse.unquote("a%20b%26c%3D1"))  # 'a b&c=1'

看到滿是 % 跟兩位 hex 的字串,你就知道:這是 URL 編碼,解開它。

XOR:CTF 最愛的「可逆魔法」(呼應第 35 章)

前面三種是「換寫法」的編碼。XOR 則站在編碼與加密的交界處:它是一個位元運算, 卻也是許多真實加密法的核心零件(第 35 章會看到它撐起一次性密碼本與串流加密)。 對 CTF 來說,你只要先抓住它一個神奇性質:XOR 同一個值兩次,會變回原樣。

XOR(互斥或,寫作 ^)的規則只有一條:兩個位元相同得 0,不同得 1。 從這條規則,可以推出那個讓它在密碼學裡無所不在的性質: 

# XOR 的可逆性,用 Python 一眼看穿
P = ord('A')      # 65
K = 0x2A          # 金鑰(隨便一個位元組)
C = P ^ K         # 「加密」
print(C)          # 111
print(C ^ K)      # 「解密」:再 XOR 一次同樣的 K -> 65,變回 'A'
⚠️
單位元組 XOR:看似加密,其實一秒破

有些題目把旗用「單一個位元組」當金鑰去 XOR,以為這樣就安全了。錯。 金鑰只有 1 個位元組,代表只有 256 種可能(0 到 255)。你可以寫個迴圈把 256 把鑰匙 全部試一遍,看哪一把解出來的結果長得像 FLAG{...}——這叫暴力破解(brute force), 現代電腦眨眼就跑完。這也再次說明:鑰匙太短的「加密」,跟沒加密差不多。練習 69.4 就讓你親手做這件事。

動手做:寫一支「編碼瑞士刀」

觀念講完,該把它們組裝成一個能用的工具了。CTF 老手人人都有一支自己的「編碼瑞士刀」—— 一個能在 文字、hex、Base64、URL、XOR 之間自由互轉的小程式。我們現在就來寫一支。 它不長,但把這一章的所有觀念全串了起來: 

#!/usr/bin/env python3
# -*- coding: utf-8 -*-
"""編碼瑞士刀:文字 / hex / Base64 / URL / XOR 互轉。CTF 入門必備。"""
import base64
import urllib.parse

# ---------- 編碼方向(把原始 bytes 換種寫法)----------
def to_hex(data: bytes) -> str:
    return data.hex()

def to_b64(data: bytes) -> str:
    return base64.b64encode(data).decode()

def to_url(text: str) -> str:
    return urllib.parse.quote(text)

# ---------- 解碼方向(還原回原始 bytes)----------
def from_hex(text: str) -> bytes:
    return bytes.fromhex(text.strip())

def from_b64(text: str) -> bytes:
    return base64.b64decode(text.strip())

def from_url(text: str) -> str:
    return urllib.parse.unquote(text)

# ---------- XOR(加密=解密,key 為 bytes,循環使用)----------
def xor(data: bytes, key: bytes) -> bytes:
    # 把 data 的每個位元組,跟 key 對應位置的位元組做 XOR
    # key 比 data 短時,用 i % len(key) 讓金鑰循環重複
    return bytes(data[i] ^ key[i % len(key)] for i in range(len(data)))

# ---------- 試跑:每種轉換都「轉過去再轉回來」驗證正確 ----------
if __name__ == "__main__":
    s = "FLAG{sw1ss_kn1f3}"
    b = s.encode()                 # str -> bytes

    print("原文     :", s)
    print("hex      :", to_hex(b))
    print("  還原   :", from_hex(to_hex(b)).decode())
    print("base64   :", to_b64(b))
    print("  還原   :", from_b64(to_b64(b)).decode())
    print("url      :", to_url(s))
    print("  還原   :", from_url(to_url(s)))

    # XOR 兩次同樣金鑰,應該變回原文
    k = b"\x5a"                    # 單位元組金鑰 0x5A
    once = xor(b, k)
    twice = xor(once, k)
    print("XOR 兩次 :", twice.decode())   # FLAG{sw1ss_kn1f3}

執行它,你會看到每一種編碼「去」與「回」都對得上,XOR 兩次也乖乖變回原文。 恭喜,你有了第一件親手打造的 CTF 武器。現在,讓我們用它來打一個大魔王。

大魔王:剝開包了三層的 flag

真實的 CTF 題,往往不會只用一層編碼,而是層層包裹,故意讓你多繞幾圈。 經典的「洋蔥題」長這樣:出題者拿到 flag,先 XOR、再 Base64、再 hex,一層層往外包; 你拿到的只是最外面那串 hex,得反著順序、一層層往內剝: 

# 接續上面的瑞士刀,加上「剝洋蔥」函式
def peel(payload_hex: str) -> str:
    """拿到最外層的 hex,反向剝開 hex -> base64 -> xor(0x5A) -> flag"""
    step1 = from_hex(payload_hex)         # 剝掉 hex 層 -> 得到 Base64 文字(bytes)
    step2 = from_b64(step1.decode())      # 剝掉 Base64 層 -> 得到 XOR 後的 bytes
    flag  = xor(step2, b"\x5a")           # 剝掉 XOR 層 -> 還原成 flag
    return flag.decode()

# 出題者給你的(只有最外層那串 hex):
challenge = "48425962485345324f794d2f4b436b464e6a4d78507755374e4155314e444d314e43633d"
print(peel(challenge))     # FLAG{layers_like_an_onion}

跑跑看,旗就跳出來了。注意解題函式 peel 的順序,正好是出題包裝的相反: 包的時候是 XOR→Base64→hex,剝的時候就是 hex→Base64→XOR。這種「認出層次、反向拆解」的思路, 會貫穿你整個 CTF 生涯。練習 69.5 留了一題類似的給你親手剝。

工具速覽:不必每次都重造輪子

自己寫瑞士刀是為了真懂原理;但實戰比賽分秒必爭,很多時候你會直接用現成工具。 兩個你一定要認識:

  • CyberChef:一個瀏覽器裡的「編碼/解碼瑞士刀」,被暱稱為「網路的瑞士刀」。 它把上百種操作做成積木,你用拖拉的方式串成一條「食譜(recipe)」——例如「From Hex → From Base64 → XOR」—— 貼上資料就即時看到結果。本章的剝洋蔥題,在 CyberChef 上三個積木就解完了。它甚至有個「Magic」功能, 能自動猜測資料被哪些編碼包過。這是入門 Crypto/Misc 題的神器,而且它純前端、可離線使用。
  • 命令列工具:你已經見過的 base64、xxd、strings、file, 加上 tr(字元替換)、rev(反轉)、od(看八進位/十六進位)。它們的好處是 能串進管線、能自動化——當你要對一千個檔案做同一件事,終端機完勝任何網頁工具。

✏️ 練習 69.1 實作

你有一個 treasure/ 資料夾,旗藏在三個不同的地方。請用終端機指令把它們挖出來:

  1. 資料夾裡有上百個 .txt 檔,其中某一個的某一行含有 FLAG{...}。怎麼一次找出來?
  2. 有個檔叫 image.bin,cat 它只看到亂碼,但裡面藏了可讀的旗。怎麼撈?
  3. 旗檔深藏在好幾層子目錄裡,你只知道它的檔名以 flag 開頭。怎麼定位它?
看解答

第 1 題——遞迴搜尋字串,用 grep -r:

$ grep -r "FLAG{" treasure/        # -r 鑽進每個子目錄,印出含 FLAG{ 的行
$ grep -rn "FLAG{" treasure/       # 加 -n 連「在哪個檔的第幾行」都告訴你

第 2 題——先認類型,再用 strings 撈可讀文字:

$ file image.bin                   # 先問它到底是什麼
$ strings image.bin | grep "FLAG"  # 把所有可讀字串撈出來,再篩出旗

第 3 題——用 find 依檔名遞迴尋找:

$ find treasure/ -name 'flag*'     # 找檔名以 flag 開頭的檔案
$ cat $(find treasure/ -name 'flag*')   # 找到後直接讀它的內容

這三招(grep -r 搜內容、strings 撈藏字、find 找檔名)是 Misc/Forensics 題的鐵三角, 幾乎每場比賽都用得到。

✏️ 練習 69.2 實作

你攔截到一段字串 RkxBR3tiYXNlNjRfaXNfbm90X2VuY3J5cHRpb259。 (a) 你怎麼一眼判斷它「可能」是 Base64?(b) 用 Python 把它解開。 (c) 這題想提醒你什麼觀念?

看解答

(a) 它只由 A–Z a–z 0–9 + / 組成、長度是 4 的倍數——這是 Base64 的典型特徵 (這串剛好沒有 = 結尾,因為長度正好對齊;有 = 結尾就更篤定)。

(b) 用 Python 的 base64 模組:

import base64
s = "RkxBR3tiYXNlNjRfaXNfbm90X2VuY3J5cHRpb259"
print(base64.b64decode(s).decode())   # 記得 .decode() 把 bytes 變成可讀文字
# FLAG{base64_is_not_encryption}

在終端機裡更快:echo RkxBR3t... | base64 -d。

(c) 旗的內容自己說了:Base64 不是加密!沒有任何金鑰,任何人都能還原。 這正是本章的核心觀念——編碼只是「換種寫法」,不提供機密性。

✏️ 練習 69.3 實作

實作 hex 與 bytes 的雙向轉換。 (a) 先用 Python 內建方法,把 b"FLAG" 轉成 hex 字串,再轉回來。 (b) 進階:不准用 .hex() 和 bytes.fromhex(),自己刻一個 hex→bytes 的函式, 證明你真的懂「兩個 hex 字元 = 一個位元組」。

看解答

(a) 內建方法最簡單:

b = b"FLAG"
h = b.hex()                 # bytes -> hex:'464c4147'
print(h)
print(bytes.fromhex(h))     # hex -> bytes:b'FLAG'

(b) 自己刻 hex→bytes:每兩個字元取一段,用 16 進位解讀成一個位元組值。 關鍵是 int(段, 16) 能把 "4c" 這種兩位 hex 轉成數字 76:

def my_from_hex(h: str) -> bytes:
    h = h.strip()
    out = bytearray()
    for i in range(0, len(h), 2):     # 每次跳 2 個字元
        pair = h[i:i+2]               # 取兩個 hex 字元,例如 '4c'
        out.append(int(pair, 16))     # int(_, 16):當 16 進位解讀 -> 76
    return bytes(out)

print(my_from_hex("464c4147"))        # b'FLAG'

# 反方向:bytes -> hex,也自己刻
def my_to_hex(b: bytes) -> str:
    return "".join(f"{x:02x}" for x in b)   # 每個位元組補零成兩位 hex

print(my_to_hex(b"FLAG"))             # '464c4147'

能自己刻出這兩個函式,代表你真正理解了「位元組」和「它的十六進位寫法」之間的關係—— 這是後面所有 Crypto 題的底層直覺。

✏️ 練習 69.4 挑戰

你拿到一段被單一位元組金鑰 XOR 過的密文(以 hex 表示),但不知道金鑰是多少:

6c666b6d51521a58751b5975405f595e754b4e4e435e43454457

請寫一支 Python 程式,暴力破解出金鑰並還原旗。提示:單位元組金鑰只有 256 種可能, 全部試一遍,看哪一把解出來的結果以 FLAG{ 開頭。

看解答

核心思路:把 256 把鑰匙(0~255)逐一拿去 XOR,用「結果是否以 FLAG{ 開頭」當作判斷對錯的訊號。

cipher = bytes.fromhex("6c666b6d51521a58751b5975405f595e754b4e4e435e43454457")

for key in range(256):                    # 試遍每一把可能的鑰匙
    # 用同一把 key 對整段密文做 XOR(XOR 加解密同一運算)
    plain = bytes(b ^ key for b in cipher)
    if plain.startswith(b"FLAG{"):        # 認得出旗的長相,就是它!
        print(f"金鑰 = {key} (十六進位 {key:#x})")
        print("旗 =", plain.decode())
        break

# 金鑰 = 42 (十六進位 0x2a)
# 旗 = FLAG{x0r_1s_just_addition}

這就是「已知明文特徵 + 暴力搜尋」的經典套路:金鑰空間小到能全試時, 你不需要任何聰明的數學,只要一個迴圈加一個「認得出答案」的判斷式。也再次印證: 鑰匙太短的加密,等於沒加密。

✏️ 練習 69.5 挑戰

大魔王回歸。下面這串 hex,是一面旗被三層包起來的結果,包裝順序是 XOR(金鑰 0x5A) → Base64 → hex。請反向剝開,拿到旗:

48425962485345324f794d2f4b436b464e6a4d78507755374e4155314e444d314e43633d

提示:剝的順序,和包的順序相反。先剝最外層的 hex……

看解答

包裝是「由內往外」XOR→Base64→hex,所以解題要「由外往內」hex→Base64→XOR,順序反過來:

import base64

payload = "48425962485345324f794d2f4b436b464e6a4d78507755374e4155314e444d314e43633d"

step1 = bytes.fromhex(payload)            # ① 剝掉 hex 層 -> 得到一段 Base64 文字
print("剝掉 hex:", step1)                 # b'HBYbHSE2OyM/KCkFNjMxPwU7NAU1NDM1NCc='

step2 = base64.b64decode(step1.decode())  # ② 剝掉 Base64 層 -> 得到 XOR 後的 bytes

# ③ 剝掉 XOR 層:每個位元組再 XOR 一次 0x5A 還原
flag = bytes(b ^ 0x5A for b in step2)
print("旗:", flag.decode())               # FLAG{layers_like_an_onion}

觀察那個結尾的 =(在 step1 印出來時可見):它出賣了「中間那層是 Base64」。 在真實比賽裡,出題者不會告訴你包了哪幾層、用什麼金鑰——你得靠這些蛛絲馬跡(= 結尾、 全是 hex 字元、可印字元的分布)自己推理。若懶得手刻,把這串 hex 丟進 CyberChef, 用 From Hex → From Base64 → XOR(0x5A) 三個積木也能解,還能用它的「Magic」自動猜層次。
