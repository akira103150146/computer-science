複雜度是這一章的共同語言

我們會反覆用大 O 記號($O(1)$、$O(\log n)$、$O(n)$)來描述操作有多快。 如果你對這套記號還不熟,可以先回顧第 26 章「演算法複雜度」。 一句話複習:大 O 描述的是「資料量 $n$ 長大時,所需步驟成長的趨勢」, 而不是真正的秒數。$O(1)$ 是「不管多大都一樣快」,$O(n)$ 是「大一倍就慢一倍」, $O(\log n)$ 則是「大一千倍也只慢十倍」——好得驚人。

抽象資料型別(ADT):先說「能做什麼」,再說「怎麼做」

在動手打造任何結構之前,先建立一個會貫穿整章的觀念。回想第 8 章 談抽象與封裝時的核心精神:把「介面」和「實作」分開。 資料結構的世界,把這個精神提煉成一個專有名詞:抽象資料型別(abstract data type, ADT)。

ADT 是一種規格,它只回答兩個問題:「這個型別裝什麼資料?」「它支援哪些操作?」—— 但它絕口不提這些操作是怎麼實現的。舉個例子,「堆疊(stack)」這個 ADT 說: 我裝一疊東西,你可以 push(放一個上去)、pop(拿最上面那個), 且保證「後進先出」。至於這疊東西底下是用陣列存、還是用鏈結串列串起來?ADT 不管。那是實作的事。

為什麼這個分離這麼重要?因為它讓使用者和實作者各自獨立。 使用堆疊的人,只要照著介面寫,程式就能運作;哪天你發現另一種實作更快, 把底層整個換掉,使用者的程式一行都不用改。這正是 Python 的 list、dict 能在不同版本間悄悄優化、而你的程式照跑不誤的原因。整章接下來介紹的每個結構, 你都可以用這個框架去看:它的介面(承諾哪些操作、各要多快)是一回事, 它的實作(用陣列?用節點+指標?)是另一回事。

陣列 vs 鏈結串列:兩種「排成一列」的方式

最基本的需求是「把一堆東西排成一列」。這件事有兩種根本不同的做法, 它們的差異,源自一個你看不見、卻決定一切的東西:資料在記憶體裡怎麼擺。

陣列:緊挨著住的連續門牌

陣列(array)把所有元素連續地擺在記憶體裡,一個緊接著一個, 像一排門牌號連續的房子。因為連續,電腦只要知道「起點位址」和「每格多大」, 就能用一條乘法直接算出第 i 個元素在哪:起點 + i × 每格大小。 這就是為什麼用索引取值是 $O(1)$——不管陣列多長,一步到位(這也呼應第 4 章 說的「索引就是離起點多遠的偏移量」)。

代價呢?連續是把雙面刃。要在中間插入或刪除一個元素,後面所有元素 都得整批往後挪或往前補,平均得搬一半,所以是 $O(n)$。而且一塊連續空間滿了, 就得另開一塊更大的、把舊資料全搬過去(這就是第 4 章說的「動態陣列搬家」, 它的攤銷成本我們稍後在雜湊表那節會細談)。

鏈結串列:散居各處、用紙條互相指路

鏈結串列(linked list)採取完全相反的策略:元素不必連續, 愛住哪住哪。每個元素被包成一個節點(node),節點裡除了資料, 還存了一張「紙條」——一個指向下一個節點的指標(pointer)。 於是這些散居各處的節點,靠著一條條紙條串成一列。你只要抓住第一個節點(稱為頭 head), 就能順著紙條一路走到尾。

這種設計的好處,正好補上陣列的弱點:要在串列中間插入一個節點, 不必搬動任何資料——只要改兩條紙條(把前一個節點指向新節點、新節點指向後一個),就是 $O(1)$。 刪除也一樣,把前一個節點的紙條繞過要刪的那個就好。代價也正好相反:鏈結串列沒有索引, 要找第 i 個元素,你只能從頭順著紙條一個一個數過去,所以隨機存取是 $O(n)$。

用 C 看見節點與指標

鏈結串列是理解指標的最佳教材。在 C 裡,一個節點就是一個 struct:裝資料, 外加一個指向「同類型 struct」的指標。每個新節點都用 malloc 跟作業系統要一塊記憶體 (這正是第 24 章講的動態配置),用完要 free 還回去。 

#include <stdio.h>
#include <stdlib.h>

/* 一個節點:存一個 int,外加指向「下一個節點」的指標 */
typedef struct Node {
    int data;
    struct Node *next;     /* 指向同型別的下一個節點 */
} Node;

/* 在串列頭部插入一個新值,回傳新的 head */
Node *push_front(Node *head, int value) {
    Node *n = malloc(sizeof(Node));   /* 跟系統要一塊記憶體 */
    n->data = value;
    n->next = head;                    /* 新節點指向原本的頭 */
    return n;                         /* 新節點變成新的頭 */
}

/* 順著指標走訪整串 */
void print_list(Node *head) {
    for (Node *p = head; p != NULL; p = p->next) {
        printf("%d -> ", p->data);
    }
    printf("NULL\n");
}

int main(void) {
    Node *head = NULL;          /* 空串列就是「頭指向 NULL」 */
    head = push_front(head, 3); /* 3 -> NULL */
    head = push_front(head, 2); /* 2 -> 3 -> NULL */
    head = push_front(head, 1); /* 1 -> 2 -> 3 -> NULL */
    print_list(head);           /* 1 -> 2 -> 3 -> NULL */

    /* 用完記得把每個節點 free 掉,否則記憶體洩漏 */
    while (head != NULL) {
        Node *tmp = head;
        head = head->next;
        free(tmp);
    }
    return 0;
}

看 n->next = head 這一行——這就是鏈結串列的靈魂。我們沒有搬動任何既有資料, 只是讓新節點的「紙條」指向舊的頭,$O(1)$ 完成插入。p->data 這個箭頭寫法, 是 C 裡「透過指標存取結構成員」的語法(等同於 (*p).data)。

用 Python 也實作一個

Python 沒有「裸指標」,但物件參考(reference)本質上就扮演了指標的角色: 一個物件的屬性指向「另一個物件」,串起來就是鏈結串列。 

class Node:
    def __init__(self, data, next=None):
        self.data = data
        self.next = next          # 指向下一個 Node(或 None 表示結尾)

class LinkedList:
    def __init__(self):
        self.head = None

    def push_front(self, value):          # 頭部插入,O(1)
        self.head = Node(value, self.head)

    def __iter__(self):                   # 讓它可以被 for 走訪
        p = self.head
        while p is not None:
            yield p.data
            p = p.next

ll = LinkedList()
for x in [3, 2, 1]:
    ll.push_front(x)
print(list(ll))        # [1, 2, 3]

雙向鏈結串列:紙條兩面都寫

上面是單向(singly)鏈結串列:只能往後走。如果每個節點再多存一個指向 前一個節點的指標,就成了雙向(doubly)鏈結串列——可以前後雙向走, 也讓「已知某節點、要刪掉它」變得很方便(因為你能直接找到它的前一個)。 代價是每個節點多花一個指標的空間。稍後我們實作 LRU 快取時,雙向鏈結串列會是主角。

ℹ️
那……到底什麼時候該用鏈結串列?

說句實話:在日常 Python 程式裡,你幾乎不會手寫鏈結串列——內建的 list 已經夠好。鏈結串列真正的價值,一是當觀念基石:堆疊、佇列、雜湊表的鏈結法、 樹、圖,全都是「節點+指標」的變奏。二是某些特定場景:需要大量在頭尾或中間插刪、 且不需要隨機索引時(例如下面的 deque、LRU 快取)。學它,是為了看懂後面一切。

堆疊與佇列:一對 LIFO/FIFO 雙胞胎

有了「排成一列」的能力,我們可以加上「規矩」,變出兩個極其常用的 ADT。 它們的差別只有一個:從哪一端進、從哪一端出。

堆疊(stack):後進先出(LIFO)

堆疊(stack)像一疊洗好的盤子:你只能往最上面放(push), 也只能從最上面拿(pop)。最後放上去的,最先被拿走—— 這叫後進先出(Last In, First Out, LIFO)。

佇列(queue):先進先出(FIFO)

佇列(queue)像排隊買飲料:從尾端排進去(enqueue), 從前端被服務(dequeue)。先來的先走—— 這叫先進先出(First In, First Out, FIFO),公平得很。

用 Python 實作:陣列版與鏈結版

堆疊用 Python 的 list 實作再自然不過——append 當 push,pop() 當 pop, 兩者都是尾端操作,都是 $O(1)$: 

class Stack:
    def __init__(self):
        self._data = []
    def push(self, x):  self._data.append(x)      # O(1)
    def pop(self):      return self._data.pop()   # O(1),拿最後一個
    def peek(self):     return self._data[-1]     # 看頂端但不拿走
    def is_empty(self): return len(self._data) == 0

s = Stack()
s.push(1); s.push(2); s.push(3)
print(s.pop())   # 3  ← 後進先出
print(s.pop())   # 2

佇列就要小心了。如果你用 list 的 pop(0) 來「從前端拿」,那是 $O(n)$—— 因為刪掉第一個元素後,後面全部要往前挪!正確做法是用標準庫的 collections.deque(底下是雙向結構,兩端進出都是 $O(1)$): 

from collections import deque

q = deque()
q.append("A"); q.append("B"); q.append("C")  # 從尾端 enqueue
print(q.popleft())   # A  ← 從前端 dequeue,O(1)
print(q.popleft())   # B

想看「鏈結版」的堆疊也很簡單:還記得上面 LinkedList.push_front 嗎? 頭部插入與頭部取出都是 $O(1)$——那其實就是一個堆疊。這正好印證了 ADT 的精神: 同一個「堆疊」介面,陣列和鏈結串列都能實作。

它們藏在哪裡?(你早就用過了)

  • 括號匹配:檢查 (){}[] 有沒有正確配對,是堆疊的招牌應用—— 遇到左括號就 push,遇到右括號就 pop 來核對。這是本章練習 1,等下你會親手寫。
  • 呼叫堆疊(call stack):還記得第 3 章的遞迴嗎? 每呼叫一個函式,系統就 push 一個「堆疊框(stack frame)」記住「回來後從哪繼續」; 函式返回就 pop。遞迴太深會「stack overflow」,正是這個堆疊被塞爆了。
  • 計算機 / 虛擬機:第 23 章的堆疊式虛擬機, 靠一個運算元堆疊來算式子——push 3; push 4; add 就把 3、4 推上去再相加。
  • 佇列 → BFS:廣度優先搜尋(breadth-first search)用佇列「一圈一圈」探索圖, 先進先出保證了「離起點越近的越早被處理」。我們在第 30 章會大用特用。
  • 佇列 → 排程與緩衝:作業系統的工作排程、印表機的列印佇列、訊息系統, 本質都是 FIFO 佇列。

雙端佇列(deque,double-ended queue):兩端都能進出的佇列, 兼具堆疊與佇列的能力(Python 的 collections.deque 就是)。 環形緩衝區(ring buffer):用一個固定大小的陣列首尾相接成「環」來實作佇列, 常見於串流、音訊處理等需要固定記憶體、持續吞吐資料的場景。

雜湊表:平均 $O(1)$ 的魔法(重頭戲)

現在來拆解整章最重要、也最迷人的結構。第 4 章你已經見過它的招牌效果: Python 的 dict 不管裝多少資料,用鍵查值都幾乎是瞬間($O(1)$)。 這背後的結構叫雜湊表(hash table)。它怎麼辦到的?

核心點子:讓「鍵」自己算出「住址」

想像你有一排編號 0 到 7 的信箱(一個陣列)。問題是:給你一個鍵 (例如字串 "小美"),該把它的值放進哪一格? 雜湊表的答案是:讓鍵自己算出來。我們用一個雜湊函數(hash function) 把任意鍵換算成一個整數,再對信箱數量取餘數,得到一個 0~7 的格號(稱為桶 bucket)。 

# 一個玩具雜湊函數:把字串每個字元的編碼加起來
def my_hash(key, num_buckets):
    total = 0
    for ch in str(key):
        total += ord(ch)          # ord() 取字元的數字編碼
    return total % num_buckets    # 取餘數,壓進 0 ~ num_buckets-1

print(my_hash("小美", 8))   # 例如 → 3,就放進第 3 格

妙處在於:存的時候算出第 3 格放進去,查的時候同一個鍵算出同一個第 3 格, 直接跳過去拿——不必逐一比對。一個好的雜湊函數要滿足兩個要求: 同樣的鍵永遠算出同樣的值(確定性),而且把不同的鍵盡量均勻打散到各桶, 避免大家擠在同一格。

碰撞(collision):兩個鍵算到同一格

但這裡有個無法迴避的問題。鍵的種類是無限的(任意字串),信箱卻只有有限格。 鴿籠原理(第 14 章)告訴我們: 遲早會有兩個不同的鍵,算出同一個格號。這就叫碰撞(collision)。 碰撞不是 bug,是必然——雜湊表的設計,有一大半就是在處理碰撞。有兩大流派:

解法一:鏈結法(chaining)

每一格不直接放值,而是掛一條鏈結串列(看,鏈結串列又出現了)。 所有算到同一格的鍵值對,就串在那一格的鏈上。查找時先算出格號,再在那條(通常很短的)鏈上線性搜尋。

解法二:開放定址(open addressing)

另一派完全不用鏈結串列,堅持「一格只放一個」。當算出的格子已被占用, 就照某個規則往後找下一個空格(最簡單的規則叫「線性探測 linear probing」: 滿了就看下一格,還滿就再下一格)。查找時也照同樣規則走,直到找到鍵或撞上空格。 好處是全部資料都待在同一塊連續陣列裡,對 CPU 快取友善;壞處是容易「結塊」, 刪除也比較麻煩(不能直接清空,否則會切斷探測路徑)。

負載因子與動態擴容:呼應攤銷分析

不管哪種解法,有一個關鍵指標決定雜湊表的死活:負載因子(load factor) $\alpha = \dfrac{\text{元素個數}}{\text{桶數}}$。它衡量「擠不擠」。負載因子越高,碰撞越多、 鏈越長,查找就從 $O(1)$ 慢慢退化。所以雜湊表會盯著它:一旦超過門檻(常見是 0.7 左右), 就擴容(resize)——開一個約兩倍大的新桶陣列,把所有元素重新雜湊(rehash) 搬過去。

「搬一整批」聽起來是 $O(n)$ 的大工程,那怎麼還敢說插入是 $O(1)$? 關鍵在攤銷分析(amortized analysis)(第 26 章): 擴容雖貴,但每次都把容量翻倍,所以擴容發生的頻率越來越低。 把這偶爾一次的大成本,平均分攤到中間那許多次便宜的插入上, 每次插入的平均成本仍是 $O(1)$。這和第 4 章說的「動態陣列搬家」是同一個道理—— 因為動態陣列本來就是雜湊表的近親。

🔑
平均 $O(1)$,最壞 $O(n)$

雜湊表的查找/插入/刪除,平均是 $O(1)$——這是它最大的賣點。 但最壞情況是 $O(n)$:如果雜湊函數很爛、或被惡意設計的輸入攻擊, 讓所有鍵全擠進同一個桶,那條鏈會長到 $n$,查找就退化成線性搜尋。 現實中,只要雜湊函數夠好、負載因子控制得當,最壞情況極少發生——所以我們才能放心地把它當 $O(1)$ 用。

從零實作一個簡單的雜湊表(鏈結法)

理論講完,動手寫一個。我們用「桶陣列 + 每桶一條 list(扮演鏈)」實作鏈結法, 並在負載因子過高時自動擴容。看完這段,你就真正理解了 dict 的靈魂: 

class HashMap:
    def __init__(self, num_buckets=8):
        self.buckets = [[] for _ in range(num_buckets)]  # 每桶一條鏈(用 list)
        self.size = 0                                    # 目前有幾組鍵值對

    def _index(self, key):
        return hash(key) % len(self.buckets)   # 用內建 hash() 算桶號

    def put(self, key, value):
        # 負載因子超過 0.7 就擴容成兩倍
        if self.size / len(self.buckets) > 0.7:
            self._resize()
        bucket = self.buckets[self._index(key)]
        for i, (k, v) in enumerate(bucket):
            if k == key:                # 鍵已存在 → 更新
                bucket[i] = (key, value)
                return
        bucket.append((key, value))     # 新鍵 → 加到鏈尾
        self.size += 1

    def get(self, key):
        bucket = self.buckets[self._index(key)]
        for k, v in bucket:             # 在那條(短)鏈上線性找
            if k == key:
                return v
        raise KeyError(key)

    def _resize(self):
        old = self.buckets
        self.buckets = [[] for _ in range(len(old) * 2)]  # 桶數翻倍
        self.size = 0
        for bucket in old:              # 把舊資料全部重新雜湊搬過去
            for k, v in bucket:
                self.put(k, v)

m = HashMap()
m.put("小明", 18)
m.put("小美", 17)
m.put("小明", 21)          # 更新,不是新增
print(m.get("小美"))       # 17
print(m.get("小明"))       # 21
print(m.size)              # 2

樹與二元搜尋樹:讓搜尋變聰明

到目前為止,我們的結構都是「線性」的——一條線串下去。但很多資料天生是階層的: 家族族譜、公司組織、檔案系統的資料夾、HTML 的標籤巢狀。描述這種「一對多」的分支關係, 要用樹(tree)。

樹的術語:先把名字搞清楚

樹是上下顛倒長的——根在最上面,葉在最下面。幾個必備詞彙:

  • 節點(node):樹上的每個資料點。
  • 根(root):最上面、唯一沒有父節點的那個。
  • 邊(edge):連接父子節點的線。
  • 父(parent)/ 子(child):直接相連的上下關係。
  • 葉(leaf):沒有任何子節點的節點(樹的末梢)。
  • 深度(depth):從根走到某節點要經過幾條邊(根的深度是 0)。
  • 高度(height):從某節點走到最深葉子的邊數;整棵樹的高度就是根的高度。

其中最常見的是二元樹(binary tree):每個節點最多兩個子節點,稱為左子與右子。

二元搜尋樹(BST):有序的二元樹

二元搜尋樹(binary search tree, BST)在二元樹上加了一條黃金規則, 讓它變成搜尋利器:

🔑
BST 性質

對每一個節點:它左子樹裡的所有值,都比它小; 它右子樹裡的所有值,都比它大。這條規則遞迴地對整棵樹的每個節點都成立。

這條規則為什麼威力強大?因為搜尋變成了「猜數字」遊戲。要找一個值, 從根開始:比根小就往左、比根大就往右,每一步都砍掉一半的可能。 這正是二分搜尋的精神,落實在樹結構上。 如果樹是「平衡」的,高度約 $\log n$,所以搜尋、插入都是平均 $O(\log n)$。

插入與搜尋

兩者邏輯幾乎一樣:都是「比大小、往左或往右走」。搜尋是走到找到或走到空;插入是走到空,然後把新節點掛上去。

class TreeNode:
    def __init__(self, val):
        self.val = val
        self.left = None
        self.right = None

class BST:
    def __init__(self):
        self.root = None

    def insert(self, val):
        self.root = self._insert(self.root, val)

    def _insert(self, node, val):
        if node is None:               # 走到空位 → 在這裡放新節點
            return TreeNode(val)
        if val < node.val:
            node.left = self._insert(node.left, val)    # 比較小,往左
        elif val > node.val:
            node.right = self._insert(node.right, val)  # 比較大,往右
        return node                    # 相等就忽略(BST 通常不放重複值)

    def search(self, val):
        node = self.root
        while node is not None:
            if val == node.val:
                return True
            node = node.left if val < node.val else node.right
        return False

bst = BST()
for x in [8, 3, 10, 1, 6, 14, 4]:
    bst.insert(x)
print(bst.search(6))   # True
print(bst.search(7))   # False

四種走訪(traversal)

「走訪」是指「把樹上每個節點都拜訪一次」。二元樹有四種經典走訪順序, 前三種是深度優先(一條路走到底再回頭),差別只在「什麼時候處理自己」:

  • 前序(pre-order):先處理自己 → 左子樹 → 右子樹。(常用來「複製」一棵樹)
  • 中序(in-order):左子樹 → 自己 → 右子樹。
  • 後序(post-order):左子樹 → 右子樹 → 自己。(常用來「釋放/刪除」整棵樹)
  • 層序(level-order):一層一層、由上而下、由左而右——這是廣度優先, 用佇列實作(看,佇列又回來了)。

其中中序走訪有個漂亮的性質:對 BST 做中序走訪,得到的會是由小到大排序好的序列! 因為「左邊都比我小、右邊都比我大」,所以「先看完左、再看自己、再看右」恰好就是升冪。 

def inorder(node, out):
    if node is None:
        return
    inorder(node.left, out)    # 1. 先走完左子樹
    out.append(node.val)       # 2. 再處理自己
    inorder(node.right, out)   # 3. 最後走右子樹

result = []
inorder(bst.root, result)
print(result)   # [1, 3, 4, 6, 8, 10, 14]  ← 自動排序好了!

# 層序走訪:用佇列
from collections import deque
def level_order(root):
    out, q = [], deque([root] if root else [])
    while q:
        node = q.popleft()
        out.append(node.val)
        if node.left:  q.append(node.left)
        if node.right: q.append(node.right)
    return out

print(level_order(bst.root))   # [8, 3, 10, 1, 6, 14, 4]  ← 一層一層

刪除:三種情況

刪除是 BST 最棘手的操作,因為刪掉一個節點不能破壞「左小右大」的規則。要分三種情況:

  1. 葉節點(沒有小孩):直接拿掉,最簡單。
  2. 只有一個小孩:用那個小孩取代自己(像鏈結串列繞過一個節點)。
  3. 有兩個小孩:最麻煩。找到「右子樹裡最小的節點」(稱為中序後繼, 它是右子樹裡最左下角那個),用它的值取代自己,再去右子樹把那個後繼刪掉。 為什麼選它?因為它剛好是「比我大的數裡最小的」,放到我的位置,左小右大依然成立。
def delete(node, val):
    if node is None:
        return None
    if val < node.val:
        node.left = delete(node.left, val)
    elif val > node.val:
        node.right = delete(node.right, val)
    else:                                   # 找到要刪的節點了
        if node.left is None:               # 情況 1、2:沒有左小孩
            return node.right
        if node.right is None:              # 沒有右小孩
            return node.left
        succ = node.right                   # 情況 3:找右子樹的最小節點
        while succ.left is not None:
            succ = succ.left
        node.val = succ.val                 # 用後繼的值覆蓋自己
        node.right = delete(node.right, succ.val)  # 再刪掉那個後繼
    return node
⚠️
BST 會「退化」成一條鏈

BST 的 $O(\log n)$ 有個大前提:樹要夠平衡(矮胖)。但如果你照順序 插入已排序的資料(例如依序插入 1, 2, 3, 4, 5),每個新值都比前一個大, 會全部往右掛——樹變成一條只往右的「斜線」,高度變成 $n$, 搜尋退化成 $O(n)$,跟鏈結串列沒兩樣!這是 BST 的致命弱點, 也正是下一節「平衡樹」要解決的問題。

平衡樹:用「旋轉」保證永遠矮胖

既然 BST 一退化就完蛋,聰明人想出了自平衡二元搜尋樹(self-balancing BST): 在每次插入/刪除後,自動微調樹的形狀,保證它永遠不會長得太歪, 高度始終維持在 $O(\log n)$。微調的核心動作,叫做旋轉(rotation)。

旋轉:不破壞 BST 性質的局部重整

旋轉是一個神奇的局部操作:它改變幾個節點的父子關係、把樹「轉正」一點, 但完全不破壞「左小右大」的 BST 性質,也不改變中序走訪的順序。 看一次「右旋」就懂了:

AVL 樹與紅黑樹:兩種平衡策略

怎麼判斷「該不該轉、怎麼轉」?兩種最著名的自平衡樹,給了兩種不同的紀律:

  • AVL 樹:最嚴格。它要求每個節點的「左右子樹高度差不超過 1」。 每次插刪後檢查,一旦違規就立刻旋轉修正。因為控制得緊,AVL 樹非常矮,查找最快; 代價是插刪時旋轉較頻繁。
  • 紅黑樹(red-black tree):較寬鬆。它給每個節點塗上紅或黑, 並用一組著色規則(例如「紅節點的小孩必須是黑」「從根到每個葉的路徑上黑節點數相同」) 間接保證「最長路徑不超過最短路徑的兩倍」。它不追求完美平衡,所以旋轉較少、插刪較快, 是工程上最常用的選擇——C++ 的 std::map、Java 的 TreeMap、 Linux 核心的排程器,底下都是紅黑樹。
🔑
為什麼平衡樹能保證 $O(\log n)$?

不論 AVL 還是紅黑樹,它們的著色/高度規則,數學上都強制樹高維持在 $O(\log n)$。 而 BST 的搜尋、插入、刪除,成本都正比於樹高(因為每步往下一層)。 樹高被鎖死在 $\log n$,這些操作自然就保證是 $O(\log n)$——連最壞情況也是, 再也不會退化成鏈。這就是平衡樹相對於普通 BST 的關鍵勝利。

完整的 AVL/紅黑樹實作(處理所有旋轉與著色情況)相當繁瑣,初學階段不必硬背。 你只要牢牢記住兩件事:(1) 普通 BST 會退化;(2) 平衡樹用旋轉把樹高鎖在 $O(\log n)$, 因此所有操作的最壞情況都是 $O(\log n)$。這個「為什麼」遠比背旋轉細節重要。

ℹ️
一句話帶過 B 樹

B 樹(B-tree)是「每個節點可以有很多個小孩」的平衡樹,專為 硬碟與資料庫設計。當資料大到放不進記憶體、必須存在磁碟上時, 「讀一次磁碟」非常昂貴,所以 B 樹刻意把每個節點做得又寬又胖(一次讀一大塊), 讓樹變得極矮,大幅減少磁碟讀取次數。幾乎所有資料庫的索引、以及許多檔案系統, 底層都是 B 樹或其變體 B+ 樹——這會在第八部「資料庫」 與檔案系統(呼應第 24 章)再相見。

堆積與優先佇列:永遠先給我「最緊急的那個」

前面的佇列是「公平」的(先進先出)。但很多時候我們要的不是公平,而是緊急優先: 醫院急診室先看最危急的病人、作業系統先跑最高優先級的任務、 導航演算法每次取「目前最短的路徑」。這需要一個新 ADT:優先佇列(priority queue)—— 你不斷丟東西進去,但每次取出時,它永遠給你目前優先級最高(或值最小)的那一個。

怎麼高效實作?用排序好的陣列?那插入要 $O(n)$。用普通 BST?會退化。 最漂亮的答案是二元堆積(binary heap)。

二元堆積:一棵「半排序」的完全二元樹

堆積是一棵特別的二元樹,滿足兩個條件:

  • 堆積性質(heap property):以最小堆積(min-heap)為例, 每個節點的值都不大於它的兩個小孩。於是最小值永遠在根。 (注意:這比 BST 寬鬆——它只管「父子」關係,不管左右誰大誰小,所以堆積只是「半排序」。)
  • 完全二元樹(complete binary tree):除了最後一層,每一層都填滿, 且最後一層從左往右填。這個整齊的形狀,讓我們可以用一個陣列就裝下整棵樹,不需要任何指標!

上浮與下沉:維持堆積性質的兩個動作

插入和取出時,堆積性質可能被破壞,我們用兩個對稱的修復動作把它救回來:

  • 上浮(sift up):插入時用。新元素先放到陣列最尾端 (樹的最後一個位置),然後不斷和「父節點」比較:如果比父親小(在 min-heap),就交換、往上爬, 直到不再比父親小或爬到根。因為樹高 $\log n$,最多爬 $\log n$ 層,所以插入是 $O(\log n)$。
  • 下沉(sift down):取出最小值時用。最小值在根,取走它後, 把最後一個元素搬到根填空,然後不斷和「較小的那個小孩」比較:如果比小孩大,就交換、往下沉, 直到比兩個小孩都小或沉到底。同樣是 $O(\log n)$。

用 Python 從零實作一個 min-heap

class MinHeap:
    def __init__(self):
        self.a = []

    def push(self, x):
        self.a.append(x)            # 1. 放到尾端
        self._sift_up(len(self.a) - 1)

    def pop_min(self):
        a = self.a
        a[0], a[-1] = a[-1], a[0]   # 把根和最後一個交換
        smallest = a.pop()          # 拿走原本的根(最小值)
        if a:
            self._sift_down(0)
        return smallest

    def _sift_up(self, i):
        while i > 0:
            parent = (i - 1) // 2
            if self.a[i] < self.a[parent]:    # 比父親小 → 往上換
                self.a[i], self.a[parent] = self.a[parent], self.a[i]
                i = parent
            else:
                break

    def _sift_down(self, i):
        n = len(self.a)
        while True:
            left, right, smallest = 2 * i + 1, 2 * i + 2, i
            if left < n and self.a[left] < self.a[smallest]:
                smallest = left
            if right < n and self.a[right] < self.a[smallest]:
                smallest = right
            if smallest == i:        # 已經比兩個小孩都小 → 停
                break
            self.a[i], self.a[smallest] = self.a[smallest], self.a[i]
            i = smallest

h = MinHeap()
for x in [5, 3, 8, 1, 9, 4]:
    h.push(x)
print(h.pop_min())   # 1  ← 永遠先吐最小的
print(h.pop_min())   # 3

還有一個技巧叫 heapify:給你一個亂序陣列,要把它整個變成合法堆積。 天真做法是逐一 push,共 $O(n \log n)$;但有個聰明做法是「從最後一個非葉節點開始, 由下往上對每個節點做 sift down」,數學上可證明總成本只要 $O(n)$! Python 的 heapq.heapify() 就是這個。

heapsort:用堆積來排序

堆積給了我們一個免費的排序法:把全部元素 heapify 成堆積($O(n)$), 再一個一個 pop 出來($n$ 次,每次 $O(\log n)$),取出的順序自然就是排序好的。 總成本 $O(n \log n)$,而且不需要額外空間(可以原地做)。這就是堆積排序(heapsort)。

實務:直接用 heapq

自己寫過一次理解原理後,實務上請直接用標準庫的 heapq(它把一個普通 list 當成 min-heap 操作):

優先佇列藏在哪?

優先佇列是許多重要演算法的引擎。Dijkstra 最短路徑每次都要取「目前距離最小的節點」, 靠的就是優先佇列(我們第 30 章會用到); 作業系統的排程(呼應第 24 章)用它來挑下一個該跑的任務; 資料壓縮的 Huffman 編碼、事件模擬、合併 K 個排序串列……到處都是它的身影。 記住:「每次都要拿最大/最小的那一個」= 該用堆積。

並查集:近乎瞬間的「他們是同一夥嗎?」

最後一個主角,解決一類特別的問題:動態的分組與連通性。 想像社群網路上有一群人,你不斷收到「A 和 B 是朋友」這樣的訊息(合併兩群), 並隨時要回答「X 和 Y 屬於同一個朋友圈嗎?」(查詢)。 專門對付這種需求的結構,叫並查集(union-find,或 disjoint set,不相交集合)。

它支援兩個操作,名字就說明了一切:

  • find(x):找出 x 屬於哪一群(回傳該群的「代表元」)。
  • union(x, y):把 x 和 y 所在的兩群合併成一群。

核心點子:每群是一棵樹,根當代表

並查集把每一群表示成一棵樹,樹根就是這群的「代表」。判斷兩個元素是否同群, 只要看它們順著父指標往上爬到的根是不是同一個。 合併兩群,就把一棵樹的根掛到另一棵樹的根底下。我們只需要一個陣列 parent[], 記住「每個元素的父親是誰」(根的父親就是它自己)。

兩個讓它快到飛起的優化

樸素版的 find 要往上爬,樹一歪就慢。兩個經典優化讓它快到近乎常數:

  • 按秩合併(union by rank/size):合併時,永遠把比較矮(或比較小)的樹 掛到比較高的樹底下,避免樹越長越高。
  • 路徑壓縮(path compression):做 find 往上爬時,順手把沿路每個節點直接接到根上, 下次再 find 就一步到位。

兩招合用,均攤下來,每次操作的成本是 $O(\alpha(n))$,其中 $\alpha$ 是 反阿克曼函數(inverse Ackermann function)——它增長得慢到變態: 對任何你在宇宙裡寫得出來的 $n$,$\alpha(n)$ 都不超過 4。所以實務上,你可以放心把它當常數時間。 

class UnionFind:
    def __init__(self, n):
        self.parent = list(range(n))   # 一開始每個元素自成一群(父親是自己)
        self.rank = [0] * n            # 樹的概略高度

    def find(self, x):
        # 路徑壓縮:把沿路節點直接接到根
        if self.parent[x] != x:
            self.parent[x] = self.find(self.parent[x])
        return self.parent[x]

    def union(self, x, y):
        rx, ry = self.find(x), self.find(y)
        if rx == ry:
            return False               # 本來就同群,不需合併
        if self.rank[rx] < self.rank[ry]:   # 按秩合併:矮的掛到高的下面
            rx, ry = ry, rx
        self.parent[ry] = rx
        if self.rank[rx] == self.rank[ry]:
            self.rank[rx] += 1
        return True

    def connected(self, x, y):
        return self.find(x) == self.find(y)

uf = UnionFind(5)
uf.union(0, 1)
uf.union(2, 3)
print(uf.connected(0, 1))   # True
print(uf.connected(0, 3))   # False
uf.union(1, 3)              # 把兩群連起來
print(uf.connected(0, 3))   # True  ← 現在連通了
ℹ️
並查集藏在哪?

最經典的應用是 Kruskal 最小生成樹(MST)演算法 (第 31 章):它由小到大挑邊, 用並查集判斷「這條邊會不會造成環」(兩端是否已連通),不會才加入。 其他像「網路節點連通性」「圖像中的連通區域」「滲流模型」等,都是它的主場。

還有兩個值得認識的名字

資料結構的世界很大,這裡用一句話各介紹兩個你日後會遇到的:

  • 字典樹(trie,或前綴樹 prefix tree):一棵專門存字串的樹, 每條從根往下的路徑拼出一個字串、共享的前綴只存一份。它讓「以某前綴開頭的單字有哪些」 變得極快——搜尋框的自動補全、輸入法的選字背後常是它。
  • 布隆過濾器(Bloom filter):一個極省空間的機率型結構, 用來快速回答「這個元素可能在集合裡嗎?」。它可能誤報 (說在、其實不在),但絕不漏報(說不在、就一定不在)。 瀏覽器用它快速排除「絕對安全」的網址、資料庫用它避免無謂的磁碟查詢。

✏️ 練習 29.1 實作

用堆疊寫一個函式 is_balanced(s),判斷一個只含 ()[]{} 的字串裡,括號是否正確配對且巢狀正確。 例如 "([]{})" → True,"([)]" → False, "((" → False。

看解答

核心直覺:遇到左括號就 push;遇到右括號,就 pop 出最近一個左括號, 看它們是不是「同一對」。最後堆疊必須剛好清空(沒有落單的左括號)。

def is_balanced(s):
    pairs = {')': '(', ']': '[', '}': '{'}   # 右括號 → 對應的左括號
    stack = []
    for ch in s:
        if ch in '([{':
            stack.append(ch)                 # 左括號,推進去
        elif ch in pairs:
            # 右括號:堆疊不能空,且頂端必須是對應的左括號
            if not stack or stack.pop() != pairs[ch]:
                return False
    return len(stack) == 0                    # 全部配對完,堆疊應為空

print(is_balanced("([]{})"))   # True
print(is_balanced("([)]"))     # False  ← 交錯,pop 出的不對
print(is_balanced("(("))       # False  ← 結束時堆疊沒清空

這就是堆疊「後進先出」最自然的應用:最晚開的括號,必須最早關。 編譯器檢查你的程式碼括號、HTML 檢查標籤巢狀,用的都是這招。

✏️ 練習 29.2 實作

在本章 HashMap 的基礎上,補上兩個方法:get(key, default) ——查不到時回傳預設值(而非報錯);以及 remove(key)——刪除一個鍵值對。 請正確處理「碰撞鏈上有多個元素」的情況。

看解答

關鍵在於:刪除時要在那條鏈(bucket)上找到正確的那一對再移除,別誤刪同桶的鄰居。

class HashMap:
    def __init__(self, num_buckets=8):
        self.buckets = [[] for _ in range(num_buckets)]
        self.size = 0

    def _index(self, key):
        return hash(key) % len(self.buckets)

    def put(self, key, value):
        bucket = self.buckets[self._index(key)]
        for i, (k, v) in enumerate(bucket):
            if k == key:
                bucket[i] = (key, value)
                return
        bucket.append((key, value))
        self.size += 1

    def get(self, key, default=None):
        bucket = self.buckets[self._index(key)]
        for k, v in bucket:
            if k == key:
                return v
        return default                # 查不到 → 回傳預設值,不報錯

    def remove(self, key):
        bucket = self.buckets[self._index(key)]
        for i, (k, v) in enumerate(bucket):
            if k == key:
                bucket.pop(i)         # 只刪這一對,同桶的其他鍵不受影響
                self.size -= 1
                return True
        return False                  # 本來就沒有這個鍵

m = HashMap()
m.put("a", 1); m.put("b", 2)
print(m.get("a"))          # 1
print(m.get("z", -1))      # -1  ← 查不到,回傳預設
print(m.remove("a"))       # True
print(m.get("a", "沒了"))  # 沒了

體會一下:因為碰撞讓多個鍵可能擠在同一個 bucket,所有操作都必須「先定位到桶, 再在桶內逐一比對」。這正是雜湊表「平均 $O(1)$、最壞 $O(n)$」的根源。

✏️ 練習 29.3 實作

實作 BST 的 insert 與中序走訪,並驗證:依序插入 [5, 2, 8, 1, 3, 7, 9] 後,中序走訪會吐出排序好的 [1, 2, 3, 5, 7, 8, 9]。

看解答

插入靠遞迴「比大小、往左或往右」;中序走訪「左 → 自己 → 右」恰好給出升冪。

class TreeNode:
    def __init__(self, val):
        self.val, self.left, self.right = val, None, None

def insert(root, val):
    if root is None:
        return TreeNode(val)
    if val < root.val:
        root.left = insert(root.left, val)
    else:
        root.right = insert(root.right, val)
    return root

def inorder(root):
    if root is None:
        return []
    return inorder(root.left) + [root.val] + inorder(root.right)

root = None
for x in [5, 2, 8, 1, 3, 7, 9]:
    root = insert(root, x)
print(inorder(root))   # [1, 2, 3, 5, 7, 8, 9]

為什麼中序會排序?因為 BST 保證「左子樹全比我小、右子樹全比我大」, 所以「先看完全部比我小的(左)、再看我、再看全部比我大的(右)」自然就是由小到大。 這也給了你一個排序法:把資料全插進 BST 再中序走訪。

✏️ 練習 29.4 實作

用 heapq 寫一個函式 top_k(nums, k), 回傳串列 nums 裡最大的 k 個元素(順序不拘)。 要求:用一個大小為 k 的最小堆積來做,不要直接排序整個串列。 (這在資料量極大、k 很小時特別有用。)

看解答

訣竅很巧妙:維護一個只有 k 個元素的 min-heap。堆裡永遠是「目前看到的最大 k 個」, 而堆頂(最小值)就是這 k 個裡的門檻。新元素若比門檻大,就把門檻擠掉、換它進來。

import heapq

def top_k(nums, k):
    heap = []
    for x in nums:
        if len(heap) < k:
            heapq.heappush(heap, x)          # 還沒滿,先放進去
        elif x > heap[0]:                    # 比堆裡最小的還大
            heapq.heapreplace(heap, x)       # 踢掉最小的,換 x 進來
    return heap

print(top_k([5, 1, 9, 3, 7, 2, 8], 3))   # [7, 8, 9](順序不保證)

複雜度是 $O(n \log k)$:掃過 $n$ 個元素,每次堆操作 $O(\log k)$。 當 $k \ll n$ 時,這比「全排序」的 $O(n \log n)$ 快得多,也只用 $O(k)$ 記憶體。 當然,若不在意效率,heapq.nlargest(k, nums) 一行就解決。

✏️ 練習 29.5 實作

用並查集解這題:有 n 個城市(編號 0 ~ n-1), 以及一串「道路」edges(每條是 (a, b) 表示 a、b 直接相連)。 寫一個函式 count_components(n, edges),回傳這張圖被分成幾個「連通分量」 (互相到得了的城市算同一群)。例如 count_components(5, [(0,1),(1,2),(3,4)]) → 2。

看解答

把每條邊都做一次 union,最後數一數「有幾個不同的根」,就是連通分量的數目。 這裡直接用本章的 UnionFind:

def count_components(n, edges):
    uf = UnionFind(n)
    for a, b in edges:
        uf.union(a, b)               # 把每條邊的兩端合併成同一群
    # 每群恰有一個根(parent[x] == x 的那個)
    roots = {uf.find(i) for i in range(n)}
    return len(roots)

print(count_components(5, [(0, 1), (1, 2), (3, 4)]))   # 2
# 群 1:{0,1,2}  群 2:{3,4}

print(count_components(4, []))                          # 4(每個城市自成一群)

一開始每個城市自成一群(n 群),每成功合併一次,群數就減一。 最後剩幾群,就是連通分量數。並查集讓「動態合併 + 隨時查連通」幾乎是常數時間, 這正是它無可取代的強項。

✏️ 練習 29.6 實作

把一個單向鏈結串列反轉。給你 head,回傳反轉後的新 head。 例如 1 → 2 → 3 → None 變成 3 → 2 → 1 → None。 請原地反轉(只改指標,不另建節點),目標 $O(n)$ 時間、$O(1)$ 額外空間。

看解答

這是經典面試題。訣竅是邊走邊「把每個節點的箭頭反過來指向前一個」。 用三個指標:prev(前一個)、cur(當前)、暫存 cur.next(下一個,免得弄丟)。

class Node:
    def __init__(self, data, next=None):
        self.data, self.next = data, next

def reverse(head):
    prev = None
    cur = head
    while cur is not None:
        nxt = cur.next      # 1. 先記住下一個(否則改了 cur.next 就找不到了)
        cur.next = prev     # 2. 把箭頭反過來,指向前一個
        prev = cur          # 3. prev 往前挪
        cur = nxt           # 4. cur 往前挪
    return prev             # 走到底時,prev 就是新的 head

# 建 1 → 2 → 3,反轉後印出
head = Node(1, Node(2, Node(3)))
head = reverse(head)
p = head
while p:
    print(p.data, end=" ")   # 3 2 1
    p = p.next

關鍵是第 1 步那個暫存 nxt:一旦你把 cur.next 改掉, 原本的下一個就斷線了,所以必須先存起來。畫個圖、用手指跟著三個指標走一遍, 這個技巧就會永遠刻進你腦裡。

✏️ 練習 29.7 挑戰

寫一個函式 is_valid_bst(root),判斷一棵二元樹是否為合法的 BST。 陷阱:只檢查「每個節點 > 左子、< 右子」是不夠的! 因為 BST 要求「整個左子樹都比我小、整個右子樹都比我大」, 不只是直接的兩個小孩。

看解答

正確做法是「邊往下走,邊收緊合法範圍」:每個節點都必須落在一個 (lower, upper) 開區間內。往左走時,上界收緊成「父節點的值」;往右走時,下界收緊成「父節點的值」。

def is_valid_bst(root, lower=float('-inf'), upper=float('inf')):
    if root is None:
        return True
    if not (lower < root.val < upper):     # 自己必須落在允許範圍內
        return False
    # 往左:值上限變成自己;往右:值下限變成自己
    return (is_valid_bst(root.left,  lower, root.val) and
            is_valid_bst(root.right, root.val, upper))

# 測試:這棵「看似」合法,其實不是 BST
#        5
#       / \
#      3   7
#         / \
#        4   8     ← 4 在 5 的右子樹,卻比 5 小!違法
root = TreeNode(5)
root.left = TreeNode(3)
root.right = TreeNode(7)
root.right.left = TreeNode(4)    # 致命的 4
root.right.right = TreeNode(8)
print(is_valid_bst(root))        # False ✓ 抓到了!

為什麼天真的「只比較父子」會出錯?上面的 4 確實比它的父親 7 小(看似合法), 但它在 5 的右子樹裡,卻比 5 小,違反了 BST。 「往下傳遞收緊的區間」優雅地解決了這個問題。 另一個等價解法:對它做中序走訪,檢查結果是否嚴格遞增—— 因為合法 BST 的中序一定是排序好的(練習 29.3 的逆用)。

✏️ 練習 29.8 挑戰

實作一個 LRU 快取(Least Recently Used cache):容量固定為 capacity,支援 get(key) 與 put(key, value), 兩者都要 $O(1)$。當容量滿了又要放新的,就淘汰最久沒被使用的那一個。 (提示:這正是「雜湊表 + 雙向鏈結串列」的招牌組合。)

看解答

這是本章的集大成題,把多個結構縫在一起。核心設計:

  • 用一條雙向鏈結串列維護「使用的新舊順序」:最近用過的放靠頭,最久沒用的在靠尾。
  • 用一個雜湊表 key → 節點,讓我們能 $O(1)$ 找到任一節點(否則在鏈上找是 $O(n)$)。
  • 每次存取,就把該節點移到頭(標記為最近使用);容量滿時,砍掉尾巴。雙向鏈結讓「移動」「刪除」都是 $O(1)$。
class Node:
    def __init__(self, key=0, val=0):
        self.key, self.val = key, val
        self.prev = self.next = None

class LRUCache:
    def __init__(self, capacity):
        self.cap = capacity
        self.map = {}                       # key → Node,O(1) 定位
        # 用兩個哨兵節點(head、tail)省掉一堆邊界判斷
        self.head, self.tail = Node(), Node()
        self.head.next, self.tail.prev = self.tail, self.head

    def _remove(self, node):                # 從鏈上摘掉一個節點,O(1)
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_front(self, node):             # 插到最前面(緊接 head),O(1)
        node.next = self.head.next
        node.prev = self.head
        self.head.next.prev = node
        self.head.next = node

    def get(self, key):
        if key not in self.map:
            return -1
        node = self.map[key]
        self._remove(node)                  # 用過了 → 移到最前面
        self._add_front(node)
        return node.val

    def put(self, key, value):
        if key in self.map:                 # 已存在 → 更新並移到最前
            self._remove(self.map[key])
        node = Node(key, value)
        self.map[key] = node
        self._add_front(node)
        if len(self.map) > self.cap:        # 超量 → 淘汰最久沒用的(尾巴)
            lru = self.tail.prev
            self._remove(lru)
            del self.map[lru.key]

cache = LRUCache(2)
cache.put(1, 1)
cache.put(2, 2)
print(cache.get(1))   # 1   (1 變成最近使用)
cache.put(3, 3)       # 容量滿,淘汰最久沒用的 key 2
print(cache.get(2))   # -1  (2 已被淘汰)
print(cache.get(3))   # 3

體會這個組合的精妙:雜湊表負責「$O(1)$ 找到節點」,雙向鏈結串列負責 「$O(1)$ 調整使用順序與淘汰」。兩者各補對方的弱點,合起來讓 get/put 都是 $O(1)$。 這也是 Python 標準庫 functools.lru_cache 與無數真實快取系統的核心思想—— 你剛剛親手重建了它。

把這一章串起來

這一章很長,因為我們真的把資料結構的引擎蓋掀開了。從第 4 章「會用容器」, 到這一章「會做、會選容器」,你跨過了一道讓很多人卡住的坎。回顧這趟旅程:

  • ADT vs 實作:先說「能做什麼」(介面),再說「怎麼做」(實作)。同一個 ADT 可有多種實作,這讓我們能自由替換更快的底層——這是抽象的勝利。
  • 陣列 vs 鏈結串列:連續記憶體(索引快、插刪慢)對上節點+指標(插刪快、無索引)。這是所有結構的兩塊基石。
  • 堆疊與佇列:LIFO 與 FIFO,藏在括號匹配、呼叫堆疊、VM、BFS、排程背後。
  • 雜湊表:雜湊定位 + 碰撞處理(鏈結法/開放定址)+ 負載因子觸發擴容,換來平均 $O(1)$、最壞 $O(n)$。dict 與 set 的真身。
  • BST 與走訪:左小右大讓搜尋變「猜數字」($O(\log n)$),但會退化成鏈($O(n)$);中序走訪給出排序序列。
  • 平衡樹:用旋轉把樹高鎖在 $O(\log n)$,連最壞情況都保證。AVL 嚴格、紅黑樹實用、B 樹為磁碟而生。
  • 堆積與優先佇列:用陣列表示的完全二元樹,上浮/下沉維持半排序,$O(\log n)$ 取極值。Dijkstra、排程、Top-K 的引擎。
  • 並查集:路徑壓縮 + 按秩合併,近乎 $O(\alpha(n))$ 的分組與連通查詢。Kruskal MST 的好搭檔。
  • 選擇:用鍵查值要快 → 雜湊表;要快又要排序 → 平衡樹;反覆取極值 → 堆積。沒有最好,只有最適合。

資料結構是地基,演算法是蓋在上面的高樓。你會發現,接下來幾章的演算法—— 圖的 BFS/DFS、最短路徑、最小生成樹、動態規劃——幾乎每一個都建立在這一章的結構之上。 把這些工具練熟,你解決問題的「武器庫」就真正成形了。
