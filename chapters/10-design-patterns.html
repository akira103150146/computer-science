設計模式與重構

當你第十次寫出「好像在哪寫過」的程式,你不是健忘——你是撞見了一個模式。前人早就替這些反覆出現的難題,磨好了一套有名字、可溝通的解法。

一段似曾相識的程式

假設你在寫一個購物網站。第一週,老闆說要支援信用卡付款,你寫了一段結帳邏輯。 第二週,要加 LINE Pay,你複製貼上、改幾行。第三週,要加超商代碼;第四週,貨到付款…… 於是你的結帳函式裡,長出了一條又臭又長的 if / elif / elif / elif。 每加一種付款方式,你就得回去動那同一坨程式碼,而且每動一次都提心吊膽,深怕弄壞別的分支。

這時你心裡冒出一個念頭:「這結構不太對勁,但我說不上來哪裡不對,更不知道該怎麼改。」 恭喜你——你正站在一道門檻前。門的另一邊,住著兩個會跟著你一輩子的工程素養: 重構(refactoring)與設計模式(design pattern)。 前者教你怎麼把已經寫歪的程式扳正,後者給你一整箱「前人驗證過、有名字」的好結構, 讓你下次一眼就認出「啊,這是策略模式可以解的」。

前面幾章,我們學會了把程式「寫出來」、讓它「跑起來」、甚至「測試它對不對」(第 5 章)。 這一章,我們要往上跨一階,談的是一件更難、卻決定你能走多遠的事: 怎麼把程式寫得「好」——好讀、好改、好擴充,讓三個月後的你(或你的隊友) 不會邊看邊咒罵。

🔑
「能跑」只是及格,「好改」才是高分

軟體的成本,絕大部分不在第一次寫,而在後續無數次的修改。 一份程式一生中被讀的次數,遠多於被寫的次數。所以「容易被讀懂、容易被安全地修改」, 才是專業程式碼真正的價值。設計模式與重構,服務的正是這個目標。

什麼是設計模式?一套設計的詞彙

先講一個容易誤會的點:設計模式不是可以直接複製貼上的程式碼,也不是某個函式庫。 它更像是「針對某一類反覆出現的設計問題,一套被無數人驗證過的解法骨架」。 它描述的是「角色怎麼分、責任怎麼擺、物件之間怎麼互動」,而具體的程式碼,由你照著情境填。

一個更好的比喻:設計模式是工程師之間的共同語言。 當你跟同事說「這裡用一個觀察者(Observer)就好」,對方立刻懂你的意思—— 要有一個被觀察的主體,一群訂閱者,主體一變動就通知大家。 你不必畫圖、不必解釋十分鐘。一個詞,傳遞了一整套結構。 這就是為什麼學模式的最大收穫,往往不是「會用」,而是「會講」。

🔑
模式是「被發現」,不是「被發明」

沒有人坐在書桌前「發明」了策略模式。是無數工程師在各自的專案裡, 不約而同地用相似的方式解決了相似的問題,後來有人把這些重複出現的好結構歸納、命名、記錄下來, 才成了「模式」。就像生物學家不是發明了「翅膀」,而是發現了「飛行」這個問題在鳥、蝙蝠、昆蟲身上 被各自獨立地用相似的方案解決。好模式是「長出來的」,不是「設計出來的」。

📜
模式語言:一個來自建築界的點子

「設計模式」這個概念其實不是程式設計師原創的,而是借自建築。 1977 年,建築師克里斯多福・亞歷山大(Christopher Alexander) 在《模式語言(A Pattern Language)》一書中提出:好的建築與城鎮設計, 可以拆解成 253 個反覆出現的「模式」,例如「面南的中庭」「窗邊的座位」「六英尺深的陽台」。 每個模式都描述「一個情境中反覆出現的問題,以及它的核心解法」。 1994 年,四位工程師(人稱 Gang of Four,GoF)把這個想法搬進軟體界, 寫出了經典的《設計模式(Design Patterns)》,收錄 23 個物件導向設計模式。 這本書,直到今天仍是這個領域的聖經。

在動手認識那些模式之前,我們得先學會看出「程式哪裡不對勁」——因為模式與重構, 都是用來「治病」的;你得先學會「診斷」。

程式碼異味:程式發出的求救訊號

程式碼異味(code smell)是個很傳神的詞,由 Kent Beck 提出、 經 Martin Fowler 發揚光大。它指的是程式裡某些「表面上能跑,但聞起來怪怪的」徵兆—— 就像冰箱飄出一絲異味:不代表東西一定壞了,但值得你打開來檢查。 異味本身不是 bug,而是「這裡的設計可能有問題,未來會讓你痛苦」的早期警報。

⚠️
幾種最常見的程式碼異味
  • 重複程式碼(Duplicated Code):同一段邏輯在好幾處出現;改一處就得記得改其他處,遲早漏掉一個。
  • 過長函式(Long Method):一個函式長到要捲動好幾頁,做了太多事,沒人想讀完。
  • 過大類別(Large Class):一個類別塞了太多欄位與方法,什麼都管,變成無所不包的「神物件」。
  • 過多參數(Long Parameter List):一個函式要傳七八個參數,呼叫時根本記不住順序。
  • 發散式修改(Divergent Change):每次需求一變,都得改同一個類別,因為它扛了太多種不同的責任。
  • 霰彈式修改(Shotgun Surgery):做一個小改動,卻要散彈打鳥地動到十幾個檔案;責任被切得太散。
  • 特性依戀(Feature Envy):某個方法老是在存取「別的類別」的資料,顯示它根本待錯了地方。
  • 資料泥團(Data Clumps):同一組欄位(如 x, y 或 start, end)總是黏在一起出現,暗示它們該被包成一個物件。

注意「發散式修改」與「霰彈式修改」剛好是一對相反的病:前者是「一個地方被太多種理由逼著改」, 後者是「一個理由逼著太多地方改」。健康的設計追求的是兩者的甜蜜點—— 一種變動,集中對應到一個地方。這正是後面許多模式的核心動機。

重構:在不改變行為下,把結構變好

聞到異味,接下來呢?你重構它。 重構(refactoring)的定義很精確,也很重要: 在不改變程式對外行為的前提下,改善它的內部結構。 關鍵在「不改變外部行為」——重構不是改功能、不是修 bug、不是加新東西, 它純粹是「把同一件事,用更好的方式重寫」。輸入什麼、輸出什麼,前後完全一樣。

🔑
重構的兩條鐵律:小步前進 + 測試保護網

第一,小步前進。不要一口氣大改特改。每次只做一個微小、安全的變動, 改完馬上確認程式還是對的,再做下一步。
第二,測試是你的安全網。重構之所以敢動,正是因為有第 5 章學到的測試撐腰: 每改一小步就跑一次測試,只要綠燈,就知道你「沒把行為改壞」。 沒有測試的重構,叫「亂改」,遲早出事。

重構有一整本目錄(Fowler 的《Refactoring》列了七十多種),但你最常用的就那幾招。 我們挑四個最基本、最高頻的,各看一次「之前 / 之後」。

重構一:提取函式(Extract Method)

這是最常用的一招:當一個函式裡有一段程式碼「自成一件事」,就把它抽出來,變成一個有名字的小函式。 名字本身就是最好的註解。看這個印發票的函式,中間那段「算總額」其實是獨立的一件事:

之前——什麼都擠在一起,還要靠註解才看得懂在算什麼:

def print_invoice(order):
    print(f"訂單編號:{order['id']}")
    print(f"客戶:{order['customer']}")
    # 計算總金額
    total = 0
    for item in order['items']:
        total += item['price'] * item['qty']
    if total > 1000:
        total *= 0.9          # 滿千打九折
    print(f"應付金額:{total}")

之後——把「算總額」提取成獨立函式,函式名字取代了註解,主流程一眼看完:

def calculate_total(items):
    """計算商品總額,滿千打九折。"""
    total = sum(item['price'] * item['qty'] for item in items)
    if total > 1000:
        total *= 0.9
    return total

def print_invoice(order):
    print(f"訂單編號:{order['id']}")
    print(f"客戶:{order['customer']}")
    print(f"應付金額:{calculate_total(order['items'])}")

好處不只是好讀:calculate_total 現在可以被單獨測試,也可以被別處重複使用。 過長函式這個異味,十之八九都靠提取函式來治。

重構二:重新命名(Rename)

最便宜、效益卻最高的重構,就是把爛名字改成好名字。 程式碼是寫給「人」看的,一個誠實的名字,勝過十行註解。

之前——calc、d、r 是什麼?鬼才知道:

def calc(d, r):
    return d * (1 + r)

之後——名字自己說明了一切,連註解都快不需要了:

def apply_interest(principal, rate):
    """回傳本金 principal 加上利率 rate 之後的金額。"""
    return principal * (1 + rate)
💡
命名的小準則

變數/函式名要回答「它是什麼 / 它做什麼」,而不是「它怎麼做」。 布林值用 is_、has_ 開頭(如 is_valid);函式用動詞(calculate_total); 別怕名字長,days_until_deadline 永遠勝過 d。 現代編輯器都有「整批改名」功能,改名幾乎零成本,別省這個。

重構三:以多型取代條件式(Replace Conditional with Polymorphism)

還記得開頭那條長長的 if / elif 嗎?當你發現一段條件式是在「根據某個型別標籤,做不同的事」, 而且這種條件式在程式裡到處出現(典型的霰彈式修改),就該考慮用多型取代它: 讓每一種「型別」變成一個類別,各自實作自己的版本。

之前——每多一種形狀,就得回來改這個 if,而且程式各處可能都有類似的判斷:

def area(shape):
    if shape['type'] == 'circle':
        return 3.14159 * shape['r'] ** 2
    elif shape['type'] == 'rectangle':
        return shape['w'] * shape['h']
    elif shape['type'] == 'triangle':
        return 0.5 * shape['base'] * shape['height']
    else:
        raise ValueError("未知形狀")

之後——每種形狀自己負責算自己的面積,條件式消失了:

import math
from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self):
        ...

class Circle(Shape):
    def __init__(self, r):
        self.r = r
    def area(self):
        return math.pi * self.r ** 2

class Rectangle(Shape):
    def __init__(self, w, h):
        self.w, self.h = w, h
    def area(self):
        return self.w * self.h

class Triangle(Shape):
    def __init__(self, base, height):
        self.base, self.height = base, height
    def area(self):
        return 0.5 * self.base * self.height

# 用法:不再需要任何 if 判斷型別
shapes = [Circle(2), Rectangle(3, 4), Triangle(6, 8)]
for s in shapes:
    print(round(s.area(), 2))

現在要新增「梯形」,你只要新增一個類別,完全不必碰任何既有的程式碼。 這呼應了一條著名的設計原則——開放封閉原則(Open–Closed Principle): 對「擴充」開放,對「修改」封閉。能加新檔案就解決,就別去改老檔案。

重構四:引入參數物件(Introduce Parameter Object)

當一個函式的參數列長到讓人記不住(過多參數的異味),而且其中某幾個參數總是「成群結隊」出現 (資料泥團),就把它們打包成一個物件。Python 的 dataclass 讓這件事輕鬆無比。

之前——六個參數一字排開,呼叫時很容易把順序搞錯:

def book_room(name, phone, email, check_in, check_out, guests):
    print(f"{name} 訂房:{check_in} ~ {check_out},{guests} 人")

book_room("王小明", "0912...", "a@b.com", "2026-07-01", "2026-07-03", 2)

之後——把「一筆訂房」包成物件,意圖清楚,也方便日後加欄位:

from dataclasses import dataclass

@dataclass
class Booking:
    name: str
    phone: str
    email: str
    check_in: str
    check_out: str
    guests: int

def book_room(booking):
    print(f"{booking.name} 訂房:{booking.check_in} ~ {booking.check_out},"
          f"{booking.guests} 人")

b = Booking(
    name="王小明", phone="0912...", email="a@b.com",
    check_in="2026-07-01", check_out="2026-07-03", guests=2,
)
book_room(b)

順帶一提:把「總是黏在一起的資料」包成物件之後,你常會發現「操作這些資料的方法」 也很自然地想搬進這個物件——這就一步步把鬆散的程序,長成了結構良好的物件。 重構與設計,本來就是一體兩面。

三大類模式:一張地圖

GoF 把 23 個模式分成三大類。你不必背全部,但記住「這三類各自在煩惱什麼」, 會讓你看到一個新模式時,馬上知道它大概在解哪種問題。

類別在煩惱什麼一句話代表模式
建立型
(Creational)		 物件「怎麼被生出來」 把「建立物件」的細節藏起來,讓你不必到處寫死建構過程 Factory Method、Abstract Factory、Builder、Prototype、Singleton
結構型
(Structural)		 物件與類別「怎麼組起來」 把小零件拼裝成更大的結構,或讓不合的介面合作 Adapter、Decorator、Composite、Facade、Proxy、Bridge、Flyweight
行為型
(Behavioral)		 物件之間「怎麼溝通、分工」 把「演算法」與「責任」在物件間妥善分配與傳遞 Strategy、Observer、Iterator、Command、State、Template Method、Visitor

粗體的那八個,是日常開發中最常遇到的。接下來我們一個一個拆開來看。每個模式我都會用同一個框架介紹: 問題情境 → 解法 → Python 程式碼 → 何時用 / 別濫用。

Strategy 策略:把「可替換的演算法」封裝起來

問題情境:回到開頭那個購物網站。你的結帳流程要支援多種付款方式, 每種方式的「怎麼付」都不一樣,而且未來還會一直新增。如果全塞進一條 if / elif, 每加一種就得回去動那同一個函式——這正是發散式修改的異味。

解法:把「會變動、可互換的那段演算法」抽出來,各自封裝成一個獨立的物件 (稱為策略)。讓使用它的人(稱為情境 Context) 持有一個策略,需要時就委派給它。換演算法,只要換一個策略物件,情境完全不用改。

Python 程式碼:

from abc import ABC, abstractmethod

class PaymentStrategy(ABC):
    @abstractmethod
    def pay(self, amount):
        ...

class CreditCard(PaymentStrategy):
    def pay(self, amount):
        print(f"用信用卡付 {amount} 元")

class LinePay(PaymentStrategy):
    def pay(self, amount):
        print(f"用 LINE Pay 付 {amount} 元")

class Checkout:
    def __init__(self, strategy: PaymentStrategy):
        self.strategy = strategy          # 持有一個策略
    def confirm(self, amount):
        self.strategy.pay(amount)         # 委派給策略去做

# 換演算法只要換策略物件,Checkout 完全不用改
Checkout(CreditCard()).confirm(300)       # 用信用卡付 300 元
Checkout(LinePay()).confirm(300)          # 用 LINE Pay 付 300 元

同樣的精神,你其實天天在用而不自知。Python 內建的 sorted 就吃一個 key 參數—— 那個 key 函式,正是一個「怎麼比較」的策略: 

students = [("小明", 88), ("小美", 95), ("阿強", 76)]

# 「按分數排序」是一個策略
by_score = sorted(students, key=lambda s: s[1])
# 「按名字排序」是另一個策略,換掉 key 就好
by_name = sorted(students, key=lambda s: s[0])
ℹ️
何時用 / 別濫用

該用:當你有「一族可互換的演算法」,而且未來還會增加,或想讓使用者在執行期切換。
別濫用:如果只有兩三種分支、而且幾乎不會再變,一個簡單的 if 反而更清楚。 別為了「看起來很物件導向」就把每個 if 都升級成策略類別。 而且在 Python 裡,策略常常用一個函式就夠了(像上面的 key),不必開一堆類別——這點我們最後一節會再談。

Observer 觀察者:一變動,就通知所有訂閱者

問題情境:你做了一個天氣監測站,溫度一更新,就要同時通知「手機 App」「網頁看板」「警報簡訊」…… 而且未來還想隨時新增或移除通知對象。如果讓天氣站直接去呼叫每一個對象, 它就得認識所有對象、跟它們緊緊綁死,每加一個就要改天氣站的程式。

解法:讓被觀察的主體(Subject)維護一份「訂閱者名單」, 提供「訂閱 / 取消訂閱」的方法;當狀態改變,就逐一通知(notify)名單上的每個 觀察者(Observer)。主體只認得「觀察者有一個 update 方法」這個抽象約定, 完全不在乎對方到底是 App 還是簡訊。

Python 程式碼:

class WeatherStation:
    def __init__(self):
        self._observers = []

    def subscribe(self, observer):
        self._observers.append(observer)

    def unsubscribe(self, observer):
        self._observers.remove(observer)

    def set_temperature(self, temp):
        print(f"\n[氣象站] 溫度更新為 {temp}°C")
        self._notify(temp)               # 狀態一變就廣播

    def _notify(self, temp):
        for obs in self._observers:
            obs.update(temp)

class PhoneApp:
    def update(self, temp):
        print(f"  App 推播:現在 {temp}°C")

class SMSAlert:
    def update(self, temp):
        if temp >= 38:
            print(f"  簡訊警報:高溫 {temp}°C,請注意!")

station = WeatherStation()
station.subscribe(PhoneApp())
station.subscribe(SMSAlert())

station.set_temperature(30)   # 只有 App 推播
station.set_temperature(39)   # App 推播 + 簡訊警報
🔑
你早就活在觀察者模式裡了

所有事件驅動(event-driven)的系統,骨子裡都是觀察者模式。 當你寫 button.addEventListener("click", handler),你就是在把一個觀察者(handler) 訂閱到一個主體(button)上;按鈕被點擊,就「通知」所有訂閱的處理函式。 GUI、前端框架的響應式更新、訊息佇列的發布/訂閱(pub/sub)、Excel 公式自動重算—— 全是這個模式的化身。理解它,你就理解了半個現代軟體的運作方式。

ℹ️
何時用 / 別濫用

該用:一個狀態改變,需要連動通知「數量不定、可動態增減」的多個對象,而你又不想讓主體跟它們綁死。
別濫用:通知鏈太長、太隱晦時,程式會變得難以追蹤——你會搞不清楚「按一下到底觸發了多少連鎖反應」。 另外要小心:忘了取消訂閱,可能造成記憶體洩漏(主體一直握著已經沒用的觀察者)。

Factory 工廠:把「建立物件」的邏輯集中

問題情境:你的程式有十幾個地方都要根據某個字串(來自設定檔、API 回應、使用者輸入) 建立對應的物件:"circle" 就 Circle(...)、"rect" 就 Rectangle(...)。 如果每個地方都自己寫一段判斷,哪天新增一種形狀,你就得霰彈式地改十幾處。

解法:把「決定該建立哪個類別」的邏輯,集中到一個地方。 最簡單的版本叫簡單工廠(Simple Factory)——就是一個負責生產物件的函式: 

import math
from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self): ...

class Circle(Shape):
    def __init__(self, r): self.r = r
    def area(self): return math.pi * self.r ** 2

class Rectangle(Shape):
    def __init__(self, w, h): self.w, self.h = w, h
    def area(self): return self.w * self.h

def make_shape(kind, **kwargs):
    """簡單工廠:把『該建立哪個類別』的知識集中在這裡。"""
    if kind == "circle":
        return Circle(kwargs["r"])
    elif kind == "rectangle":
        return Rectangle(kwargs["w"], kwargs["h"])
    raise ValueError(f"未知形狀:{kind}")

# 全程式只有這個函式知道「字串 -> 類別」的對應
shape = make_shape("circle", r=3)
print(round(shape.area(), 2))   # 28.27

現在 if 判斷只剩一處;新增形狀,只改這一個函式。 要更 Pythonic 一點,還可以用字典查表取代 if,把對應關係寫成資料: 

_REGISTRY = {
    "circle": lambda kw: Circle(kw["r"]),
    "rectangle": lambda kw: Rectangle(kw["w"], kw["h"]),
}

def make_shape(kind, **kwargs):
    if kind not in _REGISTRY:
        raise ValueError(f"未知形狀:{kind}")
    return _REGISTRY[kind](kwargs)

GoF 書裡更正式的工廠方法(Factory Method)則更進一步: 把「要生產什麼」這個決定,交給子類別去覆寫。父類別定義流程,子類別決定細節: 

class Dialog(ABC):
    @abstractmethod
    def create_button(self):       # 工廠方法:由子類別決定生哪種按鈕
        ...

    def render(self):              # 父類別定義流程,呼叫工廠方法
        button = self.create_button()
        button.render()

class WindowsButton:
    def render(self): print("畫一個 Windows 風格按鈕")

class WebButton:
    def render(self): print("畫一個 HTML 按鈕")

class WindowsDialog(Dialog):
    def create_button(self): return WindowsButton()

class WebDialog(Dialog):
    def create_button(self): return WebButton()

WindowsDialog().render()   # 畫一個 Windows 風格按鈕
WebDialog().render()       # 畫一個 HTML 按鈕
ℹ️
何時用 / 別濫用

該用:建立物件的邏輯有點複雜、或散落多處、或需要根據條件決定具體類別時,集中到工廠能省下大量重複與維護成本。
別濫用:如果你只是 obj = Circle(3) 這樣直接建立就好,硬包一層工廠只是徒增間接層。 記住:工廠的價值來自「集中變化點」,沒有變化,就不需要它。

Singleton 單例:全程只允許一個實例(且要小心)

問題情境:有些東西在整個程式裡「應該只有一份」——例如一個全域設定、一個資料庫連線池、 一個日誌記錄器。你希望不管在哪裡取用,拿到的都是「同一個」物件。

解法:單例(Singleton)確保一個類別只會有一個實例,並提供一個全域取用點。 在 Python 裡,可以覆寫 __new__(真正負責「建立」物件的方法)來達成: 

class Logger:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance.logs = []
        return cls._instance          # 永遠回傳同一個實例

    def log(self, msg):
        self.logs.append(msg)

a = Logger()
b = Logger()
print(a is b)        # True ← 是同一個物件
a.log("啟動")
print(b.logs)        # ['啟動'] ← b 看得到 a 寫的,因為它們是同一個
⛔
單例是「爭議最大」的模式,請三思

很多資深工程師認為單例是偽裝成模式的全域變數,而全域狀態正是程式難以維護的元凶之一:

  • 難以測試:單例把狀態藏在類別裡、跨測試殘留。一個測試改了它,會悄悄汙染下一個測試。測試最愛「乾淨、獨立」,單例剛好相反。
  • 隱藏的相依:任何函式都能偷偷取用單例,使得「這段程式到底依賴什麼」變得不透明,耦合無聲蔓延。
  • 並發問題:多執行緒同時第一次建立實例,可能造出兩個,得額外加鎖處理。

更好的替代:把那個「唯一的物件」當參數明確傳進去(這叫依賴注入), 需要它的人就拿得到,也方便在測試時換成假的。

💡
Python 的「天然單例」:模組

Python 的模組(module)本身就是單例——一個模組無論被 import 幾次, 整個程式裡都只有一份。所以在 Python,要一個全域共享的東西,最簡單(也最被推崇)的做法, 往往就是把它放在一個模組裡當模組層級的變數,根本不必動用 __new__ 那套機制。

Decorator 裝飾者:動態地替物件加功能

問題情境:你有一杯咖啡,顧客想加牛奶;另一位想加牛奶又加糖漿;再一位只要雙份濃縮。 如果替每種組合都開一個類別(CoffeeWithMilk、CoffeeWithMilkAndSyrup……), 組合一多就爆炸了。

解法:裝飾者(Decorator)讓你像「俄羅斯娃娃」一樣, 把一個物件層層包起來,每包一層就加上一點功能,而且每一層都維持「跟原本一樣的介面」, 所以外界用起來感覺不出差別,卻可以任意疊加。 

class Coffee:
    def cost(self): return 50
    def desc(self): return "咖啡"

class AddOn:                       # 裝飾者基底:包住一杯飲料
    def __init__(self, drink):
        self._drink = drink

class Milk(AddOn):
    def cost(self): return self._drink.cost() + 10
    def desc(self): return self._drink.desc() + " + 牛奶"

class Syrup(AddOn):
    def cost(self): return self._drink.cost() + 15
    def desc(self): return self._drink.desc() + " + 糖漿"

# 一層一層包上去,自由組合
drink = Syrup(Milk(Coffee()))
print(drink.desc(), drink.cost())   # 咖啡 + 牛奶 + 糖漿 75

如果這個「包一層、加一點功能」的味道你覺得很熟悉——沒錯,Python 的 @decorator 語法正是這個模式的語言內建版! 一個函式裝飾器,就是「拿一個函式,包一層,回傳一個加了料的新函式」: 

import time
import functools

def timer(func):
    @functools.wraps(func)            # 保留原函式的名字與說明
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)    # 呼叫被包住的原函式
        elapsed = time.perf_counter() - start
        print(f"{func.__name__} 花了 {elapsed:.4f} 秒")
        return result
    return wrapper

@timer                                # 等同於 slow = timer(slow)
def slow():
    time.sleep(0.3)

slow()    # slow 花了 0.3xxx 秒

看出對應了嗎?timer 把 slow 包起來,加上「計時」這個功能, 但 slow 用起來還是 slow(),介面沒變。這正是裝飾者模式的精神, 只是 Python 給了它一個漂亮的 @ 語法糖。日誌、快取、權限檢查、重試—— 都常用裝飾器來「外掛」上去,而不必弄髒原本的函式。

ℹ️
何時用 / 別濫用

該用:想對「個別物件」動態加功能,而且功能可以自由組合疊加時。
別濫用:包了太多層之後,除錯會變痛苦——你得剝開一層層娃娃才知道實際發生什麼, 堆疊追蹤(traceback)也會變長。功能組合不多時,直接寫清楚反而好。

Adapter 轉接器:讓不相容的介面合作

問題情境:你的程式到處呼叫 shape.area()。現在你想用一個第三方(或舊系統)的矩形類別, 但它偏偏沒有 area(),只有一個 get_dimensions()。你不能改它的原始碼, 但又不想為它在程式裡到處開特例。

解法:就像出國旅行帶的電源轉接頭——它不改變插座,也不改變你的充電器, 只是夾在中間做翻譯。轉接器(Adapter)包住那個「介面不合」的物件, 對外提供你的程式期待的介面,內部再翻譯成對方聽得懂的呼叫: 

# 這是無法修改的舊類別:介面跟我們要的不一樣
class LegacyRectangle:
    def __init__(self, w, h):
        self.w, self.h = w, h
    def get_dimensions(self):
        return (self.w, self.h)        # 它只給你長寬,沒有 area()

# 轉接器:對外提供 area(),內部翻譯成 get_dimensions()
class RectangleAdapter:
    def __init__(self, legacy_rect):
        self._rect = legacy_rect
    def area(self):
        w, h = self._rect.get_dimensions()
        return w * h

# 現在它可以跟所有「期待 .area()」的程式無縫合作
shapes = [RectangleAdapter(LegacyRectangle(3, 4))]
for s in shapes:
    print(s.area())   # 12
ℹ️
何時用 / 別濫用

該用:整合你無法修改的第三方函式庫或舊系統,讓它套進你既有的介面約定。
別濫用:如果那個介面本來就是你自己能改的,直接把它改成一致的介面就好,別多包一層轉接器。 轉接器是「不得已時的橋」,不是常態。

Composite 組合:用一致的方式處理樹狀結構

問題情境:檔案系統裡有「檔案」與「資料夾」。資料夾裡可以放檔案,也可以放別的資料夾, 層層巢狀。現在你想算「某個資料夾總共佔多少空間」。如果處理「檔案」和「資料夾」要寫兩套不同的程式、 還得到處判斷「這是檔案還是資料夾」,程式很快就會纏成一團。

解法:組合(Composite)讓「單個物件(葉節點)」與 「物件的容器(組合節點)」共用同一個介面。 如此一來,使用者可以用完全一樣的方式對待「一個檔案」和「一整棵資料夾樹」—— 不必再區分手上的到底是單一物件還是一群物件。

Python 程式碼:

class File:
    def __init__(self, name, size):
        self.name = name
        self.size = size
    def total_size(self):
        return self.size                 # 葉節點:直接回傳自己的大小

class Folder:
    def __init__(self, name):
        self.name = name
        self.children = []
    def add(self, node):
        self.children.append(node)
        return self
    def total_size(self):
        # 組合節點:把每個小孩的 total_size() 加起來(遞迴!)
        return sum(child.total_size() for child in self.children)

# 蓋一棵樹
root = Folder("專案")
src = Folder("src")
src.add(File("a.py", 300)).add(File("b.py", 500))
root.add(File("main.py", 100)).add(src).add(File("README", 50))

print(root.total_size())   # 950 ← 一行就算完整棵樹,完全不必管哪個是檔案哪個是資料夾

注意 total_size() 的呼叫者,完全不需要知道手上的是檔案還是資料夾—— 它對兩者一視同仁。這種「整體與部分一致對待」的能力,正是組合模式的魔力。 它和我們在遞迴章學過的「樹」是天作之合:組合模式幾乎就是「物件導向版的遞迴資料結構」。

ℹ️
何時用 / 別濫用

該用:資料天生是樹狀/巢狀(檔案系統、組織圖、GUI 元件、運算式樹、選單),且你想用統一的方式處理整體與部分。
別濫用:資料根本不是樹狀、或巢狀只有一層時,硬套組合只是把簡單問題複雜化。

Iterator 迭代器:把「逐一走訪」標準化

問題情境:你做了一個自訂的集合(例如一個環狀緩衝區、一棵樹), 希望別人能像走訪 list 一樣,用 for x in my_collection 一個一個拿出元素, 而不必知道你內部到底是怎麼存的。

解法:迭代器(Iterator)提供一個標準的「逐一取出下一個」的介面, 把「怎麼走訪」與「集合內部怎麼存」分開。好消息是:Python 把這個模式直接做進了語言裡。 你只要實作兩個魔術方法——__iter__(回傳迭代器)與 __next__(回傳下一個,沒了就丟 StopIteration)—— 你的物件就能直接用在 for 迴圈裡: 

class Countdown:
    """從 start 倒數到 1 的可迭代物件。"""
    def __init__(self, start):
        self.start = start

    def __iter__(self):
        self.current = self.start
        return self                       # 自己就是自己的迭代器

    def __next__(self):
        if self.current <= 0:
            raise StopIteration           # 沒東西了,for 迴圈看到這個就停
        value = self.current
        self.current -= 1
        return value

for n in Countdown(3):
    print(n)        # 3, 2, 1  ← for 迴圈背後就是反覆呼叫 __next__

但在 Python,大多數時候你連這些都不用寫——用產生器(generator)更簡潔。 只要在函式裡用 yield,Python 就自動幫你生出一個迭代器: 

def countdown(start):
    while start > 0:
        yield start          # yield 一個值,然後「暫停」在這裡,下次再接著跑
        start -= 1

for n in countdown(3):
    print(n)        # 3, 2, 1  ← 同樣的效果,但程式碼少了一大半
🔑
迭代器:一個被「吸收進語言」的模式

迭代器是「模式被語言內建」的最佳範例。在 GoF 時代,迭代器是個需要自己手刻的設計模式; 但今天的 Python(以及多數現代語言)直接把它變成語法層級的東西——for 迴圈、 yield 產生器、生成式,全都建立在迭代器協定之上。 當一個模式好到「應該人人都用」,語言設計者往往就會把它收進語言本身。這正是下一節要談的現象。

反模式與過度設計:不要為了用模式而用模式

學完一堆模式,新手最容易得一種病:看什麼都想套模式。簡單的 if 非要升級成策略類別, 三行就能解決的事硬要包進工廠加裝飾者……結果程式不但沒變好,反而多了一堆間接層, 讀的人得在十個檔案間跳來跳去才看懂一件小事。這就是過度設計(over-engineering), 本身就是一種反模式(anti-pattern)——看似在解決問題,實則製造問題。

反模式是「反覆出現的壞解法」,和設計模式恰好相反。常見的有:

  • 神物件(God Object):一個類別什麼都管,幾千行、無所不包,改它像拆炸彈。
  • 義大利麵程式(Spaghetti Code):控制流糾纏不清,沒有結構,牽一髮動全身。
  • 金錘子(Golden Hammer):「我有一把錘子,所以看什麼都像釘子」——學了某個模式就到處硬套。
  • 複製貼上程式設計:靠複製貼上累積功能,重複程式碼遍地開花。
  • 過早最佳化(Premature Optimization):為了還沒發生的效能問題,把程式寫得又複雜又難懂。
🔑
兩條保命原則:KISS 與 YAGNI

KISS(Keep It Simple, Stupid):讓它保持簡單。能用最直白的方式解決,就別繞遠路。 複雜不是高明,清楚才是。
YAGNI(You Aren't Gonna Need It):你不會用到它的。 不要為了「以後也許需要」而現在就加一堆彈性與抽象。多數「以後」永遠不會來, 你只是先付出了複雜度的代價。需要時再加,永遠不嫌晚——因為到時你有了測試,可以安心重構。

所以正確的順序是:先寫出最簡單能動的版本 → 發現它開始發臭(出現異味、改動變痛苦) → 這時才重構,引入恰好夠用的模式。 模式是用來「治痛」的藥,不是用來「預防」的保健品。沒病別亂吃藥。

模式與語言:當一級函式讓模式「消失」

最後一個重要觀念,會改變你看待整本 GoF 的方式。GoF 的 23 個模式, 誕生於 1990 年代的 C++/Java 世界——那是個「函式不能單獨存在,一切都得包進類別」的世界。 於是許多模式,本質上是在「用物件,模擬出函式該有的彈性」。

但 Python 有一級函式(first-class functions):函式可以被當成值、 存進變數、當參數傳遞、當回傳值。這項能力,讓好幾個 GoF 模式直接「蒸發」成幾行普通程式碼。 最鮮明的例子就是策略模式——回想本章一開始那個 PaymentStrategy 抽象類別加一堆子類別, 在 Python 裡,你常常一個函式就夠了: 

# Java 風格:策略 = 一個介面 + 多個實作類別(本章前面那一大段)
# Python 風格:策略 = 就是一個函式

def credit_card(amount):
    print(f"用信用卡付 {amount} 元")

def line_pay(amount):
    print(f"用 LINE Pay 付 {amount} 元")

def checkout(amount, pay_strategy):     # 策略就是傳進來的那個函式
    pay_strategy(amount)

checkout(300, credit_card)              # 用信用卡付 300 元
checkout(300, line_pay)                 # 用 LINE Pay 付 300 元

沒有抽象類別、沒有繼承、沒有樣板,策略模式「消失」了——它溶解進了「把函式當參數傳」這件再平常不過的事。 同樣地,命令模式(Command)在有閉包(closure)的語言裡也常常退化成一個函式; 迭代器,我們剛看到,被 yield 吸收了。

🔑
模式是解法,不是目的

有句話說得好:「設計模式,是語言能力不足時的補救措施。」 當一個模式好到人人都需要,語言設計者往往會把它直接做進語言(像 Python 的 for、 yield、@decorator、一級函式)。 所以學模式時,真正該學的是它背後要解決的問題與設計取捨, 而不是死記某個類別圖。換到不同語言,實作可能天差地遠,甚至根本不需要—— 但「為什麼要把會變動的部分隔離出來」這個智慧,放諸四海皆準。

動手練習

模式不是用看的,是用寫的。下面五題由淺入深,請務必先自己動手,卡住再看解答。

✏️ 練習 10.1 實作

下面這個函式有好幾種程式碼異味。請說出它至少有哪兩種異味,並把它重構得更好讀 (至少用上「提取函式」與「重新命名」)。注意:重構後對外行為要完全一樣。

def p(u):
    # u 是一個 dict,代表使用者
    print("==== 報表 ====")
    print(u["n"])
    s = 0
    for x in u["o"]:                # o 是訂單金額串列
        s = s + x
    if s > 10000:
        print("VIP")
    else:
        print("一般")
    print("總消費:" + str(s))
看解答

主要異味:(1) 名字爛透了(p、u、n、o、s、x 全靠註解才看得懂); (2) 過長函式 / 責任混雜(同一個函式又算總額、又判等級、又印報表); 另外「算總額」那段其實是可以提取、也可重用的獨立邏輯。重構後:

def total_spending(orders):
    """計算訂單金額總和。"""
    return sum(orders)

def membership_level(total):
    """依總消費判定會員等級。"""
    return "VIP" if total > 10000 else "一般"

def print_report(user):
    total = total_spending(user["orders"])
    print("==== 報表 ====")
    print(user["name"])
    print(membership_level(total))
    print(f"總消費:{total}")

# 假設資料的鍵也順手改成有意義的名字
print_report({"name": "王小明", "orders": [3000, 5000, 4000]})

每個函式現在只做一件事、名字自我說明,而且 total_spending 與 membership_level 都能被單獨測試與重用。這就是「小步重構」累積出的威力。

✏️ 練習 10.2 實作

下面是一段用 if / elif 算運費的程式。請用 Strategy 模式重寫它, 讓「新增一種運送方式」時不必修改既有的計算函式。(用函式版或類別版都可以。)

def shipping_fee(method, weight):
    if method == "standard":
        return weight * 10
    elif method == "express":
        return weight * 25 + 50
    elif method == "freezer":
        return weight * 30 + 100
    else:
        raise ValueError("未知的運送方式")
看解答

在 Python,最輕巧的做法是「策略 = 函式」,用一個字典把名稱對應到策略函式。 新增運送方式,只要寫一個新函式、在字典登記一行,完全不動既有程式:

def standard(weight):
    return weight * 10

def express(weight):
    return weight * 25 + 50

def freezer(weight):
    return weight * 30 + 100

SHIPPING_STRATEGIES = {
    "standard": standard,
    "express": express,
    "freezer": freezer,
}

def shipping_fee(method, weight):
    strategy = SHIPPING_STRATEGIES.get(method)
    if strategy is None:
        raise ValueError(f"未知的運送方式:{method}")
    return strategy(weight)

print(shipping_fee("express", 4))   # 150
# 要加「隔日到貨」?只要新增一個函式並登記,不必碰 shipping_fee:
# SHIPPING_STRATEGIES["nextday"] = lambda w: w * 40 + 200

注意原本的 if / elif 不見了,「該用哪個策略」變成一次字典查表。 這就是策略模式想達成的:把「會增長的變化點」隔離開來。

✏️ 練習 10.3 實作

實作一個簡單的事件系統(Observer)。寫一個 EventBus 類別, 支援 subscribe(handler) 訂閱、publish(data) 發布 (發布時呼叫所有訂閱者並把 data 傳給它們)。 測試:訂閱兩個處理函式,發布一則訊息,兩個都要收到。

看解答

觀察者的精髓:主體維護一份訂閱名單,發布時逐一通知。這裡訂閱者用「函式」最簡單:

class EventBus:
    def __init__(self):
        self._handlers = []

    def subscribe(self, handler):
        self._handlers.append(handler)

    def unsubscribe(self, handler):
        self._handlers.remove(handler)

    def publish(self, data):
        for handler in self._handlers:    # 廣播給所有訂閱者
            handler(data)

# 測試
bus = EventBus()
bus.subscribe(lambda msg: print(f"記錄到 log:{msg}"))
bus.subscribe(lambda msg: print(f"寄出通知信:{msg}"))
bus.publish("使用者註冊成功")
# 記錄到 log:使用者註冊成功
# 寄出通知信:使用者註冊成功

這個 30 行的小東西,就是所有 pub/sub、事件匯流排、前端事件系統的核心骨架。 你可以再進階:讓 subscribe 指定「事件類型」,publish 時只通知對應類型的訂閱者 (用一個 dict[事件名 -> 處理函式串列])。

✏️ 練習 10.4 實作

寫一個 Python 裝飾器 logged,套在任何函式上後, 每次呼叫時印出「呼叫了什麼、傳了哪些參數、回傳什麼」。 例如套在一個 add(a, b) 上,呼叫 add(2, 3) 應印出類似 呼叫 add(2, 3) -> 5 的訊息。(提示:參考本章 timer 的寫法。)

看解答

裝飾器就是「包一層、加功能、回傳新函式」。用 functools.wraps 保留原函式的名字:

import functools

def logged(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        # 把位置參數與關鍵字參數組成好看的字串
        parts = [repr(a) for a in args]
        parts += [f"{k}={v!r}" for k, v in kwargs.items()]
        signature = ", ".join(parts)

        result = func(*args, **kwargs)
        print(f"呼叫 {func.__name__}({signature}) -> {result!r}")
        return result
    return wrapper

@logged
def add(a, b):
    return a + b

add(2, 3)            # 呼叫 add(2, 3) -> 5
add(a=10, b=20)      # 呼叫 add(a=10, b=20) -> 30

同樣的骨架稍改一下就能做計時(timer)、快取(記住算過的結果)、權限檢查、自動重試。 這就是裝飾者模式在 Python 裡的日常樣貌——把橫切關注點(cross-cutting concern) 乾淨地外掛上去,不弄髒原函式。

✏️ 練習 10.5 挑戰

用 Composite 模式建一個檔案系統模型,並計算總大小。 需求:File 有名字與大小;Folder 可以裝 File 或別的 Folder。 兩者都要有 size() 方法。進階:再加一個 show(indent=0) 方法, 能縮排印出整棵樹的結構。

看解答

關鍵在「葉節點與組合節點共用同一介面」:File.size() 直接回傳自己, Folder.size() 則遞迴地把小孩們加總。呼叫者完全不必區分手上是檔案還是資料夾:

class File:
    def __init__(self, name, size):
        self.name = name
        self._size = size

    def size(self):
        return self._size

    def show(self, indent=0):
        print(" " * indent + f"📄 {self.name} ({self._size})")

class Folder:
    def __init__(self, name):
        self.name = name
        self.children = []

    def add(self, node):
        self.children.append(node)
        return self                       # 回傳自己,方便串接 add().add()

    def size(self):
        return sum(child.size() for child in self.children)   # 遞迴加總

    def show(self, indent=0):
        print(" " * indent + f"📁 {self.name}/ ({self.size()})")
        for child in self.children:
            child.show(indent + 2)        # 不管小孩是檔案還是資料夾,一律呼叫 show

# 蓋一棵樹
root = Folder("專案")
src = Folder("src")
src.add(File("a.py", 300)).add(File("b.py", 500))
root.add(File("main.py", 100)).add(src).add(File("README.md", 50))

print("總大小:", root.size())   # 總大小: 950
root.show()
# 📁 專案/ (950)
#   📄 main.py (100)
#   📁 src/ (650)
#     📄 a.py (300)
#     📄 b.py (500)
#   📄 README.md (50)

留意 show 與 size 都用了「對每個小孩做同一件事」的遞迴結構—— 這正是組合模式與樹狀遞迴的完美結合。如果你之前覺得遞迴抽象, 這個「資料夾包資料夾」的具體例子,應該讓它變得無比自然。

把這一章串起來

這一章,我們從「程式聞起來怪怪的」一路走到「用前人的智慧把它治好」。回顧路上的幾塊路標:

  • 設計模式是針對反覆出現的設計問題的成熟解法,也是工程師之間的共同語言。它是被「發現」歸納出來的,源頭可追到建築師亞歷山大的「模式語言」。
  • 程式碼異味是設計出問題的早期警報:重複、過長函式、過大類別、過多參數、發散式/霰彈式修改、特性依戀……聞到了就該檢查。
  • 重構是在不改變行為下改善結構,靠小步前進加測試保護網(第 5 章)安全進行。提取函式、重新命名、以多型取代條件式、引入參數物件,是你最常用的四招。
  • 模式分建立型、結構型、行為型三大類。本章深入了八個:Strategy、Observer、Factory、Singleton、Decorator、Adapter、Composite、Iterator——每個都記住它的「問題情境」勝過記住它的類別圖。
  • Singleton 要警惕(全域狀態、難測試);Python 的模組本身就是天然單例。
  • 最重要的心法:不要為了用模式而用模式。KISS、YAGNI,先簡單,等真的痛了再重構引入模式。過度設計本身就是反模式。
  • 模式是「解法」不是「目的」。在有一級函式的 Python,Strategy、Command、Iterator 等常常溶解成幾行普通程式碼,甚至被語言內建吸收。

你現在手上有了一整箱經過驗證的設計工具,也有了「何時別打開工具箱」的判斷力。 這份「把程式寫得好」的素養,會跟著你走過往後每一個專案。
