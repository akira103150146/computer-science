邏輯程式設計 Prolog

你寫了一輩子的程式,都在告訴電腦「怎麼做」。但如果你只描述「是什麼」——把事實與規則攤在桌上,然後說「答案自己找吧」——會發生什麼事?歡迎來到一個你只負責描述問題、由機器負責解題的世界。

回到原點:兩種知識,這次換它當主角

還記得第 1 章那碗滷肉飯嗎?我們說過,知識有兩種。命令式知識(imperative knowledge)告訴你「怎麼做」: 切肉、爆香、燉煮——一步步的食譜。宣告式知識(declarative knowledge)告訴你「是什麼」: 滷肉飯是那碗香香油亮的東西。我們當時下了一個結論:電腦只懂命令式知識。你想算平方根, 不能只丟給它「平方根的定義」,你得給它一串它能一步步執行的指令。

那個結論,其實只說對了一半。

從第 2 章一路到現在,你寫的每一行 Python、每一行 C,都是命令式的:迴圈怎麼跑、變數怎麼變、 遞迴怎麼往下鑽,全是你親手安排的「步驟」。這是命令式程式設計(imperative programming)的世界, 也是大多數人唯一見過的世界。但計算的版圖比這大得多。這一章,我們要走進一個顛倒過來的國度: 在這裡,你只描述宣告式知識——把你知道的「事實」與「規則」全部寫下來—— 然後丟出一個問題,剩下的「怎麼做」,交給語言自己去想。

這就是邏輯程式設計(logic programming),而它最經典、最純粹的代表, 是一門 1972 年誕生於法國馬賽、名字取自「programmation en logique」的語言:Prolog。

🔑
Prolog 的核心承諾

在命令式語言裡,你寫的是演算法(algorithm):解決問題的明確步驟。 在 Prolog 裡,你寫的是邏輯(logic):關於問題的事實與規則。一句口號道盡這個範式的靈魂—— 「Algorithm = Logic + Control」(Kowalski, 1979):你負責提供邏輯(知識), Prolog 的推論引擎負責提供控制(搜尋怎麼跑)。你描述問題,Prolog 找答案。

這聽起來幾乎像魔法。一台只懂命令式的機器,怎麼可能「自己想辦法」?祕密在於:Prolog 底下其實藏著一台 通用的命令式引擎(一個系統化的搜尋程序),它把你的宣告式描述,翻譯成它自己的搜尋步驟。 換句話說——有人替你把「是什麼」轉成了「怎麼做」,而那個人,就是 Prolog 的作者們。 你站在他們的抽象之上(又是第 1 章的「抽象高塔」!),從此只需要思考「什麼是真的」。

三個基本構件:事實、規則、查詢

Prolog 的整個世界,只由三種東西組成。學會它們,你就學會了 Prolog 的骨架。我們用一個「家族關係」的知識庫, 從頭把這三件事玩透——這個例子會一路陪我們到本章中段。

① 事實(fact):無條件為真的陳述

事實是你直接斷言為真的東西,就像在桌上攤開的牌。我們先寫下「誰是誰的父母」: 

parent(tom, bob).      % tom 是 bob 的父母
parent(pam, bob).      % pam 是 bob 的父母
parent(tom, liz).      % tom 是 liz 的父母
parent(bob, ann).      % bob 是 ann 的父母
parent(bob, pat).      % bob 是 pat 的父母
parent(pat, jim).      % pat 是 jim 的父母

每一行都是一個事實,以句點 . 結尾(這是 Prolog 最常被忘記的標點,漏了它整個程式就壞)。 parent 叫做述詞(predicate)——對,就是第 13 章那個「帶空格的命題」。 括號裡的 tom、bob 是原子(atom):小寫開頭的常數,代表一個具體的東西。 再加上每個人的性別: 

male(tom).    male(bob).    male(jim).
female(pam).  female(liz).  female(ann).  female(pat).
⚠️
大寫 = 變數,小寫 = 原子(Prolog 第一鐵則)

這是 Prolog 最關鍵、也最容易讓初學者翻車的規則:識別字第一個字母大寫(或以底線 _ 開頭)的, 是變數(variable),代表「某個還不知道是誰的東西」;小寫開頭的,是原子, 代表一個固定的具體值。所以 tom 是「那個叫 tom 的人」,而 X、Who、Person 是「待填的空格」。 寫成 parent(Tom, bob) 的話,Tom 會被當成變數,意思完全變了!

② 查詢(query):向知識庫提問

把上面的事實存成檔案、載入 Prolog 系統後,你會看到一個 ?- 提示符。這是你提問的地方。 最簡單的問題是「這件事是真的嗎?」: 

?- parent(tom, bob).
true.

?- parent(tom, jim).
false.

Prolog 去知識庫裡找:有 parent(tom, bob) 這個事實嗎?有 → true。 有 parent(tom, jim) 嗎?沒有,也推不出來 → false。但真正的魔法,在你把變數放進問題裡的時候: 

?- parent(tom, X).
X = bob ;
X = liz.

你問的是:「存在哪些 X,使得 tom 是 X 的父母?」這不再是「是非題」, 而是「找出所有答案」。Prolog 回答 X = bob;你按下分號 ;(意思是「還有別的嗎?」), 它再給你 X = liz。這個「用 ; 要下一個解」的動作,是 Prolog 互動的招牌, 我們稍後會看到它背後的回溯機制。你也可以反過來問「bob 的父母有誰?」: 

?- parent(X, bob).
X = tom ;
X = pam.

注意一件深刻的事:同一個 parent 述詞,可以正著問、也可以反著問。 在 Python 裡,「找某人的小孩」和「找某人的父母」是兩個不同的函式;在 Prolog 裡,它們是同一個關係的不同查法。 這就是宣告式的第一個甜頭——你描述的是「關係」,而關係沒有固定的輸入輸出方向。

③ 規則(rule):從已知推出未知

事實是攤在桌上的牌,規則則是「如果……那麼……」的推論。規則讓你定義新的關係, 而不必把每個情況都當事實寫死。語法是 結論 :- 條件.,那個 :- 唸作「if(如果)」, 是把右邊條件導向左邊結論的箭頭: 

% X 是 Y 的母親,如果:X 是 Y 的父母,而且 X 是女性
mother(X, Y) :- parent(X, Y), female(X).

% X 是 Y 的父親,如果:X 是 Y 的父母,而且 X 是男性
father(X, Y) :- parent(X, Y), male(X).

讀法:「mother(X, Y) 為真,如果 parent(X, Y) 為真而且 female(X) 為真。」 那個逗號 , 是合取(and):所有條件都得成立。這正是第 13 章述詞邏輯的化身—— 一條規則就是一個帶變數的蘊含式 $\forall X\,\forall Y\;\big(\text{parent}(X,Y) \wedge \text{female}(X) \rightarrow \text{mother}(X,Y)\big)$。

有了規則,我們就能推導「祖父母」「手足」這些沒有直接寫成事實的關係:

% X 是 Z 的祖父母:X 是某個 Y 的父母,而 Y 又是 Z 的父母
grandparent(X, Z) :- parent(X, Y), parent(Y, Z).

% X 和 Y 是手足:他們有共同的父母 P,而且 X、Y 不是同一個人
sibling(X, Y) :- parent(P, X), parent(P, Y), X \= Y.

這裡 X \= Y 唸作「X 不等於 Y」,擋掉「自己是自己的手足」這種蠢答案。現在來問問題: 

?- grandparent(tom, Who).
Who = ann ;
Who = pat.

?- sibling(ann, Who).
Who = pat.

?- mother(X, bob).
X = pam.

我們從來沒寫過「tom 是 ann 的祖父母」這個事實,但 Prolog 透過 grandparent 規則,自己推導出來了。 這就是規則的威力:你只寫下少量事實 + 幾條規則,Prolog 就能回答無數個你從沒明說的問題。

✏️ 練習 45.1 實作

沿用上面的家族知識庫,新增三條規則:

  • grandmother(X, Y):X 是 Y 的祖母。
  • aunt(X, Y):X 是 Y 的姑姑/阿姨(X 是女性,且 X 是 Y 某個父母的手足)。
  • 另外為一個全新的小世界(例如「誰喜歡誰」或「哪本書是哪個作者寫的」)寫至少 3 個事實 + 1 條規則,並寫出一個會回傳多個解的查詢。
看解答 
grandmother(X, Y) :- grandparent(X, Y), female(X).

% X 是 Y 的姑姑/阿姨:X 是女性,X 是某人 P 的手足,而 P 是 Y 的父母
aunt(X, Y) :- female(X), sibling(X, P), parent(P, Y).

% ---- 一個全新的小世界:誰喜歡誰 ----
likes(mary, food).
likes(mary, wine).
likes(john, wine).
likes(john, mary).

% John 喜歡所有 Mary 喜歡的東西
likes(john, X) :- likes(mary, X).

查詢「John 喜歡哪些東西?」會回傳多個解:

?- likes(john, What).
What = wine ;     % 直接事實
What = mary ;     % 直接事實
What = food ;     % 經由規則:Mary 喜歡 food
What = wine.      % 經由規則:Mary 喜歡 wine(會重複出現!)

注意最後 wine 出現兩次——一次來自事實、一次來自規則。Prolog 老老實實地回報每一條「能成功的推導路徑」, 即使結論重複。這是初學者常見的驚訝點,稍後談回溯時你會完全明白為什麼。

統一:變數如何與項「對上」

當你問 parent(tom, X),Prolog 拿這個問題去和事實 parent(tom, bob) 比對, 然後得出 X = bob。這個「比對並讓兩邊變得一致」的核心動作,叫做統一(unification)。 它是 Prolog 引擎的心臟——所有的查詢、所有的推論,底層都是一連串統一。

統一問的是:「能不能找到一組變數的值,讓兩個項(term)變得一模一樣?」規則很直覺:

  • 原子對原子:相同才統一成功。bob 和 bob ✔;bob 和 liz ✘。
  • 變數對任何東西:變數會「綁定(bind)」成那個東西。X 對 bob → X = bob。
  • 結構對結構:述詞名稱與參數個數要相同,然後對應位置逐一遞迴統一。

你可以直接用 = 來測試統一:

?- X = bob.
X = bob.

?- parent(tom, X) = parent(tom, bob).
X = bob.                          % 結構相同,逐位統一,X 綁定為 bob

?- point(X, 3) = point(7, Y).
X = 7,
Y = 3.                            % 兩個變數各自從對面找到值

?- parent(tom, X) = parent(pam, Y).
false.                            % tom ≠ pam,第一個位置就對不上,整個失敗

統一比一般語言的「賦值」聰明得多:它是雙向的——變數可以在等號的任何一邊, 值會自己「流」到變數那裡。而一旦一個變數在某次推導中被綁定了,它在這條推導路徑上就不能再變 (這叫「邏輯變數一次綁定」)。記住這個性質,下一節的回溯才看得懂。

🔑
查詢 = 統一 + 推論

當你問 grandparent(tom, Who),Prolog 做的事是:(1) 拿這個目標去統一規則的頭 grandparent(X, Z),綁定 X = tom、Z = Who;(2) 規則的頭成立的條件,是它的身體 parent(X, Y), parent(Y, Z) 成立,於是這兩個新目標遞迴地用同一套統一程序去解。 整個 Prolog,就是「統一 + 把目標換成子目標」的不斷重複。

推論引擎:回溯與搜尋

現在來掀開引擎蓋,看看 Prolog 到底怎麼「自己想辦法找答案」。答案是一個你已經很熟的東西: 深度優先搜尋(depth-first search)——對,就是第 30 章那個一路鑽到底、走不通再退回來的 DFS。 Prolog 解一個查詢的過程,就是在一棵「可能性的樹」上做 DFS,而那個「退回來再試別條路」的動作, 有個專屬名字:回溯(backtracking)。

它的工作流程,可以濃縮成四個動作,不斷循環:

  1. 取目前最左邊的目標,由上而下掃描知識庫,找第一個能統一的事實或規則頭。
  2. 找到了 → 若是規則,就把規則身體的條件變成新的子目標,繼續往下解(深入)。
  3. 所有子目標都成功 → 找到一個解,回報。你按 ; 要更多 → 它假裝失敗,逼自己回溯。
  4. 某個目標失敗(沒有東西能統一)→ 回溯:退回上一個選擇點,撤銷剛才的變數綁定,試下一個候選。

讓我們把 ?- grandparent(tom, Who). 的搜尋過程,一步步畫成一棵樹。 這棵樹會清楚顯示 Prolog 怎麼深入、怎麼找到解、怎麼在死路上回溯。

盯著這棵樹看,你會發現幾件事。第一,解的順序由「事實在程式裡的書寫順序」決定—— 因為 Prolog 由上而下掃描,parent(tom, bob) 寫在 parent(tom, liz) 前面,所以 bob 這條先被探索。 第二,回溯時變數綁定會被撤銷:當 Y = liz 那條失敗,Prolog 不只退回來,還會把 Y 解綁, 乾乾淨淨地像沒發生過。第三,那個分號 ; 的真面目揭曉了——你按 ;, 其實是在主動命令 Prolog「假裝剛才的解失敗,給我回溯找下一條路」。

🔗
這就是第 30 章的 DFS,換了身衣服

還記得 DFS 怎麼走迷宮嗎?選一條岔路走到底,撞牆就退回上一個路口、試下一條岔路,直到走遍所有可能。 Prolog 的回溯一模一樣:每個「有多個候選可統一」的目標,就是一個岔路口; 「某個子目標失敗」就是撞牆;「回溯撤銷綁定」就是退回路口。把 Prolog 想成「一台專門在邏輯推導樹上跑 DFS 的機器」, 你對它的直覺就全通了。

遞迴規則:讓推論深入無限

到目前為止,grandparent 只能往上找兩層。但「祖先(ancestor)」呢?祖先可以是父母、是祖父母、 是曾祖父母……深度事先不知道。在命令式語言裡,這要靠迴圈或遞迴函式;在 Prolog 裡, 答案是遞迴規則(recursive rule)——而它的長相,會讓你想起第 3 章與第 13 章的老朋友。

祖先的定義,天生就是遞迴的:「X 是 Y 的祖先,如果 X 是 Y 的父母(基底),或 X 是某人 Z 的父母、而 Z 又是 Y 的祖先(遞迴)」:

% 基底情形:父母就是(最近的)祖先
ancestor(X, Y) :- parent(X, Y).

% 遞迴情形:X 是 Z 的父母,而 Z 是 Y 的祖先
ancestor(X, Y) :- parent(X, Z), ancestor(Z, Y).

兩條規則,共用同一個述詞名 ancestor,就構成「或」的關係(任一條成立即可)。看出它和第 13 章結構歸納法、 第 3 章遞迴函式的對應了嗎?第一條是基底情形(base case),第二條把問題化小(往上挪一代)再遞迴呼叫自己。 這正是「遞迴 ⟷ 歸納:同一枚硬幣的兩面」。來查查看: 

?- ancestor(tom, jim).
true.                  % tom → bob → pat → jim,層層推導

?- ancestor(X, jim).   % 誰是 jim 的祖先?(全部找出來)
X = pat ;
X = bob ;
X = tom.

ancestor(tom, jim) 為真,是因為 Prolog 順著遞迴一路推:tom 是 bob 的父母,bob 是 pat 的祖先 (因為 bob 是 pat 的父母),pat 是 jim 的祖先(pat 是 jim 的父母)。遞迴規則就像一條會自我延伸的鏈子, 要多長有多長。

⚠️
遞迴規則的順序攸關生死

遞迴規則裡,基底情形通常要寫在前面,而且遞迴呼叫前最好先放一個「會讓問題變小」的目標 (上面是 parent(X, Z))。如果你把規則寫成 ancestor(X, Y) :- ancestor(X, Z), parent(Z, Y). ——遞迴呼叫擺在最前面、又沒先縮小問題——Prolog 會一頭栽進 ancestor(X, Z)、 再展開成 ancestor(X, Z')……無限遞迴、堆疊爆掉。這是 Prolog 的頭號陷阱: 你的「邏輯」明明正確,卻因為「控制」(搜尋順序)而跑不動。再次印證 Algorithm = Logic + Control。

✏️ 練習 45.2 實作

(a) 定義 descendant(X, Y):X 是 Y 的後代。(提示:後代是祖先的相反方向,一行就能搞定。)

(b) 換個場景呼應第 30 章:給定有向圖的邊事實 edge(a, b). edge(b, c). edge(c, d). edge(a, c)., 寫一個遞迴規則 reachable(X, Y),判斷從節點 X 能不能沿著邊走到 Y。

看解答

(a) 後代就是「把祖先反過來」:X 是 Y 的後代,等於 Y 是 X 的祖先。

descendant(X, Y) :- ancestor(Y, X).

(b) 結構和 ancestor 一模一樣:基底是「有一條直接的邊」,遞迴是「先走一步、剩下的繼續走」:

edge(a, b).  edge(b, c).  edge(c, d).  edge(a, c).

reachable(X, Y) :- edge(X, Y).                    % 基底:一步到達
reachable(X, Y) :- edge(X, Z), reachable(Z, Y).   % 遞迴:先走到 Z,再從 Z 走到 Y
?- reachable(a, d).
true.            % a → c → d(或 a → b → c → d)

?- reachable(d, a).
false.           % 有向圖,走不回去

這其實就是 Prolog 版的「圖可達性」,而它用的搜尋正是 DFS。小心:如果圖裡有環 (例如再加 edge(d, a).),這個簡單版本會繞圈圈陷入無限遞迴—— 真正的解法要帶一個「走過的節點」清單,我們在進階應用一節會再碰到這個主題。

串列:Prolog 的 [Head|Tail]

Prolog 處理串列(list)的方式,是它最優雅的部分之一,而且和你第 7 章學的「自我參照資料」 是同一個思想。一個串列要嘛是空的 [],要嘛是「一個頭元素,接著一個更短的串列(尾)」。 Prolog 用 [Head|Tail] 這個記號把頭和尾拆開——那根直槓 | 唸作「接上」: 

?- [H|T] = [a, b, c].
H = a,
T = [b, c].              % 頭是 a,尾是剩下的 [b, c]

?- [X, Y | Rest] = [1, 2, 3, 4].
X = 1, Y = 2,
Rest = [3, 4].          % 可以一次拆出前兩個

?- [H|T] = [single].
H = single,
T = [].                 % 只有一個元素時,尾是空串列

看出那個「自我參照」了嗎?串列的定義裡又出現了串列(尾巴)。這意味著:幾乎所有串列操作, 都會寫成「處理頭 + 遞迴處理尾」的形狀——恰好對上第 7 章的設計配方。我們來親手實作幾個經典, 你會發現它們短得驚人,而且常常能「反著用」。

member:某元素在不在串列裡

% 基底:X 就是串列的頭 → 是成員(尾巴是什麼不重要,用底線 _ 忽略)
member(X, [X|_]).
% 遞迴:X 不是頭,那就看看它在不在尾巴裡
member(X, [_|T]) :- member(X, T).

那個底線 _ 是匿名變數(anonymous variable):「我不在乎這裡是什麼」。來看它怎麼用:

?- member(b, [a, b, c]).
true.

?- member(X, [a, b, c]).    % 反著用:列出串列的所有元素!
X = a ; X = b ; X = c.

第二個查詢展示了 Prolog 的雙向魔法:同一個 member,既能檢查成員,又能枚舉成員。 它怎麼枚舉?靠回溯。我們把 member(X, [a,b,c]) 的搜尋樹畫出來:

append:把兩個串列接起來

% 基底:空串列接上 L,結果就是 L
append([], L, L).
% 遞迴:把第一串的頭 H 留在結果的頭,剩下的遞迴接
append([H|T], L, [H|R]) :- append(T, L, R).
?- append([1, 2], [3, 4], Result).
Result = [1, 2, 3, 4].

但 append 真正驚人的,是反著用。把結果固定、讓前兩個參數當變數,Prolog 會幫你 窮舉所有「能拼出這個串列」的切法: 

?- append(Front, Back, [1, 2, 3]).
Front = [],        Back = [1, 2, 3] ;
Front = [1],       Back = [2, 3] ;
Front = [1, 2],    Back = [3] ;
Front = [1, 2, 3], Back = [].

一個你以為只會「接串列」的述詞,反過來變成了「拆串列」的工具——這是命令式函式做不到的。 在 Python 裡,a + b 永遠只能把 a 和 b 黏起來;你沒辦法問它「哪些 a 和 b 黏起來會是 [1,2,3]」。 宣告式的關係,天生就是多向的。

length 與 reverse

% 串列長度:空串列長度 0;非空串列 = 1 + 尾巴長度
len([], 0).
len([_|T], N) :- len(T, N0), N is N0 + 1.

% 反轉串列:用一個累積器(accumulator)邊走邊把頭堆到前面
reverse(L, R) :- rev(L, [], R).
rev([], Acc, Acc).                          % 走完了,累積器就是答案
rev([H|T], Acc, R) :- rev(T, [H|Acc], R).   % 把頭 H 疊到累積器最前面
?- len([a, b, c, d], N).
N = 4.

?- reverse([1, 2, 3], R).
R = [3, 2, 1].

len 裡出現了 is——這是做算術的關鍵字,而它牽出 Prolog 一個天大的陷阱, 我們下一節專門講。reverse 則示範了累積器這個 Prolog/函數式都愛用的技巧: 多帶一個參數當「進行中的成果」,邊遞迴邊把它疊好,走到底時它就是答案。

✏️ 練習 45.3 實作

(a) 實作 last_elem(List, X):X 是 List 的最後一個元素。(提示:用 [H|T] 與遞迴; 想想「只剩一個元素」的串列長什麼樣。)

(b) 用你已經有的 append,定義 prefix(P, L):P 是 L 的前綴。 想一想:為什麼一行就夠了?

看解答

(a) 基底是「串列只剩一個元素 [X]」,此時它就是最後一個;否則丟掉頭、遞迴看尾:

last_elem([X], X).                  % 只剩一個 → 它就是最後一個
last_elem([_|T], X) :- last_elem(T, X).   % 還有更多 → 看尾巴
?- last_elem([a, b, c], X).
X = c.

(b) 「P 是 L 的前綴」等於「存在某個尾段 _,讓 P 接上它會等於 L」—— 這正是 append 反用!

prefix(P, L) :- append(P, _, L).
?- prefix(X, [1, 2, 3]).
X = [] ; X = [1] ; X = [1, 2] ; X = [1, 2, 3].

(同理 suffix(S, L) :- append(_, S, L).。看到沒?寫好一個 append, prefix、suffix、甚至「子串列」都能用它組出來——這就是宣告式的複利。)

算術與比較:那個害死人的 =

前面我們一直在玩「關係」與「結構」,還沒做過真正的數字運算。原因是:Prolog 對算術的態度, 和你的直覺完全相反,這裡藏著新手最常踩的一顆地雷。先看正確的做法: 

?- X is 3 + 4 * 2.
X = 11.                  % is 會「計算」右邊的算式,把結果綁給 X

?- X is 10 / 3.
X = 3.3333333333333335.

要做算術,你必須用 is 這個關鍵字,它的意思是「計算右邊的算式,把數值結果統一到左邊」。 比較大小則用一組運算子(注意 Prolog 的「小於等於」寫成 =<,不是 <=!):

運算子意思範例
> <大於、小於5 > 3 為真
>= =<大於等於、小於等於3 =< 3 為真
=:=算術值相等1 + 2 =:= 3 為真
=\=算術值不相等2 =\= 3 為真

有了 is 與比較,就能寫遞迴的數值運算,例如階乘:

factorial(0, 1).                    % 基底:0! = 1
factorial(N, F) :-
    N > 0,                          % 守衛:只處理正數
    N1 is N - 1,                    % 先算出 N-1
    factorial(N1, F1),              % 遞迴求 (N-1)!
    F is N * F1.                    % 再乘回 N
?- factorial(5, F).
F = 120.
⛔
最致命的誤解:= 不是賦值,是統一!

來自 Python/C 的你,看到 = 會直覺以為是「賦值」或「計算後相等」。大錯。 在 Prolog 裡 = 是統一,它不會做任何算術:

?- X = 3 + 4.
X = 3+4.            % X 被綁成「3+4 這個算式結構」,不是 7!

?- X is 3 + 4.
X = 7.             % 這才會真的算

?- 7 = 3 + 4.
false.             % 數字 7 和結構 3+4 統一不起來,當然不相等

記住口訣:要算數值,用 is;要比對結構/綁定變數,用 =;要比算術值是否相等,用 =:=。 還有一個常見錯誤:N is N - 1 永遠失敗(一個邏輯變數不能既是 N 又是 N-1),所以才要寫成 N1 is N - 1 用一個新變數。在 Prolog 裡沒有「改變數的值」這回事——變數一旦綁定就是綁定了。

控制:cut 與否定

Prolog 預設會「盡責地找出所有解」,但有時候這份盡責反而礙事:你明明知道「找到一個就夠了」, 或想砍掉某些不必要的回溯。這時你需要兩個控制工具——它們很強,也很危險。

cut(!):剪掉回溯的剪刀

cut 寫成一個驚嘆號 !。它是一個「永遠成功」的特殊目標,但有個副作用: 一旦執行到它,就『凍結』在它之前的所有選擇,Prolog 不准回溯到它左邊去試別的候選。 它像一把剪刀,剪掉搜尋樹上的某些枝枒。經典用途是寫「最大值」: 

% 沒有 cut 的版本:有重複解的隱憂
max(X, Y, X) :- X >= Y.
max(X, Y, Y) :- X < Y.

% 有 cut 的版本:第一條成功就「定案」,不再嘗試第二條
max(X, Y, X) :- X >= Y, !.
max(_, Y, Y).

在 cut 版裡,如果 X >= Y 成立,! 就把這個選擇釘死:Prolog 確定答案是 X, 絕不會再回溯去碰第二條規則。這能避免重複解、也能省下無謂的搜尋。

否定即失敗(\+):「證不出來」就當成假

Prolog 用 \+ 表示否定(negation as failure):\+ Goal 的意思是 「如果 Goal 無法被證明為真,那麼 \+ Goal 就成功」。注意這個措辭很微妙: 不是「Goal 為假」,而是「Goal 證不出來」。例如: 

person(amy).  person(bob).  person(cara).
married(amy).

% 單身 = 是一個人,而且「無法證明他結婚了」
single(X) :- person(X), \+ married(X).
?- single(bob).
true.            % bob 是人,而知識庫裡證不出 married(bob) → 視為單身

?- single(amy).
false.           % married(amy) 證得出來 → 不單身
⚠️
封閉世界假設:沒寫的,就當成假

\+ 背後是封閉世界假設(closed-world assumption):Prolog 認為「知識庫裡推不出來的事情,就是假的」。 這跟現實不同——現實中「我不知道 bob 結婚沒」不等於「bob 沒結婚」,但 Prolog 會當成沒結婚。 這也讓 \+ 對變數順序敏感:\+ 裡若含有還沒綁定的變數,結果常常出乎意料 (例如 \+ married(X) 在 X 未綁定時,問的是「能不能找到任何已婚的人」,而非「X 單身」)。 穩妥的用法是:先把變數綁定好(像上面先 person(X)),再用 \+ 做最後的篩選。

cut 與否定是 Prolog 從「純邏輯」滑向「實用語言」的妥協:它們讓你掌控搜尋、表達「例外」與「預設」, 但也讓程式的意義開始依賴執行順序(背離了純宣告式的理想)。威力與危險,一體兩面—— 善用它們,但每加一個 ! 都要問自己:「我是不是剪掉了某個我其實需要的解?」

✏️ 練習 45.4 追蹤

不要用電腦,用紙筆畫出下列查詢的回溯搜尋樹,寫出 Prolog 會依序回報哪些解:

edge(a, b).
edge(a, c).
edge(b, d).

?- edge(a, X), edge(X, Y).
看解答

這是合取目標的回溯:先解左目標 edge(a, X),對每個解再解右目標 edge(X, Y)。

  • edge(a, X) 第一個解:X = b。 接著解 edge(b, Y) → Y = d。✓ 解 1:X = b, Y = d。
  • 按 ; 回溯:edge(b, Y) 沒有別的解了 → 回到左目標。
  • edge(a, X) 第二個解:X = c。 接著解 edge(c, Y) → 知識庫裡沒有 edge(c, ...),✗ 失敗。
  • 左目標也沒有更多解了 → 整個查詢結束。

所以 Prolog 只回報一個解:X = b, Y = d。畫成樹:

edge(a, X), edge(X, Y)
├── X = b → edge(b, Y) → Y = d   ✓ 解
└── X = c → edge(c, Y) → 失敗 ✗

關鍵心得:合取的回溯像「巢狀迴圈」——外層每換一個值,內層就從頭跑一遍;內層走完才換外層。 這正是第 30 章 DFS 在「兩個串起來的目標」上的樣子。

宣告式的威力:讓 Prolog 替你解難題

前面打好的地基,現在要收割了。Prolog 最迷人的地方在於:對某些問題,你只要把規則寫清楚, 「怎麼解」就免費附送。我們看三個經典場景,感受那種「我只描述了問題,答案就自己跑出來」的震撼。

解邏輯謎題:把線索變成約束

你一定看過那種「誰養什麼寵物」的邏輯謎題。最著名的是斑馬謎題(Zebra puzzle): 五棟不同顏色的房子、五國人、五種寵物、五種飲料、五種香菸,給你十幾條交錯的線索 (「英國人住紅房子」「養狗的人住在抽駱駝牌的人隔壁」……),問「斑馬是誰養的?」。 人類解這題要畫表格、反覆消去,費神得很。Prolog 的解法,本質上只是把每條線索翻成一個約束, 然後讓回溯去窮舉所有排列、自動淘汰違反約束的。我們用一個「迷你版」展示這個套路: 

% 三個人 amy/bob/cara,各養一種寵物:cat/dog/fish,每人不同。
% 線索:amy 不養 cat;bob 不養 dog;cara 養 fish。誰養什麼?

solve(Amy, Bob, Cara) :-
    Pets = [cat, dog, fish],
    member(Amy,  Pets),         % 從可能的寵物裡各挑一個給每個人
    member(Bob,  Pets),
    member(Cara, Pets),
    Amy  \= Bob, Bob \= Cara, Amy \= Cara,   % 三人寵物互不相同
    Amy  \= cat,                % 線索1
    Bob  \= dog,                % 線索2
    Cara =  fish.               % 線索3
?- solve(Amy, Bob, Cara).
Amy = dog,
Bob = cat,
Cara = fish.

你沒有寫任何「解題演算法」——沒有迴圈、沒有消去法、沒有 if-else。你只描述了「合法的解長什麼樣子」 (每人各挑一隻、互不相同、滿足三條線索),Prolog 的回溯引擎就替你窮舉 + 篩選,把唯一的解挖了出來。 這就是「產生並測試(generate and test)」——member 負責產生候選,後面的約束負責淘汰。 斑馬謎題只是同一招放大到 25 個變數而已。

圖著色:呼應第 15 章

還記得第 15 章的圖著色(graph coloring)嗎?替地圖上每個區域塗色,讓相鄰區域不同色。 這個出名的難題,用 Prolog 寫起來短得不可思議。以澳洲本土地圖(WA、NT、SA、Q、NSW、V 六個州)為例, 三種顏色夠不夠?直接問 Prolog: 

color(red).  color(green).  color(blue).

% 替六個州上色,讓地圖上每一對相鄰的州都「不同色」
coloring(WA, NT, SA, Q, NSW, V) :-
    color(WA), color(NT), color(SA), color(Q), color(NSW), color(V),
    WA  \= NT,  WA \= SA,           % WA 與 NT、SA 相鄰
    NT  \= SA,  NT \= Q,            % NT 與 SA、Q 相鄰
    SA  \= Q,   SA \= NSW, SA \= V, % SA 與 Q、NSW、V 相鄰
    Q   \= NSW,                     % Q 與 NSW 相鄰
    NSW \= V.                       % NSW 與 V 相鄰
?- coloring(WA, NT, SA, Q, NSW, V).
WA = red, NT = green, SA = blue, Q = red, NSW = green, V = red ;
... (還有更多合法上色,按 ; 繼續看)

路徑搜尋、專家系統,與更遠的地平線

圖的路徑搜尋(呼應第 30 章)在 Prolog 裡就是練習 45.2 的 reachable 再進一步—— 帶上一個「已造訪節點」清單來避免繞圈,你就能枚舉所有從 A 到 B 的路徑,連 DFS 程式碼都不必寫, 回溯本身就是 DFS。專家系統(expert system)則是 Prolog 在 1980 年代的招牌應用: 把領域專家的知識(「如果發燒且喉嚨痛,則可能是流感」)寫成一條條規則,系統就能根據症狀推論診斷—— 本質上就是我們的家族知識庫,換成醫學或法律的規則。

再往前看兩條地平線。其一,Prolog 內建一種叫 DCG(Definite Clause Grammar,確定子句文法)的語法糖, 讓你用幾乎和文法規則一樣的寫法做自然語言剖析(呼應第 42 章)——把「句子 → 名詞片語 + 動詞片語」 直接寫成 Prolog 規則,剖析就是查詢。其二,現代 Prolog(如 SWI-Prolog 的 library(clpfd))支援 約束邏輯程式設計(constraint logic programming),能把「產生並測試」升級成更聰明的 「約束傳播」,瞬間解開數獨、排程這類大型約束問題——那是「宣告式威力」的當代巔峰。

✏️ 練習 45.5 挑戰

用「產生並測試」的套路解一個小邏輯謎題:

三位朋友 小安、小本、小卡分別來自台北、台中、高雄(三人城市互不相同)。 已知:(1) 小安不是台北人;(2) 小本住在比小卡更北的城市(台北最北、高雄最南);(3) 小卡不是台中人。 請問各自來自哪裡?

寫出 Prolog 程式,讓它自己找出答案。

看解答

把城市由北到南編號(台北=北,可用一個 north_of/2 述詞表達「更北」),其餘照線索翻成約束:

city(taipei).  city(taichung).  city(kaohsiung).

% 「更北」關係:台北 比 台中、高雄 北;台中 比 高雄 北
north_of(taipei, taichung).
north_of(taipei, kaohsiung).
north_of(taichung, kaohsiung).

solve(An, Ben, Ka) :-
    city(An), city(Ben), city(Ka),
    An \= Ben, Ben \= Ka, An \= Ka,   % 三人城市互不相同
    An  \= taipei,                    % 線索1:小安不是台北人
    north_of(Ben, Ka),                % 線索2:小本比小卡北
    Ka  \= taichung.                  % 線索3:小卡不是台中人
?- solve(An, Ben, Ka).
An = taichung,
Ben = taipei,
Ka = kaohsiung.

推理:線索 3 + 城市互異 ⇒ 小卡是台北或高雄;線索 2 要小本比小卡北,若小卡台北就沒人比他北 ⇒ 小卡是高雄。 線索 1 ⇒ 小安非台北 ⇒ 小安台中 ⇒ 小本台北。Prolog 不需要你做這串推理, 它窮舉所有排列、用約束篩掉不合的,剩下的就是答案。把線索加到二十條,程式的「形狀」一模一樣—— 這就是斑馬謎題的縮影。

✏️ 練習 45.6 挑戰

替一個三角形圖(三個節點 a、b、c,兩兩相鄰)著色。 (a) 用兩種顏色 color(red). color(blue). 去 coloring(A, B, C),Prolog 會回傳什麼? 為什麼?(b) 改成三種顏色,結果如何?這呼應了第 15 章關於「色數(chromatic number)」的什麼結論?

看解答 
coloring(A, B, C) :-
    color(A), color(B), color(C),
    A \= B, B \= C, A \= C.   % 三邊兩端都不同色

(a) 兩色:

?- coloring(A, B, C).
false.

無解!因為三個節點兩兩相鄰,需要三種互異的顏色,但只有兩色可選——根據鴿籠原理(第 13 章), 三個節點塞進兩種顏色,必有兩個同色,而它們又相鄰,違反約束。Prolog 窮舉了全部 $2^3 = 8$ 種塗法, 全部回溯失敗,所以回 false。

(b) 三色:有解,例如 A = red, B = green, C = blue(以及它們的各種排列,共 $3! = 6$ 種)。

這正印證第 15 章的結論:一個三角形(亦即 3 個點的完全圖 $K_3$)的色數是 3—— 至少要三種顏色才塗得開。更一般地,$n$ 個點的完全圖 $K_n$ 色數恰為 $n$。 Prolog 在這裡不只「解了」著色問題,還幫你實驗出了色數:從少到多試顏色數,第一個有解的數目就是色數。

邏輯程式設計的定位:它跟 SQL 是親戚

學到這裡,你可能會想:這麼特別的範式,在真實世界用在哪?Prolog 與邏輯程式設計,在三個領域留下深刻足跡—— 而其中有一個,你其實已經很熟了。

  • 人工智慧:Prolog 是 1970–80 年代「符號 AI(symbolic AI)」的旗艦語言, 當年的專家系統、自然語言處理、自動推理大量用它。日本 1982 年豪賭的「第五代電腦計畫」,核心語言就是 Prolog。 雖然後來 AI 主流轉向統計與機器學習(第 38 章),但「用規則做邏輯推論」的思想仍活在現代的知識圖譜、規則引擎、 自動定理證明裡。
  • 形式驗證:要證明一段程式、一個協定、一塊晶片「絕對正確」(呼應第 13 章「測試 ≠ 證明」), 靠的正是邏輯推論。邏輯程式設計與它的近親(如 Datalog、各種證明助理)是這個領域的核心工具。
  • 資料庫查詢:這是最貼近你的一個。你在第 37 章學的 SQL, 骨子裡也是宣告式的!
🔑
SQL 與 Prolog:同一種精神

當你寫 SELECT name FROM students WHERE score > 90,你沒有告訴資料庫 「怎麼掃描資料表、要不要用索引、用哪種演算法」——你只描述了你想要什麼(分數大於 90 的學生姓名), 剩下的「怎麼查」交給資料庫的查詢最佳化器。這跟 Prolog「你描述事實與規則,引擎找答案」是同一種宣告式精神。 事實上,資料庫理論裡有一支叫 Datalog 的語言,根本就是「Prolog 的資料庫版」—— 家族知識庫的 parent 事實,就是一張資料表;grandparent 規則,就是一個遞迴的 view。 學會 Prolog,你會用全新的眼光看懂 SQL 為什麼長那樣。

換一顆腦袋

就算你這輩子再也不寫一行 Prolog,這一章也值得。因為它逼你換一顆腦袋看待「什麼是程式」。

在命令式的世界待久了,你會不自覺地把「寫程式」等同於「設計一串步驟」。但 Prolog 給你看了另一種可能: 程式可以是「一組關於真理的陳述」,而執行,就是「在這些真理裡搜尋答案」。 你不再扮演「指揮機器每一步」的工頭,而是扮演「定義遊戲規則」的立法者,然後讓推論引擎去玩這場遊戲。

這個視角會悄悄改變你寫任何語言的方式。當你下次面對一個複雜問題,你會多問一句: 「我能不能不要去想『怎麼一步步算』,而是先把『什麼樣的答案才算正確』描述清楚?」 這個問題,正是函數式程式設計(第 6 章)、SQL(第 37 章)、約束求解、甚至現代型別系統共享的智慧。 命令式教你駕馭機器,宣告式教你描述問題——一個成熟的計算機科學家,兩種腦袋都要有, 而且知道何時該切換。

💡
把這個習慣帶走

下次卡在一個演算法上時,試試「Prolog 式提問」:先別管步驟,先寫下「一個正確的答案,必須滿足哪些條件?」 很多時候,把問題的「規格(是什麼)」描述清楚,解法(怎麼做)就呼之欲出了。這是 Prolog 留給每個程式設計師最珍貴的禮物。

把這一章串起來

我們從第 1 章的「兩種知識」出發,讓宣告式那一半終於登上主舞台。回顧這趟旅程:

  • 邏輯程式設計讓你描述「是什麼」(事實 + 規則),由推論引擎推導「怎麼做」。口訣:Algorithm = Logic + Control。
  • Prolog 三件套:事實(無條件為真)、規則(結論 :- 條件,即帶變數的蘊含)、查詢(用變數提問,用 ; 要更多解)。
  • 統一是引擎的心臟:讓兩個項對應位置逐一匹配、綁定變數。大寫是變數,小寫是原子。
  • 推論的本質是回溯搜尋——就是第 30 章的 DFS,在邏輯推導樹上跑:深入、找解、撞牆、回溯、撤銷綁定。
  • 遞迴規則(ancestor)與 list 的 [H|T](member、append、reverse)呼應了第 3、7、13 章的遞迴與歸納;許多關係還能反著用。
  • 算術要用 is;= 是統一不是賦值(頭號陷阱)。cut(!) 剪枝、\+ 否定即失敗——強大但要小心。
  • 宣告式的威力:用「產生並測試」解邏輯謎題與圖著色(呼應第 15 章),幾乎不寫演算法。
  • 它的精神延伸到 AI、形式驗證,以及——你早就在用的 SQL(第 37 章):同樣是「你描述需求,系統找答案」。

最重要的收穫,或許不是 Prolog 本身,而是那顆被換過的腦袋:程式不只是步驟的編排,也可以是真理的描述。 帶著這個雙重視角,你看待計算的眼光,從此會更寬、更深。
