一個關於插頭的故事

你家牆上的插座有固定的形狀。三孔插座只收三腳插頭,圓孔插座只收圓腳插頭。 這不是設計師找你麻煩——它是一道安全機制:形狀對不上,你根本插不進去, 也就不會把 110V 的東西塞進 220V 的孔、把訊號線插進電源孔。形狀,事先就替你擋掉了一整類災難。

程式裡也有一模一樣的東西。每一個函式的參數,就像一個插座,它「期待」收到某種形狀的值; 每一個值,就像一個插頭,它自己有某種形狀。而「檢查插頭形狀符不符合插座」這件事, 在程式語言裡有個正式的名字,叫型別檢查(type checking)。那個決定「形狀」的東西,就是——

🔑
型別是什麼

型別(type)= 一組值(values)+ 一組能對這些值做的操作(operations)。
例如 int 這個型別,它的「值」是一堆整數 …、-1、0、1、2、…;它的「操作」是加減乘除、比大小等。 str 的值是各種字串,操作是連接、取子字串、轉大寫等。型別不只是「這是什麼資料」, 更重要的是「你可以對它做什麼、不可以做什麼」。

這個定義有個立刻的推論:有些操作對某些型別根本沒有意義。把兩個字串相「減」是什麼意思? "蘋果" - "香蕉" 等於什麼?沒有答案,因為「減法」不在字串的操作清單裡。 型別系統的工作,就是在你犯這種錯之前(或之後)抓住你。

為什麼需要型別?

想像一個沒有任何型別概念的世界:所有東西都只是一串位元(bits),你想對它做什麼操作都行。 聽起來很自由,但這種自由是有毒的。你可以把一個「人的年齡」當成記憶體位址去跳轉, 可以把一張「圖片」拿去做開根號,可以把「使用者名稱」這個字串拿去做減法——機器照單全收,然後給你一堆垃圾,或直接崩潰。

型別替這個混亂的世界畫上界線。它至少幫你三件事:

  • 擋掉一整類錯誤:「把字串拿去做減法」這種錯,根本不該被允許發生。型別讓它變成「不合法」, 而不是「跑出莫名其妙的結果」。
  • 當成文件:當你看到 def send(to: str, body: str) -> bool,你不必讀任何內文, 就知道「丟兩個字串進去,會拿回一個是否成功的布林值」。型別簽名是一行永遠不會過期的註解—— 因為如果它跟程式不符,(在靜態語言裡)根本編譯不過。
  • 當成一種輕量級的證明:一個通過型別檢查的程式,等於證明了「這裡絕不會發生型別不符的操作」。 這聽起來不起眼,但它是你不必親自測試、就能確定成立的性質。型別越強,你能免費證明的東西就越多。
📜
型別,其實就是邏輯的證明

「型別是一種證明」不只是比喻。1930–60 年代,邏輯學家發現了一件深刻到令人起雞皮疙瘩的事: 程式的型別,與數學的邏輯命題,是同一個東西的兩種說法;而一個「型別正確的程式」, 恰好對應到「該命題的一個證明」。這叫做 Curry–Howard 對應(Curry–Howard correspondence)。 它是為什麼像 Coq、Agda 這類「證明助手」可以同時是程式語言:在它們裡面,寫程式跟證定理,字面上是同一件事。 本章我們只取它最樸素的精神:把性質編碼進型別,讓編譯器替你驗證。

何時檢查型別?靜態 vs 動態

現在來到本章第一條主軸。所有語言都會做型別檢查(否則 "5" - 3 不知該如何是好), 差別在於什麼時候檢查:

  • 靜態型別(static typing):在程式執行之前,由編譯器逐行檢查所有插頭與插座。 形狀不符,直接拒絕編譯——你連跑都跑不起來。C、C++、Java、Rust、Haskell 都是這一派。
  • 動態型別(dynamic typing):平時不檢查,等到程式真的執行到某個操作的那一刻, 才檢查「這個值現在的型別,支不支援這個操作」。不支援,就在那一瞬間丟出錯誤。Python、JavaScript、Ruby 屬於這一派。

這個差別聽起來抽象,但用同一個 bug 在兩種語言裡的命運一對照,你會瞬間懂。 我們來寫一個常見的真實錯誤:程式從外界拿到的輸入其實是字串(這在現實中極常見—— 鍵盤輸入、網頁表單、設定檔讀進來的全都是字串),卻直接拿去做算術。

先看 Python(動態):

def discount(price):
    return price - 10          # 想做「打折」:減 10 元

user_input = "100"             # 注意:從 input()/表單拿到的永遠是「字串」
print(discount(user_input))
# 程式照樣啟動、照樣跑……跑到上面那行才「砰」:
# TypeError: unsupported operand type(s) for -: 'str' and 'int'

關鍵在於:這個程式完全能啟動。它可能順利跑過前面九百行,直到某個使用者的某次操作走到 discount, 才當著他的面爆炸。如果這行藏在一個很少被觸發的分支裡(例如「使用者剛好用了優惠券」), 它甚至可能上線好幾個月都沒人發現——直到某天半夜把你叫醒。

現在看同一個 bug 在 C(靜態):

#include <stdio.h>

int discount(int price) {       /* 插座明寫:只收 int */
    return price - 10;
}

int main(void) {
    char *user_input = "100";   /* 字串 */
    printf("%d\n", discount(user_input));   /* ← 編譯器在這裡就翻臉 */
    return 0;
}
/* 編譯時(不是執行時!)就報錯,例如:
   error: incompatible pointer to integer conversion
          passing 'char *' to parameter of type 'int' */

在 C 裡,你永遠跑不到那個 bug——因為編譯器根本不讓你產生出可執行檔。 錯誤在你按下「編譯」的那一秒就被攤在桌上,而不是在正式環境裡、在最糟的時間點才現形。 這就是靜態與動態最核心的氣質差異。

🔑
靜態 = 事前體檢,動態 = 事後急診

靜態型別像出國前的健康檢查:慢、有點煩、會擋下一些其實沒事的人,但問題在出發前就被發現。 動態型別像沒體檢就出發,平常沒事跑得很順,真出狀況時人已經在國外了——而且可能是最不方便的時刻。 兩者沒有絕對的優劣,只有適不適合。

各自的優缺點

明白了「何時檢查」,優缺點就幾乎是自動推導出來的:

靜態型別(C / Java / Haskell)動態型別(Python / JavaScript)
抓錯時機編譯期就抓到一大類錯誤,跑都不必跑得等到執行到那一行,且只在「真的走到」時才現形
文件性型別簽名是永遠正確的文件,IDE 自動完成精準得靠命名、註解或測試去猜資料長什麼樣
重構安全改一個型別,編譯器替你標出所有受影響處大型專案重構像走鋼索,容易漏改
開發速度前期較囉嗦,編譯-修正循環較慢寫得快、改得快,極適合原型與探索
彈性有些「其實正確」的程式會被型別系統拒絕鴨子型別(duck typing):能動就好,自由度高
執行效能編譯器可依型別最佳化,通常較快執行期才確定型別,通常較慢、較吃記憶體
需要的測試型別已涵蓋一層,測試可專注在邏輯需要更多測試,來補回型別免費給的那層安全網
🦆
什麼是「鴨子型別」?

動態語言常奉行一句話:「如果牠走起來像鴨子、叫起來像鴨子,那牠就是鴨子。」(duck typing)。 意思是:Python 不在乎一個物件「宣稱」自己是什麼型別,只在乎「你要呼叫的那個方法它有沒有」。 只要一個物件有 .read() 方法,任何期待「可讀取物件」的函式就收它——管它是檔案、網路連線還是記憶體緩衝區。 這帶來巨大的彈性,代價就是:「有沒有那個方法」要到執行期才知道。

怎麼轉換型別?強 vs 弱

現在來到第二條主軸,而它是初學者最容易跟第一條搞混的地方。「靜態/動態」談的是什麼時候檢查; 「強/弱」談的是當型別對不上時,語言有多願意偷偷幫你轉換(隱式轉型,implicit conversion)。

  • 強型別(strongly typed):不亂幫你轉。型別對不上,就老實報錯,要你自己講清楚想怎麼轉。
  • 弱型別(weakly typed):很「貼心」地自作主張,幫你把一邊轉成另一邊,好讓操作「能進行下去」—— 即使結果常常不是你要的。

弱型別的「貼心」是甜蜜的陷阱。經典的反面教材是 JavaScript: 

"5" + 3      // "53"   ← 遇到 +,只要有一邊是字串,就當「字串連接」,把 3 轉成 "3"
5 + "3"      // "53"   ← 同上,順序顛倒結果一樣
"5" - 3      // 2      ← 但 - 沒有「字串版」!於是把 "5" 偷偷轉成數字 5,得到 2
"5" * "2"    // 10     ← 兩個都是字串,卻都被轉成數字相乘

看出恐怖之處了嗎?同一個值 "5",遇到 + 是字串、遇到 - 卻變數字。 語言的「自作主張」讓行為變得難以預測,而這類隱式轉型製造的 bug 又特別陰險—— 因為程式不報錯,只是默默給你一個錯的答案。

對照之下,Python 是強型別(它動態,但強): 

"5" + 3
# TypeError: can only concatenate str (not "int") to str
# Python 拒絕猜你的意思,要你自己決定:要 "5" + str(3),還是 int("5") + 3?

而 C 是靜態,卻偏弱:它在編譯期檢查型別,但對數值之間的隱式轉型相當大方, 這正是無數底層 bug 的溫床。看一個惡名昭彰的例子——有號數與無號數比較: 

#include <stdio.h>

int main(void) {
    int i = -1;
    unsigned int u = 1;

    if (i < u) {                       /* 直覺:-1 < 1,當然成立 */
        printf("i 比較小\n");
    } else {
        printf("居然沒進去?!\n");      /* ← 實際印出這行 */
    }
    return 0;
}

為什麼 -1 < 1 會是假的?因為 C 為了讓兩邊「型別一致」, 偷偷把 i(-1)轉成無號數,於是它變成一個天文數字(在 32 位元下是 4294967295), 當然不小於 1。這個轉換你看不見、沒人警告你(預設不會),它就靜靜地把你的程式邏輯顛倒過來。 這就是「弱」的代價。

⚠️
隱式轉型是「方便」與「災難」之間最細的一條線

弱型別的隱式轉型,寫起來確實少打幾個字,但它把「型別不符」這個明確的訊號, 悄悄換成了「一個你沒料到的值」。在 C/C++ 裡,養成盡量明寫轉型(explicit cast)、 打開編譯器警告(-Wall -Wextra)的習慣;在 JavaScript 裡,用 === 而非 == (前者不做隱式轉型),都是在替自己拆掉這顆地雷。

關鍵:這是兩條獨立的軸

現在把上面四個例子排在一起,你會發現一件很多人從沒意識到的事:「動態/靜態」和「強/弱」是兩條彼此獨立的軸, 一個語言在這兩條軸上的位置可以任意組合。Python 是「動態 + 強」,C 是「靜態 + 弱」, 它們在兩條軸上都站在不同邊。把四個象限畫出來:

ℹ️
「強/弱」沒有教科書級的精確定義

誠實說:相較於「靜態/動態」有清楚的界線(編譯期 vs 執行期),「強/弱」更像是個口語的光譜, 不同人用法略有出入。本書採用最通行的版本:強 = 不輕易隱式轉型、型別不符就報錯;弱 = 大量自動轉型。 你只要抓住「它跟動靜是不同的事」這個重點就夠了。

型別推論:編譯器自己想出來

靜態型別最大的抱怨是「囉嗦」——好像每個變數都得手動標型別。但這其實是個過時的印象。 現代的靜態語言有一招厲害的功夫叫型別推論(type inference):你一個型別都不寫, 編譯器卻能自己推導出每個東西的型別,而且仍然是貨真價實、編譯期檢查的完整靜態型別。

這招的源頭是 ML 家族(包含 Haskell、OCaml、SML)所用的 Hindley–Milner 型別推論。 它的直覺其實很像解一道代數聯立方程式:從你怎麼使用每個值,反推它必須是什麼型別。看一段 Haskell: 

-- 你一個型別都沒寫:
double x = x + x

-- 但編譯器推論出完整型別:
-- double :: Num a => a -> a
-- 它的推理:x 被用在 (+) 上 → x 必須是「能相加的數」;
--          回傳 x + x,所以回傳型別跟 x 一樣。

再看一個更有說服力的例子——一個處理串列的函式,完全沒寫型別,卻被推得明明白白:

-- 取串列的第一個元素,沒寫任何型別
firstOf (x : _) = x

-- 編譯器推論:firstOf :: [a] -> a
-- 「吃一個任意型別 a 的串列,回傳一個 a」——連泛型都自動幫你推出來了!

這就是 ML 家族讓人驚艷的地方:程式碼看起來像動態語言一樣乾淨,骨子裡卻有最嚴格的靜態型別保護。 你享受了動態的簡潔,又沒放棄靜態的安全。(SML 的味道也類似:fun double x = x + x 會被推成 int -> int。)

型別推論並非 ML 家族的專利,只是程度不同。C++ 從 C++11 起有 auto,Java 有 var: 

// C++:auto 讓編譯器從右邊推出左邊的型別
auto count = 42;              // 推成 int
auto name  = std::string("Akira");  // 推成 std::string
auto pi    = 3.14159;         // 推成 double
// 注意:這仍是「靜態型別」!count 一旦被推成 int,就永遠是 int,
// 你不能之後拿它去裝字串——這跟動態語言的「變數可隨時換型別」完全不同。

甚至連動態的 Python 也在玩這個遊戲,只是方向相反:它本來沒有型別宣告, 後來加上了「型別提示(type hints)」,讓你可以選擇性地把型別寫上去,給工具與讀者看。 這個我們稍後會專門細談。

💡
「不寫型別」不等於「沒有型別」

很多人以為「沒看到型別宣告 = 動態語言」。錯。Haskell 滿屏看不到型別,卻是型別系統最森嚴的語言之一。 關鍵分野永遠是那句老話:型別是在「跑之前」就釘死(靜態),還是在「跑的時候」才確定(動態)? 型別推論只是讓靜態語言「看起來」清爽,本質一點都沒鬆動。

代數資料型別:用型別拼出新型別

到目前為止,我們的型別都是現成的:int、str、bool。但程式設計真正的力量, 在於你能自己組合出新型別。組合的方式只有兩種基本積木,合稱代數資料型別(algebraic data type, ADT)。 名字裡有「代數」,因為這兩種組合恰好對應乘法與加法——等一下你會看到這個對應美得不可思議。

積型別(product type):「而且」

第一種積木:把幾個型別綁在一起、同時擁有。一個「點」同時有 x 而且有 y; 一個「人」同時有姓名、年齡、身高。這叫積型別,你早就見過它的各種化身: tuple(Python)、struct(C)、record、物件的欄位…… 

-- Haskell:一個 Point 同時有 x 和 y(積型別)
data Point = Point Double Double

-- C 的對應物:struct
-- struct Point { double x; double y; };

和型別(sum type):「或」

第二種積木更有意思,也是多數主流語言長期缺席的:一個值是幾種可能性之一(其中一種,而非全部)。 一個「形狀」是圓形,或是矩形,或是三角形——同一時間只會是其中一種。這叫和型別: 

-- Haskell:一個 Shape「是」圓、矩形、或三角形之一(和型別)
data Shape = Circle Double            -- 圓:帶一個半徑
           | Rectangle Double Double  -- 矩形:帶寬與高
           | Triangle Double Double   -- 三角形:帶底與高
-- 每個「|」都是一個 variant(變體 / 構造子)。
-- 注意每個變體可以帶自己的資料,且資料的「形狀」可以完全不同。

為什麼叫「積」與「和」?這是 ADT 最漂亮的一刻。數一數每個型別到底有幾個可能的值:

  • bool 有 2 個值(True/False)。一個 (bool, bool) 的積型別有 2 × 2 = 4 個值。 欄位的值「相乘」——所以叫積。
  • 一個「bool 或 一個空值 ()」的和型別,有 2 + 1 = 3 個值。 各變體的值「相加」——所以叫和。
🔑
積 = 而且,和 = 或

積型別(struct/tuple):「我同時擁有 A 而且 B 而且 C。」
和型別(enum/union/Option):「我是 A 或 B 或 C 其中之一。」
幾乎任何真實世界的資料,都是這兩者層層套疊出來的。能流暢地用「積與和」去思考資料, 是區分初學者與成熟工程師的一道分水嶺。

Option / Maybe:用和型別消滅 null

和型別最有名、最改變人生的應用,是消滅程式設計史上最貴的 bug 來源:null。 想想一個「在資料庫裡找使用者」的函式——它可能找到,也可能找不到。傳統作法是「找不到就回傳 null」, 但這埋下一個地雷:呼叫者很容易忘記檢查 null,直接拿去用,然後程式就 NullPointerException / 當機。

和型別的解法是:設計一個型別,明確地說「這裡可能有值,也可能沒有」, 並且強迫呼叫者面對「沒有」的情況。這個型別在 Haskell 叫 Maybe,在 ML/Rust 叫 Option: 

-- 標準庫裡就是這樣定義的(a 是任意型別,這也是泛型——下一節談):
data Maybe a = Nothing      -- 沒有值
             | Just a       -- 有值,值就在 Just 裡面

-- 安全的除法:除以 0 時,誠實回傳「沒有結果」
safeDiv :: Int -> Int -> Maybe Int
safeDiv _ 0 = Nothing
safeDiv a b = Just (a `div` b)

魔法在於:safeDiv 10 2 的型別是 Maybe Int,不是 Int。 所以你不能直接拿它去做加減乘除——你必須先「拆開」它,而拆開時編譯器會逼你同時處理 Just 和 Nothing 兩種情況。 「忘記檢查」這個錯誤,在型別層次上被消滅了。怎麼拆?這正是下一節主角:模式比對。

📜
「十億美元的錯誤」

null 參考的發明人東尼・霍爾(Tony Hoare,著名的快速排序作者)在 2009 年的一場演講中公開懺悔: 「我把它叫做我的十億美元錯誤(billion-dollar mistake)……1965 年我設計了第一個 null 參考, 純粹因為它很好實作。但它後來造成了數不清的錯誤、漏洞與系統崩潰,過去四十年大概造成了十億美元的損失。」 和型別(Option/Maybe)就是業界對這個錯誤的集體解答。

Python 怎麼模擬 ADT?

Python 沒有內建 ADT 的語法,但用幾個工具能拼得很像。積型別用 dataclass; 和型別用 Enum(若各變體不帶資料)或 Union(若帶不同資料); 「可能沒有值」則用 Optional[T](等價於 T | None): 

from dataclasses import dataclass

# 積型別:同時有 width 和 height(用 dataclass)
@dataclass
class Circle:
    radius: float

@dataclass
class Rectangle:
    width: float
    height: float

# 和型別:一個 Shape「是」Circle 或 Rectangle(用 Union,寫成 | )
Shape = Circle | Rectangle

# 「可能沒有值」:Optional,等價於 int | None,這就是 Python 版的 Maybe/Option
def safe_div(a: int, b: int) -> int | None:
    if b == 0:
        return None
    return a // b

若你的和型別各變體不帶資料,只是「一組固定選項」(例如紅綠燈的三種狀態), 用 Enum 最乾淨——它也是一種和型別: 

from enum import Enum, auto

class Light(Enum):     # 一個 Light「是」紅、黃、綠 其中之一(和型別)
    RED = auto()
    YELLOW = auto()
    GREEN = auto()

模式比對:拆開代數資料型別

我們在第 6 章(函數式程式設計)初次見到模式比對(pattern matching)。 在這裡,你會看到它與和型別簡直是天作之合——它正是「安全地拆開一個和型別」的標準方式。

回到 Haskell 的 Shape。要算面積,你必須對「圓 / 矩形 / 三角形」分別處理。 模式比對讓你一次把所有變體攤開,並順手把裡面的資料解開來綁定到變數: 

area :: Shape -> Double
area (Circle r)      = 3.14159 * r * r     -- 是圓:把半徑解出來叫 r
area (Rectangle w h) = w * h               -- 是矩形:解出寬 w、高 h
area (Triangle b h)  = 0.5 * b * h         -- 是三角形:解出底 b、高 h

同樣的事,用 Python 3.10+ 的 match:

def area(s: Shape) -> float:
    match s:
        case Circle(radius=r):
            return 3.14159 * r * r
        case Rectangle(width=w, height=h):
            return w * h
    raise ValueError("未知的形狀")

再看怎麼用模式比對安全地拆開 Maybe/Optional——這就是 null 被消滅的具體現場:

describe :: Maybe Int -> String
describe Nothing  = "沒有結果"          -- 你「被迫」處理沒有值的情況
describe (Just x) = "結果是 " ++ show x  -- 只有在這個分支,x 才存在、才拿得到

看出關鍵了嗎?你無法跳過 Nothing 那一行直接用 x——x 根本不存在於 Nothing 的世界裡。 型別系統把「有可能沒有值」這件事,變成你不得不處理的東西。

🔑
窮盡性檢查:ADT + 模式比對最強的維護力

在 Haskell、Rust、OCaml 這類語言裡,如果你的 match 漏掉了某個變體, 編譯器會直接警告或報錯。這帶來一個驚人的好處:當你未來替 Shape 新增一個 Pentagon(五邊形), 編譯器會立刻替你標出程式裡每一處還沒處理五邊形的 match。 你不必靠記性、不必靠測試去找——型別系統替你維護了一份「待辦清單」。 這就是為什麼資深工程師如此偏愛和型別。(Python 的 match 預設不做窮盡性檢查,但 mypy 在某些設定下能補上。)

泛型:一次寫好,處處可用

現在回頭看一個被我們略過的細節。剛才那個取串列首元素的 Haskell 函式,型別是 [a] -> a, 那個小寫的 a 是什麼?它是個型別變數——代表「任何型別都行」。 這個能力叫參數多型(parametric polymorphism),在主流語言裡更常被叫做泛型(generics)。

泛型解決的痛點很實際。假設你要寫一個「堆疊(stack)」資料結構。難道整數堆疊、字串堆疊、物件堆疊 要各寫一份一模一樣、只是型別不同的程式?那是 DRY 原則的災難。泛型讓你寫一次,適用所有型別: 

# Python:用 TypeVar 宣告型別變數 T
from typing import TypeVar, Generic

T = TypeVar("T")            # T 代表「某個型別,呼叫時才決定」

class Stack(Generic[T]):
    def __init__(self) -> None:
        self._items: list[T] = []

    def push(self, item: T) -> None:
        self._items.append(item)

    def pop(self) -> T:
        return self._items.pop()

# 使用時才把 T「填」進去:
numbers: Stack[int] = Stack()    # 一個只裝 int 的堆疊
numbers.push(1)
numbers.push(2)
# numbers.push("oops")   ← mypy 會抓到:期待 int,卻給了 str

其他主流語言用角括號 <...> 來寫泛型。同一個概念,三種腔調: 

// C++:用 template 寫泛型函式
template <typename T>
T max_of(T a, T b) {
    return (a > b) ? a : b;
}
// max_of(3, 5) 用 int,max_of(2.5, 1.0) 用 double——編譯器各生一份
// Java:用 <T> 寫泛型類別
class Box<T> {
    private T value;
    public void set(T v) { value = v; }
    public T get() { return value; }
}
// new Box<String>() 是裝字串的盒子;List<Integer> 是裝整數的串列

雖然概念相同,三種語言實作泛型的哲學卻大不相同,這個差別會影響效能與錯誤訊息:

語言泛型怎麼實作後果
C++ template編譯期為每個用到的型別各生成一份專屬程式碼(單型化 monomorphization)執行最快(無額外成本),但編譯較慢、錯誤訊息惡名昭彰地冗長
Java generics編譯後把型別資訊抹除(type erasure),執行期其實只剩 Object相容舊版,但執行期拿不到泛型型別、有些操作受限
Python TypeVar純粹給靜態檢查器(mypy)與讀者看,執行期完全不檢查零執行成本也零執行保護,安全網全靠你跑不跑 mypy
🔍
為什麼角括號要在 HTML 裡跳脫?

(給也在寫筆記的你)上面 C++ 的 template <typename T>、Java 的 Box<T>、List<String>, 在原始碼裡是 < 和 >,但瀏覽器會把它們當成 HTML 標籤的開頭,導致畫面壞掉。 所以寫網頁時必須把它們跳脫成 &lt; 與 &gt;。Python 用方括號 List[T] 就沒這個困擾—— 這也是 Python 型別語法的一個小確幸。

型別安全,以及型別擋不住的事

我們花了一整章歌頌型別,該誠實地談談它的邊界了。先給「型別安全」一個定義:

🔑
型別安全(type safety)

一個型別安全的語言,保證程式不會執行「型別系統認定為非法」的操作—— 例如把整數當成函式去呼叫、把字串當成記憶體位址去跳轉。羅賓・米爾納(Robin Milner)有句名言: 「Well-typed programs cannot go wrong.」(型別正確的程式不會出錯。) 但關鍵在於:這裡的「出錯(go wrong)」有一個精確而有限的定義——它指的只是「型別層次的非法操作」, 不包括你的邏輯寫錯。

這個「有限」非常重要,因為它是無數人對型別的誤解來源。型別系統擋不住的事,多到超乎想像: 

# 這個函式型別「完全正確」:吃兩個 float,回傳一個 float
def area(width: float, height: float) -> float:
    return width + height        # ← 邏輯大錯!面積是「乘」,不是「加」

# 型別檢查器看了:型別?完美。它對你的「邏輯」一無所知。
print(area(3.0, 4.0))            # 印出 7.0,而正確答案是 12.0

型別管不到的還有一長串:把 width 和 height 傳反了(型別都是 float,看不出來)、 除以零、陣列索引越界(多數語言)、無窮迴圈、競態條件、商業規則寫錯、把錢算成負的…… 這些 bug 全都「型別正確」,卻全都是 bug。

⚠️
型別不是萬靈丹,測試不能省

「我用了強型別語言,所以不太需要測試」是危險的錯覺。型別系統證明的是「結構不會錯」, 卻完全管不到「邏輯對不對」。把型別想成第一道篩網:它低成本地濾掉一大類低級錯誤, 讓你的測試與心力,能專注在它濾不掉的那些——真正關乎正確性的邏輯。型別與測試是隊友,不是替代品。

話說回來,「型別能保證多少」其實是可以往上推的,而這正是型別系統研究最迷人的前沿: 型別越強,你能編碼進去、讓編譯器替你證明的性質就越多。Rust 把「記憶體安全」與「沒有資料競爭」 塞進了型別系統(它的 borrow checker);相依型別(dependent types)的語言甚至能讓型別保證「這個陣列存取絕不越界」。 但天下沒有白吃的午餐——由於停機問題那一類的理論極限(回憶第 1 章), 沒有任何型別系統能自動證明「任意程式的任意性質」。型別的疆界,終究也是計算的疆界。

Python 的型別提示與 mypy:漸進式型別

如果靜態與動態各有千秋,有沒有可能兩個都要?這正是 Python 從 3.5 起走的路: 保留動態的本性,但讓你選擇性地加上型別標註,再用外部工具做靜態檢查。 這種「想嚴格的地方就嚴格、想隨意的地方就隨意」的折衷,叫漸進式型別(gradual typing)。

先看怎麼加型別提示(type hints)。語法就是在參數和回傳值後面標型別: 

# greet.py
def greet(name: str) -> str:         # 吃一個 str,回傳一個 str
    return "哈囉," + name

age: int = 25
print(greet(age))                    # 我們不小心傳了 int 進去

這裡有個務必記住的重點:型別提示不影響 Python 的執行。 Python 直譯器基本上把它們當註解略過。所以上面這支程式直接 python greet.py 仍會啟動, 然後在 "哈囉," + age 那行因為「字串 + 整數」當場丟 TypeError—— 型別提示本身不會幫你擋下任何東西。

真正的魔法來自一個獨立工具:mypy(或 Pyright 等)。它讀你的型別提示, 在程式跑之前幫你做一次靜態檢查,等於替動態的 Python 補上一道編譯期的安全網: 

$ mypy greet.py
greet.py:5: error: Argument 1 to "greet" has incompatible type "int"; expected "str"  [arg-type]
Found 1 error in 1 file (checked 1 source file)

漂亮——錯誤在「跑之前」就被指名道姓地抓出來,連行號都給你。mypy 還能抓到更隱晦、 不一定會當機但邏輯肯定錯的問題,例如「把結果存進型別不對的變數」: 

def first(items: list[int]) -> int:
    return items[0]

result: str = first([1, 2, 3])   # 我宣告 result 是 str,卻塞了 int 進去
# mypy: error: Incompatible types in assignment
#       (expression has type "int", variable has type "str")

漸進式型別最甜美的應用,還是回到我們的老朋友 null。用 int | None 標出「可能沒有值」, mypy 就會強迫你先檢查 None,把 Maybe/Option 的精神帶進 Python: 

def find_user(uid: int) -> str | None:    # 可能找到名字,也可能找不到
    ...

name = find_user(7)
print(name.upper())     # mypy: error: Item "None" of "str | None" has no attribute "upper"

# 正確寫法:先處理 None 的情況,mypy 才放行
if name is not None:
    print(name.upper())     # 在這個區塊裡,mypy 知道 name 一定是 str
💡
怎麼在你的專案裡用漸進式型別?

漸進式的精髓是「不必一次到位」。實務上的策略:新寫的程式碼一律加型別提示; 舊程式碼從最核心、最常被呼叫的模組開始慢慢補;把 mypy 接進 CI(持續整合), 讓它在每次提交時自動把關。你會驚訝於它幫你攔下多少「上線才會爆」的低級錯誤—— 這幾乎是動態語言能拿到的、CP 值最高的一筆投資。

C 的型別世界:typedef、struct 與指標

本書的慣例是用 C 看「底層真相」。C 是靜態 + 偏弱的型別系統——編譯期檢查, 但對數值間的隱式轉型相當縱容(還記得那個有號/無號的坑嗎)。這一節快速認識 C 裡幾個關鍵的型別工具, 它們也是理解第五部「計算機系統」的伏筆。

typedef:替型別取別名

typedef 讓你給一個型別取個更好懂的名字,純粹是為了可讀性: 

typedef unsigned long long u64;   /* 以後寫 u64 就等於 unsigned long long */
u64 big = 18000000000ULL;         /* 清爽多了 */

struct:C 的積型別

struct 把幾個欄位綁成一個新型別——它就是 C 版的積型別,常搭配 typedef 一起用: 

#include <stdio.h>

typedef struct {
    char   name[20];
    int    age;
    double height;
} Person;            /* 積型別:同時有 name、age、height */

int main(void) {
    Person p = {"小明", 18, 172.5};
    printf("%s,%d 歲,%.1f 公分\n", p.name, p.age, p.height);
    return 0;
}

union:危險版的和型別

C 也有「或」的概念,叫 union——同一塊記憶體,可以當成幾種型別之一來解讀。 但它跟 ML 的安全和型別有個致命差別:它沒有「標籤」告訴你現在裝的到底是哪一種: 

union Value {
    int    i;
    double d;
    char  *s;
};   /* 同一塊記憶體,三種解讀方式之一——但沒人替你記住「現在是哪種」 */

union Value v;
v.i = 42;            /* 現在裝的是 int */
double x = v.d;      /* ← 災難:把那塊位元當成 double 讀,得到垃圾值 */

這就是「裸 union」的危險:你存進去的是 int,卻沒東西阻止你把它當 double 讀出來。 安全的和型別需要一個額外的標籤(tag)來記住目前是哪個變體,這種寫法叫 tagged union(標籤聯合)。而 ML/Haskell 的 datatype 之所以安全, 正是因為它內建幫你管好了這個標籤,還用模式比對逼你正確地拆——這就是為什麼它們的和型別優雅又安全。

指標型別:型別決定「怎麼解讀位元」

C 最能體現「型別 = 怎麼解讀一段位元」這個本質的,是指標型別。 一個指標的型別,不只說「它是個位址」,還說「位址那頭,該怎麼解讀」: 

int   x = 42;
int  *p = &x;       /* p 的型別是「指向 int 的指標」:讀 *p 時讀 4 個位元組當 int */
char *s = "hi";     /* s 的型別是「指向 char 的指標」:讀 *s 時讀 1 個位元組當 char */

同樣一個記憶體位址,透過 int * 看是一個四位元組的整數,透過 char * 看卻是一個位元組的字元。 型別,就是那副決定你「看見什麼」的眼鏡。這個「型別其實只是『如何詮釋一串位元』」的深層真相, 我們會在第五部「計算機系統」裡掀開記憶體的蓋子,講得淋漓盡致。現在,你只要記得這個埋下的伏筆。

🔍
同一個靈魂,跨越所有語言

從 Haskell 的 data、Python 的 dataclass、到 C 的 struct 與 union, 你會發現「積型別」與「和型別」這兩個概念無所不在,只是換了腔調。 這正是第 1 章那句「同一個靈魂,不同軀殼」的延續——學會用型別的本質(值的集合 + 操作 + 怎麼組合)去看, 你就能在任何新語言裡迅速認出老朋友。

Java靜態 + 強(偏強)編譯期檢查;雖有少量數值自動升級,但不會把字串硬轉成數字
C靜態 + 弱編譯期檢查型別,但數值隱式轉型大方(有號/無號、縮窄轉換都默默發生)
C++靜態 + 弱同 C 的傳統;雖比 C 多了型別工具,隱式轉型的歷史包袱仍在
Python動態 + 強執行期才檢查,但 "5" - 3 會直接丟 TypeError,拒絕亂轉
JavaScript動態 + 弱執行期才檢查,且 "5" - 3 === 2、"5" + 3 === "53",隱式轉型滿天飛

最容易答錯的是把 Python 當成「弱」。記住:Python 動態但強——它不像 JS 那樣偷偷幫你轉, 這跟 C「靜態但弱」恰好對角。能分清這兩條軸,你就贏過一大半人了。

✏️ 練習 9.2 實作

替下面這段 Python 加上完整的型別提示(type hints),然後說明:如果有人寫 average(["a", "b"]) 或把 repeat 的兩個引數傳反,mypy 會抓到什麼?

def average(nums):
    return sum(nums) / len(nums)

def repeat(text, times):
    return text * times
看解答

加上型別提示:average 吃一串數字、回傳浮點數;repeat 吃字串與整數、回傳字串。

def average(nums: list[float]) -> float:
    return sum(nums) / len(nums)

def repeat(text: str, times: int) -> str:
    return text * times

mypy 會抓到的問題:

  • average(["a", "b"]):傳入 list[str],但參數要求 list[float]。 mypy 報 incompatible type "list[str]"; expected "list[float]"。 (注意:不加型別時,這個錯誤要到執行期 sum() 試圖把字串相加才會爆。)
  • repeat(3, "hi")(把引數傳反):第一個參數期待 str 卻收到 int, 第二個期待 int 卻收到 str,mypy 兩個都會標出來。 有趣的是,3 * "hi" 在 Python 執行期其實不會報錯(會得到 "hihihi"), 所以這是個「不當機但邏輯錯」的 bug——正是靜態檢查最有價值的地方。
✏️ 練習 9.3 實作

用 dataclass 在 Python 裡模擬一個和型別 Option:定義 Some(帶一個值) 與 Nothing(沒有值)。然後寫一個 safe_sqrt(x):若 x >= 0 回傳 Some(根號 x), 否則回傳 Nothing。最後用 match 把結果拆開印出來。

看解答 
from dataclasses import dataclass
import math

@dataclass
class Some:
    value: float

@dataclass
class Nothing:
    pass

Option = Some | Nothing        # 和型別:結果「是」Some 或 Nothing

def safe_sqrt(x: float) -> Option:
    if x >= 0:
        return Some(math.sqrt(x))
    return Nothing()

# 用模式比對拆開——你「被迫」處理 Nothing 的情況
def show(result: Option) -> str:
    match result:
        case Some(value=v):
            return f"答案是 {v}"
        case Nothing():
            return "沒有實數解"

print(show(safe_sqrt(16)))   # 答案是 4.0
print(show(safe_sqrt(-9)))   # 沒有實數解

這就是 Maybe/Option 的精神:safe_sqrt 的回傳型別是 Option 而不是 float, 所以你不能直接拿去做運算,必須先 match 拆開,順便被逼著面對「沒有結果」的情況——null 的地雷被拆掉了。

✏️ 練習 9.4 實作

設計一個泛型堆疊 Stack[T](用 Python 的 Generic 與 TypeVar), 提供 push、pop、peek(看頂端但不取出)、is_empty。 示範它能裝 int 也能裝 str,並說明為什麼 Stack[int] 裡 push("x") 會被 mypy 擋下。

看解答 
from typing import Generic, TypeVar

T = TypeVar("T")               # 型別變數:呼叫時才決定 T 是什麼

class Stack(Generic[T]):
    def __init__(self) -> None:
        self._items: list[T] = []

    def push(self, item: T) -> None:
        self._items.append(item)

    def pop(self) -> T:
        return self._items.pop()

    def peek(self) -> T:
        return self._items[-1]

    def is_empty(self) -> bool:
        return len(self._items) == 0

# 裝整數
nums: Stack[int] = Stack()
nums.push(1); nums.push(2)
print(nums.pop())              # 2

# 同一份程式,裝字串
words: Stack[str] = Stack()
words.push("hello")
print(words.peek())           # hello

# nums.push("x")              # ← mypy:Argument 1 has incompatible type "str"; expected "int"

關鍵:當你寫 nums: Stack[int],就把型別變數 T 綁定成了 int。 於是 push 的參數 item: T 在這個物件上變成 item: int, push("x") 自然違反型別。泛型的好處正在這:一份程式,不同使用點各自享有精準的型別檢查。 注意這層保護只在 mypy 階段;Python 執行期並不會真的擋下 push("x")。

✏️ 練習 9.5 挑戰

用和型別(ADT)為一棵運算式樹(expression tree)建模:一個運算式是 一個數字,或兩個運算式的加、減、乘、除其中之一。然後寫一個求值函式 eval_expr, 用模式比對算出整棵樹的值。最後算出 (1 + 2) × (8 − 3)。

附加思考:除法可能除以零。結合練習 9.3 的 Option 思維, 讓 eval_expr 回傳「可能沒有值」,優雅地處理除零。

看解答

先把運算式樹定義成和型別:每個變體要嘛是葉子(數字),要嘛是帶兩棵子樹的運算。 這是 ADT 最自然的舞台——資料的「形狀」天生就是遞迴的和型別。

from dataclasses import dataclass

@dataclass
class Num:
    value: float

@dataclass
class Add:
    left: "Expr"
    right: "Expr"

@dataclass
class Sub:
    left: "Expr"
    right: "Expr"

@dataclass
class Mul:
    left: "Expr"
    right: "Expr"

@dataclass
class Div:
    left: "Expr"
    right: "Expr"

# 和型別:一個 Expr「是」數字,或四種運算之一
Expr = Num | Add | Sub | Mul | Div

def eval_expr(e: Expr) -> float:
    match e:
        case Num(value=v):
            return v
        case Add(left=l, right=r):
            return eval_expr(l) + eval_expr(r)
        case Sub(left=l, right=r):
            return eval_expr(l) - eval_expr(r)
        case Mul(left=l, right=r):
            return eval_expr(l) * eval_expr(r)
        case Div(left=l, right=r):
            return eval_expr(l) / eval_expr(r)
    raise ValueError("未知的運算式")

# (1 + 2) × (8 − 3)
tree = Mul(Add(Num(1), Num(2)),
           Sub(Num(8), Num(3)))
print(eval_expr(tree))        # 15.0

注意 eval_expr 對自己遞迴呼叫:要算 Add 的值,先算左右子樹的值—— 這正是第 3 章「樹狀結構天生適合遞迴」的活例子,而 ADT + 模式比對讓它寫起來幾乎像在抄數學定義。

附加:用 Option 處理除零。把回傳型別改成「可能沒有值」,任何一步除零就讓整棵樹的結果變成 None:

def safe_eval(e: Expr) -> float | None:
    match e:
        case Num(value=v):
            return v
        case Add(left=l, right=r):
            return _combine(safe_eval(l), safe_eval(r), lambda a, b: a + b)
        case Sub(left=l, right=r):
            return _combine(safe_eval(l), safe_eval(r), lambda a, b: a - b)
        case Mul(left=l, right=r):
            return _combine(safe_eval(l), safe_eval(r), lambda a, b: a * b)
        case Div(left=l, right=r):
            a, b = safe_eval(l), safe_eval(r)
            if a is None or b is None or b == 0:   # 任何子樹失敗、或除以零
                return None
            return a / b
    raise ValueError("未知的運算式")

def _combine(a, b, op):
    if a is None or b is None:    # 只要有一邊算不出來,整體就算不出來
        return None
    return op(a, b)

print(safe_eval(Div(Num(1), Num(0))))                 # None(優雅地處理除零)
print(safe_eval(Add(Num(10), Div(Num(6), Num(2)))))   # 13.0

對照看 Haskell 寫同一棵樹有多精煉——而且它的 case 若漏掉某個變體,編譯器會直接警告(窮盡性檢查):

data Expr = Num Double
          | Add Expr Expr
          | Mul Expr Expr

eval :: Expr -> Double
eval (Num n)   = n
eval (Add a b) = eval a + eval b
eval (Mul a b) = eval a * eval b

你已經親手體會到:當資料的結構是「和型別」,ADT + 模式比對 + 遞迴, 幾乎是把問題的數學定義直接翻譯成程式。這就是型別替你思考的威力。

把這一章串起來

我們從牆上的插座出發,走完了型別系統的全景。回顧這趟旅程的路標:

  • 型別 = 一組值 + 一組操作。型別檢查就像確認插頭符合插座,擋掉「對字串做減法」這類無意義的操作。
  • 型別的價值有三:擋掉一整類錯誤、當永不過期的文件、當輕量級的證明(Curry–Howard 對應)。
  • 靜態 vs 動態:差別在何時檢查——編譯期(事前體檢)還是執行期(事後急診)。同一個 bug,C 編譯期被擋、Python 執行期才爆。
  • 強 vs 弱:差別在願不願意偷偷轉型。這跟動靜是兩條獨立的軸:Python 是動態但強,C 是靜態但弱,站在對角線兩端。
  • 型別推論讓靜態語言看起來像動態一樣清爽(Hindley–Milner、C++ auto);不寫型別,不等於沒有型別。
  • 代數資料型別:積型別(而且 / struct、tuple)與和型別(或 / enum、union、Option)。 Option/Maybe 用和型別消滅了「十億美元錯誤」null。
  • 模式比對是拆開和型別的標準姿勢;搭配窮盡性檢查,讓「新增一個變體」這件事變得安全。
  • 泛型(參數多型)讓一段程式適用所有型別(Stack[T]、C++ template、Java <T>),三語言的實作哲學各異。
  • 型別安全保證「不做非法的型別操作」,但管不到你的邏輯——型別是第一道篩網,不是測試的替代品。
  • Python 的型別提示 + mypy 實現了漸進式型別:保留動態的彈性,又能在跑之前抓錯。
  • C 的型別世界:typedef、struct(積)、union(危險的裸和型別)、指標型別——型別的本質, 其實是「如何解讀一段位元」,這條線會在第五部展開。

現在你看程式語言的眼光應該不一樣了:每一種語言,都是在「動/靜」與「強/弱」這張地圖上, 替「安全 vs 彈性」做出了一組取捨。沒有最好的型別系統,只有最適合手上問題的那一個。 下一站,我們會把這些「程式設計核心」的概念,帶往更接近機器、更講究效率的世界。
