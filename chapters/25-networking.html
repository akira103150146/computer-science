網際網路(Internet)是人類有史以來造過最龐大、最複雜的工程系統。沒有人能一次理解它的全部。 但好消息是:設計它的人也知道這件事,所以他們用了一招你已經很熟的武器來馴服這頭巨獸—— 正是第 1 章講的抽象、第 11 章講的分層架構。

🔑
本章的學法:由上而下(top-down)

傳統教科書喜歡從電線、電壓講起,一路往上爬到網頁。我們反過來—— 從你最熟悉、最有感的那一層(瀏覽器、網址、HTTP)開始,一路往下挖, 看每一層為了支撐上一層,默默做了什麼。這叫「由上而下」,是 Kurose & Ross 經典教材的招牌教法, 也是讓網路「有感」的最好方式。

為什麼要分層?網路的骨架

想像一下,如果叫你「從零設計網際網路」,你得同時操心多少事:電子訊號怎麼在銅線或光纖裡傳? 兩台相鄰機器怎麼知道一串 0 和 1 從哪開始、到哪結束?資料怎麼從台北送到美國,中間經過幾十台機器怎麼選路? 中途有封包掉了怎麼辦?最後瀏覽器又怎麼知道對方傳來的是一張圖還是一段 HTML?

把這些問題全部攪在一起想,任何人都會瘋掉。於是工程師用了那招老把戲:分而治之。 把整個「讓兩台電腦對話」的大問題,切成幾個堆疊起來的層(layer), 每一層只負責一件事,並且只依賴它下面那一層提供的服務,完全不管下層是怎麼做到的。

🔗
這正是第 11 章的「分層架構」

還記得分層架構的鐵律嗎?每一層只能呼叫正下方那一層,上層依賴下層的服務,下層不知道上層的存在。 網路協定堆疊(protocol stack)就是這個原則最成功、規模最大的一次實踐。 當年那張「應用層 / 傳輸層 / 網路層 / 連結層」的圖,現在我們要把每一層裡面真正在跑什麼,一一講清楚。

分層帶來的兩個禮物

分層不是為了好看,它解決了兩個真實的痛點:

  • 各司其職,獨立演化:寫瀏覽器(應用層)的人,完全不用懂光纖怎麼傳光(實體層)。 反過來,電信公司把銅線換成光纖、把 4G 升級成 5G,你的 LINE 和 Chrome 一行程式都不用改。 因為它們之間只透過固定的「介面」溝通,內部怎麼變都互不影響。
  • 同一套東西重複用:不管你傳的是網頁、電子郵件還是視訊,在傳輸層以下用的都是同一套 TCP/IP。網路層只管「把這包資料送到那台機器」,它根本不在乎裡面裝的是情書還是貓影片。

協定:機器之間的共同語言

在繼續往下之前,先講清楚一個貫穿全章的詞:協定(protocol)。

兩個人講電話,得先有默契:接起來要先說「喂?」,對方回應了才開始講正事,講完說「那就這樣囉,掰掰」再掛斷。 這套「誰先講、講什麼格式、收到要怎麼回」的規矩,如果換成兩台機器之間,就叫協定。

🔑
什麼是協定

協定(protocol)是一份事先講好的「對話規則」,規定通訊雙方交換訊息的格式、順序,以及收到訊息後該採取的動作。 HTTP、DNS、TCP、IP……這些名字後面全都是一份協定。只要雙方都遵守同一份協定, 一台 Windows 筆電就能和一台從沒見過面、跑著 Linux 的伺服器,順暢地對話。協定,就是網路世界的「共同語言」。

五層模型與 OSI 七層

現實中最常用的,是把網路切成五層。由上而下分別是:

  • 應用層(application layer):你的程式直接打交道的一層。HTTP、DNS、SMTP 都在這。處理「網頁、郵件」這種有意義的資料。
  • 傳輸層(transport layer):負責把資料可靠地(或快速地)從一支程式送到另一台機器的某支程式。主角是 TCP 與 UDP。
  • 網路層(network layer):負責把資料從一台主機送到地球另一端的另一台主機,沿途選路。主角是 IP。
  • 連結層(link layer):負責把資料送到實體上相鄰的下一台機器(例如你的筆電到家裡的路由器)。乙太網路、Wi-Fi 在這。
  • 實體層(physical layer):真正把 0 和 1 變成電壓、光、無線電波,在線材或空氣中傳出去。

你可能也聽過更老、更學術的 OSI 七層模型。它把上面的「應用層」進一步拆成三層(應用、表現、會議), 其餘大致對應。OSI 是教科書上的理想參考模型,而真正主宰整個網際網路的,是樸實的 TCP/IP(五層)。 兩者對照如下:

封裝:資料如何一層層被包起來

分層最精妙的機制,叫封裝(encapsulation)。它的點子簡單到像在寄包裹。

想像你要寄一份合約給朋友。合約本身是「資料」。你把它裝進一個信封,信封上寫收件人姓名——這是第一層包裝。 郵局收到後,把你的信封丟進一個更大的郵袋,郵袋上貼著「寄往台中分局」的標籤——第二層包裝。 貨運再把郵袋裝上一台貼著車牌的卡車——第三層。每一層都在外面多包一層、貼上自己這一層需要的資訊,卻完全不去動裡面的內容。

到了對方那台機器,過程完全倒過來:連結層先拆掉自己的訊框標頭,把裡面的東西交給網路層; 網路層拆掉 IP 標頭,交給傳輸層;傳輸層拆掉 TCP 標頭,把最裡面那份 HTTP 訊息交給應用層。 每一層都只「拆自己貼的那張標籤」,從不去管別層的事。這就是分層的優雅之處。

📜
TCP/IP 的誕生

1974 年,Vint Cerf 與 Bob Kahn 發表了一篇論文,描述一套能讓「不同網路彼此互連」的協定—— 這就是 TCP/IP 的雛形,也是「inter-net(網路之間)」這個詞的由來。1983 年 1 月 1 日, ARPANET 正式全面切換到 TCP/IP,這一天常被視為「現代網際網路的生日」。 兩人後來也因此共同獲得了被譽為計算機界諾貝爾獎的圖靈獎。

看解答

下行(封裝):

  1. 應用層產生一份 HTTP 訊息(例如 GET / HTTP/1.1 ...)。
  2. 傳輸層在前面加上 TCP 標頭(含來源埠、目的埠、序號等),變成區段(segment)。
  3. 網路層在前面加上 IP 標頭(含來源 IP、目的 IP),變成封包(packet / datagram)。
  4. 連結層在前後加上 訊框標頭與尾巴(含來源/目的 MAC、錯誤偵測碼),變成訊框(frame)。
  5. 實體層把這串訊框變成電壓/光/無線電波送出去。

上行(解封裝):對方主機反過來,連結層→網路層→傳輸層→應用層,每層拆掉自己那段標頭, 最後把純淨的 HTTP 訊息交給對方的網頁伺服器程式。每一層都只看、只拆自己負責的標頭, 這正是分層架構「各層獨立、互不干涉」的體現。

應用層:你看得見的那一層

既然要由上而下,我們就從你最熟的應用層開始。應用層的協定多到數不完,但對自學者來說, 有兩個是非懂不可的:讓網頁能跑的 HTTP,以及把 google.com 翻成數字位址的 DNS。

HTTP:網頁世界的請求與回應

HTTP(HyperText Transfer Protocol,超文字傳輸協定)是全球資訊網(Web)的核心。 它的模型簡單到不可思議:主從式(client-server)。瀏覽器是客戶端(client), 它發出請求(request);網頁伺服器是伺服器(server),它回傳回應(response)。 一來一回,故事就結束了。

一個 HTTP 請求長這樣(它其實就是一段純文字):

GET /index.html HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0
Accept: text/html

第一行是最關鍵的:方法(method)、路徑、版本。常見的方法有:

  • GET:「給我這個資源」。讀取、不改變伺服器狀態。你開網頁、點連結,基本都是 GET。
  • POST:「這是我要交給你的資料,請處理」。送出表單、上傳留言、登入,通常是 POST。
  • 還有 PUT(更新)、DELETE(刪除)等,在設計 API 時很常用,我們在後面的 Web 章節會細談。

伺服器的回應也是純文字,開頭是一行狀態碼(status code):

HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8
Content-Length: 1256

<!doctype html>
<html> ... 網頁內容 ... </html>

狀態碼用一個三位數字,精簡地告訴你「結果如何」。你一定要認得這幾個家族:

狀態碼家族意義經典代表
2xx成功200 OK(一切正常)
3xx重新導向301 永久搬家、304 沒變(用快取就好)
4xx你(客戶端)錯了404 Not Found(找不到)、403 禁止、401 未授權
5xx伺服器自己出錯500 Internal Server Error、503 暫時無法服務
💡
一秒看懂誰的錯

4 開頭=客戶端的問題(你打錯網址、沒登入);5 開頭=伺服器的問題(它程式爆了)。 下次看到 404 別怪伺服器,那通常是網址打錯;看到 500,則是對方工程師要加班了。

無狀態,與 cookie 的發明

HTTP 有一個影響深遠的設計:它是無狀態的(stateless)。意思是, 伺服器處理完一個請求後,立刻把你忘得一乾二淨。下一個請求進來,它完全不記得你是誰、剛剛做過什麼。 這讓伺服器變得簡單又好擴充(每個請求獨立,可以隨便丟給任何一台伺服器處理),但也帶來一個問題: 那「登入狀態」「購物車」這種需要記住你的功能,怎麼辦?

答案是 cookie。伺服器在回應裡夾一張小紙條(Set-Cookie: session=abc123), 瀏覽器把它存起來;之後每次請求,瀏覽器都自動把這張紙條附上去。 伺服器一看紙條就「想起你了」。所以 cookie 本質上是在無狀態協定之上,人工縫補出來的「記憶」。

HTTPS:加上一層鎖

HTTPS 就是 HTTP 再套一層加密外殼:TLS(Transport Layer Security)。 它做兩件事:加密(沿途的人偷看,只看到亂碼)與驗證身分(確認你連到的真的是 Google,不是冒牌貨)。 為什麼非要它不可?這牽涉到網路的根本弱點,我們留到本章最後「網路與安全」一節再揭曉。

DNS:把名字翻成號碼的電話簿

現在來解開開頭那道謎題最關鍵的一環。你在網址列打的是 google.com, 但網路底層其實完全不認得名字,它只認得數字位址,例如 142.250.196.110(這叫 IP 位址)。 那麼,從名字到號碼的翻譯,是誰做的?DNS(Domain Name System,網域名稱系統)。

DNS 就像全世界共用的一本超巨大電話簿:你給它一個名字,它回你一個號碼。但這本電話簿有上億筆資料、 每秒被查上兆次,不可能放在單一台機器上。於是它被設計成一個分散式、階層式的資料庫—— 這是整個網路裡最優雅的設計之一。

解析 google.com 的步驟,大致是:

  1. 你的電腦先問離你最近的 DNS 解析器(resolver)(通常是你家路由器或 ISP 提供的)。
  2. 解析器若快取(cache)裡有,直接回答,結束。(這就是為什麼第二次開同個網站特別快。)
  3. 若沒有,它先問根伺服器(root):「.com 該找誰?」根回:「去問管 .com 的 TLD 伺服器。」
  4. 再問 TLD 伺服器:「google.com 該找誰?」它回:「去問 google.com 的權威伺服器。」
  5. 最後問 權威伺服器(authoritative),拿到真正的 IP:142.250.196.110。
  6. 解析器把答案回給你,並快取一段時間(由 TTL 決定),下次就不用再跑一遍。
📜
DNS 之前的黑暗時代

1983 年 DNS 由 Paul Mockapetris 設計之前,名字到 IP 的對照是放在一個叫 HOSTS.TXT 的 單一檔案裡,由史丹佛研究院統一維護。每台電腦要定期下載這個檔案來更新。 隨著網路長大,這招很快撐不住——檔案越來越大、更新越來越慢、衝突越來越多。 DNS 的「分散式階層」設計,正是為了解決這個「中央集權撐不住」的難題。(今天你電腦裡那個 hosts 檔,就是它的活化石。)

一句話帶過:其他應用層協定

應用層還住著很多鄰居:寄電子郵件用 SMTP、收信用 IMAP/POP3;傳檔案有 FTP; 遠端登入有 SSH(你在第 16 章用過);看影片的串流、線上遊戲也各有自己的應用層協定。 它們的共通點是:都站在傳輸層的肩膀上,自己只專心定義「訊息長什麼樣、怎麼一來一往」。

主從式 vs P2P

HTTP、DNS、郵件,清一色是主從式架構:有明確的「伺服器」提供服務,「客戶端」來索取。 但這不是唯一的玩法。還有一種叫 P2P(peer-to-peer,點對點):沒有中央伺服器, 每個參與者既是客戶端、也是伺服器,大家直接互相分享。BitTorrent 下載、許多區塊鏈網路,都是 P2P。

🔗
這正是第 11 章的「主從式架構」

還記得第 11 章談過的 client-server 風格嗎?網路是它最大的舞台。 主從式的好處是好管理、好控管;缺點是伺服器一掛,全體癱瘓(單點故障)。P2P 反過來: 沒有中心、極難整個打掉,但難管理、難保證品質。要選哪種,又回到那句老話——架構是一種權衡。

傳輸層:把資料交到正確的程式手上

應用層產生了一份要寄出去的訊息。接下來,傳輸層登場。它要解決兩個問題: (1) 同一台電腦上開了一堆程式,資料來了該交給誰?(2) 我們要「可靠」還是「快」?

連接埠與多工

你的電腦此刻可能同時開著瀏覽器、LINE、收信軟體、線上遊戲。它們的封包全擠在同一條網路線上進進出出。 當一個封包抵達,作業系統怎麼知道該把它交給瀏覽器,而不是 LINE?

答案是連接埠(port):一個 16 位元的號碼(0 到 65535)。每支正在用網路的程式, 都「佔住」一個埠號。IP 位址帶你找到正確的機器(像找到一棟大樓),埠號則帶你找到機器上正確的程式(像找到大樓裡正確的房間)。 傳輸層靠「IP + 埠號」這組地址,把資料精準送到目的程式——這個「分流」的動作,就叫多工(multiplexing)/解多工(demultiplexing)。

有些埠號是全世界講好的慣例,看到就該認得:

埠號服務協定
80HTTPTCP
443HTTPSTCP
53DNS多半 UDP
22SSHTCP
25SMTP(寄信)TCP

這就是為什麼你打 http://example.com 不用特別指定埠號——瀏覽器預設就連 80 埠(HTTPS 連 443)。

UDP:簡單、不可靠、但快

傳輸層有兩位主角,先講比較簡單的這位:UDP(User Datagram Protocol)。

UDP 幾乎什麼都不做。它接過應用層的資料,加個小小的標頭(來源埠、目的埠、長度、檢查碼),就丟給網路層, 然後……就沒有然後了。它不保證送達、不保證順序、不重傳、不確認。封包掉了?它不知道,也不管。 這聽起來很爛,但它換來了兩個寶貴的東西:極低的延遲與極小的開銷。

什麼時候「快」比「可靠」重要?

  • 即時視訊/語音通話:掉一兩個畫面格沒關係(頂多花一下),但若為了補那一格而卡住,反而更糟。
  • 線上遊戲:你角色 0.1 秒前的位置,重傳到了也沒意義——遊戲早就往前跑了。新資料才有用。
  • DNS 查詢:一個小問句、一個小答案,用 UDP 一來一回就好,沒掉就沒掉,掉了重問一次比建立連線還快。

TCP:在不可靠的網路上,變出可靠

現在來到本章的智慧高峰。先認清一個殘酷的事實:底下的網路層(IP)是「盡力而為」(best-effort)的—— 它會盡力把封包送到,但從不保證。封包可能在路上塞車被丟掉、可能走不同路徑導致先發後到、可能損毀。 IP 對這些一概不負責。

然而你下載一個檔案、開一個網頁,它一個位元組都不能錯、不能漏、不能亂序。 這個「在不可靠的地基上,蓋出可靠的房子」的奇蹟,就是 TCP(Transmission Control Protocol) 的工作。 它怎麼辦到的?靠三個核心點子。

核心點子一:三向交握(建立連線)

和 UDP「丟了就跑」不同,TCP 在傳資料之前,雙方要先「打招呼、確認彼此都準備好了」。 這個開場儀式,叫三向交握(three-way handshake),正好三個來回:

交握完成後,連線就「建立」了。注意:這條「連線」其實是虛擬的——底下的 IP 根本沒有「連線」的概念, 每個封包都是各走各的。TCP 只是用雙方記住的狀態(序號等),在兩端「假裝」有一條穩定的水管。這又是一層漂亮的抽象。

核心點子二:序號 + 確認 + 重傳(達成可靠)

資料開始流動後,TCP 怎麼確保不漏、不亂?三招配合:

  • 序號(sequence number):每個位元組都有編號。就算封包亂序抵達,接收方也能照編號重新排好。
  • 確認(ACK):接收方每收到資料,就回一個 ACK:「我已經收到第 N 號之前的所有東西了。」
  • 重傳(retransmission):傳送方送出資料後會等 ACK。等太久沒等到(逾時), 就假設那包掉了,再傳一次。

這三招合起來,就是「可靠資料傳輸」的核心思想:編號讓你能重組與偵測缺漏,確認讓你知道對方收到了沒, 重傳讓你能補救遺失。掉了就補、亂了就排——無論底下的網路多麼不靠譜,上層應用看到的永遠是一條乾淨、有序、完整的資料流。

🔑
可靠傳輸的精髓:一句話

TCP 不是靠「一條不會出錯的線」做到可靠,而是靠「不斷確認 + 缺了就重傳」這個回饋迴路。 這是個深刻的工程哲學:與其追求完美的元件,不如設計一個能偵測錯誤並自我修復的系統。 你會在分散式系統、甚至 TCP 以外的無數地方,一再遇到這個思想。

核心點子三:流量控制與壅塞控制(別把人淹死、也別把路塞爆)

最後,TCP 還很有「公德心」。它管兩種「淹水」:

  • 流量控制(flow control):別傳太快,把接收方淹死。接收方會在 ACK 裡告訴你 「我還剩多少緩衝空間(視窗 window)」,傳送方就照這個量,不會塞爆對方的信箱。
  • 壅塞控制(congestion control):別傳太快,把中間的網路塞爆。 這招更聰明:TCP 一開始小心試探(慢啟動,從很慢開始,逐步加速), 一旦偵測到塞車跡象(封包開始掉),就立刻大幅降速,然後再慢慢爬回來。 這種「慢慢加、出事就猛踩煞車」的策略,讓全世界幾十億條 TCP 連線能客氣地共享有限的網路,而不是一起把它擠垮。
📜
1986 年的網路大塞車

1986 年,早期網際網路曾發生「壅塞崩潰(congestion collapse)」:大家拼命重傳塞住的封包, 結果越塞越重傳、越重傳越塞,有效傳輸量一度暴跌上千倍。Van Jacobson 在 1988 年提出的 慢啟動與壅塞避免演算法,救了整個網路,至今仍是 TCP 的基石。這提醒我們: 一個看似抽象的「控制策略」,可能就是一個系統能否規模化的生死關鍵。

參考答案:

網路層:讓封包穿越地球

傳輸層把資料切成一段段,交給網路層。網路層只有一個任務,但這任務無比艱鉅: 把這個封包,從你台北家裡的電腦,送到地球另一端某台機器手上——而它倆中間,隔著幾十台陌生的機器。

IP 位址:網路世界的門牌

要把東西送到某處,得先有地址。網路層的地址就是 IP 位址(IP address)。 目前最普及的 IPv4 用 32 個位元,寫成四個 0–255 的數字,用點隔開,例如 140.112.30.26。

一個 IP 位址其實分成兩半:網路部分(network)說明「你在哪個網路(社區)」, 主機部分(host)說明「你是那個網路裡的哪一台(門牌號)」。哪裡是分界?由子網路遮罩決定, 現代寫法用 CIDR 表示法,在位址後面加一條斜線和數字:

為什麼要分這兩半?因為這讓路由變得可行。全世界有幾十億台機器,路由器不可能記住每一台的位置; 但它可以只記「往某個網路(一整個社區)的封包,該往哪個方向送」。 到了那個社區的門口(路由器),再由它負責送到正確的門牌。用「網路 / 主機」分層,把天文數字般的位址,壓縮成可管理的規模—— 這又是分而治之。

順帶一提:32 位元的 IPv4 只有約 43 億個位址,早就不夠用了。 於是有了 IPv6,用 128 位元,位址多到幾乎用不完(足以給地球上每一粒沙子都分一個)。

路由與轉送:一跳一跳地送

現在最核心的問題來了:封包怎麼從台北「找到路」去美國?關鍵在於——沒有任何一台機器知道完整的路。 整個過程是一跳一跳(hop by hop)接力完成的。

負責接力的機器叫路由器(router)。每台路由器手上有一張轉送表(forwarding table), 記著「要去某某網路方向,就把封包交給我隔壁的哪一台路由器」。封包每抵達一台路由器,它就查表、 把封包往「比較接近目的地」的下一台丟過去。如此一台傳一台,像接力賽,直到抵達目的網路。

這裡有兩個容易混淆但重要的詞:

  • 轉送(forwarding):單一路由器當下的動作——「這個封包進來,我該把它從哪個出口送出去?」查表、轉手,瞬間完成。
  • 路由(routing):在背後默默進行的大工程——路由器們彼此交換「我能到哪些網路、距離多遠」的情報, 合力算出「整張網路的最佳路徑圖」,並填好每個人的轉送表。(這背後正是第 15 章的圖論最短路徑演算法!)

封包交換 vs 電路交換

為什麼網際網路要這樣「一包一包、各走各的」?這牽涉一個歷史性的設計抉擇:封包交換(packet switching) 對上老式電話網的電路交換(circuit switching)。

  • 電路交換(老電話):通話前,先在你和對方之間專門「接通」一條實體線路, 整通電話這條線都只給你用。優點是品質穩定;缺點是浪費——你講話中間的停頓,那條線就空著也沒人能用。
  • 封包交換(網際網路):資料切成一個個小封包,每個封包獨立地、見縫插針地在共享的線路上傳。 線路被所有人動態共享,使用率極高、極有彈性,某條線斷了封包還能繞道。代價是品質沒保證(這就是為什麼需要 TCP 來補)。

網際網路選了封包交換。這個決定,讓網路變得極度有韌性(部分損壞仍能運作)又極度有效率—— 這也是它能長成今天這個規模的根本原因之一。

順手認識:NAT、ICMP、ping、traceroute

網路層還有幾個你遲早會遇到的名詞,一句話各記一個:

  • NAT(網路位址轉換):你家裡一堆裝置,對外其實共用同一個公開 IP——路由器靠 NAT 在中間做「翻譯」, 這也是 IPv4 位址不夠用還能撐這麼久的功臣。
  • ICMP:網路層的「客服與診斷」協定,專門傳遞錯誤與測試訊息。
  • ping:用 ICMP 問對方「你還活著嗎?」並量來回時間(延遲)。網路不通時你第一個該下的指令。
  • traceroute:一個聰明的小把戲,能把封包經過的每一跳路由器列出來,讓你親眼看到「一跳一跳」是真的。
# 看封包從你的電腦到 google.com 經過了哪些路由器
$ traceroute google.com        # Windows 上是 tracert

✏️ 練習 25.4 挑戰

子網路計算。某公司分到一個網段 192.168.10.0/24。請回答: (a) 這個網段的「網路部分」佔幾位元、「主機部分」佔幾位元? (b) 理論上最多能放幾台主機(扣掉不能用的網路位址與廣播位址)? (c) 若改成 /26,主機數變多還是變少?大約幾台?

看解答

(a) /24 表示網路部分佔前 24 位元,主機部分佔剩下的 32 − 24 = 8 位元。

(b) 8 位元可表示 $2^8 = 256$ 個位址。但每個子網路有兩個位址不能分給主機: 全 0 的網路位址(.0)與全 1 的廣播位址(.255)。 所以可用主機數 = $256 - 2 = \mathbf{254}$ 台。

(c) /26 表示網路部分變多(26 位元),主機部分只剩 $32 - 26 = 6$ 位元, 所以主機數變少:$2^6 - 2 = 64 - 2 = \mathbf{62}$ 台。 (斜線後的數字越大,網路部分越長、子網路切得越細、每個子網路能容納的主機越少。)

連結層:送到隔壁那一台

最後一層(我們會碰到的)。網路層說「把這封包送到地球另一端」,但它其實沒辦法一步到位—— 它只能仰賴連結層,把封包送到實體上直接相連的下一台機器(下一跳)。 連結層管的是「同一段線路上的鄰居之間」怎麼傳。

  • MAC 位址(MAC address):每張網路卡出廠時就燒進去的、全球唯一的硬體編號 (像 3c:22:fb:0a:1b:2c)。IP 位址是「邏輯地址」(可換、可分配),MAC 是「物理身分證」(綁在硬體上)。
  • 乙太網路(Ethernet):有線區域網路最通行的連結層技術;Wi-Fi 則是它的無線版表親。
  • 交換器(switch):區域網路裡的「分流員」,看 MAC 位址,把訊框只送給該收的那台機器 (不像最原始的集線器會吵到所有人)。
  • ARP:在同一個區域網路內,把「IP 位址」翻成「MAC 位址」的小幫手——「誰是 192.168.1.5?把你的 MAC 告訴我。」
🔍
兩種「地址」,各司其職

一趟旅程同時用到兩種地址:IP 位址負責端到端(從台北到美國,全程不變,像「最終收件地址」); MAC 位址負責每一跳之間(從這台到隔壁那台,每跳都換,像「下一個中轉站」)。 封包每經過一台路由器,IP 不變,但連結層的 MAC 來源/目的會被改寫成「這一跳」的雙方。 看懂這點,你就真的懂了分層。

把整條路走一遍:輸入 google.com 到看到頁面

準備好了。現在我們把所有層串起來,完整回答那道從第 0 章欠到現在的題目。 當你在網址列敲下 google.com 按下 Enter,以下這一切,在大約一秒內發生:

  1. DNS 解析:瀏覽器先問 DNS:「google.com 的 IP 是?」(若快取有就秒回,沒有就走前面那段根→TLD→權威的階層查詢)。拿到 142.250.196.110。
  2. TCP 三向交握:瀏覽器對著那個 IP 的 443 埠,送出 SYN,完成 SYN-ACK、ACK,建立可靠連線。
  3. TLS 交握:因為是 HTTPS,雙方再協商加密金鑰、驗證伺服器憑證(確認對方真的是 Google)。之後所有資料都加密。
  4. HTTP 請求:瀏覽器送出 GET / HTTP/1.1(帶著 Host、cookie 等標頭)。
  5. 封裝下行:這份請求被加上 TCP 標頭(目的埠 443)、IP 標頭(目的 IP)、訊框標頭(下一跳的 MAC),變成位元串送出。
  6. 路由穿越:封包一跳一跳,經過你家路由器、ISP、海底電纜、無數台路由器,最終抵達 Google 的資料中心。沿途若有封包遺失,TCP 會重傳。
  7. 伺服器回應:Google 的伺服器處理請求(可能查它自己的資料庫、跑排序演算法),回傳 200 OK 加上 HTML,原路可靠送回。
  8. 瀏覽器渲染:瀏覽器解析 HTML,發現還需要 CSS、JS、圖片,於是再發起一輪輪的 HTTP 請求(每個資源重複上面的流程),最後把畫面畫出來。你看到了 Google 首頁。

從第 0 章那句模糊的「大概會查 DNS 吧」,到現在你能清楚說出這八個階段、每一階段在哪一層、用什麼協定—— 這就是你讀完這趟旅程的證明。

加分重點:能說出「每層各加了什麼標頭(封裝)」「為什麼用 TCP 而非 UDP(可靠)」「DNS 為何分階層+快取」, 就能展現你真的理解,而不只是背步驟。

用程式碼摸到網路:Python socket

講了這麼多,現在親手碰一下。作業系統提供了一個叫 socket(插座)的抽象: 它是你的程式通往網路的「插孔」。你把資料塞進 socket,它就順著 TCP/IP 那一整套機制送出去; 對方的資料也從 socket 流回來。Python 內建的 socket 模組,讓你能直接玩這個插座。

一個極簡的 echo 伺服器

我們來寫一個「回聲(echo)」伺服器:你傳什麼給它,它就原封不動傳回什麼。麻雀雖小,五臟俱全—— bind(綁定埠號)、listen(開始聽)、accept(接受連線)、recv/sendall(收發), 正是所有網路伺服器的骨架。 

# echo_server.py —— 一個極簡的 TCP 回聲伺服器
import socket

HOST = "127.0.0.1"   # 本機位址(localhost),只接受自己電腦的連線
PORT = 65432         # 挑一個沒被佔用的埠號(1024 以上比較安全)

# AF_INET = 用 IPv4;SOCK_STREAM = 用 TCP(可靠串流)
with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as s:
    s.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)  # 方便重複啟動
    s.bind((HOST, PORT))     # 把 socket 綁到「這台機器的這個埠」
    s.listen()               # 開始聆聽連線
    print(f"伺服器在 {HOST}:{PORT} 等待連線…")

    conn, addr = s.accept()  # 阻塞,直到有人連進來;回傳一條專屬連線
    with conn:
        print(f"已連線:{addr}")
        while True:
            data = conn.recv(1024)   # 一次最多收 1024 個位元組
            if not data:             # 收到空 bytes,代表對方關閉連線
                break
            print("收到:", data.decode())
            conn.sendall(data)       # 原封不動送回去(這就是 echo)
    print("連線關閉")

對應的客戶端

# echo_client.py —— 連上 echo 伺服器,傳一句話,印出回聲
import socket

HOST = "127.0.0.1"
PORT = 65432

with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as s:
    s.connect((HOST, PORT))            # 連到伺服器(背後就是 TCP 三向交握!)
    s.sendall(b"Hello, network!")      # 注意 b"":網路只認位元組 bytes,不是字串
    data = s.recv(1024)                # 收回聲
    print("伺服器回傳:", data.decode())  # bytes 轉回字串顯示

開兩個終端機:一個跑 python echo_server.py,另一個跑 python echo_client.py。 你會看到伺服器印出「收到: Hello, network!」,客戶端印出「伺服器回傳: Hello, network!」。 恭喜——你剛剛親手在兩支程式之間,用 TCP 收發了真正的位元組。

⚠️
網路傳的是位元組,不是字串

注意上面到處是 b"..." 和 .encode() / .decode()。 網路底層只搬位元組(bytes),不認得 Python 的字串型別。送出前要 encode()(字串→位元組), 收進來要 decode()(位元組→字串)。初學者最常在這裡踩到 TypeError。

用 socket 發一個原始的 HTTP 請求

最後這個最有「揭開魔法」的快感:我們不用任何網頁函式庫,只用最底層的 socket, 手動拼出一個 HTTP 請求字串送出去,然後把伺服器的回應原汁原味印出來。你會親眼看到, 本章前面講的「HTTP 其實就是一段純文字」是真的。 

# raw_http.py —— 不靠任何 HTTP 函式庫,手動發一個 GET 請求
import socket

host = "example.com"
port = 80   # HTTP 用 80 埠(這裡示範明文 HTTP;HTTPS 還要加 TLS 較複雜)

# 親手拼出 HTTP 請求文字。注意換行必須是 \r\n,結尾要空一行(\r\n\r\n)
request = (
    f"GET / HTTP/1.1\r\n"
    f"Host: {host}\r\n"
    f"Connection: close\r\n"   # 告訴伺服器:回應完就關連線
    f"\r\n"                     # 空行 = 標頭結束
)

with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as s:
    s.connect((host, port))          # DNS 解析 + TCP 交握都在這一行內完成
    s.sendall(request.encode())      # 字串編成 bytes 送出

    response = b""
    while True:                      # 持續收,直到對方關閉連線
        chunk = s.recv(4096)
        if not chunk:
            break
        response += chunk

print(response.decode("utf-8", errors="replace"))

執行後,你會看到伺服器吐回來的東西,開頭正是本章前面講過的狀態行與標頭:

HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8
Content-Length: 1256
...

<!doctype html>
<html>
<head><title>Example Domain</title></head>
...

這短短二十行,把這一整章串成了一條可以執行的鏈:connect 觸發 DNS 與 TCP 交握、 sendall 送出 HTTP 請求、回應沿著路由可靠地流回來。你平常用的 requests 函式庫、 瀏覽器,骨子裡做的也是同一件事,只是幫你把這些細節包成了更舒服的抽象。

✏️ 練習 25.6 實作

把上面的 echo 伺服器改強一點:(a) 讓它能連續服務多個客戶端 (一個服務完,還能接下一個,不要服務完就結束);(b) 如果客戶端傳來的內容是 "quit", 就回它一句 "bye" 並關閉那條連線。

看解答

關鍵是把 accept() 放進一個外層 while 迴圈,服務完一個再回頭等下一個:

import socket

HOST, PORT = "127.0.0.1", 65432

with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as s:
    s.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
    s.bind((HOST, PORT))
    s.listen()
    print(f"在 {HOST}:{PORT} 等待…")

    while True:                       # (a) 外層迴圈:服務完一個還能接下一個
        conn, addr = s.accept()
        with conn:
            print("已連線:", addr)
            while True:
                data = conn.recv(1024)
                if not data:
                    break
                msg = data.decode().strip()
                if msg == "quit":     # (b) 收到 quit 就道別並結束此連線
                    conn.sendall(b"bye")
                    break
                conn.sendall(data)    # 否則照樣回聲
        print("一位客戶端離開,繼續等待下一位…")

進階思考:這個版本一次只能服務「一個」客戶端,後面的人得排隊。 要同時服務很多人,需要用到多執行緒、多行程,或非同步 I/O(asyncio)—— 這正是「並行(concurrency)」的議題,留待後面的作業系統與系統設計章節。

✏️ 練習 25.7 實作

用 socket 寫一個程式,連到 example.com,發出一個原始的 HTTP GET 請求, 只把回應的狀態行(第一行,例如 HTTP/1.1 200 OK)印出來,其餘丟掉。

看解答

在前面 raw_http.py 的基礎上,把收到的整段回應切出第一行即可:

import socket

host = "example.com"
request = (
    f"GET / HTTP/1.1\r\n"
    f"Host: {host}\r\n"
    f"Connection: close\r\n\r\n"
)

with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as s:
    s.connect((host, 80))
    s.sendall(request.encode())
    # 狀態行就在最前面,通常前幾百個位元組就夠了
    head = s.recv(1024).decode("utf-8", errors="replace")

status_line = head.split("\r\n", 1)[0]   # 取到第一個換行為止
print(status_line)                       # 例如:HTTP/1.1 200 OK

小觀察:split("\r\n", 1) 的第二個參數 1 表示「最多切一次」, 所以 [0] 就是第一行。試著把網址改成一個不存在的路徑(如 GET /nope), 看看狀態行是不是變成了 404 Not Found。

網路與安全的交會點

讀完這一章,你應該對網路如何運作有了清晰的圖像。但這幅圖像藏著一個令人不安的真相, 它直接通往第七部「資訊安全」:

回想封包穿越網路的過程——它一跳一跳,經過了幾十台你完全不認識、也無法控制的機器。 而最原始的 HTTP、DNS 等協定,傳的全是明文。這意味著,沿途任何一台機器(惡意的路由器、 共用同一個 Wi-Fi 的陌生人),理論上都能偷看你傳的內容——你的帳號、密碼、訊息。這叫竊聽(eavesdropping)。

⛔
這就是為什麼非 HTTPS 不可

網路的開放與分享本性,讓它先天就不安全。TLS / 加密(也就是 HTTP 變成 HTTPS 的那個 S) 正是為了補這個洞:就算封包被沿途的人攔下,他看到的也只是一堆無意義的亂碼;同時,憑證機制還能確保 「你連到的真的是 Google,而不是有人冒充」。在咖啡廳連公用 Wi-Fi 卻看到網址是 http:// 而非 https:// 時,請提高警覺。

除了竊聽,還有幾種你該先聽過名字的攻擊,我們會在第七部細談:

  • 中間人攻擊(Man-in-the-Middle, MITM):攻擊者偷偷卡在你和伺服器之間,假扮成對方, 把雙方的對話全程攔截、甚至竄改。TLS 的憑證驗證就是專門防它的。
  • DDoS(分散式阻斷服務):操縱成千上萬台機器同時對一個目標發出海量請求, 把它的頻寬或運算資源塞爆癱瘓,讓正常使用者無法存取。

這些都是「網路的特性」與「惡意」交會時迸出的火花。理解了網路怎麼運作,你才看得懂它為什麼會被攻擊、 又該怎麼防——這正是第七部要接手的故事。

把這一章串起來

我們用「由上而下」的方式,從你最熟的瀏覽器一路挖到了線材。回顧這趟旅程的路標:

  • 分層是馴服網路這頭巨獸的關鍵:每層只做一件事、只依賴下一層,讓各層能獨立演化、重複利用。 這正是第 11 章「分層架構」的最大舞台。
  • 封裝讓資料下行時層層加標頭(訊息→區段→封包→訊框),上行時層層拆開。協定則是各層的共同語言。
  • 應用層:HTTP 的請求-回應與狀態碼、無狀態與 cookie;DNS 用分散式階層 + 快取,把名字翻成 IP。
  • 傳輸層:連接埠做多工;UDP 簡單快但不可靠;TCP 用三向交握、序號、ACK、重傳、壅塞控制, 在不可靠的網路上「修」出可靠——這是本章最深刻的工程思想。
  • 網路層:IP 位址(網路/主機兩部分)、封包交換、一跳一跳的路由,讓封包能穿越地球。
  • 連結層:MAC 位址與乙太網路,負責「送到隔壁那一台」。IP 管端到端,MAC 管每一跳。
  • 最後,我們用 Python socket 親手收發了位元組,並把這一整章串成一條能執行的鏈。

你現在能完整回答那道從第 0 章欠到現在的題目了。更重要的是,你獲得了一種「看穿」的能力: 下次網頁開不出來,你不會只會重開機——你會開始想,是 DNS 出問題、TCP 連不上、還是伺服器回了 5xx? 這種沿著分層去推理的能力,正是工程師和一般使用者的分水嶺。

網路把全世界的電腦連了起來,但「連起來之後要怎麼安全地用」「資料存在哪、怎麼存」「系統怎麼設計才扛得住幾億人」—— 這些更大的問題,我們會在後面的資安、資料庫與系統設計章節繼續探索。網路,是這一切的地基。
