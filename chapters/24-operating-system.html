作業系統:虛擬化、並行與持久化

你按下開機鍵之後,有一個程式比所有程式都先醒來,而且永遠不睡。它管著你電腦裡的每一顆 CPU、每一格記憶體、每一個檔案——卻幾乎從不現身。這一章,我們去認識這個沉默的統治者:作業系統。

一個忙翻天的管家

想像你住進一棟豪宅,裡面有一位無所不能、但脾氣古怪的管家。你不必知道廚房的瓦斯怎麼接、電從哪裡來、垃圾車幾點到—— 你只要拉一下鈴,說「我餓了」,飯菜就會上桌;說「我冷了」,暖氣就會打開。管家在背後協調一切: 他確保你和其他房客不會同時搶用同一個浴室,確保你睡覺時隔壁的派對不會吵到你,確保你寄放的東西明天還在。

你電腦裡就住著這樣一位管家,它叫作業系統(operating system,OS)。 你打開瀏覽器、播音樂、開編輯器寫程式——這三個程式其實都在搶同一批資源: 那幾顆 CPU、那塊記憶體、那個硬碟。是作業系統在中間調度,讓它們以為自己獨佔了整台電腦, 而且彼此井水不犯河水。它做得太好,好到你幾乎感覺不到它的存在。

上一章(第 21 章)我們看清楚了硬體的真相:CPU 笨得只會「讀一個數字、做一件事、再讀下一個」; 記憶體只是一長條格子;螢幕和鍵盤不過是一段被映射的位址。這些硬體能用,但極難用—— 它有限(CPU 只有幾顆)、原始(一個寫錯的位址就能弄花整個畫面)、而且一次只能做一件事。 作業系統的工作,就是把這堆難搞的硬體,變成你寫程式時看到的那個乾淨、寬敞、安全的世界。

🔑
作業系統的兩個身分:管理者與魔術師

作業系統同時扮演兩個角色。資源管理者(resource manager):像政府一樣,公平且有效率地分配 CPU 時間、記憶體空間、硬碟與裝置給每個程式,並且當「警察」維持秩序——不讓一個程式霸佔資源或偷看別人的資料。 魔術師(illusionist):把有限、難用的硬體,「變」成一份份乾淨好用的抽象(abstraction)—— 讓每個程式都以為自己獨佔了一顆 CPU、獨佔了整片記憶體、有一個井然有序的檔案世界。 這份「美好的假象」,正是你寫程式時習以為常、卻其實得來不易的舒適環境。

分界線:使用者模式與核心模式

作業系統握有對硬體的最高權限——它能存取任何記憶體、能直接操控磁碟、能決定哪個程式上 CPU。 但這帶來一個顯而易見的危險:如果任何程式都有這種權限,那你隨便下載的一個小遊戲,就能讀走你的密碼、 或者一個 bug 就能把整台機器弄垮。所以現代 CPU 在硬體層面提供了兩種執行模式(execution mode):

  • 核心模式(kernel mode):特權模式。可以執行任何指令、存取任何硬體。只有作業系統的核心(kernel)跑在這個模式。
  • 使用者模式(user mode):受限模式。不能直接碰硬體、不能存取別人的記憶體。你寫的所有應用程式都跑在這裡。

那問題來了:既然應用程式被關在使用者模式、不能直接碰硬體,它要怎麼讀檔案、怎麼把字印到螢幕? 答案是——它必須拜託作業系統代勞。這個「拜託」的正式機制,叫做系統呼叫(system call)。

當你的程式呼叫 printf 或 open,底層其實會執行一條特殊的陷阱指令(trap): CPU 一執行它,就會切換到核心模式,並跳進作業系統預先設定好的入口。作業系統接手、檢查你有沒有權限、 替你完成這件危險的事,做完再切回使用者模式,把結果交還給你。整個過程像隔著一道牆遞紙條: 你不能翻牆進去,但可以從牆上的小窗口,把請求遞給裡面那位有權限的管家。

ℹ️
「核心(kernel)」與「作業系統」不完全是同一件事

嚴格說,核心是作業系統中那塊跑在核心模式、最核心的程式(管行程、記憶體、檔案、裝置)。 而我們口語講的「作業系統」(像 Windows、macOS、Linux 發行版)通常還包含一大堆跑在使用者模式的東西: 視窗系統、檔案總管、終端機、各種系統工具。本章談的是核心負責的那些核心職責——也就是 OSTEP 的三件事。

虛擬化 CPU(一):行程是什麼

從「虛擬化 CPU」開始。這裡的主角,是一個你天天聽到、卻未必說得清楚的詞:行程(process)。 先把一個最常見的混淆釐清:

程式(program)是躺在硬碟上的一份檔案;行程(process)是這份程式正在執行中的一個實體。

打個比方:程式像一張食譜(寫在紙上的靜態指令),行程則是廚師正在照這張食譜煮菜的整個過程—— 他手邊有一鍋正在滾的湯(記憶體裡的資料)、看到食譜的第幾行了(程式計數器 PC)、手上拿著哪些食材(暫存器)。 同一張食譜,可以同時有三個廚師各自照著煮——於是同一個程式,可以同時跑成三個行程 (你開三個記事本視窗,就是三個行程跑同一個程式)。

所以一個行程,是「一個執行中的程式」加上它所有的狀態:它的記憶體(程式碼、資料、堆疊、堆積)、 它的暫存器(包括 PC)、它開了哪些檔案……作業系統把這些資訊打包成一個叫 行程控制塊(PCB,Process Control Block)的資料結構,用它來「記住」每個行程長什麼樣。 當作業系統要把 CPU 從一個行程切換到另一個,它做的就是「存好這個行程的狀態、載入那個行程的狀態」。

行程的三種狀態

一個行程的一生,大部分時間都在三種狀態之間切換:

  • 執行中(running):正在某顆 CPU 上跑。一顆 CPU 同一時刻只能有一個行程處於這個狀態。
  • 就緒(ready):準備好了、隨時能跑,只是還沒輪到它(CPU 正被別人用)。
  • 阻塞(blocked):在等某件事完成(例如等磁碟讀完、等網路回應、等鍵盤輸入),這段期間它不該佔著 CPU。

為什麼「阻塞」這個狀態這麼重要?因為 I/O 慢得驚人。CPU 一秒能算數十億次,但去硬碟讀一筆資料可能要等上千萬個 CPU 週期。 如果一個行程在等磁碟時還死佔著 CPU,那就是天大的浪費。所以作業系統一看到行程要等 I/O,就把它丟進「阻塞」, 立刻把 CPU 換給別的「就緒」行程跑。等 I/O 完成,再把它叫醒、放回「就緒」隊伍。這個切換,我們馬上會細談。

虛擬化 CPU(二):fork、exec、wait

行程是怎麼被「生」出來的?在 Unix/Linux 的世界裡(這也是業界與教學的主流),靠的是三個經典的系統呼叫: fork()、exec()、wait()。它們的設計初看怪異,卻優雅得令人拍案。

fork():一個行程,裂成兩個

fork() 大概是整個作業系統裡最詭異的一個函式:你呼叫它一次,它卻回傳兩次。 為什麼?因為呼叫的那一瞬間,作業系統把目前這個行程完整複製了一份—— 記憶體、暫存器、PC、開啟的檔案,全部一模一樣。於是現在有兩個幾乎相同的行程在跑, 它們都從 fork() 回傳的那一行「醒來」,繼續往下執行同一份程式碼。

那要怎麼分辨誰是誰?靠 fork() 的回傳值:在「父行程(parent)」裡,它回傳新生子行程的編號(PID); 在「子行程(child)」裡,它回傳 0。看這個範例: 

#include <stdio.h>
#include <unistd.h>   /* fork() 在這個標頭檔 */

int main(void) {
    printf("起點:現在只有我一個行程\n");

    int pid = fork();          /* ★ 就是這一行,一個行程裂成兩個 ★ */

    if (pid < 0) {
        printf("fork 失敗了\n");           /* 系統資源不足時才會發生 */
    } else if (pid == 0) {
        printf("我是子行程,fork 回傳給我 0\n");
    } else {
        printf("我是父行程,我的子行程 PID = %d\n", pid);
    }
    return 0;
}

複製整份記憶體不會很慢嗎?

會,如果真的逐位元複製的話。所以現代系統用一招叫寫時複製(copy-on-write,COW): fork 的瞬間,父子共用同一份記憶體頁(只是各自標記成唯讀);只有當某一方真的動手寫某一頁時, 才在那一刻才幫它複製出私有的一份。大多數 fork 之後馬上 exec(整份記憶體立刻被換掉)的情況,根本不必複製,省下大量功夫。 這是「分頁」這個機制(等一下會講)帶來的漂亮副產品。

exec():把自己「變身」成另一個程式

光會複製自己有什麼用?威力在於 fork() 跟 exec() 的組合。 exec() 做的事是:把目前行程的記憶體內容,整個換成另一個程式—— 程式碼、資料全部換新,從新程式的開頭開始跑,但 PID 不變。它像「換腦不換身分證」。

於是「執行一個新程式」的標準套路,就是 fork 一個自己,然後讓子行程 exec 成想跑的程式。 這正是你每次在終端機打指令、或點開一個 App 時,背後發生的事: 

#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/wait.h>

int main(void) {
    int pid = fork();

    if (pid == 0) {
        /* 子行程:把自己變身成 `ls -l` 這個程式 */
        char *args[] = { "ls", "-l", NULL };   /* 參數列,結尾要放 NULL */
        execvp("ls", args);
        /* 如果 exec 成功,下面這行「永遠不會」被執行——因為這個行程已經不是原本的程式了 */
        perror("exec 失敗");
        exit(1);
    } else {
        /* 父行程:等子行程做完,再繼續 */
        wait(NULL);
        printf("子行程結束了,換我繼續\n");
    }
    return 0;
}

注意 exec 那行詭異的特性:成功的話,它下面的程式碼永遠跑不到,因為這個行程的「靈魂」已經被 ls 取代了。只有 exec 失敗(例如找不到那個程式)時,才會落到下面的錯誤處理。

wait():父行程等子行程

範例裡的 wait(NULL) 讓父行程暫停,直到子行程結束。 這保證了輸出順序(子行程先做完,父行程才印那句話),也讓父行程有機會「收屍」—— 回收子行程結束後殘留的資源。如果父行程不 wait 就先走了,那些結束卻沒被回收的子行程會變成 殭屍行程(zombie):已經死了,卻還在行程表裡佔個位子,等著被父行程確認。

🔑
fork + exec + wait:整個系統的繁衍方式

這三個系統呼叫,撐起了整個 Unix 的行程世界:fork 複製出一個新行程、exec 把它變成想跑的程式、 wait 讓父行程等它做完。你的終端機(shell)就是個無限迴圈:讀你打的指令 → fork → 子行程 exec 成那個指令 → wait 等它跑完 → 再問你下一個指令。 你開機後看到的所有行程,追根溯源,全都是從第一個行程(init,PID 1)一路 fork 出來的——這是一棵巨大的行程家族樹。

排程:幾顆 CPU,跑上百個行程

現在有了一堆行程,但 CPU 可能只有 8 顆,行程卻有好幾百個。怎麼辦?作業系統的答案是 分時多工(time sharing):讓每個行程輪流跑一小段時間(可能只有幾毫秒), 切得夠快,你就感覺它們在同時跑——就像電影其實是一張張靜止照片快速播放,你卻看成連續動作。

「把 CPU 從行程 A 換給行程 B」這個動作,叫上下文切換(context switch)。 作業系統得先把 A 的全部狀態(暫存器、PC……)存進 A 的 PCB,再把 B 的狀態從 B 的 PCB 載回 CPU, 然後 B 就接著上次中斷的地方繼續跑。這個切換本身要花時間(純屬「行政成本」,沒做任何有用計算), 所以切太頻繁也不好——這是個需要拿捏的權衡。

負責決定「下一個換誰上」的那段程式,叫排程器(scheduler),它用的規則叫排程策略(scheduling policy)。 在比較策略好壞前,先定義兩個關鍵指標:

  • 週轉時間(turnaround time) = 完成時刻 − 抵達時刻。從「來排隊」到「全部做完」總共等了多久。批次運算(例如跑一個科學模擬)在乎這個。
  • 回應時間(response time) = 第一次被排到 CPU 的時刻 − 抵達時刻。從「來排隊」到「終於開始動」等了多久。互動式程式(例如你打字、滑網頁)在乎這個——你要的是「立刻有反應」。

策略規則優點缺點
FIFO(先到先做)誰先來誰先跑,做完才換下一個最簡單、最公平直覺一個大工作排前面,後面全被卡住(「車隊效應」)

週轉 vs 回應:一個永恆的取捨

你沒辦法同時把兩者都做到最好。要「總體最快做完」(週轉短),就該讓短工作先做、少切換; 要「人人都覺得很順」(回應快),就得頻繁輪轉、別讓任何人等太久。這正是為什麼你的手機/筆電用的是 MLFQ 這類混合策略:它偷偷觀察每個行程的行為——常常很快放掉 CPU 去等 I/O 的(像你打字的編輯器), 判定它是「互動式」,給它高優先級、讓你感覺超順;一直猛吃 CPU 的(像影片轉檔),降級到背景慢慢跑。 你感覺不到它在做這些,但這正是它身為「魔術師」的功力。

虛擬化記憶體:每個行程都以為自己獨佔整片記憶體

虛擬化完 CPU,換記憶體。這裡藏著作業系統最精彩的一個魔術。先想一個問題:你電腦同時跑著幾十個程式, 它們都在用記憶體。如果每個程式都直接讀寫實體記憶體的真實位址,會出大事—— 程式 A 一個寫錯的位址,就可能覆蓋掉程式 B 的資料,甚至覆蓋掉作業系統自己。一團亂。

作業系統的解法美得驚人:它給每一個行程都發一份「假的、私有的記憶體」,叫 位址空間(address space)。在這個行程眼裡,它彷彿從位址 0 開始、獨佔了一整片連續而廣大的記憶體, 想用哪格用哪格,完全不知道、也不需要知道別人的存在。這份美好的假象,就叫 虛擬記憶體(virtual memory)。

但實體記憶體只有一塊啊!假象怎麼變出來?關鍵在於:行程用的位址全是虛擬位址(virtual address), 是假的;每次它要存取記憶體,中間都有人偷偷把這個虛擬位址翻譯成真正的 實體位址(physical address)。這個翻譯動作叫位址轉譯(address translation), 由 CPU 裡一個專門的硬體 MMU(記憶體管理單元,Memory Management Unit)飛快完成。

分頁:把記憶體切成固定大小的方塊

要怎麼做這個翻譯?最主流的方法叫分頁(paging)。它的點子是:把虛擬記憶體切成許多固定大小的小塊, 叫分頁(page)(常見是 4 KB);把實體記憶體也切成同樣大小的小塊,叫頁框(page frame)。 然後作業系統替每個行程維護一張對照表,叫分頁表(page table),記著 「這個行程的第 N 號虛擬分頁,實際放在實體記憶體的第幾號頁框」。

於是翻譯就變成查表:CPU 拿到一個虛擬位址,MMU 把它拆成「分頁號碼 + 頁內位移」, 用分頁號碼去分頁表查出對應的頁框號碼,拼上位移,就得到真正的實體位址。妙的地方在於: 連續的虛擬分頁,在實體記憶體裡可以是東一塊、西一塊散落的——行程完全察覺不到,它眼中永遠是整齊連續的。

為什麼要這麼麻煩?三個理由

多繞這麼一層翻譯,值得嗎?太值得了。它一口氣解決了三個大問題:

  • 隔離(isolation):每個行程有自己的分頁表,它的虛擬位址只能映射到「分配給它的」頁框。 行程 A 根本沒辦法表達出「我要碰行程 B 的記憶體」這件事——它的位址空間裡壓根沒有通往 B 的路。一個行程當掉,不會拖垮別人。
  • 保護(protection):分頁表的每一筆還能附帶權限位元(可讀/可寫/可執行)。 想偷寫唯讀的程式碼區?MMU 當場攔下、丟出例外。這也是擋住許多攻擊的第一道牆。
  • 超額使用(overcommitment):行程要的記憶體總和,可以大於實體記憶體! 因為不是每一頁都隨時要用——暫時用不到的頁,可以先「換」到磁碟上,要用時再搬回來。

最後這點就是換頁(swapping / paging to disk):把暫時不用的分頁挪去磁碟,騰出實體記憶體; 等行程真的要存取那一頁時,才觸發一次「缺頁(page fault)」,作業系統再把它從磁碟搬回來。 這讓你能在 16 GB 的機器上,跑出「彷彿有更多記憶體」的效果——代價是搬進搬出很慢。 (換頁與快取的細節,我們留到第十一部第 48 章「計算機結構」再深入。)

💡
TLB:給翻譯加一個快取

每次存取記憶體都要先查一次分頁表——但分頁表自己也住在記憶體裡,豈不是每次存取都要多一次存取?太慢了。 所以 MMU 裡有一塊極快的小快取,叫 TLB(Translation Lookaside Buffer),專門記住最近用過的「虛擬頁 → 頁框」對照。 因為程式存取記憶體有很強的「區域性」(reuse 同一批分頁),TLB 命中率非常高,絕大多數翻譯一查 TLB 就有答案,不必碰分頁表。 這是「快取」這個思想無所不在的又一例證。

記憶體與 C:堆疊、堆積、與兩個經典惡夢

虛擬記憶體給了每個行程一片乾淨的位址空間。那這片空間裡面是怎麼安排的?用 C 寫程式時你會直接撞上它。 一個行程的位址空間,大致分成這幾區:

我們聚焦兩個你寫 C 時天天在用、卻最容易搞錯的區域:堆疊(stack)與堆積(heap)。

  • 堆疊:放區域變數和函式呼叫資訊。它自動管理——你進入一個函式,它的區域變數就被推上堆疊; 函式一回傳,這些變數立刻消失。快、省事,但壽命被綁死在函式裡。
  • 堆積:放你「想要它活久一點」的資料。你用 malloc() 跟它手動要一塊, 用完得手動用 free() 還回去。自由,但責任也全在你身上。
#include <stdlib.h>

int *make_array(int n) {
    int local[100];                    /* 堆疊:函式一回傳就消失 → 回傳它的位址是大忌! */
    int *arr = malloc(n * sizeof(int)); /* 堆積:跟系統要一塊,函式結束後依然存在 */
    for (int i = 0; i < n; i++) {
        arr[i] = i * i;                /* 用指標 arr 像用陣列一樣存取 */
    }
    return arr;       /* OK:回傳堆積的指標,呼叫者用完記得 free(arr) */
}

這裡出現了 C 的靈魂——指標(pointer):一個「存放某個記憶體位址的變數」。 malloc 回傳的就是一個指標,指向它在堆積上幫你劃出來的那塊空間的開頭。 想拿到某個變數的位址,用取址運算子 &;想透過指標去存取它指向的內容,用解參考運算子 *。 指標讓 C 強大又危險——而那份危險,正是接下來兩個惡夢的根源。

惡夢一:記憶體洩漏(memory leak)

跟堆積要了記憶體,卻忘了還,就叫記憶體洩漏。那塊記憶體沒有被釋放、也沒有任何指標還指著它—— 它就這樣「迷失」了,直到行程結束都拿不回來。一次洩一點看似無害,但若發生在長期運行的迴圈或伺服器裡, 記憶體會像漏水的水管一樣越積越多,最後耗盡、程式崩潰。 

void leaky(void) {
    int *p = malloc(100 * sizeof(int));  /* 跟堆積要了一塊 */
    p[0] = 42;
    /* … 用完了,但忘記 free(p) … */
}   /* ← p 這個「指標」隨函式結束消失了,但它指向的那塊堆積記憶體沒被釋放 → 洩漏 */

惡夢二:懸空指標(dangling pointer)

反過來,釋放了一塊記憶體,卻還留著指向它的指標,然後又去用——這個指標就成了懸空指標。 它指向的那塊空間已經被系統收回、隨時可能配給別人。透過它讀寫,就是踩進別人家—— 這是「未定義行為(undefined behavior)」,可能當場崩潰、可能默默讀到垃圾、最糟的是看似正常卻悄悄毀損資料。 

int *p = malloc(sizeof(int));
*p = 10;
free(p);          /* 把這塊還給系統 */
*p = 20;          /* ⛔ 危險!p 已是懸空指標,這塊記憶體可能已被別人拿去用了 */
p = NULL;         /* ✓ 好習慣:free 之後立刻把指標設成 NULL,避免誤用 */
⚠️
記憶體錯誤不只是 bug,更是資安漏洞

C/C++ 把記憶體管理的全部責任交給你,換來極致的效能與控制,代價是這些錯誤防不勝防:洩漏、懸空指標、 重複釋放(double free)、還有最惡名昭彰的緩衝區溢位(buffer overflow)——寫入超過配置範圍, 覆蓋掉旁邊的資料甚至堆疊上的「返回位址」。攻擊者能精心構造輸入,利用這類漏洞讓程式跳去執行他的程式碼。 歷史上無數重大資安事件都源於此。記憶體安全是一條貫穿系統與資安的主線,我們會在 第七部「資訊安全」(尤其是防禦性程式設計那章)正面拆解緩衝區溢位的攻防。 現代語言(Rust、Go、Java、Python)用各種手段——自動垃圾回收、所有權系統——來幫你擋掉大半, 但要真正懂它們在防什麼,你得先親眼看過 C 裡的這些坑。

並行:當很多事同時發生

到這裡,OSTEP 的第一件事(虛擬化)講完了。接下來是最燒腦、也最迷人的第二件事:並行(concurrency)。 起點是一個比「行程」更輕巧的概念:執行緒(thread)。

前面說,行程之間是隔離的——各有各的位址空間,互不相見。但有時候,你希望同一個程式裡有「好幾條執行流」 同時進行,而且它們要共享同一批資料。例如一個瀏覽器:一條執行流負責下載網頁、一條負責渲染畫面、 一條負責回應你的捲動——它們得共用同一份頁面資料。每個都開成獨立行程太重(資料還不能共享),這時就用執行緒。

執行緒,是「同一個行程的位址空間裡,多條獨立的執行流」。它們共享同一片記憶體(程式碼、堆積、全域變數),但各有各自的堆疊與暫存器(包括 PC)。

打個比方:行程是一間公寓(有自己的牆、自己的家具,鄰居進不來);執行緒則是住在同一間公寓裡的室友—— 共用客廳、廚房、冰箱(共享記憶體),但各有各的房間(各自的堆疊)。共享帶來效率與便利, 也帶來室友之間的經典衝突:兩個人同時要用同一個東西,怎麼辦?

並行的惡夢:競爭條件

看一個看似無害的程式:開兩條執行緒,各自把一個共享變數 counter 加 100 萬次。直覺上,最後應該是 200 萬。 

#include <stdio.h>
#include <pthread.h>

int counter = 0;            /* 兩條執行緒共享的變數 */

void *worker(void *arg) {
    for (int i = 0; i < 1000000; i++) {
        counter++;          /* 看似一個動作,其實是三個步驟! */
    }
    return NULL;
}

int main(void) {
    pthread_t t1, t2;
    pthread_create(&t1, NULL, worker, NULL);   /* 開第一條執行緒 */
    pthread_create(&t2, NULL, worker, NULL);   /* 開第二條執行緒 */
    pthread_join(t1, NULL);                     /* 等它們做完 */
    pthread_join(t2, NULL);
    printf("最後 counter = %d(你以為會是 2000000)\n", counter);
    return 0;
}

實際跑起來,你會得到像 1377492 這種每次都不一樣、而且小於 200 萬的怪數字。 為什麼?因為那行天真的 counter++,在 CPU 眼裡不是一個動作,而是三個:

  1. 讀:把 counter 從記憶體載入暫存器。
  2. 加:暫存器裡的值 +1。
  3. 寫:把結果寫回記憶體的 counter。

現在想像兩條執行緒「交錯」執行這三步:執行緒 A 讀到 counter = 5,還沒寫回去, 時間片到了被切換出去;執行緒 B 也讀到 counter = 5,加成 6 寫回去;然後 A 回來,它手上還握著舊的 5, 加成 6 又寫回去。兩次遞增,結果只加了 1——有一次更新被悄悄吃掉了。

🔑
競爭條件、臨界區、不可分割性

競爭條件(race condition):程式的結果取決於多條執行緒「碰巧」誰先誰後的時序—— 時序不同,結果就不同,而你無法控制時序。這種 bug 時有時無、極難重現,是並行程式最可怕的敵人。
臨界區(critical section):那段「存取共享資源、不能被打斷」的程式碼(這裡就是 counter++)。
不可分割性(atomicity):我們真正想要的——讓臨界區「要嘛整段做完、要嘛完全沒做」, 中間不被別的執行緒插隊。問題的本質,就是 counter++ 不是不可分割的。

馴服並行:鎖、條件變數、號誌

怎麼讓臨界區不被插隊?作業系統與函式庫提供了幾種同步原語(synchronization primitive)。 最基本的一個叫鎖(lock,又稱互斥鎖 mutex)。

鎖的概念像廁所門上的「使用中」插銷:進臨界區前先「上鎖(lock)」,出來時「解鎖(unlock)」。 同一時間只有一條執行緒能持有鎖;別的執行緒想上鎖時,如果鎖已被佔,就只能等,直到持有者解鎖。 這就強制讓臨界區一次只有一個人在裡面——把「不可分割性」硬是造了出來。

用鎖修好前面那個程式,只要在臨界區前後加上 lock / unlock: 

#include <stdio.h>
#include <pthread.h>

int counter = 0;
pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;   /* 宣告並初始化一把鎖 */

void *worker(void *arg) {
    for (int i = 0; i < 1000000; i++) {
        pthread_mutex_lock(&lock);     /* 進臨界區:上鎖(若被佔則在此等待) */
        counter++;                     /* 這一段現在「不可分割」了 */
        pthread_mutex_unlock(&lock);   /* 離開臨界區:解鎖,放下一個進來 */
    }
    return NULL;
}

int main(void) {
    pthread_t t1, t2;
    pthread_create(&t1, NULL, worker, NULL);
    pthread_create(&t2, NULL, worker, NULL);
    pthread_join(t1, NULL);
    pthread_join(t2, NULL);
    printf("最後 counter = %d\n", counter);   /* 現在穩穩是 2000000 */
    return 0;
}

條件變數與號誌

鎖解決「不要同時改」,但並行還有另一類需求:「等某個條件成立再繼續」。 例如一條執行緒要等另一條把資料準備好。如果用「不斷迴圈檢查」(busy-waiting / 忙碌等待),會白白燒掉 CPU。 更好的工具是:

  • 條件變數(condition variable):讓執行緒「睡著等待」某個條件,直到別人喊它「醒來(signal)」。 典型用法:持有鎖時發現條件還不成立,就 wait(會自動放掉鎖去睡);另一條執行緒改變了狀態後, signal 把它叫醒。它把「等待」做得既正確又不浪費 CPU。
  • 號誌(semaphore):一個帶計數的同步工具,由 Dijkstra 提出。它有兩個操作: wait(把計數減 1,若變負就阻塞)和 post(把計數加 1,可能喚醒一個等待者)。 計數設成 1,它就退化成一把鎖;設成 N,就能用來「最多允許 N 個同時進入」。它能同時表達互斥與排序,非常萬用。

生產者-消費者問題

這些工具最經典的應用,是生產者-消費者問題(producer-consumer problem),也叫有界緩衝區問題。 場景是:一群「生產者」執行緒不斷產出資料、放進一個固定大小的緩衝區;一群「消費者」執行緒不斷把資料取走處理。 你天天在用它——影片播放器一邊「生產」(解碼下一段)、一邊「消費」(顯示這一段); 終端機的管線 a | b 也是(a 生產輸出、b 消費它)。難點在於要同時處理好三件事:

  • 互斥:生產者和消費者不能同時亂改緩衝區(用鎖)。
  • 緩衝區滿了:生產者得停下來等,直到有空位(用條件變數/號誌)。
  • 緩衝區空了:消費者得停下來等,直到有東西(用條件變數/號誌)。

把這三件事用同步原語正確地組合起來,正是並行程式設計的核心基本功。

死結:大家都在等別人

鎖能救你,也能害你。最有名的陷阱叫死結(deadlock):一群執行緒互相卡住,誰也動不了。 經典畫面是:執行緒 A 握著鎖 1、想要鎖 2;執行緒 B 握著鎖 2、想要鎖 1。兩個都在等對方先放手, 於是永遠地等下去。這就像兩台車在窄橋上對向相遇,誰都不肯倒車。

Coffman 在 1971 年指出:死結要發生,必須同時滿足四個條件,缺一不可—— 這意味著只要破壞掉任何一個,就能避免死結:

條件英文意思怎麼破壞它
互斥mutual exclusion資源一次只能一人用用無鎖(lock-free)資料結構,讓資源可共享
持有並等待hold and wait握著一個資源時,還去等另一個要嘛一次把所有需要的鎖全拿到,要嘛一個都別先拿
不可搶奪no preemption資源不能被強制收回拿不到第二個鎖時,主動放掉手上的、稍後重試

💡
最實用的一招:固定上鎖順序

實務上對付死結,最簡單有效的就是破壞「循環等待」:替所有鎖排個全域順序,規定每條執行緒都一律按同樣的順序上鎖。 這樣就不可能出現「A 拿了 1 等 2、B 拿了 2 等 1」的環——因為大家都會「先拿 1 再拿 2」。 一個簡單的約定,就讓整類 bug 消失。並行的世界裡,紀律往往比聰明更重要。

持久化:讓資料活過斷電

最後一件大事:持久化(persistence)。前面講的記憶體再怎麼虛擬,有個致命弱點—— 它是揮發性(volatile)的:一斷電,全部歸零。但你的照片、文件、程式碼,必須在關機後依然存在。 負責這件事的,是作業系統裡管理持久性儲存(磁碟、SSD)的那一塊,它給你的核心抽象,就是 檔案系統(file system)。

檔案系統提供兩個你再熟悉不過的抽象:

  • 檔案(file):一串有名字的位元組(bytes)。作業系統不在乎裡面是照片還是文字——對它來說都只是「一串位元組」。
  • 目錄(directory):一個裝著「名字 → 對應東西」的容器。目錄裡可以放檔案,也可以放別的目錄,於是長出整棵目錄樹。

inode:檔案真正的身分證

這裡有個關鍵洞見,顛覆很多人的直覺:檔案的「名字」其實不屬於檔案本身。 檔案真正的本體,是一個叫 inode 的資料結構,它記著這個檔案的所有後設資料(metadata): 大小、權限、擁有者、建立時間,以及最重要的——它的資料實際存在磁碟的哪些區塊。 每個 inode 有一個編號。

那名字呢?名字住在目錄裡。一個目錄,本質上就是一張表,記著 「名字 → inode 編號」的對應。所以當你打開 /home/me/report.txt,作業系統做的是: 在 / 目錄查到 home 的 inode、在 home 查到 me 的 inode、 在 me 查到 report.txt 對應的 inode 編號,最後讀那個 inode、循著它記的區塊位置,把資料撈出來。

檔案 API:open / read / write / close

程式怎麼操作檔案?透過一組簡潔的系統呼叫。核心就四個:open(開啟,拿到一個代表它的整數 檔案描述子 file descriptor)、read(讀)、write(寫)、close(關閉)。 看一個完整的「寫一行字,再讀回來」: 

#include <stdio.h>
#include <fcntl.h>     /* open 與 O_* 旗標 */
#include <unistd.h>    /* read, write, close */
#include <string.h>

int main(void) {
    /* === 寫 === 開檔(沒有就建立、有就清空),取得檔案描述子 fd === */
    int fd = open("hello.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);
    if (fd < 0) { perror("open"); return 1; }

    char *msg = "Hello, file system!\n";
    write(fd, msg, strlen(msg));   /* 把這串位元組寫進去 */
    close(fd);                     /* 關閉,釋放描述子 */

    /* === 讀 === 重新開檔(唯讀),把內容讀回緩衝區 === */
    fd = open("hello.txt", O_RDONLY);
    char buf[128];
    int n = read(fd, buf, sizeof(buf) - 1);   /* 最多讀 127 個位元組 */
    buf[n] = '\0';                  /* 補上字串結尾,才能當 C 字串印 */
    close(fd);

    printf("讀到:%s", buf);         /* 讀到:Hello, file system! */
    return 0;
}

注意那個檔案描述子(file descriptor):它是個小整數,代表「這個行程開啟的某個檔案」。 你早就見過三個天生就開好的描述子——0 是標準輸入(stdin)、1 是標準輸出(stdout)、 2 是標準錯誤(stderr)。這也是為什麼終端機的「重導向」和「管線」能這麼優雅: 它們不過是在偷偷換掉這些描述子背後接的東西而已。

儲存與可靠性:當斷電發生在寫到一半

檔案系統最深的挑戰,不是「怎麼存」,而是「萬一寫到一半斷電怎麼辦」。 底層的硬體先一句話帶過:磁碟(HDD)靠機械手臂在旋轉碟片上尋找磁區,所以「隨機讀寫」慢、「循序讀寫」快; 固態硬碟(SSD)用快閃記憶體、沒有機械零件,快得多,但有寫入次數壽命與「不能原地覆寫、得整塊抹除」的脾氣。 無論哪種,它們都比記憶體慢上好幾個數量級——這個落差,主宰了檔案系統的所有設計。

緩衝與快取:用記憶體掩護慢吞吞的磁碟

因為磁碟太慢,作業系統會在記憶體裡留一塊緩衝區/快取(buffer cache / page cache): 你讀過的資料先留在記憶體,下次再讀就直接命中、不必碰磁碟;你寫的資料也先進記憶體、晚點再「一次寫一批」到磁碟。 這讓 I/O 快得多——但也埋下一個隱患:資料「已經寫了」其實只到記憶體,還沒真的落到磁碟。 這時候斷電,資料就飛了。這正是為什麼資料庫等程式會在關鍵時刻呼叫 fsync(),強迫把快取真正刷到磁碟。

崩潰一致性:日誌如何力挽狂瀾

更棘手的是崩潰一致性(crash consistency)。想想前面的 inode:新增一個檔案,可能要同時更新好幾個地方—— 寫資料區塊、更新 inode、更新目錄、更新「哪些區塊還空著」的記錄。如果這幾步只做到一半就斷電, 檔案系統會進入不一致的鬼狀態:例如 inode 說「我用了區塊 100」,但配置表卻說「區塊 100 是空的」—— 遲早被覆寫,資料毀損。

現代檔案系統的主流解法叫日誌(journaling),點子借自資料庫:在真正動手改之前, 先把「我接下來要做哪幾件事」當成一筆完整紀錄,寫進一塊專門的日誌區;確認這筆紀錄完整寫好了, 才回頭去做真正的更新。這樣一來:

  • 若斷電發生在「日誌都還沒寫完」——那就當什麼都沒發生,原本的資料完好無損。
  • 若斷電發生在「日誌寫完了、但真正更新做到一半」——開機後,檔案系統照著日誌把這批操作重做一遍(replay),補完未竟之事。

無論哪種情況,檔案系統都不會卡在「做一半」的不一致狀態。要嘛完全沒做,要嘛完整做完—— 你會發現,這跟並行那邊追求的「不可分割性(atomicity)」是同一個靈魂,只是這次對抗的不是「別的執行緒」, 而是「隨時可能發生的斷電」。電腦科學裡,好的想法總是一再回來。

ℹ️
為什麼「正確地拔隨身碟」很重要

現在你懂了:當你拷貝檔案到隨身碟、進度條跑到 100%,資料可能還只在作業系統的寫入快取裡, 尚未真正落到隨身碟。直接拔掉,就可能寫到一半、毀損檔案甚至整個檔案系統。「安全移除」這個動作, 就是叫作業系統把快取全刷乾淨(flush)、確認都落地了,才放你拔。 這個你嫌麻煩的小步驟,背後是一整套崩潰一致性的考量。

✏️ 練習 24.1 思考

看這段程式,假設 fork 都成功。它總共會印出幾行 "hi"?畫出行程樹來說明。

#include <stdio.h>
#include <unistd.h>
int main(void) {
    fork();
    fork();
    printf("hi\n");
    return 0;
}
看解答

4 行。每個 fork() 會讓當下的行程數量「翻倍」。

  • 一開始:1 個行程。
  • 第一個 fork() 之後:1 → 2 個行程。
  • 第二個 fork() 之後:每個行程又各自裂成兩個,2 → 4 個行程。

這 4 個行程都會執行到 printf,所以印 4 次。通則:n 個連續的 fork(), 會產生 2ⁿ 個行程。行程樹長這樣(P 是原始行程):

          P
        /   \         ← 第一個 fork:P 生出 C1
      P      C1
     / \    / \       ← 第二個 fork:P 生 C2、C1 生 C3
    P  C2  C1  C3
   葉子有 4 個 → 4 行 hi
✏️ 練習 24.2 思考

下面這段呢?它印幾次 "X"?(小心 && 的短路特性與 fork() 的回傳值。)

#include <stdio.h>
#include <unistd.h>
int main(void) {
    if (fork() && fork()) {     /* 兩個 fork,中間是「邏輯且」 */
        fork();
    }
    printf("X\n");
    return 0;
}
看解答

4 次。兩個關鍵:① fork() 在父行程回傳子 PID(非 0,為真)、在子行程回傳 0(為假); ② && 會短路——左邊為假時,右邊那個 fork() 根本不執行。我們追蹤每一個被製造出來的行程的命運:

  1. 第一個 fork()(&& 左運算元):原始行程 P 執行它,生出子行程 C1。行程數 1 → 2。
    • 在 C1 裡,第一個 fork 回傳 0(假)→ && 短路,不執行第二個 fork,整個 if 為假 → 直接去 printf。C1 印 1 次。
    • 在 P 裡,第一個 fork 回傳真 → 繼續算右運算元。
  2. 第二個 fork()(&& 右運算元):只有 P 會跑到這裡,它生出 C2。行程數 2 → 3。
    • 在 C2 裡,第二個 fork 回傳 0 → if 條件(真 && 假)為假 → 跳過內層 fork,直接去 printf。C2 印 1 次。
    • 在 P 裡,兩個都為真 → 進入 if 內層。
  3. 第三個 fork()(if 內層):只有 P 會跑到,它生出 C3。行程數 3 → 4。P 與 C3 都離開 if 去 printf。P 與 C3 各印 1 次。

合計四個行程都跑到 printf:C1 + C2 + P + C3 = 4 次。

本題的教學重點:① && 短路會讓某些子行程跳過後面的 fork,不能再無腦套用 2ⁿ; ② 最穩的數法是「追蹤每個被製造出來的行程,各自會不會跑到 printf」。別被嚇人的巢狀條件唬住,逐一拆解即可。

✏️ 練習 24.3 實作

下面這個轉帳程式有競爭條件:兩條執行緒同時對同一個帳戶反覆「存 1 元、再領 1 元」, 照理說餘額最後該回到 0,但實際跑出來常是個亂七八糟的非零值。請指出競爭條件在哪, 並用一把 mutex 修正它。

#include <stdio.h>
#include <pthread.h>
int balance = 0;
void *txn(void *arg) {
    for (int i = 0; i < 500000; i++) {
        balance = balance + 1;   /* 存 1 元 */
        balance = balance - 1;   /* 領 1 元 */
    }
    return NULL;
}
看解答

競爭條件在哪:balance = balance + 1 和 balance = balance - 1 都不是不可分割的 (各自是「讀-改-寫」三步)。兩條執行緒交錯時,某些更新會被覆蓋,導致最終餘額不為 0。 臨界區是「對 balance 的整段讀寫」。修法:用一把鎖把臨界區框起來。

#include <stdio.h>
#include <pthread.h>
int balance = 0;
pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;

void *txn(void *arg) {
    for (int i = 0; i < 500000; i++) {
        pthread_mutex_lock(&lock);     /* 進臨界區 */
        balance = balance + 1;
        balance = balance - 1;
        pthread_mutex_unlock(&lock);   /* 離開臨界區 */
    }
    return NULL;
}

int main(void) {
    pthread_t a, b;
    pthread_create(&a, NULL, txn, NULL);
    pthread_create(&b, NULL, txn, NULL);
    pthread_join(a, NULL);
    pthread_join(b, NULL);
    printf("balance = %d\n", balance);   /* 穩穩是 0 */
    return 0;
}

進階思考:把鎖放在迴圈外面(整個迴圈框成一個臨界區)結果也正確,而且因為上鎖/解鎖次數從 100 萬次降到 1 次, 通常更快;但缺點是:持鎖期間另一條執行緒完全動不了,失去了並行性。「鎖的粒度(granularity)」要多細, 是效能與並行度之間的經典取捨。

✏️ 練習 24.4 思考

三個工作在時刻 0 同時抵達,需要的 CPU 時間分別是:A = 30、B = 6、C = 6(單位:毫秒)。 假設不可搶占。請分別算出 FIFO(順序 A→B→C) 與 SJF 兩種排程下, 每個工作的週轉時間,以及平均週轉時間。哪個比較好?為什麼?

看解答

週轉時間 = 完成時刻 − 抵達時刻;這裡都在時刻 0 抵達,所以週轉時間 = 完成時刻。

FIFO(A→B→C):

  • A 在 0 開始,30 完成 → 週轉 30
  • B 在 30 開始,36 完成 → 週轉 36
  • C 在 36 開始,42 完成 → 週轉 42
  • 平均 =(30 + 36 + 42)/ 3 = 36

SJF(短的先,順序 B→C→A):

  • B 在 0 開始,6 完成 → 週轉 6
  • C 在 6 開始,12 完成 → 週轉 12
  • A 在 12 開始,42 完成 → 週轉 42
  • 平均 =(6 + 12 + 42)/ 3 = 20

SJF 明顯較好(平均 20 vs 36)。直覺:把又快又短的 B、C 先做掉,讓它們不必苦等那個 30 毫秒的大塊頭 A; 只有 A 一個被延後,但它本來就要花最久。事實上可以證明:對「同時抵達」的一批工作, SJF 能讓平均週轉時間達到理論最小值。它的致命缺點是:現實中你通常無法預知每個工作要跑多久—— 這正是 MLFQ 想用「觀察行為來猜」去逼近 SJF 的原因。

✏️ 練習 24.5 思考

(a) 用你自己的話解釋:虛擬記憶體(分頁 + 每個行程一張分頁表)是如何提供行程之間的「隔離」的? 為什麼行程 A「即使想」也碰不到行程 B 的記憶體?
(b) 列出死結的四個必要條件,並說明:為什麼「給所有鎖規定一個固定的上鎖順序」就能避免死結?它破壞了哪一個條件?

看解答

(a) 虛擬記憶體如何提供隔離:每個行程用的都是虛擬位址,而且每個行程有自己獨立的一張分頁表。 行程要存取記憶體時,MMU 只會用「該行程自己的分頁表」去翻譯,而這張表裡只登記了「分配給這個行程的頁框」。 所以行程 A 能表達的任何虛擬位址,翻譯出來都只會落在 A 自己的頁框上——它的位址空間裡根本沒有一條路能通到 B 的頁框。 即使 A 寫一個「想越界」的位址,要嘛被翻到 A 自己的另一頁、要嘛該頁根本沒映射而觸發例外被作業系統攔下。 關鍵在於:行程無法繞過 MMU 直接用實體位址(那是核心模式才有的特權),所以這道牆是硬體強制、無法翻越的。

(b) 死結四條件:① 互斥 ② 持有並等待 ③ 不可搶奪 ④ 循環等待。四者缺一不可, 所以破壞任何一個即可避免死結。

「固定上鎖順序」破壞的是循環等待(circular wait)。死結的環長這樣:A 持鎖 1 等鎖 2、B 持鎖 2 等鎖 1。 但如果規定「所有人都必須先鎖編號小的、再鎖編號大的」,那麼想同時持有 1 和 2 的人,一律會先拿 1 再拿 2—— 不可能出現「有人先拿了 2 再回頭等 1」的情況,環就形成不了。沒有環,就沒有死結。 這是實務上最常用、最便宜的防死結手段。

✏️ 練習 24.6 實作

下面這個 C 程式有兩個記憶體 bug。請各指出是什麼類型(從「記憶體洩漏 / 懸空指標 / 重複釋放」中選),並改正。

#include <stdlib.h>
#include <stdio.h>

int *first_square(int n) {
    int *arr = malloc(n * sizeof(int));
    for (int i = 0; i < n; i++) arr[i] = i * i;
    int *result = malloc(sizeof(int));
    *result = arr[0];
    return result;        /* bug A:這裡少做了一件事 */
}

int main(void) {
    int *r = first_square(5);
    printf("%d\n", *r);
    free(r);
    free(r);              /* bug B */
    return 0;
}
看解答

Bug A 是記憶體洩漏:函式裡 malloc 了 arr,用完(只取了 arr[0])卻沒有 free(arr) 就回傳了。 函式一結束,arr 這個區域指標就消失,但它指向的那塊堆積記憶體沒被釋放、也再沒人指著它 → 洩漏。

Bug B 是重複釋放(double free):r 被 free 了兩次。第二次 free 一塊已經還掉的記憶體是未定義行為, 可能直接崩潰、也可能破壞堆積管理結構。(它跟「懸空指標」是近親:第一次 free 後 r 就已成懸空指標,第二次 free 正是誤用它。)

改正版:

int *first_square(int n) {
    int *arr = malloc(n * sizeof(int));
    for (int i = 0; i < n; i++) arr[i] = i * i;
    int *result = malloc(sizeof(int));
    *result = arr[0];
    free(arr);            /* ✓ 修正 A:用完就還 */
    return result;
}

int main(void) {
    int *r = first_square(5);
    printf("%d\n", *r);
    free(r);
    r = NULL;             /* ✓ 修正 B:free 後設 NULL,杜絕重複釋放/懸空誤用 */
    return 0;             /*    (對 NULL 指標再 free 是安全的、什麼都不做) */
}

養成兩個習慣能擋掉大半記憶體 bug:① 每個 malloc 都先想好「誰、在哪裡負責 free」; ② free 之後立刻把指標設成 NULL。

}

int main(void) {
    pthread_t th[NTHREADS];
    for (int i = 0; i < NTHREADS; i++)
        pthread_create(&th[i], NULL, worker, NULL);
    for (int i = 0; i < NTHREADS; i++)
        pthread_join(th[i], NULL);
    printf("counter = %ld\n", counter);   /* 1000000 */
    return 0;
}

編譯時記得連結 pthread:gcc prog.c -o prog -lpthread。 試試把 lock/unlock 兩行註解掉再跑幾次,你會看到結果每次都不同、且小於一百萬——親眼見證競爭條件。

(乙)用 fork 建行程樹:

#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>

int main(void) {
    for (int i = 0; i < 3; i++) {
        int pid = fork();
        if (pid == 0) {
            /* 子行程:印自己與父親的 PID,然後結束(別再繼續 fork!) */
            printf("子行程 #%d:我的 PID=%d,父 PID=%d\n", i, getpid(), getppid());
            return 0;        /* ★ 關鍵:子行程在這裡離開迴圈,否則會繼續 fork 出孫子 */
        }
        /* 父行程:繼續迴圈,fork 下一個 */
    }
    for (int i = 0; i < 3; i++)
        wait(NULL);          /* 等三個子行程都結束 */
    printf("all children done\n");
    return 0;
}

最關鍵的一行是子行程裡的 return 0;:如果忘了讓子行程跳出迴圈, 每個子行程也會繼續往下 fork,行程數量爆炸(變成 2³ 而不是 1+3)。 這是寫 fork 迴圈最常見的坑——務必讓子行程「做完自己的事就離開」。

把這一章串起來

我們從一位忙翻天的管家出發,走完了作業系統的全景。如果只能帶走一句話,那就是 OSTEP 的那三個字—— 作業系統,本質上就在做三件事:

  • 虛擬化(讓你以為獨佔):用行程把一顆 CPU「變」成人人有份(靠分時與上下文切換、靠排程器在週轉與回應間權衡); 用虛擬記憶體(分頁、分頁表、MMU、TLB)讓每個行程都以為獨佔整片記憶體,同時換來隔離、保護與超額使用。
  • 並行(管好同時發生的事):執行緒讓一個行程裡多條執行流共享記憶體, 卻也召來競爭條件這個惡夢;我們用鎖、條件變數、號誌守住臨界區、造出不可分割性, 並小心死結的四個條件(固定上鎖順序是最實用的解藥)。
  • 持久化(資料不因斷電消失):檔案系統用檔案與目錄當抽象、用 inode 當檔案的本體、 用簡潔的 open/read/write/close 當介面;再用快取掩護慢吞吞的磁碟、用日誌對抗「寫到一半就斷電」, 守住崩潰一致性——說穿了,又是在追求那個熟悉的「不可分割性」。

回頭看,這三件事其實是同一個母題的三種變奏:把醜陋、有限、危險的硬體,變成乾淨、寬裕、安全的抽象。 這正是第 1 章就埋下的那條主線——抽象——在系統層級最壯觀的一次演出。你寫的每一行 Python、每一次 open("file")、每一個 x = x + 1,背後都有這位沉默的管家,替你打點好了一切。

但有一件事,作業系統一個人辦不到:讓你的電腦跟世界上其他幾十億台電腦對話。 當資料要離開這台機器、跨越大海去到另一端,我們需要一整套新的抽象與協定。 下一章,我們就把視野從「一台電腦」拉到「連起來的所有電腦」——進入計算機網路的世界。
