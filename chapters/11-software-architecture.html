軟體架構

蓋一間狗屋,你可以想到哪做到哪;蓋一座城市,你得先規劃。當程式從幾百行長成幾十萬行,決定它生死的,往往不是某個函式寫得多巧,而是它的「骨架」——軟體架構。

從「蓋房子」到「規劃城市」

你前幾章學會了函式、遞迴、物件導向、設計模式。這些都是「把一個房間蓋好」的功夫: 把一段邏輯寫得乾淨、把一個類別設計得優雅。但有一天,你的程式不再是一個房間, 而是一整座城市:幾十萬行程式碼、幾十個模組、好幾個團隊同時在上面工作。

這時你會發現一個殘酷的事實:就算每一個房間都蓋得很漂亮,整座城市還是可能爛得一塌糊塗—— 馬路接不起來、水電亂拉、消防車開不進去。城市的好壞,不取決於單一棟房子, 而取決於更高一層的東西:分區、幹道、管線該怎麼配置。這一層,就是架構。

🔑
什麼是軟體架構

軟體架構(software architecture)是一個系統的高層結構: 它由哪些主要元件(components)組成、每個元件負責什麼(職責)、 以及它們如何彼此溝通(連接器 connectors)。 一句話:架構回答「這個系統由哪些大塊拼成、它們怎麼合作」,而不是「某個函式裡的迴圈怎麼寫」。

注意「高層」這兩個字。架構是站在很高的地方俯瞰系統——你看到的是街區與幹道, 不是每一塊磚。那「每一塊磚怎麼砌」是誰的事?是細部設計(detailed design)的事: 某個類別有哪些方法、某個演算法選哪種、某個變數叫什麼名字。兩者是同一件事的不同「高度」:

為什麼架構決策最貴、最難改?

如果你把臥室的牆漆錯顏色,重漆就好,半天的事。但如果你把整座城市的「污水處理廠」蓋在市中心、 把「工業區」設在水源上游——這種錯誤,等城市蓋起來才發現,幾乎不可能修了, 要打掉重來。軟體一模一樣。

改一個函式的內部實作,影響範圍只在那個函式裡;但改一個架構決策——比方「我們當初決定用單一資料庫, 現在要拆成十個」——會牽動整個系統的每一個角落。架構決策的特徵是: 它們做得早(專案初期、資訊最少的時候)、影響廣(碰到所有人)、 而且改起來貴(越晚改越貴)。

⚠️
但也別過度設計

「架構很重要、改起來很貴」常被誤讀成「一開始就要把所有可能性都設計進去」。 錯。為一個還沒有用戶的產品先蓋好「能撐十億人」的架構,叫過度工程(over-engineering), 通常會害死你——你花光力氣解決不存在的問題,真正的需求卻沒空做。 好的架構是「剛好夠用、又留有演化空間」,而不是「一步到位的完美」。 後面的貫穿案例,就是要讓你看到架構如何隨需求成長。

元件與連接器:看懂任何系統的萬用眼鏡

架構師描述系統時,最常用的一副「眼鏡」叫做元件與連接器(components & connectors)觀點。 它簡單到只有兩種東西:

  • 元件(component):一個有明確職責、可以獨立思考的「計算單位」。 它可能是一個模組、一個物件、一個服務、一個程式,甚至一整台伺服器。畫圖時通常是個方塊。
  • 連接器(connector):元件之間溝通的管道。 它可能是一次函式呼叫、一個 HTTP 請求、一條資料庫連線、一個訊息佇列。畫圖時通常是條線(箭頭)。

就這兩樣,你已經能畫出世界上絕大多數系統的骨架。來看一個你天天在用的東西——一個典型的網頁應用:

模組化:把大象切成可以單獨搬動的塊

要能畫出上面那張圖,前提是系統本身真的被「切成塊」了。這種「把系統拆成相對獨立、各司其職的單元」的做法, 就是模組化(modularity)。它是人類對付複雜度的老招數——你不可能一次理解十萬行程式碼, 但你可以一次理解一個「只負責寄信」的模組。

模組化帶來四個實際的好處,每一個都直接對應到工程上的痛點:

  • 可理解:一次只需要看懂一塊,不必把整個系統塞進腦袋。
  • 可分工:不同人/團隊可以同時負責不同模組,不互相踩腳。
  • 可替換:把「寄信模組」從 Gmail 換成 SendGrid,不用動到其他地方。
  • 可測試:可以單獨拿一個模組出來測,不必把整座城市點亮。

但「切成塊」本身還不夠——怎麼切才是學問。切得好,系統像樂高,拆裝自如; 切得爛,系統像一碗打結的義大利麵,動一根就扯動全部。要判斷切得好不好,我們需要兩把尺。

兩把尺:低耦合、高內聚

這是整章——甚至整個軟體工程——最該刻進骨子裡的一句話。衡量「模組切得好不好」, 幾乎可以濃縮成兩個詞:

🔑
低耦合、高內聚

耦合(coupling):模組之間的依賴程度。A 改一下,B 就得跟著改,代表它們耦合高。 我們要的是低耦合——模組之間越獨立越好。
內聚(cohesion):模組內部各元素的相關程度。一個模組裡的東西如果都在做同一件事, 代表它內聚高。我們要的是高內聚——每個模組「只做一件事、並把它做好」。
口訣:對內抱緊(高內聚)、對外鬆綁(低耦合)。

耦合:為什麼它決定系統好不好改

想像兩種家電配線。第一種,每台電器都用一條獨立的線插到牆上的插座; 哪台壞了,拔掉換新就好,其他電器無感。第二種,所有電器的電線在牆裡纏成一團、彼此焊死; 想換一台冰箱,你得先把整面牆挖開,還很可能順手弄壞旁邊的電視。前者低耦合,後者高耦合。

左圖的連線數量會隨模組數量平方成長。5 個模組就可能有 10 條連線,10 個模組會有 45 條。 每一條線都是一個「你改我就得跟著改」的牽絆。系統一旦長到這個樣子,就進入了工程師口中的 「大泥球(big ball of mud)」——沒人敢動,因為沒人知道動一根線會扯垮哪裡。 這正是為什麼耦合直接決定一個系統好不好維護:低耦合的系統,修改是局部的; 高耦合的系統,修改是全域的賭博。

內聚:把相關的東西放在一起

內聚講的是另一面:一個模組內部是不是「名實相符」。一個叫 UserService 的模組, 如果裡面除了處理使用者,還塞了寄信、產生報表、連金流——那它內聚很低, 因為這些根本是不同的事。高內聚的模組,你光看名字就知道它做什麼,而且它只做那件事。

高內聚和低耦合常常是一體兩面:當你把「相關的東西聚在一起、不相關的東西分開」, 模組內部變得緊密(高內聚),模組之間自然就鬆了(低耦合)。我們用一段程式碼把這件事變具體。 下面是一個「下訂單」的服務,它把所有事情都攬在自己身上: 

# ✗ 反面教材:低內聚 + 高耦合
import smtplib
import mysql.connector

class OrderService:
    def place_order(self, user_email, item, qty):
        # 1. 直接連資料庫——綁死了 MySQL
        conn = mysql.connector.connect(host="localhost", user="root",
                                       password="123456", database="shop")
        cur = conn.cursor()
        cur.execute("INSERT INTO orders (item, qty) VALUES (%s, %s)", (item, qty))
        conn.commit()

        # 2. 直接寄信——綁死了 SMTP 與帳號
        server = smtplib.SMTP("smtp.gmail.com", 587)
        server.starttls()
        server.login("shop@gmail.com", "password")
        server.sendmail("shop@gmail.com", user_email, f"您的 {item} x{qty} 已成立")

        # 3. 直接印發票——把「呈現格式」也攪進來了
        print(f"===== 發票 =====\n品項:{item}\n數量:{qty}")

這個類別至少同時是「資料庫程式設計師、寄信員、會計」三種身分,內聚極低; 而且它直接依賴 MySQL 和 Gmail SMTP 的細節,耦合極高。 想換資料庫?改它。想換寄信服務?改它。想寫測試?幾乎不可能——你不會想在跑測試時真的寄一封信、真的寫進資料庫。

解法是把不同的職責拆出去,讓 OrderService 只負責「下訂單這件事」, 其他細節交給專門的模組,並且只透過介面使用它們(這個技巧叫 依賴注入,dependency injection): 

# ✓ 拆開後:每個類別高內聚,彼此低耦合
class OrderRepository:          # 只負責「存訂單」,內部用什麼資料庫是它的祕密
    def save(self, order): ...

class Notifier:                 # 只負責「通知使用者」,SMTP？簡訊？它的事
    def send(self, to, message): ...

class OrderService:             # 只負責「下訂單的業務流程」
    def __init__(self, repo: OrderRepository, notifier: Notifier):
        self.repo = repo        # 依賴「介面」,而不是某個具體實作
        self.notifier = notifier

    def place_order(self, user_email, item, qty):
        order = {"item": item, "qty": qty}
        self.repo.save(order)                                  # 怎麼存?不關我的事
        self.notifier.send(user_email, f"您的 {item} x{qty} 已成立")  # 怎麼通知?不關我的事
        return order

現在,要換資料庫只需換一個 OrderRepository 實作;測試時可以塞一個「假的」 Repository 和 Notifier 進去,完全不碰真資料庫和真郵件。OrderService 對它們的細節一無所知, 只認得介面這份「契約」。這就是低耦合、高內聚在程式碼層面的長相。

💡
一個快速的自我檢查

想知道你的耦合是高是低?問自己:「為了讀懂這個模組,我得同時打開幾個別的檔案?」 數字越大,耦合越高。想知道內聚高不高?試著用一句話說出這個模組在做什麼。 如果你的句子裡出現「而且……」「還有……」,它大概該被拆開了。

兩根樑柱:關注點分離與資訊隱藏

低耦合高內聚是「目標」,而要達成它,有兩個更基本的原則在背後撐著。 它們是整個軟體設計的承重牆,值得單獨拉出來講。

關注點分離(separation of concerns)

一個「關注點(concern)」就是系統要操心的一件事:畫面長怎樣、業務規則是什麼、資料怎麼存、 錯誤怎麼記錄……關注點分離的意思是:把這些不同的事分開處理,別攪在一起。 前面那個拆 OrderService 的例子,本質上就是把「業務流程、資料儲存、使用者通知、發票呈現」 這四個關注點分開。

這個原則無所不在。網頁開發裡經典的「HTML 管結構、CSS 管樣式、JavaScript 管行為」就是關注點分離; 後面要講的 MVC、分層架構,骨子裡也都是同一個原則在不同尺度上的展開。

資訊隱藏(information hiding)

這個概念由 David Parnas 在 1972 年的經典論文裡提出,影響至今。核心想法是: 每個模組應該把「最可能改變」的決策藏在自己內部,只對外露出一個穩定、不易變的介面。 外界只能透過介面使用它,看不到、也不該依賴它內部怎麼實作。

你早就見過它的化身了——第 1 章的抽象、物件導向的封裝(encapsulation), 都是資訊隱藏的具體手段。它和耦合的關係很直接:你把越多細節藏起來,外界能依賴的東西就越少,耦合就越低。 當 OrderRepository 把「我用的是 MySQL」這個決策藏起來,哪天改用 PostgreSQL,外界毫無感覺。

📜
Parnas 的洞見:照「變化」來切模組

Parnas 那篇論文最反直覺、也最深刻的一句話是:模組的邊界,不該照「執行步驟的先後」來切, 而該照「哪些設計決策最可能改變」來切——把每一個可能變動的決策,各自包進一個模組裡藏好。 這樣當變化真的來臨時,衝擊只會落在一個模組內,不會擴散。 五十年過去,這仍是判斷「模組該怎麼切」的最佳指南針。

架構風格巡禮

前人蓋過太多系統,於是某些「拆法」被反覆證明好用,沉澱成了固定的套路—— 我們叫它架構風格(architectural style)。 知道這些風格,你就有了一套現成的詞彙與藍圖:遇到新問題時,不必從零發明, 而是先問「這像哪一種風格?」接下來我們逐一巡禮,每種都用同一個格式: 概念 → 架構圖 → 優缺點 → 適用場景。

1. 分層架構(layered architecture)

最古老、最常見的風格。把系統切成上下堆疊的層(layer), 規則只有一條:每一層只能呼叫它「正下方」的那一層,把結果往上回傳。 上層依賴下層提供的服務,下層完全不知道上層的存在。就像蓋樓,每層樓踩在下一層之上。

優點:好懂、好分工(每層一組人)、好替換(換掉某一層,只要介面不變,其他層無感)。 缺點:請求得「穿過」每一層,可能帶來效能開銷;有時為了某個小功能, 每一層都得跟著加一點程式碼(俗稱「夾心餅乾」式的修改)。 適用:絕大多數企業應用、作業系統、網路協定——當系統的關注點能清楚分成「上下層級」時。

2. 主從式(client-server)與多層(n-tier)

把系統切成兩種角色:客戶端(client)負責提出請求與顯示結果, 伺服器(server)負責集中處理請求與管理資源。你用瀏覽器、手機 App、Email 軟體, 背後全是主從式。當伺服器端再進一步拆成「網頁伺服器 + 應用伺服器 + 資料庫伺服器」分散到不同機器上, 就成了多層架構(n-tier)——注意 tier 指「實體部署的層」,和前面 layer「邏輯的層」是不同概念。

優點:資源集中好管理(資料、權限、更新都在伺服器);客戶端可以很輕薄; 各 tier 可獨立擴充(網站流量大就多開幾台 Web 伺服器)。 缺點:伺服器是單點瓶頸與單點故障(它一掛,全部人沒得用);網路延遲無可避免。 適用:幾乎所有需要「多人共用同一份資料/服務」的系統——網站、線上遊戲、雲端服務。

3. 管線-過濾器(pipe-and-filter)

把資料想像成在水管裡流動。每個過濾器(filter)是一道加工站:從上游收資料、做一件事、 把結果丟給下游。管線(pipe)就是連接它們的水管。每個過濾器只管自己那一步, 對前後是誰一無所知——只要進出的資料格式對得上,就能任意串接、重組。

下面這行你以後會在終端機裡寫無數次,它就是一條活生生的管線:統計存取記錄裡出現最多次的「404 來源 IP」:

cat access.log | grep "404" | cut -d' ' -f1 | sort | uniq -c | sort -rn | head

優點:每個過濾器高內聚、低耦合,可以自由組合與重用(Unix 哲學:「做一件事,把它做好」); 天生適合串流處理與平行化。 缺點:不適合需要「互動」或「共享狀態」的場景;資料得在過濾器間反覆轉換格式,有開銷。 適用:編譯器、資料處理流水線(ETL)、影音串流、Shell 指令、CI/CD 流程。

4. 模型-視圖-控制器(MVC)

專為「有使用者介面」的程式而生的經典分工,把系統切成三個角色:

  • 模型(Model):資料與業務邏輯。它是真相的來源,完全不管畫面長怎樣。
  • 視圖(View):把模型畫成使用者看得到的樣子(畫面)。它只負責「顯示」。
  • 控制器(Controller):接收使用者的操作(點擊、輸入),翻譯成對模型的指令。

MVC 的靈魂是把「畫面」和「資料/邏輯」分開——你可以把同一份資料用網頁、手機、 報表三種視圖呈現,而模型一行都不用改。它正是「關注點分離」在 UI 領域的招牌應用。 優點:UI 與邏輯解耦、易測試(模型可單獨測)、同一模型多種視圖。 缺點:小程式套用反而變囉嗦;三者之間的職責邊界在實務上常有爭議, 於是衍生出 MVP、MVVM 等變體。 適用:桌面 GUI、網頁框架(Django、Rails、Spring MVC)、前端框架的思想源頭。

5. 事件驅動 / 發布-訂閱(event-driven / pub-sub)

前面的風格大多是「A 直接呼叫 B」。事件驅動反其道而行:元件不直接呼叫彼此, 而是發布事件(event),昭告天下「某件事發生了」; 對這件事有興趣的元件,事先訂閱(subscribe)它,事件一來就被通知。 中間常有一個事件匯流排(event bus)或訊息代理(broker)居中傳遞。

關鍵在於:發布者根本不知道誰在聽。它只管喊一聲,誰要回應、有幾個人回應,它一概不管。 這帶來了極致的低耦合——這不就是第 10 章觀察者模式(Observer)的精神嗎?沒錯, 發布-訂閱可以說是觀察者模式放大到架構層級的版本。

用程式碼把這個事件匯流排寫出來,你會發現它和第 10 章的觀察者幾乎是雙胞胎:

class EventBus:
    def __init__(self):
        self._subscribers = {}                       # 事件名稱 -> 處理函式清單

    def subscribe(self, event, handler):             # 訂閱
        self._subscribers.setdefault(event, []).append(handler)

    def publish(self, event, data):                  # 發布:通知所有訂閱者
        for handler in self._subscribers.get(event, []):
            handler(data)

bus = EventBus()
bus.subscribe("order_placed", lambda d: print(f"寄信:您的 {d['item']} 已成立"))
bus.subscribe("order_placed", lambda d: print(f"庫存:{d['item']} 扣 {d['qty']} 件"))

# 訂單服務只要喊一聲,完全不知道有誰在聽
bus.publish("order_placed", {"item": "鍵盤", "qty": 1})
# 寄信:您的 鍵盤 已成立
# 庫存:鍵盤 扣 1 件

優點:極致的低耦合與彈性;天生適合非同步、可平行、可水平擴展;新增功能不必改舊元件。 缺點:整體流程變得「隱晦」——很難一眼看出「按下這顆按鈕之後到底發生了什麼」, 除錯與追蹤特別困難;事件的順序與「最終一致性」要小心處理。 適用:GUI 事件、即時通知、IoT、金融交易、微服務之間的溝通。

6. 微服務 vs 單體(microservices vs monolith)

這是當代最熱門、也最容易被誤用的架構辯論。單體(monolith)是把整個應用 打包成一個可部署的單位——所有功能住在同一個程式、同一個程序裡(它內部還是可以好好分層)。 微服務(microservices)則把應用拆成許多小而獨立的服務, 每個服務管自己的一塊業務、有自己的資料庫、能獨立部署,彼此透過網路(API 或訊息)溝通。

面向單體 Monolith微服務 Microservices
開發起步簡單、快前期投入大
部署一次部署全部各服務獨立部署
擴充整包一起放大哪個熱就單獨放大哪個
技術選擇通常一套技術棧各服務可用不同語言/資料庫
故障影響容易一處掛全部掛可隔離,但網路本身會失敗
除錯/維運單純,單機可追分散式追蹤,複雜度高
團隊適合小團隊適合多個自主團隊
⚠️
別一開始就上微服務

微服務不是「進步」,它是一種權衡:用「開發單純」換「擴展彈性與團隊自主」。 對一個還在找產品定位的小團隊,微服務帶來的分散式複雜度(網路失敗、資料一致性、部署維運) 往往會把你壓垮。業界的成熟建議是「Monolith First」:先用乾淨分層的單體把產品做出來, 等真正遇到擴展或團隊瓶頸,再把該拆的部分拆成服務。記住:你拆的是問題,不是流行。

7. 儲存庫 / 黑板(repository / blackboard)

最後簡述一個常被忽略、但其實無所不在的風格:儲存庫(repository)。 它的中心是一份共享的中央資料,周圍的元件都圍著它工作——讀它、改它, 但元件之間不直接對話,只透過這份共享資料間接協作。你熟悉的資料庫就是最典型的例子: 前端、後端、報表系統、備份程式,全都圍著同一個資料庫轉,彼此卻互不認識。

當這份共享資料還會「主動觸發」元件去處理(誰看到自己能處理的資料就跳出來做), 就成了它的變體黑板(blackboard)——名字來自「一群專家圍著一塊黑板, 各自把會的部分寫上去,直到問題被解開」。黑板常見於沒有明確固定解法的領域, 如語音辨識、影像理解、早期的 AI 系統。 優點:元件徹底解耦、易於增減。 缺點:那份共享資料容易成為瓶頸與耦合點(大家都依賴它的結構)。

架構是「品質屬性」的權衡藝術

看完七種風格,你大概會冒出一個很自然的問題:「那到底哪一種最好?」 答案會讓你有點失望,卻是這一章最重要的領悟:沒有最好的架構,只有最適合的架構。

為什麼?因為架構真正在做的事,不是「追求完美」,而是在一堆互相衝突的品質屬性 (quality attributes)之間做取捨。這些屬性的英文常以 -ility 結尾, 所以工程師暱稱它們為「那些 -ilities」。它們不是程式「功能上」做不做得到, 而是程式「做得多好」:

品質屬性英文它在問什麼
效能performance反應夠快嗎?延遲多低、吞吐多高?
可擴展性scalability使用者/資料量暴增 10 倍,撐得住嗎?
可用性availability它有多常是「活著、能用」的?(那串 99.9% 的「幾個 9」)
可維護性maintainability改一個功能、修一個 bug,容不容易、會不會弄壞別的?
安全性security能不能擋住惡意攻擊、保護資料?(第七部主題)
可測試性testability容不容易為它寫出可靠的測試?

關鍵在於:這些屬性常常彼此打架。你很難同時把每一個都拉到最高:

  • 為了安全,你加了層層加密與驗證——效能就下降了。
  • 為了可用性,你把資料複製到多台機器——一致性就變難維護了。
  • 為了可擴展性與可維護性,你拆成微服務——維運的複雜度就上去了。
🔑
架構決策的本質是「取捨」

下次有人問你「哪種架構好」,正確的反問是:「對什麼而言好?」 銀行系統最在乎安全與一致性;直播平台最在乎效能與可擴展;新創 MVP 最在乎開發速度與可維護性。 同一個架構,在一個情境是神來之筆,在另一個情境是災難。先確定你要優化什麼,架構才有對錯可言。

架構模式 vs 設計模式:同一座山的不同高度

第 10 章你學了設計模式(工廠、觀察者、策略……),這一章談的是架構風格與模式。 它們聽起來很像,差別到底在哪?答案還是那個關鍵字:高度。

  • 架構模式 / 風格:影響整個系統的結構決策。例:分層、MVC、微服務、事件驅動。 一個系統通常只有少數幾個。
  • 設計模式:影響一小群類別/物件如何協作。例:觀察者、工廠、策略。 一個系統裡可以同時用上幾十個。

兩者的精神完全一致——都是把「前人驗證過的好解法」打包成可重用的範本——只是作用的尺度不同。 有趣的是它們常常彼此呼應:你用「事件驅動」這個架構模式搭起骨架, 而骨架裡每一個元件監聽事件的方式,可能正是用「觀察者」這個設計模式實作的。

用 UML 把架構畫給別人看

架構是團隊溝通的語言,而溝通需要共同的符號。UML(Unified Modeling Language,統一塑模語言) 就是這套業界通用的「畫圖文法」。你不需要精通它全部幾十種圖——對架構而言, 最常用、最該認得的就三種。

元件圖(component diagram):系統由哪些塊組成

用來表達「有哪些元件、它們提供與需要什麼介面」。UML 用一個小圖示: 提供的介面畫成「棒棒糖(○)」,需要的介面畫成「插座(⊂)」, 兩者一接,就表示「這個元件用了那個元件提供的服務」。

套件圖(package diagram):程式碼怎麼分組

當元件多到爆,就用套件(package)把相關的元件框在一起分組, 再用箭頭表示套件之間的依賴方向。它像系統的「資料夾結構俯瞰圖」, 最適合用來檢查:依賴的箭頭是不是都朝著該有的方向?有沒有出現不該有的循環依賴?

循序圖(sequence diagram):一件事按時間怎麼發生

前兩種圖是「靜態」的(系統長怎樣);循序圖則是「動態」的——它沿著時間軸由上往下, 畫出「一個情境裡,各元件之間依序送出哪些訊息」。每個元件有一條垂直的生命線, 橫向箭頭代表一次呼叫。它最適合說清楚「使用者按下結帳之後,到底發生了哪些步驟」。

💡
圖是用來溝通的,不是用來考試的

別被 UML 嚇到。實務上沒人會把所有圖都畫得一絲不苟——多數團隊只在白板上隨手畫個「方塊加箭頭」, 能讓大家對齊理解就夠了。重點永遠是把架構想法傳達清楚,而不是符號畫得標不標準。 學 UML 的目的,是讓你看得懂別人的圖、也能快速畫出讓別人看懂的圖。

架構決策、技術債與康威定律

架構不是畫一次圖就結束的,它活在持續的決策與妥協裡。這一節談三個你遲早會親身碰到的現實。

架構決策:把「為什麼」記下來

團隊裡最常見的悲劇是:三年後沒人記得「我們當初為什麼選 PostgreSQL 而不是 MongoDB」, 於是新人想改,改到一半才踩到當年那個坑。解法很簡單卻常被忽略——把重要的架構決策寫成 架構決策紀錄(Architecture Decision Record, ADR):一份短文件,記下 「我們面臨什麼問題、考慮過哪些選項、最後選了什麼、為什麼、放棄了什麼」。 架構的價值,一半在決策本身,一半在記得當初為什麼這樣決策。

技術債(technical debt):先借後還,但會生利息

這是 Ward Cunningham 提出的絕妙比喻。為了趕上線,你選了一個「快但不夠好」的做法 (例如先硬寫死、先不分層、先複製貼上)——這就像借了一筆錢, 讓你現在跑得快。但這筆債會生利息:之後每次要在這段爛程式上加功能, 都得多花時間繞過它、小心別弄壞它。利息會越滾越大,直到有一天你「只還得起利息」—— 團隊所有力氣都花在跟爛架構搏鬥,再也擠不出時間做新東西。

ℹ️
技術債不全是壞事

就像真實世界的貸款,刻意地、有意識地欠技術債有時是聰明的商業決定—— 為了搶市場先機,先借一筆「以後再重構」的債,讓產品快點上線。 危險的是無意識地欠債(根本不知道自己在寫爛架構),以及永遠不還 (從不撥時間重構)。健康的團隊會定期「還債」:安排重構,把高利息的爛區塊清乾淨。

📜
康威定律(Conway's Law):你的系統長得像你的組織

1967 年,程式設計師 Melvin Conway 提出一個半開玩笑卻一語中的的觀察: 「設計系統的組織,所產出的系統結構,會複製該組織的溝通結構。」 白話說:四個團隊去做一個編譯器,你就會得到一個四階段的編譯器。 因為團隊之間怎麼溝通,模組之間就會怎麼切。這個定律影響深遠—— 現代「微服務 + 小型自主團隊」的搭配,某種程度就是反向利用康威定律: 先把團隊照「你希望的架構」切好,架構自然會長成那個樣子。 想改架構?有時得先改組織。

貫穿案例:一個電商系統的架構演化史

理論講完了,讓我們把所有概念縫進一個故事。假設你開了一家網路商店「小書坊」, 從一個人的副業,長成一家公司。看著它的架構如何隨需求一步步演化—— 你會發現,每一次演化都是在回應新的品質屬性壓力,而每一步都是一場權衡。

第一階段:單體,而且是一團亂的單體

一開始只有你一個人,目標是「盡快上線,驗證有沒有人要買」。 你寫了一個程式,把所有東西塞在一起:處理網頁、商品邏輯、訂單、寄信、直接操作資料庫—— 全攪成一鍋。這其實就是本章開頭那個反面教材的放大版:一個大泥球。

權衡:此刻你優先的品質屬性是「開發速度」,代價是「可維護性」幾乎是零。 在還沒有用戶的階段,這是對的選擇——別急著為不存在的規模做設計。

第二階段:重構成乾淨的分層單體

生意起來了,你開始頻繁改 bug、加功能,卻發現每次都得在那團義大利麵裡提心吊膽。 是時候還技術債了。你不拆成微服務(太早),而是套用本章的核心原則, 把這個單體重構成乾淨的分層架構:展示層、業務邏輯層、資料存取層, 並把「寄信」「金流」抽成各自高內聚的模組,彼此透過介面溝通。

權衡:你用一次重構的成本,換來了長期的「可維護性」與「可測試性」, 同時完整保留了單體「部署簡單、開發單純、效能好」的優點。這往往是整個演化過程中 CP 值最高的一步。

第三階段:當壓力真的來了,才拆成微服務

又過了兩年,「小書坊」爆紅。現在你面臨真正的規模壓力: 每逢促銷,「訂單」與「金流」模組被打爆,但「商品瀏覽」明明很閒—— 然而因為是單體,你只能把整包一起加機器,浪費資源。同時你的工程師多到 二十人,大家擠在同一份程式碼裡互相卡住部署。這兩個訊號(獨立擴展需求 + 團隊自主需求) 才是該上微服務的真正理由。

權衡:你換來了夢寐以求的「可擴展性」與「團隊自主」, 但代價毫不含糊——你現在得面對分散式系統的全部難題:網路會逾時、 資料分散在多個資料庫難以保持一致、一個請求跨好幾個服務難以追蹤、維運複雜度暴增。 你需要 API 閘道、服務發現、分散式追蹤、監控告警……一整套新基礎設施。 這就是為什麼「微服務」絕不是免費的升級,而是一筆用複雜度換彈性的昂貴交易。

🔑
架構是「演化」出來的,不是一次「設計」死的

回顧「小書坊」的三個階段:每一步都不是因為某個架構「比較潮」,而是因為需求變了、 壓力變了、最該優化的品質屬性變了。好的架構師不追求一步到位的完美藍圖, 而是讓架構保持「剛好夠用、且能演化」的彈性,在對的時機做對的取捨。 這也是為什麼「低耦合、高內聚」如此重要——它讓你的系統有能力在未來被重新切分。

練習

✏️ 練習 11.1 思考

下面四個情境,各自最適合哪一種架構風格?請選出風格並說明理由(沒有唯一標準答案,重點在推理):

  1. 一個把使用者上傳的影片,依序做「轉檔 → 壓縮 → 加浮水印 → 產生縮圖」的處理服務。
  2. 一個三人小團隊要在兩個月內做出來的記帳 App MVP。
  3. 一個股票交易平台,當「成交」發生時,要同時更新報價看板、推播通知、寫入帳本、觸發風控檢查。
  4. 一個有桌面版、網頁版、手機版三種介面,但背後是同一套資料與規則的待辦事項軟體。
看解答

1. 管線-過濾器。它是典型的「資料依序流過好幾道獨立加工站」,每一步高內聚、彼此低耦合, 還能各自平行擴充。把每個步驟做成一個過濾器,日後要加一步(例如「AI 內容審查」)只要插進管線即可。

2. 單體(乾淨分層)。三人、兩個月、要的是「快點做出來驗證」。微服務的分散式複雜度會把小團隊拖垮。 先做一個分層乾淨的單體,既快又好維護;真的紅了再說。這正是「Monolith First」。

3. 事件驅動 / 發布-訂閱。「一件事發生 → 多個互不相干的後續動作」是事件驅動的招牌場景。 「成交」發布一個事件,看板、推播、帳本、風控各自訂閱;日後要新增「給推薦系統一個訊號」只要多加一個訂閱者, 成交邏輯一行都不用改。(金融場景對可靠性要求高,實務上會搭配可靠的訊息佇列。)

4. MVC(或其變體)。「同一份資料/邏輯、多種介面呈現」正是 MVC 的核心價值—— 把 Model 抽出來共用,三種介面各自是一個 View,模型完全不必為了多一種介面而改動。

✏️ 練習 11.2 實作

下面這個 ReportGenerator 高耦合又低內聚。請指出它違反了哪些原則,並重構它, 讓「資料來源」「報表格式」「輸出方式」三者可以各自獨立替換(例如改成從 API 拿資料、輸出成 PDF、寄到 email)。

class ReportGenerator:
    def run(self):
        # 從 MySQL 撈資料(綁死資料來源)
        import mysql.connector
        conn = mysql.connector.connect(host="localhost", database="shop")
        rows = conn.cursor().execute("SELECT * FROM sales").fetchall()

        # 組成 CSV 字串(綁死輸出格式)
        text = "id,amount\n"
        for r in rows:
            text += f"{r[0]},{r[1]}\n"

        # 直接寫到本機檔案(綁死輸出方式)
        with open("report.csv", "w") as f:
            f.write(text)
看解答

問題診斷:

  • 低內聚:一個類別同時做了「取資料、格式化、輸出」三件不相干的事。
  • 高耦合:直接依賴 MySQL、CSV 格式、本機檔案這三個具體實作,任何一個要換都得動到它。
  • 違反關注點分離與資訊隱藏:該藏起來的「怎麼取/怎麼格式化/怎麼輸出」全攤在同一個方法裡。

重構:把三個關注點各自抽成介面,用依賴注入組裝起來。

from abc import ABC, abstractmethod

class DataSource(ABC):              # 關注點一:從哪取資料
    @abstractmethod
    def fetch(self): ...

class Formatter(ABC):              # 關注點二:格式化成什麼
    @abstractmethod
    def format(self, rows) -> str: ...

class Output(ABC):                 # 關注點三:輸出到哪
    @abstractmethod
    def write(self, text): ...

class ReportGenerator:            # 只負責「協調流程」,高內聚
    def __init__(self, source: DataSource, fmt: Formatter, out: Output):
        self.source, self.fmt, self.out = source, fmt, out

    def run(self):
        rows = self.source.fetch()
        text = self.fmt.format(rows)
        self.out.write(text)

# 具體實作(可自由替換、可塞假物件來測試)
class MySQLSource(DataSource):
    def fetch(self): ...           # 連 MySQL

class CSVFormatter(Formatter):
    def format(self, rows):
        return "id,amount\n" + "\n".join(f"{r[0]},{r[1]}" for r in rows)

class FileOutput(Output):
    def __init__(self, path): self.path = path
    def write(self, text):
        with open(self.path, "w") as f:
            f.write(text)

# 組裝:想換成 API 來源、PDF 格式、Email 輸出,只要換掉對應實作
report = ReportGenerator(MySQLSource(), CSVFormatter(), FileOutput("report.csv"))
report.run()

現在三個關注點各自獨立、可替換、可單獨測試。這正是低耦合、高內聚、關注點分離、資訊隱藏四個原則的合體—— 也是策略模式(第 10 章)在架構層級的應用。

✏️ 練習 11.3 設計

為一個多人即時聊天室(像 Discord 的簡化版)畫出「元件與連接器」圖。 至少要包含:多個聊天客戶端、一個訊息伺服器、訊息的儲存、以及「某人發的訊息要即時送到所有在線的人」這個機制。 標出每條連接器用的是什麼方式溝通,並說一句:你會用哪種架構風格?為什麼?

看解答

一種合理的元件與連接器設計:

  • 聊天客戶端 ×N(元件):透過 WebSocket(一條雙向、持久的連接器)連到訊息伺服器—— 這比一般 HTTP 請求/回應更適合「伺服器主動推訊息給你」。
  • 訊息伺服器 / 訊息匯流排(元件):收到某人的訊息後,①寫進資料庫存檔,②把訊息發布給所有訂閱了該頻道的在線客戶端。
  • 訊息資料庫(元件):透過資料庫連線存歷史訊息,供「往上滑看舊訊息」用。

架構風格:事件驅動 / 發布-訂閱。因為「一個人發訊息 → 所有訂閱該頻道的人即時收到」 正是 pub-sub 的天職;訊息伺服器扮演匯流排,客戶端訂閱頻道。 這讓「有多少人在線、是誰」與「發訊息」這兩件事解耦——發訊息的人不需要知道有誰在聽。

✏️ 練習 11.4 思考

有一家成立兩年、十人團隊的 SaaS 公司,系統是一個分層乾淨的單體。 一位新來的工程師主張:「大公司都用微服務,我們也該全面改成微服務,這樣比較先進。」 請站在架構師的角度,寫出你會問他的三個問題,並說明在什麼情況下「不該改」、什麼情況下「該改」。

看解答

三個該問的問題:

  1. 「我們現在遇到的具體痛點是什麼?是某個模組獨立擴展不了?還是團隊互相卡部署?」—— 如果答不出具體痛點,那就是為了「先進」而改,是典型的過度工程。
  2. 「我們有沒有能力承擔微服務的維運成本?分散式追蹤、服務發現、資料一致性、監控告警, 這些基礎設施我們建得起來、養得起嗎?」——十人團隊往往沒有這個餘裕。
  3. 「如果要拆,該照什麼邊界拆?哪些業務真的彼此獨立?」—— 拆錯邊界的微服務,比單體更糟(變成「分散式大泥球」,每改一個功能要同時改好幾個服務)。

不該改的情況:單體目前運作良好、沒有明確的擴展或團隊瓶頸、改的理由只是「比較潮」。 此時微服務帶來的分散式複雜度是純粹的成本,沒有對應收益。記住:架構是權衡,不是「進步階梯」。

該改的情況(通常要同時滿足好幾個):某些模組有截然不同的擴展需求(訂單被打爆但商品很閒); 團隊大到大家擠在同一份程式碼裡互相卡部署;不同部分想用不同技術棧;單體的建置/部署慢到拖累迭代。 而且最好是「逐步把痛點模組拆出去」,而不是一次全部重寫。

✏️ 練習 11.5 挑戰

挑一個你每天在用的 App(YouTube、IG、Uber、Google Maps、線上銀行……), 憑你的理解猜測並畫出它的高層架構(元件與連接器),然後標出: 這個系統最在乎哪兩個品質屬性?為了它們,你猜它犧牲了什麼?

看參考方向(以 YouTube 為例)

這題沒有標準答案——重點是練習「用架構的眼光拆解真實系統」。以 YouTube 為例的猜測:

  • 元件:客戶端(App/網頁)、API 閘道、影片上傳服務、轉檔流水線(管線-過濾器!)、 內容傳遞網路 CDN(把影片快取到離你最近的伺服器)、推薦服務、搜尋服務、留言服務、各自的資料庫。
  • 連接器:客戶端用 HTTP/串流協定看影片;服務之間用 API 與事件匯流排;上傳後用管線做轉檔。

最在乎的品質屬性(猜測):

  1. 可擴展性 + 可用性:全球幾十億用戶、海量影片,必須能水平擴展、極少當機。 這就是為什麼它幾乎一定是微服務 + CDN 的組合。
  2. 效能(低延遲串流):影片不能轉個半天。CDN 把內容推到離你最近的節點,就是為了這個。

為此犧牲了什麼:

  • 強一致性:你按了讚,觀看數可能過幾秒才更新;新留言別人不一定立刻看到—— 這是用「最終一致性」換取規模與可用性的典型取捨。
  • 簡單性 / 維運成本:這樣的系統極度複雜,需要龐大的團隊與基礎設施來維運。

只要你能說出「它優先什麼、為此犧牲了什麼」,你就已經在用架構師的方式思考了。

把這一章串起來

我們從一座城市的規劃,走到了一家電商的演化史。把這趟旅程的路標收進口袋:

  • 軟體架構是系統的高層結構——有哪些元件、各自的職責、以什麼連接器溝通。它和細部設計的差別是「高度」,而架構決策做得早、影響廣、改起來最貴。
  • 衡量「切得好不好」的兩把尺:低耦合、高內聚。撐起它們的兩根樑柱:關注點分離與資訊隱藏。
  • 七種架構風格各有擅場:分層(好懂好分工)、主從/多層(資源集中)、管線-過濾器(可組合的流水線)、MVC(UI 與邏輯分離)、事件驅動/發布-訂閱(極致解耦)、微服務 vs 單體(彈性換複雜度)、儲存庫/黑板(共享資料協作)。
  • 沒有最好的架構,只有最適合的。架構的本質是品質屬性(那些 -ilities)的權衡——先確定要優化什麼,對錯才有意義。
  • 架構模式 vs 設計模式是同一種智慧的不同尺度;用 UML(元件圖、套件圖、循序圖)把架構畫給別人看;留意技術債與康威定律。
  • 最重要的一句:架構是演化出來的,不是一次設計死的。讓它「剛好夠用、且能演化」,在對的時機做對的取捨。

恭喜——你已經把「程式設計核心」這一部走完了。從函式、遞迴、資料結構、物件導向、設計模式, 到這一章的軟體架構,你看待程式的眼光已經從「一行一行的指令」升級到「一座可以規劃、可以演化的城市」。 接下來,我們將潛入這座城市底下的地基:電腦系統本身——硬體、作業系統與網路。
