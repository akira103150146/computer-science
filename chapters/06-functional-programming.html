函數式程式設計

如果有一種寫程式的方式,讓你的函式像數學裡的函數一樣可靠——相同輸入永遠得到相同輸出、從不偷偷改動任何東西——你會不會想學?這就是函數式程式設計的世界。

  • lambda 匿名函式、函式組合(composition)、currying 與 partial application
  • 閉包(closure):函式如何記住它「出生時」的環境

兩種世界觀:食譜步驟 vs 數學函數

還記得第 1 章那碗滷肉飯嗎?食譜是一連串「先做這、再做那」的步驟:熱油、爆香、下肉、加醬油、燉一小時。 每一步都在改變鍋子裡的狀態——生肉變熟肉,清湯變濃汁。你照著步驟,一步步把世界從一個狀態推向下一個狀態。 這,正是你到目前為止寫程式的方式,我們稱之為命令式程式設計(imperative programming)。

但數學家看世界的方式不太一樣。當數學家寫下 $f(x) = x^2$,他沒有在「改變」任何東西。 他只是描述一種對應關係:給我一個 $x$,我還你一個 $x^2$。你把 $3$ 丟進去,永遠得到 $9$—— 不管現在幾點、這個式子被算過幾次、宇宙發生了什麼事。沒有鍋子被攪動,沒有狀態被改寫, 只有一個輸入對應到一個輸出。這種「用函數的對應關係來描述計算」的風格,就是函數式程式設計(functional programming,常簡稱 FP)。

我們把同一件事——「把一串數字各自平方,再加總」——用兩種風格各寫一遍,你立刻感受得到那股不同的氣味: 

# 命令式:一步步改變狀態
nums = [1, 2, 3]
total = 0                     # 準備一個會被反覆改寫的「累加器」
for n in nums:
    total = total + n * n     # 每跑一圈,total 就被覆寫一次
print(total)                  # 14

# 函數式:描述「值如何被轉換」,不改變任何既有變數
from functools import reduce
nums = [1, 2, 3]
total = reduce(lambda acc, x: acc + x, map(lambda x: x * x, nums), 0)
print(total)                  # 14

命令式版本的核心是那個 total,它像一塊白板,被你擦了又寫、寫了又擦。 函數式版本則完全沒有「會變的變數」:map 把每個數平方產生一個新序列,reduce 把它折疊成一個總和。 資料只是「流過」一連串轉換,從來沒有誰被原地改寫。

🔑
兩種世界觀,一句話分辨

命令式問的是:「電腦,接下來該做什麼?」(一連串改變狀態的指令)
函數式問的是:「這個答案是什麼?」(用函數把輸入對應到輸出的算式)
前者像食譜步驟,後者像數學定義。FP 的核心信念是:盡量用「計算值」取代「改變狀態」。

先講清楚一件事,免得你誤會:函數式不是要取代你學過的一切,更不是說 for 迴圈有罪。 它是一套看待程式的角度與一組讓程式更好推理的紀律。學會它,你不只多會一種語法, 而是多了一雙眼睛——你會開始看見哪些程式碼之所以難懂、難測、難改,是因為「狀態」在到處亂竄。接下來,我們從最核心的那塊基石開始:純函式。

純函式與參照透明性

第 3 章我們埋過一個伏筆——純函式(pure function)。現在它要正式登場,而且是這一整章的地基。 一個函式要稱得上「純」,必須同時滿足兩個條件:

  • 相同輸入,永遠相同輸出:呼叫 f(3) 這輩子都回傳同一個值,跟現在幾點、跑過幾次、別處發生什麼事都無關。
  • 沒有副作用(side effect):它不改動函式以外的任何東西——不碰全域變數、不改傳進來的參數、不寫檔案、不印螢幕、不發網路請求。它只是「輸入 → 計算 → 回傳」。
# ✅ 純函式:答案只取決於參數,且不碰外界
def add(a, b):
    return a + b

# ❌ 不純:答案取決於外界狀態(現在時間),且每次都不同
import time
def stamp(msg):
    return f"[{time.time()}] {msg}"   # 同樣的 msg,每次輸出都不一樣

# ❌ 不純:偷偷改動了傳進來的 list(副作用)
def add_item(cart, item):
    cart.append(item)                 # 把呼叫者的 cart 改掉了!
    return cart

跟純函式緊緊相連的,是一個聽起來很學術、其實超實用的概念:參照透明性(referential transparency)。 它的意思是:你可以把一個運算式,直接換成它的計算結果,而程式的行為完全不變。

舉例。既然 add(2, 3) 永遠是 5,那麼在程式任何地方,add(2, 3) 和 5 就是可以互換的—— 這就叫參照透明。但 time.time() 不行:你不能把它換成「某個固定的數字」,因為它每次都不一樣。 參照透明性是數學算式的基本性質(你當然可以把 $2+3$ 換成 $5$),而純函式讓你的程式碼也擁有這種性質。

🔑
參照透明 = 「能用結果替換呼叫」

如果 f(x) 是純的,那麼 f(x) + f(x) 就保證等於 2 * f(x),你甚至可以「只算一次、存起來重複用」而不改變語意。 這種「可以放心代換」的特性,是編譯器最佳化、快取(memoization)、平行化能成立的根本前提。

純函式為什麼這麼香?

純度不是潔癖,它換來三個非常具體的好處:

  • 好測試:測一個純函式,只要「給輸入、比對輸出」即可,不必先佈置一堆外部狀態(假資料庫、假時間、假檔案)。 assert add(2, 3) == 5 就結束了。不純的函式則往往要大費周章地「模擬」整個世界。
  • 好推理:你光看函式的簽名(輸入、輸出)就能完全理解它的行為,不必擔心它「偷偷做了別的事」。 這正是第 3 章說的「靠參數進、靠 return 出」原則的極致。
  • 好平行化:這是最關鍵、也最有未來性的一點。

想像你要把一百萬個數字各自平方。如果「平方」是純函式,那麼算第 5 個和算第 50 萬個之間毫無瓜葛—— 沒有共享的狀態、沒有先後順序的依賴。於是你可以放心地把工作切成八份,丟給八顆 CPU 核心同時算,結果保證正確。 但如果每次計算都會偷偷去改同一個全域變數,那多執行緒同時讀寫它,就會撞成一團亂(這叫競爭條件 race condition)。

🧭
伏筆:這就是平行運算愛 FP 的原因

並行(concurrency)與平行(parallelism)程式設計裡最惡名昭彰的 bug,幾乎都源自「多個執行緒同時改動共享的可變狀態」。 純函式因為不改動、不共享狀態,天生就避開了這整類災難。這也是為什麼當摩爾定律走向「多核心」時代,函數式思維重新大紅。 我們會在第十部「平行程式設計」把這條線好好接起來——到時你會慶幸自己在這裡先打好了底。

不可變性:別動我的資料

純函式有個天然的盟友,叫不可變性(immutability):資料一旦被建立,就永遠不再被修改。 需要「不一樣的版本」?那就產生一個新的,而不是把舊的就地改掉。

為什麼要這麼「龜毛」?因為「會變的共享資料」是 bug 的超級溫床。看這個經典陷阱: 

def add_tax(prices):
    for i in range(len(prices)):
        prices[i] = prices[i] * 1.05   # 就地修改傳進來的 list!
    return prices

original = [100, 200, 300]
with_tax = add_tax(original)
print(with_tax)    # [105.0, 210.0, 315.0]
print(original)    # [105.0, 210.0, 315.0]  ← 慘了!原始資料也被改掉了

呼叫者只想「拿到一份加稅後的價格」,卻發現手上原本那份乾淨的 original 也被偷偷竄改了。 在大型程式裡,這種「某個函式偷偷改了我以為不會變的東西」是最難抓的 bug 之一。函數式的解法是:不要改,回傳新的。 

def add_tax(prices):
    return [p * 1.05 for p in prices]   # 產生一個全新的 list,原本的完全沒動

original = [100, 200, 300]
with_tax = add_tax(original)
print(with_tax)    # [105.0, 210.0, 315.0]
print(original)    # [100, 200, 300]  ← 原始資料毫髮無傷
🔑
習慣:回傳新值,而不是修改舊值

把「修改(mutate)」換成「轉換(transform)」。每個函式都像一台機器:吃進資料、吐出新的資料, 從不去動桌上別人的東西。當資料不會被偷改,你對程式的每一個「現在的值是什麼」就有了確定的答案——這就是可預測性的來源。

Python 裡的不可變工具:tuple 與 frozenset

Python 的 list、dict、set 都是可變(mutable)的;但它也內建了幾種不可變的對應物。 最常用的是 tuple(元組)——它就像一個「凍結的 list」,建立之後就不能再增刪改: 

point = (3, 4)          # tuple,用小括號
print(point[0])         # 3,讀取沒問題
point[0] = 99           # ❌ TypeError:tuple 不支援就地修改

# 需要「改一個座標」?產生一個新的 tuple
moved = (point[0] + 1, point[1])
print(moved)            # (4, 4),而 point 仍是 (3, 4)

同樣地,frozenset(凍結集合)是不可變版的 set: 

tags = frozenset({"python", "fp"})
print("fp" in tags)     # True,查詢照常
tags.add("ml")          # ❌ AttributeError:frozenset 不能新增元素

不可變物件還有一個附帶好處:因為值不會變,它可以被雜湊(hashable),於是能當 dict 的鍵、放進 set 裡。 這也是為什麼 (1, 2) 可以當字典的鍵,而 [1, 2] 不行。

⚠️
不可變 ≠ 一定省記憶體

「每次都回傳新值」聽起來很浪費——難道每改一個元素就複製整份資料?在樸素實作裡確實會。 但真正的函數式語言(如 Clojure、Haskell)用持久性資料結構(persistent data structure)解決它: 新舊版本「共享」絕大部分沒變的內部結構,只為改動的部分付出代價。所以不可變性在實務上沒你想的那麼貴。 在 Python 裡,小資料用 tuple/重建即可;大資料若真有效能需求,再考慮專門的函式庫。

一級函式:函式也是值

現在進入讓 FP 變得強大的關鍵機制。在許多語言裡,函式是一種「特別的東西」——你只能定義它、呼叫它,僅此而已。 但在 Python(以及所有函數式語言)裡,函式是「一級公民(first-class citizen)」: 它跟整數、字串、list 一樣,是一種可以被當成「值」來操作的東西。

這句話拆開來,具體是說,函式可以:

  • 被指派給變數
  • 被當成引數傳給另一個函式
  • 被當成另一個函式的回傳值
  • 被存進 list、dict 等資料結構
def shout(text):
    return text.upper() + "!"

# 1. 函式可以指派給變數(注意:不加括號,不是呼叫,是「拿函式本身」)
yell = shout
print(yell("hello"))          # HELLO!  ← yell 現在就是 shout

# 2. 函式可以存進資料結構
def whisper(text):
    return text.lower() + "..."

styles = {"big": shout, "small": whisper}   # 字典的「值」是函式!
print(styles["big"]("hi"))    # HI!
print(styles["small"]("HI"))  # hi...

# 3. 函式可以當引數傳進去(下一節的主角)
def apply_twice(func, x):
    return func(func(x))      # 把傳進來的 func 連用兩次
print(apply_twice(shout, "wow"))   # WOW!!

最容易絆倒新手的是 yell = shout 這行:注意它後面沒有括號。 shout 是「函式這個物件本身」,shout("hi") 才是「呼叫它、拿到結果」。 把這個區別記牢——「函式名」和「呼叫函式」是兩回事——後面整章都靠它。

🔍
函式只是一個「剛好能被呼叫」的物件

在 Python 裡,函式跟數字 5、字串 "hi" 一樣,都是記憶體裡的一個物件。 差別只在於它身上多了一個能力:你可以在它後面加 () 來「執行」它。 一旦你接受「函式也是值」這個觀念,接下來的高階函式、閉包、組合,全都會變得理所當然。

高階函式:map、filter、reduce

既然函式可以當引數傳遞,那就誕生了一種特別的函式:高階函式(higher-order function)—— 它接收函式當參數,或回傳一個函式(或兩者皆是)。

FP 世界有三位最出名的高階函式,合稱「三劍客」:map、filter、reduce。 它們分別對應三種最常見的「對一整個序列做事」的模式。我們的策略是:每一個,都先看你熟悉的手寫迴圈版,再重構成高階函式版,讓你看清它們其實是同一件事的兩種說法。

map:把每個元素都變個樣

map 的工作是:對序列裡的每個元素,套用同一個函式,產生一個等長的新序列。 手寫迴圈時,你會這樣做: 

# 手寫迴圈版:把每個數字平方
nums = [1, 2, 3, 4]
squares = []
for n in nums:
    squares.append(n * n)
print(squares)            # [1, 4, 9, 16]

注意這個迴圈的「骨架」:建一個空 list、跑一圈、把「轉換後的元素」塞進去。這個骨架太常見了,於是被抽象成 map:

# 高階函式版:map(轉換函式, 序列)
nums = [1, 2, 3, 4]
squares = list(map(lambda n: n * n, nums))   # map 回傳的是迭代器,用 list() 收集
print(squares)            # [1, 4, 9, 16]

你把「要對每個元素做什麼」(lambda n: n * n,下一節細講)當成函式傳給 map, map 負責「跑一圈、逐一套用」這個重複的骨架。重點轉移了:你不再描述「怎麼跑迴圈」,只描述「每個元素該怎麼變」。

ℹ️
其實,串列推導式也是函數式的好朋友

Python 老手常說:很多 map/filter 用串列推導式(list comprehension)更 Pythonic 也更好讀: [n * n for n in nums] 等價於上面的 map;[n for n in nums if n % 2 == 0] 等價於 filter。 推導式骨子裡也是「描述結果長什麼樣」的宣告式風格。本章用 map/filter/reduce 是為了讓你看清「高階函式」這個機制, 但實務上兩種寫法你都該會、都該讀得懂。

filter:篩掉不要的

filter 的工作是:保留「讓判斷函式回傳 True」的元素,丟掉其餘的。那個判斷函式回傳 True/False,稱為謂詞(predicate)。 

# 手寫迴圈版:留下偶數
nums = [1, 2, 3, 4, 5, 6]
evens = []
for n in nums:
    if n % 2 == 0:
        evens.append(n)
print(evens)              # [2, 4, 6]

# 高階函式版:filter(謂詞, 序列)
evens = list(filter(lambda n: n % 2 == 0, nums))
print(evens)              # [2, 4, 6]

一樣的故事:迴圈的骨架(跑一圈、用 if 挑、塞進新 list)被 filter 吃掉了,你只需提供「什麼樣的元素值得留下」這條規則。

reduce:把一整串折疊成一個值

前兩位都「序列進、序列出」。reduce 不一樣:它「序列進、單一值出」。 它的工作是:拿一個「累加器」,從頭到尾把每個元素「併」進去,最後得到一個總結果。 加總、連乘、找最大值,本質上全都是 reduce。它住在標準函式庫的 functools 裡: 

# 手寫迴圈版:加總
nums = [1, 2, 3, 4]
total = 0                       # 累加器的「初始值」
for n in nums:
    total = total + n           # 把 n 併進累加器
print(total)                    # 10

# 高階函式版:reduce(合併函式, 序列, 初始值)
from functools import reduce
total = reduce(lambda acc, n: acc + n, nums, 0)
print(total)                    # 10

看那個 lambda acc, n: acc + n:它收兩個參數——acc 是「目前累積到的結果」,n 是「這次要併入的元素」—— 回傳「併入之後的新累積值」。reduce 就拿著這個函式,從初始值 0 開始,一個一個把元素折進去。把過程攤開來看就懂了: 

reduce(+, [1, 2, 3, 4], 0)
= ((((0 + 1) + 2) + 3) + 4)
=   (((1)     + 2) + 3) + 4
=    ((3)          + 3) + 4
=     (6)               + 4
=      10

只要換掉那個「合併函式」,reduce 就能變出各種總結。連乘、找最大,都只是一行之差:

from functools import reduce

product = reduce(lambda acc, n: acc * n, [1, 2, 3, 4], 1)   # 連乘:24
biggest = reduce(lambda acc, n: acc if acc > n else n, [3, 9, 2, 7])  # 找最大:9
print(product, biggest)         # 24 9
💡
reduce 用得巧,但別硬用

reduce 很強,但也很容易寫出「只有作者看得懂」的程式。 Python 之父 Guido 本人就不太愛它——所以 map/filter 是內建的,reduce 卻被「流放」到 functools。 實務建議:加總用內建的 sum()、找最大用 max(),它們更清楚。 當你的「折疊」邏輯沒有現成函式、又真的是「逐步累積」時,才搬出 reduce。清楚永遠優先於炫技。

三劍客最美的地方,是它們能串起來,像水管接水管,讓資料流過一連串轉換。 「把所有偶數平方後加總」可以寫成一條清晰的管線: 

from functools import reduce
nums = [1, 2, 3, 4, 5, 6]
result = reduce(lambda acc, n: acc + n,           # 3. 折疊:加總
                map(lambda n: n * n,              # 2. 轉換:平方
                    filter(lambda n: n % 2 == 0,  # 1. 篩選:留偶數
                           nums)), 0)
print(result)            # 4 + 16 + 36 = 56

lambda、函式組合與部分套用

lambda:用完即丟的匿名函式

你已經在上面看了一堆 lambda,現在正式認識它。lambda 是用來建立匿名函式(anonymous function)的語法—— 一個小到「連名字都懶得取」的函式。它和 def 做的事一樣,只是更精簡: 

# 這兩個是等價的:
def square(x):
    return x * x

square = lambda x: x * x        # lambda 參數: 回傳的運算式

# lambda 的威力在於「即場產生一個函式,馬上交給高階函式」
nums = [3, 1, 4, 1, 5, 9, 2, 6]
print(sorted(nums, key=lambda x: -x))   # 用「負值」當排序鍵 → 由大到小

lambda 參數: 運算式——冒號左邊是參數,右邊是「直接回傳的那個運算式」(不能寫多行語句,不能有 return,因為它本身就是回傳值)。 lambda 最常見的用途,就是當高階函式(map、filter、sorted……)需要一個「一次性的小函式」時,即場生一個給它。

⚠️
lambda 不是越多越好

lambda 只適合「一行就講得完」的小邏輯。如果你的 lambda 開始變長、變複雜,或你想重複使用它, 請退回去用 def 給它一個好名字——一個有意義的名字,本身就是最好的註解。 把 lambda x: (x - 32) * 5 / 9 命名為 def fahrenheit_to_celsius(x):,讀者立刻就懂了。

函式組合:把小函式接成大函式

數學裡,$(f \circ g)(x) = f(g(x))$——先用 $g$ 處理 $x$,再把結果餵給 $f$。這叫函式組合(function composition)。 既然函式是一級值,我們可以寫一個高階函式,把兩個函式「黏」成一個新函式: 

def compose(f, g):
    return lambda x: f(g(x))     # 回傳一個「先做 g 再做 f」的新函式

def double(x):  return x * 2
def increment(x): return x + 1

double_then_inc = compose(increment, double)   # 先 double,再 increment
print(double_then_inc(5))        # (5*2)+1 = 11

inc_then_double = compose(double, increment)   # 順序反過來,結果不同!
print(inc_then_double(5))        # (5+1)*2 = 12

注意 compose 回傳的是一個函式——它是個不折不扣的高階函式(吃函式、吐函式)。 組合的精神是:寫一堆小而純的函式,再像樂高一樣把它們拼成複雜的行為。每塊樂高都好測、好懂,拼出來的整體也跟著可靠。

currying 與 partial application

最後兩個聽起來很玄、其實很實用的概念。先看 currying(柯里化): 把一個「吃多個參數」的函式,改寫成「一次只吃一個參數、回傳一個函式去吃下一個」的連環形式。名字來自邏輯學家 Haskell Curry(對,Haskell 語言也是以他命名)。 

# 一般的雙參數函式
def add(a, b):
    return a + b
print(add(3, 4))          # 7

# 柯里化版本:吃一個 a,回傳一個「還在等 b」的函式
def add_curried(a):
    return lambda b: a + b

print(add_curried(3)(4))  # 7   ← 連續呼叫兩次
add3 = add_curried(3)     # 先餵一半:得到一個「永遠加 3」的函式
print(add3(10))           # 13
print(add3(100))          # 103

關鍵收穫在 add3 = add_curried(3):我們只先提供了一部分參數,就得到一個「記住 3、等著收第二個數」的新函式。 這個「先固定一部分參數,生出一個更專用的函式」的動作,就叫 部分套用(partial application)。

你不必每次都手寫柯里化。Python 的 functools.partial 能對任何函式做部分套用: 

from functools import partial

def power(base, exp):
    return base ** exp

square = partial(power, exp=2)   # 把 exp 固定成 2,得到「平方」函式
cube   = partial(power, exp=3)   # 把 exp 固定成 3,得到「立方」函式
print(square(5))                 # 25
print(cube(5))                   # 125

# 實用例:把常用設定「凍」進去,得到更專用的函式
import functools
to_int_base2 = partial(int, base=2)   # int(x, base=2) 的專用版
print(to_int_base2("1011"))      # 11
🔑
currying 與 partial,差在哪?

currying 是把「一個 n 參數函式」改造成「n 個各吃一個參數的單參數函式鏈」——這是一種結構轉換。
partial application 是「固定掉其中幾個參數,得到一個參數更少的新函式」——這是一種具體動作。
在 ML、Haskell 這類語言裡,所有函式預設都是柯里化的,所以部分套用是家常便飯;在 Python 裡,我們靠 functools.partial 達成同樣效果。

閉包:函式記得它出生的地方

你有沒有注意到,剛剛的 add_curried(3) 回傳的那個 lambda b: a + b, 在 add_curried 早就執行完、它的區域變數 a 照理該消失之後,竟然還記得 a 是 3? 這個「函式記住它出生時周圍環境的變數」的現象,就是 FP 裡最迷人的機制之一:閉包(closure)。

我們用一個更乾淨的例子——make_adder——把它看個透徹: 

def make_adder(n):
    def adder(x):           # 內層函式,用到了外層的 n
        return x + n        # n 不是 adder 的參數,是「外面那層」的變數
    return adder            # 回傳內層函式本身(注意:沒有括號)

add5 = make_adder(5)        # n=5,回傳一個「記住 5」的函式
add10 = make_adder(10)      # n=10,回傳另一個「記住 10」的函式

print(add5(100))            # 105
print(add10(100))           # 110  ← 兩個函式各自記得不同的 n!

當 make_adder(5) 執行完,它的堆疊框照理該被丟掉,n 也該煙消雲散。但 adder 這個內層函式「用到了 n」, 於是 Python 很貼心地:讓回傳的 adder 函式把它需要的那塊環境(n=5)一起「背走」。 這個「函式 + 它捕獲的環境」的組合,就是閉包。add5 和 add10 是兩個獨立的閉包,各自背著自己的 n。

閉包還能拿來做「有記憶的函式」。下面這個計數器,把狀態藏在閉包裡,每次呼叫都記得上次數到幾: 

def make_counter():
    count = 0
    def counter():
        nonlocal count      # 宣告:我要「修改」外層的 count,不是新建區域變數
        count += 1
        return count
    return counter

c = make_counter()
print(c(), c(), c())        # 1 2 3   ← 它記得自己數到哪了
d = make_counter()          # 另一個獨立的計數器
print(d())                  # 1       ← 全新的,跟 c 互不干擾

這裡出現了一個新關鍵字 nonlocal。回想第 3 章:在函式裡對一個變數賦值,Python 預設會把它當成新的區域變數。 nonlocal count 是在說:「不,我指的是外層那個 count,我要修改它。」沒有它,count += 1 會報 UnboundLocalError。

🔍
閉包 vs 物件,其實是一體兩面

上面的計數器「把狀態藏起來、只透過函式介面操作」——這聽起來是不是很像物件(object)? 沒錯。有句名言:「閉包是窮人的物件,物件是窮人的閉包。」兩者都是「把資料和操作它的程式碼綁在一起」的手段, 只是出發點不同。等你讀到第 8 章物件導向設計,可以回頭對照這裡,會有「啊,原來是同一件事」的頓悟。

順帶一提:你最常遇到閉包的場合,其實是裝飾器(decorator)——Python 那個 @ 語法。 裝飾器就是「一個吃函式、回傳新函式」的高階函式,而那個新函式通常是個閉包(它記住了被包裝的原函式)。本章練習就有一題會帶你親手做一個。

用遞迴取代迴圈

到這裡你可能發現一件事:函數式風格似乎刻意避開了 for、while 這種會改變迴圈變數的迴圈。 為什麼?因為 for i in range(n) 裡的 i、while 裡那個被你 += 1 的計數器,本質上都是「會被反覆改寫的可變狀態」——正是 FP 想避免的東西。

那不用迴圈,怎麼「重複做事」?答案你第 3 章已經學過了:遞迴(recursion)。 在純函數式語言(如 Standard ML、Haskell、Scheme)裡,遞迴就是主要的迴圈手段。 比較「把一串數字加總」的兩種寫法: 

# 命令式:用會變的 total 和 for 迴圈
def sum_loop(nums):
    total = 0
    for n in nums:          # i 與 total 都在反覆改變
        total += n
    return total

# 函數式:遞迴,沒有任何變數被改寫
def sum_rec(nums):
    if not nums:            # base case:空串列,總和是 0
        return 0
    return nums[0] + sum_rec(nums[1:])   # 第一個 + 「剩下的總和」

print(sum_loop([1, 2, 3, 4]))   # 10
print(sum_rec([1, 2, 3, 4]))    # 10

遞迴版裡沒有「被覆寫的變數」——每一層呼叫都有自己全新的 nums,算完就回傳。整個計算是「一串值的對應」,而非「一塊白板被反覆塗改」。這正是函數式的精神。

尾遞迴:讓遞迴不吃堆疊

但第 3 章也提醒過:遞迴會吃呼叫堆疊,遞太深會堆疊溢位。 如果遞迴要當迴圈用,這豈不是天生的致命傷?函數式語言的解法是尾遞迴(tail recursion)。

所謂尾遞迴,是指遞迴呼叫是函式做的最後一件事——它的回傳值直接就是整個函式的回傳值,呼叫回來後不再做任何運算。 對照一下: 

# 不是尾遞迴:呼叫回來後「還要再加 nums[0]」
def sum_rec(nums):
    if not nums:
        return 0
    return nums[0] + sum_rec(nums[1:])      # 等子呼叫算完,還得做一次加法

# 尾遞迴:用 acc 累積,呼叫就是最後一步,回來後什麼都不做
def sum_tail(nums, acc=0):
    if not nums:
        return acc
    return sum_tail(nums[1:], acc + nums[0])  # 回傳值「就是」這個遞迴呼叫

為什麼這很關鍵?因為當「遞迴呼叫是最後一步」時,目前這層的堆疊框其實已經沒用了——沒有東西還等著它算完。 於是聰明的編譯器可以「重用」這個框,不再往上疊新框,把遞迴變成跟迴圈一樣只佔固定的記憶體。 這個最佳化叫尾呼叫消除(tail call optimization, TCO)。

在 Scheme、ML、Haskell 等語言,TCO 是語言標準的一部分——所以它們可以放心地「用尾遞迴當迴圈」,要遞迴幾百萬層都不會爆。 這就是為什麼這些語言裡很少看到 for:他們根本不需要。

⚠️
Python 偏偏不做 TCO

很現實:Python 刻意不支援尾呼叫消除(Guido 認為它會破壞除錯用的呼叫堆疊追蹤資訊)。 所以在 Python 裡,即使你寫成漂亮的尾遞迴,它照樣一層層疊框,照樣會撞上預設約 1000 層的遞迴上限。 結論:在 Python,需要大量重複時請直接用迴圈或 map/reduce,別指望尾遞迴幫你省堆疊。 但「尾遞迴」這個概念本身,在你讀 ML/Scheme/Haskell 時是必備知識。

嚐一口 ML 與 Racket/Scheme

到目前為止,我們都用 Python「假扮」函數式。但 Python 骨子裡是命令式的多範式語言,它的 FP 是「外掛」。 要真正感受函數式的味道,你得嚐一口為它而生的語言。OSSU 的 UW CSE341 Programming Languages 這門課, 正是用兩種「血統純正」的函數式語言帶你打開眼界:Standard ML(ML 家族)與 Racket(Lisp/Scheme 家族)。 我們各看一小段,你不需要會寫,只要感受它們的「腔調」。

Standard ML:靜態型別 + 模式比對的優雅

ML 家族(包含 Standard ML、OCaml、F#)的特色是:強大的靜態型別系統(編譯時就抓型別錯誤,卻幾乎不用你手寫型別, 靠「型別推論」自動推出來)、以及模式比對(pattern matching)讓程式碼貼近數學定義。看階乘: 

(* Standard ML:階乘。用「模式比對」分情況定義,像數學一樣 *)
fun factorial 0 = 1
  | factorial n = n * factorial (n - 1)

(* let ... in ... end 綁定區域值,本身是個運算式 *)
fun circle_area r =
    let
        val pi = 3.14159    (* val 綁定一個不可變的值,不是「變數」 *)
    in
        pi * r * r
    end

val result = factorial 5    (* 120;注意呼叫不用括號:factorial 5 *)

看那兩行 factorial:第一行說「0 的階乘是 1」,第二行說「n 的階乘是 n 乘以 (n-1) 的階乘」—— 這不就是數學課本的定義嗎?| 把不同情況分開,ML 會自動依照傳進來的值「比對」該用哪一條。 還有,val pi = 3.14159 裡的 val 不是「變數」,而是「綁定一個不可變的值」——在 ML 裡,值預設就是不可變的,跟我們前面強調的習慣完全一致。

Racket/Scheme:一切皆括號的 Lisp 之美

Racket 屬於 Lisp 家族(源自 Scheme)。它最鮮明的特徵是滿滿的括號與前綴記法(prefix notation): 運算子永遠寫在最前面,(+ 1 2) 就是 1 + 2。一開始你會覺得彆扭,但這套極簡的語法背後藏著驚人的一致性與威力。 

; Racket:define 定義、lambda 匿名函式、map/filter/foldl 高階函式
(define (square x) (* x x))          ; 定義函式 square
(define (factorial n)
  (if (= n 0)
      1
      (* n (factorial (- n 1)))))    ; 遞迴:n * (n-1)!

(square 5)                           ; => 25
(factorial 5)                        ; => 120

; 高階函式三劍客,在 Racket 是日常
(map square '(1 2 3 4))              ; => '(1 4 9 16)
(filter even? '(1 2 3 4 5 6))        ; => '(2 4 6)
(foldl + 0 '(1 2 3 4 5))             ; => 15   (foldl 就是 reduce)

; 匿名函式:lambda 直接用
((lambda (x) (* x x)) 6)             ; => 36

把 Racket 和 Python 對照,你會發現觀念完全相同:map 還是 map、filter 還是 filter、foldl 就是 reduce、lambda 還是 lambda。 只是 Lisp 把它們寫成 (函式 引數...) 的統一括號形式。這種「程式碼長得跟資料一模一樣」(都是括號列表)的特性, 讓 Lisp 擁有獨步天下的「巨集(macro)」能力——這是後話了。

📜
兩大家族,半部程式語言史

Lisp 誕生於 1958 年(John McCarthy),是世界第二古老的高階語言,把「函式是一級值」「程式即資料」帶給了世界。 ML 誕生於 1970 年代(Robin Milner),原本是用來寫「定理證明器」的工具,卻發展出影響深遠的型別推論系統。 今天你看到的許多現代語言特性——Python/JS 的 lambda、Rust/Swift 的模式比對與 Option 型別、各語言的型別推論—— 源頭都能追到這兩個家族。學它們,等於回到許多想法的發源地。

ℹ️
它們在 OSSU 裡的角色

CSE341 Programming Languages 用 ML、Racket、Ruby 三種語言,帶你比較「靜態 vs 動態型別」「函數式 vs 物件導向」等核心取捨—— 重點不是學三種語法,而是學會跳出單一語言、思考「程式語言設計」本身。 這也是為什麼它被放在「程式設計核心」:它讓你成為一個能評斷語言、而不只是使用語言的人。

代數資料型別與模式比對初探

ML 那段 factorial 用到的「模式比對」,搭配函數式語言另一個招牌——代數資料型別(algebraic data type, ADT)—— 會威力倍增。別被名字嚇到,它其實很直覺:ADT 是一種「用『或』把幾種可能組合起來」的型別。

舉例:一個「形狀」要嘛是圓形(需要一個半徑),要嘛是矩形(需要寬和高)。在 ML 裡你可以這樣定義它,再用模式比對拆解它: 

(* 定義一個 ADT:shape「是」Circle 或 Rect 其中之一 *)
datatype shape = Circle of real
               | Rect of real * real

(* 用模式比對,依照「實際是哪一種」分別處理 *)
fun area (Circle r)      = 3.14159 * r * r
  | area (Rect (w, h))   = w * h

val a1 = area (Circle 2.0)      (* 12.56636 *)
val a2 = area (Rect (3.0, 4.0)) (* 12.0 *)

看 area 怎麼定義的:它直接「攤開」每一種可能(Circle r 一行、Rect (w, h) 一行), 模式比對同時做了兩件事——判斷是哪一種,並把裡面的資料解開綁定到名字(r、w、h)。 這比 Python 傳統的 if isinstance(...) 一層層判斷再 .r、.w 取值優雅太多。

而且,ML 編譯器會檢查你「有沒有漏掉某種情況」——如果你忘了處理 Rect,它會警告你。這種「窮盡性檢查」是型別系統送給你的安全網。

Python 從 3.10 起也引進了結構化模式比對(match 語句),搭配 dataclass,能寫出神似 ML 的程式碼: 

from dataclasses import dataclass

@dataclass
class Circle:
    r: float

@dataclass
class Rect:
    w: float
    h: float

def area(shape):
    match shape:                       # 結構化模式比對(Python 3.10+)
        case Circle(r):                # 判斷是 Circle,並把半徑綁到 r
            return 3.14159 * r * r
        case Rect(w, h):               # 判斷是 Rect,並把寬高綁到 w, h
            return w * h
        case _:                        # 萬用模式,接住其餘情況
            raise ValueError("未知形狀")

print(area(Circle(2.0)))              # 12.56636
print(area(Rect(3.0, 4.0)))           # 12.0
🧭
與第 9 章「型別系統」的連結

代數資料型別不只是語法糖,它是型別系統的核心題目之一。ML 的型別推論、ADT 的窮盡性檢查、 現代語言用 Option/Result 型別「逼你處理空值與錯誤」,全都建立在這套基礎上。 我們在第 9 章「靜態與動態型別系統」會正式回來,談靜態型別如何在「程式還沒執行」時就替你擋下一整類錯誤—— 到時你會看到,本節嚐到的 ML 味道,正是那場大戲的序曲。

何時用 FP、何時收手

讀到這裡,你可能熱血沸騰,想把所有程式都寫成純函式的組合。先冷靜——FP 不是萬靈丹,它有一個無法迴避的根本張力。

純函式不能有副作用。但一個有用的程式,終究要跟真實世界互動:讀寫檔案、查詢資料庫、印東西到螢幕、回應網路請求、接收使用者輸入。 這些全都是副作用(I/O)。一個完全純的程式,算得再漂亮,也只是「在記憶體裡自言自語」,對外界毫無影響,等於沒用。

所以真正的智慧,不是「消滅副作用」,而是:

🔑
把純計算與副作用「分開」

讓程式的核心邏輯盡量是純函式(好測、好推理、好平行),把不得不做的副作用(I/O)集中、隔離到程式的邊緣。 這就是業界常說的「Functional Core, Imperative Shell(函數式核心,命令式外殼)」:中間那顆心臟是純的,只有最外面那層薄殼負責跟世界打交道。

具體什麼時候 FP 風格特別適合?什麼時候命令式更自然?給你一張務實的對照:

函數式風格大放異彩命令式可能更自然
資料轉換管線(過濾、映射、彙總一堆資料)大量、效能敏感的就地修改(如改寫巨大陣列)
需要平行/並行,怕競爭條件本質上就是一連串「步驟與狀態變化」的流程
需要高度可測試、可預測的核心邏輯與硬體、I/O 緊密互動的底層程式
數學/演算法性質強的計算團隊更熟悉、可讀性更高的命令式寫法

好消息是:你不必二選一。Python 是多範式(multi-paradigm)語言——你可以在同一支程式裡, 用純函式 + map/filter 處理資料核心,用 for 迴圈和物件處理流程與狀態,用 class 組織大型結構。 FP 對你最大的價值,往往不是「逼你全用它」,而是給你一種新的預設傾向:

  • 能寫純函式,就別讓它有副作用;
  • 能回傳新值,就別就地修改;
  • 能用小函式組合,就別寫一個巨大的函式;
  • 能避免共享的可變狀態,就避免——你未來的除錯人生會感謝你。
💡
FP 是一把尺,不是一個牢籠

就算你一輩子主要寫命令式或物件導向的程式,函數式思維仍會讓你變強:你會更警覺於「狀態在哪裡偷偷流動」, 更傾向寫出好測試的小函式,更懂得用 map/filter 把意圖講清楚。 把 FP 當成一套讓程式更好推理的紀律,而不是一個非此即彼的教條——這才是它送給你最實在的禮物。

動手練習

函數式這種東西,光看會點頭、動手才會懂。以下五題難度遞增,請務必先自己寫,卡住再看解答。 每寫一題,問自己:這裡有沒有「會變的狀態」?能不能把它換成「值的轉換」?

✏️ 練習 6.1 實作

下面這段命令式程式,把一串名字裡「長度大於 3」的名字,各自轉成大寫,收進新 list。 請用 map 和 filter(或串列推導式)把它改寫成函數式風格,不要用迴圈。

names = ["Al", "Bob", "Carol", "Dan", "Eve", "Frank"]
result = []
for name in names:
    if len(name) > 3:
        result.append(name.upper())
# 期望 result == ['CAROL', 'FRANK']
看解答

先 filter 篩出長度大於 3 的,再 map 轉大寫:

names = ["Al", "Bob", "Carol", "Dan", "Eve", "Frank"]

# 寫法一:filter + map
result = list(map(lambda s: s.upper(),
                  filter(lambda s: len(s) > 3, names)))
print(result)            # ['CAROL', 'FRANK']

# 寫法二:串列推導式(更 Pythonic,通常更推薦)
result = [s.upper() for s in names if len(s) > 3]
print(result)            # ['CAROL', 'FRANK']

兩種都好。串列推導式把「結果長什麼樣」一行講完,在 Python 裡通常比巢狀的 map/filter 更好讀。

✏️ 練習 6.2 實作

自己實作一個 my_map(func, seq),行為跟內建的 map 一樣:對 seq 的每個元素套用 func,回傳結果組成的 list。 不准用內建的 map。動手做一次,你就真正懂 map 在做什麼了。

看解答

核心就是第 6.3 節說的那個「迴圈骨架」——只是這次由你親手把它包起來:

def my_map(func, seq):
    result = []
    for x in seq:
        result.append(func(x))   # 對每個元素套用 func
    return result

print(my_map(lambda x: x * x, [1, 2, 3, 4]))   # [1, 4, 9, 16]
print(my_map(str.upper, ["a", "b", "c"]))      # ['A', 'B', 'C']

進階:同理可以寫 my_filter 與 my_reduce:

def my_filter(pred, seq):
    return [x for x in seq if pred(x)]

def my_reduce(func, seq, init):
    acc = init
    for x in seq:
        acc = func(acc, x)       # 把每個元素「折」進累加器
    return acc

print(my_filter(lambda x: x % 2 == 0, [1, 2, 3, 4]))   # [2, 4]
print(my_reduce(lambda a, b: a + b, [1, 2, 3, 4], 0))  # 10
✏️ 練習 6.3 實作

用 functools.reduce 完成兩件事(都不准用迴圈,也不准用內建的 max):

  • (a) 寫 factorial(n),用 reduce 計算 n!(提示:對 range(1, n+1) 做連乘)。
  • (b) 寫 maximum(nums),用 reduce 找出 list 裡的最大值。
看解答

階乘 = 把 1 到 n 全部「乘」在一起,初始值是 1;找最大 = 每次「留下兩者中較大的那個」:

from functools import reduce

def factorial(n):
    return reduce(lambda acc, x: acc * x, range(1, n + 1), 1)

def maximum(nums):
    return reduce(lambda acc, x: acc if acc > x else x, nums)
    # 不給初始值時,reduce 會自動拿第一個元素當起點

print(factorial(5))            # 120
print(maximum([3, 9, 2, 7, 5]))  # 9

注意 maximum 沒給第三個參數(初始值):此時 reduce 直接用序列的第一個元素當起點,從第二個開始折疊。 但這也代表——對空 list 它會丟 TypeError。永遠記得想邊界情況!

✏️ 練習 6.4 挑戰

用閉包做一個 memoize(func) 裝飾器:它包裝任何函式,把算過的結果記在一個快取裡; 下次用同樣的引數呼叫時,直接回傳快取的答案,不再重算。用它來「加速」第 3 章那個慢吞吞的樸素遞迴費氏數列。

提示:裝飾器就是「吃一個函式、回傳一個新函式」的高階函式,而那個新函式是個閉包,它要記住 func 和一個 cache 字典。

看解答

關鍵:cache 這個字典被閉包「背」在身上,跨多次呼叫持續存在。用引數當鍵,查得到就回傳、查不到才真的算:

def memoize(func):
    cache = {}                     # 被閉包捕獲,跨呼叫存活
    def wrapper(*args):
        if args not in cache:      # 沒算過才呼叫原函式
            cache[args] = func(*args)
        return cache[args]         # 算過的直接查表
    return wrapper                 # wrapper 是個閉包,記住 func 與 cache

@memoize                           # 等同 fib = memoize(fib)
def fib(n):
    if n < 2:
        return n
    return fib(n - 1) + fib(n - 2)

print(fib(50))                     # 12586269025,瞬間算完!

原本 $O(2^n)$ 的指數級樸素遞迴,被快取一包,瞬間降到 $O(n)$——因為每個 fib(k) 只會真正被算一次。 這個技巧叫記憶化(memoization),是動態規劃的入門票,我們會在第六部好好招待它。 (順帶一提,Python 標準庫已內建 functools.lru_cache 與 functools.cache 替你做這件事——但親手寫一次,你才真懂閉包與快取的魔法。)

✏️ 練習 6.5 挑戰

寫一個 pipe(*funcs),把任意多個單參數函式接成一條管線(pipeline): pipe(f, g, h)(x) 應該由左到右依序執行,等於 h(g(f(x)))。 用它把 「去頭尾空白 → 轉小寫 → 把空格換成底線」 串成一個「網址 slug 產生器」。

附加思考:再寫一個數學味的 compose(*funcs),它由右到左執行(compose(f, g, h)(x) == f(g(h(x)))), 符合數學上 $f \circ g \circ h$ 的慣例。你能用 reduce 寫出 compose 嗎?

看解答

pipe 回傳一個閉包,內部讓 x 依序流過每個函式(這其實就是「用迴圈做 reduce」):

def pipe(*funcs):
    def inner(x):
        for f in funcs:        # 由左到右,一個一個套用
            x = f(x)
        return x
    return inner

slugify = pipe(
    str.strip,                 # 去頭尾空白
    str.lower,                 # 轉小寫
    lambda s: s.replace(" ", "-"),  # 空格換成連字號
)
print(slugify("  Hello Functional World  "))   # hello-functional-world

附加:用 reduce 寫 compose。關鍵是把「函式串」折疊成「一個大函式」, 而因為要由右到左,我們從右邊開始一層層往外包:

from functools import reduce

def compose(*funcs):
    # 兩個函式合成:先做 g,再做 f
    return reduce(lambda f, g: lambda x: f(g(x)), funcs)

def double(x):    return x * 2
def increment(x): return x + 1
def negate(x):    return -x

h = compose(negate, increment, double)   # negate(increment(double(x)))
print(h(5))      # double(5)=10 → +1=11 → 取負 = -11

觀察 pipe 與 compose 互為鏡像:pipe(f, g, h) == compose(h, g, f)。 很多函式庫(如 JavaScript 的 Ramda、Python 的 toolz)同時提供這兩者,讓你依「讀起來順不順」自由選擇方向。

把這一章串起來

這一章,我們從「食譜步驟 vs 數學函數」的對比出發,學會用「計算值」取代「改變狀態」的眼光看程式。回顧幾個關鍵路標:

  • 兩種世界觀:命令式問「接下來做什麼」(改變狀態);函數式問「答案是什麼」(用函數對應輸入到輸出)。
  • 純函式(相同輸入→相同輸出、無副作用)帶來參照透明性——好測試、好推理、好平行化(第十部的伏筆)。
  • 不可變性:回傳新值而非就地修改;Python 用 tuple/frozenset 表達不可變資料。
  • 一級函式:函式是值,可指派、可傳遞、可回傳、可入資料結構——這是一切的基石。
  • 高階函式三劍客:map(逐一轉換)、filter(篩選)、reduce(折疊成單值),每個都是某種迴圈骨架的抽象。
  • lambda 造匿名小函式;compose 把小函式拼成大函式;currying / partial 先固定部分參數造出更專用的函式。
  • 閉包 = 函式 + 它捕獲的環境;它讓函式「記得出生地」,也是裝飾器與「有記憶的函式」背後的機制。
  • 遞迴取代迴圈:純函數式語言用(尾)遞迴當迴圈,靠 TCO 不吃堆疊——但 Python 不做 TCO。
  • ML 與 Racket 讓你嚐到血統純正的函數式;代數資料型別 + 模式比對讓程式碼貼近數學定義,並通往第 9 章的型別系統。
  • 務實心法:Functional Core, Imperative Shell——純計算放核心,副作用趕到邊緣。Python 是多範式語言,FP 是一把尺、不是牢籠。

你現在手上多了一套全新的思考工具:把問題看成「資料流過一連串純粹的轉換」。 下一章,我們要把這份對「函式」與「資料」的理解,升級成一套系統化的設計方法—— 教你面對任何問題,都能有章法地從資料定義一路推導出正確的程式。系好安全帶,我們要從「會寫」邁向「會設計」。
