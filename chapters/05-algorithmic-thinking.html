程式能跑,不等於問題解決了

前幾章你學會了變數、迴圈、函式,也許已經能寫出幾十行能跑的程式。但你大概也遇過這種場景: 程式「看起來」對了,丟個例子進去也沒報錯,結果上線後卻在某個你從沒想過的輸入上整個爆掉。

差別在哪?差在思維。寫程式只是把想法翻譯成語言;而在動手翻譯之前, 你得先把「想法」本身想清楚。這一章談的不是某個新語法,而是一整套解決問題的方法論—— 以及兩個讓程式真正可靠、卻最常被初學者跳過的環節:除錯(debugging)與 測試(testing)。這也是第一部的收尾:把它讀透,你就準備好迎接第六部那場關於演算法的盛宴了。

解決問題的方法論:六個步驟

厲害的工程師面對新問題時,腦中跑的並不是「趕快開始打字」,而是一條固定的流水線。 我們先把這條流水線攤開來,再用一個真實的小問題完整走一遍。

我們挑一個看似簡單、其實藏著魔鬼的問題當主角:

給你一串數字,找出裡面「第二大」的數。

① 理解問題:先把題目問清楚

新手最愛犯的錯,是題目都還沒讀懂就開始寫。「找第二大的數」聽起來不能更明確了,但魔鬼藏在細節:

  • 如果輸入是 [7, 7, 3],第二大是 7(就位置而言的第二名)還是 3(不重複的第二大)?
  • 如果輸入是 [5, 5, 5],根本沒有「第二大」,該回傳什麼?報錯?
  • 如果輸入只有一個數,甚至是空的呢?

這些問題沒有「標準答案」——它們是規格(specification)的一部分,得由你(或需求方)決定。 我們在這裡定義清楚:「第二大」指的是不重複的數值裡排第二的那個,所以 [7,7,3] 的答案是 3; 若不存在(全部相等、或數量不足),就丟出錯誤。

🔑
沒搞懂題目就動手,是 bug 的頭號來源

一段「正確實作了錯誤理解」的程式,比語法錯誤危險一百倍——因為它不會報錯,它只會默默給你錯的答案。 動手前花三分鐘把「輸入長什麼樣、輸出該是什麼、邊界情況怎麼處理」寫下來,是回報率最高的投資。

② 拆解(decomposition):把大問題切成小問題

拆解是 CS 思維的核心動作:把一個你不知道怎麼解的大問題,切成幾個你知道怎麼解的小問題。 「找第二大」可以怎麼拆?最直覺的拆法:

  1. 先把重複的數值去掉(因為我們要的是「不重複的第二大」)。
  2. 在剩下的數值裡找「最大」——這我們第 1 章就會了。
  3. 找「第二大」,等於「拿掉最大之後,剩下的最大」。

看!原本陌生的問題,被拆成了「去重」+「找最大」這兩個我們熟悉的零件。這就是拆解的威力。

③ 先寫虛擬碼(pseudocode):用人話打草稿

在敲下任何 Python 語法之前,先用虛擬碼(介於人話與程式之間的草稿)把邏輯寫順。 虛擬碼沒有語法規則,目的只有一個:讓你專心想「邏輯」,而不是分心於「分號、括號、縮排」。 

演算法:找第二大的數(不重複)
輸入:一串數字
1. 如果數字少於兩個,沒辦法找 → 報錯
2. 設兩個擂台:first(冠軍)與 second(亞軍),都先放「負無窮大」
3. 逐一檢查每個數字 x:
      如果 x 比 first 大:
          舊冠軍退居亞軍(second = first),x 登上冠軍(first = x)
      否則如果 x 比 second 大、而且 x 不等於 first:
          x 成為新亞軍(second = x)
4. 如果 second 還是「負無窮大」,代表根本沒有第二名 → 報錯
5. 輸出 second

注意第 3 步那個「而且 x 不等於 first」——這正是步驟①把規格想清楚的回報。 如果當初沒釐清重複值要怎麼處理,這裡就會埋下一個 bug。

💡
虛擬碼可以直接變成註解

一個好習慣:把虛擬碼貼進程式檔裡當成註解框架,再逐行「填」進真正的程式碼。 寫完後,那些註解剛好就成了文件。你會發現,難的從來不是語法,而是把邏輯想清楚。

④ 實作:把虛擬碼翻成 Python

有了清楚的虛擬碼,翻譯成程式幾乎是水到渠成:

def second_largest(nums):
    if len(nums) < 2:
        raise ValueError("至少需要兩個數字")

    first = second = float("-inf")    # 兩個擂台,先放負無窮大
    for x in nums:
        if x > first:
            first, second = x, first  # 舊冠軍退居亞軍,x 登頂
        elif x > second and x != first:
            second = x                # x 當不上冠軍,但夠格當亞軍

    if second == float("-inf"):
        raise ValueError("找不到第二大的數(數值可能全部相同)")
    return second

print(second_largest([3, 9, 2, 7, 5]))   # 7
print(second_largest([7, 7, 3]))         # 3(不重複的第二大)

這個版本只「掃過資料一次」,效率很好。但其實還有個更短、更好懂的寫法——用內建工具: 

def second_largest_simple(nums):
    distinct = sorted(set(nums))      # set 去重,sorted 排序
    if len(distinct) < 2:
        raise ValueError("找不到第二大的數")
    return distinct[-2]               # 倒數第二個就是第二大

⑤ 測試:證明它對,也找出它何時錯

程式寫完不是結束,而是另一個開始。我們得拿一堆例子去「拷問」它, 尤其是那些邊界情況(edge cases)。先手動試幾個關鍵案例: 

# 用 assert 當小考官:條件不成立就立刻喊停
assert second_largest([3, 9, 2, 7, 5]) == 7   # 一般情況
assert second_largest([7, 7, 3]) == 3         # 有重複
assert second_largest([1, 2]) == 1            # 剛好兩個
assert second_largest([-5, -2, -9]) == -5     # 全是負數
print("全部通過 ✅")

最後兩個案例特別重要:如果你當初把擂台初始化成 0 而不是 float("-inf"), 那麼「全是負數」這個案例就會把你的 bug 揪出來。邊界情況就是 bug 最愛躲的地方—— 測試這件事,我們稍後會專門展開講。

⑥ 優化:先求對,再求快

最後才談優化。注意這個順序:先讓它對,再讓它快。一段又快又錯的程式毫無價值。 我們這個問題,「掃一遍」的版本已經是線性時間,很難更好了。 但「先排序再取倒數第二」的版本其實做了多餘的工——它把整串都排好了,只為了拿一個數。 當資料量很大時,這個差別會變得很明顯。而「什麼時候該優化、優化什麼」, 答案幾乎永遠藏在下一個主題裡——演算法的選擇。

同一個問題,不同演算法,效率天差地別

現在來看本章最重要的一課。我們用一個再普通不過的任務當主角: 在一串數字裡,找出某個目標數在哪個位置。 這個任務有兩種經典解法,而它們的效率差距,大到會讓你重新理解「演算法」這兩個字的份量。

解法一:線性搜尋(linear search)

最直覺的做法:從頭到尾一個一個看,看到目標就回報位置。就像在沒有目錄的書裡找某個字,只能一頁一頁翻。 

def linear_search(arr, target):
    for i in range(len(arr)):
        if arr[i] == target:
            return i        # 找到了,回傳索引
    return -1               # 整串看完都沒有,回傳 -1 表示找不到

nums = [2, 5, 8, 12, 16, 23, 38, 45, 56, 72]
print(linear_search(nums, 23))   # 5
print(linear_search(nums, 99))   # -1

最壞情況下(目標在最後一個、或根本不存在),你得看完全部 $n$ 個元素。 我們說它的時間複雜度是 $O(n)$:資料有多長,工作量就跟著等比例增加。

解法二:二分搜尋(binary search)

但如果這串數字事先已經排好序,我們可以聰明得多。想想你查紙本字典找「machine」: 你不會從第一頁翻起,而是直接翻到中間,看到「M」之後的字,就知道該往前找;翻到「P」,就往後找。 每翻一次,你就砍掉一半的搜尋範圍。這就是二分搜尋。 

def binary_search(arr, target):
    low, high = 0, len(arr) - 1      # 搜尋範圍的左右邊界
    while low <= high:
        mid = (low + high) // 2      # 取中間位置(// 是整數除法)
        if arr[mid] == target:
            return mid               # 命中!
        elif arr[mid] < target:
            low = mid + 1            # 目標比中間大 → 丟掉左半,往右找
        else:
            high = mid - 1           # 目標比中間小 → 丟掉右半,往左找
    return -1                        # 範圍縮到空了還沒找到

nums = [2, 5, 8, 12, 16, 23, 38, 45, 56, 67, 72, 80, 91, 95, 99]
print(binary_search(nums, 23))   # 5
print(binary_search(nums, 99))   # 14
print(binary_search(nums, 100))  # -1

讓我們把這「每次砍半」的過程畫出來,在 15 個元素裡找 23:

差距有多大?把數字放大來看

因為每次砍半,二分搜尋最多只要 $\log_2 n$ 次比較。這個 $\log$ 帶來的差別,在小資料上不明顯, 放大之後卻是降維打擊:

資料量 $n$線性搜尋(最壞)二分搜尋(最壞)
1,0001,000 次約 10 次
1,000,0001,000,000 次約 20 次
1,000,000,00010 億次約 30 次

在十億筆資料裡找一個數,線性搜尋要看十億次,二分搜尋只要約 30 次—— 差距是三千多萬倍。這不是「快一點」,這是「能用」與「不能用」的分界線。

🔑
好演算法 > 快電腦

把線性搜尋跑在一台超級電腦上,還是會輸給跑在你手機上的二分搜尋——只要資料夠大。 換更快的硬體,頂多讓你快「幾倍」;換更好的演算法,可以讓你快「幾百萬倍」。 這就是為什麼計算機科學的核心,是「想出更聰明的方法」,而不是「買更貴的機器」。

⚠️
二分搜尋的前提:資料必須已排序

天下沒有白吃的午餐。二分搜尋的飛快,建立在「資料已經排好序」之上。 如果你只為了找一次就先花 $O(n \log n)$ 去排序,那還不如直接線性掃一遍。 二分搜尋真正划算的場景,是「排序一次,搜尋很多次」—— 這也是為什麼資料庫會幫常查的欄位建「索引」。

複雜度的直覺:資料變大 10 倍,時間變幾倍?

前面一直冒出 $O(n)$、$O(\log n)$ 這種符號。別緊張,我們現在只建立直覺, 正式定義(那些 $\exists c, n_0$ 的數學)留到第六部第 26 章再嚴謹處理。

Big-O 想回答的問題其實很白話:「當資料變大,工作量會跟著長多快?」 我們不在乎「跑了幾毫秒」(那跟電腦、語言有關),只在乎「成長的形狀」。 最好的體會方式,就是問自己:「資料變成 10 倍,時間大約會變幾倍?」

複雜度名稱資料 ×10,時間約 ×?感覺例子
$O(1)$常數不變幾乎免費,跟資料量無關用索引取陣列元素、查雜湊表
$O(\log n)$對數只多一點點好到不可思議二分搜尋
$O(n)$線性×10公平,成正比線性搜尋、找最大值、加總
$O(n \log n)$線性對數約 ×10 多一點很好,實務上的「快」合併排序、快速排序
$O(n^2)$平方×100資料一大就吃力氣泡/插入排序、兩兩比較
$O(2^n)$指數幾乎直接爆炸只能用在很小的 $n$暴力枚舉所有子集

感受一下那條 $O(2^n)$ 有多可怕:$n$ 每增加 1(不是變 10 倍,只是加 1),時間就翻倍。 $n = 50$ 時,$2^{50}$ 已經超過一千兆;就算每秒算十億次,也要跑超過十天。 $n = 100$?宇宙年齡都不夠你跑完。這就是為什麼當你聽到一個問題「只有指數解法」時, 那幾乎等於宣告「在大輸入上無解」。

ℹ️
Big-O 看的是「最壞情況」與「成長趨勢」

兩個約定要先記住:第一,Big-O 通常描述最壞情況(worst case), 因為我們想要的是「保證不會更慘」。第二,它忽略常數與低階項—— $O(2n + 100)$ 就是 $O(n)$,因為當 $n$ 夠大,那個「2」和「100」都無關緊要, 重要的只有「成長的形狀」。背後嚴謹的理由,第 26 章見。

兩個暖身排序:它們慢,但教會你思考

排序(sorting)是 CS 的「Hello, World 等級」演算法練習場。第六部會教你又快又漂亮的 $O(n \log n)$ 排序;但在那之前,我們先用兩個慢一點、卻直覺得多的排序熱身。 它們都是 $O(n^2)$,正好讓你親手摸到「平方級」的手感。

氣泡排序(bubble sort):大的數字「浮」到右邊

想像一串泡泡:比較相鄰的兩個,如果左邊比右邊大就交換。一輪掃下來,最大的數會像氣泡一樣, 一路「浮」到最右邊。重複這個過程,第二大的浮到倒數第二……直到全部排好。 

def bubble_sort(arr):
    n = len(arr)
    for i in range(n):                  # 要跑 n 輪
        swapped = False
        for j in range(n - 1 - i):      # 每輪比較相鄰兩數(已浮上去的不用再比)
            if arr[j] > arr[j + 1]:
                arr[j], arr[j + 1] = arr[j + 1], arr[j]   # 交換
                swapped = True
        if not swapped:                 # 一整輪都沒交換 → 已經排好,提早收工
            break
    return arr

print(bubble_sort([5, 2, 9, 1, 7]))   # [1, 2, 5, 7, 9]

插入排序(insertion sort):像整理手上的撲克牌

你打牌時怎麼理牌?左手已經是排好的一疊,右手每抽一張新牌,就從右往左比, 找到它該插入的位置塞進去。插入排序就是這個直覺:把陣列分成「左邊已排序」與「右邊待處理」兩區, 每次拿右區第一張,往左插到對的位置。 

def insertion_sort(arr):
    for i in range(1, len(arr)):     # arr[0] 自己算已排序,從第二張牌開始
        key = arr[i]                 # 手上這張要插入的牌
        j = i - 1
        while j >= 0 and arr[j] > key:   # 把比 key 大的數,通通往右挪一格
            arr[j + 1] = arr[j]
            j -= 1
        arr[j + 1] = key             # 空出來的位置,就是 key 的家
    return arr

print(insertion_sort([5, 2, 4, 6, 1]))   # [1, 2, 4, 5, 6]

為什麼這兩個都是 $O(n^2)$?直覺上:它們都有「一個迴圈套另一個迴圈」的結構—— 外層要跑約 $n$ 次,內層每次又要比較到約 $n$ 個元素。$n \times n = n^2$。 資料變 10 倍,時間就變約 100 倍。當 $n = 100{,}000$,$n^2 = 100$ 億次操作,你會等到睡著。

📚
慢排序也有它的價值

別小看它們。插入排序在資料很小或幾乎已排好時其實非常快, 因此許多工業級排序(如 Python 的 Timsort)會在小區段上偷偷改用插入排序。 至於真正能對付大資料的 $O(n \log n)$ 排序——合併排序、快速排序、堆積排序—— 是第六部的重頭戲。這裡你只要先把「平方很慢」這個手感刻進骨子裡。

除錯:這是常態,不是失敗

我要先幫你卸下一個心理包袱。許多初學者一看到紅色錯誤訊息就心跳加速,覺得「我是不是不適合寫程式」。 錯。連最頂尖的工程師,寫程式的時間有一大半都花在除錯上。 Bug 不是你的失敗,它是寫程式這件事的常態。差別只在於:高手有一套抓 bug 的系統方法,而新手只會瞎改。

📜
第一隻「bug」是真的蟲

1947 年,先驅工程師 Grace Hopper 的團隊在 Harvard Mark II 電腦裡, 發現一隻飛蛾卡在繼電器中導致故障。他們把那隻蛾貼進工作日誌,寫下 「First actual case of bug being found」。從此 debug(除蟲/除錯)就成了程式設計的日常用語。

第一步:讀懂錯誤訊息(traceback)

Python 出錯時會吐出一段 traceback(回溯)。新手常一看到就直接略過, 但它其實是在「報案」,而且把線索都給你了。關鍵是:從最下面往上讀。 

def average(nums):
    return sum(nums) / len(nums)

print(average([]))   # 不小心傳了空串列
Traceback (most recent call last):
  File "demo.py", line 4, in <module>
    print(average([]))
  File "demo.py", line 2, in average
    return sum(nums) / len(nums)
ZeroDivisionError: division by zero

traceback 的閱讀順序:

  • 最後一行是 bug 的類型與描述:ZeroDivisionError: division by zero——除以零了。
  • 往上看,它告訴你出事的確切位置:line 2, in average 的 sum(nums) / len(nums)。
  • 再往上是「呼叫鏈」:誰呼叫了誰。原來是 line 4 用空串列呼叫了 average。

三行字,就把「什麼錯、在哪錯、怎麼被觸發」全交代了。讀懂 traceback,你已經解決了一半的 bug。

print 除錯法:最樸素,也最好用

當程式沒報錯、但結果不對時,最簡單的武器就是在可疑處插入 print, 把變數的「實際值」攤開來看。你猜測的程式狀態,和它真實的狀態,往往就差在這裡。 

def buggy_sum_to(n):
    total = 0
    for i in range(n):        # 想加 1+2+...+n,但這裡有 bug
        total += i
    return total

# 加 print 觀察:i 到底跑了哪些值?total 怎麼變?
def debug_sum_to(n):
    total = 0
    for i in range(n):
        total += i
        print(f"i={i}, total={total}")   # 把過程攤開來看
    return total

debug_sum_to(5)
# i=0, total=0   ← 啊哈!i 從 0 開始,還少加了 5
# i=1, total=1
# ...

一看輸出就真相大白:range(n) 產生的是 0..n-1, 既多算了 0、又漏了 n。正解是 range(1, n + 1)。這種「邊界差一」的錯,我們等等會專門講。

小黃鴨除錯法(rubber duck debugging)

這招聽起來像玩笑,卻出奇有效:找一隻橡皮鴨(或任何東西、任何人),把你的程式一行一行、 逐字逐句講給它聽。當你被迫把「我以為它在做什麼」用嘴巴說出來時, 大腦會自動發現「咦,它其實沒在做這件事」。很多 bug 在你講到一半時就自己現形了—— 因為說明的過程,逼你從「腦中模糊的假設」切換成「精確的事實」。

二分法定位:把搜尋 bug 也變成「砍半」

還記得二分搜尋嗎?同樣的「砍半」思想可以用來抓 bug。當你有一段長程式不知道哪裡壞了, 不要從頭一行行盯——在程式中段插一個檢查點(print 或 assert), 看看「到這裡為止,狀態還對嗎?」對 → bug 在後半;不對 → bug 在前半。 每次都砍掉一半的嫌疑範圍,幾步就能逼近真凶。版本控制裡的 git bisect(第四部會學) 就是把這招用在「哪一次提交引入了 bug」上。

用真正的除錯器:pdb 與 breakpoint()

當 print 不夠用,該請出專業工具:Python 內建的除錯器 pdb。 在你想暫停的那一行寫上 breakpoint()(Python 3.7+),程式跑到那裡就會停下來, 把控制權交給你,讓你像「按下慢動作」一樣逐步檢視: 

def find_second_largest(nums):
    first = second = float("-inf")
    for x in nums:
        breakpoint()          # 程式跑到這裡會暫停,進入互動除錯模式
        if x > first:
            first, second = x, first
        elif x > second:
            second = x
    return second

find_second_largest([3, 9, 2])

進入 pdb 後,在那個 (Pdb) 提示符號下,幾個最常用的指令:

指令全名作用
nnext執行下一行(不跳進函式內部)
sstep執行下一行(會跳進被呼叫的函式)
ccontinue繼續跑,直到下一個 breakpoint 或結束
p xprint印出變數 x 目前的值
llist顯示目前停在哪幾行
qquit離開除錯器

用 n 一步步走、用 p first / p second 隨時查看擂台狀態, 你就能親眼看著變數怎麼變化,而不必用 print 把畫面洗版。除錯器是你從新手邁向熟手的重要工具,值得花時間練熟。

💡
除錯心法:先形成假設,再驗證

不要亂槍打鳥地瞎改。科學的除錯流程是:觀察現象 → 提出「我猜是因為 X」的假設 → 設計一個能證明或推翻它的實驗(加 print、下 breakpoint)→ 看結果 → 修正假設。 這跟科學家做研究一模一樣。你不是在「修程式」,你是在「查案」。

測試:怎麼知道你的程式「真的」對?

你怎麼確定 second_largest 是對的?跑一次 [3,9,2,7,5] 得到 7? 那只證明它在這一個輸入上對。程式可能在你沒試過的輸入上默默出錯。 測試(testing)就是有系統地拿各種輸入去「拷問」程式,讓 bug 在上線前就現形。

最小的測試:assert 斷言

assert(斷言)是你最輕便的測試工具:它後面接一個「你認為一定為真」的條件, 如果條件成立,什麼都不發生;如果不成立,程式立刻停下並報錯。 

assert second_largest([3, 9, 2, 7, 5]) == 7
assert second_largest([7, 7, 3]) == 3
# 如果哪一行的等號不成立,程式會在那裡丟出 AssertionError

邊界情況(edge cases):bug 最愛躲的角落

測「正常」輸入誰都會,真正考驗你的是邊界情況。寫測試時,養成習慣把這份清單過一遍:

  • 空輸入:空串列 []、空字串 ""
  • 單一元素:只有一個的 [42]
  • 重複值:[5, 5, 5]、[7, 7, 3]
  • 極大 / 極小:超大的數、超長的串列
  • 負數與零:[-3, -1, 0](很多 bug 來自「以為輸入都是正的」)
  • 已排序 / 反向排序:對排序與搜尋演算法特別關鍵

把這份清單套用到我們的函式,寫成一個手刻的測試函式:

def test_second_largest():
    # 正常情況
    assert second_largest([3, 9, 2, 7, 5]) == 7
    # 邊界:剛好兩個
    assert second_largest([1, 2]) == 1
    # 邊界:有重複值
    assert second_largest([7, 7, 3]) == 3
    # 邊界:全是負數
    assert second_largest([-5, -2, -9]) == -2
    # 邊界:不合法輸入應該要報錯
    try:
        second_largest([5])
        assert False, "單一元素應該要丟出 ValueError"
    except ValueError:
        pass   # 有正確報錯,測試通過
    print("test_second_largest 全部通過 ✅")

test_second_largest()
🔑
測試驅動的心態:先想「怎樣算對」

有經驗的工程師常在動手寫功能之前,就先把測試寫好——這叫 測試驅動開發(Test-Driven Development, TDD)。即使你不嚴格照 TDD 做, 那個心態也價值連城:在寫程式前,先逼自己清楚回答「這個函式怎樣才算對? 哪些輸入會讓它出錯?」當你能寫出測試,代表你真的想清楚規格了。

升級到框架:unittest 與 pytest

當測試變多,手刻會變亂。Python 內建 unittest 框架, 幫你把測試組織起來、自動執行、印出漂亮的報告: 

import unittest

class TestSecondLargest(unittest.TestCase):
    def test_normal(self):
        self.assertEqual(second_largest([3, 9, 2, 7, 5]), 7)

    def test_duplicates(self):
        self.assertEqual(second_largest([7, 7, 3]), 3)

    def test_too_short_raises(self):
        with self.assertRaises(ValueError):   # 預期它會丟 ValueError
            second_largest([5])

if __name__ == "__main__":
    unittest.main()

業界更流行的是第三方的 pytest,語法更輕巧——你甚至可以直接用熟悉的 assert: 

# 檔名以 test_ 開頭,函式也以 test_ 開頭,pytest 會自動找到並執行
import pytest

# 終端機執行:pytest  →  它會自動跑完所有 test_ 函式並回報

現在你不用記住每個細節,只要建立一個觀念:專業的程式碼,都附帶一整套會自動執行的測試。 每次改完程式,跑一遍測試,就能立刻知道「有沒有把原本好好的東西改壞」(這叫 回歸測試 regression testing)。測試是你重構與改進程式時的安全網。

防禦性思考:不要相信任何輸入

測試問的是「我的程式在各種輸入下對不對」;防禦性程式設計(defensive programming) 則更進一步問:「如果有人餵給我惡意或離譜的輸入,會怎樣?」 一個成熟的程式,會在門口先驗證輸入(input validation),把不合法的東西擋在外面。 

def average(nums):
    # 先驗證,再動工
    if not isinstance(nums, list):
        raise TypeError("nums 必須是 list")
    if len(nums) == 0:
        raise ValueError("不能對空串列求平均")
    return sum(nums) / len(nums)

這背後有個貫穿整個 CS 的鐵律,尤其在資訊安全領域:

⛔
永遠不要相信使用者輸入

「Never trust user input.」當你的程式接收外部資料(使用者打的字、網路來的封包、檔案內容), 你必須假設它可能是錯的、缺的、甚至是攻擊者精心構造來搞垮你的。 史上多數重大資安漏洞——SQL injection、緩衝區溢位、XSS——根源都是「相信了不該相信的輸入」。 這顆種子我們先埋在這裡,第七部「資訊安全」會讓它長成大樹。

C 對照:同樣的搜尋,底層的真相

Python 幫我們隱藏了很多細節。換成 C,你會看見演算法在記憶體層面「真正」長什麼樣子—— 也會撞見一個經典到值得單獨拿出來講的真實世界 bug。

線性搜尋(C 版)

#include <stdio.h>

/* C 的陣列不知道自己有多長,所以必須額外把長度 n 一起傳進來 */
int linear_search(int arr[], int n, int target) {
    for (int i = 0; i < n; i++) {
        if (arr[i] == target) {
            return i;          /* 找到,回傳索引 */
        }
    }
    return -1;                 /* 找不到 */
}

int main(void) {
    int data[] = {2, 5, 8, 12, 16, 23, 38, 45};
    int n = sizeof(data) / sizeof(data[0]);   /* 計算元素個數的慣用法 */
    printf("%d\n", linear_search(data, n, 23));   /* 5 */
    printf("%d\n", linear_search(data, n, 99));   /* -1 */
    return 0;
}

注意第一個關鍵差異:C 的陣列傳進函式時會「退化」成指標,丟失長度資訊。 所以你必須把長度 n 當成另一個參數一起傳——這是 C 程式設計最基本、也最常被忘記的紀律。 忘了它,你的迴圈就會讀出陣列邊界、踩進不屬於你的記憶體。

二分搜尋(C 版)與那個著名的 overflow bug

int binary_search(int arr[], int n, int target) {
    int low = 0, high = n - 1;
    while (low <= high) {
        int mid = low + (high - low) / 2;   /* 注意!不是 (low + high) / 2 */
        if (arr[mid] == target) {
            return mid;
        } else if (arr[mid] < target) {
            low = mid + 1;     /* 丟掉左半 */
        } else {
            high = mid - 1;    /* 丟掉右半 */
        }
    }
    return -1;
}

看到那行 mid = low + (high - low) / 2 的註解了嗎?為什麼不寫成更直覺的 (low + high) / 2?這裡藏著一個潛伏了幾十年的真實 bug。

⚠️
整數溢位:潛伏 20 年的二分搜尋 bug

在 C / Java 這類語言,int 能表示的最大值有上限(常見是約 21 億,$2^{31}-1$)。 當陣列很大,low 和 high 都接近上限時,low + high 會超過上限而溢位, 變成一個負數,讓 mid 算出荒謬的值,程式崩潰。 改寫成 low + (high - low) / 2 在數學上完全等價,卻永遠不會溢位 (high - low 一定是個不大的正數)。
這不是學院派的杞人憂天:2006 年,Google 的 Joshua Bloch 公開指出, Java 官方函式庫 Arrays.binarySearch 以及無數教科書裡的二分搜尋, 都犯了這個錯——而它整整潛伏了將近二十年沒人發現。一個「看起來絕對正確」的式子, 也可能藏著魔鬼。

Python 為什麼不用擔心?因為 Python 的整數會自動長大,要多大有多大,沒有溢位這回事—— 這正是 Python 「幫你隱藏底層真相」的又一例。C 不幫你,所以 C 逼你看見真相。 兩種語言,兩種收穫。

動手練習

看懂和會做是兩回事。下面五題由淺入深,請務必先自己動手,卡住了再看解答。

✏️ 練習 5.1 實作

自己實作一個二分搜尋 binary_search(arr, target)(假設 arr 已由小到大排序)。 找到就回傳索引,找不到要回傳 -1。請特別測試這些情況:目標在開頭、在結尾、在中間、不存在、以及空陣列。

看解答

關鍵在迴圈條件 low <= high 與邊界更新 mid + 1 / mid - 1。 少了那個 =,或忘了 ±1,都可能讓「找不到」時陷入無窮迴圈或漏掉答案。

def binary_search(arr, target):
    low, high = 0, len(arr) - 1
    while low <= high:
        mid = (low + high) // 2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            low = mid + 1
        else:
            high = mid - 1
    return -1

# 測試(含邊界)
assert binary_search([1, 3, 5, 7, 9], 1) == 0    # 開頭
assert binary_search([1, 3, 5, 7, 9], 9) == 4    # 結尾
assert binary_search([1, 3, 5, 7, 9], 5) == 2    # 中間
assert binary_search([1, 3, 5, 7, 9], 4) == -1   # 不存在
assert binary_search([], 1) == -1                # 空陣列
print("通過 ✅")

空陣列為什麼自動就對?因為 high = -1,迴圈條件 0 <= -1 一開始就不成立,直接回傳 -1。好的邊界設計,常常讓特例「自然地」被涵蓋。

✏️ 練習 5.2 實作

寫一個函式 most_common(items),回傳串列中出現次數最多的那個元素(眾數)。 例如 most_common([1, 3, 3, 2, 3, 1]) 應回傳 3。想想:它的時間複雜度是多少?

看解答

用一個字典(雜湊表)記錄每個元素出現幾次,邊數邊更新目前的冠軍。只掃一遍,所以是 $O(n)$。

def most_common(items):
    if not items:
        raise ValueError("空串列沒有眾數")
    counts = {}
    best, best_count = items[0], 0
    for x in items:
        counts[x] = counts.get(x, 0) + 1   # 累加這個元素的次數
        if counts[x] > best_count:
            best, best_count = x, counts[x]
    return best

assert most_common([1, 3, 3, 2, 3, 1]) == 3
assert most_common(["a", "b", "a"]) == "a"
print("通過 ✅")

進階:Python 標準函式庫的 collections.Counter 一行就能搞定(Counter(items).most_common(1))。 但親手寫一遍,你才真正理解它在做什麼。「會用工具」與「懂工具怎麼運作」,都重要。

✏️ 練習 5.3 思考|測試

有人寫了一個 clamp(x, low, high) 函式,作用是「把 x 限制在 [low, high] 範圍內」 (小於 low 就回傳 low,大於 high 就回傳 high,否則回傳 x 本身)。 請你不看實作,只憑這段描述,寫出一組涵蓋邊界情況的測試。你能想到幾種邊界?

看解答

好的測試應該涵蓋:在範圍內、剛好等於上下界、低於下界、高於上界、以及 low == high 的退化情況、負數等。

def clamp(x, low, high):
    if x < low:
        return low
    if x > high:
        return high
    return x

def test_clamp():
    assert clamp(5, 0, 10) == 5      # 在範圍內
    assert clamp(-3, 0, 10) == 0     # 低於下界 → 夾到 low
    assert clamp(99, 0, 10) == 10    # 高於上界 → 夾到 high
    assert clamp(0, 0, 10) == 0      # 邊界:剛好等於 low
    assert clamp(10, 0, 10) == 10    # 邊界:剛好等於 high
    assert clamp(7, 7, 7) == 7       # 退化:low == high
    assert clamp(-5, -10, -1) == -5  # 全負數範圍
    print("test_clamp 通過 ✅")

test_clamp()

重點不在這幾行 assert,而在你「動手前先列出所有邊界」的思考過程。 能列出邊界清單,代表你真的把規格想透了——這正是測試最有價值的副作用。

✏️ 練習 5.4 除錯

下面這個 contains 想判斷 target 在不在陣列裡。它在「找得到」時運作正常, 但只要 target 不存在,就會崩潰。請找出這個「差一(off-by-one)」的 bug 並修正。

def contains(arr, target):
    i = 0
    while i <= len(arr):        # 這裡有問題
        if arr[i] == target:
            return True
        i += 1
    return False

print(contains([1, 2, 3], 2))   # True(沒問題)
print(contains([1, 2, 3], 9))   # 崩潰!IndexError
看解答

bug 在迴圈條件 i <= len(arr)。陣列 [1,2,3] 的合法索引是 0, 1, 2, 但這個條件會讓 i 一路跑到 3(等於 len(arr)),於是 arr[3] 越界, 丟出 IndexError。為什麼「找得到」時沒事?因為它在越界之前就 return 了, bug 被提早結束的迴圈「遮住」了——這正是邊界 bug 陰險的地方。

修正:把 <= 改成 <。

def contains(arr, target):
    i = 0
    while i < len(arr):         # 改成 <,索引就不會越界
        if arr[i] == target:
            return True
        i += 1
    return False

「差一錯誤」是程式設計界最常見的 bug 類型之一,常出在迴圈邊界、陣列索引、 「要不要包含端點」的判斷上。看到 < 與 <=、n 與 n-1 時,請多停三秒。

✏️ 練習 5.5 挑戰

用我們學到的「直覺法」估計下面三段程式的 Big-O 複雜度(用 $n$ 表示 len(nums))。 別急著查公式,先問自己:「最內層那行,總共會被執行幾次?」

# A
def f_a(nums):
    return nums[0]

# B
def f_b(nums):
    count = 0
    for i in range(len(nums)):
        for j in range(i + 1, len(nums)):
            if nums[i] + nums[j] == 0:
                count += 1
    return count

# C
def f_c(nums):
    n = len(nums)
    steps = 0
    while n > 1:
        n = n // 2        # 每次砍半
        steps += 1
    return steps
看解答

A:$O(1)$。不管陣列多長,它只取第一個元素,做的事是固定的。

B:$O(n^2)$。兩層巢狀迴圈。雖然內層因為 i + 1 越跑越短, 總比較次數是 $(n-1) + (n-2) + \dots + 1 = \frac{n(n-1)}{2}$。這是個 $n^2$ 量級的式子 (常數 $\frac{1}{2}$ 與低階項在 Big-O 裡都被忽略),所以是 $O(n^2)$。 這是「枚舉所有配對」的典型複雜度。

C:$O(\log n)$。關鍵在 n = n // 2——每一圈都把 n 砍半。 從 $n$ 砍到 1,需要的次數正是 $\log_2 n$。任何「每次把問題規模砍半」的結構,都帶著對數的味道, 二分搜尋也是同一個道理。

進階思考:如果把 B 裡面的內層改成另一個會砍半的迴圈,整體會變成什麼?(答案:$O(n \log n)$—— 外層線性 × 內層對數。學會把複雜度「相乘」與「相加」,你就能徒手估計絕大多數程式了。)

把這一章串起來

這一章,我們從「程式能跑 ≠ 問題解決」出發,補上了讓你從「會寫程式」進化到「會解決問題」的關鍵內功:

  • 方法論:理解 → 拆解 → 虛擬碼 → 實作 → 測試 → 優化。最重要的是那條回頭的虛線——解題是循環,不是直線。
  • 演算法的選擇能決定一切:線性搜尋 $O(n)$ vs 二分搜尋 $O(\log n)$,在大資料上是「能用」與「不能用」之差。好演算法勝過快電腦。
  • Big-O 是看「成長的形狀」:問「資料 ×10,時間 ×幾?」就抓到了直覺;嚴謹定義留待第 26 章。
  • 暖身排序:氣泡與插入都是 $O(n^2)$,讓你摸到平方的手感。$O(n \log n)$ 的高效排序是第六部的主菜。
  • 除錯是日常:讀懂 traceback、print 法、小黃鴨、二分定位、pdb——把抓 bug 當查案,先假設再驗證。
  • 測試與防禦:用 assert、邊界清單、unittest/pytest 證明程式對;並謹記「永遠不要相信輸入」。

到這裡,第一部「計算機科學入門」就告一段落了。你已經知道計算是什麼、會用 Python 表達想法、 懂得像工程師一樣拆解問題與保證品質。手上的工具齊了,接下來,該認真磨「演算法」這把劍—— 第六部見,我們會把這一章埋下的每一顆種子,養成參天大樹。
