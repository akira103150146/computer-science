系統化程式設計法

大多數人寫程式靠瞎猜、亂改、碰運氣。但有一群人,把寫程式變成了像蓋房子一樣——先有藍圖,再砌磚。這一章,要把那張藍圖交到你手上。

先有藍圖,再蓋房子

想像有兩個人要蓋房子。

第一個人直接搬磚開砌。砌到一半發現門忘了留,敲掉重來;牆砌好了才想到水管要走牆裡, 再鑿開;二樓蓋上去才發現一樓的柱子撐不住,整棟拆掉。他不是不努力,他甚至非常努力—— 只是他把所有力氣都花在「試錯」上。

第二個人先畫藍圖:幾房幾廳、承重牆在哪、水電怎麼走、地基要多深。畫圖時他想清楚了所有關係, 等到真正動工,砌磚只是「照圖施工」,順得不得了。

你大概已經猜到我要說什麼了。大多數初學者寫程式,是第一個人:打開編輯器, 憑著一股「我覺得應該這樣」的靈感開始敲,跑不過就改一行再跑、再改再跑, 最後拼湊出一坨「能跑但自己也說不清為什麼能跑」的程式。這種寫法,我們上一章已經給過它一個名字—— 瞎猜試錯。

但程式設計其實可以是第二個人。1990 年代末,一群電腦科學教育者提出一個主張: 寫一個函式,不該靠靈感,而該照著一條固定、可重複、誰來做都一樣的流程。 他們把這條流程寫成一本書,書名就叫 How to Design Programs(HtDP), 而這條流程的核心,叫做設計配方(design recipe)。

🔑
配方的承諾

「設計配方」不是要把你變成機器人,也不保證你想得出聰明的演算法。它做的事更基本、也更重要: 它保證你永遠不會對著空白螢幕發呆。不管問題多陌生,你都有「下一步該做什麼」的明確指示。 它把「寫程式」這件充滿焦慮的事,變成一條讓你穩穩往前走的流水線。

📜
HtDP 與它的「教學語言」

How to Design Programs 由 Matthias Felleisen、Robert Bruce Findler、Matthew Flatt、 Shriram Krishnamurthi 四位學者合著,第一版 2001 年由 MIT Press 出版,第二版 2018 年問世, 並完整免費放在 htdp.org 上。它用的不是一般的程式語言,而是 Racket (前身為 PLT Scheme)裡一系列循序漸進的「教學語言」(BSL、ISL……)。 本章我們用 Python 來示範配方的精神,並在關鍵處附上 Racket 原版的對應寫法,讓你看見它原本的長相。

設計配方:把寫程式變成一條紀律

設計配方的核心洞見是:「寫一個函式」這件事,可以拆成六個獨立、有順序的步驟。 每一步都產出一個具體的、寫得出來的東西——不是「在腦中想想」,而是真的打進檔案裡。 你一步步填,函式自己就長出來了。

這六步乍看囉嗦——「不就是寫個函式,哪需要這麼多儀式?」但請先忍住這個念頭。 我們馬上用一個再簡單不過的例子完整走一遍,你會發現:囉嗦的是把它寫下來, 而那正是重點。把它寫下來,就逼你想清楚;想清楚了,程式自己就浮現了。

六步配方走一遍:計算圓面積

我們挑一個你國小就會的問題當白老鼠:給一個圓的半徑,算出它的面積。 面積公式是 $A = \pi r^2$,小學數學。正因為它「太簡單」,我們才能把全副注意力放在配方本身, 而不是被問題難度分心。

步驟 1:資料定義(data definition)

第一個問題不是「怎麼算」,而是「資料長什麼樣」。半徑在程式裡,該用什麼型別表示? 它是一個數,而且不會是負的。我們把這個認知寫成一段註解,給這種資料取個名字: 

# 資料定義
# 一個「半徑(Radius)」是一個 >= 0 的浮點數(float)
# 解讀:代表一個圓的半徑長度,單位不限
# 範例:0.0、1.5、10.0

看起來只是廢話?但它逼你回答了一個常被略過的問題:「負的半徑算合法輸入嗎?」(不算。) 這一行註解,就是你和未來的自己簽下的合約。

ℹ️
Racket 原版怎麼寫

在 HtDP 的教學語言裡,這一步長這樣:
; A Radius is a non-negative Number
一模一樣的精神——先替資料命名、說清楚它是什麼。

步驟 2:簽章(signature)

簽章說的是:這個函式吃進什麼型別、吐出什麼型別。它是函式的「規格標籤」。 我們的 area 吃一個 Radius(其實就是 float),吐出一個面積(也是 float): 

# 簽章:Radius -> float
# (吃一個半徑,回傳一個面積)
def area(r: float) -> float:
    ...

Python 的型別提示(type hints) r: float 和 -> float 剛好就是簽章的程式化身。Racket 原版寫成註解:; area : Number -> Number。 無論哪種寫法,你都被迫先回答:輸入是什麼、輸出是什麼。

步驟 3:目的陳述(purpose statement)

簽章說的是「型別」,目的陳述說的是「意義」:用一句人話講清楚這函式做什麼。 在 Python 裡,最自然的家就是 docstring: 

def area(r: float) -> float:
    """計算半徑為 r 的圓面積。"""
    ...

「一句話」是個刻意的限制。如果你發現一句話講不完,通常代表這個函式想做的事太多了, 該拆成兩個函式——這是一個免費附送的設計警報。

步驟 4:範例與測試(examples / tests)

關鍵的一步來了。在寫任何實作之前,我們先寫下「我期待它給我什麼答案」。 半徑 0,面積該是 0;半徑 1,面積該是 $\pi \approx 3.14159$;半徑 2,面積該是 $4\pi \approx 12.566$。 把這些期待寫成可以自動檢查的測試: 

import math

# 範例與測試(先寫!此時 area 都還沒實作)
def test_area():
    assert area(0) == 0
    assert math.isclose(area(1), math.pi)
    assert math.isclose(area(2), 4 * math.pi)

為什麼用 math.isclose 而不是 ==?因為浮點數有微小誤差,直接用 == 比兩個小數, 常常會「看起來相等卻判定不等」。這是浮點運算的經典陷阱,math.isclose 幫你容忍那點誤差。

ℹ️
Racket 原版:check-expect

HtDP 內建一個漂亮的測試語法,直接把例子寫在函式旁邊:
(check-expect (area 1) 3.14)
它和我們的 assert 是同一件事:在實作之前,先白紙黑字寫下「正確答案應該長這樣」。 這就是測試先行(test-first)的精神,本章稍後會專門展開。

步驟 5:樣板(template)

樣板是函式的「骨架」——依資料的形狀先把結構搭出來,但還不寫邏輯。 對 area 來說,輸入是一個單純的原子值(一個 float),沒有分支、沒有結構, 所以樣板簡單到幾乎是廢話:函式體就是「對 r 做點什麼」。 

# 樣板(原子資料 → 直接運用那個值)
def area(r: float) -> float:
    return ...  # 這裡會用到 r

你可能覺得「這步根本沒做什麼」。沒錯——但那是因為這個資料太簡單。 等一下遇到列舉、區間、清單、樹,你就會看到樣板真正的威力:資料一旦有結構, 樣板就會自動長出對應的分支與遞迴,讓你「照著框框填」就好。這正是設計配方最神奇的地方, 我們很快就會碰到。

步驟 6:函式本體(function body)

最後一步,才是真正寫邏輯:把樣板的 ... 填滿。有了前面五步,這一步往往水到渠成: 

import math

# 簽章:Radius -> float
def area(r: float) -> float:
    """計算半徑為 r 的圓面積。"""
    return math.pi * r ** 2

# 跑測試!
def test_area():
    assert area(0) == 0
    assert math.isclose(area(1), math.pi)
    assert math.isclose(area(2), 4 * math.pi)

test_area()
print("area 全部通過 ✅")

寫完立刻跑測試。綠燈,代表你的實作符合你一開始寫下的期待;紅燈,代表實作或期待有一個錯了—— 無論哪種,你都立刻知道,而不是等到三個月後上線才發現。

💡
配方的副產品:文件

走完六步,你會發現除了那一行 return,你還順手寫出了:資料的定義、輸入輸出的型別、 一句話說明、一組測試。這些不是額外負擔,而是免費的文件與保險。 三個月後回來看這個函式,你(或你的同事)一眼就懂它要幹嘛、怎麼用、怎麼驗證。 靠瞎猜寫的程式,從來不會留下這些。

資料驅動設計:讓函式跟著資料的形狀走

現在要講設計配方背後那個最深刻、也最實用的思想了。剛才 area 的樣板之所以平淡, 是因為它的輸入「沒有形狀」——就一個數。但大多數真實資料是有形狀的:它可能「是好幾種情況之一」、 可能「由好幾個欄位組成」、甚至「裡面還包著一個自己」。而設計配方的核心主張是:

函式的結構,應該映照資料的結構。資料長什麼形狀,函式的骨架就長什麼形狀。

這叫做資料驅動設計(data-driven design)。它聽起來抽象,但用起來具體得驚人: 你先看清楚「資料是怎麼定義的」,就能機械式地推導出「函式的樣板長怎樣」。 資料有幾種情況,函式就有幾個分支;資料裡包著自己,函式就得遞迴呼叫自己。

列舉型資料:當輸入只有「幾種固定選項」

列舉(enumeration)是最單純的「幾選一」:資料只可能是有限的、一一列得出來的幾個值。 紅綠燈只有紅、黃、綠;T 恤尺寸只有 S、M、L;一週只有七天。當你看到「就那幾種」,你就遇到了列舉。

用紅綠燈當例子,先寫資料定義:

# 資料定義
# 一個「號誌(TrafficLight)」是下列之一:
#   - "red"
#   - "yellow"
#   - "green"
# 解讀:代表紅綠燈當下的狀態

關鍵來了:資料有三種情況,樣板就有三個分支。這不是巧合,是資料驅動設計的直接結果。 我們先把樣板搭出來——每一種情況都給它一個 if 分支,內容先空著: 

# 樣板(列舉 → 每種情況一個分支)
def fn_for_light(tl: str):
    if tl == "red":
        return ...
    elif tl == "yellow":
        return ...
    elif tl == "green":
        return ...

有了這個骨架,寫一個「下一個燈號是什麼」的函式,就只是把每個 ... 填上答案。 走完整套配方: 

# 簽章:TrafficLight -> TrafficLight
def next_light(tl: str) -> str:
    """回傳號誌的下一個狀態(紅→綠→黃→紅 循環)。"""
    if tl == "red":
        return "green"
    elif tl == "yellow":
        return "red"
    elif tl == "green":
        return "yellow"
    else:
        raise ValueError(f"不是合法的號誌:{tl!r}")

# 範例與測試
assert next_light("red") == "green"
assert next_light("green") == "yellow"
assert next_light("yellow") == "red"
print("next_light 通過 ✅")

注意最後那個 else: raise。資料定義說「只有三種」,所以收到第四種值,代表呼叫方違約了—— 與其默默回傳錯的東西,不如大聲報錯。這是防禦性思考在配方裡的自然落點。

🔑
分支數 = 情況數

處理列舉資料時,有個簡單到不可思議的檢查法:數一數資料定義裡有幾種情況, 再數一數你的函式有幾個分支。兩個數字必須相等。少了一個分支,代表你漏處理了一種情況—— 這正是 bug 最愛躲的地方。配方把「別漏掉任何情況」這件事,變成了可以用手指頭數的機械檢查。

區間資料:當輸入是「分段的範圍」

列舉是「幾個離散的值」;區間(interval)則是「一段連續的範圍,被切成幾段」。 分數是 0 到 100 的連續範圍,但我們把它切成 A、B、C、D、F 五段;所得稅是連續的金額, 但被切成好幾個級距,每段套不同稅率。只要你看到「落在哪個範圍,就怎麼處理」,那就是區間資料。

用「分數轉等第」當主角。資料定義要把「邊界切在哪」講清楚:

# 資料定義
# 一個「分數(Score)」是 0 到 100 之間的整數。
# 它被切成幾個區間:
#   - 90 ~ 100 → "A"
#   - 80 ~ 89  → "B"
#   - 70 ~ 79  → "C"
#   - 60 ~ 69  → "D"
#   - 0  ~ 59  → "F"

和列舉一樣,有幾個區間,樣板就有幾個分支。差別只在分支的條件從「等於某個值」 變成「落在某個範圍」。樣板: 

# 樣板(區間 → 每個區間一個分支)
def fn_for_score(s: int):
    if 90 <= s <= 100:
        return ...
    elif 80 <= s:
        return ...
    elif 70 <= s:
        return ...
    elif 60 <= s:
        return ...
    else:
        return ...

填滿它:

# 簽章:Score -> str
def grade(s: int) -> str:
    """把 0~100 的分數轉成等第 A/B/C/D/F。"""
    if not (0 <= s <= 100):
        raise ValueError("分數必須在 0 到 100 之間")
    if s >= 90:
        return "A"
    elif s >= 80:
        return "B"
    elif s >= 70:
        return "C"
    elif s >= 60:
        return "D"
    else:
        return "F"

# 範例與測試:每段都測中間值,而且「一定要測邊界」!
assert grade(95) == "A"
assert grade(90) == "A"   # 邊界:剛好 90
assert grade(89) == "B"   # 邊界:剛好掉到 B
assert grade(60) == "D"   # 邊界:及格邊緣
assert grade(59) == "F"
assert grade(0)  == "F"
print("grade 通過 ✅")
⚠️
區間資料的頭號殺手:邊界

區間資料最容易出錯的地方,永遠是邊界:90 分到底算 A 還是 B?「差一」的 bug (把 >= 寫成 >)在這裡層出不窮。所以配方有一條鐵律: 每個區間的「邊界值」都必須有一個對應的測試。上面我們特地測了 90、89、60、59—— 這些剛好卡在邊界上的值,才是真正能抓出 bug 的測試。測 95 很安心,但它什麼也證明不了。

稅率級距是一模一樣的結構,只是「回傳值」從等第換成稅金。你可以把它當成同一個樣板的變奏: 

# 簽章:所得(float) -> 稅率(float)
# (簡化的累進級距示意,非真實稅法)
def tax_rate(income: float) -> float:
    """依所得落在哪個級距,回傳對應稅率。"""
    if income <= 540_000:
        return 0.05
    elif income <= 1_210_000:
        return 0.12
    elif income <= 2_420_000:
        return 0.20
    else:
        return 0.30

看出來了嗎?「分數轉等第」和「所得算稅率」表面是兩個完全不同的問題,但因為它們的資料形狀相同 (都是區間),它們的函式骨架也相同。這就是資料驅動設計給你的超能力: 認出資料的形狀,你就已經知道程式的一半長相。

複合資料:一個東西有很多欄位

真實世界的東西,常常「一個就包含好幾項資訊」。一個點有 x 和 y;一個人有姓名、生日、電話; 一筆訂單有商品、數量、金額。這種「把好幾個欄位綁成一個整體」的資料,叫做複合資料(compound data)。

在 Python 裡,表達複合資料最清楚的工具是 dataclass(資料類別)。 我們以平面上的一個點 Point 為例: 

from dataclasses import dataclass

# 資料定義
# 一個「點(Point)」由兩個欄位組成:
#   - x:float,水平座標
#   - y:float,垂直座標
@dataclass
class Point:
    x: float
    y: float

p = Point(3, 4)
print(p.x, p.y)   # 3 4
print(p)          # Point(x=3, y=4) ← dataclass 免費送你漂亮的印法

@dataclass 幫你自動產生建構式、__repr__(漂亮印法)、__eq__(可比較相等)等等, 讓你少寫一堆樣板碼。如果你想要的是「不可變」的版本,標準函式庫的 namedtuple 也是常見選擇: 

from collections import namedtuple
Point = namedtuple("Point", ["x", "y"])
p = Point(3, 4)
print(p.x, p.y)   # 3 4(但 p.x = 5 會報錯,因為它不可變)

現在是重點:複合資料的樣板長什麼樣?規則是——函式體要「碰到」這個資料的每一個欄位。 既然 Point 有 x 和 y 兩個欄位,樣板就先把這兩個欄位都「掏出來」備用: 

# 樣板(複合 → 觸及每一個欄位)
def fn_for_point(p: Point):
    return ... p.x ... p.y ...

用它來寫「一個點到原點的距離」,完整走配方:

import math

# 簽章:Point -> float
def distance_to_origin(p: Point) -> float:
    """計算點 p 到原點 (0, 0) 的距離。"""
    return math.sqrt(p.x ** 2 + p.y ** 2)

# 範例與測試
assert distance_to_origin(Point(0, 0)) == 0
assert distance_to_origin(Point(3, 4)) == 5      # 經典 3-4-5 直角三角形
assert math.isclose(distance_to_origin(Point(1, 1)), math.sqrt(2))
print("distance_to_origin 通過 ✅")
💡
樣板讓你「不漏欄位」

複合資料的樣板,本質上是一張檢查清單:把每個欄位都先列出來,提醒你「這些資訊你都可以用」。 很多 bug 來自「忘了用到某個欄位」——例如算距離時只用了 x 忘了 y。先寫樣板,把欄位攤開, 這類疏漏就難以發生。資料的形狀,再一次,替你擋掉了一整類錯誤。

自我參照資料與結構遞迴:清單與樹

我們來到了整套配方最美的部分。前面的資料,大小都是「固定的」:一個半徑、一個號誌、一個有兩欄的點。 但有些資料,大小是不固定的——一個清單可能有 0 個、3 個、或一百萬個元素; 一棵家族樹可能有任意多代。要怎麼定義「任意長」的資料?答案優雅得令人吃驚:讓它參照自己。

清單,是一個遞迴定義的東西

在第 3 章,我們見過遞迴函式;現在我們要見到遞迴定義的資料。一個「數字清單」是什麼? 經典的定義是這樣的(這也是 Lisp/Racket 看待清單的方式): 

# 資料定義(自我參照!)
# 一個「數字清單(ListOfNumber)」是下列之一:
#   - 空清單 []                              ← 基底情形(base case)
#   - 一個數字「接上」一個 ListOfNumber       ← 自我參照:[first] + rest
#
# 也就是說:一個清單,要嘛是空的,
# 要嘛是「第一個元素」加上「一個更短的清單」。

盯著這個定義看一會兒。它在用「清單」定義「清單」——這不是循環論證嗎?不是。 因為它有一個不參照自己的出口:空清單。每次「剝掉第一個元素」,剩下的清單就更短一點, 最終一定會剝到空清單為止。這個「自己包含一個更小的自己,加上一個終止情況」的結構, 正是遞迴的靈魂(回顧第 3 章的「鏡子裡的鏡子」)。

自我參照資料 → 遞迴樣板

既然資料定義有「兩種情況」(空 / 非空),樣板就有兩個分支;既然「非空」那種情況裡 包含一個更小的清單,樣板的對應位置就出現一個對自己的遞迴呼叫。 這個「資料參照自己 → 函式遞迴呼叫自己」的對應,就叫結構遞迴(structural recursion): 

# 樣板(自我參照資料 → 結構遞迴)
def fn_for_lon(lon: list):
    if lon == []:                       # 基底情形:空清單
        return ...
    else:
        first = lon[0]                  # 第一個元素(一個 Number)
        rest  = lon[1:]                 # 其餘清單(又是一個 ListOfNumber!)
        return ... first ... fn_for_lon(rest) ...   # ← 對「其餘清單」遞迴

這個樣板幾乎是「免費」的:你只是把資料定義「翻譯」成程式骨架。空清單那一支對應基底情形, 非空那一支把資料拆成 first 與 rest,而 rest 是個更小的清單, 所以理所當然地把它交給函式自己處理。剩下的,只是想清楚「怎麼把第一個元素的貢獻, 和其餘清單的結果合起來」。

把樣板填滿:sum_list 與 length

第一個:把清單裡的數字全部加起來。基底情形「空清單的總和是 0」;非空就是「第一個 + 其餘的總和」:

# 簽章:ListOfNumber -> Number
def sum_list(lon: list) -> float:
    """回傳清單中所有數字的總和。"""
    if lon == []:
        return 0                        # 空清單,總和是 0
    else:
        return lon[0] + sum_list(lon[1:])   # 第一個 + 其餘的總和

assert sum_list([]) == 0
assert sum_list([5]) == 5
assert sum_list([3, 1, 4]) == 8
print("sum_list 通過 ✅")

第二個:算清單長度。基底情形「空清單長度是 0」;非空就是「1 + 其餘的長度」。注意骨架幾乎一字不差:

# 簽章:ListOfNumber -> int
def length(lon: list) -> int:
    """回傳清單的元素個數。"""
    if lon == []:
        return 0                        # 空清單,長度是 0
    else:
        return 1 + length(lon[1:])      # 1(算進第一個)+ 其餘的長度

assert length([]) == 0
assert length([5]) == 1
assert length([3, 1, 4]) == 3
print("length 通過 ✅")

看看 sum_list 和 length 多像!它們的骨架完全相同,只有「第一個元素貢獻什麼」 和「基底回傳什麼」不一樣。這不是因為這兩題碰巧相似,而是因為它們處理的是同一種形狀的資料, 所以共用同一個樣板。一旦你內化了清單的結構遞迴樣板,「對清單做某件事」這類問題, 你幾乎是用填空的速度就能解掉。

🔑
結構遞迴的口訣

資料怎麼遞迴定義,函式就怎麼遞迴呼叫。資料定義裡每出現一次「自己」, 樣板裡就對應一次遞迴呼叫;資料定義裡的「終止情況」(空清單、空樹),就是遞迴的「基底情形」。 掌握這條對應,你就不必再害怕遞迴——它不是需要靈光一閃的魔法,而是跟著資料形狀走的機械步驟。

ℹ️
Racket 原版的清單長相

HtDP 處理的是真正的 Lisp 清單:用 (cons first rest) 把元素接到清單前面, 用 empty 表示空清單,用 (first lst) / (rest lst) 取出兩部分, 用 (empty? lst) 判斷是不是空的。我們的 lon[0] / lon[1:] / lon == [] 就是這套概念在 Python 的對應。底層機制不同(Python 的 list[1:] 會複製, Lisp 的 rest 不會),但「第一個 + 其餘」的結構完全一致。

測試先行的威力

你或許注意到,在配方裡,測試(步驟 4)排在實作(步驟 6)前面。這個順序是刻意的,而且威力驚人。 我們上一章已經見過 assert 與 pytest;這裡要談的是:為什麼「先寫測試」會讓你寫得更好。

  • 逼你想清楚規格。你沒辦法替一個自己都說不清楚的函式寫測試。當你能寫出 assert grade(90) == "A",代表你已經確定了「90 分到底算 A 還 B」這個曾經模糊的問題。 測試先行,其實是「需求釐清」先行。
  • 給你明確的完工定義。有了測試,「做完了沒?」這個問題有了客觀答案:全部綠燈,就是做完了。 不再憑感覺說「應該好了吧」。
  • 當場抓錯。實作一寫完立刻跑測試,bug 在它出生的那一刻就被逮到,而不是潛伏三個月。

而測試最被低估的價值,是它在你修改程式之後才顯現的。假設你想把 sum_list 改寫成 更快的迴圈版本: 

def sum_list(lon: list) -> float:
    """回傳清單中所有數字的總和(改寫成迴圈版)。"""
    total = 0
    for x in lon:
        total += x
    return total

# 一行都不用改,直接重跑原本的測試:
assert sum_list([]) == 0
assert sum_list([5]) == 5
assert sum_list([3, 1, 4]) == 8
print("改寫後仍然通過 ✅")

那組你一開始就寫好的測試,現在搖身一變成了安全網:它替你保證「新寫法和舊寫法行為一致, 我沒有在加速的同時偷偷弄壞東西」。這種「改完程式後重跑舊測試,確認沒改壞」的做法, 就是上一章提過的回歸測試(regression testing)。你寫過的每個範例,都不會白寫—— 它們會一直站在那裡,守著你的程式。

🔑
範例不是用過就丟,它們是資產

很多人手動試幾個例子、看到對了就把例子刪掉。配方告訴你:把每個你想得到的例子, 都變成一行不會消失的測試。它們是你重構時的膽量來源、是新人理解程式的活文件、 是你三個月後敢動這段程式的底氣。測試先行,不是多做一件事,而是把「反正都要試」這件事, 做成了永久有效的版本。

完整實戰:設計一個 BMI 分類器

紙上談兵夠了。我們拿一個五臟俱全的真實小問題,從零開始、一步不漏地把整套配方走一遍, 讓你看見它在「正常難度」的問題上怎麼運作。

給一個人的體重(公斤)與身高(公尺),算出 BMI 並分類成「過輕 / 正常 / 過重 / 肥胖」。

這個問題剛好同時用到我們學過的兩種形狀:一個人是複合資料(體重 + 身高), 而 BMI 分類是區間資料。配方會幫我們把它們自然地組合起來。

步驟 1:資料定義。先定義「一個人的身體資料」這個複合資料,以及分類結果這個列舉:

from dataclasses import dataclass

# 一個「身體資料(Body)」由兩個欄位組成:
#   - weight:float,體重,單位公斤,必須 > 0
#   - height:float,身高,單位公尺,必須 > 0
@dataclass
class Body:
    weight: float
    height: float

# 一個「BMI 分類(BMICategory)」是下列之一:
#   - "過輕"  (BMI < 18.5)
#   - "正常"  (18.5 <= BMI < 24)
#   - "過重"  (24   <= BMI < 27)
#   - "肥胖"  (BMI >= 27)
# (區間採台灣衛福部成人標準)

步驟 2 + 3:簽章與目的。我們其實要設計兩個函式——一個算 BMI 數值,一個做分類。 拆成兩個,每個都只做一件事(還記得「一句話講不完就該拆」嗎?): 

# bmi : Body -> float          計算一個人的 BMI 數值
# classify_bmi : float -> BMICategory   把 BMI 數值分類成等級

步驟 4:範例與測試。實作之前,先把期望的答案釘死。BMI 公式是 $\text{體重} / \text{身高}^2$:

import math

def test_bmi():
    # 170cm / 65kg → BMI ≈ 22.49,屬「正常」
    assert math.isclose(bmi(Body(65, 1.70)), 65 / 1.70**2)
    assert classify_bmi(17.0) == "過輕"
    assert classify_bmi(22.0) == "正常"
    assert classify_bmi(25.0) == "過重"
    assert classify_bmi(30.0) == "肥胖"
    # 邊界!這些卡在界線上的值,才是測試的重點
    assert classify_bmi(18.5) == "正常"   # 剛好 18.5 算正常
    assert classify_bmi(24.0) == "過重"   # 剛好 24 算過重
    assert classify_bmi(27.0) == "肥胖"   # 剛好 27 算肥胖

步驟 5:樣板。bmi 吃複合資料 → 觸及每個欄位;classify_bmi 吃區間資料 → 每個區間一個分支: 

def bmi(body: Body) -> float:
    return ... body.weight ... body.height ...   # 複合:碰每個欄位

def classify_bmi(value: float) -> str:
    if value < 18.5:                              # 區間:每段一個分支
        return ...
    elif value < 24:
        return ...
    elif value < 27:
        return ...
    else:
        return ...

步驟 6:本體,然後跑測試。把樣板填滿,加上輸入驗證:

def bmi(body: Body) -> float:
    """計算一個人的 BMI = 體重(kg) / 身高(m) 的平方。"""
    if body.weight <= 0 or body.height <= 0:
        raise ValueError("體重與身高都必須是正數")
    return body.weight / body.height ** 2

def classify_bmi(value: float) -> str:
    """把 BMI 數值分類成 過輕/正常/過重/肥胖。"""
    if value < 18.5:
        return "過輕"
    elif value < 24:
        return "正常"
    elif value < 27:
        return "過重"
    else:
        return "肥胖"

# 把兩個函式組起來用:
def report(body: Body) -> str:
    """產生一句 BMI 健康報告。"""
    value = bmi(body)
    return f"你的 BMI 是 {value:.1f},屬於「{classify_bmi(value)}」。"

test_bmi()
print("BMI 測試全部通過 ✅")
print(report(Body(65, 1.70)))   # 你的 BMI 是 22.5,屬於「正常」。

停下來看看你做了什麼。你沒有「想出」這個程式——你是推導出它的。資料定義決定了型別, 型別決定了簽章,資料的形狀決定了樣板,而你最後只是把樣板的空格填滿。一個複合資料的函式 加一個區間資料的函式,自然地組合成一個有用的程式。這就是有藍圖地蓋房子的感覺。

💡
把配方寫進註解,然後逐格填

實務上的好習慣:開新函式時,直接把六步當成註解骨架打進去—— # 簽章 / # 目的 / # 範例 / 樣板——再逐格填滿。寫完,那些註解剛好成了文件。 你會發現,當你卡住時,十之八九是因為某個步驟跳過了:不知道怎麼寫,通常是資料定義還沒想清楚。 回到卡住的那一步,把它補上,路就通了。

✏️ 練習 7.1 實作

用完整的六步配方,設計一個 is_leap_year(year),判斷某個西元年是不是閏年。 規則:能被 4 整除的是閏年,但能被 100 整除的不是,除非它也能被 400 整除。 (所以 2000 是閏年,1900 不是,2024 是。)請先寫簽章、目的、範例,再實作。

看解答

輸入是一個原子值(整數年份),所以樣板簡單。難點在把那三層「但是 / 除非」的規則翻成布林邏輯。 範例一定要涵蓋四種代表情況:能被 400(2000)、能被 100 不能被 400(1900)、能被 4 不能被 100(2024)、不能被 4(2023)。

# 資料定義:一個「年份(Year)」是一個正整數
# 簽章:Year -> bool
def is_leap_year(year: int) -> bool:
    """判斷 year 是否為閏年(能被4整除,但能被100不被400整除者除外)。"""
    return year % 4 == 0 and (year % 100 != 0 or year % 400 == 0)

# 範例與測試(四種代表情況都要測)
assert is_leap_year(2000) is True    # 能被 400 → 閏年
assert is_leap_year(1900) is False   # 能被 100 不能被 400 → 平年
assert is_leap_year(2024) is True    # 能被 4 不能被 100 → 閏年
assert is_leap_year(2023) is False   # 不能被 4 → 平年
print("通過 ✅")

那一行布林運算式不好讀?把它拆成註解版會更清楚,效果完全一樣: 「能被 4」and(「不能被 100」or「能被 400」)。 用配方逼自己先寫好四個範例,你就不會漏掉 1900 這種陰險的反例。

✏️ 練習 7.2 實作

這是一個列舉練習。設計一個 letter_to_gpa(letter),把等第字母轉成 GPA 積點: "A"→4.0、"B"→3.0、"C"→2.0、"D"→1.0、"F"→0.0。 先寫資料定義(列出所有合法的等第),再依「幾種情況 = 幾個分支」搭出樣板,最後填滿。 遇到不合法的字母要報錯。

看解答

資料定義有五種情況,樣板就該有五個分支(加一個 else 擋非法輸入)。 用 if/elif 是最貼近配方的寫法;熟練後也可以改用字典查表,但分支版更能看出「列舉樣板」的結構。

# 資料定義:一個「等第(Letter)」是下列之一:"A"/"B"/"C"/"D"/"F"
# 簽章:Letter -> float
def letter_to_gpa(letter: str) -> float:
    """把等第字母轉成 GPA 積點(A=4.0 ... F=0.0)。"""
    if letter == "A":
        return 4.0
    elif letter == "B":
        return 3.0
    elif letter == "C":
        return 2.0
    elif letter == "D":
        return 1.0
    elif letter == "F":
        return 0.0
    else:
        raise ValueError(f"不是合法的等第:{letter!r}")

assert letter_to_gpa("A") == 4.0
assert letter_to_gpa("F") == 0.0
print("通過 ✅")

# 進階寫法:用字典查表(資料即程式)
GPA = {"A": 4.0, "B": 3.0, "C": 2.0, "D": 1.0, "F": 0.0}
def letter_to_gpa_v2(letter: str) -> float:
    if letter not in GPA:
        raise ValueError(f"不是合法的等第:{letter!r}")
    return GPA[letter]

兩種寫法都對。分支版讓你看清「列舉樣板」的骨架;字典版更精簡,適合選項很多時。 這正是配方的好處:先用樣板把邏輯走通,再決定要不要換更漂亮的寫法。

✏️ 練習 7.3 實作

這是一個複合資料練習。沿用本章的 Point(有 x、y 兩欄), 設計 distance(p1, p2),計算平面上兩個點之間的距離。 (公式:$\sqrt{(x_1-x_2)^2 + (y_1-y_2)^2}$。)函式吃兩個複合資料—— 想想樣板該怎麼觸及這四個欄位?先寫簽章、目的、範例。

看解答

當函式吃兩個複合資料時,樣板要觸及兩個資料的每個欄位: p1.x, p1.y, p2.x, p2.y 四個都要用到。範例別忘了「兩點相同 → 距離 0」這個邊界, 以及經典的 3-4-5。

import math
from dataclasses import dataclass

@dataclass
class Point:
    x: float
    y: float

# 簽章:Point Point -> float
def distance(p1: Point, p2: Point) -> float:
    """計算平面上兩點 p1、p2 之間的直線距離。"""
    return math.sqrt((p1.x - p2.x) ** 2 + (p1.y - p2.y) ** 2)

assert distance(Point(0, 0), Point(0, 0)) == 0      # 邊界:同一點
assert distance(Point(0, 0), Point(3, 4)) == 5      # 經典 3-4-5
assert math.isclose(distance(Point(1, 1), Point(2, 2)), math.sqrt(2))
print("通過 ✅")

注意 distance(p, Point(0,0)) 其實就是本章的 distance_to_origin(p)—— 後者是前者的特例。看出這種關係,代表你開始「用資料的形狀」思考了。

✏️ 練習 7.4 遞迴

這是一個自我參照資料練習。為「數字清單」設計一個 count_positive(lon), 回傳清單中正數(> 0)的個數。請套用本章的清單遞迴樣板:先處理基底情形(空清單), 再處理「第一個 + 其餘清單」。先寫簽章、目的、範例(務必包含空清單!),再實作。

看解答

直接套清單樣板:空清單 → 0 個正數;非空 → 「第一個元素是不是正的(是就算 1,不是算 0)」 加上「其餘清單裡的正數個數」。基底情形與遞迴情形,剛好對應資料定義的兩種情況。

# 簽章:ListOfNumber -> int
def count_positive(lon: list) -> int:
    """回傳清單中大於 0 的數字個數。"""
    if lon == []:                          # 基底情形
        return 0
    else:
        first_count = 1 if lon[0] > 0 else 0    # 第一個的貢獻
        return first_count + count_positive(lon[1:])  # + 其餘清單的結果

assert count_positive([]) == 0             # 邊界:空清單
assert count_positive([-1, -2, -3]) == 0   # 全負
assert count_positive([1, -2, 3, 0, 5]) == 3   # 0 不算正數!
print("通過 ✅")

把它和本章的 sum_list、length 並排看:骨架一模一樣,只有「第一個元素貢獻什麼」 和「基底回傳什麼」不同。這就是結構遞迴的威力——同一個樣板,解掉一整類「對清單做某事」的問題。 (當然,實務上你會直接寫 sum(1 for x in lon if x > 0);但親手遞迴一次,你才真懂它。)

✏️ 練習 7.5 挑戰

進階的自我參照練習:二元樹。一棵二元樹要嘛是空的(None), 要嘛是一個節點,含一個值與左右兩棵子樹(子樹又各自是二元樹)。資料定義給你:

from dataclasses import dataclass
from typing import Optional

# 資料定義(自我參照,而且參照「兩次」!)
# 一棵「二元樹(BinaryTree)」是下列之一:
#   - None                          ← 空樹(基底情形)
#   - Node(value, left, right)      ← left、right 各自又是 BinaryTree
@dataclass
class Node:
    value: int
    left: "Optional[Node]"    # 一棵 BinaryTree
    right: "Optional[Node]"   # 一棵 BinaryTree

請設計兩個函式:count_nodes(tree) 回傳節點總數; depth(tree) 回傳樹的深度(空樹深度 0,只有根的樹深度 1)。 提示:資料定義裡「自己」出現了兩次(left 與 right),所以樣板裡會有兩個遞迴呼叫。 先各自寫簽章、目的、範例,再實作。

看解答

樹的樣板:基底情形是 None;遞迴情形要觸及 value,並對 left、right 各遞迴一次,再把兩邊的結果合起來。count_nodes 把兩邊相加(再 +1 算自己); depth 取兩邊較深者(再 +1 算這一層)。

# 樣板(樹 → 兩個遞迴呼叫)
# def fn_for_tree(t):
#     if t is None:
#         return ...
#     else:
#         return ... t.value ... fn_for_tree(t.left) ... fn_for_tree(t.right) ...

# 簽章:BinaryTree -> int
def count_nodes(tree) -> int:
    """回傳二元樹的節點總數。"""
    if tree is None:
        return 0
    else:
        return 1 + count_nodes(tree.left) + count_nodes(tree.right)

# 簽章:BinaryTree -> int
def depth(tree) -> int:
    """回傳二元樹的深度(空樹為 0)。"""
    if tree is None:
        return 0
    else:
        return 1 + max(depth(tree.left), depth(tree.right))

# 建一棵測試用的樹:
#        1
#       / \
#      2   3
#         /
#        4
t = Node(1, Node(2, None, None),
            Node(3, Node(4, None, None), None))

assert count_nodes(None) == 0      # 邊界:空樹
assert count_nodes(t) == 4
assert depth(None) == 0            # 邊界:空樹
assert depth(t) == 3               # 1 → 3 → 4 三層
print("通過 ✅")

再次驗證那條口訣:資料定義裡「自己」出現幾次,樣板裡就遞迴幾次。 清單只參照自己一次(rest),所以一個遞迴呼叫;二元樹參照自己兩次(left、right),所以兩個。 你完全不需要「想像整棵樹怎麼被走遍」——只要相信「子樹會自己處理好自己」, 照著資料形狀填樣板,遞迴就對了。這就是結構遞迴最解放人心的地方。

把這一章串起來

我們從「蓋房子要先有藍圖」出發,學會了一套讓寫程式不再靠瞎猜的紀律。回顧這趟旅程:

  • 設計配方把寫一個函式拆成六步:資料定義 → 簽章 → 目的 → 範例 → 樣板 → 本體。 它的承諾是:你永遠知道下一步該做什麼。
  • 資料驅動設計是核心思想:函式的結構,映照資料的結構。看懂資料的形狀,你就推導得出程式的骨架。
  • 四種資料形狀,四種樣板:原子(直接用)、列舉/區間(幾種情況=幾個分支)、 複合(觸及每個欄位)、自我參照(遞迴呼叫自己)。
  • 結構遞迴讓遞迴不再神祕:資料怎麼遞迴定義,函式就怎麼遞迴呼叫。清單、樹,都是同一招。
  • 測試先行逼你想清規格、給你完工定義、當場抓錯,還順手織成一張回歸測試的安全網。

最重要的一句話,請你帶走:好程式是「設計」出來的,不是「想」出來的。 當你下次面對一個陌生問題、不知從何下手時,別急著敲鍵盤——先問「資料長什麼形狀?」, 然後讓配方帶你一步步往前走。你會驚訝地發現,大部分「不知道怎麼寫」的卡關, 其實是「還沒把資料想清楚」。把藍圖畫好,砌磚自然順手。
