物件導向設計

當程式越長越大,函式與資料散落一地,你會開始懷念真實世界的秩序——一切都由「東西」組成,每個東西都知道自己能做什麼。這就是物件導向的起點。

  • 用魔術方法(dunder methods)讓你的物件像內建型別一樣自然

當程式長大:為什麼需要 OOP

想像你在寫一個遊戲。一開始只有一隻狗,你用幾個變數記住牠: 

dog_name = "旺財"
dog_energy = 100
dog_hungry = False

def dog_bark(name):
    print(f"{name}: 汪汪!")

def dog_eat(name):
    print(f"{name} 吃飽了")

還行。但接著你要加第二隻狗、第三隻狗,還有貓、有 NPC、有玩家。 每多一隻動物,就多出一票 xxx_name、xxx_energy、xxx_hungry 變數, 還有一堆得小心翼翼把「正確的名字」傳進去的函式。資料散落各處, 行為又住在另一邊,兩者之間只靠你「記得要配對」維繫——這正是 bug 的溫床。

問題的根源是:在現實裡,「一隻狗」是一個完整的東西。 牠的名字、體力、餓不餓,還有牠會叫、會吃,這些本來就該是「同一包」。 我們的程式卻把它拆成了散裝的變數加散裝的函式。當東西一多,這種散裝就管不動了。

物件導向程式設計(Object-Oriented Programming, OOP)提出一個樸素卻威力強大的主張: 把資料(它的狀態)和操作這些資料的行為(它的方法) 綁成一個自給自足的單位,叫做物件(object)。 一隻狗就是一個物件,牠自己帶著自己的名字與體力,你叫牠 bark(),牠自己會處理。 程式不再是「一堆變數 + 一堆函式」,而是「一群會互相合作的物件」—— 就像真實世界,是由一個個各司其職的東西組成的。

🔑
OOP 的一句話本質

把資料和操作那份資料的行為包在一起,變成一個會「自己照顧自己」的物件。 當系統的複雜度上升,這種「就近綁定」會讓程式更好理解、更好修改、更難寫錯。

類別與物件:藍圖 vs 實體

我們不可能一隻一隻手工打造狗。我們需要一張藍圖:上面寫好「一隻狗應該有哪些資料、會做哪些事」, 然後照著藍圖,要幾隻就蓋幾隻。這張藍圖,就是類別(class); 照藍圖蓋出來的、活生生的個體,就是物件(object),也叫實例(instance)。

類別是「狗」這個概念;物件是「旺財」這隻具體的狗。 一張藍圖可以蓋出無數棟房子,一個類別也可以產生無數個物件, 每個物件各有各的資料,但共用同一套行為。來看 Python 怎麼寫: 

class Dog:
    def __init__(self, name, energy=100):   # 建構子:蓋一隻新狗時自動呼叫
        self.name = name        # 屬性:這隻狗的名字
        self.energy = energy    # 屬性:這隻狗的體力

    def bark(self):             # 方法:這隻狗會做的事
        print(f"{self.name}: 汪汪!")

    def play(self):
        self.energy -= 10       # 玩耍會消耗自己的體力
        print(f"{self.name} 玩得很開心,剩下 {self.energy} 體力")


# 照藍圖蓋出三隻不同的狗(三個 instance)
d1 = Dog("旺財")
d2 = Dog("來福", energy=80)
d3 = Dog("小黑", energy=50)

d1.bark()    # 旺財: 汪汪!
d2.play()    # 來福 玩得很開心,剩下 70 體力
print(d1.energy, d2.energy)   # 100 70  ← 各自獨立,互不干擾

幾個一定要搞懂的字眼:

  • class Dog: ——宣告一個類別(藍圖)。慣例上類別名用大寫開頭的駝峰式(CamelCase)。
  • __init__ ——建構子(constructor),更精確地說是「初始化方法」。每次你寫 Dog(...) 蓋一隻新狗,Python 就自動呼叫它,讓你把這隻狗的初始狀態設好。
  • self ——指向「這一個物件自己」。self.name 就是「我這隻狗的名字」。它是方法的第一個參數,Python 會自動把「是誰呼叫的」塞進去。
  • 屬性(attribute) ——掛在物件身上的資料,如 self.name、self.energy。
  • 方法(method) ——定義在類別裡的函式,如 bark()、play()。它能透過 self 存取自己的屬性。
ℹ️
self 不是關鍵字,是約定

當你寫 d1.bark(),Python 在背後其實是執行 Dog.bark(d1)—— 把 d1 當作第一個參數傳進去,而那個參數的名字,慣例上就叫 self。 理論上你可以取別的名字,但請不要:全世界的 Python 程式都用 self,違反它只會讓人讀不懂。

💡
類別屬性 vs 實例屬性

寫在 __init__ 裡、掛在 self 上的是實例屬性(每個物件一份)。 直接寫在 class 區塊裡、不靠 self 的,是類別屬性(所有物件共用一份), 適合放「所有狗都一樣」的常數,例如 species = "犬"。初學者常把可變的東西(如 list) 誤設成類別屬性,結果「一隻狗加了玩具,全部的狗都有了」——這是經典坑,要小心。

封裝:把內部藏好,只露出按鈕

回想第 1 章那台車:你開車只需要方向盤、油門、煞車,引擎多複雜都被藏在蓋子底下。 這種「把內部狀態藏起來、只透過明確的介面操作」的設計,就是物件導向的第一根支柱—— 封裝(encapsulation)。

為什麼要藏?舉個會痛的例子。假設我們做一個溫度感測器物件,它存著攝氏溫度。 物理上,溫度不可能低於絕對零度(約 -273.15°C)。如果我們讓外界直接改這個值: 

class Thermometer:
    def __init__(self, celsius):
        self.celsius = celsius     # 完全公開,誰都能改

t = Thermometer(25)
t.celsius = -500    # 😱 一個荒謬的溫度,但程式照單全收

現在這個物件處於一個不可能的狀態,後面任何依賴它的計算都會出錯, 而且 bug 發生的「現場」(設成 -500 的那行)和「爆炸」的地方可能離得很遠,難追到死。 我們需要一個守門員,確保「不管誰來改,溫度永遠合理」。這個「永遠該成立的條件」, 有個正式名字叫不變式(invariant)。

Python 的慣例:底線與 name mangling

和 Java、C++ 不同,Python 沒有真正強制的 private。它走的是「我們都是成年人」的文化路線, 用命名慣例來表達意圖:

  • 單底線 _x:一個君子之約。意思是「這是內部用的,請別從外面碰」。Python 不會阻止你,但你碰了就是你的錯。
  • 雙底線 __x:會觸發名稱改寫(name mangling)——Python 偷偷把 __x 改名成 _ClassName__x,讓它在子類別裡不容易被意外覆蓋。它不是為了「保密」,主要是為了避免命名衝突。

property:讓「方法」假裝成「屬性」

如果只是把 celsius 改名成 _celsius 再寫 get_celsius() / set_celsius(), 用起來會很醜(到處 t.set_celsius(30))。Python 給了優雅的解法:@property。 它讓你「用存取屬性的語法,執行方法的邏輯」——外界感覺像在讀寫一個普通屬性, 背後卻偷偷跑了你的檢查程式: 

class Thermometer:
    def __init__(self, celsius):
        self.celsius = celsius          # 注意:這會走下面的 setter!

    @property
    def celsius(self):                  # 讀取 t.celsius 時呼叫
        return self._celsius

    @celsius.setter
    def celsius(self, value):           # 寫入 t.celsius = ... 時呼叫
        if value < -273.15:             # 守門:維護不變式
            raise ValueError("溫度不可低於絕對零度")
        self._celsius = value

    @property
    def fahrenheit(self):               # 唯讀的「衍生屬性」,沒有 setter
        return self._celsius * 9 / 5 + 32


t = Thermometer(25)
print(t.celsius)      # 25      ← 看起來像讀屬性,其實跑了 getter
print(t.fahrenheit)   # 77.0    ← 即時換算,不必另存一份
t.celsius = 30        # 看起來像寫屬性,其實跑了 setter(通過檢查)
t.celsius = -500      # ❌ ValueError:溫度不可低於絕對零度

漂亮的地方在於:外界的寫法完全沒變(還是 t.celsius = 30), 但我們已經偷偷裝上了守門員。這就是封裝的精髓—— 把「怎麼存、怎麼檢查」的細節藏在介面背後,外界只管用。

🔑
為什麼封裝這麼重要

封裝帶來兩個關鍵好處。其一,保護不變式:所有改動都必須經過守門員,物件永遠處於合法狀態。 其二,降低耦合(coupling):外界只依賴「介面」,不依賴「實作」。 哪天你想把攝氏改成內部存克耳文,只要 getter/setter 換算一下, 用到這個類別的幾千行程式碼一行都不用改。介面是承諾,實作是自由。

✏️ 練習 8.1 實作

設計一個 BankAccount 類別,維護一個重要的不變式:餘額永遠不能是負數。 它要有 deposit(amount)(存款)與 withdraw(amount)(提款)兩個方法。 要求:餘額用封裝藏好(不能讓外界直接 account.balance = -999); 存入金額必須為正;提款若超過餘額要拒絕並報錯。

看解答

把餘額存成 _balance,只開放一個唯讀的 balance property(沒有 setter,外界改不了), 所有變動都得走 deposit / withdraw,在裡面把關:

class BankAccount:
    def __init__(self, owner, balance=0):
        self.owner = owner
        self._balance = 0
        if balance > 0:
            self.deposit(balance)      # 連初始餘額都走同一道關

    @property
    def balance(self):                 # 唯讀:外界只能看,不能直接設
        return self._balance

    def deposit(self, amount):
        if amount <= 0:
            raise ValueError("存款金額必須為正")
        self._balance += amount

    def withdraw(self, amount):
        if amount <= 0:
            raise ValueError("提款金額必須為正")
        if amount > self._balance:     # 維護「餘額不為負」的不變式
            raise ValueError("餘額不足")
        self._balance -= amount


acc = BankAccount("小明", 100)
acc.deposit(50)
acc.withdraw(120)
print(acc.balance)        # 30
# acc.balance = 999999    # ❌ AttributeError:沒有 setter,改不了
acc.withdraw(999)         # ❌ ValueError:餘額不足

關鍵心法:讓非法狀態無法被表示。只要所有入口都把關, 這個物件就永遠不可能餘額為負——你不必在程式各處重複檢查,因為物件自己保證了。

抽象:介面與實作的分離

封裝藏的是「一個物件」的內部;抽象(abstraction)則是更高一層的思考: 當我們設計一群類別時,先想清楚「它們能做什麼(介面)」, 把「怎麼做(實作)」留到各自去填。抽象讓我們能對一整族物件「一視同仁」。

舉例:不管是圓形、矩形還是三角形,它們都「是一種圖形」,而且都「能算面積」。 「能算面積」是所有圖形共同的契約,至於圓形用 $\pi r^2$、矩形用 $w \times h$——那是各自的事。 Python 用 abc 模組(Abstract Base Class,抽象基底類別)來表達這種契約: 

from abc import ABC, abstractmethod

class Shape(ABC):                  # 繼承 ABC,宣告這是抽象類別
    @abstractmethod
    def area(self):                # 抽象方法:只規定「要有」,不給實作
        ...

    @abstractmethod
    def perimeter(self):
        ...


# s = Shape()   # ❌ TypeError:不能實例化抽象類別(它只是契約)

Shape 是一個抽象基底類別:它規定「任何想當圖形的人, 都必須提供 area() 和 perimeter()」,但它自己不能被實例化—— 因為「一個沒有具體形狀的圖形」本身沒有意義。它是一份規格,不是一個東西。 誰沒實作完所有抽象方法,誰就也不能被實例化。這把「忘了實作」的錯誤, 從「半夜執行時才爆炸」提早到了「一建立物件就報錯」。

ℹ️
抽象 ≠ 封裝

兩者常被搞混。一個記法:封裝是「隱藏」(把內部細節藏在介面後面); 抽象是「忽略」(只關注「能做什麼」這個本質,忽略「怎麼做」的差異)。 封裝是手段,抽象是目的——我們封裝細節,正是為了能在更高的抽象層次上思考。

繼承:重用與特化

現在讓 Shape 真正開枝散葉。圓形和矩形都「是一種」圖形, 它們可以繼承(inheritance) Shape,自動獲得它的身分與契約, 再各自把抽象方法填上具體實作。被繼承的叫父類別(superclass / base class), 繼承者叫子類別(subclass / derived class)。 

import math

class Circle(Shape):                   # Circle 是一種 Shape
    def __init__(self, radius):
        self.radius = radius

    def area(self):
        return math.pi * self.radius ** 2

    def perimeter(self):
        return 2 * math.pi * self.radius


class Rectangle(Shape):                # Rectangle 也是一種 Shape
    def __init__(self, width, height):
        self.width = width
        self.height = height

    def area(self):
        return self.width * self.height

    def perimeter(self):
        return 2 * (self.width + self.height)


c = Circle(5)
r = Rectangle(3, 4)
print(round(c.area(), 2))   # 78.54
print(r.area())             # 12

super():站在父類別的肩膀上

繼承的重點是重用。如果子類別的建構子想先做父類別已經寫好的初始化, 再加上自己的部分,就用 super() 呼叫父類別的方法。 假設我們有個 Animal,以及一個「會額外記住品種」的 Dog: 

class Animal:
    def __init__(self, name):
        self.name = name

    def describe(self):
        return f"我是 {self.name}"


class Dog(Animal):
    def __init__(self, name, breed):
        super().__init__(name)      # 先做 Animal 的初始化(設好 name)
        self.breed = breed          # 再補上 Dog 特有的屬性

    def describe(self):             # 方法覆寫(override)
        base = super().describe()   # 重用父類別的描述
        return f"{base},品種是 {self.breed}"


d = Dog("旺財", "柴犬")
print(d.describe())   # 我是 旺財,品種是 柴犬

這裡示範了兩件事。方法覆寫(override):子類別定義一個和父類別同名的方法, 就會「蓋掉」父類別的版本。用 super() 重用:覆寫時常常不是要完全取代, 而是「在父類別的基礎上加東西」,super() 讓你呼叫被蓋掉的那個原版。

方法解析順序(MRO)簡介

當 Python 看到 d.describe(),它怎麼知道該用哪一版?它會沿著一條叫 方法解析順序(Method Resolution Order, MRO)的清單, 從物件自己的類別開始,一層層往父類別找,用第一個找到的版本。 單純的單一繼承很直觀,但 Python 允許多重繼承(一個類別有多個父類別), 這時順序就微妙了。你可以用 __mro__ 看這條清單: 

print(Dog.__mro__)
# (<class 'Dog'>, <class 'Animal'>, <class 'object'>)
# 找方法時:先看 Dog,沒有再看 Animal,最後看所有類別的祖先 object
⚠️
多重繼承與菱形問題

若 D 同時繼承 B 和 C,而 B、C 又都繼承 A, 就形成一個「菱形」。A 該被走訪一次還是兩次?方法該用誰的? Python 用一套叫 C3 線性化的演算法算出一個一致、合理的 MRO 來解決, 但多重繼承本身容易讓人困惑、容易出錯。初學階段請盡量避開它, 優先用單一繼承,或用我們稍後會講的「組合」。

多型:同一個介面,不同的行為

繼承鋪好了路,多型(polymorphism)才是真正的回報。 多型的字面意思是「多種形態」:同一個呼叫,作用在不同物件上,會產生各自合適的行為。 看看這段把所有圖形「一視同仁」的程式: 

shapes = [Circle(5), Rectangle(3, 4), Circle(1)]

total = 0
for s in shapes:
    total += s.area()      # 不管 s 是圓是方,都呼叫 .area()
                           # Python 自動跑「那個物件真正的」area()
print(round(total, 2))     # 95.69

注意那個 for 迴圈:它完全不在乎 s 到底是圓還是方。 它只說「我要你的面積」,每個物件自己回答。今天你新增一個 Triangle, 這段加總的程式碼一個字都不用改——這就是多型帶來的「對擴充開放、對修改封閉」的威力。

鴨子型別(duck typing):Python 的多型哲學

這裡有個 Python 獨有的味道。在 C++/Java 這類靜態型別語言裡, 要享受多型,通常 s 必須被宣告成 Shape 型別,而且 area() 得是「虛擬函式」 (稍後 C++ 對照會講)。但 Python 不檢查型別,它信奉鴨子型別(duck typing):

「如果牠走起來像鴨子、叫起來像鴨子,那牠就是鴨子。」

Python 不問 s 「你是不是 Shape 的子類別?」,它只問「你有沒有 area() 這個方法?」。 只要有,就能用。s 甚至根本不必繼承 Shape——任何長得「像圖形」的物件都行: 

class Donut:                       # 完全沒繼承 Shape!
    def area(self):                # 但它「會算面積」,長得像鴨子
        return 10.0

things = [Circle(5), Donut()]      # 混在一起也無妨
print(sum(t.area() for t in things))   # 88.54 —— 照樣運作

這讓 Python 的多型極為彈性,但代價是:錯誤要到執行時才會浮現 (如果某個物件其實沒有 area(),跑到那一行才 AttributeError)。 靜態語言則是在編譯時就攔下這類錯——彈性與安全,是一組永恆的取捨。

四大支柱速記

封裝:把資料與行為包起來,藏住內部。抽象:只暴露「做什麼」,隱藏「怎麼做」。 繼承:子類別重用並特化父類別。多型:同一介面、不同物件、各自的行為。 四者環環相扣:封裝與抽象定義出乾淨的介面,繼承建立「是一種」的關係,多型則在這個關係上收割彈性。

✏️ 練習 8.2 實作

延續 Shape 階層,新增一個 Triangle(三角形,以底 base 與高 height 表示), 並寫一個函式 total_area(shapes),接受一個圖形串列、回傳它們的總面積。 要證明你的函式對「任何圖形」都通用——包含你剛加的三角形。

看解答

新增 Triangle 時,只要繼承 Shape 並實作 area() / perimeter() (這裡為簡化,假設是直角三角形以方便算周長)。重點是 total_area 靠多型, 對既有與新增的圖形一律適用,完全不用改:

class Triangle(Shape):
    def __init__(self, base, height):
        self.base = base
        self.height = height

    def area(self):
        return self.base * self.height / 2

    def perimeter(self):
        hyp = (self.base ** 2 + self.height ** 2) ** 0.5
        return self.base + self.height + hyp


def total_area(shapes):
    return sum(s.area() for s in shapes)   # 多型:對每個 s 呼叫它自己的 area()


shapes = [Circle(2), Rectangle(3, 4), Triangle(6, 8)]
print(round(total_area(shapes), 2))   # 36.57

體會一下:total_area 是在 Triangle 還不存在時就寫好的, 卻能無痛吃下新圖形。這就是多型「對擴充開放」的具體樣貌。

組合優於繼承

繼承很迷人,初學者常會「繼承上癮」——看到任何相似性就想用繼承串起來,蓋出一座深不見底的階層塔。 但業界有句廣為流傳的箴言:組合優於繼承(composition over inheritance)。 為什麼?因為繼承有它的陷阱。

繼承表達的是「是一種(is-a)」的關係,而且它把子類別和父類別緊緊綁死。 改一下父類別,可能莫名其妙弄壞所有子類別——這叫脆弱基底類別問題(fragile base class)。 經典的反例:有人讓 Stack 繼承 list,想重用 list 的功能。 結果呢?list 的 append、insert、pop(0) 全都漏了出去, 外界可以從堆疊的任何位置塞東西、拿東西——它根本不再是個堆疊了。子類別繼承了父類別全部的能力,連你不想要的也一併繼承。

組合(composition)表達的則是「有一個(has-a)」的關係: 一個物件「擁有」另一個物件當零件,透過呼叫它來借用功能,而不是「變成」它。 用組合重寫堆疊: 

# ❌ 繼承:Stack「是一種」list —— 連不該有的 insert/append 都暴露了
class BadStack(list):
    def push(self, x):
        self.append(x)
    # 慘:外界還能 s.insert(0, x)、s[2] = 9 ... 堆疊的規矩被破壞

# ✅ 組合:Stack「有一個」list 當內部容器,只露出堆疊該有的操作
class Stack:
    def __init__(self):
        self._items = []            # 借用 list 的能力,但藏在內部

    def push(self, x):
        self._items.append(x)

    def pop(self):
        return self._items.pop()    # 只能從頂端拿

    def peek(self):
        return self._items[-1]

    def is_empty(self):
        return len(self._items) == 0


s = Stack()
s.push(1); s.push(2)
print(s.pop())        # 2
# s.insert(0, 99)     # ❌ AttributeError:Stack 根本沒有這個方法,規矩守住了
💡
何時用繼承,何時用組合?

問自己:這真的是「是一種」嗎?貓是一種動物——繼承合理。 但汽車不是一種引擎,汽車有一個引擎——這該用組合。 當你只是想「重用某段程式碼」,那幾乎總是組合的工作,不是繼承。 繼承請留給真正的「is-a 而且想要多型」的場合。階層越淺越好,兩三層通常就夠了。

✏️ 練習 8.3 思考

有人這樣設計員工系統:Manager(經理) 繼承 Engineer(工程師), 理由是「經理也要會寫程式,所以重用工程師的 write_code()」。 請說明這個設計為什麼有問題,並用組合重構它。

看解答

問題:這不是真正的「is-a」。經理不是一種工程師(一個從不寫程式的純管理職經理也存在)。 用繼承會逼著每個經理都「是工程師」,而且哪天 Engineer 改了介面, Manager 可能莫名其妙壞掉。正確的關係是:經理擁有一些可被指派的技能/角色。 用組合把「寫程式」這個能力抽成一個零件:

class CodingSkill:
    def write_code(self):
        return "寫了一段程式"


class Engineer:
    def __init__(self, name):
        self.name = name
        self.coding = CodingSkill()       # 有一個寫程式技能

    def work(self):
        return f"{self.name} {self.coding.write_code()}"


class Manager:
    def __init__(self, name, can_code=False):
        self.name = name
        self.reports = []                 # 帶領的部屬
        self.coding = CodingSkill() if can_code else None  # 經理「可以」會寫程式,也可以不會

    def work(self):
        actions = [f"{self.name} 安排了 {len(self.reports)} 位部屬的工作"]
        if self.coding:                   # 有這個技能才用
            actions.append(self.coding.write_code())
        return ";".join(actions)

現在「寫程式」是一個可插拔的零件,經理要不要會寫程式變成一個選擇, 而不是被繼承強迫。Engineer 和 Manager 也不再彼此綁死。

魔術方法:讓物件融入語言

到目前為止,我們的物件還是「外來客」:你不能用 + 把它們相加, print 它會吐出一串難看的 <__main__.Vector object at 0x7f...>, == 比較也常常不如預期。Python 提供一套魔術方法(magic methods), 也叫 dunder methods(double underscore,雙底線), 讓你的自訂物件能「接上」語言的內建語法,用起來像原生型別一樣自然。

我們做一個 2D 向量,把幾個最常用的 dunder 都裝上:

class Vector:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __repr__(self):                 # 給「程式設計師」看的、明確的表示法
        return f"Vector({self.x}, {self.y})"

    def __str__(self):                  # 給「使用者」看的、友善的字串(print 用這個)
        return f"({self.x}, {self.y})"

    def __eq__(self, other):            # 定義 == 的意義:內容相同就相等
        return self.x == other.x and self.y == other.y

    def __add__(self, other):           # 定義 + 的意義:對應分量相加
        return Vector(self.x + other.x, self.y + other.y)

    def __len__(self):                  # 定義 len() —— 這裡示範回傳維度數
        return 2


a = Vector(1, 2)
b = Vector(3, 4)
print(a + b)            # (4, 6)        ← 觸發 __add__,再用 __str__ 印
print(a == Vector(1, 2)) # True         ← 觸發 __eq__,比的是內容
print(repr(a))          # Vector(1, 2)  ← 觸發 __repr__
print([a, b])           # [Vector(1, 2), Vector(3, 4)] ← 容器裡的元素用 __repr__
魔術方法你寫的語法意義
__init__Vector(1, 2)建立並初始化物件
__repr__repr(v)、在容器裡顯示給工程師看的精確表示(求歧義最少)
__str__str(v)、print(v)給人看的友善字串(沒寫就退回 __repr__)
__eq__v == w相等的定義
__add__v + w加法(運算子多載)
__len__len(v)長度 / 大小
__lt__v < w小於(讓物件能被 sorted() 排序)
ℹ️
__str__ vs __repr__

一個好記的分工:__str__ 是「給使用者看的」(漂亮、易讀); __repr__ 是「給工程師看的」(明確、最好能讓人照著重建這個物件)。 慣例上,__repr__ 的輸出最好是一段「貼回程式裡就能造出同樣物件」的字串,如 Vector(1, 2)。 如果只想實作一個,選 __repr__——因為 __str__ 沒定義時會自動退回用它。

⚠️
定義了 __eq__,記得也想想 __hash__

一旦你自訂了 __eq__,Python 會把物件的 __hash__ 設成 None, 讓它不能再放進 set 或當 dict 的鍵(因為「相等」的定義變了,雜湊也得跟著變)。 若你需要它可雜湊,得自己實作 __hash__,通常用一個不可變特徵的 tuple, 例如 return hash((self.x, self.y))。

✏️ 練習 8.4 實作

幫一個 Money(金額)類別實作 __repr__ 與 __eq__, 讓兩個金額能正確比較與列印。Money(100, "TWD") 應印成易懂的樣子, 且只有「幣別相同、金額相同」才算相等。

看解答

__eq__ 要同時比金額與幣別;穩健起見,先檢查對方是不是 Money (否則拿 Money 和數字比會出錯)。回傳 NotImplemented 是 Python 的慣例, 讓它能優雅地處理「型別不匹配」:

class Money:
    def __init__(self, amount, currency="TWD"):
        self.amount = amount
        self.currency = currency

    def __repr__(self):
        return f"Money({self.amount}, {self.currency!r})"

    def __eq__(self, other):
        if not isinstance(other, Money):
            return NotImplemented          # 交給 Python 決定後續(通常得 False)
        return self.amount == other.amount and self.currency == other.currency

    def __hash__(self):
        return hash((self.amount, self.currency))   # 自訂 __eq__ 後補上,才能放進 set


print(Money(100))                       # Money(100, 'TWD')
print(Money(100) == Money(100, "TWD"))  # True
print(Money(100, "USD") == Money(100))  # False  ← 幣別不同
print(Money(100) == 100)                # False  ← 型別不同,安全處理

{self.currency!r} 裡的 !r 表示用 repr() 來格式化, 所以字串會帶引號('TWD'),讓 __repr__ 的輸出更貼近「可重建」的理想。

SOLID:五個讓設計不腐爛的原則

當你開始寫比較大的物件導向系統,會需要一些「設計品味」的指引。 最有名的一組是 SOLID 五大原則(由 Robert C. Martin 等人整理推廣)。 這裡每個只給一句直覺,細節與對應的「設計模式」留到第 10 章深入。

原則一句話小例子
S ─ 單一職責
(Single Responsibility)		 一個類別只該有一個「改變的理由」。 別讓 Report 同時負責「計算內容」和「存成 PDF」——拆成兩個類別,各管一件事。
O ─ 開放封閉
(Open–Closed)		 對「擴充」開放,對「修改」封閉。 新增 Triangle 時,不必動到既有的 total_area——靠多型,新增即可。
L ─ 里氏替換
(Liskov Substitution)		 子類別必須能無痛頂替父類別。 若 Square 繼承 Rectangle 卻讓「分別設長寬」失效,就違反了它——子類別不該打破父類別的承諾。
I ─ 介面隔離
(Interface Segregation)		 別逼別人依賴用不到的方法。 與其一個肥大的 Machine(印、掃、傳真),不如拆成 Printer、Scanner 等小介面。
D ─ 依賴反轉
(Dependency Inversion)		 依賴抽象,不要依賴具體實作。 程式該依賴抽象的 Storage 介面,而非寫死 MySQLDatabase——這樣才換得掉。
🔑
SOLID 的共同精神

五個原則看似各說各話,骨子裡都在追求同一件事:讓「會一起變的東西」靠在一起, 讓「不相干的東西」彼此獨立。如此一來,當需求改變,你只需要動一小塊, 而不是牽一髮動全身。這就是好設計的終極目標——擁抱變化。

C++ 對照:同樣的概念,更嚴格的規矩

Python 的物件導向輕巧而寬鬆;C++ 的物件導向則嚴謹而明確。 把同樣的 Animal → Dog 用 C++ 寫一次,你會看到許多 Python 幫你「自動處理」的東西, 在 C++ 裡都得親手講清楚: 

#include <iostream>
#include <string>
#include <vector>
#include <memory>

class Animal {
private:                                  // 明確的存取控制:外界碰不到
    std::string name_;
public:
    // 建構子(constructor):用初始化列表設定成員
    Animal(const std::string& name) : name_(name) {}

    // virtual 才會有「動態分派」(多型);沒寫 virtual 就是靜態綁定
    virtual std::string describe() const {
        return "我是 " + name_;
    }

    std::string name() const { return name_; }   // 唯讀存取器(getter)

    // 基底類別的解構子(destructor)務必設成 virtual,否則透過
    // 基底指標刪除子類別物件時,子類別的解構子不會被呼叫(資源洩漏)
    virtual ~Animal() = default;
};

class Dog : public Animal {               // public 繼承 = is-a 關係
private:
    std::string breed_;
public:
    Dog(const std::string& name, const std::string& breed)
        : Animal(name), breed_(breed) {}  // 先呼叫父類別建構子,再初始化自己

    std::string describe() const override {   // override:明確表示要覆寫
        return Animal::describe() + ",品種是 " + breed_;
    }
};

int main() {
    // 用基底類別的指標,裝著不同的子類別物件 —— 多型的舞台
    std::vector<std::unique_ptr<Animal>> zoo;
    zoo.push_back(std::make_unique<Dog>("旺財", "柴犬"));
    zoo.push_back(std::make_unique<Animal>("某動物"));

    for (const auto& a : zoo) {
        // 因為 describe() 是 virtual,這裡會呼叫「物件真正型別」的版本
        std::cout << a->describe() << std::endl;
    }
    return 0;   // unique_ptr 離開作用域時自動釋放記憶體(RAII)
}

對照 Python,幾個關鍵差異值得你刻進腦海:

  • 編譯期型別:C++ 在編譯時就檢查型別,寫錯型別程式根本編不過。Python 是執行時才知道,彈性高但錯誤晚發現。
  • 明確的存取控制:C++ 的 private / public / protected 是編譯器強制的,外界真的碰不到 private。Python 只有「君子之約」的底線慣例。
  • virtual 才有多型:這是最容易踩雷的一點。C++ 的方法預設是靜態綁定,只有標上 virtual 的方法,才會根據「物件實際的型別」動態分派。Python 則所有方法天生都是多型的(因為鴨子型別)。
  • 解構子與 RAII:C++ 物件有明確的「死亡時刻」,解構子(destructor) ~Animal() 會在物件被銷毀時自動執行,用來釋放資源(記憶體、檔案、鎖)。這套「資源的生命週期綁定物件的生命週期」的慣用法叫 RAII(Resource Acquisition Is Initialization),是 C++ 的核心智慧。Python 則靠垃圾回收自動管理記憶體,你很少需要操心。
⚠️
忘了 virtual 的經典 bug

在 C++ 裡,如果你忘了把 describe() 標成 virtual, 那 a->describe() 會永遠呼叫基底類別的版本, 不管 a 實際指向的是 Dog 還是 Animal——多型直接失效,而且編譯器不會報錯。 同理,基底類別忘了寫 virtual ~Animal(),透過基底指標刪除子類別物件會造成資源洩漏。 這兩個是 C++ 新手最常見、也最難察覺的坑。

把它們串起來:一個小小銀行系統

理論講完了,讓我們蓋一個麻雀雖小、五臟俱全的例子,把封裝、繼承、多型、魔術方法 全部串在一起。需求:一個銀行有不同種類的帳戶,每種帳戶「月結」時的行為不同 (儲蓄帳戶給利息;支票帳戶收手續費),但我們希望能用同一套程式碼處理所有帳戶。 

from abc import ABC, abstractmethod

class Account(ABC):
    """所有帳戶的抽象基底:封裝餘額,規定月結契約。"""
    def __init__(self, owner, balance=0):
        self.owner = owner
        self._balance = balance          # 封裝:餘額藏起來

    @property
    def balance(self):                   # 唯讀,維護「外界不可亂改」
        return self._balance

    def deposit(self, amount):
        if amount <= 0:
            raise ValueError("存款須為正數")
        self._balance += amount

    def withdraw(self, amount):
        if amount > self._balance:       # 不變式:不可透支
            raise ValueError("餘額不足")
        self._balance -= amount

    @abstractmethod
    def monthly_update(self):            # 抽象:每種帳戶月結方式不同
        ...

    def __repr__(self):                  # 魔術方法:好讀的表示
        return f"{type(self).__name__}({self.owner!r}, balance={self._balance})"


class SavingsAccount(Account):           # 繼承:儲蓄帳戶是一種帳戶
    def __init__(self, owner, balance=0, rate=0.02):
        super().__init__(owner, balance) # 重用父類別初始化
        self.rate = rate

    def monthly_update(self):            # 覆寫:加利息
        self._balance += self._balance * self.rate


class CheckingAccount(Account):          # 繼承:支票帳戶也是一種帳戶
    def __init__(self, owner, balance=0, fee=10):
        super().__init__(owner, balance)
        self.fee = fee

    def monthly_update(self):            # 覆寫:扣手續費(但不扣到負)
        self._balance = max(0, self._balance - self.fee)


# 多型:用同一個迴圈處理一籃子不同型別的帳戶
bank = [
    SavingsAccount("小明", 10000),
    CheckingAccount("小美", 5000),
    SavingsAccount("阿強", 20000, rate=0.05),
]

for acc in bank:
    acc.monthly_update()     # 各自跑各自的月結邏輯,呼叫端完全不必分類
    print(acc)               # 觸發 __repr__

# SavingsAccount('小明', balance=10200.0)
# CheckingAccount('小美', balance=4990)
# SavingsAccount('阿強', balance=21000.0)

停下來欣賞那個 for 迴圈:它完全不知道自己處理的是哪種帳戶, 卻能讓每種帳戶各自做對的事。這就是物件導向四大支柱合奏的成果—— 封裝守住每個帳戶的餘額不變式;抽象用 Account 定下「都要會月結」的契約; 繼承讓兩種帳戶重用共通的存提款邏輯;多型讓呼叫端用一套程式碼一視同仁。 明天要加一種「外幣帳戶」?寫個新子類別、實作 monthly_update 就好, 這段主程式一個字都不用改。

✏️ 練習 8.5 挑戰

設計一個更完整的 2D 向量類別 Vector2D,支援運算子多載: 加法 +、減法 -、純量乘法 *(向量乘以一個數字)、相等 ==、 以及用 abs(v) 取得向量長度(模)。請讓它印出來也好看。

看解答

每個運算子對應一個 dunder:__add__、__sub__、__mul__、 __eq__、__abs__。長度用畢氏定理 $\sqrt{x^2 + y^2}$:

import math

class Vector2D:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __repr__(self):
        return f"Vector2D({self.x}, {self.y})"

    def __add__(self, other):
        return Vector2D(self.x + other.x, self.y + other.y)

    def __sub__(self, other):
        return Vector2D(self.x - other.x, self.y - other.y)

    def __mul__(self, scalar):              # v * 3
        return Vector2D(self.x * scalar, self.y * scalar)

    def __rmul__(self, scalar):             # 3 * v 也能用(右側乘法)
        return self * scalar

    def __eq__(self, other):
        return isinstance(other, Vector2D) and self.x == other.x and self.y == other.y

    def __abs__(self):                      # abs(v) = 向量長度
        return math.hypot(self.x, self.y)


a = Vector2D(3, 4)
b = Vector2D(1, 2)
print(a + b)        # Vector2D(4, 6)
print(a - b)        # Vector2D(2, 2)
print(a * 2)        # Vector2D(6, 8)
print(2 * a)        # Vector2D(6, 8)   ← 靠 __rmul__
print(abs(a))       # 5.0              ← 3-4-5 直角三角形
print(a == Vector2D(3, 4))   # True

小細節:__rmul__(right multiply)讓 3 * a 也能用—— 當 Python 發現左邊的 int 不知道怎麼乘 Vector2D 時, 會回頭問右邊的物件「你會不會被乘?」,這時 __rmul__ 就派上用場了。

✏️ 練習 8.6 挑戰

用組合(而非繼承 list)實作一個泛用的堆疊 Stack,支援 push、pop、peek,並讓它能用 len(s) 取得元素數、 用 if s: 直接判斷空不空(空的時候為 False)。 提示:後兩個需求分別對應 __len__ 與 __bool__。

看解答

內部藏一個 list 當容器(組合),只開放堆疊該有的操作; __len__ 讓 len() 可用,而 __bool__ 沒寫時 Python 會退而求其次用 __len__ (長度為 0 即視為 False),所以其實只實作 __len__ 也能讓 if s: 正確運作—— 這裡兩個都寫出來讓意圖更清楚:

class Stack:
    def __init__(self):
        self._items = []                 # 組合:內含一個 list

    def push(self, x):
        self._items.append(x)

    def pop(self):
        if not self._items:
            raise IndexError("從空堆疊 pop")
        return self._items.pop()

    def peek(self):
        if not self._items:
            raise IndexError("從空堆疊 peek")
        return self._items[-1]

    def __len__(self):
        return len(self._items)

    def __bool__(self):                  # if s: 為空時走這裡 → False
        return len(self._items) > 0

    def __repr__(self):
        return f"Stack({self._items})"


s = Stack()
print(bool(s))      # False ← 空的
s.push(1); s.push(2); s.push(3)
print(len(s))       # 3
print(s.pop())      # 3
print(s.peek())     # 2
while s:            # 利用 __bool__,優雅地清空
    print(s.pop())  # 2, 然後 1

注意這個 Stack 沒有 insert、append(到任意位置) 之類會破壞堆疊規矩的方法—— 因為我們用組合「只借需要的」,而不是繼承 list「全盤接收」。這正是練習 8.3 的精神。

把這一章串起來

我們從「程式長大後資料與行為散落一地」的痛點出發,一路走到能設計出乾淨、好擴充的物件系統。回顧這趟旅程:

  • OOP 的本質:把資料與操作它的行為綁成「會自己照顧自己」的物件,用以馴服日益膨脹的複雜度。
  • 類別 vs 物件:類別是藍圖,物件是照藍圖蓋出的實體。__init__ 初始化、self 指向自己、屬性存資料、方法是行為。
  • 封裝:藏住內部狀態,只透過介面操作,保護不變式、降低耦合。Python 用 _ / __ 慣例與 property 達成。
  • 抽象:用 abc 定義「能做什麼」的契約,把「怎麼做」留給子類別。
  • 繼承:子類別重用並特化父類別,super() 站在肩膀上,覆寫改寫行為,MRO 決定方法去哪找。
  • 多型:同一介面、不同物件、各自的行為。Python 靠鴨子型別,不問身分只問能力;靜態語言靠虛擬函式與型別宣告。
  • 組合優於繼承:能用「有一個」就別用「是一種」。繼承留給真正的 is-a 與多型,避開脆弱基底類別與過深階層。
  • 魔術方法:__repr__、__eq__、__add__、__len__ 等讓自訂物件融入語言,用起來像原生型別。
  • SOLID:五個原則,一個精神——讓會一起變的靠在一起,讓不相干的彼此獨立。
  • C++ 對照:編譯期型別、強制存取控制、virtual 才有動態分派、解構子與 RAII——同樣的概念,更嚴格的規矩。

物件導向不是「語法」,而是一種建模世界的思考方式:把問題拆成一群各司其職、互相合作的物件。 你現在握有的,是組織大型程式最重要的工具之一。下一章我們會把視野從「設計單一程式」拉高, 而真正把這些原則用到極致、形成可複用的解法套路,則是第 10 章「設計模式」的主題—— 你會在那裡看到 SOLID 開花結果。
