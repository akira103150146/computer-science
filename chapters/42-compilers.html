翻譯,是電腦科學的母題

第 23 章結尾,我們站在一座親手蓋起來的高塔前:一行 let sum = sum + i;, 像水一樣往下流過編譯器、VM 翻譯器、組譯器,最後變成電子在 CPU 裡奔跑。那一章,我們是鳥瞰—— 三個處理站各給了一個玩具版,夠你看懂全貌。但每一站裡面,其實都藏著數十年的理論與一整個學門。 這一章,我們要俯衝下去,把每一站講深。

先停下來,想一件你可能從沒想過、但其實很奇妙的事:編譯器,是一個「讀程式、寫程式」的程式。 它的輸入是文字(你的原始碼),輸出也是文字(組合語言、位元組碼、或機器碼)。 寫編譯器的人是程式設計師,用它的人也是程式設計師——所以有人說,編譯器是 「程式設計師寫給程式設計師的程式」。它站在人類與機器之間,當一個忠實的翻譯官: 把我們腦中的意圖(「把分數加總」),翻成 CPU 唯一聽得懂的命令(一串 0 與 1)。

而「翻譯」這件事,根本就是電腦科學反覆出現的母題。組譯器把組語翻成機器碼;VM 翻譯器把位元組碼翻成組語; 瀏覽器把 HTML 翻成畫面;資料庫把 SQL 翻成查詢計畫……你會發現,只要有「兩種表示法」, 中間就站著一個翻譯程式。學會編譯器,你學到的不只是「怎麼做一個編譯器」,而是一套 「如何嚴謹地把一種結構化的語言,轉換成另一種」的通用功夫。

編譯器的本質:意義保留的翻譯

編譯器要做的,不是逐字硬翻,而是保住意義(semantics)的翻譯: 翻完之後的程式,跑起來的行為必須跟原本一模一樣,只是換了一種機器聽得懂的形式。 這跟人類翻譯一首詩很像——重點不是每個字都對上,而是意思要對。 只不過編譯器面對的是形式語言,所以它的「信達雅」可以做到數學等級的精確。

編譯、直譯、JIT:三種讓程式跑起來的方式

在拆管線之前,先把一個常被混淆的問題講清楚:同樣是「讓高階語言跑起來」, 為什麼有人說 C 是「編譯語言」、Python 是「直譯語言」、Java 跑得越久越快?差別到底在哪?

關鍵在「翻譯」與「執行」這兩件事,什麼時候發生、由誰來做。我們用一個比喻:你拿到一本外文食譜。

  • 編譯器(compiler):像請翻譯社事先把整本食譜翻成中文,印成一本新書交給你。 翻譯花的時間是一次性的;之後你照中文書下廚,快得很。代價是:翻完才能煮,而且那本中文書只適用於「會中文的廚師」(特定 CPU)。 C、C++、Rust、Go 走這條路:提前(ahead-of-time, AOT)把整個程式翻成機器碼,產生一個獨立執行檔。
  • 直譯器(interpreter):像請一位口譯站在你旁邊,你讀一句,他翻一句、你做一步。 不用等整本翻完就能開煮,很有彈性;但每做一次都要現場翻一次,慢。而且沒有一本「翻好的書」留下來。 Python、Ruby、傳統的 JavaScript 走這條路:程式邊讀邊執行,不產生獨立的機器碼檔。
  • JIT(Just-In-Time,即時編譯):最聰明的折衷。一開始像口譯一樣邊翻邊做; 但口譯發現「這一段你已經做第 100 次了」,就乾脆把這段當場翻成中文寫下來,之後直接用,跳過翻譯。 它在執行期間把反覆執行的「熱點(hot spot)」即時編譯成原生機器碼。 現代的 JVM(HotSpot)、JavaScript 的 V8、.NET、PyPy 都靠 JIT,讓「直譯的彈性」與「編譯的速度」兼得。
ℹ️
「編譯語言 vs 直譯語言」其實是個誤稱

嚴格說,語言本身不分編譯或直譯,只有「實作」分。同一個語言,你可以替它寫編譯器、也可以寫直譯器。 C 通常被編譯,但也有 C 直譯器(如 Cling);Python 通常被直譯,但會先被編譯成位元組碼(.pyc), 再由 CPython 虛擬機直譯那些位元組碼——它其實是「先編譯、後直譯」的混血。所以更精確的說法是: 「CPython 是 Python 的一種直譯式實作」「GCC 是 C 的一種編譯式實作」。 這一章講的「編譯器」,指的是前面那一大段「原始碼 → 某種低階表示」的翻譯流程—— 無論最後是被執行檔直接跑、還是被 VM 直譯,這段管線都長得一樣。

編譯器的全景:前端、中端、後端

第 23 章把編譯器切成三站:詞法、語法、程式碼產生。那是夠用的簡化。真實的編譯器,習慣用另一種更有用的切法, 分成三大塊——而這個切法,本身就是一個關於抽象與解耦的漂亮設計:

  • 前端(front end):只跟「來源語言」有關。負責「讀懂」原始碼:詞法分析、語法分析、語意分析。 它的產物是一棵帶著完整意義資訊的樹(或一份中間表示)。前端不在乎你最後要跑在哪顆 CPU 上。
  • 中端(middle end):既不管來源語言、也不管目標機器。它在一種中性的 中間表示(intermediate representation, IR)上做最佳化:把程式改寫得更快、更小,但行為不變。
  • 後端(back end):只跟「目標機器」有關。負責把 IR 翻成特定 CPU 的機器碼:指令選擇、暫存器配置、指令排程。 它不在乎這份 IR 原本是 C 還是 Rust 編出來的。

這個「前端管語言、後端管機器、中端居中最佳化」的三分法,就是第 23 章「M×N 降成 M+N」那招的升級版—— 現在連最佳化都被抽到中間,讓所有語言、所有機器都能共享。LLVM 這個現代最有名的編譯器框架, 整個設計就建立在這條界線上:Clang(C/C++)、Swift、Rust 全都把程式翻成同一種 LLVM IR, 共用同一套中端最佳化器與後端;要支援一顆新晶片,只要寫一個後端,上面所有語言立刻受惠。

前端第一站 — 詞法分析:把字元流切成 token

編譯器最初拿到的,是一堆毫無結構的字元:x、=、空格、a、+…… 在它眼中,連「這是一個單字、那是一個運算子」都還不知道。詞法分析(lexical analysis,又叫 scanning、tokenizing) 就是第一刀:把字元流切成一個個有意義的最小單位——token(詞元),並貼上類別標籤。

第 23 章我們手刻過一個 tokenizer。這次,我們要看清它底下的兩塊理論基石:正規表示式與有限自動機。

token 有哪些種類

對大多數程式語言,token 不外乎這幾類:

  • 關鍵字(keyword):語言保留的字,如 if、while、return、let。
  • 識別字(identifier):程式設計師自取的名字,如變數 score、函式 main。
  • 常值(literal):寫死在程式裡的值,如數字 42、字串 "hi"。
  • 運算子與標點(operator / punctuation):+、*、==、(、;、{。
  • 空白與註解:通常被丟掉(但要先正確辨認出來才能丟)。

tokenizer 只負責斷詞,不負責理解。你餵它 ) ) + while,它照樣乖乖切出四個 token—— 「這串 token 拼起來合不合語法」是下一站的事。詞法分析,就像把一句沒有空格的英文切成一個個單字,但還沒讀懂句子。

正規表示式:token 的「規格書」

每一類 token 長什麼樣,要怎麼精確描述?答案是你在第 16 章用 grep 玩過的老朋友—— 正規表示式(regular expression)。它正是描述「一組字串長相」的標準工具:

token 類別正規表示式白話
整數[0-9]+一個以上的數字
識別字[A-Za-z_][A-Za-z0-9_]*字母或底線開頭,後面接任意多個字母/數字/底線
浮點數[0-9]+\.[0-9]+數字、一個小數點、再數字
運算子==|!=|<=|>=|[-+*/()]雙字元運算子,或單一符號

這不是巧合。第 16 章的 grep 拿正規表示式來搜尋文字;編譯器拿一模一樣的正規表示式來切分文字。 同一套理論,兩種用途。而這引出一個更深的問題:電腦到底是怎麼判斷一串字元「符不符合」某個正規表示式的? 答案,藏在計算理論裡——一台簡單到不可思議的抽象機器。

有限自動機:正規表示式背後的引擎

每一個正規表示式,背後都對應一台有限自動機(finite automaton)——更精確地說,一台 決定性有限自動機(DFA,Deterministic Finite Automaton)。 這是第十二部「計算理論」的主角(第 49 章會嚴格展開),這裡先給你直覺。

DFA 是一台極度健忘的機器:它只有有限個狀態(state),一次讀一個字元, 然後只根據「目前在哪個狀態」與「讀到什麼字元」決定跳到哪個狀態。它沒有記憶體、不會數數, 只記得「我現在站在哪」。神奇的是,光憑這點,它就足以辨認任何正規表示式描述的字串。 來看一台同時辨認「整數」與「識別字」的 DFA:

正規表示式、DFA、tokenizer,其實是同一件事的三張臉: 正規表示式是「規格」(token 長什麼樣)、DFA是「實作」(怎麼一個字元一個字元辨認它)、 tokenizer是「應用」(用它把原始碼切開)。lex/flex 這類工具的工作, 就是讀進你寫的正規表示式,自動幫你生出對應的 DFA 程式碼。第十二部會證明一個漂亮的定理: 「正規表示式能描述的語言」恰好等於「DFA 能辨認的語言」——理論與工程在此完美閉環。

動手寫一個 tokenizer

實務上,我們很少真的手刻 DFA。一個又快又乾淨的技巧,是把每類 token 的正規表示式用具名群組(named group) 合併成「一條大 regex」,讓 Python 的 re 模組(它內部就是自動機)替我們一次比對。 每次比對成功,用 lastgroup 問「是哪一類中的」: 

import re

# 每類 token 配一條正規表示式,順序很重要(先比對到的優先)
TOKEN_SPEC = [
    ("NUMBER",   r"\d+"),                 # 整數
    ("ID",       r"[A-Za-z_]\w*"),        # 識別字
    ("OP",       r"[+\-*/()=]"),          # 運算子與括號
    ("NEWLINE",  r"\n"),                  # 換行(用來分隔敘述)
    ("SKIP",     r"[ \t]+"),              # 空白:辨認出來,但丟掉
    ("MISMATCH", r"."),                   # 其他任何字元 → 錯誤
]
# 合併成一條大 regex:(?P<NUMBER>\d+)|(?P<ID>[A-Za-z_]\w*)|...
_master = re.compile("|".join(f"(?P<{name}>{pat})" for name, pat in TOKEN_SPEC))

def tokenize(code):
    tokens = []
    for m in _master.finditer(code):
        kind, val = m.lastgroup, m.group()   # 是哪一類?比到什麼字串?
        if kind == "SKIP":
            continue                          # 空白直接跳過
        elif kind == "MISMATCH":
            raise SyntaxError(f"非法字元 {val!r}")
        elif kind == "NUMBER":
            tokens.append(("NUMBER", int(val)))   # 順手轉成 int
        elif kind == "NEWLINE":
            tokens.append(("NEWLINE", "\n"))
        else:
            tokens.append((kind, val))
    tokens.append(("EOF", None))              # 結尾哨兵,方便 parser 判斷
    return tokens

print(tokenize("ans = 12 + x * 3"))
# [('ID', 'ans'), ('OP', '='), ('NUMBER', 12), ('OP', '+'),
#  ('ID', 'x'), ('OP', '*'), ('NUMBER', 3), ('EOF', None)]

注意 TOKEN_SPEC 的順序暗藏玄機:MISMATCH(萬用的 .)放在最後, 所以只有當前面都比對失敗時才會輪到它報錯。如果語言有關鍵字,通常的做法是:照樣用 ID 的規則切出來, 切完再查一張關鍵字表,把 while 這種「剛好是保留字」的識別字改標成關鍵字—— 這樣就不必擔心 while 的正規表示式誤吃掉 whileLoop 這種變數名(最長匹配,maximal munch 的經典坑)。

前端第二站 — 語法分析:從 token 長出一棵樹

有了一排扁平的 token,真正的硬仗才開始。語法分析(syntactic analysis,俗稱 parsing) 要搞懂這些 token 之間的結構:誰修飾誰、誰先算、哪裡是一個完整的敘述。這一節是整章的重頭戲。

用文法描述語言:CFG 與 BNF

要讓機器判斷「這串 token 合不合語法」,我們得先把語法寫成規則。 詞法層用的正規表示式不夠力——它沒有記憶,數不出「左括號跟右括號有沒有配對」這種需要「記住巢狀深度」的結構。 我們需要更強的工具:上下文無關文法(context-free grammar, CFG)。

CFG 由一堆產生規則(production rule)組成,每條規則說「某個語法成分,可以由哪些更小的成分組成」。 最常見的書寫格式叫 BNF(Backus–Naur Form),1950 年代為了描述 ALGOL 語言而發明。 我們用 BNF 寫出「算術運算式」的文法(角括號 < > 框住的是語法成分,::= 讀作「可以是」): 

<expr>   ::= <term> (("+" | "-") <term>)*       // 運算式 = 一串用 +/- 連起來的 term
<term>   ::= <factor> (("*" | "/") <factor>)*    // term = 一串用 */÷ 連起來的 factor
<factor> ::= NUMBER | ID | "(" <expr> ")"        // factor = 數字、變數、或括號裡的運算式

這三行藏著兩個極深的設計。第一,它是遞迴的:<factor> 又繞回去引用 <expr> (括號的情況)。正因如此,括號才能無限巢狀——((1+2)) 完全合法。這種「自己引用自己」的能力, 正是 CFG 比正規表示式強的地方,也是「context-free」的威力來源。

第二,也更巧妙:運算子優先序被悄悄編碼進了文法的「分層」裡。把 *、/ 放在比 +、- 更「裡層」的 <term>,就自動讓乘除「先算」——因為要拆解一個 <expr>, 必須先把它的 <term> 拆完。我們等下會看到,這個分層如何讓 parser「不費力氣」就處理對優先序。

📜
喬姆斯基階層(Chomsky hierarchy)

語言學家 Noam Chomsky 在 1956 年把形式語言依「描述它需要多強的文法」分成四層: 正規語言(regular)⊂ 上下文無關(context-free)⊂ 上下文相關(context-sensitive)⊂ 遞迴可枚舉(recursively enumerable)。 這條階層,恰好對應到辨認它們所需的機器:DFA、下推自動機(堆疊!)、線性有界自動機、圖靈機。 編譯器漂亮地用上了前兩層:詞法用正規語言(夠切 token),語法用上下文無關語言(夠處理巢狀結構)。 這是計算理論直接落地到工程的最佳範例,我們留到第十二部細講。

剖析樹 vs 抽象語法樹(AST)

parser 的產物是一棵樹。為什麼是樹?因為程式天生有階層:一個運算式包含子運算式, 一個 if 包含條件與主體,主體又包含更多敘述……這種「裡面還有裡面」的結構,用樹表達最自然。但樹有兩種:

  • 剖析樹(parse tree,又叫具體語法樹):忠實記錄文法每一步推導,連 <expr>、 <term> 這些中間語法成分、連括號標點都留著。它完整,但臃腫。
  • 抽象語法樹(AST,Abstract Syntax Tree):只留下意義所需的東西——運算子、運算元、結構; 丟掉那些只為了「解析」而存在的中間節點與標點。它精煉,是後續每一站真正在用的東西。

同一個小運算式 2 + 3,兩種樹擺在一起,差別一目了然:

歧義:同一句話,兩種解讀

文法最怕一個東西:歧義(ambiguity)——同一串 token,能長出不只一棵合法的剖析樹。 最惡名昭彰的例子,是程式語言裡的「懸盪 else(dangling else)」: 

if (a) if (b) x = 1; else x = 2;

那個 else 到底是配給外層的 if (a),還是內層的 if (b)? 兩種解讀對應兩棵不同的樹,語意天差地遠。一個歧義的文法,等於沒把話講清楚,parser 無所適從。 解法有幾種:(1) 改寫文法消除歧義;(2) 訂一條消歧規則(主流語言都規定「else 配最近的 if」); (3) 強迫加大括號。我們前面那個算術文法之所以「漂亮」,正因為它無歧義:優先序與結合性都被分層與重複結構釘死了, 每個運算式只有唯一一棵樹。

遞迴下降:把文法直接變成程式

現在進入最迷人的部分:parser 到底怎麼從一排 token 長出樹?最直覺、也最常被手寫編譯器採用的方法,叫 遞迴下降(recursive descent)。它的核心點子簡單到像作弊:

文法裡的每一條規則,寫成一個函式。規則之間互相引用,函式就互相呼叫(包括遞迴)。

對應關係機械到近乎自動:規則的「選項 |」變成 if 分支;「重複 *」變成 while 迴圈; 「引用另一條規則」變成函式呼叫;「自我引用」變成遞迴。把前面那個 BNF 文法一條一條翻成 Python: 

class Parser:
    def __init__(self, tokens):
        self.toks = tokens
        self.i = 0

    def peek(self):                          # 看目前 token(不前進)
        return self.toks[self.i]

    def eat(self, val=None):                 # 吃掉目前 token,前進
        tok = self.toks[self.i]
        if val is not None and tok[1] != val:
            raise SyntaxError(f"預期 {val!r},卻看到 {tok[1]!r}")
        self.i += 1
        return tok

    def parse_expr(self):                    # <expr> ::= <term> (("+"|"-") <term>)*
        node = self.parse_term()
        while self.peek()[1] in ("+", "-"):  # 「重複」→ while
            op = self.eat()[1]
            node = BinOp(op, node, self.parse_term())   # 左結合的關鍵(見下)
        return node

    def parse_term(self):                    # <term> ::= <factor> (("*"|"/") <factor>)*
        node = self.parse_factor()
        while self.peek()[1] in ("*", "/"):
            op = self.eat()[1]
            node = BinOp(op, node, self.parse_factor())
        return node

    def parse_factor(self):                  # <factor> ::= NUMBER | ID | "(" <expr> ")"
        kind, val = self.peek()
        if kind == "NUMBER":
            self.eat();  return Num(val)
        if kind == "ID":
            self.eat();  return Var(val)
        if val == "(":
            self.eat("(")
            node = self.parse_expr()         # 遞迴!括號裡又是完整的 expr
            self.eat(")")
            return node
        raise SyntaxError(f"非預期的 token:{(kind, val)}")

三個函式,對應三條規則,長得幾乎跟文法一模一樣。parse_factor 在遇到括號時回頭呼叫 parse_expr—— 這個「下降到底、又繞回頂端」的迴圈,就是「遞迴下降」名字的由來。

優先序與結合性,是怎麼「自動」對的

這是遞迴下降最美的地方。我們從沒寫過任何一條「乘法優先」的特判,但它就是對了。為什麼?

結合性(associativity)也藏在程式碼的一個小細節裡。看 parse_expr 的迴圈: 每抓到一個新的 +,我們是把「目前已建好的整棵樹」當成新運算子的左子樹 (node = BinOp(op, node, ...))。於是 1 - 2 - 3 會長成 (1 - 2) - 3 而非 1 - (2 - 3)—— 這就是左結合,也正是減法該有的行為(答案是 −4,不是 2)。 如果你想要右結合(像次方 2 ** 3 ** 2 該是 2 ** (3 ** 2)), 只要把迴圈改成遞迴呼叫自己即可——這是本章一道練習。

🔑
遞迴下降的精髓

把語法的三件難事——巢狀(靠遞迴)、優先序(靠函式分層)、結合性(靠迴圈 vs 遞迴)—— 全部交給「函式之間的呼叫結構」去自動處理,而不是寫一堆特判。你只要把文法設計對,parser 幾乎是「抄」出來的。 這就是為什麼幾乎每一本編譯器教材、每一個真實語言(GCC、Clang、V8、Rust 的 parser)都先教、且大量採用遞迴下降。

LL 與 LR:由上而下 vs 由下而上

遞迴下降屬於一個更大的家族,叫 LL 剖析(由上而下,top-down):從文法的「起始符號」出發, 一路往下猜「接下來該套哪條規則」,像在填一棵從根長到葉的樹。它直覺、好手寫、錯誤訊息友善, 缺點是對付某些文法(尤其「左遞迴」)需要先改寫。

另一派是 LR 剖析(由下而上,bottom-up):不先猜,而是把 token 一個個堆到一個堆疊上, 一旦堆頂湊出了某條規則的右側,就把它「歸約(reduce)」成左側的語法成分,像從葉子往根組裝。 它能處理的文法範圍更廣、更適合機器自動生成,但人類難手寫、出錯時訊息也較難懂。 yacc、bison 這類「parser 產生器」走的就是 LR 路線。

LL(由上而下)LR(由下而上)
方向從根往葉(先猜規則)從葉往根(先堆再歸約)
典型做法遞迴下降(常手寫)表格驅動(常用工具生成)
代表工具ANTLR、手刻 parseryacc / bison
優點好懂、好寫、錯誤訊息清楚能吃更多文法、效率高
痛點需處理左遞迴難手寫、衝突難除錯

一句話總結:LL 是「邊讀邊猜這是什麼」,LR 是「先收集材料、湊齊了再說這是什麼」。本章手寫的,是 LL。

前端第三站 — 語意分析:符號表、型別、作用域

語法分析只保證程式「合文法」,但合文法不代表「有意義」。 x = y + 1 在文法上完美無瑕,但如果 y 從沒被宣告、或者 y 是個字串不能加 1, 這程式就是錯的。抓出這類「文法對、意義錯」的問題,是語意分析(semantic analysis)的工作。 它走訪 AST,做三件大事:建符號表、解析作用域、做型別檢查。

符號表:程式裡每個名字的戶口名簿

程式裡滿是「名字」:變數名、函式名、型別名。每個名字背後,編譯器都得記住一串資訊—— 它是什麼型別?是區域變數還是全域?住在記憶體哪一格(或哪個段、哪個暫存器)? 這本「名字的戶口名簿」就是符號表(symbol table),本質上是一個從「名字」對應到「屬性」的查找結構 (通常就是第 29 章的雜湊表)。

作用域:同一個名字,在不同地方指不同東西

麻煩的是,同一個名字 x,在不同的作用域(scope)裡可能指完全不同的東西: 函式裡的區域 x 會遮蔽(shadow)外層的全域 x。所以符號表通常是一疊的: 進入一個新作用域(函式、區塊)就疊一張新表,查名字時由內往外一層層找,離開作用域就把那張表丟掉。 這個「疊起來、由內往外查」的結構,你應該覺得眼熟——又是堆疊。

型別檢查:把錯誤擋在執行之前

語意分析最重要的一項,是型別檢查(type checking)——呼應第 9 章談過的型別系統。 編譯器走訪 AST,替每個運算式推導出它的型別,並檢查每個運算「合不合理」: 整數 + 整數 沒問題、字串 + 整數 在多數語言是錯誤、把布林值拿去當陣列索引也是錯誤。 這正是第 9 章靜態型別(static typing)的價值所在:很多 bug 在編譯期就被攔下, 根本不必等到程式跑起來才爆炸。型別檢查靠的就是 AST + 符號表:從符號表查出每個變數的型別, 由葉往根逐步「算出」整棵樹的型別,一發現矛盾就報錯。

ℹ️
語法錯誤 vs 語意錯誤

語法錯誤(syntax error)是「話沒講通」:x = + ; 連文法都不合,parser 那一關就過不了。 語意錯誤(semantic error)是「話講通了但沒道理」:x = y + 1 文法完美, 但 y 沒宣告、或型別不對。前者是 parser 的職責,後者是語意分析的職責。 你平常看到的「undefined variable」「type mismatch」錯誤訊息,絕大多數都來自這一站。

中端 — 中間表示(IR)與最佳化

前端結束,我們手上有一棵帶著完整型別資訊的 AST。理論上可以直接從這裡產生目標碼(第 23 章的玩具就是這麼幹的)。 但真實編譯器會先把 AST 降階成一種更「平」、更接近機器、卻又不綁定任何特定 CPU 的形式—— 中間表示(intermediate representation, IR),然後在 IR 上動最佳化的刀。

為什麼要有 IR?

理由跟第 23 章「為什麼要 VM」如出一轍,只是用在編譯器內部:解耦。 AST 太貼近來源語言(每種語言的 AST 長得都不一樣),機器碼又太貼近目標 CPU。 在中間插一層中性的 IR,讓最佳化器寫一次、所有語言與所有機器共用。 這就是圖 42-2 裡「中端」存在的理由,也是 LLVM 整個生態的地基——大家都翻成 LLVM IR,共享同一套最佳化。

三位址碼:IR 的經典長相

IR 有很多種形式,最經典、最好懂的一種叫三位址碼(three-address code, TAC)。 規則很簡單:每一條指令最多只有一個運算子、最多三個位址(兩個來源、一個目的), 形如 結果 = 運算元1 運算 運算元2。複雜運算式被拆成一串這種小步驟,中間結果存進編譯器自動產生的 臨時變數(temporary) t1、t2……。看 x = a + b * 2 變成 TAC: 

t1 = b * 2      // 先算乘法,結果放進臨時變數 t1
t2 = a + t1     // 再算加法
x  = t2         // 存回 x

為什麼要拆這麼碎?因為「攤平、規律、每步只做一件事」的形式,最適合機器分析與改寫。 接下來的每一種最佳化,都是在這種規律的指令串(或它的圖形版本)上動手。

最佳化巡禮:用更少的力氣,做一樣的事

最佳化(optimization)的鐵律只有一條:可以改快、改小,但行為絕不能變。 它是一門「在保證等價的前提下重寫程式」的藝術。我們看四個最經典、你一定會反覆遇到的招式,每招給你 before / after。

① 常數摺疊(constant folding)

編譯期就能算出來的常數運算,何必留到執行期?直接先算好:

// before                 // after
int sec = 60 * 60 * 24;    int sec = 86400;   // 編譯時就乘好了

它常跟常數傳播(constant propagation)聯手:如果 x = 5 之後 x 沒被改過,就把後面用到 x 的地方都換成 5,往往又能再觸發新一輪摺疊。

② 死碼消除(dead code elimination)

算了卻沒人用的結果、永遠到不了的程式碼,通通刪掉:

// before                          // after
int x = expensive();   // 算了但      return a + b;
int y = a + b;         // 後面只用 y
return y;              // x 是死碼 → 整行刪掉,連同 expensive() 的呼叫
                       // (前提:它沒有副作用)

注意那句但書:只有當 expensive() 沒有副作用(不印東西、不改全域狀態)時才能安全刪除——這正是最佳化最考驗功力的地方:判斷「刪掉它,行為真的不變嗎?」

③ 共同子運算式消除(common subexpression elimination, CSE)

同一個運算式算了兩遍?算一次、存起來、重用:

// before                          // after
a = (x + y) * 2;                   t = x + y;        // x+y 只算一次
b = (x + y) * 3;                   a = t * 2;
                                   b = t * 3;

④ 迴圈最佳化:迴圈不變式外提(loop-invariant code motion)

迴圈裡每圈都算、但結果根本不變的東西,搬到迴圈外面只算一次。迴圈跑 n 次,就省下 n−1 次:

// before                          // after
for i in range(n):                 t = a * b        # 跟 i 無關 → 提到迴圈外
    arr[i] = a * b + i             for i in range(n):
                                       arr[i] = t + i

迴圈是程式的熱點,所以迴圈最佳化的回報最大,招式也最多(還有強度削減:把昂貴的乘法換成便宜的加法; 迴圈展開:減少跳躍與判斷的次數)。

🔑
最佳化是「等價變換」的科學

每一種最佳化,骨子裡都是同一句話:找出一個跑得更快、但行為完全等價的寫法,然後替換。 難的從來不是「想到更快的寫法」,而是嚴格證明「行為真的沒變」—— 有沒有副作用?有沒有別名(alias)?有沒有例外路徑?這需要編譯器先做大量分析(analysis) (資料流分析、控制流圖、別名分析),才敢動手變換(transformation)。 分析在前、變換在後,是所有最佳化的鐵律。而這些分析,大量用到第 30 章的圖論與第 32 章的動態規劃。

後端 — 程式碼產生

最後一站:把最佳化過的 IR(或 AST)翻成真正的目標碼——可能是第 23 章的 VM 命令、第 22 章的組合語言, 或某顆 CPU 的機器碼。第 23 章已經給過一個美麗的洞見,值得在此重述,因為它是整個程式碼產生的靈魂:

把運算式樹用「後序走訪(先左子樹、再右子樹、最後自己)」,產生的命令順序,剛好就是正確的堆疊機程式。

為什麼?因為堆疊機要求「運算子動作前,運算元得先在堆疊上」。後序走訪恰恰保證:輪到處理一個運算子時, 它的兩個子樹早已被處理、值已 push 上堆疊。優先序與順序,全都「免費」地從樹的形狀掉出來。後端真正的硬骨頭, 其實在針對真實 CPU 時才出現:

  • 指令選擇(instruction selection):同一個運算,目標 CPU 可能有好幾種指令可達成,挑最划算的。
  • 暫存器配置(register allocation):CPU 的暫存器只有少少幾個(比記憶體快上百倍), 但程式裡的變數成千上百。該讓哪些變數住進寶貴的暫存器、哪些只好放回記憶體?這被巧妙地轉化成第 30 章的 圖著色(graph coloring)問題——一個 NP-hard 難題,實務上用啟發式快速求近似解。
  • 指令排程(instruction scheduling):重排指令順序,讓 CPU 的管線(第 21 章)不必空等。

堆疊機(像第 23 章的 VM、像 JVM)之所以「好產生碼」,正是因為它沒有暫存器——運算元位置永遠固定在堆疊頂端, 後端省掉了最痛的暫存器配置。這是當年設計 JVM 用堆疊架構的重要考量。代價是:堆疊機跑起來通常比暫存器機慢, 所以最後 JVM 還是得靠 JIT 把它再翻成用暫存器的原生碼。一切又繞回了「時機」與「取捨」。

大整合 — 用 Python 造一個會直譯也會編譯的小語言

理論講夠了。現在把整條前端管線串起來,做一個真的能跑的東西:一個迷你語言,支援 + - * /、括號、變數與賦值,而且——同一棵 AST,我們既能直接直譯求值,也能編譯成第 23 章式的堆疊機碼。 這正好親手體現「編譯 vs 直譯,差別只在你拿 AST 來做什麼」。

先定義 AST 節點。比起第 23 章用 tuple,我們這次用 dataclass 類別,讓每種節點有清楚的名字與欄位——這也是真實編譯器的做法:

from dataclasses import dataclass

@dataclass
class Num:    value: float                 # 數字常值,如 42
@dataclass
class Var:    name: str                    # 變數參照,如 x
@dataclass
class BinOp:  op: str; left: object; right: object   # 二元運算 a op b
@dataclass
class Unary:  op: str; operand: object     # 一元運算,如 -x
@dataclass
class Assign: name: str; expr: object      # 賦值 name = expr

tokenizer 沿用前面那支 regex 版(此處不再重貼)。parser 在前面遞迴下降的基礎上,多加兩件事:支援一元負號,以及把整個程式看成一串敘述(每行一句,可以是賦值或裸運算式):

class Parser:
    def __init__(self, tokens):
        self.toks = tokens; self.i = 0
    def peek(self): return self.toks[self.i]
    def eat(self, val=None):
        tok = self.toks[self.i]
        if val is not None and tok[1] != val:
            raise SyntaxError(f"預期 {val!r},卻看到 {tok[1]!r}")
        self.i += 1; return tok

    def parse_program(self):                # 程式 = 一串以換行分隔的敘述
        stmts = []
        while self.peek()[0] != "EOF":
            if self.peek()[0] == "NEWLINE":
                self.eat(); continue
            stmts.append(self.parse_statement())
        return stmts

    def parse_statement(self):              # 敘述 = 賦值 | 裸運算式
        # 偷看「ID =」開頭 → 賦值;否則當成運算式
        if self.peek()[0] == "ID" and self.toks[self.i + 1][1] == "=":
            name = self.eat()[1]; self.eat("=")
            return Assign(name, self.parse_expr())
        return self.parse_expr()

    def parse_expr(self):                   # <expr> ::= <term> (("+"|"-") <term>)*
        node = self.parse_term()
        while self.peek()[1] in ("+", "-"):
            op = self.eat()[1]
            node = BinOp(op, node, self.parse_term())
        return node

    def parse_term(self):                   # <term> ::= <factor> (("*"|"/") <factor>)*
        node = self.parse_factor()
        while self.peek()[1] in ("*", "/"):
            op = self.eat()[1]
            node = BinOp(op, node, self.parse_factor())
        return node

    def parse_factor(self):                 # <factor> ::= "-" <factor> | NUM | ID | "(" <expr> ")"
        kind, val = self.peek()
        if val == "-":                      # 一元負號
            self.eat(); return Unary("-", self.parse_factor())
        if kind == "NUMBER":
            self.eat(); return Num(val)
        if kind == "ID":
            self.eat(); return Var(val)
        if val == "(":
            self.eat("("); node = self.parse_expr(); self.eat(")")
            return node
        raise SyntaxError(f"非預期的 token:{(kind, val)}")

def parse(code):
    return Parser(tokenize(code)).parse_program()

現在有了 AST,兵分兩路。第一條路:直譯。寫一個遞迴函式走訪 AST,當場算出每個節點的值, 變數存在一個字典 env 裡(這個 env 就是最陽春的符號表): 

def interpret(stmts):
    env = {}; result = None
    for node in stmts:
        result = ev(node, env)
    return result, env

def ev(node, env):
    if isinstance(node, Num):    return node.value
    if isinstance(node, Var):
        if node.name not in env:
            raise NameError(f"未定義變數 {node.name}")
        return env[node.name]
    if isinstance(node, Unary):  return -ev(node.operand, env)
    if isinstance(node, Assign):
        env[node.name] = ev(node.expr, env)      # 求值後存進環境
        return env[node.name]
    if isinstance(node, BinOp):
        a, b = ev(node.left, env), ev(node.right, env)   # 後序:先算兩個子樹
        return {"+": a + b, "-": a - b, "*": a * b, "/": a / b}[node.op]
    raise TypeError(node)

第二條路:編譯。走訪同一棵 AST,但這次不求值,而是吐出第 23 章那種堆疊機 VM 命令。 順便用一張符號表把變數名對應到 local 段的索引——正是前面語意分析講的東西,在這裡真的派上用場: 

class Compiler:
    def __init__(self):
        self.code = []
        self.symtab = {}                     # 變數名 → local 索引(符號表!)
    def slot(self, name):                    # 查/配置變數的 local 槽位
        if name not in self.symtab:
            self.symtab[name] = len(self.symtab)
        return self.symtab[name]
    def emit(self, line): self.code.append(line)

    def gen(self, node):                      # 後序走訪,產生堆疊機命令
        if isinstance(node, Num):
            self.emit(f"push constant {node.value}")
        elif isinstance(node, Var):
            self.emit(f"push local {self.slot(node.name)}")
        elif isinstance(node, Unary):
            self.gen(node.operand); self.emit("neg")
        elif isinstance(node, Assign):
            self.gen(node.expr)               # 先把右邊算到堆疊頂端
            self.emit(f"pop local {self.slot(node.name)}")   # 再存進變數
        elif isinstance(node, BinOp):
            self.gen(node.left)               # 先左子樹
            self.gen(node.right)              # 再右子樹
            self.emit({"+":"add","-":"sub","*":"mul","/":"div"}[node.op])

def compile_program(stmts):
    c = Compiler()
    for s in stmts:
        c.gen(s)
    return c.code, c.symtab

為了證明這串命令是真貨,我們再寫一台 30 行的小堆疊機,實際執行它產生的碼——閉環驗證:

def run_vm(code, nlocals):
    stack, local = [], [0] * nlocals
    for line in code:
        p = line.split(); op = p[0]
        if op == "push" and p[1] == "constant": stack.append(int(p[2]))
        elif op == "push" and p[1] == "local":  stack.append(local[int(p[2])])
        elif op == "pop"  and p[1] == "local":   local[int(p[2])] = stack.pop()
        elif op == "neg":                        stack.append(-stack.pop())
        else:
            y = stack.pop(); x = stack.pop()
            stack.append({"add":x+y, "sub":x-y, "mul":x*y, "div":x//y}[op])
    return stack, local

✏️ 練習 42.1 實作

替一個迷你語言寫 tokenizer。這個語言的 token 集是:關鍵字 if/else/while、 識別字、整數、以及符號 { } ( ) ; = < > + -。 讓 tokenize("if (x > 0) y = 1;") 切出正確的 token 串(空白要丟掉,關鍵字要跟普通識別字區分開)。

看解答

策略:先用「合併大 regex」切出所有 token,把字母開頭的一律當識別字切出來,切完再查一張關鍵字表, 把剛好是保留字的改標成關鍵字。這樣 whileLoop 就不會被誤判成關鍵字 while + 識別字 Loop。

import re

KEYWORDS = {"if", "else", "while"}
SPEC = [
    ("ID",       r"[A-Za-z_]\w*"),
    ("NUMBER",   r"\d+"),
    ("SYM",      r"[{}();=<>+\-]"),
    ("SKIP",     r"[ \t]+"),
    ("MISMATCH", r"."),
]
_master = re.compile("|".join(f"(?P<{n}>{p})" for n, p in SPEC))

def tokenize(code):
    out = []
    for m in _master.finditer(code):
        kind, val = m.lastgroup, m.group()
        if kind == "SKIP":
            continue
        if kind == "MISMATCH":
            raise SyntaxError(f"非法字元 {val!r}")
        if kind == "ID" and val in KEYWORDS:   # 切完再查關鍵字表
            kind = "KEYWORD"
        if kind == "NUMBER":
            val = int(val)
        out.append((kind, val))
    return out

print(tokenize("if (x > 0) y = 1;"))
# [('KEYWORD','if'), ('SYM','('), ('ID','x'), ('SYM','>'), ('NUMBER',0),
#  ('SYM',')'), ('ID','y'), ('SYM','='), ('NUMBER',1), ('SYM',';')]
print(tokenize("whileLoop = 3"))
# [('ID','whileLoop'), ('SYM','='), ('NUMBER',3)]  ← while 沒被誤切!

注意 SPEC 裡 ID 排在 NUMBER 前、MISMATCH 排最後; SYM 字元類別裡的 - 要放結尾(或跳脫)以免被當成範圍。「先當識別字、再查表升級成關鍵字」是真實 lexer 的標準手法。

✏️ 練習 42.2 思考

用 BNF 寫一個文法,描述「逗號分隔的整數清單,外面包一對中括號」,例如 []、[1]、[1, 2, 3] 都合法, 但 [1,]、[1 2]、[1,,2] 不合法。

看解答

關鍵是把「空清單」與「非空清單」分開處理,並用「第一個元素 + (逗號 + 元素)的重複」來避免結尾多餘的逗號:

<list>     ::= "[" "]"                       // 空清單
             | "[" <items> "]"               // 非空清單
<items>    ::= NUMBER ("," NUMBER)*           // 一個數字,後面跟「,數字」零或多次
NUMBER     ::= [0-9]+

為什麼這樣就排除了 [1,]?因為 ("," NUMBER)* 規定逗號後面一定得跟一個 NUMBER, 不可能出現「逗號後面直接是 ]」。[1 2] 也不合法,因為兩個數字之間沒有逗號, <items> 拆完第一個 1 後,下一個不是 , 也不是 ],卡住。 這個「頭 + 重複的(分隔符 + 項)」模式,是描述各種「分隔清單」的萬用範本(函式引數、陣列元素都用它)。

進階思考:如果改用 <items> ::= NUMBER | NUMBER "," <items>(右遞迴)行不行?也行,而且這版更純粹地展示了 CFG 的遞迴本質。試著畫出 [1, 2, 3] 在兩種寫法下的剖析樹,比較它們的形狀。

不靠程式,純手動,把運算式 a * (b + c) - d / 2 畫成 AST(抽象語法樹)。 標出哪個運算子在樹根(代表最後才算)。再依後序走訪,寫出它對應的堆疊機命令序列。

看解答

先想優先序與結合性:*、/ 高於 +、-;括號逼 b + c 先算; 最外層是減法,所以 - 在樹根。樹長這樣(用縮排表示階層):

✏️ 練習 42.4 實作

擴充本章的遞迴下降 parser,加入次方運算子 **。它的優先序高於乘除,而且是 右結合:2 ** 3 ** 2 要解析成 2 ** (3 ** 2) = 512,而不是 (2 ** 3) ** 2 = 64。 (提示:在 term 與 factor 之間插一層新規則 power,並讓它用遞迴而非迴圈來達成右結合。)

看解答

右結合的祕訣:把「右運算元」交給遞迴呼叫自己(而不是像左結合那樣用 while 把樹往左掛)。 新文法層:power ::= factor ("**" power)?——注意右側又是 power,這個右遞迴正是右結合的來源。

    def parse_term(self):                 # term 現在呼叫 power(不再直接呼叫 factor)
        node = self.parse_power()
        while self.peek()[1] in ("*", "/"):
            op = self.eat()[1]
            node = BinOp(op, node, self.parse_power())
        return node

    def parse_power(self):                # power ::= factor ("**" power)?  ← 右結合
        node = self.parse_factor()
        if self.peek()[1] == "**":
            self.eat()
            node = BinOp("**", node, self.parse_power())   # 右遞迴!
        return node

(別忘了 tokenizer 要先能切出 **:在符號規則裡加 \*\*,且放在單一 * 之前, 以「最長匹配」原則優先切兩個星號。)驗證:

2 ** 3 ** 2   →  BinOp('**', 2, BinOp('**', 3, 2))   = 512   ✓ 右結合
2 ** 3 * 2    →  BinOp('*', BinOp('**', 2, 3), 2)    = 16    ✓ ** 比 * 先

對照本章 parse_expr 的迴圈(左結合)與這裡的遞迴(右結合), 你就徹底掌握了「結合性是怎麼用程式碼結構決定的」這件事。

✏️ 練習 42.5 實作

寫一個常數摺疊最佳化函式 fold(node):它走訪本章的 AST, 把「兩個運算元都是常數」的子樹,就地換成計算好的單一 Num 節點。 例如 2 + 3 * 4 + x 摺疊後應變成 14 + x(即 BinOp('+', Num(14), Var('x')))。

看解答

標準做法是後序遞迴:先把左右子樹各自摺疊好,再看「摺完之後」兩邊是不是都變成了 Num——若是,就當場算出來、回傳一個新的 Num;否則回傳一個子樹已摺疊過的新節點。

def fold(node):
    if isinstance(node, BinOp):
        l, r = fold(node.left), fold(node.right)      # 先遞迴摺疊兩邊
        if isinstance(l, Num) and isinstance(r, Num): # 兩邊都成了常數 → 直接算
            v = {"+": l.value + r.value, "-": l.value - r.value,
                 "*": l.value * r.value, "/": l.value / r.value}[node.op]
            return Num(v)
        return BinOp(node.op, l, r)                    # 否則保留結構(但子樹已摺疊)
    if isinstance(node, Unary):
        o = fold(node.operand)
        if isinstance(o, Num):
            return Num(-o.value)
        return Unary(node.op, o)
    return node                                        # Num / Var 葉節點原樣回傳

tree = parse("2 + 3 * 4 + x")[0]
print(fold(tree))
# BinOp(op='+', left=Num(value=14), right=Var(name='x'))   ← 14 + x

體會「分析在前、變換在後」:我們必須先(遞迴地)把子樹算到底,才能判斷一個節點能不能摺。 這個迷你函式,跟真實編譯器裡的常數摺疊,骨架完全相同——差別只在真實版還要小心溢位、浮點誤差與副作用。 進階:摺疊後接著做一輪死碼消除(例如 x * 0 直接變 0、x + 0 直接變 x), 叫做代數化簡(algebraic simplification),試著加進 fold 裡。

✏️ 練習 42.7 思考

回到「懸盪 else」這個歧義。對於 if (a) if (b) x = 1; else x = 2;, 畫出兩種可能的解讀(else 配內層 if、else 配外層 if),說明它們在 a 為真、b 為假時行為有何不同。 主流語言選了哪一種?你會怎麼修改文法或加上規則來消除這個歧義?

看解答

解讀一(else 配最近的內層 if,主流選擇):

if (a) {
    if (b) x = 1;
    else   x = 2;     // else 屬於內層 if(b)
}

解讀二(else 配外層 if):

if (a) {
    if (b) x = 1;
}
else x = 2;           // else 屬於外層 if(a)

當 a 為真、b 為假時:解讀一會執行 x = 2(進了外層,內層 b 假走 else); 解讀二則什麼都不做(外層 a 真,不走它的 else;內層 b 假又沒有自己的 else)。 同一段碼,兩種命運——這就是歧義的危險。

主流語言(C、Java、C# 等)一律規定:else 配對「最近、且尚未配對」的 if,也就是解讀一。 消歧的辦法:(1) 訂這條「最近匹配」規則,讓 parser 在遇到 else 時優先綁給最內層 if(遞迴下降天生就會這樣); (2) 改寫文法,區分「有配對 else 的完整 if」與「沒有 else 的開放 if」,從文法層面就只允許一種樹; (3) 最務實的——強制使用大括號,讓程式設計師自己講清楚。許多團隊的程式風格規範直接要求「if 一律加 {}」, 根源就是為了躲開這個經典陷阱。

把這一章串起來

第 23 章我們鳥瞰了翻譯鏈;這一章,我們把編譯器這節車廂徹底拆開,看清了每一根齒輪。回顧路標:

  • 編譯、直譯、JIT 的差別,在於「翻譯」發生的時機:事先全翻、完全現場翻、或先跑再挑熱點翻。語言本身不分編譯或直譯,只有實作分。
  • 編譯器的全景分三塊:前端(管來源語言)、中端(在中性 IR 上最佳化)、後端(管目標機器)。這條界線是 LLVM 等現代框架的地基。
  • 詞法分析用正規表示式(呼應 grep)描述 token、用有限自動機(DFA)辨認它們——理論與工具在此握手。
  • 語法分析用上下文無關文法(CFG/BNF)描述結構,產出AST。遞迴下降把文法直接變成互相呼叫的函式,讓優先序(靠分層)、結合性(靠迴圈 vs 遞迴)、巢狀(靠遞迴)全部自動成立。LL 由上而下、LR 由下而上。
  • 語意分析用符號表(疊成作用域)記住每個名字,用型別檢查把「文法對、意義錯」的 bug 擋在執行之前。
  • 中端把 AST 降成三位址碼等 IR,在上面做常數摺疊、死碼消除、CSE、迴圈最佳化——全都是「保證等價」的重寫;分析在前、變換在後。
  • 後端後序走訪樹產生目標碼;面向真實 CPU 時還有指令選擇、暫存器配置(化約成圖著色)、指令排程。
  • 我們用一百多行 Python,做了一個會直譯也會編譯的小語言——兩條路共用 tokenizer 與 parser,只在拿到 AST 後分岔。
  • 編譯器是整個 CS 的縮影:形式語言、樹/圖/雜湊、DP/貪婪/圖著色、型別系統與執行期,全在這裡交會。

從第 1 章的「演算法 vs 程式」,到第 23 章親手蓋出翻譯鏈,再到這一章把每一階段講透——你對「程式如何變成執行」這件事的理解, 已經完整繞了一圈。下一站,我們會把目光投向更廣的進階程式設計領域。但無論你之後寫哪種語言、用哪個框架, 你都會記得:在那行 print("Hello") 與螢幕上跳出的字之間,站著一整座你已經看穿的精密機器。
