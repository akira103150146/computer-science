指令,其實也只是數字

我們花了二十章,從演算法、Python、資料結構,一路爬到了離散數學,還在第 20 章用邏輯閘親手造出了 ALU(算術邏輯單元)和記憶體。現在,讓我們問一個你也許從沒認真想過的問題: 當你按下「執行」,你那段漂漂亮亮的程式,到底在那塊指甲蓋大小的晶片裡發生了什麼事?

答案會讓你既失望又震撼。失望的是:CPU 其實笨得可以。它不懂什麼是「函式」、什麼是「迴圈」、什麼是「物件導向」。 它一輩子只會做一件事——讀一個數字,照那個數字做一件極簡單的事,然後讀下一個數字。 震撼的是:就靠這件笨事,每秒重複數十億次,它撐起了整個數位文明。

這裡藏著一個關鍵字:「讀一個數字」。CPU 真正執行的東西,不是你看到的 x = a + b,而是一串 0 與 1。這串 0/1,我們叫它機器語言(machine language), 是電腦唯一真正聽得懂的語言。一條機器語言指令,長得就像: 

1110000010010000

十六個位元。沒有任何文字、沒有任何符號。它就是一個數字(寫成十進位是 57488)。 而 CPU 看到這個數字,就知道要去做某件事——比方說「把 D 暫存器和 A 暫存器相加,結果存回 D」。 指令,本身就是一個數字。

🔑
指令即資料(instruction as data)

這是整本書最深刻的洞見之一:在電腦眼裡,程式和資料沒有本質區別,它們都只是記憶體裡的一串數字。 數字 57680 可以是「某張照片裡某個像素的顏色值」,也可以是「一條把兩數相加的指令」—— 差別只在於 CPU 怎麼看待它。正因為「指令也只是資料」,我們才可能寫出 「能讀進別的程式、再吐出另一個程式」的程式:組譯器、編譯器、直譯器、甚至病毒,全都建立在這個基礎上。 這一章的後半,我們就會用它來爆破一個你以為很神祕的東西。

但「一台機器怎麼把一串數字變成有意義的行為」?要回答它,我們得先看清楚整台電腦的骨架—— 一張畫出來八十年、至今幾乎沒變過的藍圖。

馮紐曼架構:一張統治了八十年的藍圖

1945 年,數學家馮紐曼(John von Neumann)在一份關於 EDVAC 電腦的草稿裡, 勾勒出一個極其簡潔的設計。八十年後的今天,從你手機裡的晶片到資料中心的伺服器, 骨架幾乎一模一樣。我們稱它為馮紐曼架構(von Neumann architecture)。它只有三個大塊:

  • 中央處理器(CPU,Central Processing Unit):大腦。負責「做」——算數、做決定、搬資料。
  • 記憶體(memory):筆記本。存放兩種東西——程式(要做的指令)和資料(要處理的數字)。
  • 輸入/輸出(I/O,Input/Output):與外界的窗口。鍵盤、螢幕、硬碟、網路。

它們之間靠匯流排(bus)——一束電線——來回搬運位址與資料。整張圖長這樣:

圖中最容易被滑過、卻最重要的一點,是中間那塊記憶體裡程式和資料並排放在一起。 在馮紐曼之前,許多早期機器(像 ENIAC)要改程式得重新接線——程式不是「存」在機器裡,而是「焊」在機器裡的。 馮紐曼架構把程式也當成普通數字存進記憶體,於是「換一個程式」變成「把另一串數字載入記憶體」這麼簡單的事。 這個想法叫儲存程式(stored-program)概念,是現代電腦的起點。

📜
「馮紐曼架構」這個名字有點不公平

這個名字來自馮紐曼 1945 年那份《First Draft of a Report on the EDVAC》。但「儲存程式」的概念並非他一人之功: 機器的工程實作來自 Eckert 與 Mauchly(ENIAC/EDVAC 的設計者),而「一台能讀程式來執行任何程式的通用機器」這個更深的理論, 早在 1936 年就由圖靈(Alan Turing)以「通用圖靈機(universal Turing machine)」奠定。 因為那份草稿只掛了馮紐曼一個人的名字、又流傳最廣,功勞就這樣記在他頭上了。歷史常常這樣不公平。

ℹ️
一個誠實的小提醒:Hack 其實偷了個懶

本章主要的教學範例是 Nand2Tetris 的 Hack 電腦。為了讓硬體更好懂,Hack 把指令放在一塊唯讀的 指令記憶體(ROM),資料放在另一塊資料記憶體(RAM),兩者分開——這嚴格說來是 哈佛架構(Harvard architecture),而不是純粹的馮紐曼(程式與資料共用一塊記憶體)。 真實的桌機/手機是馮紐曼式的(共用)。這個差別不影響你理解本章的任何概念,但既然我們講求誠實,就先講清楚。

拆開 CPU:它的四個器官

把鏡頭推近到 CPU 內部。它看似神祕,其實只由四種零件組成。我們在第 20 章已經造好了其中最關鍵的兩種(ALU 和暫存器), 這裡把它們組裝起來,再補上另外兩種:

器官英文它做什麼比喻
算術邏輯單元ALU做加減、AND/OR/NOT 等運算(第 20 章造的)計算機的「手」
暫存器registersCPU 內部極快的小儲存格,放正在用的數字手邊的便利貼
控制單元control unit解碼指令,指揮其他零件「現在該做什麼」樂團的指揮
程式計數器PC(Program Counter)記住「下一條指令在記憶體的哪個位址」讀到第幾行的書籤

為什麼需要暫存器?因為記憶體雖然大,但相對於 CPU 的速度卻「很遠、很慢」。 CPU 算一筆數要先把資料從記憶體搬進來、算完再搬回去,太浪費時間。 於是 CPU 內部留了幾個超快的小格子當「草稿紙」。Hack 平台只有三個你需要認識的:

名稱全名 / 角色說明
DData register(資料暫存器)純粹存一個數值,給 ALU 當運算元。
AAddress register(位址暫存器)身兼兩職:既可當一個數值,也可當成「要存取的記憶體位址」。
M記憶體裡的那一格不是真的暫存器,而是 RAM[A] 的簡寫——「A 指到的那格記憶體」。

記住 M 就是 RAM[A],這一點之後寫程式時無比關鍵:想存取記憶體第 100 格,你得先把 100 放進 A,然後才能透過 M 去碰它。下圖把這四個器官的接線畫出來:

指令週期:電腦永不停止的心跳

現在四個器官都到位了,它們怎麼協作?答案是一個一刻不停重複的循環,叫做指令週期(instruction cycle), 或俗稱取指-解碼-執行週期(fetch-decode-execute cycle)。它就是 CPU 的心跳。每一拍,CPU 做四件事:

  1. 取指(fetch):看 PC 說下一條指令在哪,就去記憶體把那串 0/1 讀進來。
  2. 解碼(decode):控制單元拆解這串位元,搞懂「這是什麼指令、要動用哪些零件」。
  3. 執行(execute):照指令做事——ALU 算數、或讀寫記憶體、或準備跳躍。
  4. 回寫(write-back):把結果存回該存的地方(某個暫存器或記憶體格)。

做完這四步,PC 前進到下一條(通常是 PC = PC + 1),然後一切從頭再來。 除非——指令是一個「跳躍」,那 PC 就不是 +1,而是被改成某個指定位址(這就是迴圈與 if 的祕密,等等會看到)。

走一遍:跑一條 D=D+A

抽象講完,我們用一條具體指令把這四步走一遍。假設 PC 現在指向位址 5,而位址 5 存著一條指令,它的意思是 「把 D 和 A 相加,結果存回 D」(它的組合語言寫法就是 D=D+A)。執行前 D = 2、A = 3:

  1. 取指:PC = 5,CPU 去讀 ROM[5],拿到那串 16 位元 1110000010010000。
  2. 解碼:控制單元看開頭,發現這是一條「運算指令」;再拆出欄位,得知「運算 = D+A,結果要存到 D,不跳躍」。
  3. 執行:控制單元命令 ALU,把 D(=2)當一個輸入、A(=3)當另一個輸入,做加法,得到 5。
  4. 回寫:把 5 寫進 D 暫存器。現在 D = 5。
  5. PC 前進:因為沒有跳躍,PC 變成 6,準備讀下一條。

就這樣。沒有魔法。你看到的每一個 App、每一場遊戲、每一次 AI 推論,拆到最底,都是這四步,重複幾十億次。 但我們剛剛偷渡了一句「控制單元拆出欄位」——它到底怎麼從一串 0/1 看出「運算 = D+A」? 是時候真正讀懂機器碼了。

Hack 機器語言:最小可用的一套指令

真實 CPU(像 x86)有上千條指令,讀起來像一本天書。Nand2Tetris 的 Hack 平台聰明地砍到極簡: 整套機器語言只有兩種指令。學會這兩種,你就掌握了所有 CPU 的共通骨架。 每條 Hack 指令都是 16 個位元,靠最高位(最左邊那一位)來區分是哪一種:

  • 最高位是 0 → A 指令(A-instruction):把一個數字載入 A 暫存器。
  • 最高位是 1 → C 指令(C-instruction):做一次運算(這是主力,幾乎所有事都靠它)。

A 指令:把一個數字搬進 A

A 指令最單純。它的格式是 0vvvvvvvvvvvvvvv:開頭一個 0,後面 15 個位元就是要載入的數值(0 到 32767)。 組合語言寫成 @值。例如 @21: 

組合語言:   @21
機器碼:     0000000000010101
            └┬┘└──────┬─────┘
           最高位 0    21 的 15 位元二進位(16+4+1)

執行 @21 之後,A 暫存器就變成 21。A 指令有兩個用途:一是「給 ALU 一個常數運算元」(例如想加 21,先 @21), 二是「指定一個記憶體位址」(想碰第 21 格記憶體,先 @21,之後就能用 M = RAM[21])。

C 指令:Hack 的主力

C 指令是真正幹活的指令,它一口氣回答三個問題:算什麼?(comp)算完存到哪?(dest)算完要不要跳?(jump) 組合語言的完整格式是: 

dest = comp ; jump

其中 dest= 和 ;jump 都可以省略。例如 D=D+A(只有 dest 和 comp)、0;JMP(只有 comp 和 jump)。 對應到 16 位元,它被切成這幾個欄位:

comp 欄位:直接接回第 20 章的 ALU

這是全章最漂亮的接點。還記得第 20 章我們造 ALU 時,它有 6 個控制位元嗎? zx, nx, zy, ny, f, no——分別控制「要不要把 x 歸零、要不要把 x 反相、……、要不要把輸出反相」。 當時你可能覺得這 6 個位元有點抽象。現在揭曉:C 指令 comp 欄位的那 6 個位元,字面上就是餵給 ALU 的那 6 個控制位元!

前面再加一個 a 位,決定 ALU 的第二個運算元是 A 暫存器(a=0) 還是 記憶體 M(a=1)。 所以 comp 欄位一共 7 位(a + 6 個 ALU 位元),控制單元什麼都不用「想」,直接把這 6 位原封不動接到 ALU 的控制腳, ALU 就會算出正確的東西。組合語言裡那些好記的寫法,只是這些位元組合的別名:

想算的 (comp)azxnxzynyfnoALU 在做什麼
D0001100原封不動輸出 D
A0110000原封不動輸出 A
M1110000輸出 RAM[A]
D+10011111D 加一
D+A0000010D 加 A(f=1 是加法)
D-A0010011D 減 A
D&A0000000D 與 A 逐位元 AND(f=0 是邏輯)

看最後兩欄的 f 位:它是 1 就做加法、是 0 就做 AND——這正是第 20 章 ALU 那個「f 選功能」的位元。 一條高階的「相加」,到了最底層,不過是把某幾個開關撥到對的位置而已。

🔑
控制單元其實「不太聰明」

很多人以為控制單元是 CPU 的「大腦中的大腦」,會思考。其實它笨得驚人:它做的事,大半只是把指令裡的位元 當成電線,直接接到對應零件的控制腳。comp 的 6 位接到 ALU、dest 的 3 位接到「要不要載入這個暫存器」的開關、 jump 的 3 位接到「要不要把 PC 換成 A」的判斷。沒有思考,只有佈線(wiring)。 這就是為什麼純硬體就能執行程式——因為「解碼」本質上是一張固定的接線圖。

dest 與 jump 欄位

dest(目的地)是 3 個位元 d1 d2 d3,各自對應「要不要把 ALU 的結果存進 A、D、M」。 三個獨立的開關,可以同時打開,所以一次能存到多個地方:

位元 d1d2d3助記符結果存到哪
000(省略)哪都不存(通常配 jump 使用)
001M存進 RAM[A]
010D存進 D
100A存進 A
011MD同時存進 M 和 D
111AMDA、M、D 三個都存

jump(跳躍)也是 3 個位元 j1 j2 j3,意思是「拿 comp 算出的值跟 0 比較,符合條件就跳」。 「跳」的意思是:把 PC 設成 A 暫存器目前的值,於是下一輪取指就從那裡開始(而不是順順地 +1)。 這就是迴圈與分支的物理本質。

位元 j1j2j3助記符跳躍條件(對 comp 的結果)
000(省略)永不跳
001JGT結果 > 0 才跳
010JEQ結果 = 0 才跳
011JGE結果 ≥ 0 才跳
100JLT結果 < 0 才跳
101JNE結果 ≠ 0 才跳
110JLE結果 ≤ 0 才跳
111JMP無條件跳(永遠跳)

所以 0;JMP 的意思是:comp 算 0(這裡算什麼根本不重要),dest 省略(結果不存), jump 是 JMP(無條件跳)。效果就是「不管三七二十一,跳到 A 指的地方」——這就是 goto。

用組合語言寫程式:if 與迴圈的底層真相

理論夠了,來真的。組合語言(assembly)就是機器碼的「人類可讀版」:把 1110000010010000 寫成 D=D+A。它跟機器碼是一對一的——每一行組語剛好對應一條機器指令(這也是它跟 C、Python 最大的差別)。

例一:兩數相加

我們的目標:算 2 + 3,把答案存進記憶體第 0 格 RAM[0]。回想 M 就是 RAM[A], 所以要先把 A 設成 0,才能透過 M 寫進第 0 格: 

// 目標:RAM[0] = 2 + 3
@2          // A = 2
D=A         // D = A = 2          (把 2 搬進 D)
@3          // A = 3
D=D+A       // D = D + A = 2 + 3 = 5
@0          // A = 0              (準備寫到第 0 格)
M=D         // RAM[0] = D = 5

注意那個一再出現的節奏:「想用一個常數或位址,就先 @它,把它放進 A」。 因為 A 暫存器只有一個,你得不斷地把它「借」來放常數、再「借」來當位址。 一開始會覺得綁手綁腳,但這正是看清楚硬體有多原始的好方法。

例二:用 jump 蓋出迴圈(累加 1 到 n)

高階語言裡你寫 for 或 while,CPU 並不認得這些字。迴圈在底層, 是「做一段事 → 檢查條件 → 條件成立就跳回去重做」。我們來算 1 + 2 + … + n,其中 n 放在 RAM[0](也就是 R0),答案存到 RAM[1](R1)。 我們用兩個變數:i(目前數到幾)和 sum(累加和)。括號 (LOOP) 是標籤(label), 標記一個位址,讓 jump 有地方可跳: 

// 目標:R1 = 1 + 2 + ... + R0
    @R1
    M=0          // sum = 0
    @i
    M=1          // i = 1            (i 是我們自取名的變數,組譯器會配一格 RAM 給它)

(LOOP)
    @i
    D=M          // D = i
    @R0
    D=D-M        // D = i - n
    @END
    D;JGT        // if (i - n) > 0,即 i > n,就跳到 END,結束迴圈

    @i
    D=M          // D = i
    @R1
    M=D+M        // sum = sum + i
    @i
    M=M+1        // i = i + 1
    @LOOP
    0;JMP        // 無條件跳回 LOOP,再來一輪

(END)
    @END
    0;JMP        // 收尾:原地無限迴圈,讓程式「停」在這(Hack 慣例)

把它對照高階語言,你會看見一個一對一的翻譯:

Python / 高階概念組合語言怎麼蓋出來
while i <= n:在迴圈開頭算 i - n,用 D;JGT 在「不該繼續時」跳出去
sum += i@R1 / M=D+M
i += 1@i / M=M+1
迴圈「繞回去」@LOOP / 0;JMP(無條件跳)
if 分支就是一個條件跳躍(JGT/JEQ/...):條件不成立就跳過某段
🔑
所有控制流,底層都是跳躍

第 2 章你學的 if、while、for,甚至函式呼叫,到了 CPU 這一層,通通只是「改 PC」這一個動作。 條件分支 = 條件跳躍;迴圈 = 跳回去;函式呼叫 = 跳過去、記住回來的位址、做完再跳回來。 高階語言給了你漂亮的結構,但底層只有一招:goto。這也是為什麼前輩們說「goto 是組合語言的母語」。 你寫的每一個優雅的 for 迴圈,落地都是一串 @LOOP / 0;JMP。

記憶體映射 I/O:螢幕和鍵盤,其實只是一段記憶體

到這裡你可能有個疑問:CPU 只會算數和讀寫記憶體,那它怎麼畫圖到螢幕、怎麼知道你按了哪個鍵? 難道有什麼特別的「畫圖指令」?答案優雅得令人吃驚:沒有。Hack(以及大量真實系統)用的招數叫 記憶體映射 I/O(memory-mapped I/O)——把輸入輸出裝置,偽裝成一段普通的記憶體位址。

在 Hack 裡,記憶體的位址空間是這樣劃分的:

於是「畫一個點到螢幕左上角」這種看似需要硬體魔法的事,變成一條再普通不過的記憶體寫入: 

@SCREEN     // A = 16384(SCREEN 是內建符號,等於螢幕第一格的位址)
M=1         // RAM[16384] = 1 → 左上角第一個像素亮起

反過來,「讀鍵盤」就是去讀那一格記憶體的值:它存著目前被按下那個鍵的代碼(沒按鍵就是 0)。 這個設計的威力在於:CPU 完全不需要為每種裝置設計專屬指令。要支援新硬體?分一段位址給它就好。 程式設計師也只要會 M=... 和 D=M,就能操控全世界的周邊。這就是輸入輸出的底層真相。

放眼真實世界:定址、ISA、RISC 與 CISC

Hack 是個刻意極簡的教學平台,但它的每個概念,在真實 CPU 裡都有對應的、更豐富的版本。我們快速巡禮幾個關鍵詞。

定址模式(addressing modes)

「指令怎麼指出它要的運算元在哪」,有好幾種方式,統稱定址模式。你其實已經在 Hack 裡見過它們了:

  • 立即定址(immediate):運算元直接寫在指令裡。@21 的 21 就是。
  • 暫存器定址(register):運算元在暫存器裡。D=A 用的是 A。
  • 直接定址(direct):指令給一個記憶體位址,去那格取值。@100 之後用 M。
  • 間接定址(indirect):位址本身存在某格記憶體裡,要「轉一手」。例如 @ptr / A=M / D=M:先讀出 ptr 裡存的位址,再去那個位址取值——這正是「指標(pointer)」的硬體本質。

指令集架構(ISA)

一顆 CPU「聽得懂哪些指令、有哪些暫存器、記憶體怎麼定址」,這整套規格叫 指令集架構(ISA,Instruction Set Architecture)。它是硬體與軟體之間的契約, 也是一道關鍵的抽象界線:只要遵守同一套 ISA,Intel 和 AMD 可以用完全不同的電路去實作 x86, 而你的程式在兩家的晶片上都能跑。Hack 的 ISA,就是我們上面學的那「兩種指令」。

ℹ️
RISC vs CISC,一句話

RISC(精簡指令集):指令少、簡單、長度固定,把複雜留給軟體——ARM、RISC-V、以及我們的 Hack 都是這一派。 CISC(複雜指令集):指令多、強大、長度不一,一條指令能幹很多事——代表是桌機的 x86。 你手機(ARM/RISC)和筆電(x86/CISC)的哲學差異,一句話就是這個。

對照 x86 與 ARM

真實的 x86 或 ARM,指令數以千計、暫存器更多、還有快取、管線(pipeline)、亂序執行等一堆 Hack 沒有的把戲。 但骨架完全相同:都是馮紐曼(或類馮紐曼)架構,都有 ALU、暫存器、PC、控制單元,都跑著取指-解碼-執行週期, 都把控制流化約成跳躍,也都用記憶體映射或類似機制做 I/O。你在 Hack 上建立的每一個直覺,都會原封不動地遷移過去—— 差別只是規模與優化的細節,那些我們留到第十一部「進階計算機架構」再深入。

抽象的力量:同一個迴圈,從 Python 到 0 與 1

我們繞了一大圈,現在可以把整本書最核心的主題——抽象(第 1 章就埋下的伏筆)——用一個畫面收束起來。 以同一個再簡單不過的迴圈「把 1 加到 n」為例,看它在抽象高塔的每一層長什麼樣:

盯著這張圖多看一會兒。你寫的 for i in range(1, n+1): s += i, 最終會變成最底下那串看似毫無意義的 0 與 1,在矽晶片裡以光速流動。 中間每一層之間的「翻譯」,都是一個了不起的程式:把組合語言變成機器碼的,叫組譯器(assembler); 把 C 變成組合語言的,叫編譯器(compiler)。

而還記得本章開頭那個洞見嗎?指令也只是數字。正因如此,「組譯器」才可能存在——它不過是一個 「讀進一堆文字(組合語言)、吐出一堆數字(機器碼)」的普通程式。下一章,我們就要親手把這個看似神祕的東西拆開, 寫一個自己的組譯器,完成從「人話」到「0 與 1」的最後一哩路。

動手練習

✏️ 練習 21.1 思考

當一回控制單元,親手解碼下面兩條 16 位元的 Hack 機器碼,寫出它們對應的組合語言。 (提示:先看最高位判斷 A 指令還是 C 指令;C 指令的欄位順序是 1 11 a cccccc ddd jjj,可對照本章的三張表。)

(a)  1111110000010000
(b)  1110001100000001
看解答

(a) 1111110000010000

  • 最高位是 1 → C 指令。開頭 111 確認。
  • a 位 = 1 → 運算元用 M(而非 A)。
  • comp 的 6 位 = 110000,配上 a=1,查表是 M。
  • dest 3 位 = 010 → 存到 D。
  • jump 3 位 = 000 → 不跳。

合起來就是 D=M:把 A 指到的那格記憶體的值,讀進 D。

(b) 1110001100000001

  • C 指令,a 位 = 0。
  • comp = 001100(a=0)→ 查表是 D。
  • dest = 000 → 哪都不存。
  • jump = 001 → JGT(結果 > 0 才跳)。

合起來是 D;JGT:如果 D 的值大於 0,就跳到 A 暫存器指的位址。這正是 if (D > 0) goto ...。

✏️ 練習 21.2 實作

用 Hack 組合語言寫一段程式:把 RAM[0] 和 RAM[1] 兩個數相加,結果存進 RAM[2]。 (這次兩個運算元都來自記憶體,不是常數。想清楚 A 和 M 的關係。)

看解答

關鍵:M 永遠是「A 目前指到的那格」。所以每次要碰不同的記憶體格,都得先用 @ 改 A。 先把 RAM[0] 讀進 D,再「加上」RAM[1],最後寫到 RAM[2]:

// RAM[2] = RAM[0] + RAM[1]
@R0
D=M         // D = RAM[0]
@R1
D=D+M       // D = D + RAM[1]   (此時 M 是 RAM[1])
@R2
M=D         // RAM[2] = D

(R0、R1、R2 是內建符號,分別等於位址 0、1、2,寫起來比 @0 好讀。)

✏️ 練習 21.3 實作

用 jump 寫一個迴圈:用重複相加的方式做乘法,把 RAM[0] × RAM[1] 的結果存進 RAM[2]。 (思路:把 RAM[0] 累加 RAM[1] 次。每加一次,就把次數減一,直到次數歸零。)

看解答

這題同時用到「while(用 JEQ 在計數歸零時跳出)」和「無條件跳回(0;JMP)」:

// RAM[2] = RAM[0] * RAM[1]  (把 RAM[0] 累加 RAM[1] 次)
@R2
M=0          // product = 0

(LOOP)
    @R1
    D=M      // D = 剩餘次數
    @END
    D;JEQ    // 次數 == 0 就結束
    @R0
    D=M      // D = RAM[0]
    @R2
    M=D+M    // product += RAM[0]
    @R1
    M=M-1    // 次數 -= 1
    @LOOP
    0;JMP    // 跳回去再來一輪

(END)
    @END
    0;JMP    // 收尾:原地無限迴圈

進階思考:如果 RAM[1] 一開始就是 0,程式對嗎?(對:第一次進迴圈 D;JEQ 就直接跳到 END, product 維持 0,正確。)如果是負數呢?(會出問題——這版只處理非負數,是個值得補上的邊界情況。)

✏️ 練習 21.4 思考

假設 PC = 10,而 ROM[10] 是指令 M=D+1,執行前 D = 7、A = 200、RAM[200] = 99。 請用「取指 → 解碼 → 執行 → 回寫 → PC 前進」的格式,一步步描述這一輪結束後, D、A、RAM[200]、PC 各是多少。

看解答
  1. 取指:PC=10,CPU 讀 ROM[10],取得 M=D+1 的那串 16 位元。
  2. 解碼:控制單元判定這是 C 指令;comp=D+1、dest=M、jump=無。 因為 comp 是 D+1(只用到 D),a 位是 0,跟 A/M 無關。
  3. 執行:ALU 計算 D + 1 = 7 + 1 = 8。
  4. 回寫:dest 是 M,也就是 RAM[A] = RAM[200]。把 8 寫進去。
  5. PC 前進:沒有跳躍,PC = 10 + 1 = 11。

結算:D = 7(沒被改)、A = 200(沒被改)、RAM[200] = 8(從 99 被覆蓋成 8)、PC = 11。 注意一個容易錯的點:dest 是 M 不是 D,所以 D 本身不變,改的是記憶體那一格。

✏️ 練習 21.5 思考

承上「記憶體映射 I/O」:如果有人不小心寫了一段程式,讓 A 暫存器在迴圈裡從 16384 一路遞增、 並把每一格都 M=-1(也就是 16 個位元全 1),畫面上會發生什麼事?為什麼這說明了 「記憶體映射」既強大又危險?

看解答

16384 開始就是螢幕的記憶體區。把每一格都填成全 1,等於把整個螢幕的每個像素都點亮——畫面會變成全白。 這正是「清空 / 填滿螢幕」程式的做法。

它揭示了記憶體映射的雙面性:強大在於——操控硬體不需要特殊指令,一條普通的 M=... 就行, 程式設計模型極其統一。危險在於——既然螢幕只是「一段記憶體」,一個寫錯位址的程式 (例如指標算錯、迴圈邊界寫歪)就可能不小心踩進去,把畫面搞爛,甚至覆蓋掉它不該碰的東西。 在沒有記憶體保護的系統上,「一個小 bug 弄花整個螢幕」是家常便飯。這也是為什麼現代作業系統要費盡心力做 記憶體保護(後面的章節會談)。

✏️ 練習 21.6 挑戰

用 Python 寫一個極簡 CPU 模擬器:它要有程式計數器(PC)、幾個暫存器、一塊資料記憶體, 能吃進幾條指令,並且跑得動一個用跳躍寫成的迴圈。讓它計算 1 + 2 + … + 5, 把答案存進記憶體第 0 格,然後印出來。

(這題是本章的總驗收:如果你能寫出它,代表你真的懂了「取指-解碼-執行 + PC + 跳躍」這整套機制。)

看解答

我們設計一套迷你指令集(每條指令是一個 tuple),刻意做得像 Hack 的精神:有暫存器、有記憶體、靠改 PC 來跳躍。 注意 step() 裡的結構,完全就是「取指(讀 program[pc])→ PC 先 +1 → 解碼(看 op)→ 執行/回寫」:

class CPU:
    def __init__(self, program):
        self.program = program        # 指令記憶體(ROM):一串指令
        self.reg = {"A": 0, "D": 0}   # 暫存器
        self.ram = [0] * 16           # 資料記憶體(RAM)
        self.pc = 0                   # 程式計數器:下一條指令的位置

    def step(self):
        op, *args = self.program[self.pc]   # 取指:讀出 PC 指的指令
        self.pc += 1                        # PC 先前進(跳躍指令會再覆蓋它)

        # 解碼 + 執行 + 回寫
        if op == "SET":                     # SET reg, value:reg = value
            self.reg[args[0]] = args[1]
        elif op == "ADD":                   # ADD r1, r2:r1 = r1 + r2
            self.reg[args[0]] += self.reg[args[1]]
        elif op == "DEC":                   # DEC reg:reg = reg - 1
            self.reg[args[0]] -= 1
        elif op == "STORE":                 # STORE addr, reg:ram[addr] = reg
            self.ram[args[0]] = self.reg[args[1]]
        elif op == "JGT":                   # JGT reg, target:reg > 0 就跳
            if self.reg[args[0]] > 0:
                self.pc = args[1]           # 跳躍 = 直接改 PC!
        elif op == "HALT":
            return False                    # 停機
        return True

    def run(self):
        while self.step():                  # 不停跑指令週期,直到 HALT
            pass


# 程式:用迴圈算 sum = 5 + 4 + 3 + 2 + 1,存進 ram[0]
program = [
    ("SET", "D", 0),     # 0: sum = 0          (D 當累加器)
    ("SET", "A", 5),     # 1: i = 5            (A 當計數器)
    ("ADD", "D", "A"),   # 2: sum += i         ← 迴圈起點(位址 2)
    ("DEC", "A"),        # 3: i -= 1
    ("JGT", "A", 2),     # 4: if i > 0 goto 2  (還沒歸零就跳回去)
    ("STORE", 0, "D"),   # 5: ram[0] = sum
    ("HALT",),           # 6: 停機
]

cpu = CPU(program)
cpu.run()
print(cpu.ram[0])        # 輸出 15

跑一遍追蹤:A 從 5 開始,每輪把 A 加進 D 再把 A 減 1,當 A 減到 0 時 JGT 不成立、不再跳回, 於是落到 STORE。累加順序是 5+4+3+2+1 = 15。

體會三件事:(1) self.pc += 1 就是「PC 前進」;(2) JGT 直接覆寫 self.pc, 就是「跳躍」的全部祕密;(3) 整個 while self.step() 迴圈,就是真實 CPU 那個永不停止的指令週期。 你剛剛,用幾十行 Python,造了一台會跑迴圈的電腦。

把這一章串起來

這一章,我們從「按下執行之後到底發生什麼」一路鑽到了矽晶片的底層。回顧這趟旅程的路標:

  • 機器語言是電腦唯一真懂的語言,而最深的洞見是:指令也只是數字——程式與資料,在記憶體裡沒有本質差別。
  • 馮紐曼架構(CPU + 記憶體 + I/O)靠「儲存程式」的概念,讓換程式變成換一串數字這麼簡單。
  • CPU 有四個器官:ALU、暫存器、控制單元、PC,協作跑著永不停止的取指-解碼-執行週期。
  • Hack 機器語言只有兩種指令:A 指令(載入數值)與 C 指令(運算)。C 指令的 comp 欄位,字面上就是第 20 章那顆 ALU 的 6 個控制位元。
  • 高階的 if 與迴圈,在底層全都化約成同一招——改 PC,也就是跳躍。
  • 記憶體映射 I/O 把螢幕、鍵盤偽裝成一段記憶體,讓 CPU 不需要任何特殊指令就能操控硬體。
  • 真實的 x86、ARM 規模大得多,但骨架(ISA、定址、週期、跳躍)與 Hack 完全相通。
  • 同一個迴圈,從 Python → C → 組合語言 → 機器碼層層對應——這座抽象高塔,正是整本書的主軸。

我們已經能讀懂機器碼,也能手寫組合語言。但人總不能一輩子手動把 D=D+A 翻成 1110000010010000 吧?既然「指令也只是數字」,那就讓程式來做這件翻譯。 下一章,我們要寫一個組譯器(assembler):讀進組合語言文字,自動算出標籤位址、查表轉碼,吐出一串 0 與 1。 你即將親手打造抽象高塔中,從「人話」通往「機器」的第一道橋。
