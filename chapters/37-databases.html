資料庫系統

幾乎每一個你用過的 App,背後都躺著一個資料庫。它默默記住你是誰、你買過什麼、你的錢還剩多少。這一章,我們要學會跟它對話——用一種叫 SQL 的語言。

一個記帳本的災難

想像你開了一家小書店,決定用最樸素的方法管理庫存:開一個 books.txt 純文字檔, 一行記一本書——書名、作者、庫存、售價,用逗號隔開。一開始一切美好。直到——

  • 查詢的痛:老闆問「定價超過 500 元、而且還有庫存的書有哪些?」。你只能寫一支程式, 把整個檔案從頭到尾掃一遍,一行一行比對。十萬本書,每查一次都得讀十萬行。
  • 並發的痛:門市和網路訂單同時賣出最後一本《SICP》。兩支程式都讀到「庫存 = 1」, 都把它改成 0、各自寫回檔案——結果你賣了兩本不存在的書。這叫競態條件(race condition)。
  • 一致性的痛:程式寫到一半停電了。檔案只寫了一半,留下一行殘缺的亂碼, 整個檔案從此再也讀不回來。
  • 規模的痛:檔案大到 50 GB,記憶體根本裝不下。而且「書」和「訂單」「客戶」之間的關係, 用一堆平面文字檔,簡直是一場惡夢——你得在好幾個檔案之間手動對照編號。

這四種痛——查詢慢、並發亂、一致性崩、規模爆——不是你程式寫得爛, 而是「自己拿檔案硬幹」這條路本質上走不通。於是人們發明了一種專門的軟體, 把這些難題一次解決,它就是資料庫管理系統(DBMS,Database Management System)。

🔑
DBMS 替你扛下的四件事

持久化(persistence):安全地把資料存進磁碟,當機也不弄壞。
查詢(query):你只要宣告「我要什麼」,它自己想辦法怎麼又快又省地找出來。
交易(transaction):把一串操作綁成「全做或全不做」的一包,杜絕做一半的殘局。
並發控制(concurrency control):讓成千上萬人同時讀寫,卻像排隊一樣不打架。
後兩件事的底層,正是第 24 章作業系統那套鎖(lock)與同步機制——資料庫把它包裝成你用得起的承諾。

本章我們聚焦在最主流、也最值得先學透的一種:關聯式資料庫(relational database), 以及操作它的通用語言 SQL。為了讓你有個一以貫之的場景,整章我們都圍繞一個小例子: 一所學校——它有學生、有課程,學生會去選課。

關聯模型:把世界裝進表格

1970 年,IBM 的數學家 Edgar F. Codd 提出了一個影響至今的想法: 所有資料,都可以用最樸素的東西來表示——表格(table)。 這就是關聯模型(relational model),它的威力來自簡單與嚴謹的數學基礎(集合與關係)。

看一張「學生」表,你會發現你早就懂它了,只是還沒學會它的正式名字:

student_idnamemajorenrolled_yeargpa
1王小明資工20233.8
2林小美電機20233.5
3陳大文資工20242.9
  • 表 / 關聯(table / relation):整張表,代表一類事物的集合(所有學生)。
  • 列 / 元組(row / tuple):橫的一行,代表一個個體(一名學生)。
  • 欄 / 屬性(column / attribute):直的一行,代表一種性質(姓名、主修)。每欄有固定的資料型別。
  • 綱要(schema):這張表「長什麼樣」的設計藍圖——有哪些欄、各是什麼型別、有哪些規則。綱要是結構,資料是內容。

鍵(key):每一列的身分證

如果有兩個學生都叫「王小明」,你怎麼分辨誰是誰?你需要一個保證唯一的欄位來當身分證。 這就是主鍵(primary key):能唯一識別表中每一列的欄位(這裡是 student_id)。 主鍵不能重複、也不能是空值(NULL)。

但學校不只有學生,還有課程,而且學生會去選課——這是一種「關係」。 關聯模型怎麼表達「小明選了資料庫導論」?答案是再開一張選課表(enrollments), 裡面用編號去指向另外兩張表:

student_id ↘course_id ↘semesterscore
1CS1012024-Fall92
1EE2012024-Fall85
2CS1012024-Fall78

這裡的 student_id 與 course_id 就是外鍵(foreign key): 它們不是這張表自己的身分證,而是指向別張表主鍵的「參照」。 「選課表的第一列,student_id=1」意思就是「這筆選課,屬於學生表裡 id 為 1 的那位(王小明)」。 外鍵就像一條無形的連線,把分散在不同表的資料串起來,而且資料庫會幫你維護 參照完整性(referential integrity):它不會讓你新增一筆「選了根本不存在的課」的選課紀錄。

🔑
關聯模型的精髓:不重複存,用鍵連起來

學生的姓名、主修只存一次(在學生表)。選課表只記「哪個 id 選了哪門課」。 要看完整資訊?用外鍵把它們接起來(這個動作叫 JOIN,稍後登場)。 這種「資料只存一份、靠鍵連結」的設計,正是避免資料矛盾的關鍵——也是「正規化」整節要講的事。

SQL 第一張臉:DDL——蓋好房子的骨架

SQL(Structured Query Language)是操作關聯式資料庫的標準語言, 念作「sequel」或逐字母念都行。它有三張臉,我們一張一張認識。第一張是 DDL(Data Definition Language,資料定義語言),用來定義結構——也就是「蓋表」。

本章所有範例都能在 SQLite 上跑。SQLite 是一個極輕量、零設定的資料庫, 整個資料庫就是一個檔案,而且 Python 內建,稍後我們就用它。先來建第一張表: 

-- 建立學生表
CREATE TABLE students (
    student_id    INTEGER PRIMARY KEY,   -- 主鍵:唯一、不可空
    name          TEXT    NOT NULL,      -- 不可空:每個學生都得有名字
    major         TEXT,                  -- 可空:還沒決定主修也行
    enrolled_year INTEGER,
    gpa           REAL    CHECK (gpa >= 0.0 AND gpa <= 4.3)  -- 範圍檢查
);

每一欄都標了資料型別(data type)。SQLite 的型別相當寬鬆,核心就五種:

型別存什麼例子
INTEGER整數學號、年份、數量
REAL浮點數GPA、價格、平均
TEXT文字字串姓名、課名、日期字串
BLOB原始位元組圖片、檔案內容
NULL「沒有值」(不是 0、也不是空字串)還沒填的欄位

(在 MySQL、PostgreSQL 等其他資料庫,型別更講究,例如 VARCHAR(50)、DATE、 BOOLEAN、DECIMAL(10,2) 等,概念相通。)

約束(constraint):讓壞資料進不來

DDL 最強大的地方,是你能在綱要層級設下約束(constraint)——一旦定好, 資料庫就會幫你強制把關,任何違反規則的寫入都會被擋下。這比在應用程式裡寫一堆 if 檢查可靠得多, 因為不管哪支程式來寫,都逃不過這道關。常見約束:

  • NOT NULL:這欄一定要有值。
  • UNIQUE:這欄(或這組欄)的值不能重複(例如 email)。
  • PRIMARY KEY:主鍵 = UNIQUE + NOT NULL,每張表一個。
  • FOREIGN KEY:外鍵,值必須是另一張表裡真實存在的主鍵。
  • CHECK(條件):自訂條件,例如分數必須介於 0 到 100。
  • DEFAULT 值:沒給值時用的預設值。

把課程表與選課表也建起來,這次用上外鍵與複合主鍵:

-- 課程表:course_id 用字串當主鍵(像 'CS101')
CREATE TABLE courses (
    course_id  TEXT    PRIMARY KEY,
    title      TEXT    NOT NULL,
    credits    INTEGER DEFAULT 3,
    department TEXT
);

-- 選課表:橋表,連接 students 與 courses
CREATE TABLE enrollments (
    student_id INTEGER NOT NULL,
    course_id  TEXT    NOT NULL,
    semester   TEXT    NOT NULL,
    score      INTEGER CHECK (score >= 0 AND score <= 100),
    -- 複合主鍵:同一個學生、同一門課、同一學期,只能有一筆
    PRIMARY KEY (student_id, course_id, semester),
    -- 外鍵:把這兩欄綁回 students / courses 的主鍵
    FOREIGN KEY (student_id) REFERENCES students(student_id),
    FOREIGN KEY (course_id)  REFERENCES courses(course_id)
);
ℹ️
複合主鍵(composite key)

選課表沒有單一欄能當身分證:同一個學生會選很多課,同一門課有很多人選。 但「哪個學生 + 哪門課 + 哪個學期」三者合起來就唯一了。 用多個欄一起當主鍵,就叫複合主鍵——它順便保證了「不能重複選同一門課同一學期」。

其他常用的 DDL 還有「改結構」與「砍表」:

ALTER TABLE students ADD COLUMN email TEXT UNIQUE;  -- 加一欄
DROP TABLE enrollments;   -- ⚠️ 整張表連同資料一起刪除,不可逆!

SQL 第二張臉:DML——把資料搬進搬出

結構蓋好了,房子還是空的。DML(Data Manipulation Language,資料操作語言) 負責新增、修改、刪除資料本身。先用 INSERT 把人塞進來: 

-- 一次插一列(明確指定欄位,推薦這樣寫,順序與漏欄都不怕)
INSERT INTO students (student_id, name, major, enrolled_year, gpa)
VALUES (1, '王小明', '資工', 2023, 3.8);

-- 一次插多列
INSERT INTO students (student_id, name, major, enrolled_year, gpa) VALUES
    (2, '林小美', '電機', 2023, 3.5),
    (3, '陳大文', '資工', 2024, 2.9),
    (4, '黃淑芬', '資工', 2024, 3.95),
    (5, '張志豪', '數學', 2023, 3.2);

INSERT INTO courses (course_id, title, credits, department) VALUES
    ('CS101', '計算機概論', 3, '資工'),
    ('CS202', '資料庫系統', 3, '資工'),
    ('EE201', '電路學',     4, '電機'),
    ('MA101', '微積分',     4, '數學');

INSERT INTO enrollments (student_id, course_id, semester, score) VALUES
    (1, 'CS101', '2024-Fall', 92),
    (1, 'CS202', '2024-Fall', 88),
    (2, 'CS101', '2024-Fall', 78),
    (2, 'EE201', '2024-Fall', 85),
    (3, 'CS101', '2024-Fall', 65),
    (4, 'CS202', '2024-Fall', 95),
    (4, 'MA101', '2024-Fall', 90);

UPDATE 修改既有資料,DELETE 刪除。兩者都務必配上 WHERE:

-- 把陳大文的 GPA 改成 3.0
UPDATE students SET gpa = 3.0 WHERE student_id = 3;

-- 一次改多欄
UPDATE students SET major = '資工', gpa = 3.1 WHERE student_id = 3;

-- 刪除某一列
DELETE FROM enrollments WHERE student_id = 3 AND course_id = 'CS101';
⛔
忘了 WHERE = 全表災難

UPDATE students SET gpa = 0; 沒有 WHERE,會把每一個學生的 GPA 都歸零。 DELETE FROM students; 沒有 WHERE,會刪光整張表的資料。 這是真實世界中最常見、最痛的事故之一。動 UPDATE/DELETE 前, 先把同樣的 WHERE 拿去 SELECT 一次,確認「正好命中你要的那幾列」再下手。

SQL 第三張臉:SELECT——查詢的藝術

終於到了 SQL 最迷人的部分。SELECT 是宣告式(declarative)的: 你只說「我要什麼」,完全不必說「怎麼找」。資料庫內部有個查詢最佳化器(query optimizer), 會自己盤算最快的執行方式。這正是第 1 章「宣告式 vs 命令式知識」的活生生範例——你描述目標,機器規劃步驟。

最基本的形狀:SELECT 要哪些欄 FROM 哪張表 WHERE 條件。

SELECT * FROM students;              -- * 代表「所有欄位」

SELECT name, gpa FROM students;      -- 只要這兩欄

-- WHERE:篩選列。注意 >= 在 HTML 裡寫,實際打字就是 >=
SELECT name, gpa FROM students WHERE gpa >= 3.5;

-- 多個條件用 AND / OR;字串用單引號
SELECT name FROM students WHERE major = '資工' AND gpa > 3.0;

-- 不等於:標準 SQL 寫 <>(也可用 !=)
SELECT name FROM students WHERE major <> '資工';

WHERE 裡的好用運算子

運算子意思例子
= <> < <= > >=比較gpa >= 3.5
BETWEEN a AND b介於(含兩端)score BETWEEN 60 AND 80
IN (...)是清單之一major IN ('資工','電機')
LIKE模糊比對(%=任意多字,_=一個字)name LIKE '王%'(姓王)
IS NULL / IS NOT NULL是否為空值major IS NULL
⚠️
NULL 不是普通的值

NULL 代表「未知 / 沒有」,它有自己的詭異邏輯:gpa = NULL 永遠不成立 (連 NULL = NULL 都不為真!),因為「未知是否等於未知」答案也是未知。 要判斷空值,必須用 IS NULL / IS NOT NULL。這是初學 SQL 最常踩的坑。

ORDER BY、LIMIT、DISTINCT

-- 排序:預設由小到大(ASC),DESC 由大到小
SELECT name, gpa FROM students ORDER BY gpa DESC;

-- 取前 N 名:排序後配 LIMIT,就是「成績排行榜前 3 名」
SELECT name, gpa FROM students ORDER BY gpa DESC LIMIT 3;

-- DISTINCT:去除重複,看「總共有哪幾種主修」
SELECT DISTINCT major FROM students;

-- 欄位可以運算 / 改名(AS 取別名)
SELECT name, gpa * 25 AS percent_score FROM students;

常用內建函數

SELECT 的欄位位置可以放函數,對每一列各算一次:

SELECT
    UPPER(name)          AS upper_name,    -- 轉大寫(對英文)
    LENGTH(name)         AS name_len,      -- 字數
    ROUND(gpa, 1)        AS gpa_rounded    -- 四捨五入到小數一位
FROM students;

-- 處理 NULL:COALESCE 回傳第一個非 NULL 的值
SELECT name, COALESCE(major, '未定') AS major FROM students;
🔑
SELECT 子句的「思考順序」

雖然你寫的順序是 SELECT … FROM … WHERE … GROUP BY … HAVING … ORDER BY … LIMIT, 但資料庫實際執行的順序大致是:先 FROM(找來源)→ WHERE(篩列) → GROUP BY(分組)→ HAVING(篩組)→ SELECT(算欄位) → ORDER BY(排序)→ LIMIT(截斷)。記住這個順序,很多「為什麼這樣寫會錯」就通了。

✏️ 練習 37.1 實作

用上面的 students 表,寫一句 SQL:找出所有資工系、且 GPA 大於等於 3.5 的學生, 只回傳姓名與 GPA,並依 GPA 由高到低排序。

看解答 
SELECT name, gpa
FROM students
WHERE major = '資工' AND gpa >= 3.5
ORDER BY gpa DESC;

結果會是黃淑芬(3.95)、王小明(3.8)。 注意 WHERE 同時用 AND 串了兩個條件,ORDER BY ... DESC 讓最高分排最前面。 這四個子句(SELECT / FROM / WHERE / ORDER BY)就是日常九成查詢的骨架。

進階查詢:讓多張表開口說話

到目前為止,我們一次只問一張表。但真正的問題往往橫跨多表: 「每個學生選了哪些課、考幾分?」——學生姓名在 students, 課名在 courses,分數在 enrollments。把它們接起來,靠的是 JOIN。

INNER JOIN:取兩表的交集

JOIN 用一個條件(通常是「外鍵 = 主鍵」)把兩張表的列配對起來,拼成更寬的列。 最常用的 INNER JOIN 只保留「兩邊都配得上」的列: 

-- 列出每筆選課的「學生姓名 + 課名 + 分數」
SELECT s.name, c.title, e.score
FROM enrollments AS e
INNER JOIN students AS s ON e.student_id = s.student_id
INNER JOIN courses  AS c ON e.course_id  = c.course_id
ORDER BY s.name;

這裡用 AS 給每張表取了短別名(e、s、c), 之後用 表.欄 點出來自哪張表。ON 後面是配對條件: 「選課表的 student_id,等於學生表的 student_id」——這就是沿著外鍵把資料串回去。

四種 JOIN:差別在「配不上的列怎麼辦」

想像左右兩張表。INNER JOIN 只留交集;但有時你想保住某一邊「沒配對成功」的列(填上 NULL)。 這就分出了四種 JOIN:

LEFT JOIN 特別實用。例如「列出每一個學生與他選的課——包含一門課都沒選的」: INNER JOIN 會讓沒選課的學生整個消失(因為他在選課表裡沒有任何配對列),LEFT JOIN 則會保住他、課名欄填 NULL: 

SELECT s.name, c.title
FROM students AS s
LEFT JOIN enrollments AS e ON s.student_id = e.student_id
LEFT JOIN courses     AS c ON e.course_id  = c.course_id
ORDER BY s.name;
-- 張志豪沒選任何課 → 仍會出現,title 欄是 NULL
ℹ️
SQLite 與 RIGHT/FULL JOIN

舊版 SQLite 只支援 INNER 與 LEFT JOIN(新版 3.39+ 才補上 RIGHT/FULL)。 但其實任何 RIGHT JOIN 都能把左右表對調寫成 LEFT JOIN,所以實務上 LEFT JOIN 用得最多、最該熟練。

GROUP BY 與聚合:從「一堆列」算出「一個數」

前面的查詢都是「列進、列出」。但我們常想要統計:每門課幾個人選?平均幾分? 這要靠聚合函數(aggregate function)搭配 GROUP BY。聚合函數把「一整組列」壓縮成一個值:

函數算什麼
COUNT(*)列數(幾筆)
SUM(欄)總和
AVG(欄)平均
MAX(欄) / MIN(欄)最大 / 最小
 
-- 全校總共有幾個學生?
SELECT COUNT(*) AS total_students FROM students;

-- GROUP BY:把列依某欄分組,每組各算一次聚合
-- 「每門課的選課人數與平均分數」
SELECT course_id,
       COUNT(*)        AS num_students,
       ROUND(AVG(score), 1) AS avg_score
FROM enrollments
GROUP BY course_id
ORDER BY num_students DESC;

想像 GROUP BY course_id 把選課表依課程「分堆」:CS101 一堆、CS202 一堆…… 然後對每一堆分別數人數、算平均。沒有 GROUP BY 時,聚合函數會把整張表當成一大堆。

HAVING:篩選「組」,不是篩選「列」

WHERE 在分組之前篩列,HAVING 在分組之後篩組。 想找「平均分數低於 80 的課」,你不能用 WHERE(那時還沒算出平均),要用 HAVING: 

SELECT course_id, ROUND(AVG(score), 1) AS avg_score
FROM enrollments
GROUP BY course_id
HAVING AVG(score) < 80      -- 篩「組」:只留平均不到 80 的課
ORDER BY avg_score;
🔑
WHERE 對個體,HAVING 對群體

一句話記住:WHERE 篩的是「列」(分組前),HAVING 篩的是「組」(分組後)。 所以聚合函數(AVG、COUNT…)只能出現在 HAVING,不能出現在 WHERE。 兩者可以同時用:先 WHERE 把不要的列丟掉,再 GROUP BY,再 HAVING 篩剩下的組。

子查詢(subquery):把一個查詢塞進另一個

有時你需要「查詢的結果」當作「另一個查詢的條件」。這就是子查詢——括號裡先算出一個答案, 外層再拿來用。例如「找出 GPA 高於全校平均的學生」: 

SELECT name, gpa
FROM students
WHERE gpa > (SELECT AVG(gpa) FROM students);   -- 括號裡先算出平均

子查詢也常配 IN 用,先撈出「一串 id」,外層再依此篩選。「找出有選 CS202 的學生姓名」:

SELECT name FROM students
WHERE student_id IN (
    SELECT student_id FROM enrollments WHERE course_id = 'CS202'
);

集合運算:UNION 把兩份結果疊起來

如果兩個查詢的欄位數與型別相容,可以用 UNION 把結果上下疊在一起(自動去重複; 要保留重複用 UNION ALL)。其他集合運算還有 INTERSECT(交集)、EXCEPT(差集)。 

-- 把「資工系學生」與「有選 CS101 的學生」兩份名單合併(去重)
SELECT name FROM students WHERE major = '資工'
UNION
SELECT s.name FROM students s
JOIN enrollments e ON s.student_id = e.student_id
WHERE e.course_id = 'CS101';
✏️ 練習 37.2 實作

用 JOIN + GROUP BY 寫一句查詢:統計每個學生選了幾門課、平均分數是多少, 結果顯示學生姓名,並只列出「至少選了 2 門課」的學生,依平均分數由高到低排序。

看解答

核心:用 JOIN 把姓名接進來、用 GROUP BY s.student_id 依學生分組、 用 HAVING COUNT(*) >= 2 篩出選課數達標的人:

SELECT s.name,
       COUNT(*)              AS num_courses,
       ROUND(AVG(e.score),1) AS avg_score
FROM students AS s
JOIN enrollments AS e ON s.student_id = e.student_id
GROUP BY s.student_id, s.name
HAVING COUNT(*) >= 2
ORDER BY avg_score DESC;

用本章資料,王小明(CS101 92、CS202 88)、林小美(CS101 78、EE201 85)、 黃淑芬(CS202 95、MA101 90)都選了 2 門,會依平均排出。 注意:GROUP BY 同時放 student_id, name 是好習慣,確保 SELECT 的每個非聚合欄都在分組鍵裡。

資料庫設計與正規化:把肥表拆乾淨

現在回到設計。假設有個菜鳥圖省事,把所有東西塞進一張肥表 big:

student_idnamemajoradvisorcourse_idtitlescore
1王小明資工李教授CS101計算機概論92
1王小明資工李教授CS202資料庫系統88
2林小美電機陳教授EE201電路學85

這張表看起來能用,卻埋了三顆地雷,統稱資料異常(anomaly):

  • 更新異常(update anomaly):王小明改名,你得找出每一列有他的紀錄逐一改; 漏改一列,資料庫裡就同時存在兩個名字,自相矛盾。
  • 插入異常(insertion anomaly):新開一門還沒人選的課,你沒辦法把它存進這張表 ——因為每列都得有個學生,你被迫塞假資料。
  • 刪除異常(deletion anomaly):林小美退選了唯一的課,你刪掉那列, 結果連「電路學這門課存在」「林小美這個人存在」的資訊都一起消失了。

根源:函數相依(functional dependency)

這些異常的病根,是表裡混進了不該待在一起的資訊。要講清楚,得引入一個概念: 函數相依。我們寫 A → B,意思是「知道 A 就能唯一決定 B」。在這張肥表裡: 

student_id → name, major      (知道學號就知道姓名、主修)
major      → advisor          (知道主修就知道指導教授)
course_id  → title            (知道課號就知道課名)
(student_id, course_id) → score   (知道「誰選了哪門課」才知道分數)

看出問題了嗎?只有最後那條 score 真正依賴「完整主鍵」。其他欄各自只依賴主鍵的一部分 (name 只靠 student_id),或根本不依賴主鍵(advisor 靠的是 major)。 正規化(normalization)就是一套規則,把這些「依賴錯對象」的欄位拆出去,讓每張表都「乾淨單純」。

逐步拆解:1NF → 2NF → 3NF

第一正規化(1NF):每個欄位都是單一、不可再分的值,沒有「一格塞多筆」。

假設原本有人把選課寫成 courses = 'CS101, CS202' 塞在一格——這違反 1NF。 修法:拆成多列,一列一門課(這正是我們上面那張 big 表的樣子)。1NF 達成。

第二正規化(2NF):在 1NF 基礎上,消除「部分相依」—— 非鍵欄位不可以只依賴複合主鍵的一部分。

big 的主鍵是 (student_id, course_id)。但 name/major/advisor 只靠 student_id, title 只靠 course_id——都是部分相依。把它們各自搬出去: 

students (student_id PK, name, major, advisor)
courses  (course_id  PK, title)
enrollments (student_id, course_id, score)   主鍵 (student_id, course_id)

第三正規化(3NF):在 2NF 基礎上,消除「遞移相依」—— 非鍵欄位不可以依賴另一個非鍵欄位。

看 students 表:advisor 其實是靠 major 決定的 (student_id → major → advisor),而 major 不是鍵。 這條「繞了一手」的相依就是遞移相依——它正是更新異常的元兇(換系所的指導教授,要改一堆學生列)。 把它再拆出去:

🔑
3NF 的口訣

有個經典順口溜:每個非鍵欄位都必須描述 「the key(整把鍵)、the whole key(完整的鍵,不是一部分)、and nothing but the key(只有鍵,不繞別人) ——so help me Codd」。對應的正是 1NF(有鍵)、2NF(完整的鍵)、3NF(只有鍵)。 記住這句,九成的日常設計就不會出大錯。

反正規化:有時故意「弄髒」

正規化的代價是:資料拆得越散,查一份完整資訊就要 JOIN 越多張表,而 JOIN 是有成本的。 在讀取極度頻繁、效能至上的場景(例如報表系統、首頁排行榜), 工程師有時會刻意反正規化(denormalization):把常一起查的欄位重複存一份、 或預先算好統計值存起來,用「資料冗餘」換「查詢速度」。

這不是「設計錯了」,而是清醒的取捨:你接受了「要小心維護重複資料的一致性」這個負擔, 換來查詢飛快。原則是——先正規化把結構做對,確定有效能瓶頸、且有資料佐證時,才針對性地反正規化。

✏️ 練習 37.3 思考

有一張訂單肥表 orders(order_id, customer_name, customer_phone, product_name, unit_price, qty)。 指出它存在哪些資料異常,並把它正規化到 3NF(列出拆解後的各張表與主鍵/外鍵)。

看解答

異常:同一個客戶下多筆訂單,customer_name/phone 會重複存(更新異常:客戶改電話要改一堆列); 新客戶還沒下單就無處可存(插入異常);客戶唯一一筆訂單被刪,客戶資料也跟著消失(刪除異常)。 此外 product_name → unit_price 是遞移相依。

3NF 拆解:

customers (customer_id PK, customer_name, customer_phone)
products  (product_id  PK, product_name, unit_price)
orders    (order_id PK, customer_id FK)
order_items (order_id FK, product_id FK, qty)   主鍵 (order_id, product_id)

對應的 DDL 片段:

CREATE TABLE customers (
    customer_id    INTEGER PRIMARY KEY,
    customer_name  TEXT NOT NULL,
    customer_phone TEXT
);
CREATE TABLE products (
    product_id   INTEGER PRIMARY KEY,
    product_name TEXT NOT NULL,
    unit_price   REAL CHECK (unit_price >= 0)
);
CREATE TABLE orders (
    order_id    INTEGER PRIMARY KEY,
    customer_id INTEGER NOT NULL,
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);
CREATE TABLE order_items (
    order_id   INTEGER NOT NULL,
    product_id INTEGER NOT NULL,
    qty        INTEGER NOT NULL CHECK (qty > 0),
    PRIMARY KEY (order_id, product_id),
    FOREIGN KEY (order_id)   REFERENCES orders(order_id),
    FOREIGN KEY (product_id) REFERENCES products(product_id)
);

注意:unit_price 該存在 products(現價);若要保留「下單當時的價格」, 可以在 order_items 額外存一欄 price_at_order——這是一個合理的、刻意保留的冗餘, 因為歷史價格本來就是另一個獨立事實。

索引:為什麼有時快、有時慢

回到開頭那本書店的「查詢慢」之痛。當你下 SELECT * FROM students WHERE name = '王小明', 如果沒有特別安排,資料庫只能做全表掃描(full table scan):一列一列翻,直到找到。 十萬列就翻十萬列,$O(n)$。資料一大,慢到無法忍受。

解法是索引(index)。它就像一本書後面的「索引頁」:你想查「遞迴」這個詞在哪幾頁, 不必把整本書讀一遍,翻到索引頁(已按筆畫排好),一下就定位。資料庫索引也一樣—— 它替某個欄位額外維護一份排好序、可快速查找的結構。

底層通常是 B 樹

多數資料庫的索引,底層是一種叫 B 樹(B-tree) 的平衡樹 (更精確地說常是 B+ 樹)。你在第 29 章看過二元搜尋樹與平衡樹的概念; B 樹是它面向磁碟的「加寬版」——每個節點存很多個鍵、有很多個分支, 因此樹非常矮胖。為什麼要矮?因為從磁碟讀一塊資料很慢,樹越矮,查一筆要讀的磁碟次數就越少。 百萬筆資料,B 樹高度也才三、四層,查找是 $O(\log n)$,快得驚人。 

-- 在 name 欄上建索引,之後 WHERE name = ? 會飛快
CREATE INDEX idx_students_name ON students(name);

-- 也可建「複合索引」,加速同時用多欄的查詢
CREATE INDEX idx_enroll_course ON enrollments(course_id, semester);
⚠️
索引不是免費的,別亂建

索引讓讀變快,卻讓寫變慢——每次 INSERT/UPDATE/DELETE, 資料庫都得順手維護每一個相關索引。索引也佔額外磁碟空間。 所以原則是:替「常用來查詢/JOIN/排序」的欄建索引(尤其外鍵、常用 WHERE 條件欄); 別替很少查、或寫入超頻繁的欄亂建。主鍵與 UNIQUE 欄,資料庫通常會自動幫你建索引。

交易與 ACID:全做,或全不做

回到開頭「並發」與「一致性」的痛。考慮一個經典場景:從帳戶 A 轉帳 300 元給帳戶 B。 這在 SQL 裡是兩個動作: 

UPDATE accounts SET balance = balance - 300 WHERE id = 'A';  -- ① A 扣錢
UPDATE accounts SET balance = balance + 300 WHERE id = 'B';  -- ② B 收錢

如果 ① 執行完、② 還沒執行時程式當機/停電了,會怎樣?A 的 300 元憑空蒸發,B 卻沒收到。 錢不見了。這種「做一半的殘局」絕對不能發生在銀行系統。

解法是交易(transaction):把「一串應該被當成一個整體」的操作,用 BEGIN ... COMMIT 包起來。它要嘛全部成功生效(COMMIT), 要嘛全部當沒發生過(ROLLBACK,中途出錯時自動回復)。沒有中間狀態。 

BEGIN TRANSACTION;
    UPDATE accounts SET balance = balance - 300 WHERE id = 'A';
    UPDATE accounts SET balance = balance + 300 WHERE id = 'B';
COMMIT;     -- 兩步都成功,才一起「定案」生效

-- 若中途偵測到問題(例如 A 餘額不足),可主動撤銷:
-- ROLLBACK;  -- 撤回這個交易裡的所有改動,彷彿什麼都沒發生

ACID:交易的四個保證

交易帶來四個著名的保證,首字母合起來就是 ACID:

  • 原子性(Atomicity):全做或全不做,沒有半套(剛剛的轉帳)。
  • 一致性(Consistency):交易前後,資料庫都滿足所有約束(外鍵、CHECK…)。轉帳前後總額不變。
  • 隔離性(Isolation):多個交易並發執行時,彼此互不干擾,結果就像「一個接一個輪流跑」一樣。
  • 持久性(Durability):一旦 COMMIT,資料就永久保存,即使下一秒停電也不會丟。

並發三大災難與隔離級別

「隔離性」最微妙。如果完全不隔離,多個交易同時讀寫同一筆資料,會冒出三種典型錯亂:

  • 髒讀(dirty read):交易 B 讀到了交易 A 還沒 COMMIT 的暫時資料,結果 A 後來 ROLLBACK,B 等於讀到了「不存在的數字」。
  • 不可重複讀(non-repeatable read):同一個交易裡,前後讀同一列兩次,值卻不一樣(因為中間別人改了並 COMMIT)。
  • 幻讀(phantom read):同一個交易裡,前後用同一條件查兩次,第二次多出 / 少了幾列(別人插入/刪除了符合條件的列)。

防得越嚴,效能越低。所以 SQL 提供隔離級別(isolation level)讓你權衡—— 從寬鬆到嚴格,逐一堵住上面的災難:

隔離級別髒讀不可重複讀幻讀
Read Uncommitted可能可能可能
Read Committed✗ 杜絕可能可能
Repeatable Read✗ 杜絕✗ 杜絕可能
Serializable✗ 杜絕✗ 杜絕✗ 杜絕

最嚴的 Serializable 保證「並發跑的結果,等同於某種一個接一個的順序」,最安全但最慢。 底層怎麼做到?靠的正是第 24 章那套鎖(lock):交易在動一筆資料前先「上鎖」, 別人想動就得排隊等待。鎖用得太兇還會引發死結(deadlock)—— 兩個交易各握著對方要的鎖、互相空等。資料庫會偵測死結,挑一個交易把它 ROLLBACK 來打破僵局。 (SQLite 比較單純,以整個資料庫層級的鎖來保證序列化,代價是並發寫入受限。)

✏️ 練習 37.4 思考

演唱會搶票:某場次只剩最後 1 張票,卻有 5000 人在同一秒按下「購買」。 請說明:(a) 如果不用交易、各自「先讀庫存=1,再寫庫存=0」,會發生什麼? (b) 交易的哪幾個 ACID 性質在這裡救了你?

看解答

(a) 典型的競態條件:多個請求同時讀到「庫存 = 1」, 都認為「還有票」,於是都把庫存寫成 0、都回覆使用者「購票成功」。結果超賣: 一張票賣給了好幾個人。這正是開頭書店「賣了兩本不存在的書」的同一個病。

(b) 把「檢查庫存 → 扣庫存 → 建立訂單」包成一個交易,並讓資料庫對庫存那列上鎖:

BEGIN TRANSACTION;
    -- 讀並鎖住這列(寫法依資料庫而異,如 SELECT ... FOR UPDATE)
    -- 只有庫存 > 0 時這筆 UPDATE 才會真的改到一列
    UPDATE tickets SET stock = stock - 1
    WHERE event_id = 42 AND stock > 0;
    -- 接著依「實際改到幾列」決定成功或失敗;失敗就 ROLLBACK
    INSERT INTO orders (event_id, user_id) VALUES (42, 777);
COMMIT;
  • 隔離性:讓 5000 個交易實質上排隊,一次只有一個能改庫存,杜絕「同時讀到 1」。
  • 原子性:「扣庫存」與「建訂單」綁在一起,不會出現「扣了庫存卻沒建訂單」的鬼票。
  • 一致性:配合 CHECK (stock >= 0) 之類約束,保證庫存永不為負——也就永不超賣。
  • 持久性:成功的訂單 COMMIT 後不會因當機消失。

結論:第一個搶到鎖的交易把庫存改成 0、建好訂單;其餘 4999 個交易再去 UPDATE ... WHERE stock > 0 時,一列都改不到,程式據此回覆「售罄」。沒有交易,這種高並發場景幾乎必定出事。

從程式連資料庫:Python 的 sqlite3

SQL 很好,但真實應用是「程式」在跟資料庫對話。Python 內建了 sqlite3 模組, 不必安裝任何東西,就能完整建表、寫入、查詢。下面是一個從頭到尾可跑的完整範例: 

import sqlite3

# 連線:檔案不存在就自動建立(用 ":memory:" 則建在記憶體,程式結束就消失)
conn = sqlite3.connect("school.db")
conn.execute("PRAGMA foreign_keys = ON")   # SQLite 預設不強制外鍵,手動開啟
cur = conn.cursor()                          # cursor 是下指令、拿結果的把手

# 1) 建表(DDL)。三引號字串方便寫多行 SQL
cur.execute("""
CREATE TABLE IF NOT EXISTS students (
    student_id INTEGER PRIMARY KEY,
    name       TEXT NOT NULL,
    major      TEXT,
    gpa        REAL
)
""")

# 2) 插入資料(DML)。executemany 一次塞多列,? 是「參數佔位符」
cur.executemany(
    "INSERT INTO students (name, major, gpa) VALUES (?, ?, ?)",
    [("王小明", "資工", 3.8),
     ("林小美", "電機", 3.5),
     ("陳大文", "資工", 2.9)]
)
conn.commit()    # 別忘了 commit,否則改動不會真正寫進檔案!

# 3) 查詢(SELECT)。把使用者輸入透過參數傳入,千萬別用字串拼接
threshold = 3.0
cur.execute(
    "SELECT name, gpa FROM students WHERE gpa >= ? ORDER BY gpa DESC",
    (threshold,)          # 參數一定要是 tuple,單一參數也要寫成 (x,)
)
for name, gpa in cur.fetchall():   # fetchall 取回所有列;fetchone 取一列
    print(f"{name}: {gpa}")

conn.close()    # 關閉連線
# 輸出:
# 王小明: 3.8
# 林小美: 3.5
ℹ️
幾個 sqlite3 慣用法

commit() 才會把改動落地;conn 也能當context manager (with conn: 區塊正常結束就自動 commit、出例外就自動 rollback——這正是交易!)。 取結果用 fetchone() / fetchall(),或直接 for row in cur 逐列迭代。 其他語言(Java 的 JDBC、Node 的各種 driver)概念完全一樣:連線 → 下 SQL → 取結果。

參數化查詢:擋下 SQL injection

為什麼上面一直強調「用 ? 傳參數,別用字串拼接」?因為字串拼接會打開一個惡名昭彰的安全漏洞: SQL 注入(SQL injection)。看這個錯誤示範: 

# ⛔ 危險!絕對不要這樣寫
name = input("請輸入要查詢的姓名:")
cur.execute("SELECT * FROM students WHERE name = '" + name + "'")

如果使用者乖乖輸入「王小明」,組出來的 SQL 沒問題。但如果他輸入這一串: 

' OR '1'='1

那麼拼接後的 SQL 變成:

SELECT * FROM students WHERE name = '' OR '1'='1'

'1'='1' 永遠成立,於是整張表的資料全被撈出來!更狠的輸入甚至能 '; DROP TABLE students; -- 把你整張表刪掉。攻擊者「把資料當成程式碼注入」的這一招, 多年來是 Web 應用最常見的重大漏洞之一(這部分我們會在第 36 章資訊安全深入)。

正確做法:用參數化查詢。把值交給 ?,資料庫驅動會把它當成純資料, 絕不會被解讀成 SQL 語法——再怎麼刁鑽的輸入,都只是一個普通的字串值:

# ✅ 安全:用 ? 佔位,值另外傳
name = input("請輸入要查詢的姓名:")
cur.execute("SELECT * FROM students WHERE name = ?", (name,))
⛔
一條鐵律

永遠不要用字串拼接把使用者輸入塞進 SQL。永遠用參數化查詢(? 或具名參數)。 這不是「比較好的寫法」,而是「唯一可接受的寫法」。它同時還順便幫你處理引號跳脫、型別轉換, 一舉數得。把這條刻進肌肉記憶。

✏️ 練習 37.5 實作

用 Python 的 sqlite3 寫一支完整小程式:在記憶體資料庫(:memory:)裡建一張 books(id, title, price, stock) 表,插入至少 3 本書,然後用參數化查詢 找出「價格低於某金額、且還有庫存」的書並印出來。

看解答 
import sqlite3

conn = sqlite3.connect(":memory:")   # 純記憶體,測試最方便
cur = conn.cursor()

cur.execute("""
CREATE TABLE books (
    id    INTEGER PRIMARY KEY,
    title TEXT NOT NULL,
    price REAL CHECK (price >= 0),
    stock INTEGER NOT NULL DEFAULT 0
)
""")

cur.executemany(
    "INSERT INTO books (title, price, stock) VALUES (?, ?, ?)",
    [("SICP", 600, 3),
     ("龍書(編譯器)", 450, 0),
     ("CLRS 演算法", 700, 5),
     ("Python 入門", 350, 8)]
)
conn.commit()

max_price = 500
cur.execute(
    "SELECT title, price, stock FROM books WHERE price < ? AND stock > 0 ORDER BY price",
    (max_price,)
)
for title, price, stock in cur.fetchall():
    print(f"{title}:{price} 元,庫存 {stock}")

conn.close()
# 輸出:
# Python 入門:350 元,庫存 8
# (龍書雖然便宜但 stock=0 被排除;SICP/CLRS 太貴被排除)

重點全在那句 WHERE price < ? AND stock > 0 與 (max_price,): 條件值透過參數傳入,既安全又乾淨。:memory: 資料庫在 close() 後就消失, 很適合拿來練習與寫測試。

當關聯模型不夠用:NoSQL 與半結構化資料

關聯模型優雅又嚴謹,但它有個前提:你的資料得能整齊地裝進固定欄位的表格。 現實中常有不那麼乖的資料:每筆商品的規格欄位都不一樣、社群貼文巢狀著留言又巢狀著回覆、 資料量大到一台機器塞不下……這時人們轉向半結構化資料與 NoSQL。

半結構化資料:JSON 與 XML

半結構化(semi-structured)資料沒有死板的表格綱要,而是用「自帶標籤」的格式描述自己。 最主流的是 JSON(JavaScript Object Notation): 

{
  "student_id": 1,
  "name": "王小明",
  "major": "資工",
  "courses": [                         // 巢狀!一個學生直接帶著他的選課陣列
    { "id": "CS101", "score": 92 },
    { "id": "CS202", "score": 88 }
  ]
}

看到差別了嗎?關聯模型要把選課拆到另一張表用外鍵連;JSON 直接把它巢狀進來, 一個物件就是一筆完整資料。較老的 XML 用成對標籤(<name>王小明</name>) 表達同樣的階層結構。現代 SQL 資料庫其實也都支援存取 JSON 欄位了,界線越來越模糊。

NoSQL 四大家族

NoSQL(意為「不只是 SQL」)是一群「非關聯式」資料庫的統稱,各擅勝場,一句話認識它們:

  • 文件資料庫(document store),代表 MongoDB:直接存一份份 JSON 風格的文件,結構彈性,適合每筆欄位都可能不同的資料。
  • 鍵值資料庫(key-value store),代表 Redis:就是一個超大、超快的字典(鍵 → 值),常拿來做快取與工作階段(session)。
  • 欄式資料庫(column-family / wide-column),代表 Cassandra:資料按「欄」而非「列」儲存,極擅長海量寫入與大規模分析。
  • 圖資料庫(graph database),代表 Neo4j:把資料存成「節點 + 邊」,專門對付社群網路、推薦、路徑這類關係密集的問題(正是第 15 章的圖論在工業界的化身)。

CAP 定理:分散式系統的「不可能三角」

當資料大到要分散在很多台機器上,就會撞上一個深刻的限制:CAP 定理。它說, 一個分散式系統最多只能同時保證三者中的兩者:

所以,該用 SQL 還是 NoSQL?

面向關聯式(SQL)NoSQL
資料結構固定綱要、表格彈性、可巢狀、隨筆不同
關係 / JOIN強項,外鍵 + JOIN較弱,常靠應用端處理或反正規化
交易 / ACID完整支援,強一致多偏向「最終一致」,部分支援交易
水平擴展(多機)較難(傳統上偏垂直擴展)天生為分散式設計,易橫向擴展
查詢語言標準 SQL,通用各家自有 API/語法,不統一
最適合資料關係複雜、需強一致(金融、ERP、後台)海量、結構多變、可容忍弱一致(日誌、IoT、社群動態、快取)
💡
務實的建議

對絕大多數應用,從關聯式資料庫開始,通常是對的選擇:它成熟、ACID 保證強、SQL 通用、 資料一致性最省心。等到你真的遇到關聯模型扛不住的特定問題(超大規模、極彈性結構), 再針對那個部分引入合適的 NoSQL。很多大型系統其實是兩者並用:核心交易用 SQL, 快取用 Redis,搜尋與日誌用別的——這叫「polyglot persistence(多元儲存)」。

真實世界的最後一哩

離開教室前,幾個你一定會在業界聽到的詞,各用一句話收進口袋:

  • OLTP vs OLAP:OLTP(線上交易處理)是「日常一筆筆的小讀寫」——下單、轉帳、登入,要求快與一致; OLAP(線上分析處理)是「對海量歷史資料做大範圍統計分析」——本季各區營收趨勢,要求掃得廣、算得動。
  • 資料倉儲(data warehouse):專為 OLAP 打造的大型資料庫,把各系統的歷史資料集中、整理、存起來, 供分析與決策使用(常搭配反正規化與欄式儲存來加速分析)。
  • ORM(物件關聯對映):像 Django ORM、SQLAlchemy、Hibernate 這類工具, 讓你用程式語言的物件來操作資料庫,自動幫你產生 SQL——寫起來更順手, 但仍建議懂底層 SQL,才不會寫出又慢又怪的查詢(這呼應第八部軟體工程「站在抽象之上,但別忘了抽象底下」的精神)。

動手練習

SQL 是「越寫越通」的技能。下面幾題從建表一路到設計,請務必先打開 SQLite(或任何資料庫)親手敲過。

✏️ 練習 37.6 實作

從零寫出 students 與 enrollments 兩張表的完整 CREATE TABLE: 學生要有主鍵、姓名不可空;選課表要有複合主鍵、分數要有 0~100 的範圍檢查,並用外鍵連回學生表。

看解答 
CREATE TABLE students (
    student_id INTEGER PRIMARY KEY,
    name       TEXT    NOT NULL,
    major      TEXT,
    gpa        REAL    CHECK (gpa >= 0 AND gpa <= 4.3)
);

CREATE TABLE enrollments (
    student_id INTEGER NOT NULL,
    course_id  TEXT    NOT NULL,
    semester   TEXT    NOT NULL,
    score      INTEGER CHECK (score >= 0 AND score <= 100),
    PRIMARY KEY (student_id, course_id, semester),
    FOREIGN KEY (student_id) REFERENCES students(student_id)
);

檢查清單:主鍵 ✓、NOT NULL ✓、複合主鍵 ✓、CHECK 範圍 ✓、外鍵 ✓。 這五樣是日常建表的標準配備。

✏️ 練習 37.7 實作

用本章的學校資料寫一句查詢:找出「選課平均分數最高的那一門課」, 顯示課名與平均分數。(提示:JOIN 取課名、GROUP BY 算平均、ORDER BY 配 LIMIT 取第一名。)

看解答 
SELECT c.title, ROUND(AVG(e.score), 1) AS avg_score
FROM enrollments AS e
JOIN courses AS c ON e.course_id = c.course_id
GROUP BY e.course_id, c.title
ORDER BY avg_score DESC
LIMIT 1;

思路:先把選課表依課程分組算平均,JOIN 進來課名,最後排序取榜首。 如果可能「並列第一」想全取,則要改用子查詢比對最大平均值,而非 LIMIT 1—— 這是一個值得想想的延伸:HAVING AVG(e.score) = (SELECT MAX(...) FROM ...)。

✏️ 練習 37.8 挑戰

設計一個小型圖書館借閱系統的資料庫(達到 3NF),至少包含「書」「會員」「借閱紀錄」。 畫出表結構(欄位 + 主鍵/外鍵),並寫出三個查詢: (1) 某會員目前還沒還的書;(2) 最熱門(被借次數最多)的前 3 本書;(3) 逾期未還的借閱紀錄。

看解答

表結構(一本實體書可有多個館藏副本,故拆出 copies;借閱發生在「副本」上):

CREATE TABLE books (
    book_id   INTEGER PRIMARY KEY,
    title     TEXT NOT NULL,
    author    TEXT,
    isbn      TEXT UNIQUE
);
CREATE TABLE members (
    member_id INTEGER PRIMARY KEY,
    name      TEXT NOT NULL,
    phone     TEXT
);
CREATE TABLE copies (                       -- 每本書的實體館藏副本
    copy_id INTEGER PRIMARY KEY,
    book_id INTEGER NOT NULL,
    FOREIGN KEY (book_id) REFERENCES books(book_id)
);
CREATE TABLE loans (                         -- 借閱紀錄
    loan_id     INTEGER PRIMARY KEY,
    copy_id     INTEGER NOT NULL,
    member_id   INTEGER NOT NULL,
    loan_date   TEXT NOT NULL,               -- 借出日 'YYYY-MM-DD'
    due_date    TEXT NOT NULL,               -- 應還日
    return_date TEXT,                        -- 還書日;NULL = 尚未歸還
    FOREIGN KEY (copy_id)   REFERENCES copies(copy_id),
    FOREIGN KEY (member_id) REFERENCES members(member_id)
);

(1) 某會員(id=7)還沒還的書:

SELECT b.title, l.due_date
FROM loans l
JOIN copies c ON l.copy_id = c.copy_id
JOIN books  b ON c.book_id = b.book_id
WHERE l.member_id = 7 AND l.return_date IS NULL;

(2) 被借次數最多的前 3 本書:

SELECT b.title, COUNT(*) AS times_borrowed
FROM loans l
JOIN copies c ON l.copy_id = c.copy_id
JOIN books  b ON c.book_id = b.book_id
GROUP BY b.book_id, b.title
ORDER BY times_borrowed DESC
LIMIT 3;

(3) 逾期未還(應還日已過、卻還沒還):

SELECT m.name, b.title, l.due_date
FROM loans l
JOIN members m ON l.member_id = m.member_id
JOIN copies  c ON l.copy_id   = c.copy_id
JOIN books   b ON c.book_id   = b.book_id
WHERE l.return_date IS NULL
  AND l.due_date < DATE('now');     -- DATE('now') 是 SQLite 的今天

觀察:這三個查詢全都靠 return_date IS NULL 來表達「在借中」、靠多重 JOIN 串起 loans→copies→books,正是本章 JOIN + 聚合 + NULL 處理的綜合演練。 用 copies 把「書名」與「實體副本」分開,是為了正確支援「同一本書有多本」——這也是 3NF 的體現。

✏️ 練習 37.9 思考

你的查詢 SELECT * FROM orders WHERE customer_id = ? 在百萬筆訂單上慢到爆。 你想建一個索引解決。請說明:該在哪一欄建索引?建了之後「讀」與「寫」各會受什麼影響? 有沒有可能建了反而幫倒忙?

看解答

應在 customer_id 欄建索引(CREATE INDEX idx_orders_cust ON orders(customer_id);), 因為它是 WHERE 的篩選條件。建了之後:

  • 讀:這類查詢從 $O(n)$ 全表掃描變成 $O(\log n)$ 的索引查找,巨幅加速。
  • 寫:每次對 orders 做 INSERT/UPDATE customer_id/DELETE, 資料庫都要順手更新這個索引,寫入略變慢;索引本身也佔額外磁碟空間。

可能幫倒忙的情況:(a) 若 customer_id 重複度極高 (例如全部訂單只屬於少數幾個大客戶,一查就回傳半張表),索引幫助有限,最佳化器甚至會放棄用它直接全掃。 (b) 若這張表寫入極度頻繁、查詢卻很少,維護索引的成本可能超過它省下的查詢時間。 所以索引要針對實際查詢模式來建,並用資料庫的 EXPLAIN 觀察它到底有沒有被用上。

把這一章串起來

我們從一本會釀災的 books.txt 出發,走完了關聯式資料庫的全貌。把這趟旅程收進腦海:

  • 為什麼需要資料庫:檔案硬幹會在查詢、並發、一致性、規模上四面楚歌;DBMS 提供持久化、查詢、交易、並發控制。
  • 關聯模型:用表(列/欄)裝資料,用主鍵當身分證、外鍵作連線,用綱要定義結構;多對多關係靠一張橋表。
  • SQL 三張臉:DDL 蓋結構(CREATE TABLE + 約束)、DML 改資料(INSERT/UPDATE/DELETE)、SELECT 查資料。
  • 進階查詢:JOIN 串多表(INNER/LEFT 最常用)、GROUP BY 配聚合做統計、HAVING 篩組、子查詢、UNION。
  • 正規化:用 1NF→2NF→3NF 消除更新/插入/刪除異常,讓每個事實只存一份;必要時才為效能反正規化。
  • 索引:B 樹讓查詢 $O(\log n)$,但會拖慢寫入、佔空間,要針對查詢模式精準地建。
  • 交易與 ACID:用 BEGIN/COMMIT/ROLLBACK 保證「全做或全不做」;隔離級別權衡並發災難(髒讀/不可重複讀/幻讀),底層靠鎖。
  • 從程式連:Python sqlite3 完整可跑;永遠用參數化查詢擋下 SQL injection。
  • NoSQL 與 CAP:資料太彈性或太大時改用文件/鍵值/欄式/圖資料庫;分散式系統 C/A/P 三選二;多數場景「先 SQL,有特定需求再 NoSQL」。

資料庫是幾乎所有真實系統的地基。你現在不只會「用」它,還理解了它為什麼這樣設計—— 從關聯代數的優雅,到交易的鐵血保證,再到分散式世界的無奈取捨。下一步,把這些 SQL 拿去接上你寫的程式, 你就擁有了打造一個真正「會記得事情」的應用的能力。
