走到這裡,你已經很不一樣了。在第十八部前面那幾章裡,你學會了在終端機裡尋寶、剝開層層編碼 (第 69 章)、破譯古典與現代密碼(第 70 章)、對授權靶機做 Web 偵察、讀封包、挖出檔案裡的隱寫 (第 71 章),甚至讀懂組合語言、把有漏洞的程式打出一面旗來(第 72 章)。 你能解題了——這本身就是一件很多人一輩子沒摸到的事。

想像一個畫面:深夜兩點,你和隊友擠在一個 Discord 語音頻道裡打一場線上 CTF。 記分板上,你們隊離前十名只差一題。那是一道 Pwn 題:連到伺服器,它會丟給你三十道隨機的算式, 每道限你一秒內答完,全對才吐出旗。你用手算?第三題就超時了。 這時候,真正能救你的,不是更強的心算,而是一支幾行的 Python 腳本—— 它連上去、讀題、算、送回去,三十題在零點幾秒內全部答完,旗到手。

📜
CTF 從哪來?

「奪旗(Capture The Flag)」這個詞借自野外遊戲:兩隊各守一面旗、互相搶奪。 資安界的 CTF 公認起源於 1996 年的駭客大會 DEF CON——一群人把攻防做成競賽, 用「搶到對方系統裡的一段秘密字串(flag)」來計分。三十年後,它長成了全世界上千場、 從國中生到職業戰隊都在玩的學習與競技文化。你正要加入的,是一個有歷史、有傳承的社群。

自動化解題:當手動再也不夠快

資安人有一句半開玩笑的信條:「會重複三次的事,就該寫成腳本。」 在 CTF 裡,這句話不是偷懶,而是必要。很多題目你非自動化不可:

  • 連續答題型:伺服器丟給你 30、100、甚至 1000 道題,每題有時間限制,人手根本來不及。
  • 互動式 Pwn / Crypto:你得跟一個遠端程式一來一往——它問你一個值,你算出回應、送回去,反覆好幾輪,最後它才漏出旗。
  • 需要精準位元組的 payload:Pwn 題要送進去的,是一串混著數字、位址、不可見字元的二進位資料,手動在鍵盤上根本打不出來(第 72 章你已經見過)。

這些題目的共同點是:你要寫一個程式,去跟另一個程式(網路服務)一問一答。 而要做到這件事,你需要兩樣東西:一點點 socket 知識,和一個叫 pwntools 的好工具。

先打地基:Python socket(呼應第 25 章)

還記得第 25 章講網路時,我們說過——網路上的程式溝通,底層靠的是 socket(套接字): 你開一條到「某個 IP 的某個埠」的連線,然後就能在這條管子上收位元組、送位元組。 Python 標準庫的 socket 模組,讓這件事只要幾行: 

# 最小可用的 socket 客戶端:連到題目伺服器,收一行、送一行
import socket

# 題目通常會給你 host 與 port,例如 nc ctf.example.com 9000
s = socket.create_connection(("ctf.example.com", 9000))

data = s.recv(1024)          # 收最多 1024 個位元組(回來的是 bytes)
print(data.decode())         # 解碼成文字印出來

s.sendall(b"42\n")           # 送出答案;注意要自己補上換行 \n
print(s.recv(1024).decode()) # 看看伺服器怎麼回

s.close()                    # 收工,關閉連線

這就是「nc(netcat)能做的事」的程式版。但裸用 socket 有點煩: 你得自己處理「一直收到看到某個提示字為止」「收一整行」這類瑣事(看看第 70、72 章原始教材裡那個 手寫的 recvline() 迴圈就知道了)。於是,CTF 圈發明了一個把這些麻煩通通包好的瑞士刀。

升級裝備:pwntools

pwntools 是 CTF 圈幾乎人手一套的 Python 函式庫。它把「跟服務互動」「組裝二進位 payload」 「跟本機程式對話」這些事,做成了極順手的 API。安裝(建議在第 72 章那套 WSL/Linux 環境裡): 

pip install pwntools

最常用的,就是這四個動作——它們幾乎能解決所有「互動題」:

pwntools 動作它做什麼對應的 socket 苦工
io = remote(host, port)連到遠端服務socket.create_connection
io.recvuntil(b"= ?")一直收,直到出現這段字自己寫 while 迴圈一個個 byte 收
io.recvline()收一整行手寫 recvline()
io.sendline(b"42")送出一行(自動補換行)s.sendall(b"42\n")

除此之外,pwntools 還能用 process("./prog") 跟本機程式對話(逆向、Pwn 常用)、 用 p32()/p64() 把數字打包成第 72 章那種小端位元組、用 cyclic() 產生定位溢位偏移的字串…… 這些是第 72 章 Pwn 的延伸武器。本章我們聚焦最通用的一招:自動跟服務一問一答。

實戰:寫一個自動答題腳本

來解一個經典的「連續答題」題型。假設伺服器一連線就開始丟題,格式長這樣,答對就進下一題, 全部答完就把旗給你: 

第 1 題: 12 + 7 = ?
（你要在這裡送回 19）
第 2 題: 45 - 18 = ?
（送回 27）
...
第 30 題: 8 * 6 = ?
（送回 48）
恭喜! FLAG{pwntools_automate_the_boring_stuff}

手動?三十題、每題限時一秒,沒人辦得到。但程式可以。先看純標準庫 socket 版 ——它把第 25 章的網路知識直接用上,不依賴任何外部套件: 

# 自動答題(純 socket 版)——不需要 pwntools
import socket, re

def recvline(s):
    """一個位元組一個位元組地收,直到收到一整行(以 \n 結尾)。"""
    buf = b""
    while not buf.endswith(b"\n"):
        ch = s.recv(1)
        if not ch:           # 連線被對方關掉了
            break
        buf += ch
    return buf.decode("utf-8", "replace").strip()

s = socket.create_connection(("127.0.0.1", 9000))
flag = None
while True:
    line = recvline(s)
    if not line:
        break
    print(line)                       # 把伺服器說的話印出來,方便除錯
    if "FLAG{" in line:               # 拿到旗了!
        flag = re.search(r"FLAG\{[^}]*\}", line).group(0)
        break
    if "= ?" in line:                 # 這是一道算式題
        expr = line.split(":")[1].replace("= ?", "").strip()  # 取出 "12 + 7"
        ans = eval(expr)              # 算出答案
        s.sendall((str(ans) + "\n").encode())                 # 送回去
s.close()
print("\n取得的旗:", flag)

同樣一件事,用 pwntools 寫,會短得多、也讀得更順——因為收行、送行的苦工都被它包掉了: 

# 自動答題(pwntools 版)——一樣的邏輯,更短
from pwn import *

io = remote("127.0.0.1", 9000)
while True:
    line = io.recvline(timeout=5).decode("utf-8", "replace").strip()
    if not line:
        break
    log.info(line)                    # pwntools 的漂亮日誌輸出
    if "FLAG{" in line:
        log.success(line)
        break
    if "= ?" in line:
        expr = line.split(":")[1].replace("= ?", "").strip()
        io.sendline(str(eval(expr)).encode())   # sendline 自動補換行
io.close()

看出 pwntools 的價值了嗎?同樣的解題邏輯,你不必再手寫那個逐位元組的 recvline。 程式跑起來的那一刻——三十題在你眨眼之間答完、旗安靜地出現在終端機上—— 你會第一次體感到「自動化」這個詞的重量。

⚠️
eval 很方便,但要知道它的危險

上面用 eval(expr) 直接算字串算式,在 CTF 解題腳本裡是常見的偷懶—— 因為「題目伺服器」是比賽方提供的、格式可預期的。但請記住:對不受信任的輸入用 eval, 等於把你自己的電腦交給對方(它能塞進任意 Python 程式碼)。這正是第 34 章「永遠不要信任邊界外的輸入」 的反面教材。在正式程式裡,該用安全的解析(例如自己拆出數字與運算子),別把 eval 帶出 CTF 的沙箱。

自動化是貫穿全部類別的槓桿

別把 pwntools 當成「只有 Pwn 才用」。只要題目需要跟一個程式或服務多次互動,這套技能就用得上: Crypto 的互動式預言機(oracle)、Misc 的連續答題機器人、Web 的自動化請求(雖然 Web 更常用 requests)、 Reverse 後拿腳本去跑本機 binary……自動化是把你從「重複的手工勞動」裡解放出來的槓桿, 讓你的腦力專注在真正難的那一步上。

參加 CTF 比賽

會自動化之後,你該去打真正的比賽了。靶場(像 picoGym、OverTheWire)能養手感, 但限時、有對手、有記分板的正式比賽,是完全不同的體驗——它逼出你的速度、抗壓性與團隊默契。 先認識兩種主要賽制。

兩種賽制:Jeopardy 與 Attack-Defense

絕大多數你會遇到的比賽,是 Jeopardy(解題式):像益智節目一樣,主辦方擺出一面題板, 分成 Web、Crypto、Pwn 等類別,每題標好分數(越難越高分)。你解出題目、拿到 flag、提交得分。 新手友善,單人或小隊都能玩,picoCTF 就是這種。

另一種是 Attack-Defense(攻防式):每隊拿到一台一模一樣、跑著同樣(有漏洞)服務的伺服器。 你要一邊找出漏洞去打別隊、偷他們的 flag 得分,一邊修補自己的服務、別被別人打進來。 它更接近真實世界的紅藍對抗,強度高、節奏快,通常是進階戰隊在玩(像 DEF CON CTF 決賽)。

怎麼找比賽

別擔心「沒比賽可打」——全世界幾乎每個週末都有 CTF。問題只是挑哪一場。幾個給新手的起點:

平台 / 賽事是什麼適合誰
picoCTF卡內基美隆大學辦的教育型 CTF,題目超友善;比賽結束後 picoGym 全年開放重練所有新手的第一場,強烈推薦
AIS3 myFirst CTF台灣 AIS3 計畫辦的入門賽,中文友善、難度親民,常在暑期台灣學生入門首選
CTFtime全球 CTF 的「行事曆」與戰隊排名站,列出每場比賽的時間、權重、報名連結找下一場比賽、追蹤戰績
校際 / 全國賽各國各校自辦的賽事(台灣如 HITCON、各大學資安社)有一點底子後的進階挑戰

新手的標準路線很清楚:先把 picoGym 各類別的入門題刷個三五十題養手感(這正是前面幾章帶你練的), 然後上 CTFtime 找一場標著「beginner-friendly」的線上賽,揪幾個同學組隊下場。 第一場你大概只解得出一兩題——那完全正常。重點是你下場了。

組隊與分工:沒有人是全能的

前面說過,CTF 是群體運動。一支理想的小隊,會讓成員依六大類別的專長分工, 各守一塊、互相補位。一個典型的分配長這樣:

類別這個人負責什麼主要吃哪幾章的底
Web網站漏洞:注入、認證繞過、IDOR、SSRF……第 52 章(Web 安全)、第 71 章
Crypto密碼學:古典密碼、RSA 弱參數、雜湊、編碼第 35 章(密碼學)、第 70 章
Reverse逆向:讀組語、用 Ghidra/gdb 還原邏輯、解 crackme第 21、22、42 章、第 72 章

Forensics鑑識:封包分析、檔案修復、隱寫、記憶體取證第 53 章(鑑識)、第 71 章
Misc雜項:雜七雜八的腦洞題、自動化、OSINT本章(自動化)、什麼都沾一點

新手的分工:先當「主攻一類 + 共用 Misc」

四五個人的學生隊,不必每類都配一個專家。比較實際的做法是:每個人挑一類當主攻 (你最喜歡、最有把握的那類),Misc 與自動化大家一起上;再指定一個人當「記分板協調者」—— 盯著還有哪些題沒人碰、誰卡住了需要支援。CTF 最迷人的時刻之一,就是一道題被兩個不同專長的人 「Web 找到入口、Crypto 解開最後一層」接力打下來。1+1 在 CTF 裡常常大於 2。

比賽中的時間與心態管理

正式比賽通常 24 到 48 小時,是一場耐力賽。幾條老手用血淚換來的守則:

  • 先掃一遍全部題目,挑「投報率」最高的:開賽別急著埋頭打第一題。花十分鐘把所有題目讀過,先解你最有把握、分數又不錯的。記分板是動態的——別在一題上鑽到天亮。
  • 設「卡關停損點」:一題卡超過(比如)90 分鐘還沒進展,就先放下、換一題,把卡點寫進共用筆記。常常你打完別題回來,答案突然就浮現了。
  • 邊解邊記:把你試過什麼、哪條路不通、目前進度寫在隊伍共用文件(HackMD/Discord)。這既避免隊友重複勞動,也是你賽後 writeup 的草稿。
  • 照顧身體:48 小時不睡的硬撐是新手最常見的錯。睡飽、吃飯、輪班,清醒的腦袋遠比多熬的兩小時值錢。
  • 沒解出來,不等於失敗:第一場能解一兩題就很好。比賽結束後讀別人的 writeup、把你卡住的題弄懂——那才是 CTF 真正的學習發生的地方。

writeup 文化:把過程寫下來,回饋社群

如果這一章只能記住一件事,我希望是這個:寫 writeup。

writeup(解題紀錄),就是你把「這題我怎麼解出來的」從頭到尾寫成一篇文章。 它是 CTF 社群的血液——你今天能靠 Google 搜到別人的 writeup 學會一招, 是因為昨天有人願意把過程寫下來分享。而現在,輪到你回饋了。

為什麼一定要寫

  • 整理你自己的知識:解出來的當下你以為懂了,但逼自己「寫到別人能照著重現」時,你會發現很多地方其實只是矇對的。寫 writeup 是把「矇對」變成「真懂」的過濾器。
  • 回饋社群:你免費學了別人的 writeup,寫一篇還回去,這個圈子才轉得下去。這正是第 58 章說的開源精神,在資安圈的版本。
  • 建立個人品牌:一個整理乾淨、持續更新的 writeup 部落格或 GitHub,是你最強的履歷。它讓面試官看見你怎麼思考、怎麼卡關又怎麼爬出來——這比「我會資安」三個字有說服力一百倍(呼應第 58 章的作品集觀念)。

一份好的 writeup 長什麼樣

好的 writeup 有固定的骨架。核心是讓讀者(包括三個月後的你)能照著步驟重現你的解法。 順序大致是:題目資訊 → 觀察 → 嘗試 → 解法 → flag → 學到什麼。 直接給你一份可以複製來用的範本(改寫自第十八部原始教材的 writeup 範本):

    # 例:你下的指令

🔑
writeup 的黃金標準:可重現

判斷一篇 writeup 好不好,只有一個問題:「一個沒解過這題的人,照著你寫的能不能解出來?」 這意味著:指令要完整(別省略關鍵參數)、腳本要能跑(別只貼片段)、思路要連貫(說清楚「為什麼這樣試」而不只是「我這樣做」)。 尤其別漏了失敗的嘗試——「我先試了 X 但不行,因為……,於是改試 Y」往往比直接給答案更有教學價值。

發表在哪裡

寫完了,要讓人看見才有意義(這也是第 58 章說的「發表,你的努力才會產生複利」)。常見平台:

  • 個人部落格:最能累積個人品牌。用 GitHub Pages、Hugo、Hexo 等架一個靜態部落格,把 writeup 一篇篇放上去。
  • GitHub repo:開一個 ctf-writeups 倉庫,用 Markdown 寫、按比賽分資料夾。這同時練到第 18 章的 git——每篇 writeup 就是一次 commit,你的解題史一目了然。
  • HackMD:即時協作的 Markdown 編輯器,超適合比賽當下全隊共筆,賽後稍微整理就能直接發表。
⚖️
發表的禮貌:別在比賽還沒結束就公開

有兩條社群默契要守:(1) 正在進行中的比賽,絕對不要公開解法或 flag——那等於作弊、毀掉別人的比賽, 多數賽事明文禁止。等比賽結束再發。(2) 如果題目改編自真實系統或第三方素材,寫清楚出處、尊重授權。 回饋社群是好事,但要用社群認可的方式回饋。

自辦 CTF:從解題者變成出題者

當你解過夠多題、寫過夠多 writeup,有一天你會冒出一個念頭:「換我來出題目考考別人。」 這是一個巨大的跨越——能出一道好題,代表你對那個漏洞的理解,已經深到能反過來設計它。 這也是第十八部原始課程的「結業關」:自己辦一場班際 CTF。

怎麼設計一道好題

出題比解題難。一道好題要同時滿足好幾個條件,缺一個都會讓玩家罵聲連連:

  • 難度要評估準:你覺得「簡單」的題,對玩家可能極難(知識的詛咒)。出完一定要找一個沒看過解答的人試解,用他卡住的地方校準難度。一場好的比賽,難度要有梯度——從人人能解的入門題,到區分高手的難題。
  • 唯一解、且解得出來:flag 要明確,解法路徑要通。最糟的題是「非預期解」——玩家用你沒想到的偏門打進來、或根本無解卡死全場。多找人試解,就是在抓這個。
  • 防作弊:flag 不能被猜到、不能藏在前端原始碼裡裸奔、附件不能不小心包進內部解答。動態 flag(每隊不同)能防「互相抄 flag」。
  • 線索充足、不靠通靈:好題考的是「你會不會這個技術」,不是「你猜不猜得到出題者在想什麼」。適時給提示(hint)。
⛔
出題者的紅線:別把真實犯罪當題目

出題給了你「設計攻擊」的權力,但這條線必須守死:(1) 題目要在隔離、授權的環境裡(你自己的容器、CTFd 平台), 絕不能拿真實的、別人的、線上的系統當靶子。(2) 不要教真正能拿去犯罪的完整攻擊鏈—— CTF 的漏洞是刻意為教學設計的縮影,目的是讓人理解原理、學會防禦,不是給人現成的犯罪工具。 (3) 題目素材不要涉及真實個資、不要冒犯特定對象。 這呼應第 34、41 章一再強調的:技術中立,用途有責;技術力越強,越要守住倫理界線。

用 CTFd + Docker 架站

有了題,你需要一個平台來「上架題目、收 flag、即時計分、顯示記分板」。 最主流的開源選擇是 CTFd。而最省事的部署方式,是用 Docker (還記得第 40 章講的容器嗎?它把整套服務連同環境打包,讓你一行指令就跑起來): 

# 把 CTFd 抓下來
git clone https://github.com/CTFd/CTFd
cd CTFd

# 用 Docker Compose 一鍵啟動(它會把網站 + 資料庫 + 快取全部架好)
docker compose up -d

# 開瀏覽器到 http://localhost:8000,跟著設定精靈建立你的比賽
# 之後就能在後台「新增題目、設定分數、上傳附件、設定 flag」

這正是第 40 章那套「容器化部署」的縮影:你不必手動裝資料庫、配環境, docker compose up 一行,一個能對全班開放的比賽網站就上線了。 要讓校外的人也連得到,再把它部署到一台有公開網址的雲端主機即可(概念同第 58 章的「部署上線」)。

辦一場 CTF 的完整流程

從一個念頭到賽後檢討,辦賽是一條有節奏的流水線:

一道完整的入門題長什麼樣

理論講完,來看一道麻雀雖小、五臟俱全的範例題(改寫自原始教材的 crypto_easy)。 它示範了一道題該包含的所有零件:給選手的題敘、產生附件的腳本、出題者內部解答、以及部署描述檔。 題目的點子很單純:把 flag「包了兩層」(先 ROT13、再 Base64),讓選手反過來拆。

① 給選手看的題敘(challenge.md)——只給線索,不給答案:

標題:Double Wrap（Crypto, 100 分）

我把通關碼包了兩層才寄出,應該沒人解得開吧?

附件:message.txt
提示:先想想 Base64,再想想字母是怎麼被「轉」了一圈的。

② 產生附件的腳本(build_sample.py)——出題者用它做出要發給選手的 message.txt。 注意最後的 assert:出題時務必程式化驗證「包進去能解回來」,免得出一道無解的題:

# 產生附件:把 flag 先 ROT13、再 Base64(解法剛好相反)
import base64, codecs

FLAG = "FLAG{double_wrap_base64_then_rot13}"   # 出題請改成你自己的前綴與內容

wrapped = base64.b64encode(codecs.encode(FLAG, "rot13").encode()).decode()
open("message.txt", "w", encoding="utf-8").write(wrapped + "\n")

# 自我驗證:反向解回來,確認題目「解得出來」
back = codecs.decode(base64.b64decode(wrapped).decode(), "rot13")
assert back == FLAG, back
print("message.txt 已產生;驗證解回 =", back)

③ 出題者內部解答(solution.md)——絕不公開給選手,只給審題者與賽後 writeup 用:

包法:flag → ROT13 → Base64 → message.txt
解法:message.txt → Base64 解碼 → ROT13 → flag
# 指令一行解(Base64 解碼後做 ROT13)
cat message.txt | base64 -d | tr 'A-Za-z' 'N-ZA-Mn-za-m'
# 或用 Python 解
import base64, codecs
print(codecs.decode(base64.b64decode(open("message.txt").read().strip()).decode(), "rot13"))

④ 部署描述檔(challenge.yml)——用 ctfcli 之類的工具,能據此一鍵把題目灌進 CTFd:

name: "Double Wrap"
author: "你的社團名"
category: crypto
description: |
  我把通關碼包了兩層才寄出,應該沒人解得開吧?
  (提示:先 Base64,再想想字母怎麼被「轉」了。)
value: 100
type: standard
flags:
  - FLAG{double_wrap_base64_then_rot13}
files:
  - message.txt
state: visible

看懂了嗎?一道題不只是「一個 flag」,而是題敘 + 附件 + 可驗證的解 + 部署設定四件套。 當你能把這四件事打理乾淨、讓別人順順地解開你的題,你就完成了第十八部最後的里程碑—— 從「解題者」正式升級為「出題者」。

白帽的道路:職涯、社群與那條線

你在 CTF 裡磨出來的能力——像攻擊者一樣思考、快速學會陌生技術、團隊協作、把過程寫清楚—— 並不是只能用在比賽。它們是真實資安職業的核心肌肉。CTF 是業界公認最好的練功房之一。 往外看,白帽(white hat)的路有好幾條:

  • 滲透測試(penetration testing):在客戶書面授權下,模擬攻擊者去找出系統的漏洞,寫成報告交給對方修補。你的 Web、Pwn 技能在這裡直接變現。
  • 紅隊 / 藍隊(Red / Blue Team):紅隊扮演攻擊方、演練真實入侵;藍隊負責偵測、防禦、應變。兩邊對抗(紫隊則是讓兩者協作),讓組織的防禦在演習中變強。
  • 漏洞賞金(bug bounty):很多公司(透過 HackerOne、Bugcrowd 等平台)公開邀請研究者來找漏洞,找到了依嚴重程度發獎金。這是把 CTF 技能用在正途、又能合法賺錢的途徑——但務必待在每個專案明訂的「授權範圍(scope)」內,超出範圍就從白帽變成犯罪。
  • 負責任揭露(responsible disclosure):當你(哪怕是無意間)發現了某個系統的漏洞,正確的做法是私下通報給對方、給他們合理時間修補,而不是公開炫耀或拿去利用。這呼應第 53 章談的資安治理與第 34 章的倫理底線。
⛔
最重要的一段:把技能用在正途

整個第十八部、乃至這本書,一路反覆強調同一件事,在這裡做最後一次、也是最鄭重的一次提醒: 你學的所有攻擊技術,只能用在你「擁有」或「獲得明確授權」的系統上—— 你自己的機器、合法的 CTF 平台、書面授權的滲透測試、明訂範圍的漏洞賞金。 未經授權去掃描、入侵、破壞他人的系統,在世界各地幾乎都是犯罪(在台灣對應《刑法》妨害電腦使用罪), 即使你「只是好奇、沒有惡意」、即使你「沒造成損害」。CTF 給了你一把鋒利的刀; 這把刀能讓你成為守護者,也能讓你毀掉自己的人生——差別只在你對準誰。 技術中立,用途有責;技術力越強,越要守住那條線。

🔑
持續學習,是這條路唯一的常數

資安是一場沒有終點的軍備競賽(第 34 章說過)。今天的最新攻擊,明天就成了基本功; 今天的防禦,明天可能被繞過。能在這條路上走遠的人,靠的不是「學完了什麼」,而是 「持續在社群裡學、在比賽中練、在 writeup 裡分享」的習慣。 你已經有了這個習慣——這才是 CTF 留給你最珍貴的東西。

動手練習:把這一章變成你的能力

這一章的練習,和第 58 章一樣——不是算一道題,而是真的去做。 每一題都附了範例或範本。別只是讀過;打開終端機、打開編輯器,真的動手。 做完這五題,你就完整走過了「自動化 → 參賽 → writeup → 出題 → 辦賽」的全程。

✏️ 練習 73.1 實作

用 socket 或 pwntools 寫一個自動答題腳本。題型自選:可以連到第十八部原始教材的 challenge_server.py(連續算式題),或自己寫一個會「連續問加減乘除、答對才給旗」的小伺服器來打。 目標:程式自動答完所有題、印出 FLAG{...}。

看參考解答

pwntools 版(對付「... = ?」型連續算式題):

from pwn import *

io = remote("127.0.0.1", 9000)
while True:
    line = io.recvline(timeout=5).decode("utf-8", "replace").strip()
    if not line:
        break
    log.info(line)
    if "FLAG{" in line:
        log.success(line)            # 拿到旗
        break
    if "= ?" in line:
        expr = line.split(":")[1].replace("= ?", "").strip()  # "12 + 7"
        io.sendline(str(eval(expr)).encode())   # 算完送回(僅因題源可信才用 eval)
io.close()

不想裝 pwntools?把連線改成 socket.create_connection、自己寫一個逐行收的 recvline() 即可(見本章內文的純 socket 版)。關鍵心法:認出「收一行 → 判斷 → 算 → 送 → 迴圈」這個固定骨架(圖 73-1),換湯不換藥地套到任何互動題上。

✏️ 練習 73.2 寫作

挑一題你真的解過的題(picoGym、本書前幾章的練習、或任何 CTF 都行),用本章的範本寫一份 完整的 writeup。檢驗標準只有一個:請一個沒解過這題的人照著你的 writeup,他能不能重現?

看範本與一份迷你範例

骨架(複製去填):

# Writeup — 題名
## 題目資訊（題名 / 類別 / 分數 / 賽事）
## 一、題目給了什麼
## 二、觀察與猜測
## 三、解題過程（附實際指令/程式碼/截圖,要能重現）
## 四、拿到的 flag
## 五、學到什麼 / 卡在哪
## 六、參考

迷你範例(以本章的 Double Wrap 為例):

# Writeup — Double Wrap
## 題目資訊
- 類別:Crypto / 100 分 / 班際 CTF
## 一、題目給了什麼
一個 message.txt,內容是一串看起來像 Base64 的字。題敘說「包了兩層」。
## 二、觀察與猜測
字串只含 A-Za-z0-9+/、結尾有 =,典型 Base64 → 先解一次看看。
## 三、解題過程
  cat message.txt | base64 -d        # 解出來還是亂碼,但長度像英文句子
  → 想到題敘「字母被轉了一圈」= ROT13
  cat message.txt | base64 -d | tr 'A-Za-z' 'N-ZA-Mn-za-m'
## 四、拿到的 flag
  FLAG{double_wrap_base64_then_rot13}
## 五、學到什麼
看到「像 Base64 但解開仍是亂碼」,要想到「可能還有第二層編碼」。
## 六、參考:CyberChef(From Base64 → ROT13)

注意「五、學到什麼」這段——它把一次解題沉澱成下次更快的直覺,是 writeup 最值錢的部分。

✏️ 練習 73.3 規劃

假設你要揪 4 個同學組一支隊伍打你人生第一場 CTF。請規劃這支隊伍的分工: 每個人主攻哪一類(六大類別)、誰當記分板協調者、比賽時的時間管理與停損規則怎麼訂。

看參考規劃

(範例,合理即可。)

  • 小華 — 主攻 Crypto(數學好、熟第 35 章),兼看 Misc。

時間 / 心態規則:① 開賽全員先各掃自己類別,十分鐘後同步「哪些題投報率高」。 ② 單題卡 90 分鐘無進展 → 寫下卡點、換題。③ 所有嘗試即時記共筆(順便當賽後 writeup 草稿)。 ④ 排輪休,不硬撐通宵。⑤ 賽後一起讀官方/他隊 writeup,把沒解出的題弄懂。

✏️ 練習 73.4 出題

設計一道入門 CTF 題(類別自選,Crypto/Misc 最好上手)。交出四件套: ① 給選手的題敘、② 產生附件的腳本(或附件本身)、③ 出題者內部解答、④ 一句難度評估與「防作弊」說明。 鐵律:你的解答腳本要能真的跑出你設定的 flag。

看範例題(XOR 單鍵)

① 題敘:「我把通關碼跟一個祕密數字做了 XOR。可惜這個祕密只有一個位元組……」附件 cipher.txt(一串 hex)。

② 產生附件 + ③ 內部解答(同一支腳本,含自我驗證):

FLAG = b"FLAG{xor_one_byte_is_too_weak}"
KEY  = 0x5A                                   # 出題者選的祕密單一位元組

# 出題:逐位元組 XOR,輸出 hex 當附件
cipher = bytes(b ^ KEY for b in FLAG)
open("cipher.txt", "w").write(cipher.hex())

# 解法(選手該做的):單位元組金鑰只有 256 種,全試一遍,挑出印得出 "FLAG{" 的
data = bytes.fromhex(open("cipher.txt").read().strip())
for k in range(256):
    guess = bytes(b ^ k for b in data)
    if guess.startswith(b"FLAG{"):
        print("key =", hex(k), "->", guess.decode())

④ 難度與防作弊:難度「入門」——只要知道「單位元組金鑰可暴力枚舉 256 種」就能解, 適合放在比賽開頭暖身。防作弊:flag 不直接出現在附件(已 XOR)、不同隊可給不同 KEY 產生不同附件以防互抄。 出題前用上面的解法段跑過,確認真的解得回 FLAG——這就是「可驗證的唯一解」。

✏️ 行動 73.5 挑戰 · 辦一場

進階,但這才是第十八部的終點:規劃一場小型班際 CTF 的完整流程。寫出你的計畫: 要幾道題(難度梯度怎麼鋪)、怎麼審題、用什麼平台架站、賽程多長、怎麼監控與答疑、賽後怎麼檢討。 有條件的話,真的用 CTFd + Docker 架起來,放一道你在練習 73.4 出的題進去,讓同學連進來解一次。

看參考流程

架站(第 40 章的容器化部署):

git clone https://github.com/CTFd/CTFd
cd CTFd
docker compose up -d        # 開瀏覽器到 http://localhost:8000 設定比賽

流程(對照圖 73-3):

  1. 出題:湊齊 8~10 題,難度鋪成梯度——3 題人人能解的入門(暖身)、4 題中等、2~3 題難題(區分高手)。涵蓋多個類別。
  2. 審題(最關鍵):每題找一個沒看過解答的同學試解,校準難度、揪出非預期解、確認附件沒夾帶內部解答、flag 沒裸奔在前端。
  3. 上架:用 CTFd 後台(或 challenge.yml + ctfcli)逐題建立:題敘、分數、附件、flag、提示。
  4. 開賽:公告開始/結束時間(班際賽抓 3~6 小時即可)、規則、計分方式;比賽中盯記分板與 Discord,即時答疑、必要時放提示或修正出包的題。
  5. 賽後檢討:頒獎、放出官方 writeup、收集「哪題太難/太簡單/有非預期解」的回饋,帶進下一場。

當你出的題第一次被別人解開、記分板上跳出那一聲歡呼——恭喜,你完成了第十八部的結業關。

把這一章串起來

這一章,我們把第十八部從「會解題」推到了「成為社群的一員」:

  • 自動化解題:手動撞牆的地方(連續答題、互動式 Pwn/Crypto),用 Python socket(第 25 章)打底、用 pwntools 的 remote/recvuntil/sendline 收尾。所有互動題都是同一個「收 → 算 → 送 → 迴圈」(圖 73-1),這把槓桿貫穿全部類別。
  • 參賽:認識 Jeopardy 與 Attack-Defense 兩種賽制(圖 73-2);用 CTFtime/picoCTF/AIS3 找比賽;依六大類別組隊分工;靠時間管理與停損規則打好耐力賽。
  • writeup 文化:寫,是為了整理知識、回饋社群、建立個人品牌;好 writeup 的黃金標準是可重現;用部落格/GitHub/HackMD 發表(但比賽中別爆雷)。
  • 自辦 CTF:一道好題要難度準、唯一解、防作弊;用 CTFd + Docker 一鍵架站(第 40 章);走完「出題→審題→上架→開賽→檢討」的流水線(圖 73-3);一道題是「題敘+附件+可驗證解+部署檔」四件套。
  • 白帽的路與那條線:滲透測試、紅藍隊、漏洞賞金、負責任揭露——把技能用在正途;只在授權範圍內動手,技術中立、用途有責。
