視覺與圖學:一對逆問題

想像兩位工匠。第一位是偵探:你給他一張照片,他要從一片像素中推理出「這裡有一張桌子、桌上有顆蘋果、光從左邊來」。 第二位是畫家:你給他「桌子、蘋果、光從左邊來」這份劇本,他要把它畫成一張看起來真實的圖。 這兩位工匠做的,正好是相反方向的事。

偵探就是電腦視覺(computer vision)——這一部前面十幾章都在訓練他:邊緣、特徵、分類、偵測、多視幾何…… 全都是「從影像反推出世界」。畫家則是電腦圖學(computer graphics)——也就是你在第 39 章讀過的主題: 用向量、矩陣、光照,把「世界正向算成影像」。一句話總結:

🔑
看與畫,互為逆問題

視覺:影像 → 世界(分析、反推);圖學:世界 → 影像(合成、正算)。 它們處理的是同一套幾何(同樣的相機模型、同樣的投影、同樣的座標變換),只是走向相反。 正因如此,當你想把「虛擬的東西」無縫放進「真實的影像」裡(也就是 AR),你必須同時當偵探和畫家—— 先看懂相機在哪、桌面在哪,再把虛擬物件畫到對的位置。這一章就是這場握手。

2D 圖學:從「分析像素」到「算出像素」

這一部一路走來,你都在分析既有的像素(濾波、找邊、辨識)。2D 圖學是一個漂亮的轉折點: 現在換你主動算出每個像素該是什麼顏色。Illustrator 的向量圖、SVG、資料視覺化、字型渲染、遊戲場景, 底層都是 2D 圖學。

座標系與「畫一條線」一句話

先講一個常絆倒人的小東西:螢幕座標系。數學課的 $y$ 軸朝上,但影像的 $y$ 軸朝下—— 原點 $(0,0)$ 在左上角,$y$ 越大越往下。這是因為螢幕是「一列一列」從上往下掃描出來的。所以你之後會常看到 y 要做一次翻轉,別被它咬到。

至於最底層的「怎麼把一條線、一個三角形真的填成像素」——這在第 39 章已經談過(Bresenham 畫線、掃描線填充)。 一句話帶過:畫線就是在離散網格上找出最接近這條理想直線的那一串像素;填充就是逐列找出多邊形的左右邊界、把中間塗滿。 實務上我們直接呼叫 cv2.line、cv2.polylines 當畫筆,把心力放在「畫什麼」而不是「怎麼點亮像素」。

變換矩陣:生成藝術的引擎

2D 圖學最好玩的入口是生成藝術(generative art):用數學規則,而不是一筆一畫,生出對稱或碎形的圖案。 它的引擎就是你在第 39、54 章看過的齊次座標(homogeneous coordinates)變換矩陣——把 2D 點 $(x,y)$ 補成 $(x,y,1)$, 平移、旋轉、縮放就全部統一成 $3\times 3$ 矩陣乘法,還能用 @ 自由串接:

$$M = \underbrace{T(c_x,c_y)}_{\text{平移到中心}} \cdot \underbrace{R(\theta)}_{\text{旋轉}} \cdot \underbrace{S(s,s)}_{\text{縮放}}$$

記住第 39 章的鐵律:矩陣相乘不可交換,作用在點上的順序是從右到左讀——上式是「先縮放、再旋轉、最後平移」。 我們把一片花瓣的幾個頂點,旋轉複製 12 份,就得到一朵對稱的曼陀羅: 

import numpy as np, cv2

# 三個基本 2D 變換(齊次 3x3)
def T(tx, ty): return np.array([[1, 0, tx], [0, 1, ty], [0, 0, 1.]])   # 平移
def R(deg):                                                            # 旋轉
    a = np.radians(deg); c, s = np.cos(a), np.sin(a)
    return np.array([[c, -s, 0], [s, c, 0], [0, 0, 1.]])
def S(sx, sy): return np.array([[sx, 0, 0], [0, sy, 0], [0, 0, 1.]])  # 縮放

def apply(M, pts):
    """對一群 (x,y) 點套變換:補齊次的 1 → 乘 M → 取回 (x,y)。"""
    h = np.c_[pts, np.ones(len(pts))]      # 每點補一個 1 → (x, y, 1)
    return (M @ h.T).T[:, :2]              # 乘上矩陣後丟掉最後一維

img = np.zeros((500, 500, 3), np.uint8)
petal = np.array([[0, 0], [40, -15], [70, 0], [40, 15]], np.float32)   # 一片花瓣
for i in range(12):                        # 旋轉 12 份 → 對稱曼陀羅
    M = T(250, 250) @ R(i * 30) @ S(1, 1)  # 右到左:先縮放 → 旋轉 → 平移到中心
    pts = apply(M, petal).astype(np.int32)
    cv2.polylines(img, [pts], True, (120, 200, 255), 2)
cv2.imwrite("mandala.png", img)

把 R(i * 30) 換成隨機角、把 petal 換成更花俏的形狀、或把上面的迴圈改成遞迴(畫一段樹枝、再從末端分兩枝、每枝更短), 你就能長出一棵分形樹(fractal tree)。生成藝術的精神就在這裡:少少幾條規則 + 變換 + 重複/遞迴,長出複雜的美。

3D 渲染管線:從模型座標到螢幕像素

現在升上 3D。一個 3D 物體要變成螢幕上的像素,要走一條固定的流水線,叫繪圖管線(graphics pipeline)。 第 39 章畫過它的全貌;這裡我們聚焦它的幾何骨架——也就是業界俗稱的 MVP——並親手把它寫出來。 每一次「換座標系」,本質上就是乘上一個 $4\times 4$ 齊次矩陣:

$$\mathbf{p}_{\text{clip}} = \underbrace{P}_{\text{Projection 投影}} \cdot \underbrace{V}_{\text{View 視圖}} \cdot \underbrace{M}_{\text{Model 模型}} \cdot \mathbf{p}_{\text{local}}$$

  • Model(模型矩陣):把物體從「自己的座標」(以茶壺中心為原點)擺進場景,變成世界座標。旋轉、縮放、移動物體就改這個。
  • View(視圖矩陣):用「攝影機在哪、朝哪看、哪邊是上」把整個世界轉成從鏡頭看出去的座標。等於「移動相機」。
  • Projection(投影矩陣):把 3D 壓扁成 2D,並製造「近大遠小」。這是最關鍵的一步。

投影矩陣的魔法藏在輸出的第四個分量 $w$:它和「離相機多遠」成正比。最後一步透視除法(perspective divide)把所有座標除以 $w$—— 遠的東西 $w$ 大、除完就縮小,近的東西 $w$ 小、相對放大。這就是近大遠小的全部祕密:

$$(x, y, z, w) \xrightarrow{\ \div\, w\ } \left(\tfrac{x}{w}, \tfrac{y}{w}, \tfrac{z}{w}\right) \quad\Longrightarrow\quad \text{得到 } [-1, 1]^3 \text{ 的標準座標(NDC)}$$

動手:用矩陣畫出旋轉的線框立方體

把上面整條管線寫成程式,就是 M23 的旗艦範例 m23_wireframe。我們定義立方體的 8 個頂點與 12 條邊, 每一影格給它一個旋轉的 Model 矩陣,跑完 MVP + 透視除法,再把每條邊用 cv2.line 連起來——一個會轉的線框(wireframe)立方體就誕生了: 

import numpy as np, cv2

# 立方體 8 個頂點(每軸取 ±1 的所有組合)與 12 條邊
V = np.array([[x, y, z] for x in (-1, 1) for y in (-1, 1) for z in (-1, 1)], float)
E = [(0,1),(0,2),(0,4),(1,3),(1,5),(2,3),(2,6),(3,7),(4,5),(4,6),(5,7),(6,7)]

def perspective(fov, aspect, n, f):
    """透視投影矩陣:負責『近大遠小』(關鍵在輸出的 w)。"""
    t = 1 / np.tan(np.radians(fov) / 2)
    return np.array([[t/aspect, 0, 0, 0], [0, t, 0, 0],
                     [0, 0, (f+n)/(n-f), 2*f*n/(n-f)], [0, 0, -1, 0]])

def render(angle, W=400, H=400):
    img = np.zeros((H, W, 3), np.uint8)
    c, s = np.cos(angle), np.sin(angle)
    model = np.array([[c,0,s,0],[0,1,0,0],[-s,0,c,0],[0,0,0,1.]])  # 繞 y 軸旋轉
    view = np.eye(4); view[2, 3] = -5             # 相機沿 z 後退 5(才看得到)
    proj = perspective(50, W/H, 0.1, 100)
    mvp = proj @ view @ model                      # 串成一個總變換(右到左作用)
    hom = np.c_[V, np.ones(len(V))]                # 頂點補齊次的 1
    clip = (mvp @ hom.T).T                          # 套 MVP → 裁切空間
    ndc = clip[:, :3] / clip[:, 3:4]               # ★透視除法:除以 w → 近大遠小
    sx = ((ndc[:,0] + 1) / 2 * W).astype(int)      # x: [-1,1] → [0,W]
    sy = ((1 - ndc[:,1]) / 2 * H).astype(int)      # y: 翻轉(螢幕 y 向下)
    for a, b in E:
        cv2.line(img, (sx[a], sy[a]), (sx[b], sy[b]), (120, 220, 255), 2)
    return img

cv2.imwrite("wireframe.png", render(0.6))
⚠️
三個最常見的線框 bug

① 忘了透視除法(沒除以 w)→ 變成正交投影,完全沒有遠近感。 ② 螢幕 y 沒翻轉 → 物體上下顛倒(NDC 的 $y$ 朝上,螢幕的 $y$ 朝下)。 ③ 相機沒後退(忘了 view[2,3] = -5)→ 立方體卡在相機裡,看到一團亂線或全黑。

光柵化與著色器:把三角形填成有光的像素

線框只是骨架。要畫出實心、有明暗的面,接力棒交給光柵化(rasterization):把每個三角形「填」成一格格像素。 為什麼基本圖元是三角形?因為三個點一定共面、一定是平的;任何複雜曲面都能用一堆小三角形(網格 mesh)拼出來。 光柵化要回答三個問題,我們逐一擊破。

問題一:哪些像素在三角形內?重心座標

圖學最愛的判定法是重心座標(barycentric coordinates):把三角形內任一點 $\mathbf{p}$ 寫成三頂點的加權平均,

$$\mathbf{p} = \alpha A + \beta B + \gamma C, \qquad \alpha + \beta + \gamma = 1$$

當且僅當三個權重都 $\ge 0$ 時,點落在三角形內(任一個是負的,就在某條邊外面)。 這套權重還有超能力:同一組 $\alpha,\beta,\gamma$ 能把頂點上的屬性(顏色、深度、紋理座標)平滑內插到內部每個像素—— 後面的上色與深度全靠它。

問題二:誰擋住誰?z-buffer

三角形會互相遮擋。關鍵觀念:不是「後畫的蓋先畫的」,而是「近的蓋遠的」。 做法是開一張和畫面一樣大的深度緩衝(z-buffer),每像素先記「無限遠」。畫每個像素時用重心座標插出它的深度 $z = \alpha z_A + \beta z_B + \gamma z_C$, 比目前記錄更近($z$ 更小)才覆蓋顏色並更新深度,否則丟棄。這樣不管三角形以什麼順序送進來,最後留下的永遠是最靠近相機的那個。

動手:一個迷你軟體光柵器的核心

把這兩件事寫進雙重迴圈,就是 M24 旗艦範例 m24_rasterizer 的核心。為了看懂原理,這裡用最白話(也最慢)的逐像素寫法, 實務上會向量化或交給 GPU: 

import numpy as np

def edge(ax, ay, bx, by, px, py):
    """邊函數:點 (px,py) 在有向邊 a→b 的哪一側。"""
    return (bx - ax) * (py - ay) - (by - ay) * (px - ax)

def bary(P, A, B, C):
    """回傳 P 對三角形 ABC 的重心座標 (α, β, γ)。"""
    area = edge(*A, *B, C[0], C[1])
    a = edge(*B, *C, P[0], P[1]) / area
    b = edge(*C, *A, P[0], P[1]) / area
    return a, b, 1 - a - b

def fill(img, zbuf, tri, zs, cols):
    A, B, C = tri                                  # 三頂點的螢幕座標 (x, y)
    xs = [p[0] for p in tri]; ys = [p[1] for p in tri]
    for y in range(int(min(ys)), int(max(ys)) + 1):       # 只掃外接矩形,省時間
        for x in range(int(min(xs)), int(max(xs)) + 1):
            a, b, g = bary((x, y), A, B, C)
            if a >= 0 and b >= 0 and g >= 0:             # 三權重皆 ≥0 → 在三角形內
                z = a * zs[0] + b * zs[1] + g * zs[2]     # 重心插值深度
                if z < zbuf[y, x]:                        # 比現有更近才畫
                    zbuf[y, x] = z
                    img[y, x] = a*cols[0] + b*cols[1] + g*cols[2]   # 顏色也用重心插值

問題三:這個面該多亮?著色器與 Phong

最後是「上什麼顏色」。最經典的Phong 光照把一個面的顏色拆成三項相加: 環境光(ambient)給基本底亮度、漫反射(diffuse)讓「越正對光越亮」、鏡面反射(specular)給光滑表面的高光點:

$$C = \underbrace{C_a}_{\text{環境}} + \underbrace{(\mathbf{N}\cdot\mathbf{L})\,C_d}_{\text{漫反射}} + \underbrace{(\mathbf{R}\cdot\mathbf{V})^{n}\,C_s}_{\text{鏡面}}$$

漫反射項就是第 39 章說的「光照計算的心臟」——表面法線 $\mathbf{N}$ 與光線方向 $\mathbf{L}$ 的內積:面正對光時內積接近 1(最亮),側對時趨近 0(暗)。 切記法線一定要先正規化,否則亮度全錯。

在真實 GPU 上,這段「對每個像素算顏色」的程式叫片段著色器(fragment shader)。它最迷人的一點是: GPU 會同時、平行地對幾百萬個像素各跑一次這段小程式——這正呼應第 43 章的平行運算與第 48 章的 GPU 架構。 下面這段 GLSL(語法像 C,跑在 GPU)就是上面漫反射項的硬體版: 

// 片段著色器(GLSL):GPU 對每個像素各跑一次,而且幾百萬個同時平行算
#version 330
in  vec3 v_normal;        // 從上一階段插值進來的「這個像素的法線」
out vec4 f_color;         // 要輸出的顏色(RGBA)
uniform vec3 light_dir;   // 所有像素共用的參數:光線方向(Python 端傳入)
void main() {
    float diff = max(dot(normalize(v_normal), normalize(light_dir)), 0.0);  // N·L,漫反射
    f_color = vec4(vec3(0.1) + diff * vec3(0.2, 0.6, 1.0), 1.0);            // 環境光 + 漫反射
}
🔑
shader 是什麼,一句話

著色器(shader)是一段「對每個頂點或每個像素各跑一次、由 GPU 大規模平行執行」的小程式。 軟體光柵器(純 NumPy)讓你看懂「每個像素到底怎麼算」;GLSL 著色器則是把同一件事交給 GPU 平行做、快上千倍。 它和第 2 章卷積「對每個像素做運算」是同一種思想,只是搬上了 GPU。

反過來:從影像重建 3D 與點雲

到這裡,我們都在做「給定 3D,畫成 2D」。M25 把箭頭反過來——這正是視覺與圖學的交會點:從真實照片或深度,重建出 3D 結構。 取得 3D 有兩條主要路徑:

  • SfM / MVS(運動恢復結構):拿好幾張不同角度的照片,用特徵匹配找出同一個點在各張照片的位置,再靠多視幾何(第 66 章)反推出每個點的 3D 座標。一邊也順便解出相機在哪——這和第 66 章的相機標定是同一套數學。常用工具:COLMAP。
  • 深度相機:RealSense、Kinect、LiDAR 直接給出每個像素的深度,一步到位變成 3D 點。

兩條路徑的產物都是點雲(point cloud)——一大堆帶 $(x,y,z)$ 的 3D 點。點雲通常又多又髒,要經過一條清理流水線, 再「包」成連續的表面網格,才能上色、量測、3D 列印。業界標準工具是 Open3D: 

import open3d as o3d

pcd = o3d.io.read_point_cloud("scan.ply")           # 讀進點雲
pcd = pcd.voxel_down_sample(0.005)                  # 體素降採樣:每小格只留一個代表點
pcd, _ = pcd.remove_statistical_outlier(20, 2.0)    # 去離群點(看鄰居距離,太突出的當雜訊)
pcd.estimate_normals()                              # 估每點法線(表面重建必需)
mesh, _ = o3d.geometry.TriangleMesh.create_from_point_cloud_poisson(pcd, depth=9)  # Poisson 表面重建
o3d.io.write_triangle_mesh("mesh.ply", mesh)        # 輸出網格

其中「體素降採樣」的原理一行就能說清楚:把空間切成小立方格,同一格內的點通通合併成一個平均點。 用純 NumPy 就能離線理解(這也是 m25_pointcloud 附的版本): 

import numpy as np

def voxel_downsample(points, voxel=0.05):
    keys = np.floor(points / voxel).astype(np.int64)   # 每點落在哪個立方格(整數座標)
    _, inv = np.unique(keys, axis=0, return_inverse=True)  # inv:每點屬於第幾格
    out = np.zeros((inv.max() + 1, 3)); cnt = np.zeros(inv.max() + 1)
    np.add.at(out, inv, points)        # 同格座標累加
    np.add.at(cnt, inv, 1)             # 數每格幾個點
    return out / cnt[:, None]          # 總和 ÷ 個數 = 每格的平均點(代表點)

若有多次掃描(一次只能掃到一面),還要用 ICP(疊代最近點)把多片點雲配準(registration)對齊成完整模型—— 反覆「找最近對應點 → 算出怎麼旋轉平移貼合 → 移動」直到收斂。

擴增實境 AR:讓虛擬物站進真實世界

終於來到重頭戲。擴增實境(Augmented Reality, AR)是這整部書的會師點:它要把一個虛擬的 3D 物件, 以正確的位置、正確的姿態,疊到真實相機畫面上——IKEA 把沙發擺進你家、虛擬試戴眼鏡、AR 名片掃出 3D 模型,核心都是這件事。 為什麼它需要整部書?因為它必須同時當偵探和畫家:

  1. 看(視覺):在畫面裡找到一個「參考」(這裡用 ArUco 標記),偵測出它的四個角。
  2. 算(多視幾何):由四個角 + 相機內參 $K$(來自第 66 章的相機標定),反推出相機相對標記的姿態——也就是旋轉 $R$ 與平移 $t$。這一步在 OpenCV 裡叫 solvePnP。
  3. 畫(圖學):有了 $R, t, K$,就能用針孔相機模型把虛擬物件的 3D 頂點投影到畫面上的正確位置——這正是本章前半的渲染管線!

第三步的數學,就是把 3D 世界點 $\tilde{\mathbf{P}}$ 投影到 2D 影像點 $\tilde{\mathbf{p}}$ 的針孔相機公式——它其實就是 MVP 管線在真實相機上的具體化身:

$$s\,\tilde{\mathbf{p}} = K\,[\,R \mid t\,]\,\tilde{\mathbf{P}}$$

其中 $[R \mid t]$ 是「外參」(相機在世界中的姿態,對應 View 矩陣),$K$ 是「內參」(焦距、主點,對應 Projection)。 看出來了嗎?AR 的「畫」就是渲染管線,只是 View 與 Projection 矩陣不再是你隨手設的,而是從真實影像「看」出來的。這就是視覺與圖學握手的精確時刻。

動手:把虛擬立方體釘在標記上

旗艦專題 m26_aruco 走「貼 2D 圖」的免標定路線(偵測四角 → 用單應性 warp 把圖貼上去),適合先玩起來。 而下面這段是「貼 3D 物件」的完整版——它需要第 66 章標定好的 $K$ 與畸變係數 dist,正是 AR 名片/桌面 AR 的骨架: 

import cv2, numpy as np
aruco = cv2.aruco
DICT = aruco.getPredefinedDictionary(aruco.DICT_4X4_50)
# K, dist 來自第 66 章的相機標定;marker_len 是標記實際邊長(公尺),決定距離尺度
K, dist, marker_len = K_calib, dist_calib, 0.05

def draw_cube(frame, gray):
    corners, ids, _ = aruco.detectMarkers(gray, DICT)              # ① 偵測標記四角
    if ids is None:
        return frame
    rvecs, tvecs, _ = aruco.estimatePoseSingleMarkers(             # ② 估姿態 R, t
        corners, marker_len, K, dist)
    # ③ 一個立著的虛擬立方體的 8 個 3D 頂點(以標記中心為原點,單位:公尺)
    h = marker_len / 2
    cube = np.float32([[-h,-h,0],[h,-h,0],[h,h,0],[-h,h,0],
                       [-h,-h,2*h],[h,-h,2*h],[h,h,2*h],[-h,h,2*h]])
    edges = [(0,1),(1,2),(2,3),(3,0),(4,5),(5,6),(6,7),(7,4),(0,4),(1,5),(2,6),(3,7)]
    for rvec, tvec in zip(rvecs, tvecs):
        pts, _ = cv2.projectPoints(cube, rvec, tvec, K, dist)      # 3D → 2D:K[R|t]
        pts = pts.reshape(-1, 2).astype(int)
        for i, j in edges:
            cv2.line(frame, tuple(pts[i]), tuple(pts[j]), (0, 255, 0), 2)
    return frame

cv2.projectPoints 做的,正是把每個 3D 頂點丟進 $s\tilde{\mathbf p}=K[R\mid t]\tilde{\mathbf P}$ 算出 2D 像素位置——這就是你前面親手寫過的投影管線。 當你把標記在鏡頭前傾斜、移動,$R,t$ 隨之更新,立方體就會像被釘在紙卡上一樣,跟著轉、跟著走。

⚠️
AR 對不準?先查這三件事

① 沒先標定相機:$K$、dist 不對,姿態與疊合一定歪——務必先做第 66 章的標定。 ② 標記邊長填錯:marker_len 是真實公尺數,填錯會讓物件大小/距離整個跑掉。 ③ 抖動:每影格獨立估姿態會微微抖,對 $R,t$ 做時間平滑(或用更穩的標記板)會好很多。

動手練習

✏️ 練習 68.1 實作

用矩陣對 3D 點做投影,畫出一個線框立方體。請定義立方體的 8 個頂點與 12 條邊,套上 Model(隨意旋轉)→ View(相機後退)→ Projection(透視),做透視除法後映射到螢幕,連出每一條邊。驗證「忘了除以 $w$」會發生什麼事。

看解答

直接用本章 render 函式即可(見「動手:用矩陣畫出旋轉的線框立方體」)。關鍵三步:

hom = np.c_[V, np.ones(len(V))]        # 1) 補齊次的 1
clip = (proj @ view @ model @ hom.T).T  # 2) 套 MVP(右到左作用)
ndc = clip[:, :3] / clip[:, 3:4]        # 3) 透視除法:除以 w

若把第 3 步改成 ndc = clip[:, :3](不除 $w$),等於做正交投影——立方體前後面一樣大,完全沒有近大遠小的立體感。把 w 加回來,遠的面才會收縮、稜線才會向遠方聚攏。記得螢幕 $y$ 要翻轉(1 - ndc[:,1]),否則上下顛倒。

✏️ 練習 68.2 思考

有兩個三角形投影後在某些像素重疊。請解釋 z-buffer 如何保證「近的擋住遠的」,而且與繪製順序無關。如果不用 z-buffer,改成「先畫的會被後畫的蓋掉」,會出什麼錯?有沒有 z-buffer 也解決不了的情況?

看解答

z-buffer 的機制:除了顏色緩衝,再開一張同樣大小、初值為「無限遠」的深度緩衝。畫每個像素時,用重心座標插出它的深度 $z=\alpha z_A+\beta z_B+\gamma z_C$,只有當 z < zbuf[y,x](比目前記錄的更近)時,才覆蓋顏色並更新 zbuf;否則丟棄。因為每個像素永遠只留「目前看過最近的那個」,所以不管三角形以什麼順序送進來,結果都一樣——這就是與繪製順序無關的原因。

不用 z-buffer 的後果:若採「後畫蓋先畫」,只要先畫近的、後畫遠的,遠的就會錯誤地蓋住近的(畫家演算法的失敗)。先排序「從遠到近」能救一部分,但遇到互相穿插的三角形(A 的一半在 B 前、另一半在 B 後)就無法排出正確順序。

z-buffer 也搞不定的情況:半透明物體。z-buffer 是「非黑即白」地保留最近片元,但透明需要把前後顏色混合(alpha blending),這得另外把透明物排序後從後往前畫。此外深度精度有限(z-fighting),兩個非常接近的面會閃爍。

✏️ 練習 68.3 實作

用重心座標判斷一個點是否落在三角形內,並把三個頂點的顏色平滑內插上色。給定三角形 $A=(50,50)$、$B=(200,80)$、$C=(120,220)$,三頂點顏色分別為紅、綠、藍,請填出整個三角形(內部顏色應呈現平滑漸層)。

看解答

用本章的 edge / bary 函式,逐像素判斷與上色:

import numpy as np

def edge(ax, ay, bx, by, px, py):
    return (bx-ax)*(py-ay) - (by-ay)*(px-ax)

A, B, C = (50,50), (200,80), (120,220)
cA, cB, cC = np.array([0,0,255]), np.array([0,255,0]), np.array([255,0,0])  # BGR
img = np.zeros((260, 260, 3), np.uint8)
area = edge(*A, *B, C[0], C[1])
for y in range(260):
    for x in range(260):
        a = edge(*B, *C, x, y) / area
        b = edge(*C, *A, x, y) / area
        g = 1 - a - b
        if a >= 0 and b >= 0 and g >= 0:          # 三權重皆 ≥0 → 在三角形內
            img[y, x] = (a*cA + b*cB + g*cC).astype(np.uint8)   # 顏色用重心插值
import cv2; cv2.imwrite("tri.png", img)

結果是一個從紅、綠、藍三個角各自漸層、在中心混成灰白的三角形。$\alpha,\beta,\gamma$ 同時扮演「在不在裡面」的判定器與「離各頂點多近」的權重——這就是光柵化又判定又插值的雙重威力。

✏️ 練習 68.4 思考

解釋 AR 如何用「相機姿態」把虛擬物件對齊真實世界。具體說明:從一張含標記的影像,到一個立方體穩穩站在標記上,中間經過哪幾步?式子 $s\,\tilde{\mathbf p}=K[R\mid t]\tilde{\mathbf P}$ 裡的每個符號代表什麼?它和本章前半的渲染管線是什麼關係?

看解答

三步:(1) 看——偵測標記,取得它在影像上的四個角(已知它們對應標記平面上 $(\pm h,\pm h,0)$ 這些 3D 點)。(2) 算——已知「3D 點 ↔ 2D 角」的對應與相機內參 $K$,用 solvePnP 解出相機相對標記的姿態:旋轉 $R$ 與平移 $t$。(3) 畫——把虛擬立方體的 8 個 3D 頂點代入投影式,得到它們在畫面上的 2D 位置,連線疊上去。

符號:$\tilde{\mathbf P}$ 是齊次的 3D 世界點;$[R\mid t]$ 是外參(相機姿態,把世界點轉到相機座標,對應 View 矩陣);$K$ 是內參(焦距、主點,把相機座標投影到像素,對應 Projection);$s$ 是因透視除法產生的尺度因子($\tilde{\mathbf p}$ 算完要除以最後一個分量)。

關係:這就是渲染管線本身!差別在於:自己渲染時 View/Projection 是你隨手設的;AR 裡 $[R\mid t]$ 與 $K$ 是從真實影像「看」出來的(標定 + 姿態估計)。因為虛擬物件用的是「真實相機的眼睛」去投影,它才會和真實畫面對齊——這正是視覺與圖學握手的時刻。

✏️ 練習 68.5 挑戰

設計一個整合「視覺 + 圖學」的小專題。請寫出:① 要解決誰的什麼問題;② 資料流三段(輸入 → 處理用到哪些章 → 輸出);③ 最小可用版本(MVP)要先做到什麼;④ 用什麼指標評估、它在什麼情況下會失敗。

看參考範例

以「AR 互動名片」為例:

  • ① 問題:讓求職者的紙本名片,被手機掃到時浮出一個可旋轉的 3D 作品集模型,加深印象。成功標準:在一般室內光下,標記出現後 1 秒內穩定疊上模型,旋轉名片時抖動不明顯。
  • ② 資料流:輸入(webcam 影像)→ 處理〔灰階(M0)→ 偵測 ArUco 四角(本章)→ 相機標定 $K$/姿態 $R,t$(第 66 章 + 本章)→ 用 projectPoints 投影虛擬模型(本章渲染管線)〕→ 輸出(疊好虛擬物的即時畫面)。
  • ③ MVP:先從 m26_aruco 改起,做到「偵測到標記就在上面畫一個綠色線框立方體」。能動之後,再把立方體換成載入的 .obj 模型、加上時間平滑、做個切換多個模型的 UI。
  • ④ 評估與限制:指標用 FPS(是否即時)與對位誤差(座標軸是否貼齊標記)。誠實列出失敗情況:光線太暗或標記反光 → 偵測不穩;標記太小/太斜 → 姿態抖;沒重新標定就換相機 → 模型大小跑掉。

(其他可行題目:虛擬試戴、桌面 3D 掃描器、手勢控制的 3D 場景——重點是至少串起兩條技術線,並且「先能動再求好」。)

把這一章串起來

這一章,我們替整個第十七部畫下句點——把「看」與「畫」接成了一個圓。回顧這趟路:

  • 視覺與圖學是一對逆問題:視覺從影像反推世界,圖學把世界正算成影像;兩者共用同一套相機與投影數學。
  • 2D 圖學讓你從「分析像素」轉為「算出像素」,用變換矩陣串接做出生成藝術。
  • 3D 渲染管線(MVP):模型 → 世界 → 視圖 → 投影 → 透視除法,你親手寫出了會轉的線框立方體。
  • 光柵化的兩個心臟是重心座標(判定 + 內插)與 z-buffer(遮擋);著色器是 GPU 上每像素平行跑的小程式。
  • 3D 重建把箭頭反過來:從多張影像或深度相機,經點雲與表面重建,還原出 3D 世界。
  • AR 是會師點:用視覺看出相機姿態、用多視幾何算出 $R,t$、再用圖學畫出對齊真實世界的虛擬物件。

最重要的一句話帶走:當你能同時當偵探(看)和畫家(畫),虛擬與真實的邊界就消失了。 這正是整個第十七部從第 65 章一路鋪陳到這裡的目的——不是會呼叫 API,而是真正懂得「影像 ↔ 世界」這座雙向橋,並能親手把一個視覺想法,做成一件可用、可展示的作品。
