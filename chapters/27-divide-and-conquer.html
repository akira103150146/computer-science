分治法與遞迴關係

如果只能教你一種演算法設計的「招式」,我會選這一招:把一個大到無從下手的問題,劈成兩半、各自擊破、再合而為一。它的名字,叫做分治。

羅馬人怎麼統治半個世界

有句格言叫「分而治之(divide and conquer)」,據說源自古羅馬與後來的帝國統治術: 面對一塊大到無法直接掌控的疆土,別硬碰硬——把它切成幾塊小的,各個擊破,再各自管理。 一個你搞不定的大麻煩,常常是好幾個你搞得定的小麻煩拼起來的。

換個更接地氣的比喻。你要把一塊超大的披薩公平分給一群人,人多到你數不清。與其盯著整塊餅發愁, 不如把它對半切開,左半交給一個朋友去分、右半交給另一個朋友去分; 他們各自又會把自己那半再對半切、再交給別人……一路切到「每個人面前剛好一塊」為止。 最後,大家手上的小塊「合起來」,就是整塊餅被公平分完的結果。

這個「切一半 → 各自處理 → 合起來」的念頭,聽起來像生活智慧,但它其實是計算機科學裡最強大、 最優雅的演算法設計範式之一。許多你每天間接用到的飛快演算法——排序、搜尋、大數運算、 甚至影像處理裡的快速傅立葉轉換——骨子裡都是同一招。這一章,我們就來把這招練到家。

🔑
分治法的三個動作

面對規模為 $n$ 的問題,分治法做三件事:
① 分解(divide):把問題拆成幾個規模更小、但結構相同的子問題。
② 解決(conquer):遞迴地解這些子問題;小到不能再小時(基底情況),直接給答案。
③ 合併(combine):把子問題的答案,組裝成原問題的答案。
威力與陷阱,往往都藏在第 ① 步和第 ③ 步——怎麼切、怎麼併。

還記得第 3 章嗎?我們說遞迴是「自己呼叫自己」,而每個遞迴都要有基底情況(base case)讓它停手。 分治法就是遞迴的一種特別有用的形態:它不只是「把問題變小再丟給自己」,還在拿回子答案之後, 多做了一步合併。你可以說——分治 = 遞迴 + 合併策略。接下來這一整章, 都是在回答兩個問題:「怎麼設計這個合併?」以及「這樣切、這樣併,到底要花多少時間?」

主角登場:合併排序(merge sort)

第 5 章我們玩過氣泡排序與插入排序,還親手摸到了「平方很慢」的手感($O(n^2)$)。也預告過, 真正能對付大資料的排序是 $O(n \log n)$ 的。現在,第一位主角登場了—— 合併排序,一個教科書等級漂亮的分治演算法。它的點子,只要一句話:

要排好一整串數字?先把它對半切,分別排好左右兩半,再把兩個「已排序」的半邊「合併」成一個有序的整體。

看出三個動作了嗎?分解=對半切;解決=遞迴排好兩半; 合併=把兩個有序的半邊併成一個有序的整體。基底情況是:長度 0 或 1 的陣列, 本來就已經排好了,直接回傳。

關鍵在「合併」:把兩個有序串合成一個

分治演算法的靈魂常常在合併這一步,我們先把它單獨拿出來想清楚。假設你手上有兩疊各自已經由小到大排好 的牌,要併成一疊有序的。怎麼做最快?很簡單:每次只比較兩疊最上面那張,把比較小的那張抽出來放到結果裡, 如此反覆。因為兩疊都已排序,「最小的牌」必定在某一疊的頂端,所以你永遠只需要看兩張就能決定下一個。 當一疊先空了,把另一疊剩下的整段接上去就行。 

def merge(left, right):
    """把兩個『已排序』的陣列合併成一個有序陣列。"""
    result = []
    i = j = 0
    # 兩邊都還有東西時,每次挑出較小的那個放進結果
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:
            result.append(left[i])
            i += 1
        else:
            result.append(right[j])
            j += 1
    # 其中一邊先空了,把另一邊剩下的(本來就有序)整段接上
    result.extend(left[i:])
    result.extend(right[j:])
    return result

print(merge([2, 5, 8], [1, 4, 7, 9]))   # [1, 2, 4, 5, 7, 8, 9]

合併兩個總長 $n$ 的有序陣列,只要掃一遍——每比一次就確定一個元素的歸宿,所以是漂亮的 $O(n)$。 記住這個 $O(n)$,它待會兒會是整個複雜度分析的關鍵。

合起來:完整的合併排序

有了 merge,合併排序的本體短得驚人,因為「排好兩半」這件難事,被它優雅地丟給了自己:

def merge_sort(arr):
    # 基底情況:長度 0 或 1,本身就已排序 → 停手
    if len(arr) <= 1:
        return arr

    mid = len(arr) // 2
    left  = merge_sort(arr[:mid])    # ① 分解 + ② 遞迴解決左半
    right = merge_sort(arr[mid:])    #            遞迴解決右半
    return merge(left, right)        # ③ 合併兩個有序半邊

print(merge_sort([5, 2, 8, 4, 7, 1, 3, 6]))
# [1, 2, 3, 4, 5, 6, 7, 8]

三行核心,精準對應三個動作。這就是分治寫出來的樣子:本體負責「切」與「併」,遞迴負責「啃」中間那塊難的。 讓我們把整個過程攤開來看——先一路往下切到底(分解),再一路往上併回來(合併)。

為什麼是 $O(n \log n)$?

現在來回答那個關鍵問題。看圖 27-3,把目光放在每一層:

  • 最底層:8 個單元素,各併一下——但合併本身的總工作量,正比於那一層所有元素的總數,也就是 $n$。
  • 上一層:4 次合併,每次處理約 2 個元素,$4 \times 2 = 8 = n$。
  • 再上一層:2 次合併,每次約 4 個元素,$2 \times 4 = 8 = n$。
  • 最頂層:1 次合併,處理全部 8 個元素,$= n$。

看出規律了嗎?每一層的合併總工作量,恆為 $O(n)$。為什麼?因為往下一層時,子問題的數量加倍、 但每個的大小減半,兩者相乘剛好抵消。所以總時間就是「每層 $O(n)$」乘以「總共有幾層」。 而從 $n$ 一路對半砍到 1,需要的層數正是 $\log_2 n$。於是:

$$ \text{總時間} = \underbrace{O(n)}_{\text{每層}} \times \underbrace{O(\log n)}_{\text{層數}} = O(n \log n). $$

🔑
$n \log n$ 的直覺:層數 × 每層工作

很多分治演算法的複雜度,都可以這樣「拆成兩個因子」來想:遞迴有幾層(取決於每次把問題縮小多少), 乘上每一層做多少事(取決於分解與合併的成本)。合併排序之所以快,是因為「砍半」讓層數只有 $\log n$, 而合併便宜到每層只要 $O(n)$。把這個「層數 × 每層工作」的直覺記牢,等一下的主定理不過是把它公式化而已。

ℹ️
合併排序的代價:它要額外的空間

天下沒有白吃的午餐。注意 merge 開了一個新的 result 陣列——合併排序需要 $O(n)$ 的 額外記憶體,不像氣泡、插入排序那樣「原地(in-place)」操作。這也是為什麼實務上, 當記憶體很吃緊、或資料量沒那麼大時,人們有時會選擇快速排序(quicksort,平均也是 $O(n \log n)$ 但幾乎原地); 而當你需要「穩定排序」或在意最壞情況保證時,合併排序就非常香。第 28 章我們會把這幾種排序好好比一比。

把快慢寫成方程式:遞迴關係

剛才我們是「看著圖、數層數」算出 $O(n \log n)$ 的。但畫圖數層終究是土法煉鋼,我們需要更系統的工具。 第 15 章埋過一顆種子:遞迴演算法的執行時間,天生就是一條遞迴關係(recurrence relation)。 現在這顆種子要發芽了。

設 $T(n)$ 是合併排序處理 $n$ 個元素所花的時間。讀一遍它的程式碼,把成本逐項翻譯成數學:

  • 把陣列切成兩半,再各自遞迴排好——這是兩個規模 $n/2$ 的子問題,花 $2\,T(n/2)$。
  • 切割(切片)與合併(merge)——花 $O(n)$。

合起來,就得到那條我們在第 15 章見過的式子:

$$ T(n) = 2\,T(n/2) + O(n), \qquad T(1) = O(1). $$

這就是合併排序的「時間方程式」。所謂「解遞迴關係」,就是從這條遞迴定義,推出一個直接用 $n$ 表示的 封閉解(closed form),例如 $T(n) = O(n \log n)$。我們有兩種主力解法。

解法一:遞迴樹法(recursion tree)

遞迴樹法,其實就是把我們剛剛「數層數」的直覺畫成一棵樹、然後加總。把 $O(n)$ 寫成具體的 $c\,n$(某個常數 $c$), 畫出每個節點「自己要做多少工」:

把這個過程寫成式子,就是一個等比/等差的加總:

$$ T(n) = \sum_{i=0}^{\log_2 n} 2^i \cdot c\,\frac{n}{2^i} = \sum_{i=0}^{\log_2 n} c\,n = c\,n\,(\log_2 n + 1) = O(n \log n). $$

遞迴樹法的好處是直覺、看得見:你能一眼看出「工作量是平均分散在每一層,還是集中在頂端或底部」。 這個觀察,正是等一下理解主定理三種情況的鑰匙。

解法二:代入法(substitution method)

遞迴樹幫你「猜」出答案;代入法則用數學歸納法幫你「證」這個猜測是對的(回想第 15 章的「猜測 + 歸納」)。 步驟是:先大膽猜一個封閉解,代回遞迴關係,驗證它撐得住。

以 $T(n) = 2\,T(n/2) + c\,n$ 為例。我們猜 $T(n) \le d\,n \log_2 n$(對某個夠大的常數 $d$)。 歸納假設:對所有比 $n$ 小的規模,這個上界都成立,特別是 $T(n/2) \le d\,\frac{n}{2}\log_2 \frac{n}{2}$。代入:

$$ \begin{aligned} T(n) &= 2\,T(n/2) + c\,n \\ &\le 2\left(d\,\tfrac{n}{2}\log_2 \tfrac{n}{2}\right) + c\,n \\ &= d\,n\,(\log_2 n - 1) + c\,n \\ &= d\,n \log_2 n - d\,n + c\,n \\ &\le d\,n \log_2 n \qquad (\text{只要 } d \ge c). \end{aligned} $$

最後一步:只要我們把 $d$ 選得 $\ge c$,那個 $-d\,n + c\,n$ 就 $\le 0$,於是不等式成立。歸納完成, 證得 $T(n) = O(n \log n)$。代入法的精髓是:你得先有個正確的猜測(通常來自遞迴樹), 然後驗證它在歸納中「自我支撐」。猜錯了(比如猜成 $O(n)$),代入時就會卡住、湊不出來——那正是提醒你回去重猜。

⚠️
代入法的陷阱:別漏了低階項

新手常犯一個錯:想證 $T(n) \le d\,n$,代入後得到 $T(n) \le d\,n + c\,n$,就宣稱「這也是 $O(n)$」。 錯!歸納要求你證出一模一樣的形式 $T(n) \le d\,n$,而你卻多出了一個 $+c\,n$ 甩不掉—— 這代表你的猜測太小了。嚴格的歸納不允許「常數會悄悄長大」。這種卡關,正是數學在告訴你:答案不是 $O(n)$。

本章重頭戲:主定理(Master Theorem)

遞迴樹和代入法都管用,但每次都要畫圖、加總、做歸納,未免太累。好消息是:幾乎所有「規模除以 $b$」型的分治遞迴, 都長同一個樣子,於是數學家替我們把答案一次算好、整理成一張查表卡。這張卡,就是 主定理(Master Theorem)——本章的重頭戲,也是你分析分治演算法時最常掏出來的瑞士刀。

所有這類遞迴都能寫成統一的形狀(第 15 章已預告過):

$$ T(n) = a\,T(n/b) + f(n), \qquad a \ge 1,\ b > 1. $$

三個符號的意思,務必記牢:

  • $a$:分解出幾個子問題(分支數)。合併排序 $a=2$。
  • $b$:每個子問題的規模是原來的 $1/b$(縮小倍率)。合併排序 $b=2$。
  • $f(n)$:分解 + 合併這一層自己要做的工(不含遞迴)。合併排序 $f(n)=O(n)$。

核心:一場「葉子 vs 根」的拔河

主定理的全部精神,可以濃縮成一個比較:把 $n^{\log_b a}$ 和 $f(n)$ 放上天秤,看誰重。 為什麼是這兩個量?回想遞迴樹:

  • $f(n)$ 是樹根(最頂層)那一個節點自己做的工。
  • $n^{\log_b a}$ 則代表所有葉子(最底層)加起來的工。為什麼?樹有 $\log_b n$ 層, 每往下一層分支數乘 $a$,所以葉子數是 $a^{\log_b n} = n^{\log_b a}$ 個——這個指數 $\log_b a$ 是整件事的主角, 我們叫它臨界指數(critical exponent)。

於是問題變成:工作量集中在頂端(根),還是底部(葉子),還是平均分散? 誰重,誰就決定了總複雜度。這就是下面三種情況的由來。

主定理的完整陳述

設 $T(n) = a\,T(n/b) + f(n)$,其中 $a \ge 1,\ b > 1$,並令臨界指數對應的量為 $n^{\log_b a}$。 比較 $f(n)$ 與 $n^{\log_b a}$,分三種情況:

情況條件(誰主導)結論 $T(n)$直覺
Case 1 $f(n) = O\!\left(n^{\log_b a - \varepsilon}\right)$,某 $\varepsilon > 0$
(合併的工明顯小於葉子)		 $\Theta\!\left(n^{\log_b a}\right)$ 葉子主導:成本壓倒性地堆在最底層,總和被葉子數量決定。
Case 2 $f(n) = \Theta\!\left(n^{\log_b a}\right)$
(合併的工恰好等於葉子量級)		 $\Theta\!\left(n^{\log_b a} \log n\right)$ 每層相當:每一層都做差不多的工,所以乘上層數 $\log n$。
Case 3 $f(n) = \Omega\!\left(n^{\log_b a + \varepsilon}\right)$,某 $\varepsilon > 0$,
且滿足規則性條件 $a\,f(n/b) \le c\,f(n)$,某 $c < 1$
(合併的工明顯大於葉子)		 $\Theta\!\left(f(n)\right)$ 根部主導:最頂層那一次合併就最貴,成本被根決定。
💡
套用主定理的四步驟 SOP

看到一條 $T(n) = a\,T(n/b) + f(n)$,照這個流程走,幾秒就有答案:
1. 讀出 $a$、$b$、$f(n)$。
2. 算臨界量 $n^{\log_b a}$。
3. 把 $f(n)$ 和它比大小。
4. 小 → Case 1(答案 $n^{\log_b a}$);一樣大 → Case 2(再乘 $\log n$);大 → Case 3(答案就是 $f(n)$)。

四個套用範例:把卡片用熟

主定理的價值,要實際套幾次才感受得到。我們挑四個代表:

演算法遞迴關係$n^{\log_b a}$$f(n)$情況$T(n)$
合併排序$2T(n/2)+n$$n^{\log_2 2}=n$$n$Case 2$\Theta(n\log n)$
二分搜尋$T(n/2)+1$$n^{\log_2 1}=n^0=1$$1$Case 2$\Theta(\log n)$
Karatsuba 乘法$3T(n/2)+n$$n^{\log_2 3}\approx n^{1.585}$$n$Case 1$\Theta(n^{1.585})$

看出二分搜尋那一列的妙處了嗎?$a=1$ 表示「只遞迴一個子問題」,於是 $n^{\log_2 1} = n^0 = 1$, 而 $f(n)=1$(每次只做一次比較),落在 Case 2,答案是 $\Theta(\log n)$。一條遞迴,主定理替你秒解。

⚠️
主定理不是萬能的

它只處理「子問題大小都是 $n/b$、且符合三種情況之一」的遞迴。有些遞迴會掉進三種情況的縫隙裡—— 例如 $T(n)=2T(n/2)+n\log n$,$f(n)$ 比 $n$ 大,但又沒「多項式倍」地大(只大了一個 $\log$ 因子), 三個 case 都不適用(答案其實是 $\Theta(n\log^2 n)$,需要進階版本或遞迴樹處理)。另外,像 $T(n)=T(n-1)+n$ 這種「規模減 1」而非「除以 $b$」的遞迴,根本不是主定理的菜——那要用第 15 章的展開法。 主定理是利器,但別忘了遞迴樹與代入法這兩把老刀。

✏️ 練習 27.2 思考

用主定理(或在不適用時,改用遞迴樹/展開法)求下列遞迴的複雜度。先各自寫出 $a,b,f(n)$ 與 $n^{\log_b a}$,再判斷情況。

  1. $T(n) = 4\,T(n/2) + n$
  2. $T(n) = 2\,T(n/2) + n^2$
  3. $T(n) = 9\,T(n/3) + n^2$
  4. $T(n) = T(n/2) + 1$
  5. $T(n) = T(n-1) + n$(注意:這條主定理管不到)
看解答

(1) $a=4,b=2,f=n$。$n^{\log_2 4}=n^2$。$f=n=O(n^{2-\varepsilon})$,比 $n^2$ 小 → Case 1 → $T(n)=\Theta(n^2)$。

(2) $a=2,b=2,f=n^2$。$n^{\log_2 2}=n$。$f=n^2=\Omega(n^{1+\varepsilon})$,比 $n$ 大;規則性條件 $2\,(n/2)^2 = n^2/2 \le c\,n^2$($c=\tfrac12<1$)成立 → Case 3 → $T(n)=\Theta(n^2)$。

(3) $a=9,b=3,f=n^2$。$n^{\log_3 9}=n^2$。$f=n^2=\Theta(n^2)$,正好相等 → Case 2 → $T(n)=\Theta(n^2\log n)$。

(4) $a=1,b=2,f=1$。$n^{\log_2 1}=n^0=1$。$f=1=\Theta(1)$ → Case 2 → $T(n)=\Theta(\log n)$。(這正是二分搜尋。)

(5) 這是「規模減 1」型,主定理不適用。用展開法:$T(n)=T(n-1)+n = T(n-2)+(n-1)+n=\dots = n+(n-1)+\dots+1 = \frac{n(n+1)}{2}=\Theta(n^2)$。 (這也正是選擇排序/插入排序最壞情況的形狀。)

分治經典巡禮

工具齊備,該來看看分治這把劍能砍倒哪些經典問題了。有些你已經見過(二分搜尋、快速冪), 我們用「分治 + 主定理」的新眼光重看一遍;有些則是會讓你眼睛一亮的新傢伙。

① 二分搜尋:其實是「退化的」分治

第 5 章的二分搜尋,本質上就是分治——只是它特別「省」:分解成兩半後, 它只需要遞迴其中一半(因為目標只可能在一邊),另一半直接丟掉,也不需要合併。 

def binary_search(arr, target, lo=0, hi=None):
    if hi is None:
        hi = len(arr) - 1
    if lo > hi:
        return -1                                       # 範圍空了 → 找不到
    mid = (lo + hi) // 2
    if arr[mid] == target:
        return mid
    elif arr[mid] < target:
        return binary_search(arr, target, mid + 1, hi)  # 只往右半遞迴
    else:
        return binary_search(arr, target, lo, mid - 1)  # 只往左半遞迴

print(binary_search([2, 5, 8, 12, 16, 23, 38], 23))   # 5

它的遞迴關係是 $T(n) = T(n/2) + O(1)$:一個子問題、規模減半、每層只做 $O(1)$ 的比較。

ℹ️
「減治」:分治的近親

像二分搜尋、快速冪這種「分解後只解一個子問題」的策略,有時被細分為 減治法(decrease and conquer)——它把問題「縮小」而非「切開成多塊」。 名稱不必太計較;重點是體會:不論叫什麼,核心都是「把規模快速縮小,讓層數變成 $\log n$」。

② 快速冪:把次方從 $O(n)$ 壓成 $O(\log n)$

第 15 章的快速冪(平方求冪)也是同一個家族。要算 $a^n$,笨方法是乘 $n$ 次($O(n)$); 分治的洞見是:$a^n = (a^{n/2})^2$——只要算一半的次方,再平方一下: 

def fast_pow(base, exp):
    if exp == 0:
        return 1
    half = fast_pow(base, exp // 2)   # 把問題砍半:只算一次
    if exp % 2 == 0:
        return half * half            # 偶數次方:平方即可
    else:
        return half * half * base     # 奇數次方:平方後再補一個 base

print(fast_pow(3, 10))   # 59049

遞迴關係 $T(n) = T(n/2) + O(1) = O(\log n)$,和二分搜尋同款。注意實作上的一個細節: 我們先把 half 算出來、存起來,再平方。如果寫成 fast_pow(base, exp//2) * fast_pow(base, exp//2),那就變成兩個子問題、 $T(n)=2T(n/2)+O(1)=O(n)$——分治的威力瞬間蒸發。「算一次、用兩次」這個小動作,是這裡的命脈。

③ Karatsuba:大數相乘,打破 $O(n^2)$ 的直覺

現在來看分治最令人拍案的一個成就。兩個 $n$ 位數相乘,你小學學的直式乘法,要把上面每一位乘下面每一位, 共約 $n \times n = n^2$ 次個位數乘法——$O(n^2)$。幾百年來大家都覺得「相乘就是這麼貴,沒辦法」。 直到 1960 年,一位 23 歲的研究生 Karatsuba(卡拉楚巴)證明了:可以更快。

用分治。把一個 $n$ 位數 $x$ 從中間切成高位 $x_1$ 和低位 $x_0$(各約 $n/2$ 位),$y$ 同理:

$$ x = x_1 \cdot 10^{n/2} + x_0, \qquad y = y_1 \cdot 10^{n/2} + y_0. $$

那麼乘積展開是:

$$ x \cdot y = \underbrace{x_1 y_1}_{z_2}\cdot 10^{n} + \underbrace{(x_1 y_0 + x_0 y_1)}_{\text{中間項}}\cdot 10^{n/2} + \underbrace{x_0 y_0}_{z_0}. $$

直接算,需要 4 次 小乘法($x_1y_1, x_1y_0, x_0y_1, x_0y_0$),遞迴關係 $T(n)=4T(n/2)+O(n)$, 主定理 Case 1 一算 → $n^{\log_2 4}=n^2$,還是 $O(n^2)$,白忙一場。

Karatsuba 的神來一筆是:那個中間項 $x_1 y_0 + x_0 y_1$,可以不必另外算兩次乘法。注意到:

$$ (x_1 + x_0)(y_1 + y_0) = x_1 y_1 + (x_1 y_0 + x_0 y_1) + x_0 y_0 = z_2 + \text{中間項} + z_0. $$

所以只要算一次 $z_1 = (x_1+x_0)(y_1+y_0)$,中間項 $= z_1 - z_2 - z_0$,用減法換掉了一次乘法! 於是只需 3 次 遞迴乘法($z_2, z_0, z_1$):

def karatsuba(x, y):
    if x < 10 or y < 10:          # 基底:夠小就直接交給硬體乘
        return x * y

    n = max(len(str(x)), len(str(y)))
    half = n // 2
    p = 10 ** half

    x_hi, x_lo = divmod(x, p)     # x = x_hi * p + x_lo
    y_hi, y_lo = divmod(y, p)

    z2 = karatsuba(x_hi, y_hi)                          # 1 次
    z0 = karatsuba(x_lo, y_lo)                          # 2 次
    z1 = karatsuba(x_hi + x_lo, y_hi + y_lo) - z2 - z0 # 3 次

    return z2 * (10 ** (2 * half)) + z1 * p + z0

print(karatsuba(1234, 5678))   # 7006652(= 1234 * 5678,答對!)

遞迴關係 $T(n) = 3\,T(n/2) + O(n)$。主定理:$n^{\log_2 3} \approx n^{1.585}$,$f(n)=n$ 小於它 → Case 1 → $T(n) = \Theta(n^{1.585})$。對 $n$ 不大時看不出差別,但當你要乘的是上千位的大數(密碼學裡天天發生), $n^{1.585}$ 對 $n^2$ 是貨真價實的勝利。這是分治第一次打破一個大家以為理所當然的速度下限—— 而它不會是最後一次。

📜
一堂課推翻的猜想

1960 年,大數學家 Kolmogorov(柯爾莫哥洛夫)在一場研討會上猜測:$n$ 位數相乘 「不可能比 $O(n^2)$ 更快」。台下 23 歲的 Karatsuba 不到一週就想出了上面這個 $O(n^{1.585})$ 的方法, 當場推翻了大師的猜想。Kolmogorov 大為震撼,還幫他把結果寫成論文發表。這個故事提醒我們: 「看起來顯然的下限」,往往只是還沒人想到更聰明的切法。

✏️ 練習 27.3 思考

(a) 親手用主定理驗證 $T(n)=3T(n/2)+O(n)$ 確實是 $\Theta(n^{\log_2 3})$,並算出 $\log_2 3$ 到小數點後三位。 (b) 假設有人發明了「更猛」的方法,只用 2 次遞迴乘法(即 $T(n)=2T(n/2)+O(n)$), 複雜度會變成多少?這比 Karatsuba 好嗎?(c) 反過來,如果手滑寫成 4 次乘法,又會退回多少?

看解答

(a) $a=3,b=2$,$n^{\log_2 3}$。$f(n)=n=n^1$,而 $\log_2 3 \approx 1.585 > 1$, 所以 $f(n)=O(n^{\log_2 3 - \varepsilon})$(取 $\varepsilon=0.5$ 即可),落 Case 1 → $\Theta(n^{\log_2 3})=\Theta(n^{1.585})$。 計算 $\log_2 3 = \frac{\ln 3}{\ln 2} = \frac{1.0986}{0.6931} \approx 1.585$。

(b) $T(n)=2T(n/2)+O(n)$ → Case 2 → $\Theta(n\log n)$。這遠優於 Karatsuba! 事實上,把整數乘法做到接近 $O(n\log n)$ 是真實的研究目標——2019 年 Harvey 與 van der Hoeven 證明了 $O(n\log n)$ 的整數乘法演算法(用的是快速傅立葉轉換 FFT 這類更深的分治,不是簡單的二分)。

(c) $T(n)=4T(n/2)+O(n)$ → $n^{\log_2 4}=n^2$ → Case 1 → $\Theta(n^2)$, 退回樸素直式乘法。這說明 Karatsuba 的全部魔法,就濃縮在「把 4 次降成 3 次」這一個動作上。

④ 最大子陣列:在起伏的數列裡找最賺的一段

新問題。給你一串有正有負的數(想像成某支股票每天的漲跌),找出連續的一段,讓這段的總和最大。 例如 [-2, 1, -3, 4, -1, 2, 1, -5, 4],最賺的一段是 [4, -1, 2, 1],總和 6。 暴力枚舉所有「起點 + 終點」是 $O(n^2)$。分治能不能更快?

分治的切入點:把陣列從中間切成左右兩半。那麼,「最大的一段」只可能是三種情況之一:

  • 完全落在左半——遞迴去左半找。
  • 完全落在右半——遞迴去右半找。
  • 橫跨中線——這段必須同時包含中點與中點右邊的元素。

前兩種交給遞迴;關鍵是第三種「橫跨」怎麼算。其實不難:從中線往左累加、記下最大的左半和,再從中線往右累加、記下最大的右半和,兩者一拼就是最佳的橫跨段——這一步只要掃一遍,$O(n)$。

def max_subarray(arr):
    return _solve(arr, 0, len(arr) - 1)

def _solve(arr, lo, hi):
    if lo == hi:                       # 基底:只剩一個元素
        return arr[lo]
    mid = (lo + hi) // 2
    best_left  = _solve(arr, lo, mid)        # 全在左半
    best_right = _solve(arr, mid + 1, hi)    # 全在右半
    best_cross = _crossing(arr, lo, mid, hi) # 橫跨中線
    return max(best_left, best_right, best_cross)

def _crossing(arr, lo, mid, hi):
    # 從 mid 往左累加,取最大的左半和(這段一定含 mid)
    s, left_best = 0, float("-inf")
    for i in range(mid, lo - 1, -1):
        s += arr[i]
        left_best = max(left_best, s)
    # 從 mid+1 往右累加,取最大的右半和(這段一定含 mid+1)
    s, right_best = 0, float("-inf")
    for i in range(mid + 1, hi + 1):
        s += arr[i]
        right_best = max(right_best, s)
    return left_best + right_best

print(max_subarray([-2, 1, -3, 4, -1, 2, 1, -5, 4]))   # 6

遞迴關係又是熟悉的 $T(n) = 2\,T(n/2) + O(n)$(兩個半邊 + 一次 $O(n)$ 的橫跨掃描)→ 主定理 Case 2 → $O(n \log n)$。 比暴力的 $O(n^2)$ 好。不過先別太得意——

📚
伏筆:這題其實有 $O(n)$ 的解

最大子陣列有一個叫 Kadane 演算法 的線性解,只要掃一遍($O(n)$),比分治版還快。 那為什麼還教分治版?因為它是練習分治思維的絕佳教材,也因為「分治不一定是最優解」本身就是重要一課。 Kadane 的精神是「動態規劃」——而動態規劃,正是當分治遇上「重疊子問題」時該換上的武器。 這個故事我們留到第 32 章。

✏️ 練習 27.4 實作

擴充上面的 max_subarray,讓它不只回傳最大總和,還回傳那段子陣列的起訖索引 (start, end)。 測試 [-2, 1, -3, 4, -1, 2, 1, -5, 4] 應得到總和 6、索引 (3, 6)。

看解答

把每個函式的回傳值從「一個數」升級成「(總和, 起, 訖)」的三元組,在比較時帶著索引一起傳遞:

def max_subarray(arr):
    return _solve(arr, 0, len(arr) - 1)   # 回傳 (sum, start, end)

def _solve(arr, lo, hi):
    if lo == hi:
        return (arr[lo], lo, hi)
    mid = (lo + hi) // 2
    left  = _solve(arr, lo, mid)
    right = _solve(arr, mid + 1, hi)
    cross = _crossing(arr, lo, mid, hi)
    return max(left, right, cross, key=lambda t: t[0])   # 比第一個元素(總和)

def _crossing(arr, lo, mid, hi):
    s, left_best, left_i = 0, float("-inf"), mid
    for i in range(mid, lo - 1, -1):
        s += arr[i]
        if s > left_best:
            left_best, left_i = s, i
    s, right_best, right_j = 0, float("-inf"), mid + 1
    for j in range(mid + 1, hi + 1):
        s += arr[j]
        if s > right_best:
            right_best, right_j = s, j
    return (left_best + right_best, left_i, right_j)

print(max_subarray([-2, 1, -3, 4, -1, 2, 1, -5, 4]))   # (6, 3, 6)

小技巧:用 max(..., key=lambda t: t[0]) 讓 max 只比三元組的第一個欄位(總和), 但回傳整個三元組,索引就跟著被選出來了。

⑤ 逆序對計數:把排序「順便」拿來數亂度

最後一個、也是最巧妙的一個。給一個陣列,數出有多少對 逆序對(inversion)—— 也就是有多少對 $(i, j)$ 滿足 $i < j$ 但 $a[i] > a[j]$(前面的數比後面的大,「順序不對」的一對)。 逆序對的數量,衡量了一個陣列「離排好序有多遠」:完全排好的陣列有 0 對,完全顛倒的有最多。 (它也是推薦系統裡比較「兩個人品味差異」的常用指標。)

暴力檢查所有配對是 $O(n^2)$。神奇的是,我們可以魔改合併排序,在排序的同時把逆序對「順便」數出來, 達到 $O(n \log n)$。關鍵洞察都在合併這一步:當我們從右半取出一個元素 right[j] 放進結果時—— 如果此刻左半還剩下 k 個元素沒取,那這 k 個全都比 right[j] 大 (因為左半已排序,剩下的都是較大的),而它們的位置又都在 right[j] 前面—— 於是一口氣產生了 k 個逆序對! 

def count_inversions(arr):
    _, inv = _sort_count(arr)
    return inv

def _sort_count(arr):
    if len(arr) <= 1:
        return arr, 0
    mid = len(arr) // 2
    left,  inv_l = _sort_count(arr[:mid])
    right, inv_r = _sort_count(arr[mid:])
    merged, inv_split = _merge_count(left, right)
    return merged, inv_l + inv_r + inv_split   # 左半內 + 右半內 + 橫跨的

def _merge_count(left, right):
    result, i, j, inv = [], 0, 0, 0
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:
            result.append(left[i]); i += 1
        else:
            result.append(right[j]); j += 1
            inv += len(left) - i      # 左半剩 (len-i) 個都比 right[j] 大
    result.extend(left[i:])
    result.extend(right[j:])
    return result, inv

print(count_inversions([2, 4, 1, 3, 5]))   # 3  → (2,1),(4,1),(4,3)

這就是分治的優雅:逆序對數量 = 左半內部的逆序對 + 右半內部的逆序對 + 橫跨兩半的逆序對。 前兩項由遞迴給你,第三項在合併時「順手」數完。和合併排序同款的遞迴關係,$O(n \log n)$。 本來要 $O(n^2)$ 的計數,被排序「捎帶」著解決了——這種「借力使力」的設計品味,值得細細回味。

✏️ 練習 27.5 實作

(a) 寫一個 $O(n^2)$ 的暴力逆序對計數當「對照組」。(b) 用隨機資料,驗證上面 $O(n\log n)$ 的版本和暴力版答案永遠一致。 (c) 一個 $n$ 個相異元素的陣列,逆序對最多有幾個?哪種排列達到這個最大值?

看解答 
import random

def brute_inversions(arr):
    inv = 0
    for i in range(len(arr)):
        for j in range(i + 1, len(arr)):
            if arr[i] > arr[j]:
                inv += 1
    return inv

# 對拍:跑 500 次隨機測試
for _ in range(500):
    a = [random.randint(0, 20) for _ in range(random.randint(0, 25))]
    assert count_inversions(a) == brute_inversions(a)
print("分治版與暴力版答案一致 ✅")

(c) 最多是 $\binom{n}{2} = \frac{n(n-1)}{2}$ 個——當陣列完全反向排序(由大到小)時, 任意一對 $(i,j)$ 都是逆序對,全部 $\binom{n}{2}$ 對都中。這也是個 $\Theta(n^2)$ 的數量, 卻能在 $O(n\log n)$ 內數完,正是這個演算法漂亮的地方:答案本身可能很大,但我們從不真的「一對一對」去數。

Strassen:連矩陣相乘的下限都能被打破

一句話帶過、但震撼力十足的最後一個例子。兩個 $n \times n$ 矩陣相乘,定義式要算 $n^3$ 次乘法,$O(n^3)$。 把矩陣切成四個 $\frac{n}{2}\times\frac{n}{2}$ 的小塊做分治,樸素做法要 8 次小矩陣乘法,$T(n)=8T(n/2)+O(n^2)$, 主定理算下來還是 $O(n^3)$。但 1969 年 Strassen(史特拉森)用一組精巧的加減組合, 硬是把 8 次乘法降成 7 次(精神和 Karatsuba 如出一轍!),得到 $T(n)=7T(n/2)+O(n^2)$, 於是 $n^{\log_2 7} \approx n^{2.807}$——

$$ O(n^3) \;\longrightarrow\; O(n^{2.807}). $$

又一次,分治打破了一個「大家以為不可能更快」的下限。Strassen 之後,矩陣相乘的指數被一路壓低 (目前理論最佳已逼近 $n^{2.37}$),成為理論計算機科學一個著名的長跑賽。你不必記住細節, 只要記住這個母題:「少做一次遞迴乘法」這個小聰明,常常就足以撬動整個複雜度的指數。

🔑
Karatsuba 與 Strassen 的共同心法

兩者都遵循同一個套路:分治時,把子問題的數量 $a$ 想辦法減少(Karatsuba 從 4 降到 3, Strassen 從 8 降到 7),代價是多做幾次便宜的加減法($f(n)$ 不變)。因為複雜度的指數是 $\log_b a$, 只要 $a$ 變小,整條曲線就往下彎。這提醒你:優化分治演算法,最值錢的方向往往不是「讓合併更快」, 而是「想辦法少遞迴幾次」。

分治的疆界:它什麼時候不是答案?

學會一把好用的鎚子,容易把每個問題都看成釘子。所以最後,我們得誠實談談分治的限制。 分治之所以高效,有一個常被忽略的前提:

切出來的子問題,必須彼此獨立——解一個不需要另一個的結果,而且它們不會重複求解同樣的東西。

合併排序、Karatsuba、最大子陣列都滿足這點:左半和右半井水不犯河水。但有一大類問題, 切出來的子問題會嚴重重疊——同一個小問題被反覆求解成千上萬次。最經典的例子, 就是第 3 章那個樸素遞迴版的費氏數列: 

def fib(n):
    if n < 2:
        return n
    return fib(n - 1) + fib(n - 2)   # 兩個子問題,但它們大量重疊!

算 fib(5) 要算 fib(4) 和 fib(3);算 fib(4) 又要算 fib(3) 和 fib(2)…… fib(3) 被算了兩次,fib(2) 被算了三次,越往下重複越誇張, 整體爆炸成指數時間 $O(\varphi^n)$(第 15 章算過)。這時候硬套分治,只是在做大量無謂的重工。

🔑
分治 vs 動態規劃:看「子問題重不重疊」

這是一條重要的分水嶺:
子問題彼此獨立、不重複 → 用分治(切開、各自解、合併)。
子問題大量重疊 → 用動態規劃(dynamic programming):把每個子問題的答案算一次就存起來, 下次直接查表,避免重工。
兩者都靠「把大問題拆成小問題」,差別只在「小問題會不會撞車」。記住這把尺,你就知道面對新問題該掏哪把刀—— 動態規劃的完整故事,第 32 章見。

✏️ 練習 27.6 挑戰

實作 quickselect:在未排序的陣列裡,找出第 $k$ 小的元素(例如中位數), 而不要先把整個陣列排好。提示:借用快速排序「選一個基準點 pivot、把元素分成『比它小/相等/比它大』三堆」的點子—— 但你只需要遞迴其中一堆(這又是「減治」)。分析它的平均與最壞複雜度。

看解答 
import random

def quickselect(arr, k):
    """回傳 arr 中第 k 小的元素(k 從 1 開始計)。"""
    if not 1 <= k <= len(arr):
        raise ValueError("k 超出範圍")

    pivot = random.choice(arr)                 # 隨機選基準,避開最壞情況
    smaller = [x for x in arr if x < pivot]
    equal   = [x for x in arr if x == pivot]
    larger  = [x for x in arr if x > pivot]

    if k <= len(smaller):
        return quickselect(smaller, k)         # 答案在「小」這堆
    elif k <= len(smaller) + len(equal):
        return pivot                           # 正好落在 pivot 這群
    else:                                      # 答案在「大」這堆
        return quickselect(larger, k - len(smaller) - len(equal))

data = [7, 2, 9, 4, 1, 6, 3]
print(quickselect(data, 3))   # 第 3 小 = 3(排序後 [1,2,3,4,6,7,9])

複雜度。 每次分割花 $O(n)$,然後只遞迴一邊。若每次都幸運切在中間附近,規模呈幾何級數遞減, 遞迴關係近似 $T(n)=T(n/2)+O(n)$,主定理 Case 3 → 平均 $O(n)$(注意:這比「先排序再取」的 $O(n\log n)$ 還快!)。 但最壞情況(每次 pivot 都選到最大或最小)會退化成 $T(n)=T(n-1)+O(n)=O(n^2)$——所以我們隨機選 pivot, 讓最壞情況幾乎不可能發生。想要 $O(n)$ 的最壞保證,有個叫「中位數的中位數(median of medians)」的進階技巧, 那又是另一段分治的故事了。

✏️ 練習 27.7 挑戰

(概念題,不必寫完整程式)平面上有 $n$ 個點,要找出距離最近的一對點。 暴力比較所有配對是 $O(n^2)$。請描述一個 $O(n \log n)$ 的分治策略, 特別說明「橫跨中線的那一對」這個棘手情況該怎麼在線性時間內處理。

看解答

這是分治的經典壓軸題「最近點對(closest pair of points)」。骨架和最大子陣列一模一樣:

  1. 分解:先把所有點依 $x$ 座標排序,用一條垂直線把點分成左右各半。
  2. 解決:遞迴求左半的最近點對距離 $d_L$、右半的 $d_R$,令 $d = \min(d_L, d_R)$。
  3. 合併:最近的一對也可能橫跨中線。關鍵洞察:這樣的一對,兩個點都必須落在中線左右各寬 $d$ 的狹長帶狀區內(否則它們的距離就超過 $d$ 了,不可能更近)。

狹長帶裡的點看似還是很多,但有一個漂亮的幾何事實救了我們:把帶內的點依 $y$ 座標排序後, 每個點只需要和它後面少數幾個點(可證明至多 7 個)比較就夠了——因為在一個 $d \times 2d$ 的小框裡, 彼此距離都 $\ge d$ 的點裝不下太多個。於是「合併」這步是 $O(n)$。

遞迴關係 $T(n) = 2\,T(n/2) + O(n)$,主定理 Case 2 → $O(n \log n)$。 (實作細節上,通常會在遞迴中一併維護依 $y$ 排序的點集,避免每層重新排序。) 這題完美收束了本章:看似只能 $O(n^2)$ 暴力的幾何問題,靠「切一半 + 聰明地處理橫跨情況」, 一樣被壓進了 $n\log n$。

把這一章串起來

我們從一塊對半切開的披薩出發,抵達了能撼動矩陣相乘指數的演算法前沿。回顧這趟旅程的路標:

  • 分治法 = 遞迴 + 合併:分解 → 解決(遞迴)→ 合併。威力與陷阱都在「怎麼切、怎麼併」。
  • 合併排序是分治的典範:每層合併 $O(n)$、共 $\log n$ 層 → $O(n\log n)$。「層數 × 每層工作」是貫穿全章的直覺。
  • 遞迴演算法的時間天生是遞迴關係:用遞迴樹法畫出來加總(看得見),用代入法(歸納)嚴格驗證。
  • 主定理是查表卡:$T(n)=aT(n/b)+f(n)$,比較 $f(n)$ 與 $n^{\log_b a}$ 的大小——葉子贏(Case 1)、平手(Case 2,多乘 $\log n$)、根贏(Case 3)。
  • 經典群像:二分搜尋、快速冪($O(\log n)$);Karatsuba($O(n^{1.585}))$、Strassen($O(n^{2.807})$)用「少遞迴一次」打破速度下限;最大子陣列、逆序對計數($O(n\log n)$)。
  • 疆界:子問題獨立 → 分治;子問題重疊 → 動態規劃(第 32 章)。

分治教會我們一種看世界的方式:再龐大的難題,只要找對「切口」,都能化約成幾個你已經會解的小問題。 下一章,我們會把這套思維帶進更多排序與選擇演算法,看看快速排序如何把分治玩出隨機化的火花。 把這一章的遞迴樹與主定理練成肌肉記憶——它們會是你往後分析任何遞迴演算法時,最先掏出來的兩把刀。
