終端機與 Shell 指令稿

滑鼠能點到的事,終究有限;真正的力量,藏在那個閃著游標、看起來不太友善的黑底白字視窗裡。學會它,你就從「使用電腦的人」變成「指揮電腦的人」。

滑鼠點不到的地方

想像你接到一個任務:把某個資料夾裡 3,000 張照片,檔名從 IMG_xxxx.jpeg 全部改成 photo_xxxx.jpg。用滑鼠的話,你得一張一張右鍵、重新命名、改副檔名…… 手指按到抽筋,大概要花一整個下午,而且難保不出錯。

但如果你會用終端機,這件事是一行指令、不到一秒就完成的事。而且明天再來 3,000 張, 你只要把同一行指令再敲一次(或按一下「向上鍵」叫回歷史紀錄)。

這就是滑鼠(graphical user interface, GUI)與鍵盤(command-line interface, CLI)最根本的差異。 GUI 親切、好上手、所見即所得,適合探索;但它有四件事天生做不好,而這四件事,正是工程的日常:

  • 力量(power):很多功能根本沒有按鈕。GUI 只能露出開發者預先想到的那幾個選項;命令列則直通系統底層,你想得到的它幾乎都做得到。
  • 可自動化(automation):滑鼠的每一次點擊都得由「人」親自執行;指令可以寫進檔案,讓電腦半夜三點自己跑一萬次,你在睡覺。
  • 可組合(composability):GUI 的每個程式各自為政,彼此很難協作;命令列工具天生就被設計成「可以串起來」,小工具拼成大工具。
  • 可重現(reproducibility):「我剛剛點了哪幾個地方?」很難說清楚;但一行指令可以原封不動貼給同事、寫進文件、存進版本控制,別人照著跑就得到一模一樣的結果。
🔑
滑鼠是給「人」用的,命令列是給「人 + 機器」用的

GUI 的操作活在你的肌肉記憶裡,別人看不到、機器重複不了。命令列把「做一件事」變成一段可以被儲存、重播、分享、組合的文字。 一旦一件事能被寫成文字,它就能被自動化——這是工程生產力的分水嶺。

這一章不會要你背一百個指令。我們的目標是讓你「住進」終端機:建立直覺、看懂它的設計哲學, 然後手裡握著二十來個最常用、最能組合的工具,就足以應付絕大多數日常。剩下的,你會在用的過程中自然學會。

終端機、Shell、Bash:先把名詞理清楚

初學者最容易在這三個詞上打結,我們花一分鐘講清楚,因為觀念對了,後面都順:

  • 終端機(terminal):那個視窗本身——負責顯示文字、接收你的鍵盤輸入。它只是個「螢幕 + 鍵盤」的門面,本身不懂任何指令。在 macOS 叫 Terminal 或 iTerm,Linux 上有 GNOME Terminal 等,Windows 上是 Windows Terminal。
  • Shell(殼層):躲在終端機背後、真正讀你的指令、解讀它、然後叫作業系統去做事的程式。它之所以叫「殼」,是因為它包在作業系統核心(kernel)外面,當你和核心之間的翻譯官。
  • Bash / Zsh:Shell 不只一種,它們是具體的 Shell 程式。Bash(Bourne Again SHell)是 Linux 上最經典、最通用的;Zsh 則是 macOS 現在的預設,功能更花俏,但語法和 Bash 高度相容。本章教的東西兩者通用。
ℹ️
一個比喻

把終端機想成「電話聽筒」(你對它講話、它把聲音放出來),把 Shell 想成「電話另一端聽你說話、幫你轉達的接線生」。 Bash 和 Zsh 則是兩位風格不同、但都很能幹的接線生。你對著聽筒講的話(指令),由接線生負責真正執行。

打開終端機,你會看到一個提示字元(prompt),長得大概像這樣: 

akira@laptop:~/projects$

別被它嚇到,它其實在跟你報告很多事:akira 是你的使用者名稱,laptop 是這台機器的名字, ~/projects 是你目前所在的資料夾(~ 代表你的家目錄), 最後那個 $ 表示「我準備好了,請下指令」(若是 # 則代表你是超級使用者 root,要格外小心)。 游標就在 $ 後面閃爍,等你說話。來下你的第一個指令吧: 

$ echo "Hello, Shell"
Hello, Shell

$ date
2026 年 6 月 29 日 週一 14:03:01 CST

$ whoami
akira

echo 把你給它的文字原樣印出來;date 報時間;whoami 告訴你你是誰。 恭喜,你已經跟電腦對上話了。（本章往後的範例,行首的 $ 都代表「這是你輸入的指令」, 下一行起沒有 $ 的就是電腦的回應,你自己打的時候不用把 $ 打進去。)

在檔案系統裡導航:你現在在哪?

用 GUI 時,你「看得到」自己在哪個資料夾;用命令列,你得自己心裡有數。所以第一組要學的指令, 就是回答三個問題:我在哪?這裡有什麼?我要去哪?

在這之前,先建立一個關鍵心智模型:整個檔案系統是一棵樹(tree)。最頂端是根目錄(root), 寫作 /;從它往下長出一層層的資料夾與檔案。你隨時都「站在」這棵樹的某個節點上,那個位置叫 目前工作目錄(current working directory)。

pwd、ls、cd:三個你每天會用上百次的指令

pwd(print working directory)告訴你「我在哪」:

$ pwd
/home/akira/projects

ls(list)告訴你「這裡有什麼」。它本身很樸素,但搭配旗標(flag)就威力大增:

$ ls
notes.txt   photos   report.pdf   src

$ ls -l            # -l (long):一行一個,附上權限、大小、修改時間等細節
-rw-r--r--  1 akira staff   2048 Jun 29 14:03 notes.txt
drwxr-xr-x  5 akira staff    160 Jun 28 09:11 photos
-rw-r--r--  1 akira staff  88123 Jun 20 17:45 report.pdf
drwxr-xr-x  8 akira staff    256 Jun 29 13:30 src

$ ls -a            # -a (all):連「隱藏檔」(檔名以 . 開頭的)也列出
.   ..   .git   .env   notes.txt   photos   report.pdf   src

$ ls -lah          # 旗標可以合併!-l -a -h 寫成 -lah,-h 讓大小變「人看得懂」
-rw-r--r--  1 akira staff   2.0K Jun 29 14:03 notes.txt
drwxr-xr-x  8 akira staff   256B Jun 29 13:30 src

留意 ls -l 開頭那串 drwxr-xr-x:第一個字母 d 代表這是目錄(directory), - 則代表普通檔案;後面那串 rwx 是權限,我們稍後會專門拆解它。

cd(change directory)讓你「移動」到別的資料夾。這裡有幾個你必須熟到反射的縮寫:

$ cd photos        # 進入子目錄 photos(這是「相對路徑」)
$ cd ..            # .. = 上一層。回到 photos 的母目錄
$ cd ~             # ~ = 你的家目錄。等同單獨打一個 cd
$ cd /usr/local    # 以 / 開頭 = 「絕對路徑」,從根目錄算起
$ cd -             # - = 回到「上一個」待過的目錄(像瀏覽器的上一頁)
$ cd               # 不帶參數,直接回家
🔑
絕對路徑 vs 相對路徑

絕對路徑(absolute path)以 / 開頭,從根目錄一路寫到底,無論你現在站在哪,它都指向同一個地方: /home/akira/projects/app.py。
相對路徑(relative path)從你「目前位置」算起:src/main.py、../notes.txt。 記住兩個特殊符號:. 是「目前目錄」,.. 是「上一層目錄」。

💡
兩個會改變你一生的按鍵

Tab 鍵:打一半按 Tab,Shell 會幫你自動補完檔名或指令(打 cd pho 按 Tab → 變成 cd photos/)。 這不只省打字,更能防打錯。
向上鍵 ↑:叫回你剛剛下過的指令,不必重打。配上 history 指令看完整歷史,你會發現自己很少需要把一整行重打第二次。

動手操作檔案:建立、複製、移動、刪除、查看

知道怎麼走路之後,該學怎麼「搬東西」了。這組指令是你每天的勞動工具:

$ mkdir backup                  # make directory:建立資料夾
$ mkdir -p project/src/utils    # -p:一次把整條不存在的路徑都建好
$ touch app.py                  # 建立一個空檔(若檔案已存在,則更新它的時間戳)

$ cp notes.txt notes.bak        # copy:複製(原檔保留)
$ cp -r photos photos_backup    # -r (recursive):整個資料夾連同內容一起複製

$ mv notes.bak archive/         # move:把檔案搬進 archive/ 資料夾
$ mv old.txt new.txt            # 來源與目的在同一層 = 等於「改名」!

$ rm notes.bak                  # remove:刪除檔案
$ rmdir backup                  # 刪除「空」目錄

有個冷知識會讓你恍然大悟:Unix 沒有「重新命名」這個指令。改名,其實就是「搬到同一個地方、換個名字」, 所以用的是 mv。這種「一個工具做多種事」的設計,是 Unix 一貫的簡約風格。

⛔
rm -rf:命令列裡最危險的四個字元

rm 的兩個旗標合起來威力驚人也致命:-r 遞迴刪除整個資料夾,-f(force)不問你、不進回收桶、直接抹除。 rm -rf old_project/ 會把那個資料夾連同裡面所有東西瞬間蒸發,沒有「還原」這回事。

最惡名昭彰的事故是 rm -rf /(刪掉整台機器)或 rm -rf $DIR/——如果變數 $DIR 不小心是空的, 這行就變成了 rm -rf /。下這種指令前,務必先用 ls 確認路徑、養成把變數加引號的習慣, 重要操作前先 echo 一遍看看會展開成什麼。在命令列,「謹慎」不是美德,是保命技能。

查看檔案內容,有四個工具,各有適用場合:

$ cat notes.txt             # 把整個檔案一次倒到螢幕上(短檔案最方便)
$ less report.txt           # 分頁瀏覽長檔案(按空白鍵翻頁、/ 搜尋、q 離開)
$ head notes.txt            # 預設看「前 10 行」
$ head -n 3 notes.txt       # 只看前 3 行
$ tail -n 20 server.log     # 看「最後 20 行」
$ tail -f server.log        # -f (follow):持續追蹤新寫入的內容——看即時 log 的神器

cat 適合短檔;檔案一長,整個畫面被洗過去,這時該用 less 慢慢看。 而 tail -f 是工程師盯伺服器記錄檔的標準姿勢:它會卡在那裡,有新的一行寫進去就即時冒出來。

萬用字元與尋找:一次處理一整批檔案

前面我們都在處理「一個」檔案。但命令列真正的爆發力,在於一次對付一整批。 關鍵是萬用字元(wildcard / globbing)——讓 Shell 幫你把符合某個模式的檔名全部展開: 

$ ls *.txt              # *  = 任意長度的任意字元。抓所有 .txt 結尾的檔
$ ls report?.pdf        # ?  = 剛好一個任意字元。配 report1.pdf、reportA.pdf
$ ls img_[0-9].png      # [] = 中括號內任一字元。配 img_0.png ~ img_9.png
$ ls *.{jpg,png}        # {} = 大括號展開。同時抓 .jpg 和 .png
$ cp *.txt backup/      # 把所有 .txt 一次複製進 backup/
⚠️
是 Shell 在展開,不是指令

當你打 ls *.txt,其實是 Shell 先把 *.txt 展開成 a.txt b.txt c.txt,再把這串檔名交給 ls。ls 自己根本沒看過那個 *。 這解釋了一個常見的坑:如果目錄裡沒有任何 .txt,*.txt 展不開,多數 Shell 會把它原樣傳過去,於是出現「找不到 *.txt」這種怪訊息。

萬用字元只能在「目前這層」配對檔名。要在整棵目錄樹裡深入尋找,得請出 find——它會遞迴鑽進每一層子目錄: 

$ find . -name "*.py"             # 從目前目錄(.)往下,找所有 .py 檔
$ find . -type f -name "*.log"    # -type f:只要「檔案」
$ find . -type d -name "test*"    # -type d:只要「目錄」
$ find . -size +10M               # 大於 10MB 的檔(找出空間殺手)
$ find . -mtime -7                # 最近 7 天內修改過的(modified time)
$ find . -name "*.tmp" -delete    # 找到符合的就刪掉(危險,先不加 -delete 確認)

find 一開始的語法看起來有點怪(它把搜尋條件當成一連串的「測試」),但它是命令列裡最強大的工具之一。 等一下你會看到,把 find 找到的結果「餵」給別的指令,能組合出極大的威力。

重新導向與管線:Unix 哲學的精華

接下來這一節,是整個命令列世界最美、也最重要的概念。如果你只能從這一章帶走一件事,請帶走這個。

先建立一個基本認知:每個正在執行的程式(process),預設都連著三條資料流(stream):

  • 標準輸入 stdin(編號 0):程式從這裡「讀」資料進來,預設來源是你的鍵盤。
  • 標準輸出 stdout(編號 1):程式把「正常結果」往這裡「寫」,預設去處是你的螢幕。
  • 標準錯誤 stderr(編號 2):程式把「錯誤訊息」往這裡寫,預設也是螢幕——但它和 stdout 是兩條獨立的管道,這個區分等下很有用。

重新導向(redirection):改變資料的去向與來源

既然這些流預設連著螢幕和鍵盤,我們就能用幾個運算子改道,把它們接到檔案上: 

$ echo "第一筆紀錄" > app.log      # >  把 stdout 寫進檔案(會「覆蓋」原內容!)
$ echo "第二筆紀錄" >> app.log     # >> 「附加」到檔案結尾(不會蓋掉舊的)
$ sort < names.txt                 # <  把檔案內容餵給程式的 stdin
$ ls /not_exist 2> err.txt         # 2> 只把 stderr(錯誤)導進檔案
$ make > build.log 2>&1            # 把 stderr「併入」stdout,一起寫進同一個檔
$ noisy_cmd > /dev/null 2>&1       # 全部丟進 /dev/null(系統的黑洞/垃圾桶)

其中 2>&1 是新手最常卡住的咒語,拆開來看就懂了:2> 是「把 stderr 導向」, &1 是「到 stdout 現在去的地方」。合起來就是「讓錯誤跟著正常輸出走」。 而 /dev/null 是一個特殊檔案,你寫進去的東西會直接消失——專門用來丟掉你不想看的輸出。

管線(pipe):把一個程式的輸出,直接接到下一個的輸入

重新導向是把流接到檔案;而管線符號 | 更神奇:它把前一個程式的 stdout, 直接當成後一個程式的 stdin——資料完全不落地,像水一樣從一根水管流進下一根。 

$ ls -l | grep ".txt"             # ls 的輸出,交給 grep 過濾出含 .txt 的行
$ cat access.log | grep "404" | wc -l    # 串三個:讀檔 → 篩 404 → 數行數
$ history | grep git | tail -5    # 從歷史紀錄裡,找出最近 5 次和 git 有關的指令

為什麼這個設計如此優雅?因為它讓每個工具都可以做得很小、很專一:grep 只管過濾、 sort 只管排序、wc 只管計數。沒有任何一個工具試圖「什麼都做」。但透過 |, 你可以把它們自由排列組合成你當下需要的任何流程。三十個小工具,能組出的花樣是天文數字。

📜
Unix 哲學:做一件事,並把它做好

1978 年,管線的發明者之一、貝爾實驗室的 Doug McIlroy 把 Unix 的設計精神濃縮成一句傳世名言:

「Write programs that do one thing and do it well. Write programs to work together. Write programs to handle text streams, because that is a universal interface.」
(寫只做一件事、並把它做好的程式;寫能彼此協作的程式;寫處理文字流的程式,因為文字是通用的介面。)

「用文字當作程式之間的通用介面」是其中最深刻的一句:正因為大家都讀寫純文字,任何工具的輸出,都能成為任何工具的輸入。這就是命令列能無限組合的祕密。

文字處理三寶:grep、sed、awk

既然「文字是通用介面」,那麼會處理文字,就等於掌握了整個命令列世界的通用貨幣。有三個工具是這個領域的傳奇, 合稱文字處理三寶。它們各司其職:grep 找、sed 改、awk 算。

grep:在文字大海裡撈針

grep 做一件事:印出符合某個模式的行。名字來自老編輯器指令 g/re/p (globally search a regular expression and print)。它是你最常用的工具之一: 

$ grep "error" server.log         # 印出所有含 "error" 的行
$ grep -i "error" server.log      # -i:忽略大小寫(Error、ERROR 也算)
$ grep -n "def " app.py           # -n:同時顯示行號
$ grep -c "404" access.log        # -c:不印內容,只回報「幾行符合」
$ grep -v "DEBUG" server.log      # -v:反向,印出「不」含 DEBUG 的行
$ grep -r "TODO" .                # -r:遞迴搜尋整個目錄樹
$ grep -rl "API_KEY" .            # -l:只列出「哪些檔案」有,不印行內容

grep 真正的魔力,在於它的搜尋模式可以是正規表示式(regular expression, regex)—— 一種描述「文字長相」的迷你語言。用 grep -E 啟用擴充語法,來認識幾個最常用的符號:

符號意思例子
.任意一個字元a.c 配 abc、a7c
*前一個東西重複 0 次以上ab* 配 a、ab、abbb
+前一個東西重複 1 次以上ab+ 配 ab、abbb(但不配 a)
?前一個東西出現 0 或 1 次colou?r 配 color、colour
^ / $行的開頭 / 結尾^Error 配「以 Error 開頭」的行
[...]字元集合[0-9] 配任一數字
{n}前一個東西剛好重複 n 次[0-9]{4} 配剛好 4 位數字
|或cat|dog 配 cat 或 dog
 
$ grep -E "^[0-9]+"       data.txt        # 以一串數字開頭的行
$ grep -E "[0-9]{3}-[0-9]{4}" contacts.txt  # 像電話的 123-4567
$ grep -E "warn|error|fail" server.log     # 任一關鍵字命中就印

正規表示式是一門可以講一整章的學問,而且不只在 Shell 裡有用——你會在幾乎每種程式語言、每個編輯器裡再遇到它。 現在先記住幾個最常用的符號,建立「原來文字可以這樣描述」的直覺就夠了。

sed:串流編輯,專長是「取代」

sed(stream editor)像一條生產線上的工人:文字一行一行流過它,它照你的規則動手腳,最常見的就是「找到 A,換成 B」: 

$ sed 's/foo/bar/'  file.txt      # 把每行「第一個」foo 換成 bar
$ sed 's/foo/bar/g' file.txt      # g (global):把每行「所有」foo 都換掉
$ sed 's/foo/bar/g' file.txt > out.txt   # 結果導向新檔(原檔不動)
$ sed -i 's/foo/bar/g' file.txt   # -i (in-place):直接改寫原檔(小心!先備份)
$ sed -n '10,20p' file.txt        # 只印出第 10 到 20 行(-n 關閉預設輸出,p 印出)

那串 s/舊/新/ 是 sed 的招牌語法:s 代表 substitute(取代),斜線是分隔符。

awk:把每一行當成「欄位」來處理

awk(以三位作者 Aho、Weinberger、Kernighan 的姓氏命名)是三寶裡最像「程式語言」的一個。 它的世界觀很特別:它把每一行自動切成欄位(field),$1 是第一欄、$2 是第二欄…… $0 是一整行。這讓它處理「表格狀」的文字時無比順手: 

$ awk '{print $1}'      data.txt       # 印出每行的第 1 欄
$ awk '{print $1, $3}'  data.txt       # 印第 1 與第 3 欄
$ awk -F, '{print $2}'  data.csv       # -F,:改用逗號當欄位分隔(處理 CSV)
$ awk '$3 > 100 {print $1}' data.txt   # 只在「第 3 欄大於 100」時,印第 1 欄
$ awk '{sum += $1} END {print sum}' nums.txt   # 把第 1 欄全部加總(END 在最後執行)

更多好幫手:wc、sort、uniq、cut

還有幾個小工具,組合起來威力驚人:

$ wc -l file.txt        # 數「行數」(word count 的 -l = lines)
$ wc -w file.txt        # 數「字數」
$ sort names.txt        # 把每行排序(預設依字典序)
$ sort -n nums.txt      # -n:依「數值」大小排(不然 10 會排在 2 前面)
$ sort -rn nums.txt     # -r:反向(由大到小)
$ uniq sorted.txt       # 去掉「相鄰」的重複行(所以通常先 sort)
$ uniq -c sorted.txt    # -c:在每行前面標上它出現的次數
$ cut -d, -f1 data.csv  # -d, 用逗號切,-f1 取第 1 欄
$ cut -c1-5 file.txt    # 取每行的第 1~5 個字元

現在,我們把這些零件組合起來,完成一個經典又實用的任務(這正是第 11 章提過的那個例子): 從伺服器存取記錄裡,找出造成最多 404 錯誤的前 10 個 IP。一行管線就搞定: 

$ grep " 404 " access.log | awk '{print $1}' | sort | uniq -c | sort -rn | head
   1532 203.0.113.7
    991 198.51.100.23
    640 192.0.2.55
    ...

把這條管線逐站讀一遍,你就真正懂了 Unix 哲學的威力:

  1. grep " 404 ":從幾百萬行裡,只留下含 404 的那些。
  2. awk '{print $1}':每行只取第一欄(也就是 IP)。
  3. sort:把相同的 IP 排在一起(這樣 uniq 才數得對)。
  4. uniq -c:把連續相同的 IP 收成一筆,並標上出現次數。
  5. sort -rn:依「次數」由大到小排序。
  6. head:只看前 10 名。

六個小工具,沒有一個「知道」整件事的全貌,但串起來就解決了一個真實的維運問題。 而且這條管線可以原封不動貼進你的筆記、塞進腳本、明天再跑一次——這就是「可組合 + 可重現」的力量。

✏️ 練習 16.1 實作

用一行管線,統計一篇英文文章 article.txt 裡最常出現的前 10 個單字(不分大小寫)。 提示:你需要先把文章「打散成一行一個單字」,再用我們學過的 sort / uniq / head 組合起來。

看解答

關鍵的第一步,是把「一行很多字」變成「一行一個字」。tr(translate)能做這件事:

$ tr -cs 'A-Za-z' '\n' < article.txt | tr 'A-Z' 'a-z' | sort | uniq -c | sort -rn | head -10
    312 the
    198 of
    187 and
    ...

逐站拆解:

  • tr -cs 'A-Za-z' '\n':把所有「不是字母」的字元(-c = complement,取補集)都換成換行, 並把連續的多個壓成一個(-s = squeeze)。於是每個單字各佔一行。< article.txt 把檔案餵進去。
  • tr 'A-Z' 'a-z':全部轉小寫,這樣 The 和 the 才會被當成同一個字。
  • sort | uniq -c:排序後計算每個單字出現幾次。
  • sort -rn | head -10:依次數由大到小排,取前 10 名。

這條管線是文字分析的萬用骨架,把 article.txt 換成任何文字資料,它都能告訴你「什麼最常出現」。

✏️ 練習 16.2 實作

你接手了一個陌生的程式專案,想知道哪些原始碼檔案裡寫了 TODO(待辦標記),好把它們清掉。 請寫出指令,從目前目錄往下遞迴尋找,只列出「含有 TODO 的檔案名稱」。再進階:只在 .py 檔裡找。

看解答

最直接的寫法,用 grep 的遞迴(-r)加「只列檔名」(-l):

$ grep -rl "TODO" .

只想找 .py 檔的話,可以用 --include 限定:

$ grep -rl --include="*.py" "TODO" .

或者用 find 找出檔案,再把它們「餵」給 grep——這示範了兩個工具怎麼接力:

$ find . -type f -name "*.py" -exec grep -l "TODO" {} +

這裡 find 的 -exec ... {} + 會把找到的每個檔名,填進 {} 的位置交給 grep 執行。 這正是命令列「小工具接力」精神的體現。

環境變數、PATH 與引號

Shell 不只是個下指令的地方,它本身也是個有「記憶」的環境——這些記憶,存在變數裡。 其中一部分叫環境變數(environment variable),是整個系統共享的設定: 

$ echo $HOME            # 你的家目錄
/home/akira
$ echo $USER            # 你的使用者名稱
akira
$ echo $PATH            # 一串「指令的搜尋路徑」,用冒號隔開
/usr/local/bin:/usr/bin:/bin:/home/akira/.local/bin

$ NAME="Akira"          # 設定一個變數(只有目前這個 shell 看得到)
$ export NAME="Akira"   # export:讓它變成環境變數,連子程序也看得到
$ echo "Hello, $NAME"   # 用 $ 取出變數的值
Hello, Akira

PATH:電腦怎麼知道指令在哪?

當你打 ls,電腦怎麼知道 ls 這個程式存在哪個資料夾?答案就是 PATH: 它是一串用冒號隔開的目錄清單,Shell 會依序到每個目錄裡找有沒有叫 ls 的可執行檔,找到第一個就執行。 這也是為什麼你自己寫的程式如果不在 PATH 裡,就得用 ./我的程式 明確指出位置。要把自己的工具加進去: 

$ export PATH="$HOME/bin:$PATH"   # 把 ~/bin 加到最前面,從此那裡的程式可直接呼叫
$ which python                    # which:告訴你某指令實際是哪個檔案被執行
/usr/bin/python

$?:上一個指令成功了嗎?

每個指令執行完都會留下一個結束狀態碼(exit status),存在特殊變數 $? 裡。 慣例是:0 代表成功,任何非 0 代表某種失敗。這在寫腳本判斷「上一步成不成功」時極為關鍵: 

$ ls /home
akira
$ echo $?
0              # 0 = 成功

$ ls /not_exist
ls: /not_exist: No such file or directory
$ echo $?
1              # 非 0 = 失敗

引號:單引號與雙引號差在哪?

這是新手最常踩的坑之一,但規則其實很簡單:雙引號會展開 $ 變數,單引號則一字不漏地照搬。

$ name="Akira"
$ echo "Hi, $name"      # 雙引號:$name 被展開
Hi, Akira
$ echo 'Hi, $name'      # 單引號:原封不動,連 $name 都照印
Hi, $name

實務上的鐵則:變數幾乎永遠要用雙引號包起來,寫成 "$file" 而不是 $file。 為什麼?因為若檔名裡有空白(例如 My Photo.jpg),沒加引號的話 Shell 會把它拆成兩個東西,釀成災難。

別名(alias):把常用的長指令縮短

$ alias ll='ls -lah'    # 以後打 ll 就等於 ls -lah
$ alias gs='git status'
$ alias ..='cd ..'
💡
讓設定「永久生效」

在終端機裡直接打的 export 或 alias,關掉視窗就沒了。想讓它們每次開啟都自動生效, 要把它們寫進你的 Shell 設定檔:Bash 是 ~/.bashrc,Zsh 是 ~/.zshrc。 這個檔案會在每次開新終端機時被自動執行——它就是你打造個人化工作環境的地方。

權限:rwx 與 chmod

Unix 從一開始就是多人共用的系統,所以每個檔案都有一套「誰能對它做什麼」的權限。 還記得 ls -l 開頭那串神祕符號嗎?現在我們把它拆開:

三組權限分別管三種人:擁有者(owner,通常是建立者你)、同群組的人(group)、以及其他所有人(others)。 每組都有讀(read)、寫(write)、執行(execute)三道開關。對檔案來說,x 代表「可以當程式跑」; 對目錄來說,x 代表「可以進去(cd)」。

改變權限用 chmod(change mode),有兩種寫法:

$ chmod +x script.sh       # 符號模式:幫所有人加上「可執行」
$ chmod u+x,go-w file      # u=owner, g=group, o=others;+加 -減
$ chmod 755 script.sh      # 數字模式:rwxr-xr-x(7=rwx, 5=r-x, 5=r-x)
$ chmod 644 notes.txt      # rw-r--r--(擁有者可讀寫,其他人只能讀)
$ chmod 700 secret.key     # rwx------(只有自己能碰,別人完全沒權限)
$ chmod -R 755 mydir/      # -R:遞迴套用到整個資料夾

$ chown akira:staff file   # 改檔案的「擁有者:群組」(通常需要 sudo)
🔒
這是你資安直覺的起點

「最小權限原則(principle of least privilege)」——只給剛好夠用的權限,不多給——是整個資訊安全的基石。 一個放密鑰的檔案設成 600(只有自己能讀寫),而不是任何人都能讀的 644,就是這個原則的日常實踐。 我們會在第七部「資訊安全」把這套思維展開講透。

✏️ 練習 16.3 實作

你有兩個檔案需要設定權限:

  • deploy.sh 是部署腳本,只有你自己能讀、寫、執行,其他任何人都不能碰。
  • notes.txt 是公開筆記,所有人都能讀,但只有你能修改。

請寫出對應的 chmod 指令(用數字模式),並說明每個數字的含義。

看解答 
$ chmod 700 deploy.sh    # rwx------
$ chmod 644 notes.txt    # rw-r--r--

解讀:

  • 700 = rwx------:擁有者 7(讀 4 + 寫 2 + 執行 1),群組 0,其他人 0。除了你,沒人能讀、寫、或執行它。
  • 644 = rw-r--r--:擁有者 6(讀 4 + 寫 2),群組 4(只讀),其他人 4(只讀)。大家都能讀,但只有你能改。

順帶一提,644 是一般檔案最常見的權限,755 則是程式與目錄最常見的權限——記住這兩個,日常就夠用了。

Shell 指令稿:讓電腦替你重複勞動

到目前為止,我們都是在終端機裡「一行一行」下指令。但只要把這些指令寫進一個檔案, 它就成了一支可以反覆執行的程式——這就是 Shell 指令稿(shell script)。它是「自動化」這件事最直接的入口。

每支腳本的第一行,通常是這個看起來很奇怪的東西:

#!/bin/bash

這叫 shebang(井號 # + 驚嘆號 !)。它告訴系統:「請用 /bin/bash 這個程式來執行底下的內容」。 有了它,你的腳本就能像一般程式一樣被直接呼叫(別忘了先 chmod +x 給它執行權限)。

變數、引數與命令替換

#!/bin/bash
# greet.sh — 跟使用者打招呼

name="$1"                        # $1 是第一個「命令列引數」
greeting="Hello"                 # 自訂變數(注意 = 兩邊不能有空格!)

echo "$greeting, $name!"
echo "今天是 $(date +%Y-%m-%d)"  # $(...) 命令替換:把指令的輸出塞進這裡

執行它:

$ chmod +x greet.sh
$ ./greet.sh Akira
Hello, Akira!
今天是 2026-06-29

幾個關鍵字彙:$1、$2…… 是傳給腳本的第一、第二個引數;$@ 是「全部引數」; $# 是「引數的數量」;$0 是腳本自己的名字。而 $(...) 命令替換(command substitution) 是最常用的招式之一:它先執行括號裡的指令,再把輸出「貼」回原位。

判斷:if

#!/bin/bash
name="$1"

if [ -z "$name" ]; then          # -z:字串是否為「空」
    echo "用法:$0 <名字>"        # 沒給引數就印出用法
    exit 1                       # 以非 0 狀態碼結束,代表「出錯了」
elif [ "$name" = "root" ]; then  # 字串相等用單一個 =
    echo "嗨,管理員!"
else
    echo "Hello, $name!"
fi                               # if 用反過來寫的 fi 結尾

Bash 的 [ ... ] 裡用的是特殊的「測試」寫法,常見的有:-z(字串為空)、-n(字串非空)、 -f(檔案存在)、-d(目錄存在)、-e(檔案或目錄存在);數字比較用 -eq(等於)、 -lt(小於)、-gt(大於)。注意數字比較不用 < >—— 那兩個符號在 Shell 裡是重新導向,會釀成意外。

迴圈:for 與 while

#!/bin/bash
# 對每個 .txt 檔做點事
for file in *.txt; do
    echo "處理中:$file"
    wc -l "$file"               # 別忘了用 "$file" 包起來,防範檔名有空白
done

# while:條件成立就一直跑
count=1
while [ "$count" -le 5 ]; do
    echo "第 $count 次"
    count=$((count + 1))        # $((...)) 做整數運算
done

函式

#!/bin/bash
# 定義一個會印出帶時間戳記的 log 函式
log() {
    echo "[$(date +%H:%M:%S)] $*"   # $* = 傳給函式的所有引數
}

log "開始執行"
log "處理了 42 筆資料"
# 輸出:
# [14:30:01] 開始執行
# [14:30:01] 處理了 42 筆資料

把它們組起來:一支真正有用的備份腳本

現在把學到的零件拼成一支實用工具:給它一個資料夾,它就打包成一個帶有日期時間戳記的壓縮檔。 這種腳本你以後會天天用到: 

#!/bin/bash
# backup.sh — 把指定資料夾打包成帶時間戳的壓縮檔
#   用法:./backup.sh <資料夾>

set -e                                   # 任何指令一出錯就立刻整支中止(安全網)

src="$1"

# 1. 先驗證輸入(防禦性思考)
if [ -z "$src" ]; then
    echo "用法:$0 <要備份的資料夾>" >&2   # 用法訊息寫到 stderr
    exit 1
fi
if [ ! -d "$src" ]; then                 # ! -d:這個目錄「不」存在
    echo "錯誤:找不到資料夾 '$src'" >&2
    exit 1
fi

# 2. 用命令替換產生時間戳,組出輸出檔名
stamp=$(date +%Y%m%d_%H%M%S)             # 例如 20260629_143010
dest="backup_${stamp}.tar.gz"            # ${stamp} 大括號避免和後面文字黏在一起

# 3. 動手打包(tar:-c 建立、-z gzip 壓縮、-f 指定檔名)
tar -czf "$dest" "$src"

echo "✅ 已備份 '$src' → '$dest'"
echo "   大小:$(du -h "$dest" | cut -f1)"
$ ./backup.sh photos
✅ 已備份 'photos' → 'backup_20260629_143010.tar.gz'
   大小:24M

短短二十幾行,它就做到了:驗證輸入、產生不重複的檔名、壓縮、回報結果。而且它可重現、可排程—— 你可以把它交給系統的 cron 排程器,讓它每天半夜自動幫你備份,你完全不用動手。這就是命令列把「勞動」變成「資產」的方式。

💡
每支腳本都該有的三行護身符

專業的 Bash 腳本常在開頭加上 set -euo pipefail:-e 出錯即停、 -u 用到未定義的變數就報錯(能擋下前面講的 rm -rf $空變數/ 慘案)、 -o pipefail 讓管線中任一站失敗都算失敗。三個字母,幫你擋掉大量隱藏的災難。

✏️ 練習 16.4 實作

寫一支腳本 rename.sh,把目前目錄裡所有 .jpeg 檔,改成 .jpg (例如 vacation.jpeg → vacation.jpg),檔名主體保持不變。這正是本章開頭那個「3,000 張照片」問題的核心。

看解答 
#!/bin/bash
# rename.sh — 把目前目錄所有 .jpeg 改成 .jpg
for f in *.jpeg; do
    [ -e "$f" ] || continue          # 若一個 .jpeg 都沒有,*.jpeg 不會展開,跳過
    mv -- "$f" "${f%.jpeg}.jpg"      # ${f%.jpeg} 砍掉結尾的 ".jpeg",再接上 ".jpg"
    echo "改名:$f → ${f%.jpeg}.jpg"
done

兩個關鍵技巧:

  • ${f%.jpeg} 是「參數展開」的後綴刪除:把變數 f 結尾的 .jpeg 砍掉。 所以 vacation.jpeg 變成 vacation,再手動接上 .jpg。
  • mv -- 的 -- 告訴 mv「後面都是檔名,別當成旗標解讀」, 這能保護你免於檔名剛好以 - 開頭的怪狀況。
  • [ -e "$f" ] || continue 處理「目錄裡根本沒有 .jpeg」的邊界情況。

想想:如果改名後可能覆蓋到已存在的同名 .jpg,你會怎麼加一道保護?(提示:用 [ -e "目標" ] 先檢查。)

站在巨人肩膀上:幾個讓你功力倍增的工具

最後,介紹幾個進階但極度實用的工具,讓你知道命令列的世界還有多寬廣:

xargs:把「一串輸出」變成「一串引數」

有些指令(像 rm、mkdir)從引數讀檔名,而不是從 stdin。但 find、grep 吐出來的是 stdin 流。xargs 就是這兩個世界之間的轉接頭——把「流」轉成「引數」: 

$ find . -name "*.tmp" | xargs rm              # 把找到的 .tmp 全部當引數丟給 rm
$ find . -name "*.log" -print0 | xargs -0 rm   # -print0/-0:用 NUL 分隔,安全處理含空白的檔名
$ cat urls.txt | xargs -n1 curl -O             # 對清單裡每個網址各下載一次(-n1 = 一次一個)

tmux:一個視窗,多個分身,還能離線續用

tmux(terminal multiplexer)讓你在「一個」終端機視窗裡分割出多個面板、開多個分頁; 更強的是,你可以把工作階段「丟到背景」,關掉視窗(甚至斷線)後再回來,所有東西原封不動—— 這對「在遠端伺服器上跑長時間任務」是救命功能。

ssh:在另一台電腦的 Shell 裡工作

ssh(secure shell)讓你安全地登入遠端主機,然後在它的 Shell 裡下指令, 彷彿你就坐在那台機器前面——這是管理伺服器、雲端機器最基本的方式: 

$ ssh akira@server.example.com         # 登入遠端,進入它的 shell
$ scp report.pdf akira@server:/backup/ # scp:透過 ssh 安全地把檔案複製到遠端

這些工具每個都能再講一整章,這裡只先讓你「知道它們存在」。等你真正需要時,你會很慶幸自己聽過它們的名字。

如果你在 Windows:PowerShell 對照

本章的 Bash 世界主要活在 Linux 與 macOS。但如果你是 Windows 使用者,好消息是:概念完全相通,只是腔調不同。 Windows 內建的 PowerShell 同樣有提示字元、管線、變數、腳本,只是指令名字換了套樣子(它愛用「動詞-名詞」命名,如 Get-ChildItem)。 對照表如下:

一個關鍵的哲學差異:文字流 vs 物件流

記得 Unix 哲學說「用文字串接一切」嗎?PowerShell 走了一條不同的路:它的管線傳遞的是結構化的物件,而非純文字。 所以 Get-ChildItem | Where-Object { $_.Length -gt 1MB } 可以直接讀物件的 Length 屬性,不必像 Bash 那樣用 awk 去「切字串」。 兩種設計各有優劣,但都體現了同一個核心精神:把小工具用管線組合成大威力。

另外,Windows 使用者也可以裝 WSL(Windows Subsystem for Linux) 或 Git Bash,直接在 Windows 上跑道地的 Bash——兩個世界你都能擁有。

✏️ 練習 16.5 挑戰

寫一支腳本 healthcheck.sh,接收一個網址當引數,檢查該網站是否存活: 如果回應 HTTP 200(正常),印出成功訊息並以狀態碼 0 結束;否則印出警告(寫到 stderr)並以非 0 結束。 提示:curl 可以只取回應的 HTTP 狀態碼。

看解答 
#!/bin/bash
# healthcheck.sh — 檢查網站是否回應 HTTP 200
#   用法:./healthcheck.sh https://example.com
set -euo pipefail

url="$1"
if [ -z "$url" ]; then
    echo "用法:$0 <網址>" >&2
    exit 2
fi

# curl 旗標:-s 安靜、-o /dev/null 丟掉內文、-w 只輸出我們要的狀態碼
code=$(curl -s -o /dev/null -w "%{http_code}" "$url")

if [ "$code" -eq 200 ]; then
    echo "✅ $url 正常(HTTP $code)"
    exit 0
else
    echo "❌ $url 異常(HTTP $code)" >&2
    exit 1
fi

執行起來像這樣:

$ ./healthcheck.sh https://example.com
✅ https://example.com 正常(HTTP 200)
$ echo $?
0

真正威力在於組合與自動化:把它和 cron 排程結合,讓它每 5 分鐘自動檢查一次, 一旦回傳非 0,就觸發寄信通知你——一支幾行的腳本,就成了一套陽春的網站監控系統。 想練更多?試試讀一個「網址清單檔」,用 while read url; do ... done < sites.txt 一次檢查全部。

把這一章串起來

我們從一個「改 3,000 個檔名」的苦差事出發,一路走到了能寫腳本、監控網站的境界。回顧這趟旅程:

  • 為什麼用命令列:因為它有力量、可自動化、可組合、可重現——這四件事,正是把「勞動」變成「資產」的關鍵。
  • 導航與檔案操作:pwd / ls / cd 讓你穿梭於檔案系統樹,mkdir / cp / mv / rm 搬動檔案(但 rm -rf 要當心)。
  • 重新導向與管線:> >> < 改變資料的去向與來源,| 把小工具串成流水線。這是 Unix 哲學的精華:「做一件事,並做好;用文字串接一切。」
  • 文字三寶:grep 找、sed 改、awk 算,搭配 sort / uniq / wc / cut,一行管線就能完成真實的資料分析。
  • 環境與權限:環境變數、PATH、$?、引號的差異,以及 rwx 權限與 chmod——後者是你資安思維的第一課。
  • Shell 指令稿:shebang、變數、if / for / while、$(...)、引數、函式——把一連串指令變成一支可反覆執行、可排程的程式。

命令列不是要你「背熟所有指令」,而是要你建立一種思維:把任何重複的工作,想像成「能不能用幾個小工具串起來自動化」。 一旦你習慣這樣想,終端機就會從一個嚇人的黑盒子,變成你最忠誠、最強大的助手。下一章,我們會把這個思維推向更高處—— 當你的程式碼開始演化、需要和別人協作時,你會需要一台「時光機」:版本控制與 Git。
