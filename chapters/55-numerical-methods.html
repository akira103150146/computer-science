數值方法

真實世界的數學問題,大多沒有漂亮的「封閉解」,甚至根本解不出來。數值方法(numerical methods)是電腦科學家的祕密武器:用「逼近 + 迭代」算出一個夠好的答案。它是科學計算、工程模擬與機器學習底層那台默默運轉的引擎。

當數學「算不出來」的時候

你在第 12 章學過,要算 $\int_0^1 x^2\,dx$,只要找個反導數 $\frac{x^3}{3}$,代入端點相減就好, 答案精確是 $\frac{1}{3}$,分秒不差。那種「找到一條公式,代數字進去就得到確切答案」的解, 叫封閉解(closed-form solution),也叫解析解。它很美。問題是—— 在真實世界,這種美麗的封閉解是稀有的例外,不是常態。

舉幾個你以為「應該算得出來」其實算不出來的例子:

  • 積分 $\displaystyle\int_0^1 e^{-x^2}\,dx$(高斯鐘形曲線下的面積,統計學天天在用)沒有用基本函數寫得出來的反導數。它不是「很難找」,是數學上被證明不存在。
  • 五次以上的多項式方程式 $x^5 - x - 1 = 0$,根據阿貝爾–魯菲尼定理,一般情況下沒有用加減乘除與開根號寫成的求根公式(高中背的「公式解」到四次方就到頂了)。
  • 單擺的運動方程式 $\theta'' = -\frac{g}{L}\sin\theta$ 是個微分方程,只要不做「小角度近似」,它的解就寫不成基本函數。

那這些問題就無解了嗎?當然不是——飛機照飛、天氣照報、神經網路照訓練。 祕訣在於:我們不堅持要那個「精確的符號答案」,改而追求一個「誤差小到無所謂」的數值答案。 要算 $\int_0^1 e^{-x^2}\,dx$?把曲線切成幾千個小薄片加起來(第 12 章的黎曼和,本章會升級它)。 要解 $x^5-x-1=0$?先猜一個數,再用某條規則一次次修正這個猜測,讓它越來越靠近真正的根。

🔑
數值方法的兩個動詞:逼近與迭代

逼近(approximation):用算得動的簡單東西(多項式、矩形、線性步進)去代替算不動的複雜東西, 換來一點點可控的誤差。
迭代(iteration):從一個粗略的猜測出發,反覆套用一條改進規則,讓答案一步步收斂到目標。
這兩件事貫穿整章——求根、解方程組、算積分、解微分方程,骨子裡全是「逼近 + 迭代」。 這也呼應了第 1 章說的:電腦只懂命令式知識(一步步怎麼做),而數值方法,正是把「求解」這件事翻譯成命令式步驟的藝術。

但天下沒有白吃的午餐。一旦我們從「精確」退到「近似」,就得睜大眼睛盯著一個東西:誤差。 誤差從哪來?會不會越滾越大?我的答案到底準到第幾位?要回答這些,我們得先掀開一個你每天用、 卻很可能誤會它的東西——電腦裡的小數。

浮點數的真相:電腦眼中的數字

你在第 2 章已經被劇透過一個驚悚事實:在 Python 裡 0.1 + 0.2 不等於 0.3。 當時我們點到為止。現在做數值方法,這件事從「冷知識」升級成「攸關生死的基本功」—— 因為你接下來算的每一個數,都活在這套規則底下。

為什麼 0.1 存不進去

電腦用二進位存數字。十進位的 $0.1$,換成二進位是 $0.0001100110011001100\ldots_2$ —— 一個無限循環小數,就像十進位寫不完 $\frac{1}{3}=0.333\ldots$ 一樣。 但記憶體是有限的(雙精度浮點數總共只有 64 個位元),電腦只好在某一位截斷並四捨五入, 存進去的其實是一個離 $0.1$ 極近、但不完全相等的數。 

from decimal import Decimal

# 看看電腦「真正」存的 0.1 是什麼
print(Decimal(0.1))
# 0.1000000000000000055511151231257827021181583404541015625
#   ↑ 不是 0.1,是離它最近的那個「可表示的數」

print(0.1 + 0.2)          # 0.30000000000000004
print(0.1 + 0.2 == 0.3)   # False

這套「用有限位元逼近實數」的規格,就是 IEEE 754 標準。一個雙精度(double, Python 的 float) 浮點數,把 64 個位元切成三段:1 個符號位、11 個指數位、52 個尾數位(mantissa)。 它本質上是「科學記號的二進位版」:$\pm\, 1.\underbrace{m_1 m_2 \ldots m_{52}}_{\text{尾數}} \times 2^{e}$。 指數讓它能表示極大與極小的數,尾數則決定了精度——而精度,是有上限的。

機器 epsilon:精度的天花板

既然精度有限,自然會有一個「最小有感的相對誤差」。這個數叫機器 epsilon(machine epsilon,$\varepsilon_{\text{mach}}$): 它是「$1.0$ 與比它大的下一個可表示浮點數之間的距離」。對雙精度而言,$\varepsilon_{\text{mach}} \approx 2.22\times 10^{-16}$。 直覺上,它告訴你浮點數大約只有 15 到 16 位有效數字,再細的差別電腦根本「看不見」。 

import numpy as np

eps = np.finfo(float).eps
print(eps)                    # 2.220446049250313e-16

# 比 epsilon 還小的擾動,加上去等於沒加
print(1.0 + 1e-16 == 1.0)     # True  ← 1e-16 < eps,被「吃掉」了
print(1.0 + 1e-15 == 1.0)     # False ← 這個還勉強看得見

記住 $\varepsilon_{\text{mach}}$ 這個數量級($\sim 10^{-16}$),它是本章所有「能準到多少位」討論的天花板。 沒有任何演算法能在雙精度下,給你比這更精確的答案。

兩種誤差:捨入 vs 截斷

數值計算裡的誤差,主要來自兩個完全不同的源頭,務必分清楚:

誤差類型來自哪裡例子
捨入誤差(round-off error)浮點數位元有限,每次運算都被迫四捨五入$0.1$ 存不準;一千萬次加總後誤差累積
截斷誤差(truncation error)我們用有限步逼近一個本該無限的數學過程泰勒級數只取前幾項;積分只切有限個矩形

關鍵洞見是:這兩種誤差常常互相拉扯。以數值微分為例(等一下會詳談),你用越小的步長 $h$, 截斷誤差越小——但 $h$ 太小時,$f(x+h)$ 與 $f(x)$ 兩個數幾乎一樣,相減後捨入誤差反而爆炸。 於是存在一個「最佳的 $h$」,在兩種誤差之間取得平衡。這個拉扯,我們後面會用一張圖看清楚。

災難性抵消:數值計算的頭號殺手

所有捨入誤差的災難裡,最陰險的一種叫災難性抵消(catastrophic cancellation): 當你把兩個非常接近的數相減,它們前面相同的有效位數會彼此抵消、消失不見, 只剩下尾巴那幾位「本來就不太準」的數字撐場面——相對誤差因此被瞬間放大幾個數量級。

看一個具體例子:計算 $f(x) = \sqrt{x+1} - \sqrt{x}$。當 $x$ 很大時,$\sqrt{x+1}$ 與 $\sqrt{x}$ 幾乎一模一樣, 直接相減就是在自找麻煩: 

import math

def naive(x):
    return math.sqrt(x + 1) - math.sqrt(x)   # 兩個幾乎相等的數相減 → 抵消

def stable(x):
    # 代數恆等式:乘上「共軛」再除回去,把減法消掉
    # (√(x+1) - √x)(√(x+1) + √x) = 1
    return 1.0 / (math.sqrt(x + 1) + math.sqrt(x))

x = 1e16
print("天真寫法:", naive(x))    # 0.0           ← 完全錯!兩數在浮點下已相等
print("穩定寫法:", stable(x))   # 5e-09         ← 正確答案 ≈ 1/(2√x)

天真寫法直接吐出 0.0——因為在浮點數眼中 $\sqrt{10^{16}+1}$ 和 $\sqrt{10^{16}}$ 已經是同一個數了, 相減當然是零。而穩定寫法用一個代數小技巧把減法轉化掉,完全避開了抵消,給出正確答案。 同一個數學式,兩種寫法,一個全錯一個全對——這就是數值方法和純數學最不一樣的地方: 運算的「順序」與「形式」會決定答案的對錯。

⚠️
看到「相近的數相減」就要警鈴大作

災難性抵消潛伏在無數地方:用 $b^2-4ac$ 的求根公式解 $x^2+10^8 x + 1=0$(練習會看到)、 算變異數時用 $E[x^2]-(E[x])^2$、算 $1-\cos x$ 當 $x$ 很小時…… 症狀都一樣:結果的有效位數莫名其妙地少、甚至變成 0。
解法通常是「換個數學上等價、但不需要相減的式子」(乘共軛、用三角恆等式、改用 Vieta 公式、Kahan 求和等)。 這也是為什麼numpy/scipy 的函式值得信賴——它們的作者花了幾十年把這些陷阱一個個填平。

誤差會累積

單次運算的捨入誤差小到 $10^{-16}$,看似無害。但數值方法動輒做幾百萬、幾十億次運算, 誤差會一點一滴累積。最經典的示範:把 $0.1$ 加一千萬次,理論上該是一百萬,實際呢? 

total = 0.0
for _ in range(10_000_000):
    total += 0.1
print(total)              # 999999.9999839197   ← 不是 1000000.0!
print(repr(1000000.0 - total))   # 1.6080332940444350e-05 的誤差

每次加 $0.1$ 都帶進一點點捨入誤差,累積一千萬次後,就偏掉了百萬分之十六。 在科學計算與長時間模擬裡,如何抑制誤差累積(例如 Kahan 補償求和、選用穩定的演算法) 本身就是一門學問。先記住這個直覺:誤差不只會發生,還會長大。

條件數與穩定性:有些問題天生難算

當一個數值答案算壞了,你該怪誰?這裡有個極重要、卻常被混為一談的區別: 到底是問題本身太敏感(怎麼算都難準),還是你的演算法太爛(把好問題算壞了)? 這兩件事分別叫條件(conditioning)與穩定性(stability)。

  • 條件數(condition number)是問題的性質,與你用什麼方法無關。它衡量: 「輸入動一點點,輸出會跟著動多少?」條件數大(ill-conditioned,病態)的問題, 輸入一丁點的誤差(別忘了輸入本身就帶著捨入誤差)就會被放大成輸出的巨大誤差——它天生就難算準。
  • 穩定性是演算法的性質。一個穩定的演算法,不會在計算過程中額外放大誤差; 不穩定的演算法(例如剛剛那個會抵消的天真寫法)會把一個好端端的問題算爛。

對函數求值 $y=f(x)$ 而言,相對條件數大約是 $\kappa \approx \left|\dfrac{x\,f'(x)}{f(x)}\right|$ ——看到沒,它就是用第 12 章的導數來衡量「敏感度」。當 $f(x)$ 接近 0(例如求根時),$\kappa$ 會爆大, 這正是為什麼「求根」附近數值上特別棘手。

🔑
一句話分清楚這兩者

條件數差,是問題的錯;不穩定,是演算法的錯。 一個經驗法則:在雙精度下,如果問題的條件數是 $\kappa \approx 10^{k}$, 你大約會損失 $k$ 位有效數字——本來有 16 位,病態到 $\kappa=10^{10}$ 時就只剩 6 位可信。 這也是為什麼專業的數值函式庫常會回報「估計的條件數」:它在誠實地告訴你「這個答案可以信幾位」。 你無法用更聰明的演算法去拯救一個病態的問題(那是問題的天性),但你絕對可以用爛演算法毀掉一個好問題。

求根:解 f(x) = 0

我們從第一個經典任務開始:給一個函數 $f$,找出讓 $f(x)=0$ 的那個 $x$(叫它的根 root)。 這看似冷僻,其實無所不在:「兩條曲線在哪相交」是求 $f-g=0$;「利潤最大在哪」是求導數 $f'(x)=0$; 「這個方程式的解是多少」更是直接求根。而正如開頭說的,大多數方程式沒有求根公式,只能用迭代逼近。

二分法:慢,但穩如老狗

第一個方法,你其實在第 5 章的二分搜尋已經見過它的靈魂。觀念極簡: 如果一個連續函數在 $a$ 點是負的、在 $b$ 點是正的,那它中間一定穿過零(這叫介值定理)。 於是我們看中點 $m=\frac{a+b}{2}$ 的正負,就能把根所在的區間砍掉一半;反覆砍,區間越縮越小, 最後夾出那個根。 

def bisection(f, a, b, tol=1e-12, max_iter=200):
    """前提:f(a) 與 f(b) 異號,根必在 [a, b] 之間"""
    assert f(a) * f(b) < 0, "端點必須異號,才保證區間內有根"
    for i in range(max_iter):
        m = (a + b) / 2
        if abs(f(m)) < tol or (b - a) / 2 < tol:
            return m, i + 1
        if f(a) * f(m) < 0:   # 根在左半邊
            b = m
        else:                 # 根在右半邊
            a = m
    return m, max_iter

# 求 √2:就是解 f(x) = x² − 2 = 0 在 [1, 2] 的根
root, iters = bisection(lambda x: x*x - 2, 1, 2)
print(f"√2 ≈ {root:.12f},用了 {iters} 次迭代")   # ≈ 1.414213562373,約 40 次

二分法的優點是保證收斂:只要起始區間端點異號,它一定會夾到根,絕不發散。 缺點是慢:每次只把誤差砍一半。每次誤差變成上一次的 $\frac{1}{2}$ 倍, 這種「誤差以固定比例縮小」叫線性收斂(linear convergence)。要多準一位十進位,得多做約 3.3 次迭代。

牛頓法:快如閃電,但需要好起點

想更快?用上第 12 章的導數。牛頓法(Newton's method)的點子很漂亮: 在你目前的猜測 $x_n$ 處,畫出函數的切線(切線是函數最好的局部直線近似), 然後看這條切線打在 $x$ 軸的哪裡,把那個交點當成更好的下一個猜測 $x_{n+1}$。 切線斜率就是 $f'(x_n)$,簡單推一下交點,就得到牛頓法的招牌公式:

$$x_{n+1} = x_n - \frac{f(x_n)}{f'(x_n)}$$

def newton(f, df, x0, tol=1e-12, max_iter=100):
    x = x0
    for i in range(max_iter):
        fx = f(x)
        if abs(fx) < tol:
            return x, i
        x = x - fx / df(x)            # 沿切線滑到 x 軸
    return x, max_iter

# 一樣求 √2:f(x)=x²−2,f'(x)=2x
root, iters = newton(lambda x: x*x - 2, lambda x: 2*x, x0=2.0)
print(f"√2 ≈ {root:.15f},只用了 {iters} 次迭代")   # ≈ 1.414213562373095,約 5 次!

二分法用了約 40 次才到 12 位精度,牛頓法只用了5 次就到 15 位。差距怎麼這麼大? 因為牛頓法是二次收斂(quadratic convergence):每一步正確位數大約翻倍 (誤差 $e_{n+1} \approx C\,e_n^2$)。第一步準 1 位、下一步準 2 位、再下一步 4、8、16……一下就撞到機器 epsilon 的天花板。

⚠️
牛頓法的代價:它不保證收斂

速度是有條件的。牛頓法需要(1)會算導數,(2)一個夠好的起點。 起點不好時,它可能衝過頭、在兩點間來回跳、甚至飛到無窮遠;若某步踩到 $f'(x_n)\approx 0$(切線幾乎水平), 下一步會被甩到天邊。二分法又慢又笨,但絕不背叛你;牛頓法又快又聰明,但需要呵護。 實務上常把兩者結合(例如 Brent's method):平時用快的,一旦發現要飛出去就退回穩的。

割線法:沒有導數時的折衷

牛頓法要你提供導數 $f'$,但很多時候 $f$ 是個複雜的黑盒子,導數根本寫不出來。 割線法(secant method)的補救很自然:用最近兩個點連成的割線斜率, 去近似那條切線的斜率(這正是第 12 章「割線是切線的前身」的逆用):

$$x_{n+1} = x_n - f(x_n)\,\frac{x_n - x_{n-1}}{f(x_n) - f(x_{n-1})}$$

它不需要導數,收斂階約為 $p=\frac{1+\sqrt 5}{2}\approx 1.618$(就是黃金比例!)—— 比二分法的線性($p=1$)快,比牛頓法的二次($p=2$)略慢,是個漂亮的折衷。

「線性 vs 二次收斂」聽起來抽象,我們把三種方法求 $\sqrt 2$ 時的誤差隨迭代次數變化畫出來。 注意縱軸是對數刻度(每往下一格,誤差小 10 倍),這樣才看得出天差地別的速度:

ℹ️
收斂階(order of convergence)的定義

若誤差滿足 $|e_{n+1}| \approx C\,|e_n|^{p}$,則 $p$ 就是收斂階。$p=1$ 是線性(誤差等比例縮小,如二分法); $1<p<2$ 是超線性(如割線法 $p\approx 1.618$);$p=2$ 是二次(正確位數翻倍,如牛頓法)。 階數越高,衝到機器精度所需的迭代次數越少——但通常也要求越多資訊(牛頓法要導數)或越好的起點。 這是貫穿數值方法的核心權衡:速度、穩健、與所需資訊,三者難以兼得。

一句話:不動點迭代

其實上面這些都可以放進一個更一般的框架——不動點迭代(fixed-point iteration): 把方程式改寫成 $x = g(x)$ 的形式,然後反覆代入 $x_{n+1} = g(x_n)$, 當 $|g'(x)|<1$(這叫壓縮映射)時,序列就會收斂到那個讓 $x=g(x)$ 成立的「不動點」——它正是原方程式的根。 牛頓法,其實就是一個收斂得特別快的不動點迭代特例。

解線性方程組:直接法 vs 迭代法

從「解一個方程式」升級到「解一整組聯立方程式」:$A\mathbf{x} = \mathbf{b}$,其中 $A$ 是矩陣、$\mathbf{x}$ 與 $\mathbf{b}$ 是向量。 這是第 54 章線性代數的核心問題,也是整個科學計算最高頻的運算——電路分析、結構力學、PageRank、 最小平方擬合、偏微分方程離散化……到最後幾乎都化約成「解一個(可能超大的)線性系統」。 解法分兩大門派。

直接法:高斯消去與 LU 分解

你高中解聯立方程式用的「消元法」,正式名稱是高斯消去法(Gaussian elimination): 用列運算把矩陣化成上三角形,再由下往上回代。把這個過程「記錄下來」, 就得到 LU 分解(LU decomposition):把 $A$ 拆成一個下三角矩陣 $L$ 乘一個上三角矩陣 $U$,即 $A=LU$。 分解一次之後,要解任何右側 $\mathbf{b}$,都只需兩次便宜的三角回代。直接法的特點是 有限步驟得到(理論上的)精確解,對中小型稠密矩陣是首選。它的代價是 $O(n^3)$ 的運算量。 

import numpy as np

A = np.array([[3.0, 2.0, -1.0],
              [2.0, -2.0, 4.0],
              [-1.0, 0.5, -1.0]])
b = np.array([1.0, -2.0, 0.0])

x = np.linalg.solve(A, b)     # 內部就是做 LU 分解 + 回代
print(x)                      # [ 1.  -2.  -2. ]
print(np.allclose(A @ x, b))  # True:驗證 Ax 確實等於 b

為什麼「永遠別算反矩陣」

數學課教你 $A\mathbf{x}=\mathbf{b}$ 的解是 $\mathbf{x}=A^{-1}\mathbf{b}$,於是初學者很自然想寫 np.linalg.inv(A) @ b。請不要這樣做。這是新手最常見的數值錯誤之一,理由有三:

  • 更慢:算出整個反矩陣 $A^{-1}$ 比直接解一個方程組貴得多(大約 3 倍運算量),而你根本用不到反矩陣的全部。
  • 更不準:求逆再相乘會引入額外的捨入誤差,放大條件數的傷害;solve 走的路徑數值上穩定得多。
  • 沒必要:你要的是 $\mathbf{x}$,不是 $A^{-1}$。np.linalg.solve(A, b) 直奔答案。
⛔
記住這條鐵律

看到 $A^{-1}\mathbf{b}$,心裡翻譯成「解 $A\mathbf{x}=\mathbf{b}$」,用 np.linalg.solve(A, b), 而不是 np.linalg.inv(A) @ b。同理,$A^{-1}B$ 也用 solve(A, B)。 幾乎沒有任何正當理由需要顯式地把一個反矩陣算出來。這條原則,連資深工程師都常踩雷。

迭代法:為什麼大稀疏系統不用直接法

當矩陣巨大(比方 $10^6 \times 10^6$,在偏微分方程模擬裡稀鬆平常)但稀疏(sparse,絕大多數元素是 0)時, $O(n^3)$ 的直接法會直接撐爆記憶體與時間(光是存 $L$、$U$ 就把好不容易的稀疏性「填滿」毀掉)。 這時改用迭代法(iterative method):從一個猜測 $\mathbf{x}^{(0)}$ 出發,反覆改進,讓它收斂到解—— 又是「逼近 + 迭代」的老精神。每一步只需做矩陣–向量乘法(對稀疏矩陣超便宜),記憶體也省。

最入門的兩個迭代法是 Jacobi 法與 Gauss–Seidel 法。 想法都是「把第 $i$ 條方程式拿來解出第 $i$ 個未知數,其餘變數先用上一輪的估計值代入」:

$$x_i^{(k+1)} = \frac{1}{a_{ii}}\left(b_i - \sum_{j\neq i} a_{ij}\,x_j^{(k)}\right)$$

兩者差別只在一個小細節:Jacobi 整輪都用舊值;Gauss–Seidel 一算出新值就立刻拿來用,所以通常收斂更快。 

import numpy as np

def jacobi(A, b, iters=50):
    n = len(b)
    x = np.zeros(n)
    D = np.diag(A)                       # 對角線元素
    R = A - np.diagflat(D)               # 其餘部分
    for _ in range(iters):
        x = (b - R @ x) / D             # 一次更新全部分量
    return x

A = np.array([[10.0, -1.0, 2.0],
              [-1.0, 11.0, -1.0],
              [2.0, -1.0, 10.0]])
b = np.array([6.0, 25.0, -11.0])

print(jacobi(A, b))               # ≈ [1.043 2.270 -1.082]
print(np.linalg.solve(A, b))      # 對照直接法的答案,幾乎相同

迭代法不保證對任意矩陣收斂(常需要矩陣「對角佔優」之類的條件),且只給近似解。 但對巨大稀疏系統,它往往是唯一可行的選擇。實務上會用更強的迭代法(共軛梯度 CG、GMRES 等), scipy.sparse.linalg 都備好了。

內插與逼近:從離散點還原連續曲線

你手上常常只有一堆離散的資料點(實驗測量、感測器讀數),卻想知道「點與點之間」的值, 或想用一條平滑曲線概括整體趨勢。這就是內插(interpolation)與逼近(approximation)。 兩者有個關鍵區別:內插要求曲線剛好穿過每一個點;逼近只要求曲線整體貼近(允許不穿過,以換取平滑與抗雜訊)。

多項式內插與 Runge 現象

給定 $n+1$ 個點,理論上存在唯一一條 $n$ 次多項式穿過它們全部。 拉格朗日內插(Lagrange interpolation)給了一條漂亮的顯式公式來寫出它。 聽起來很美——點越多,用越高次的多項式,不就越準嗎?大錯特錯。

當你用等距的點去做高次多項式內插,多項式會在區間兩端劇烈地上下震盪, 誤差不減反增,而且增得很恐怖。這個違反直覺的災難叫 Runge 現象(Runge's phenomenon), 以函數 $f(x)=\frac{1}{1+25x^2}$ 為經典反例:

樣條:用低次多項式分段拼接

解法不是用一條高次多項式硬撐全場,而是分段:每兩個相鄰點之間用一小段低次(通常三次)多項式, 再要求接縫處平滑銜接——這就是樣條內插(spline),工程繪圖與動畫曲線的標準工具,既貼合又不亂震 (scipy.interpolate.CubicSpline 一行搞定)。

最小平方法:當資料有雜訊,別硬穿過去

如果你的資料來自有雜訊的測量,「硬穿過每一點」根本是錯誤的目標——你會把雜訊也一起內插進去。 正確的做法是逼近:選一個簡單模型(例如一條直線 $y=ax+b$),找出讓它整體離所有點最近的參數。 「最近」用「各點殘差平方和最小」來定義,這就是最小平方法(least squares)—— 而它,正是你在第 38 章見過的線性迴歸的數學本體! 

import numpy as np

# 一組帶雜訊、大致呈線性的資料
x = np.array([0, 1, 2, 3, 4, 5], dtype=float)
y = np.array([1.1, 2.9, 5.2, 6.8, 9.1, 11.2])

# 用最小平方擬合一條直線 y = a·x + b(deg=1 表示一次多項式)
a, b = np.polyfit(x, y, deg=1)
print(f"擬合直線:y = {a:.3f}·x + {b:.3f}")   # ≈ y = 2.017·x + 0.962

# polyfit 背後解的是「正規方程式」,等價於最小化 Σ(yᵢ − (a·xᵢ+b))²

第 38 章用梯度下降迭代地找這條線;這裡 polyfit 直接解正規方程式 $\mathbf{w}=(X^\top X)^{-1}X^\top \mathbf{y}$ 一步到位(當然,它內部明智地用 solve 而非真的求逆)。 同一個「線性迴歸」,一個是 ML 視角(梯度下降),一個是數值線性代數視角(最小平方)—— 你現在能看出它們是同一件事的兩張臉了,這正是把不同章節串起來的樂趣。

數值微分與積分

第 12 章教過微分(斜率)與積分(面積)的概念與封閉解。但當函數沒有公式、只有一堆數值(或反導數寫不出來)時, 我們得回到定義,用數值的方式硬算。

數值微分:有限差分

導數的定義是 $f'(x)=\lim_{h\to 0}\frac{f(x+h)-f(x)}{h}$。把極限拿掉、$h$ 取一個小但非零的數, 就得到前向差分(forward difference):

$$f'(x)\approx \frac{f(x+h)-f(x)}{h}\qquad(\text{截斷誤差 }O(h))$$

更聰明的是對稱地取兩側的中央差分(central difference),誤差小一個數量級:

$$f'(x)\approx \frac{f(x+h)-f(x-h)}{2h}\qquad(\text{截斷誤差 }O(h^2))$$

但這裡藏著本章開頭預告的那場拉扯:$h$ 越小,截斷誤差越小;可是 $h$ 太小時, $f(x+h)$ 與 $f(x-h)$ 幾乎相等,相減觸發災難性抵消,捨入誤差爆炸。 兩股力量一拉一扯,總誤差呈現一個 V 形,存在一個「最佳的 $h$」:

數值積分:從黎曼和到辛普森

第 12 章用黎曼和(一堆矩形)逼近積分,它能用,但收斂慢(誤差 $O(h)$)。 只要把「矩形」換成更貼合曲線的形狀,精度立刻飛躍:

  • 梯形法(trapezoidal rule):用梯形(連接相鄰兩點的直線)而非矩形去逼近,誤差 $O(h^2)$。
  • 辛普森法(Simpson's rule):更進一步,用拋物線(二次曲線)去逼近每一小段,誤差 $O(h^4)$—— 步長減半,誤差只剩 $\frac{1}{16}$,快得驚人。

$$\text{辛普森法:}\quad \int_a^b f\,dx \approx \frac{h}{3}\Big[f_0 + 4(f_1+f_3+\cdots) + 2(f_2+f_4+\cdots) + f_n\Big]$$

來算 $\int_0^{\pi}\sin x\,dx$(真值精確是 2),比較三種方法在相同切片數下的誤差:

import numpy as np

def riemann(f, a, b, n):
    x = np.linspace(a, b, n, endpoint=False)
    return np.sum(f(x)) * (b - a) / n           # 左端點矩形

def trapezoid(f, a, b, n):
    x = np.linspace(a, b, n + 1)
    y = f(x)
    h = (b - a) / n
    return h * (y[0]/2 + y[1:-1].sum() + y[-1]/2)

def simpson(f, a, b, n):                          # n 必須是偶數
    x = np.linspace(a, b, n + 1)
    y = f(x)
    h = (b - a) / n
    return h/3 * (y[0] + 4*y[1:-1:2].sum() + 2*y[2:-1:2].sum() + y[-1])

f, a, b, true = np.sin, 0, np.pi, 2.0
for n in [4, 8, 16]:
    print(f"n={n:2d}  矩形誤差={abs(riemann(f,a,b,n)-true):.2e}  "
          f"梯形誤差={abs(trapezoid(f,a,b,n)-true):.2e}  "
          f"辛普森誤差={abs(simpson(f,a,b,n)-true):.2e}")

# n= 4  矩形誤差=1.81e-01  梯形誤差=1.29e-01  辛普森誤差=4.56e-03
# n= 8  矩形誤差=7.72e-02  梯形誤差=3.23e-02  辛普森誤差=2.69e-04
# n=16  矩形誤差=3.80e-02  梯形誤差=8.06e-03  辛普森誤差=1.59e-05

看那一整排數字:同樣切 16 段,矩形法誤差還在 $10^{-2}$,辛普森法已經到 $10^{-5}$,精度差了上千倍。 這就是選對演算法的威力——不是算更多次,而是算得更聰明。實務上直接用 scipy.integrate.quad(自適應、高精度)就好,但理解背後這幾招,你才知道它在做什麼、什麼時候會失準。

常微分方程的數值解

終於來到最能展現「逼近 + 迭代」威力的舞台:常微分方程(ODE)。 一個 ODE 長這樣:$\frac{dy}{dt}=f(t, y)$,給定起點 $y(t_0)=y_0$,問「之後 $y$ 怎麼隨時間演變」。 它描述了世界上幾乎所有「會隨時間變化」的系統:人口、放射衰變、彈簧、行星、傳染病、電路。 而正如開頭所說,絕大多數 ODE 沒有封閉解——只能一步步往前「模擬」。

歐拉法:最樸素的時間步進

導數 $\frac{dy}{dt}$ 就是「$y$ 此刻變化的速率」。歐拉法(Euler method)的想法直白到近乎天真: 既然我知道現在的位置 $y_n$ 和現在的變化率 $f(t_n, y_n)$,那就假設「在接下來一小段時間 $h$ 內, 變化率維持不變」,直接沿著這個斜率走一步:

$$y_{n+1} = y_n + h\,f(t_n, y_n)$$

這其實就是把切線當成曲線,踏著一段段小直線往前「拼」出整條軌跡。$h$ 越小,拼得越像, 但步數越多。歐拉法的全域誤差是 $O(h)$——精度不高,卻是理解一切 ODE 解法的起點。 我們拿它模擬指數成長 $\frac{dy}{dt}=y,\ y(0)=1$(它的封閉解剛好是 $y=e^t$,方便驗證): 

import numpy as np

def euler(f, y0, t0, t_end, h):
    ts = np.arange(t0, t_end + h, h)
    ys = np.zeros(len(ts))
    ys[0] = y0
    for i in range(len(ts) - 1):
        ys[i+1] = ys[i] + h * f(ts[i], ys[i])   # 沿當前斜率走一步
    return ts, ys

f = lambda t, y: y                  # dy/dt = y
ts, ys = euler(f, y0=1.0, t0=0, t_end=2, h=0.5)
for t, y in zip(ts, ys):
    print(f"t={t:.1f}  歐拉={y:7.4f}  真值 e^t={np.exp(t):7.4f}")

# t=0.0  歐拉= 1.0000  真值 e^t= 1.0000
# t=0.5  歐拉= 1.5000  真值 e^t= 1.6487
# t=1.0  歐拉= 2.2500  真值 e^t= 2.7183
# t=1.5  歐拉= 3.3750  真值 e^t= 4.4817
# t=2.0  歐拉= 5.0625  真值 e^t= 7.3891   ← 步子太大,越走越偏

Runge–Kutta(RK4):業界主力

歐拉法的毛病在於「整段只用起點的斜率」,難怪會偏。四階 Runge–Kutta 法(RK4)聰明在: 它在一步之內試探多個位置的斜率(起點、兩個中點、終點),再加權平均出一個更有代表性的斜率才走, 全域誤差一舉降到 $O(h^4)$。同樣的步長,RK4 的精度遠勝歐拉法,因此成為大多數 ODE 求解器的預設主力。 

def rk4(f, y0, t0, t_end, h):
    ts = np.arange(t0, t_end + h, h)
    ys = np.zeros(len(ts))
    ys[0] = y0
    for i in range(len(ts) - 1):
        t, y = ts[i], ys[i]
        k1 = f(t,         y)
        k2 = f(t + h/2,   y + h/2 * k1)      # 用 k1 試探中點的斜率
        k3 = f(t + h/2,   y + h/2 * k2)      # 再修正一次中點斜率
        k4 = f(t + h,     y + h   * k3)      # 終點的斜率
        ys[i+1] = y + h/6 * (k1 + 2*k2 + 2*k3 + k4)   # 加權平均
    return ts, ys

ts, ys = rk4(lambda t, y: y, 1.0, 0, 2, 0.5)
print(f"RK4 在 t=2 的值 = {ys[-1]:.5f},真值 e² = {np.exp(2):.5f}")
# RK4 在 t=2 的值 = 7.38380,真值 e² = 7.38906   ← 只走 4 步就準到小數第 2 位!

實務上你會直接用 scipy.integrate.solve_ivp(內建 RK45 等自適應步長演算法,會自動在曲線陡的地方縮小步長)。 但親手寫過歐拉與 RK4,你才真正懂「模擬一個動態系統」是怎麼一回事—— 從遊戲物理引擎、SpaceX 的軌道計算,到天氣預報模型,核心都是這套「時間步進」。

一句話帶過的幾位鄰居

數值方法是一片大陸,本章只走了主幹道。幾個重要的鄰近領域,先在你心裡種下名字:

  • 最佳化(optimization):找函數的極值。你其實已經學過它最重要的工具—— 第 12、38 章的梯度下降;更進階的有牛頓法的多維版、L-BFGS、共軛梯度等,是 ML 訓練的引擎。
  • 蒙地卡羅方法(Monte Carlo):用隨機取樣來估計積分或機率(呼應第 14 章)。 當維度極高(例如百維積分),傳統的「切格子」徹底失效,撒隨機點反而又快又準——這是金融、物理、渲染的祕密武器。
  • 快速傅立葉變換(FFT):把訊號在「時域」與「頻域」之間瞬間切換的神級演算法, $O(n\log n)$ 取代了笨方法的 $O(n^2)$,撐起了音訊/影像壓縮、通訊與訊號處理的半邊天。

那為什麼還要懂原理?因為這些函式不是魔法,它們會在你看不見的地方失準或失敗: 牛頓法可能發散、矩陣可能病態、$h$ 太小反而更不準、高次內插會 Runge 震盪。 當 solve_ivp 跑出一條離譜的軌跡、當你的擬合曲線扭曲、當答案的有效位數莫名變少時, 只有懂原理的人才知道發生了什麼、該怎麼救。工具讓你跑得快,原理讓你跌不死。

數值素養:用工具,但保持懷疑

一個有數值素養的工程師,看到任何浮點計算的結果,心裡都會默默問三句: (1) 這個問題的條件數好嗎?(答案天生能信幾位) (2) 我用的演算法穩定嗎?(有沒有災難性抵消之類的陷阱) (3) 我的誤差來自截斷還是捨入?往哪個方向調參數? 會問這三句,你就從「會呼叫函式庫的人」升級成「真正懂數值計算的人」了。

動手做:練習題

✏️ 練習 55.1 實作

用二分法與牛頓法兩種方式求 $\sqrt 2$(即解 $f(x)=x^2-2=0$), 各自跑到誤差小於 $10^{-12}$,並印出兩者分別用了幾次迭代。比較看看差距有多大,並解釋為什麼。

看解答

關鍵是在每次迭代記錄「離真值 $\sqrt 2$ 的誤差」,直到小於門檻才停。

import math
TRUE = math.sqrt(2)

def bisection_count(f, a, b, tol=1e-12):
    n = 0
    while (b - a) / 2 > tol:
        m = (a + b) / 2
        n += 1
        if f(a) * f(m) < 0:
            b = m
        else:
            a = m
    return (a + b) / 2, n

def newton_count(f, df, x0, tol=1e-12):
    x, n = x0, 0
    while abs(f(x)) > tol:
        x = x - f(x) / df(x)
        n += 1
    return x, n

f  = lambda x: x*x - 2
df = lambda x: 2*x

xb, nb = bisection_count(f, 1, 2)
xn, nn = newton_count(f, df, 2.0)
print(f"二分法:√2 ≈ {xb:.13f},迭代 {nb} 次")   # 約 40 次
print(f"牛頓法:√2 ≈ {xn:.13f},迭代 {nn} 次")   # 約 5 次

二分法約 40 次、牛頓法約 5 次。原因就是收斂階:二分法是線性收斂 (每步誤差只減半,要多準一位十進位得多做 $\log_2 10 \approx 3.3$ 次), 牛頓法是二次收斂(正確位數每步翻倍)。差距會隨要求的精度越高而越懸殊。

✏️ 練習 55.2 思考

展示一個災難性抵消的例子。用標準求根公式 $x = \frac{-b + \sqrt{b^2-4ac}}{2a}$ 去解 $x^2 + 10^8 x + 1 = 0$ 的較小那個根(接近 0 的根), 你會發現算出來嚴重失準。找出失準的原因,並用一個數值穩定的方法修正它。 (提示:Vieta 公式說兩根乘積 $x_1 x_2 = c/a$。)

看解答

當 $b=10^8$ 很大時,$\sqrt{b^2-4ac}\approx b$,於是 $-b+\sqrt{b^2-4ac}$ 是「兩個幾乎相等的大數相減」, 有效位數大量抵消,算出的小根幾乎全是雜訊。

import math
a, b, c = 1.0, 1e8, 1.0

# 天真寫法:會抵消
disc = math.sqrt(b*b - 4*a*c)
x_naive = (-b + disc) / (2*a)

# 穩定寫法:先算「不抵消」的大根,再用 Vieta x1*x2=c/a 反推小根
x_big   = (-b - disc) / (2*a)        # 這個是 -b - √… ,兩個負的大數相加,不抵消
x_small = (c / a) / x_big            # 由乘積反推小根

print("天真小根:", x_naive)    # -7.45e-09  ← 錯得離譜
print("穩定小根:", x_small)    # -1.0000...e-08  ← 正確(理論值 ≈ -1e-8)
print("驗證(代回方程):", a*x_small**2 + b*x_small + c)  # ≈ 0

訣竅:永遠讓 $-b$ 和 $\pm\sqrt{\cdots}$ 同號相加(這裡 $b>0$ 故先取 $-b-\sqrt{\cdots}$ 算大根), 避開相減;另一根用乘積關係 $x_1 x_2 = c/a$ 安全地反推。這是處理二次公式的標準數值技巧。

✏️ 練習 55.3 實作

用最小平方法擬合一組帶雜訊的資料(呼應第 38 章線性迴歸)。 自己造一組大致符合 $y=3x+2$ 但加了雜訊的資料,用 numpy.polyfit 擬合一條直線, 印出擬合出的斜率與截距,看看是否接近真實的 3 與 2。進階:不用 polyfit, 改用 numpy.linalg.lstsq 直接解最小平方,得到一樣的答案。

看解答 
import numpy as np
rng = np.random.default_rng(0)

x = np.linspace(0, 10, 30)
y = 3*x + 2 + rng.normal(0, 1.5, size=x.size)   # 真實 y=3x+2 加高斯雜訊

# 方法一:polyfit
a, b = np.polyfit(x, y, 1)
print(f"polyfit:   斜率={a:.3f}, 截距={b:.3f}")   # 接近 3 與 2

# 方法二:自己組「設計矩陣」X,解最小平方 min ‖Xβ − y‖²
X = np.column_stack([x, np.ones_like(x)])        # 每列是 [xᵢ, 1]
beta, *_ = np.linalg.lstsq(X, y, rcond=None)
print(f"lstsq:     斜率={beta[0]:.3f}, 截距={beta[1]:.3f}")

兩種方法給出幾乎一樣的結果,都會落在 $(3, 2)$ 附近(因雜訊而略有偏差)。 這正是第 38 章「線性迴歸」的數值本體:lstsq 在最小化殘差平方和 $\sum_i (y_i - (ax_i+b))^2$, 等價於解正規方程式 $X^\top X\,\beta = X^\top y$。差別只在第 38 章用梯度下降迭代逼近這個解, 這裡用直接法一步解出。注意我們沒有顯式算 $(X^\top X)^{-1}$——lstsq 內部用更穩定的 QR/SVD 分解。

✏️ 練習 55.4 實作

用梯形法與辛普森法數值計算 $\displaystyle\int_0^1 e^{-x^2}\,dx$ (還記得嗎?這個積分沒有封閉解,只能數值算)。用同樣的切片數 $n=10$ 算兩者, 並以 scipy.integrate.quad 的高精度結果當「真值」,比較兩種方法的誤差。 你會看到辛普森法為什麼是甜蜜點。

看解答 
import numpy as np
from scipy import integrate

f = lambda x: np.exp(-x**2)

def trapezoid(f, a, b, n):
    x = np.linspace(a, b, n + 1); y = f(x); h = (b - a) / n
    return h * (y[0]/2 + y[1:-1].sum() + y[-1]/2)

def simpson(f, a, b, n):              # n 須為偶數
    x = np.linspace(a, b, n + 1); y = f(x); h = (b - a) / n
    return h/3 * (y[0] + 4*y[1:-1:2].sum() + 2*y[2:-1:2].sum() + y[-1])

true, _ = integrate.quad(f, 0, 1)    # 高精度「真值」≈ 0.7468241328
print(f"真值(quad)    = {true:.10f}")
print(f"梯形法 (n=10)  誤差 = {abs(trapezoid(f,0,1,10) - true):.2e}")
print(f"辛普森 (n=10)  誤差 = {abs(simpson(f,0,1,10)   - true):.2e}")

# 真值(quad)    = 0.7468241328
# 梯形法 (n=10)  誤差 = 6.20e-04
# 辛普森 (n=10)  誤差 = 3.92e-07   ← 同樣 10 片,精度高了三個數量級

梯形法誤差 $O(h^2)$,辛普森法 $O(h^4)$。同樣只切 10 片,辛普森就準了上千倍—— 因為它用拋物線而非直線去逼近每一小段。進階觀察:把 $n$ 加倍,梯形法誤差約變 $\frac14$, 辛普森法約變 $\frac{1}{16}$,完全符合各自的誤差階。

✏️ 練習 55.5 實作

用歐拉法解一個簡單 ODE:放射性衰變 $\frac{dy}{dt}=-0.5\,y,\ y(0)=100$。 它的封閉解是 $y(t)=100\,e^{-0.5t}$。用步長 $h=0.5$ 與 $h=0.1$ 各模擬到 $t=4$, 比較兩者在終點與真值的誤差,驗證「步長減半,誤差大約也減半」(歐拉法是一階方法)。

看解答 
import numpy as np

def euler(f, y0, t_end, h):
    n = int(round(t_end / h))
    y, t = y0, 0.0
    for _ in range(n):
        y = y + h * f(t, y)
        t += h
    return y

f = lambda t, y: -0.5 * y
true = 100 * np.exp(-0.5 * 4)        # 真值 y(4) ≈ 13.5335

for h in [0.5, 0.1, 0.05]:
    approx = euler(f, 100.0, 4.0, h)
    print(f"h={h:4}  歐拉終值={approx:8.4f}  誤差={abs(approx-true):.4f}")

# h= 0.5  歐拉終值=13.3484  誤差=0.1851
# h= 0.1  歐拉終值=13.4990  誤差=0.0345    ← h 縮 5 倍,誤差約縮 5 倍
# h=0.05  歐拉終值=13.5164  誤差=0.0171    ← 再減半,誤差又約減半

誤差與步長 $h$ 大致成正比(一階方法 $O(h)$),這就是歐拉法的特徵。 想用更少步數達到更高精度?換 RK4($O(h^4)$):同樣 $h=0.5$,RK4 的誤差會小到 $10^{-4}$ 等級, 碾壓歐拉法。試著把練習裡的歐拉換成本章的 rk4 跑跑看。

✏️ 練習 55.6 挑戰

進階題,二選一(或都做):

(A) 牛頓法的發散。用牛頓法求多項式 $f(x)=x^3-2x+2$ 的根。 試試起點 $x_0=0$,觀察它會發生什麼事;再試 $x_0=-2$,看是否收斂。解釋為何某些起點會讓牛頓法卡住或發散。

(B) 直接法 vs 迭代法。對一個對角佔優的線性系統,分別用 numpy.linalg.solve(直接) 與你手寫的 Jacobi(迭代)求解,比較結果與收斂行為。

看解答

(A) 起點 $x_0=0$ 會掉進一個週期循環:$f'(0)=-2$, $x_1 = 0 - \frac{2}{-2} = 1$;$f(1)=1,\ f'(1)=1$,$x_2 = 1 - 1 = 0$……於是在 $0,1,0,1,\dots$ 之間 永遠來回跳,根本不收斂。而 $x_0=-2$ 離真實的根(約 $-1.7693$)夠近,會快速收斂。

def newton_trace(f, df, x0, steps=8):
    x = x0
    for i in range(steps):
        print(f"  step {i}: x = {x:.6f}")
        if df(x) == 0:
            print("  導數為 0,牛頓法當機!"); break
        x = x - f(x) / df(x)
    return x

f  = lambda x: x**3 - 2*x + 2
df = lambda x: 3*x**2 - 2

print("起點 x0 = 0(會循環):")
newton_trace(f, df, 0.0)          # 在 0, 1, 0, 1, ... 之間跳,不收斂
print("起點 x0 = -2(會收斂):")
newton_trace(f, df, -2.0)         # 迅速收斂到 ≈ -1.769292

教訓:牛頓法的二次收斂只在「起點夠靠近根」時成立。起點不好,可能落入循環、發散、 或踩到 $f'(x)=0$(切線水平,下一步飛向無窮)。這就是為什麼穩健的求根器會把牛頓法和二分法混搭。

(B) 直接法一步給出精確解;Jacobi 需要數十次迭代逐步逼近,但每步極便宜。

import numpy as np

A = np.array([[10., -1., 2., 0.],
              [-1., 11., -1., 3.],
              [2., -1., 10., -1.],
              [0., 3., -1., 8.]])
b = np.array([6., 25., -11., 15.])

x_direct = np.linalg.solve(A, b)          # 直接法:精確解

# 迭代法:Jacobi,印出每幾步離直接解多遠
D = np.diag(A); R = A - np.diagflat(D); x = np.zeros(4)
for k in range(1, 26):
    x = (b - R @ x) / D
    if k % 5 == 0:
        print(f"Jacobi 第 {k:2d} 次,離精確解差距 = {np.linalg.norm(x - x_direct):.2e}")

print("直接解 =", x_direct)
print("迭代解 =", x)

你會看到 Jacobi 的誤差每幾次迭代就縮小一個數量級,逐步逼近直接法的答案。 對這種小稠密矩陣,直接法當然完勝;但若矩陣是 $10^6$ 維的稀疏矩陣,直接法會爆記憶體, 迭代法才是唯一能跑的選擇——這正是兩派分工的根本原因。

把這一章串起來

我們從「大多數數學問題沒有漂亮的封閉解」這個樸素事實出發,一路看見電腦如何用「逼近 + 迭代」征服它們。回顧路標:

  • 浮點數的真相:IEEE 754 用有限位元逼近實數,機器 epsilon($\sim 10^{-16}$)是精度天花板; 誤差分捨入與截斷兩種,而災難性抵消(相近的數相減)是頭號殺手。
  • 條件數 vs 穩定性:條件數差是問題天生難算;不穩定是演算法把好問題算爛。兩者要分清楚。
  • 求根:二分法穩但慢(線性收斂),牛頓法快但需導數與好起點(二次收斂),割線法折衷(超線性)。
  • 線性方程組:小稠密用直接法(LU 分解),大稀疏用迭代法(Jacobi/Gauss–Seidel);永遠用 solve,別算反矩陣。
  • 內插與逼近:高次多項式內插會 Runge 震盪,改用樣條;有雜訊時用最小平方逼近——它就是第 38 章的線性迴歸。
  • 數值微積分與 ODE:有限差分求導(留意最佳 $h$)、梯形/辛普森求積分、歐拉/RK4 解微分方程,全是「逼近 + 迭代」。

這一章其實在講一件大事:數值方法是「理論數學」與「真實計算」之間的那座橋。 微積分、線性代數告訴你「答案存在、長什麼樣」;數值方法告訴你「在一台只有有限位元、會犯捨入誤差的真實機器上, 怎麼真的把它算出來」。你每天用的每一個 ML 框架、每一個物理模擬器、每一套 CAD 軟體、每一次天氣預報, 掀開蓋子,底層轉動的都是這一章的這些齒輪。當你下次呼叫 scipy 的某個函式時, 希望你會心一笑——因為你現在知道蓋子底下發生了什麼,也知道它什麼時候會出錯。
