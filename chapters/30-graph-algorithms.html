從一張靜止的圖,到會「走路」的演算法

回想一下第 15 章。我們把捷運路線、好友關係、城市航線、網頁連結,全都抽象成同一個東西: 圖(graph)——一堆頂點(vertex),加上連接它們的邊(edge)。 我們甚至學會了怎麼把圖塞進電腦:用鄰接矩陣,或更常用的鄰接串列。

但那時候的圖,是靜止的。它只是被「存」在記憶體裡,像一張攤開在桌上的地圖。 真正讓它發揮威力的,是接下來這件事:在圖上「走路」。從某個頂點出發, 沿著邊一步步探索,回答各種問題——

  • 從台北車站到淡水,最少要轉幾站?(最短步數)
  • 這個社交圈裡,有沒有人和我完全沒有任何共同朋友鏈?(連通性)
  • 這學期想修「機器學習」,我得先把哪些先修課修完、按什麼順序?(拓樸排序)
  • Google 地圖怎麼在零點幾秒內,從上億條道路裡挑出開車最快的那條路?(加權最短路徑)

這些問題的答案,全都來自同一個家族的演算法。而這個家族的兩位族長,就是 BFS 與 DFS—— 所有圖演算法的根。把它們學透,後面的拓樸排序、環偵測、最短路徑,都只是在它們身上加裝幾個零件而已。 別怕,我們會從最直覺的地方開始。

先把工具擺好:這一章的範例圖

為了全章一致,我們先定義一個小圖,當作貫穿後面所有走訪示範的「白老鼠」。它是一個無向、無權圖, 6 個頂點 A~F、6 條邊。我們沿用第 15 章的結論:鄰接串列是絕大多數圖演算法的預設選擇 (因為現實的圖通常稀疏),所以這裡用 Python 的字典來存它,每個頂點對應一份鄰居清單。 

# 用鄰接串列(字典)存圖:每個頂點 → 它的鄰居清單
graph = {
    'A': ['B', 'C'],
    'B': ['A', 'D', 'E'],
    'C': ['A', 'F'],
    'D': ['B'],
    'E': ['B', 'F'],
    'F': ['C', 'E'],
}
# 想知道 B 連到誰?直接看 graph['B'] → ['A', 'D', 'E']
🔑
圖走訪的鐵律:標記「看過的」頂點

圖和樹最大的不同,是圖可以有環。如果你不記錄「哪些頂點已經拜訪過」,走到環裡就會 無限繞圈、永遠停不下來。所以幾乎每個圖走訪演算法的骨架都長一樣: 一個「待辦容器」(佇列或堆疊)+ 一個「已訪集合」(visited)。 兩者的差別,只在那個容器是「先進先出」還是「後進先出」——而這個小小的差別, 就分出了 BFS 和 DFS 兩條路。

廣度優先搜尋 BFS:像水波一樣擴散

想像你在一片平靜的池塘正中央丟下一顆石頭。水波會怎麼擴散?一圈一圈、由近到遠: 先是緊鄰落點的那一圈,再來是外面一圈,然後更外面一圈……永遠是「把目前這一圈走完, 才往外踏一步」。廣度優先搜尋(Breadth-First Search, BFS)就是這個水波。

從起點出發,BFS 先拜訪所有「距離 1 步」的頂點,再拜訪所有「距離 2 步」的,以此類推。 它層層擴散(level by level),所以又叫「層序走訪」。要做到這件事,我們需要一個 佇列(queue)——還記得第 29 章嗎?佇列是「先進先出(FIFO)」的容器, 正好能保證「先排隊的(較近的)先被處理」。

BFS 怎麼跑:手動走一遍

用圖 30-1,從 A 出發。我們維護一個佇列(隊伍)和一個「已看過」集合:

  1. 起點 A 入列,標記已看過。隊伍:[A]
  2. 取出 A,看它的鄰居 B、C:都沒看過 → 標記並入列。隊伍:[B, C]
  3. 取出 B,鄰居 A(看過)、D、E:D、E 沒看過 → 入列。隊伍:[C, D, E]
  4. 取出 C,鄰居 A(看過)、F:F 沒看過 → 入列。隊伍:[D, E, F]
  5. 取出 D,鄰居只有 B(看過)。隊伍:[E, F]
  6. 取出 E,鄰居 B、F 都看過了。隊伍:[F]
  7. 取出 F,鄰居都看過。隊伍空了 → 結束。

走訪順序:A → B → C → D → E → F。注意這個順序漂亮地分成三「層」: {A}(第 0 層,距離 0)、{B, C}(第 1 層,距離 1)、{D, E, F}(第 2 層,距離 2)。 這就是 BFS 最珍貴的性質——它走到每個頂點時,走的恰好是「邊數最少」的路。

Python 實作 BFS

我們用 collections.deque 當佇列——它的 popleft() 是 $O(1)$ 的「從前端取出」, 比 list 的 pop(0)($O(n)$)快得多。這是第 29 章學過的細節,現在派上用場了。 

from collections import deque

def bfs(graph, start):
    visited = {start}              # 已看過的頂點(集合,查詢 O(1))
    order = []                     # 記錄走訪順序
    queue = deque([start])         # 佇列:先進先出
    while queue:
        node = queue.popleft()     # 從隊伍「前端」取出
        order.append(node)
        for neighbor in graph[node]:
            if neighbor not in visited:
                visited.add(neighbor)   # 關鍵:一「入列」就立刻標記
                queue.append(neighbor)  # 放到隊伍「尾端」
    return order

print(bfs(graph, 'A'))   # ['A', 'B', 'C', 'D', 'E', 'F']
⚠️
常見陷阱:要在「入列時」標記,不是「出列時」

很多初學者把 visited.add 寫在 popleft() 之後。這在小圖看起來「也會動」, 但同一個頂點可能被多個鄰居重複塞進佇列,造成大量重複工作,最壞情況退化成指數級。 記住:一旦把某頂點放進佇列,它的命運就定了(最短距離已確定),立刻標記,別讓它再被加入第二次。

BFS 的招牌應用:無權圖的最短路徑

BFS 為什麼這麼重要?因為它免費就解出了一個大問題:在一張無權圖裡 (每條邊都算「1 步」),從起點到任何頂點的最少邊數是多少。理由很簡單: BFS 是按距離一層一層擴散的,所以它第一次碰到某個頂點時,走的必然是最短的那條路。

把 order 換成「距離字典」,稍微改一下就好:

from collections import deque

def bfs_distances(graph, start):
    dist = {start: 0}              # 距離字典兼當「已看過」用
    queue = deque([start])
    while queue:
        node = queue.popleft()
        for neighbor in graph[node]:
            if neighbor not in dist:            # 還沒有距離 = 還沒看過
                dist[neighbor] = dist[node] + 1 # 比上一層多走一步
                queue.append(neighbor)
    return dist

print(bfs_distances(graph, 'A'))
# {'A': 0, 'B': 1, 'C': 1, 'D': 2, 'E': 2, 'F': 2}

如果你不只要「幾步」,還要「實際走哪條路」,只要多記一個 parent(前驅) 字典: 每次把某頂點加入佇列時,記下「我是從誰過來的」。最後從終點沿著 parent 一路回溯到起點, 反轉就是最短路徑: 

from collections import deque

def bfs_path(graph, start, goal):
    parent = {start: None}        # 記錄每個頂點是「從誰」第一次到達的
    queue = deque([start])
    while queue:
        node = queue.popleft()
        if node == goal:
            break
        for neighbor in graph[node]:
            if neighbor not in parent:
                parent[neighbor] = node
                queue.append(neighbor)
    if goal not in parent:
        return None               # 起點走不到終點
    # 從終點沿 parent 回溯,再反轉
    path, cur = [], goal
    while cur is not None:
        path.append(cur)
        cur = parent[cur]
    path.reverse()
    return path

print(bfs_path(graph, 'A', 'F'))   # ['A', 'C', 'F']
🔑
BFS 的時間複雜度:$O(V + E)$

每個頂點最多入列、出列一次($O(V)$);每條邊在掃描鄰居時最多被看兩次($O(E)$)。 合起來是 $O(V + E)$——和「圖的大小」成線性。這已經是「必須把每個頂點和每條邊 至少看一次」的問題所能達到的最佳速度了。空間複雜度同樣是 $O(V)$(visited 與佇列)。

BFS 還能做什麼

同一套「層層擴散」的骨架,稍微改裝就能解一票問題:

  • 最少步數問題:魔術方塊最少轉幾次能還原?騎士從棋盤這格到那格最少跳幾步? 字梯(word ladder)從一個單字變到另一個最少改幾個字母?只要「每一步代價相同」,就是 BFS 的主場。
  • 連通元件(connected components):從某點 BFS,能走到的所有頂點就是同一塊。稍後詳談。
  • 二分圖檢測(bipartite check):邊走邊把相鄰頂點塗成相反顏色,若發現「相鄰卻同色」就不是二分圖。稍後實作。
  • 洪水填充(flood fill):小畫家的「油漆桶」工具、踩地雷點開一片空白,本質都是在格子圖上跑 BFS/DFS。

深度優先搜尋 DFS:不撞牆不回頭

BFS 是謹慎的探勘隊,把附近搜乾淨才前進。深度優先搜尋(Depth-First Search, DFS) 則是個莽撞的冒險家:挑一條路,一頭栽到底,直到走不下去(死路),才退回上一個岔路口 試另一條。就像走迷宮時你沿著右手邊的牆一直走——這正是 DFS 的精神。

BFS 用佇列(先進先出),DFS 則用堆疊(stack,後進先出):最晚發現的岔路, 最先被探索。而堆疊最自然的化身,就是函式呼叫自己——遞迴。所以 DFS 通常寫成遞迴, 簡潔得不可思議。

DFS 怎麼跑:遞迴版

一樣從 A 出發,鄰居按清單順序嘗試:

def dfs(graph, start):
    visited = set()
    order = []
    def visit(node):
        visited.add(node)
        order.append(node)
        for neighbor in graph[node]:
            if neighbor not in visited:
                visit(neighbor)      # 對鄰居遞迴:一路鑽到底
    visit(start)
    return order

print(dfs(graph, 'A'))   # ['A', 'B', 'D', 'E', 'F', 'C']

走訪順序 A → B → D → E → F → C,和 BFS 截然不同。跟著走一遍:A 先衝到 B, B 衝到 D(死路,退回),再從 B 衝到 E,E 衝到 F,F 衝到 C(C 的鄰居都看過了,退回), 然後一路退回 A。注意 C 雖然是 A 的直接鄰居,卻最後才被走到——因為 DFS 先把 B 那條路鑽到天荒地老。

迭代版:自己管理一個堆疊

遞迴很漂亮,但 Python 的遞迴深度有上限(預設約 1000 層),圖一大就可能「堆疊溢位」。 這時可以改用迭代版,自己拿一個 list 當堆疊。技巧是把鄰居逆序壓入, 讓走訪順序和遞迴版一致: 

def dfs_iter(graph, start):
    visited = set()
    order = []
    stack = [start]
    while stack:
        node = stack.pop()          # 從「頂端」取出(後進先出)
        if node in visited:
            continue                # 可能被重複壓入,出列時再檢查
        visited.add(node)
        order.append(node)
        for neighbor in reversed(graph[node]):   # 逆序壓入
            if neighbor not in visited:
                stack.append(neighbor)
    return order

print(dfs_iter(graph, 'A'))   # ['A', 'B', 'D', 'E', 'F', 'C']
ℹ️
迭代 DFS 的小差異:在「出列時」才標記

注意迭代版把 visited 檢查放在 pop() 之後。這是因為同一個頂點可能在被處理前 就被好幾個鄰居壓進堆疊;我們允許它重複入堆疊,但靠出列時的 if node in visited: continue 確保它只被真正處理一次。這和 BFS「入列即標記」的策略不同,是堆疊行為帶來的小細節。

發現時間與完成時間:DFS 的隱藏寶藏

DFS 有個 BFS 沒有的迷人結構。我們替每個頂點記兩個時間戳:發現時間(discovery time)—— 第一次踏進它的時刻;以及完成時間(finish time)——它所有子孫都走完、要退出它的時刻。 想像一個碼表,每發生一次「進入」或「退出」就 +1。 

def dfs_times(graph, start):
    color = {v: 'white' for v in graph}   # white=未發現
    disc, fin = {}, {}
    clock = [0]                            # 用 list 當可變的計時器
    def visit(u):
        clock[0] += 1
        disc[u] = clock[0]                 # 進入 u:記下發現時間
        color[u] = 'gray'                  # gray=發現了、還在堆疊上
        for v in graph[u]:
            if color[v] == 'white':
                visit(v)
        color[u] = 'black'                 # black=完成(子孫全走完)
        clock[0] += 1
        fin[u] = clock[0]                  # 退出 u:記下完成時間
    visit(start)
    return disc, fin

disc, fin = dfs_times(graph, 'A')
for v in 'ABCDEF':
    print(v, disc[v], '/', fin[v])
# A 1 / 12   B 2 / 11   D 3 / 4   E 5 / 10   F 6 / 9   C 7 / 8

看那些數字:每個頂點的 [發現, 完成] 區間,要嘛完全互相包含,要嘛完全不重疊—— 絕不會「交錯」。這叫括號定理(parenthesis theorem):DFS 的進出, 就像一串配對工整的括號。這個結構不是花拳繡腿——它正是拓樸排序與 強連通元件演算法的數學地基,等一下你就會看到它發光。

DFS 的應用

  • 找出兩點間是否有路徑:從起點 DFS,看走不走得到終點。是「連通嗎?」最直接的解法。
  • 偵測環:DFS 過程中若碰到「還在堆疊上(gray)」的頂點,代表繞回去了 → 有環。稍後詳談。
  • 走迷宮、解謎:迷宮就是格子圖,DFS 的「一路走到底、撞牆回頭」正是人走迷宮的直覺。
  • 拓樸排序、強連通元件:都建立在 DFS 的完成時間之上。

來看「找路徑」這個最基本的應用,順便示範遞迴怎麼「提早回報成功」:

def has_path(graph, start, goal):
    visited = set()
    def visit(node):
        if node == goal:
            return True
        visited.add(node)
        for neighbor in graph[node]:
            if neighbor not in visited:
                if visit(neighbor):       # 任何一條子路找到了,就一路回報 True
                    return True
        return False
    return visit(start)

print(has_path(graph, 'A', 'F'))   # True
print(has_path(graph, 'D', 'C'))   # True(D-B-...-C 走得到)

BFS vs DFS:該用哪一個?

兩者都是 $O(V+E)$、都能走遍整張圖,差別只在「探索的順序」。但這個順序差異, 決定了它們各自擅長的題型。先看一張概念圖,再看取捨表。

面向BFS(廣度優先)DFS(深度優先)
資料結構佇列(queue,FIFO)堆疊(stack,LIFO)/ 遞迴
探索順序由近到遠,一層一層挑一條路鑽到底,再回頭
無權最短路徑✅ 天生就會(第一次碰到即最短)❌ 不保證最短
記憶體(最壞)$O(寬度)$:寬圖會吃很多(整層都在佇列裡)$O(深度)$:深圖會吃很多(遞迴堆疊)

💡
一句話記住怎麼選

看到「最少步數 / 最近 / 最短(無權)」就用 BFS; 看到「有沒有一條路 / 偵測環 / 相依排序 / 回溯窮舉」就用 DFS。 兩者骨架幾乎一樣,差別只在那個容器是隊伍還是堆疊——這是整個圖演算法世界裡 CP 值最高的一個認知。

連通元件:這張圖分成幾座「島」?

一張圖不一定是「整片相連」的。它可能像一片群島:有幾塊互相連通,塊與塊之間卻沒有任何邊。 每一塊「內部彼此走得到、與外界走不到」的最大子圖,就是一個連通元件(connected component)。

找出所有連通元件的演算法非常自然:對每個「還沒被走過」的頂點,發動一次 BFS 或 DFS, 把這次能碰到的所有頂點圈成一塊。掃完所有頂點,就分好了: 

def connected_components(graph):
    visited = set()
    components = []
    for start in graph:                  # 確保每個頂點都被考慮到
        if start not in visited:
            comp = []
            stack = [start]              # 這裡用 DFS;換成 deque 就是 BFS
            while stack:
                node = stack.pop()
                if node in visited:
                    continue
                visited.add(node)
                comp.append(node)
                for nb in graph[node]:
                    if nb not in visited:
                        stack.append(nb)
            components.append(comp)
    return components

islands = {
    'A': ['B'], 'B': ['A', 'C'], 'C': ['B'],   # 第一座島
    'D': ['E'], 'E': ['D'],                      # 第二座島
    'F': [],                                     # 孤島(沒有鄰居)
}
print(connected_components(islands))
# [['A', 'B', 'C'], ['D', 'E'], ['F']]

有向圖的進階版:強連通元件(SCC)

上面講的是無向圖。換到有向圖,「連通」要更講究:光是「A 走得到 B」還不夠, 得「A 走得到 B,而且 B 也走得回 A」才算同一夥。一群頂點若兩兩互相可達, 就構成一個強連通元件(Strongly Connected Component, SCC)。

把每個 SCC 縮成一個點,整張有向圖就會塌成一個沒有環的有向圖(DAG)——這個「凝聚圖」 在編譯器、相依分析裡用途極廣(例如找出程式裡互相遞迴呼叫的一群函式)。

ℹ️
兩個經典 SCC 演算法(一句話版)

Kosaraju 演算法:做兩次 DFS——先在原圖跑一次記下完成時間,再把所有邊「反向」, 按完成時間由大到小跑第二次,每棵 DFS 樹就是一個 SCC。Tarjan 演算法:只跑一次 DFS, 用一個「low-link 值」追蹤每個頂點能繞回的最早祖先,巧妙地一趟就切出所有 SCC。兩者都是 $O(V+E)$; 細節留到進階,但你已經看到——它們全都站在 DFS 的「完成時間」這個肩膀上。

拓樸排序:替「有先後的任務」排隊

你想修「機器學習」,但它要先修「演算法」;「演算法」又要先修「資料結構」和「離散數學」…… 這些「A 必須在 B 之前」的限制,構成一個有向無環圖(Directed Acyclic Graph, DAG): 邊 A → B 代表「A 要排在 B 前面」。問題是:能不能把所有課排成一條線性順序, 讓每門課都排在它所有先修課之後?

這就是拓樸排序(topological sort)。它的存在有個前提:圖裡不能有環—— 如果 A 要先於 B、B 要先於 C、C 又要先於 A,那是死結,誰也排不出來(這也是「無環」這個詞的由來)。

做法一:Kahn 演算法(看「入度」)

Kahn 的想法非常符合直覺:沒有任何先修課的科目(入度為 0)可以馬上修。 修完它,就把它「移除」,它指向的科目的「待修先修數」各減一;若某科目減到 0,代表它的先修都修完了, 輪到它。這其實就是一個用佇列驅動的 BFS。入度(in-degree)就是「指向我的箭頭數」。 

from collections import deque

def topo_sort_kahn(graph):
    # 1. 算每個頂點的入度
    indeg = {u: 0 for u in graph}
    for u in graph:
        for v in graph[u]:
            indeg[v] += 1
    # 2. 所有入度為 0 的頂點先入列
    queue = deque(u for u in graph if indeg[u] == 0)
    order = []
    while queue:
        u = queue.popleft()
        order.append(u)
        for v in graph[u]:           # 「移除」u,等於拔掉它的所有出邊
            indeg[v] -= 1
            if indeg[v] == 0:        # v 的先修都修完了
                queue.append(v)
    if len(order) != len(graph):
        raise ValueError("圖中有環,無法拓樸排序")   # 順帶偵測了環!
    return order

courses = {
    '程式設計': ['資料結構', '離散數學'],
    '資料結構': ['作業系統', '演算法'],
    '離散數學': ['演算法', '機器學習'],
    '作業系統': [],
    '演算法': ['機器學習'],
    '機器學習': [],
}
print(topo_sort_kahn(courses))
# ['程式設計', '資料結構', '離散數學', '作業系統', '演算法', '機器學習']

留意最後那個 if len(order) != len(graph):如果圖裡有環,環上的頂點入度永遠減不到 0、 進不了佇列,最後 order 會少幾個。所以 Kahn 演算法順手就能偵測有向環——一魚兩吃。

做法二:DFS + 完成時間

還記得 DFS 的完成時間嗎?這裡它大顯身手。觀察:在 DAG 裡,一個頂點 u 一定比它指向的所有頂點 「晚完成」(因為要先把子孫走完才會輪到 u 退出)。所以把所有頂點按完成時間由大到小 排,就是一個合法的拓樸序。實作上,只要在 DFS 「完成」一個頂點時把它推進清單,最後反轉即可: 

def topo_sort_dfs(graph):
    visited = set()
    order = []
    def visit(u):
        visited.add(u)
        for v in graph[u]:
            if v not in visited:
                visit(v)
        order.append(u)          # 關鍵:子孫都走完(「完成」)才加入
    for u in graph:              # 圖可能不連通,每個未訪點都要起跑
        if u not in visited:
            visit(u)
    order.reverse()             # 完成順序的反向 = 拓樸序
    return order

print(topo_sort_dfs(courses))
# ['程式設計', '離散數學', '資料結構', '演算法', '作業系統', '機器學習'](也是合法的)
🔑
拓樸排序無所不在

只要有「A 必須在 B 之前」的相依關係,背後就是拓樸排序:課程先修規劃、 make / Bazel 等建置系統決定先編譯哪個檔、套件管理員(pip、npm) 安裝相依套件的順序、試算表公式的重算順序、專案管理的任務排程。 下次你 npm install 看到一堆套件依序裝好,那就是一次拓樸排序在跑。

環偵測:有沒有「繞回原地」?

「圖裡有沒有環」是個極常見的問題:相依關係有沒有死結?電路有沒有迴路?死結偵測(deadlock)裡, 資源等待圖有沒有環?有向圖和無向圖的判法略有不同,我們各看一種漂亮的解法。

有向圖:用三種顏色追蹤 DFS

在有向圖裡,環的特徵是 DFS 過程中出現回邊(back edge):走到一個還在目前遞迴路徑上 (還沒完成)的頂點。我們用三種顏色標記每個頂點的狀態:

  • 白色(WHITE):還沒拜訪。
  • 灰色(GRAY):正在拜訪中——它還在當前的遞迴堆疊上。
  • 黑色(BLACK):已完成,它和它的子孫都走完了。

規則很簡單:DFS 時如果遇到一個灰色的鄰居,就代表繞回了自己的祖先 → 找到環。

def has_cycle_directed(graph):
    WHITE, GRAY, BLACK = 0, 1, 2
    color = {u: WHITE for u in graph}
    def visit(u):
        color[u] = GRAY                  # 進入:標記「在堆疊上」
        for v in graph[u]:
            if color[v] == GRAY:         # 碰到灰色 → 回邊 → 有環
                return True
            if color[v] == WHITE and visit(v):
                return True
        color[u] = BLACK                 # 完成:標記「已離開堆疊」
        return False
    return any(color[u] == WHITE and visit(u) for u in graph)

dag = {'A': ['B', 'D'], 'B': ['C'], 'C': [], 'D': []}
cyclic = {'A': ['B', 'D'], 'B': ['C'], 'C': ['A'], 'D': []}
print(has_cycle_directed(dag))     # False
print(has_cycle_directed(cyclic))  # True(A→B→C→A)
⚠️
別把「黑色」也當成環

只有碰到灰色頂點才算環。碰到黑色頂點只代表「這條路我以前走過、已經處理完了」, 那是 DAG 裡完全正常的「交會」(例如兩條路最後都通到同一個已完成的點),不是環。 初學者最常見的 bug,就是用單一個 visited 集合(只分白/非白),把正常交會誤判成環。 有向圖環偵測一定要分清楚「在路徑上(灰)」與「已完成(黑)」。

無向圖:用第 29 章的並查集

無向圖判環有個特別優雅的方法,直接呼應第 29 章的並查集(Union-Find / Disjoint Set Union)。 想法是:一條一條把邊加進來,每條邊連接兩個頂點。如果某條邊的兩個端點原本就已經在同一個集合裡, 代表它們之間早就有路可通——這條新邊一接,就多出一個環。 

class UnionFind:
    def __init__(self, items):
        self.parent = {x: x for x in items}   # 一開始每個點自成一國
    def find(self, x):
        while self.parent[x] != x:
            self.parent[x] = self.parent[self.parent[x]]  # 路徑壓縮
            x = self.parent[x]
        return x
    def union(self, a, b):
        ra, rb = self.find(a), self.find(b)
        if ra == rb:
            return False        # 已在同一集合 → 這條邊製造了環
        self.parent[ra] = rb    # 合併兩國
        return True

def has_cycle_undirected(vertices, edges):
    uf = UnionFind(vertices)
    for u, v in edges:
        if not uf.union(u, v):  # union 回傳 False 表示形成環
            return True
    return False

V = ['A', 'B', 'C', 'D']
print(has_cycle_undirected(V, [('A','B'), ('B','C'), ('C','D')]))        # False(一條鏈)
print(has_cycle_undirected(V, [('A','B'), ('B','C'), ('C','A')]))        # True(三角形)

這個「加邊、若兩端已連通就成環」的並查集技巧,等到第 31 章講最小生成樹(MST)的 Kruskal 演算法時 會再次登場——Kruskal 正是「由小到大加邊,但跳過會成環的邊」。你看,工具一旦學會,就到處都用得上。

最短路徑:這一章的重頭戲

終於到了壓軸。BFS 已經解了「無權圖」的最短路徑,但真實世界的邊往往有權重: 兩城市間的公里數、網路的延遲、轉乘的時間、轉帳的手續費。這時「邊數最少」不等於「總權重最小」—— 繞個遠路但每段都短,可能反而更划算。我們需要更強的工具。

三位主角依「能力」與「代價」排排站:

  • Dijkstra:單一起點到所有點,要求邊權非負。又快又常用,地圖導航的基礎。
  • Bellman-Ford:單一起點到所有點,容許負權,還能偵測負環。比較慢,但更通用。
  • Floyd-Warshall:一口氣算出所有點對之間的最短路徑。三行迴圈,優雅但較貴。

核心動作:鬆弛(relaxation)

三個演算法共用一個核心動作,叫鬆弛(relax)。我們替每個頂點維護一個「目前已知到它的最短距離」 dist[v](一開始除了起點是 0,其餘都是無限大 ∞)。鬆弛一條邊 u → v(權重 w)的意思是:

「如果先到 u、再走這條邊,會比我現在記的 dist[v] 更短,那就更新它。」
也就是:if dist[u] + w < dist[v]: dist[v] = dist[u] + w

所有最短路徑演算法,本質上都是「用對的順序,把每條邊鬆弛足夠多次」。差別只在於那個「順序」和「次數」。

Dijkstra:每次貪心地確定「最近的未定點」

Dijkstra 的策略是貪心的:維護一群「距離已經確定」的頂點,每一步從「還沒確定」的頂點中, 挑出目前 dist 最小的那個,宣告它「定案」,再用它去鬆弛它的鄰居。 為什麼挑最小的就能「定案」?因為所有邊權非負,沒有任何尚未確定的路徑能「繞回來」讓它變得更短。

「每次挑出最小的」——這不正是優先佇列(priority queue)的拿手好戲嗎?第 29 章的 最小堆積(min-heap)正好提供 $O(\log V)$ 的「取出最小」。我們用 Python 的 heapq: 

import heapq

def dijkstra(graph, start):
    dist = {start: 0}
    pq = [(0, start)]                       # 最小堆:(目前距離, 頂點)
    while pq:
        d, u = heapq.heappop(pq)            # 取出目前最近的未定點
        if d > dist.get(u, float('inf')):
            continue                        # 過期的舊紀錄,跳過
        for v, w in graph[u]:               # 鬆弛 u 的每條出邊
            nd = d + w
            if nd < dist.get(v, float('inf')):
                dist[v] = nd                # 找到更短的路 → 更新
                heapq.heappush(pq, (nd, v)) # 把新距離丟進堆積
    return dist

graph = {
    'A': [('B', 4), ('C', 1)],
    'B': [('E', 3)],
    'C': [('B', 2), ('D', 5)],
    'D': [('E', 1)],
    'E': [],
}
print(dijkstra(graph, 'A'))
# {'A': 0, 'C': 1, 'B': 3, 'D': 6, 'E': 6}

逐步追蹤一次(· 表示 ∞),體會「鬆弛」如何一步步把距離修短:

取出(定案)ABCDE動作
(初始)0····起點 A = 0
A(0)041··鬆弛 A→B=4、A→C=1
C(1)0316·A→C→B=3 比 4 短!D=6
B(3)03166鬆弛 B→E=3+3=6
D(6)03166D→E=7,沒比 6 短,不更新
E(6)03166全部定案,結束
⚠️
為什麼 Dijkstra 不能有負權邊?

Dijkstra 的貪心建立在一個假設上:「一旦某頂點被取出定案,就不可能再有更短的路了。」這在邊權非負時成立, 但負邊會打破它。看下圖:從 A 出發,A→B = 4,A→C = 5,C→B = −3。Dijkstra 會先取出 B(距離 4) 並宣告定案——但真正的最短是 A→C→B = 5 + (−3) = 2!那條更短的路,因為要先經過「比較遠」的 C, 被貪心策略錯過了。有負權?請改用 Bellman-Ford。

🔑
Dijkstra 的複雜度

用二元堆積實作的 Dijkstra 是 $O((V + E) \log V)$:每個頂點取出一次($V \log V$), 每條邊最多觸發一次 push($E \log V$)。這也是為什麼第 29 章要先把堆積學好—— 沒有那個「$O(\log V)$ 取最小」的優先佇列,Dijkstra 跑不快。 (用更進階的費氏堆積可降到 $O(E + V \log V)$,但實務上二元堆積已經夠用。)

Bellman-Ford:慢,但什麼都吃

Bellman-Ford 不耍貪心,改走「暴力但可靠」的路:把每一條邊都鬆弛,重複 $V-1$ 輪。 為什麼是 $V-1$ 輪?因為任何最短路徑最多經過 $V-1$ 條邊(不會重複頂點);每跑一輪, 至少能讓「多走一條邊才到」的頂點距離定下來。$V-1$ 輪後,所有最短距離都收斂了。 

def bellman_ford(vertices, edges, start):
    dist = {v: float('inf') for v in vertices}
    dist[start] = 0
    # 鬆弛 V-1 輪
    for _ in range(len(vertices) - 1):
        for u, v, w in edges:
            if dist[u] + w < dist[v]:
                dist[v] = dist[u] + w
    # 第 V 輪:若還能鬆弛,代表存在「負權環」
    for u, v, w in edges:
        if dist[u] + w < dist[v]:
            raise ValueError("圖中存在負權環,最短路徑無定義")
    return dist

V = ['A', 'B', 'C', 'D']
edges = [('A','B',4), ('A','C',5), ('C','B',-3), ('B','D',3), ('C','D',6)]
print(bellman_ford(V, edges, 'A'))
# {'A': 0, 'B': 2, 'C': 5, 'D': 5}  ← B = 2,正確抓到了 A→C→B 的負邊捷徑

注意它正確算出 B = 2,正是 Dijkstra 會搞錯的那個負邊例子。代價是速度: Bellman-Ford 是 $O(V \cdot E)$,比 Dijkstra 慢一截。但它能處理負權,而且——

🔑
偵測負環:Bellman-Ford 的獨門絕活

如果圖裡有一個負權環(繞一圈總權重為負),那「最短路徑」根本不存在——你可以一直繞那個環, 距離無限往下掉。Bellman-Ford 能偵測它:跑完 $V-1$ 輪後再多跑一輪, 如果還有任何邊能被鬆弛,就證明存在負環。這個能力在金融套利偵測(貨幣兌換繞一圈賺到錢=負環) 等場景非常有用。

Floyd-Warshall:三行迴圈,算出「所有點對」

前兩個都是「單源(single-source)」:從一個起點出發。如果你想要的是每一對頂點之間 的最短距離(例如算出城市兩兩之間的最短車程表),可以用 Floyd-Warshall。它是個漂亮的 動態規劃(dynamic programming)——這正是第 32 章的主題,這裡先嚐一口。

它的核心想法美得令人讚嘆:逐一考慮「允許經過頂點 $k$ 當中繼站」。對每一對 $(i, j)$, 檢查「繞經 $k$」會不會比「目前的 $i$ 到 $j$」更短。把 $k$ 從第一個頂點試到最後一個, 答案就全部出來了——整個演算法只有三層迴圈: 

INF = float('inf')

def floyd_warshall(dist):
    # dist[i][j] = i 到 j 的直達邊權(沒有邊就填 INF,自己到自己填 0)
    n = len(dist)
    for k in range(n):              # 中繼站 k
        for i in range(n):         # 起點 i
            for j in range(n):     # 終點 j
                if dist[i][k] + dist[k][j] < dist[i][j]:
                    dist[i][j] = dist[i][k] + dist[k][j]
    return dist

# 4 個頂點 0~3 的距離矩陣(INF 代表沒有直達邊)
g = [
    [0,   3,   INF, 7],
    [8,   0,   2,   INF],
    [5,   INF, 0,   1],
    [2,   INF, INF, 0],
]
for row in floyd_warshall(g):
    print(row)
# [0, 3, 5, 6]  /  [5, 0, 2, 3]  /  [3, 6, 0, 1]  /  [2, 5, 7, 0]

Floyd-Warshall 是 $O(V^3)$,只用一個 $V \times V$ 矩陣,程式短到不可思議。 對「點不多但要查全部點對」的情境(例如數百個節點的路網),它常常是最省事的選擇。 它也容許負邊(但同樣不能有負環)。

演算法解什麼負權?負環偵測?複雜度關鍵工具
BFS單源、無權最短路——$O(V+E)$佇列
Dijkstra單源,邊權非負❌❌$O((V{+}E)\log V)$最小堆積
Bellman-Ford單源,可負權✅✅$O(V \cdot E)$反覆鬆弛
Floyd-Warshall全點對✅(可改裝)$O(V^3)$動態規劃

不只最短路:再往前一步

圖演算法的世界比這一章大得多。兩個你接下來會遇到的重要主題,先用一句話讓你有個鉤子:

  • 最小生成樹(Minimum Spanning Tree, MST):用「最便宜的一組邊」把所有頂點連起來、不留環, 像是用最少的纜線串連所有城市。經典解法 Kruskal(由小到大加邊,用並查集跳過成環者)與 Prim(從一點長出,用優先佇列挑最便宜的邊)——是第 31 章的主角,你會發現這一章的並查集與堆積又回來了。
  • 最大流(maximum flow):把圖想成水管網路,每條邊有「容量」,問從源頭到匯點最多能送多少流量? Ford-Fulkerson 與 Edmonds-Karp(其實就是在「殘餘網路」上反覆跑 BFS)能解它。最大流還和「最小切割」 這個看似無關的問題完美對偶,是組合最佳化裡最迷人的結果之一。

圖演算法在真實世界

這一章學的不是抽象玩具,而是你每天在用的科技骨架:

  • 地圖導航:Google Maps、車載導航的「最快路線」,核心就是 Dijkstra 及其加速版 A*(用「離終點還有多遠」的估計值當啟發式,大幅剪枝)。路網就是加權圖,權重是時間或距離。
  • 社群網路:LinkedIn 的「第幾度人脈」就是 BFS 求最短步數;「你可能認識的人」靠分析共同鄰居; 找出緊密社群(community detection)則和連通元件、圖分割有關。
  • 網頁排名(PageRank):Google 把整個網際網路看成一張巨大有向圖(網頁是點、超連結是邊), 用「一個隨機點擊者長期停留在每頁的機率」來衡量頁面重要性——這是圖上的一種隨機漫步,本質仍是圖演算法。
  • 相依解析:套件管理員(pip、npm、apt)、建置系統(make、Bazel)、試算表重算, 全靠拓樸排序決定先做什麼;順便用環偵測抓出「循環相依」這個惡夢。
  • 推薦系統:電商與串流平台把「使用者—商品」建成二分圖,用圖上的關聯與隨機漫步替你推薦 「買了這個的人也買了……」。
  • 其他:編譯器的暫存器配置(圖著色)、IC 佈線、航班排程、生物的蛋白質交互作用網路…… 只要問題裡有「東西」與「關係」,圖演算法就在那裡。

動手練習

圖演算法是「看懂很容易、自己寫才會卡」的典型。下面九題由淺入深,務必先動手再看解答。

✏️ 練習 30.1 實作

用 BFS 實作 shortest_steps(graph, start),回傳起點到每個頂點的最少邊數(無權圖)。 用本章的範例 graph 從 'A' 驗證,應得 A:0, B:1, C:1, D:2, E:2, F:2。

看解答 
from collections import deque

def shortest_steps(graph, start):
    dist = {start: 0}
    queue = deque([start])
    while queue:
        u = queue.popleft()
        for v in graph[u]:
            if v not in dist:            # 第一次碰到 = 最短
                dist[v] = dist[u] + 1
                queue.append(v)
    return dist

graph = {'A':['B','C'],'B':['A','D','E'],'C':['A','F'],
         'D':['B'],'E':['B','F'],'F':['C','E']}
print(shortest_steps(graph, 'A'))
# {'A': 0, 'B': 1, 'C': 1, 'D': 2, 'E': 2, 'F': 2}

重點:BFS 按層擴散,所以「第一次把 v 放進佇列」時記下的距離,保證就是最短。 不需要任何比較或更新——這是 BFS 解無權最短路的精髓。

✏️ 練習 30.2 實作

用 DFS 實作 has_path(graph, start, goal),判斷兩點間是否存在路徑。 試試 has_path(graph, 'A', 'F')(True)與一個走不到的例子。

看解答 
def has_path(graph, start, goal):
    visited = set()
    def visit(u):
        if u == goal:
            return True
        visited.add(u)
        for v in graph[u]:
            if v not in visited and visit(v):
                return True
        return False
    return visit(start)

print(has_path(graph, 'A', 'F'))   # True
# 若把 D 改成孤立(graph['D']=[] 且沒人指向它),則 has_path 到 D 會是 False

也可以用迭代 + 堆疊寫,避免遞迴過深。核心都是「從起點探索,看能否觸及終點」。

✏️ 練習 30.3 實作

用 BFS 實作 is_bipartite(graph):能否把所有頂點塗成兩色,使得每條邊兩端顏色不同? (等價於「圖中沒有奇數長度的環」。)提示:BFS 時把鄰居塗成相反色,發現「相鄰卻同色」就回 False。

看解答 
from collections import deque

def is_bipartite(graph):
    color = {}
    for start in graph:                  # 處理可能不連通的圖
        if start in color:
            continue
        color[start] = 0
        queue = deque([start])
        while queue:
            u = queue.popleft()
            for v in graph[u]:
                if v not in color:
                    color[v] = 1 - color[u]   # 塗相反色
                    queue.append(v)
                elif color[v] == color[u]:
                    return False              # 相鄰同色 → 不是二分圖
    return True

square = {'A':['B','D'],'B':['A','C'],'C':['B','D'],'D':['A','C']}  # 4-環
triangle = {'A':['B','C'],'B':['A','C'],'C':['A','B']}              # 3-環
print(is_bipartite(square))     # True(偶環可二分)
print(is_bipartite(triangle))   # False(奇環不行)

二分圖在「配對問題」(學生選社團、工作分派)裡是核心結構。檢測它就是一次帶顏色的 BFS。

✏️ 練習 30.4 實作

用 Kahn 演算法(入度)實作 topo_sort(graph),回傳一個合法的拓樸序;若圖有環則回傳 None。用本章的 courses 字典驗證。

看解答 
from collections import deque

def topo_sort(graph):
    indeg = {u: 0 for u in graph}
    for u in graph:
        for v in graph[u]:
            indeg[v] += 1
    queue = deque(u for u in graph if indeg[u] == 0)
    order = []
    while queue:
        u = queue.popleft()
        order.append(u)
        for v in graph[u]:
            indeg[v] -= 1
            if indeg[v] == 0:
                queue.append(v)
    return order if len(order) == len(graph) else None   # 少了 = 有環

courses = {'程式設計':['資料結構','離散數學'],'資料結構':['作業系統','演算法'],
           '離散數學':['演算法','機器學習'],'作業系統':[],
           '演算法':['機器學習'],'機器學習':[]}
print(topo_sort(courses))
# ['程式設計', '資料結構', '離散數學', '作業系統', '演算法', '機器學習']

若把 courses['機器學習']=['程式設計'] 造一個環,函式會回傳 None——Kahn 順手偵測了環。

✏️ 練習 30.5 實作

用 heapq 實作 dijkstra(graph, start),圖以 {u: [(v, w), ...]} 表示。 回傳起點到所有頂點的最短距離。用本章的加權 graph 從 'A' 驗證 (A:0, C:1, B:3, D:6, E:6)。

看解答 
import heapq

def dijkstra(graph, start):
    dist = {start: 0}
    pq = [(0, start)]
    while pq:
        d, u = heapq.heappop(pq)
        if d > dist.get(u, float('inf')):
            continue                     # 過期紀錄,跳過
        for v, w in graph[u]:
            nd = d + w
            if nd < dist.get(v, float('inf')):
                dist[v] = nd
                heapq.heappush(pq, (nd, v))
    return dist

graph = {'A':[('B',4),('C',1)],'B':[('E',3)],
         'C':[('B',2),('D',5)],'D':[('E',1)],'E':[]}
print(dijkstra(graph, 'A'))   # {'A':0,'C':1,'B':3,'D':6,'E':6}

進階:把 dist 旁邊再記一個 parent 字典(鬆弛成功時 parent[v]=u), 就能像 BFS 那樣回溯出「實際的最短路徑」,而不只是距離。

✏️ 練習 30.6 實作

實作 has_cycle_directed(graph),用「白/灰/黑」三色 DFS 判斷有向圖是否有環。 用一個 DAG 與一個含環圖各驗證一次。

看解答 
def has_cycle_directed(graph):
    WHITE, GRAY, BLACK = 0, 1, 2
    color = {u: WHITE for u in graph}
    def visit(u):
        color[u] = GRAY
        for v in graph[u]:
            if color[v] == GRAY:                 # 回邊 → 有環
                return True
            if color[v] == WHITE and visit(v):
                return True
        color[u] = BLACK
        return False
    return any(color[u] == WHITE and visit(u) for u in graph)

print(has_cycle_directed({'A':['B'],'B':['C'],'C':[]}))        # False
print(has_cycle_directed({'A':['B'],'B':['C'],'C':['A']}))     # True

切記:只有遇到灰色(還在當前遞迴路徑上)的頂點才是環;遇到黑色只是「以前走完的點」,不算環。

✏️ 練習 30.7 挑戰

實際問題:迷宮最少步數。給一個 0/1 的方格 grid(0 可走、1 是牆),求從左上角 (0,0) 走到右下角的最少步數(只能上下左右移動);走不到回 -1。 這是 BFS 在隱式格子圖上的經典應用。

看解答 
from collections import deque

def shortest_maze(grid):
    rows, cols = len(grid), len(grid[0])
    goal = (rows - 1, cols - 1)
    if grid[0][0] == 1 or grid[goal[0]][goal[1]] == 1:
        return -1
    queue = deque([((0, 0), 0)])         # ((列, 行), 已走步數)
    seen = {(0, 0)}
    while queue:
        (r, c), steps = queue.popleft()
        if (r, c) == goal:
            return steps
        for dr, dc in [(-1,0),(1,0),(0,-1),(0,1)]:
            nr, nc = r + dr, c + dc
            if 0 <= nr < rows and 0 <= nc < cols \
               and grid[nr][nc] == 0 and (nr, nc) not in seen:
                seen.add((nr, nc))
                queue.append(((nr, nc), steps + 1))
    return -1

maze = [
    [0, 0, 1, 0],
    [1, 0, 1, 0],
    [0, 0, 0, 0],
    [0, 1, 1, 0],
]
print(shortest_maze(maze))   # 6

關鍵體悟:格子迷宮根本不必先「建出一張圖」。每個格子是頂點,「上下左右相鄰且不是牆」就是邊—— 我們在 BFS 過程中即時推算鄰居(這叫「隱式圖」)。因為每步代價都是 1,BFS 自然給出最少步數。

✏️ 練習 30.8 挑戰

字梯(Word Ladder)。給起始單字 begin、目標單字 end 和一份字典 word_list, 每次只能改一個字母且改完必須仍是字典裡的字。求從 begin 變到 end 的最短變換序列長度 (含頭尾),走不到回 0。提示:把每個單字當頂點,「差一個字母」當邊,跑 BFS。

看解答 
from collections import deque
import string

def word_ladder(begin, end, word_list):
    words = set(word_list)
    if end not in words:
        return 0
    queue = deque([(begin, 1)])          # (目前單字, 序列長度)
    seen = {begin}
    while queue:
        word, length = queue.popleft()
        if word == end:
            return length
        for i in range(len(word)):       # 嘗試改每一位
            for ch in string.ascii_lowercase:
                nxt = word[:i] + ch + word[i+1:]
                if nxt in words and nxt not in seen:
                    seen.add(nxt)
                    queue.append((nxt, length + 1))
    return 0

wl = ["hot","dot","dog","lot","log","cog"]
print(word_ladder("hit", "cog", wl))   # 5:hit→hot→dot→dog→cog

這題完美展示「把問題建模成圖」的威力:單字之間並沒有現成的邊,但只要定義好「什麼算相鄰」, 一個看似文字遊戲的問題,瞬間變成最單純的 BFS 最短路。建模,往往比寫演算法更關鍵。

✏️ 練習 30.9 挑戰

實作 bellman_ford(vertices, edges, start),其中 edges 是 (u, v, w) 的清單。 回傳最短距離字典;若圖中存在負權環,回傳字串 "NEGATIVE CYCLE"。 用一個含負邊但無負環的圖,以及一個含負環的圖各驗證一次。

看解答 
def bellman_ford(vertices, edges, start):
    dist = {v: float('inf') for v in vertices}
    dist[start] = 0
    for _ in range(len(vertices) - 1):       # 鬆弛 V-1 輪
        for u, v, w in edges:
            if dist[u] + w < dist[v]:
                dist[v] = dist[u] + w
    for u, v, w in edges:                     # 第 V 輪還能鬆弛 → 負環
        if dist[u] + w < dist[v]:
            return "NEGATIVE CYCLE"
    return dist

V = ['A','B','C','D']
ok = [('A','B',4),('A','C',5),('C','B',-3),('B','D',3),('C','D',6)]
print(bellman_ford(V, ok, 'A'))   # {'A':0,'B':2,'C':5,'D':5}

# 製造負環:B→C→B 總和 -3 + 1 = -2 < 0
neg = [('A','B',1),('B','C',1),('C','B',-3)]
print(bellman_ford(['A','B','C'], neg, 'A'))   # NEGATIVE CYCLE

「鬆弛 $V-1$ 輪後,若還能再鬆弛就有負環」——這個第 $V$ 輪的檢查,是 Bellman-Ford 相對 Dijkstra 最珍貴的能力,也是貨幣套利偵測等應用的理論基礎。

把這一章串起來

我們讓第 15 章那張靜止的圖,真正「走」了起來。回顧這趟旅程的路標:

  • BFS(佇列、層層擴散)解無權圖最短步數;DFS(堆疊/遞迴、一路到底)擅長找路、偵測環、相依分析。 兩者都是 $O(V+E)$,差別只在「待辦容器」是隊伍還是堆疊。
  • 連通元件用一次次走訪圈出「島」;有向圖的強連通元件(SCC)則靠 Kosaraju / Tarjan, 而它們都站在 DFS 完成時間的肩膀上。
  • 拓樸排序替有相依關係的任務排出合法順序——Kahn(入度 + 佇列)或 DFS(完成時間反轉), 順便偵測有向環。
  • 環偵測:有向圖用「白/灰/黑」三色找回邊;無向圖用第 29 章的並查集「加邊即成環」。
  • 最短路徑:核心是「鬆弛」。Dijkstra(非負權、最小堆積、貪心)、 Bellman-Ford(可負權、能偵測負環、反覆鬆弛)、Floyd-Warshall(全點對、動態規劃、三層迴圈)。
  • 這些演算法撐起了地圖導航、社群網路、PageRank、相依解析、推薦系統——你每天都在使用它們。

你也許注意到了:這一章幾乎沒有發明新的資料結構,而是把第 29 章的佇列、堆積、並查集 拿來「組裝」成強大的演算法。這正是演算法學習的精髓——少數幾個紮實的零件,排列組合出無窮的解法。 下一章(第 31 章)我們會用同樣的零件解最小生成樹,接著(第 32 章)正式進入 動態規劃的世界,把這裡偷瞄到的 Floyd-Warshall 講個透徹。
