軟體除錯與測試理論

寫程式只佔工程師一小半的時間,另一大半,花在「為什麼它不動?」與「我怎麼知道它真的會動?」這兩個問題上。這一章,我們把抓 bug 從「手藝」升級成「科學」,把測試從「跑跑看」升級成「理論」。

一隻蛾,與一場永無止境的戰爭

1947 年 9 月,在哈佛大學一台叫 Harvard Mark II 的巨型電機式電腦裡,一隻飛蛾不小心飛進了繼電器(relay), 被金屬接點夾死,造成電腦運算錯誤。工程師團隊(其中包括日後的程式語言先驅 葛麗絲・霍普(Grace Hopper))把那隻蛾用膠帶黏進工作日誌,旁邊寫下一行字: 「First actual case of bug being found.」(發現的第一個真正的「蟲」。)

📜
「bug」其實比那隻蛾更老

把工程故障叫做「bug」,並不是 1947 年才發明的——愛迪生(Thomas Edison)早在 1870 年代的信件裡就用過 bug 形容電路裡的小毛病。 但那隻被黏進日誌的飛蛾,是第一個「貨真價實、有實體」的 bug,也讓 debug(除蟲/除錯) 這個詞從此深植程式設計的血液裡。 那本日誌與那隻蛾,如今收藏在史密森尼學會(Smithsonian)。我們在第 5 章提過這段故事,這一章,我們要正式把「除蟲」變成一門學問。

從那隻蛾到今天,軟體的世界翻天覆地,但有一件事從沒變過:程式總是有 bug,而且找 bug 通常比寫程式還久。 研究與業界經驗都指向同一個殘酷數字——軟體開發的總成本裡,有將近一半花在「除錯與測試」上。一個剛入行的工程師最大的震撼, 往往是發現「寫」原來是最快、最爽的部分;真正吃掉你人生的,是「為什麼它不動」。

為什麼會這樣?因為寫程式是創造(從無到有,你掌握全局),而除錯是調查(從一個壞掉的結果, 反推到那行你看了十遍都沒看出來的錯)。這一章的目標,就是讓你的調查不再靠運氣與直覺,而是靠一套可重複、可教學、可累積的方法。 我們會先把「bug 到底是什麼」這件事,從根本上講清楚。

bug 的本質:缺陷、感染、失效

日常我們把所有問題都籠統地叫「bug」,但要科學地除錯,得先把它拆成三個不同的東西。 這個區分來自軟體除錯大師 安德烈亞斯・澤勒(Andreas Zeller) 的經典著作《Why Programs Fail》, 它是整章的地基:

  • 缺陷(defect / fault):程式碼裡那行寫錯的東西。它是靜態的、躺在原始碼裡的事實。例如某個鍵打錯字、某個 < 該寫成 <=。
  • 感染(infection):當程式執行到缺陷時,產生了一個錯誤的程式狀態(某個變數變成了不該有的值)。注意——此刻畫面上往往什麼都沒發生,錯誤只是悄悄潛伏在記憶體裡。
  • 失效(failure):那個被感染的錯誤狀態,一路傳播、擴大,最後外顯成你看得見的錯誤——崩潰、錯誤的輸出、當掉。這是你唯一直接觀察到的東西。

為什麼要這麼龜毛地分三層?因為它解釋了除錯最反直覺、也最讓新手抓狂的一件事——

🔑
失效的地方,幾乎不是缺陷的地方

程式在 A 函式崩潰,不代表 bug 在 A。很可能是 B 函式很早以前就把一個變數弄錯了(缺陷+感染), 那個錯值安安靜靜地被傳來傳去,直到 A 拿它來用才終於爆炸(失效)。 新手會死盯著崩潰那一行猛改,卻怎麼也改不對——因為兇手根本不在現場。 除錯的第一個心法,就是別把「失效點」當成「缺陷點」。

來看一個小到不能再小、卻完美演示這條鏈的例子:

def build_profile(name, age):
    return {"name": name, "aeg": age}     # ① 缺陷:鍵打錯字,"aeg" 應為 "age"

def greet(profile):
    return f"Hi {profile['name']}, you are {profile['age']} years old"

p = build_profile("Mei", 20)              # ② 感染:p 帶著錯誤狀態誕生,但此刻毫無異狀
# ......中間可能隔了幾十行、好幾個函式,p 被傳來傳去,一切看起來都很正常......
print(greet(p))                           # ③ 失效:KeyError: 'age'

Python 會在 greet 的 profile['age'] 這行丟出 KeyError: 'age'。 一個新手會在 greet 裡翻來覆去找錯——但 greet 一個字都沒寫錯!真正的缺陷遠在 build_profile, 在一個此刻看起來「執行得很順、毫無問題」的地方。失效在 greet,缺陷在 build_profile—— 這就是傳播鏈在現實裡的長相。理解了這一點,你才會養成「往回走」而不是「盯著現場」的除錯本能。

科學除錯法:像科學家一樣抓 bug

我們在第 5 章埋過一句話:「不要亂槍打鳥地瞎改,要先形成假設,再驗證。」這一節,我們把它展開成一套完整的方法論—— 科學除錯法(scientific debugging)。它的核心洞見是:除錯,本質上就是科學研究。 你面對一個無法解釋的現象(失效),你要找出背後的因果機制(缺陷),而科學方法,正是人類發明出來「從現象逆推原因」最強的工具。

把五個步驟落實到一個真實的 bug。假設我們寫了一個算「中位數(median)」的函式:

def median(xs):
    """回傳一串數字的中位數"""
    s = sorted(xs)
    n = len(s)
    return s[n // 2]          # 看起來很合理……真的嗎?

print(median([1, 2, 3, 4]))  # 預期 2.5,實際印出 3

現在,啟動科學除錯的引擎:

  1. 觀察現象:median([1, 2, 3, 4]) 回傳 3,但正確答案應是中間兩數 2 與 3 的平均,即 2.5。
  2. 提出假設:「我猜當串列長度為偶數時,沒有單一的中間元素,而這段程式硬取了 s[n // 2] 一個值,所以錯了。」
  3. 推導預測:如果假設成立,那麼奇數長度應該是對的,只有偶數長度會錯。具體預測:median([1, 2, 3]) 會正確回傳 2。
  4. 設計實驗:跑 median([1, 2, 3]) 與 median([1, 2, 3, 4]),觀察奇偶兩種情況。
  5. 觀察結果:奇數那個確實回傳 2(對),偶數那個回傳 3(錯)。結果符合預測,假設成立!缺陷被定位:偶數長度的處理。

假設一旦被證實,修正就水到渠成:

def median(xs):
    s = sorted(xs)
    n = len(s)
    if n % 2 == 1:                       # 奇數:正中間那一個
        return s[n // 2]
    return (s[n // 2 - 1] + s[n // 2]) / 2   # 偶數:中間兩個的平均
💡
寫下你的「除錯日誌」

科學家做實驗會記錄,你除錯也該記。開一個小筆記,寫下:「假設 1:______;實驗:______;結果:______(成立/推翻)」。 這看起來很慢,卻能救你一命——它防止你重複試同一個沒用的東西(除錯到後來最常發生的鬼打牆), 也讓你在求助別人時能說清楚「我已經排除了 A、B、C」。沒記錄的除錯,是在原地繞圈。

⚠️
一次只改一件事

最常見的除錯陷阱,是「同時改五個地方,然後它好了」——你永遠不會知道是哪一個改動真正修好的, 甚至可能其中四個是新的 bug,只是剛好互相抵消。科學實驗的鐵律是控制變因:一次只改一個變數,看結果,再決定下一步。 除錯時請守住同一條紀律。

重現:沒有穩定的重現,就沒有除錯

科學方法的第一步是「觀察現象」,但前提是現象要能被你穩定地觀察到。一個「偶爾才出現」的 bug, 是工程師的噩夢——因為你改完根本無法確認到底修好了沒。所以除錯真正的第一步,是讓 bug 穩定重現(reproduce)。

最小可重現案例(minimal reproducible example)

當你拿到一個「在三萬行的系統裡、載入某個 50 MB 檔案後、點某個按鈕才會崩潰」的 bug,你不可能在這團迷霧裡除錯。 你要做的是不斷簡化:能不能用更小的輸入觸發?能不能拿掉一半的步驟還是會壞?最終目標是一個 最小可重現案例——一段短到不能再短、卻仍然穩定觸發 bug 的程式與輸入。

這件事的回報高得驚人:

  • 簡化的過程本身,往往就讓你在還沒開始除錯前就看見了缺陷(每拿掉一塊,你都在排除嫌疑)。
  • 最小案例讓「實驗」變得飛快——你不必每次都跑完整個系統。
  • 當你要求助(問同事、發 GitHub issue、貼 Stack Overflow)時,一個最小可重現案例是別人願意幫你的前提。沒人想讀你三萬行的程式。

delta debugging:讓電腦幫你縮小失敗輸入

手動簡化很累。Andreas Zeller 提出了一個漂亮的點子:既然「簡化」本身就是一套機械化的流程,何不讓電腦自動做? 這就是 delta debugging(差量除錯) 的核心,它的演算法叫 ddmin。

想像一個 4000 字元的輸入會讓某個解析器崩潰。ddmin 的思路很像二分搜尋: 把輸入切成幾塊,試著刪掉一塊——如果刪掉後還是會崩潰,代表那塊是無辜的,丟掉它,問題規模就小了一半; 如果刪掉後不崩潰了,代表那塊裡有關鍵,保留它,再往更細的粒度切。如此反覆,最後逼出一個「再刪任何一個字元就不崩潰」的最小輸入。

ℹ️
這個概念,你其實會反覆遇到

「自動縮小失敗輸入」不是學院派的玩具。本章稍後會講的性質測試工具 Hypothesis, 在找到反例後會自動把它「縮小(shrink)」成最簡單的形式;Git 的 git bisect(第 18 章) 則是把同樣的二分思想用在「哪一次提交引入了 bug」上。看到一個概念在三個不同地方出現,你就知道它是真功夫了。

觀察與定位:把缺陷逼到牆角

重現之後,進入「觀察」與「定位」的核心戰場。第 5 章已經給過你基本工具(traceback、print、小黃鴨、pdb), 這裡我們把它們組織成一套「由輕到重」的武器庫,並補上更進階的招式。

觀察:看見程式真正的狀態

除錯的本質是「我以為的狀態」與「實際的狀態」之間有落差。所有觀察工具都在做同一件事:把實際狀態攤開給你看。

  • print / logging:最樸素也最萬用。差別在於 print 是臨時的,用完要刪; logging(日誌)是長久的基礎建設——用 logging.debug(...) 寫下的訊息,平時靜音,出事時把等級調到 DEBUG 就全部現形,而且自帶時間戳、模組名、嚴重等級。生產環境靠 logging,不靠 print。
  • 斷言(assertion):把「我假設這裡一定成立」的條件寫成 assert。它是會自我檢查的觀察點——一旦狀態被感染、違反了你的假設,程式會在感染剛發生的當下就停下來,而不是等它傳播到很遠才失效。斷言把「失效點」往前拉到更靠近「缺陷點」的地方,這正是它珍貴之處。
  • 除錯器(debugger):Python 的 pdb、C/C++ 的 gdb。它讓你設斷點(breakpoint)暫停程式、單步執行、檢視堆疊(stack)看「現在是誰呼叫誰一路走到這裡」、監看(watch)變數看某個值何時被改動。當 bug 牽涉複雜的控制流或狀態變化,除錯器遠勝過洗版式的 print。
import logging
logging.basicConfig(level=logging.DEBUG)   # 把等級開到 DEBUG,debug 訊息才會顯示

def withdraw(balance, amount):
    logging.debug(f"進入 withdraw: balance={balance}, amount={amount}")
    assert amount >= 0, "提款金額不該是負數"        # 斷言:守住一個你深信的前提
    balance -= amount
    assert balance >= 0, "餘額不該變成負數(感染就在這裡發生!)"
    return balance

# 若有人呼叫 withdraw(100, 150),第二個斷言會立刻喊停,
# 把失效點釘在「餘額剛變負」的瞬間,而不是等到很久以後對帳才發現對不上
⚠️
斷言是除錯工具,不是輸入驗證

斷言用來檢查「程式內部不該發生的事」(即你認為永遠為真的不變量),不該用來檢查「外部使用者的輸入」。 原因有二:一是語意不同(輸入錯誤是預期內的,該丟 ValueError 並好好處理;斷言失敗代表「程式有 bug」); 二是技術上,Python 以 -O 最佳化旗標執行時,所有 assert 會被整個移除——你拿它擋輸入,上線就破功了。 輸入驗證請用 if ... raise(第 5 章與第 36 章的防禦性程式設計)。

定位:用二分法與向後推理找到源頭

觀察讓你看見「哪裡的狀態錯了」,定位則是「順著它找到缺陷」。兩個最有威力的策略:

  • 二分定位(bisection):第 5 章講過——在程式中段插一個檢查點,「到這裡為止狀態還對嗎?」對 → bug 在後半;不對 → 在前半。每次砍掉一半嫌疑。把這招用在時間維度上,就是 git bisect(第 18 章):在一堆 commit 裡用二分法,幾步就找出「是哪一次提交把它弄壞的」。對 1000 次提交,只要約 10 次測試就能定位。
  • 向後推理(backward reasoning):這正是對付圖 46-1 那道「失效≠缺陷」鴻溝的利器。從失效點開始問:「這個錯誤的值,是哪一行賦給它的?」找到後再問:「那一行用到的輸入,又是哪裡來的、為什麼是錯的?」一路逆流而上,沿著資料的「來路」回溯,直到抵達第一個「輸入正確、輸出卻錯誤」的地方——那就是缺陷。
💡
隔離變因 & 橡皮鴨:最便宜的兩招

隔離變因:把可疑的程式碼單獨抽出來跑(就是在做最小可重現案例),排除掉「會不會是別的東西干擾」。 橡皮鴨除錯法:把程式一行一行講給一隻橡皮鴨(或任何人)聽。當你被迫把「我以為它在做什麼」說出口, 大腦會自動撞見「咦,它其實沒在做這件事」。這兩招不花一毛錢,卻常常在你還沒動用除錯器前就破案。

測試:你怎麼知道它「真的」對?

除錯是在 bug 出現之後把它揪出來;測試則是主動地、有系統地在 bug 害到使用者之前先逼它現形。 但在學怎麼寫測試之前,我們得先面對一個冷酷的理論事實——這個事實會徹底改變你對測試的期待。

為什麼不能「測完所有情況」

新手常有的幻想是:「我把所有可能的輸入都測一遍,不就證明程式完全正確了嗎?」聽起來合理, 但「所有可能的輸入」這個空間,大到超乎想像。

考慮一個再簡單不過的函式:它吃兩個 32 位元整數,回傳它們的和。輸入的組合有多少種? 每個整數有 $2^{32}$ 種可能,兩個就是 $2^{32} \times 2^{32} = 2^{64}$ 種,大約是 1844 京($1.8 \times 10^{19}$)種組合。 就算你的電腦每秒能測十億次,跑完也要超過 500 年。而這還只是「兩個整數相加」!一個吃字串的函式,輸入空間是無限大的。

既然測不完,測試到底能保證什麼?這就要請出計算機科學史上最有名的一句關於測試的話。

「測試只能證明 bug 的存在,永遠無法證明 bug 不存在。」——艾茲赫爾・戴克斯特拉(Edsger Dijkstra)

這句話我們在第 13 章(離散數學)當作引子用過。它的意思是:測試通過了,只代表「在你試過的這些輸入上沒出錯」, 不代表「在所有輸入上都對」。一個測試失敗,能確定地告訴你「這裡有 bug」;但一萬個測試全過, 也無法保證「沒有 bug」。要真正「證明」程式正確,你需要的是數學上的形式化驗證(formal verification), 那是另一個世界(第 13 章鋪的地基)。對絕大多數軟體而言,測試是務實的折衷:它無法給你確定性,但能給你信心。

🔑
測試的目的是「找出 bug」,不是「證明沒 bug」

這個心態轉換很關鍵。一個「全部通過」的測試套件不該讓你沾沾自喜——它可能只是測得太弱。 一個好的測試,是那種最有可能讓程式露出馬腳的測試。所以寫測試時,別問「我要怎麼證明它對」, 要問「我要怎麼弄垮它」。帶著找碴的心態,你才會去戳那些 bug 真正會躲的角落。

覆蓋率(coverage):衡量「測到多少」

既然測不完,我們至少想知道「測試到底碰到了多少程式碼」。這就是覆蓋率(code coverage)。常見有三種,嚴格程度遞增:

  • 敘述覆蓋率(statement coverage):每一行程式碼是否至少被執行過一次。最寬鬆。
  • 分支覆蓋率(branch coverage):每個 if 的真與假兩條岔路,是否都各走過一次。比敘述嚴格。
  • 路徑覆蓋率(path coverage):每一條可能的執行路徑(分支的所有組合)是否都走過。最嚴格,但通常無法達成——$k$ 個獨立的 if 就有 $2^k$ 條路徑,迴圈更是讓路徑數爆炸成天文數字。

用一個例子看「敘述」與「分支」覆蓋率的差別:

def abs_value(x):
    if x < 0:
        x = -x        # 這一行
    return x

# 只用一個測試案例:
assert abs_value(-3) == 3

只測 abs_value(-3),程式裡每一行都被執行到了——所以敘述覆蓋率 100%! 看起來很完美。但 if 的「假」那條岔路(x >= 0 時跳過)從來沒被走過,所以分支覆蓋率只有 50%。 假如缺陷剛好藏在「正數」這條路上(例如不小心寫成 return -x),這個「100% 敘述覆蓋」的測試完全抓不到。 這就帶出覆蓋率最重要的一課:

⚠️
高覆蓋率 ≠ 沒有 bug

覆蓋率只告訴你「哪些程式碼被執行過」,不告訴你「執行時有沒有好好檢查結果」。 你可以寫一個跑遍每一行、卻一個 assert 都沒有的測試——覆蓋率 100%,抓 bug 能力是 0。 反過來,90% 的覆蓋率,也不代表剩下 10% 沒事(更何況輸入空間根本測不完)。 把覆蓋率當地圖(找出「完全沒測到」的盲區)很好用;把它當 KPI 門檻(規定一定要 95%)就會逼出一堆假裝在測試、其實什麼都沒驗證的爛測試。第 40 章說得好:覆蓋率是好僕人,壞主人。

測試設計:有系統地挑出「該測哪些」

既然不能全測、又想測得聰明,我們需要方法,從浩瀚的輸入空間裡挑出最有代表性、最可能抓到 bug 的少數案例。 這就是測試設計技術。三個經典且實用的:

等價分割(equivalence partitioning)

核心洞見:很多輸入「本質上是同一類」,程式對它們的處理方式相同。既然如此,每一類挑一個代表來測就夠了—— 測 add(2, 3) 對了,你不太需要再測 add(2, 4),因為它們走的是同一條路。 我們把輸入空間切成幾個等價類(equivalence class),每類取一個代表。

舉例:一個 letter_grade(score) 函式,把 0–100 的分數轉成等第(F: 0–59、D: 60–69、C: 70–79、B: 80–89、A: 90–100)。 輸入空間 0–100 有 101 個值,但等價類只有幾個:[0,59]→F、[60,69]→D、[70,79]→C、 [80,89]→B、[90,100]→A,外加兩個不合法的等價類:< 0 與 > 100。 每類挑一個代表(例如 30、65、75、85、95、−10、150),七個案例就涵蓋了所有「行為類別」。

邊界值分析(boundary value analysis)

等價分割告訴你「類別之間」,而 bug 最愛躲在類別的交界處——也就是邊界。這就是第 1、5 章一再強調的 邊界情況(edge case)。「差一錯誤(off-by-one)」幾乎都發生在這裡:到底 >= 還是 >? 59 算 F 還是 D?所以,除了每類取代表,更要專門測每個邊界的兩側。

對 letter_grade,關鍵邊界與它們的兩側是:

決策表(decision table):多條件的組合

當輸出取決於多個條件的組合時,光靠分區還不夠,容易漏掉某些組合。決策表把所有條件的真假組合列成一張表, 確保每一種組合都被考慮到。例如一個「是否核發貸款」的規則,取決於兩個條件:信用良好嗎?有穩定收入嗎?

案例信用良好?收入穩定?預期結果
1是是核准
2是否婉拒
3否是婉拒
4否否婉拒

兩個布林條件就有 $2^2 = 4$ 種組合。決策表逼你把每一格都想清楚並各寫一個測試,避免「只測了核准跟一種婉拒,漏掉另一種婉拒」這種常見疏漏。 條件越多,決策表越能保護你不漏組合(雖然 $n$ 個條件會有 $2^n$ 列,這時又得回頭用等價的概念合併不影響結果的組合)。

測試的層次與類型

測試不只一種,它們在不同的層級運作,各有任務。第 40 章(軟體工程)已經完整講過測試金字塔(testing pyramid), 這裡快速複習,讓你把它和本章的理論接起來:

  • 單元測試(unit test):測一小塊——通常是單一函式或類別,並把它和外界(資料庫、網路、檔案)隔離。又快又準,壞了立刻知道是哪裡。這是本章程式範例的主角。
  • 整合測試(integration test):測「幾塊湊在一起」能不能正確合作,例如「你的程式 + 真的資料庫」。
  • 系統測試 / 端對端(E2E):把整個系統當黑盒子,模擬真人從頭到尾跑完一條完整流程(如登入→下單→付款)。
  • 驗收測試(acceptance test):站在使用者/客戶角度,驗證「這是不是他們真正要的東西」。

還有兩個跨層級、按「目的」分類的重要類型,你一定會反覆聽到:

  • 回歸測試(regression test):每修好一個 bug,就為它補一個測試,確保這個老毛病永遠不會再復發。回歸測試套件是你重構與改版時的安全網——它讓你敢動程式,因為一改壞就立刻有紅燈。
  • 冒煙測試(smoke test):名字來自硬體界——「通電後會不會冒煙?」。它是一組極快、極淺的測試,只驗證「系統最基本的功能有沒有整個掛掉」(例如服務能不能啟動、首頁打不打得開)。改完先跑冒煙測試,過了再跑完整套件,免得在一個已經根本壞掉的版本上浪費時間。

自動化測試:用 pytest 把測試變成日常

理論談夠了,來寫真的測試。第 5 章介紹過 pytest 的入門(檔名與函式都以 test_ 開頭、直接用 assert); 這裡我們補上兩個讓測試從「能用」變「專業」的利器:parametrize 與 fixtures。

parametrize:一個測試,餵很多組資料

前面用等價分割與邊界值挑出了一堆案例。如果每個案例都複製貼上寫一個 def test_...,既囉嗦又難維護。 @pytest.mark.parametrize 讓你用一個測試函式,自動跑過多組輸入與預期輸出: 

# test_grade.py
import pytest
from grade import letter_grade

@pytest.mark.parametrize("score, expected", [
    (30,  "F"),   # F 區代表
    (59,  "F"),   # ★ 邊界:59 仍是 F
    (60,  "D"),   # ★ 邊界:60 跳到 D
    (75,  "C"),   # C 區代表
    (89,  "B"),   # ★ 邊界:89 仍是 B
    (90,  "A"),   # ★ 邊界:90 跳到 A
    (100, "A"),   # 範圍端點
])
def test_letter_grade(score, expected):
    assert letter_grade(score) == expected

# 不合法輸入應該要報錯,單獨測:
@pytest.mark.parametrize("bad_score", [-1, 101, 150])
def test_letter_grade_rejects_out_of_range(bad_score):
    with pytest.raises(ValueError):
        letter_grade(bad_score)

在終端機跑 pytest,它會把上面每一組參數當成一個獨立的測試來執行並回報。哪一組掛了, pytest 會清楚告訴你是哪組參數(例如 test_letter_grade[59-F]),定位超快。 你會發現,測試設計技術(等價分割+邊界值)與 parametrize,是天作之合。

fixtures:準備與清理測試所需的環境

很多測試需要先「準備好某個東西」才能測——一個資料庫連線、一份範例資料、一個暫存檔。 如果每個測試都自己準備、自己善後,程式碼會重複又容易出錯。fixture(測試夾具)把這件事抽出來, 測試只要把 fixture 的名字寫進參數,pytest 就會自動幫它準備好: 

import pytest

@pytest.fixture
def sample_accounts():
    """提供一份乾淨的範例帳戶資料給每個用到它的測試"""
    return {"alice": 100, "bob": 50}

def test_transfer_ok(sample_accounts):       # 把 fixture 名字寫進參數即可
    transfer(sample_accounts, "alice", "bob", 30)
    assert sample_accounts["alice"] == 70
    assert sample_accounts["bob"] == 80

def test_transfer_insufficient(sample_accounts):
    with pytest.raises(ValueError):          # 餘額不足應該報錯
        transfer(sample_accounts, "bob", "alice", 999)

關鍵好處:每個測試拿到的都是全新、互不干擾的 sample_accounts。 測試之間若會互相影響(一個測試改了資料,害下一個測試掛掉),是測試套件最惡名昭彰的問題之一。 fixture 幫你守住「每個測試都從乾淨狀態出發」這條紀律。(進階的 fixture 還能用 yield 做「測試後自動清理」,例如關閉連線、刪除暫存檔。)

TDD:先寫測試,再寫程式

測試驅動開發(Test-Driven Development, TDD)把順序顛倒過來:先寫一個會失敗的測試,再寫剛好能讓它通過的程式。 它的節奏是一個三拍循環:

用一個經典範例完整走一遍:開發一個 to_roman(n),把整數轉成羅馬數字(1→"I"、4→"IV"、9→"IX"…)。

第一圈 🔴→🟢:先寫一個最簡單的失敗測試。此刻 to_roman 還不存在,測試當然失敗(紅燈)。

# test_roman.py
from roman import to_roman

def test_simple_ones():
    assert to_roman(1) == "I"
    assert to_roman(2) == "II"
    assert to_roman(3) == "III"

寫剛好能讓它過的程式——別想太多,最笨的寫法也行(綠燈):

# roman.py
def to_roman(n):
    return "I" * n          # 笨,但 1,2,3 確實過了

第二圈 🔴→🟢:加一個新測試,逼程式長大。"I" * 4 會得到 "IIII",但羅馬數字 4 是 "IV"——紅燈。

def test_subtractive():
    assert to_roman(4) == "IV"
    assert to_roman(5) == "V"
    assert to_roman(9) == "IX"

修改程式讓新舊測試全部通過。這時自然會想到「用一張對照表,從大到小不斷扣」:

def to_roman(n):
    table = [(10, "X"), (9, "IX"), (5, "V"), (4, "IV"), (1, "I")]
    result = []
    for value, symbol in table:
        while n >= value:
            result.append(symbol)
            n -= value
    return "".join(result)

第三圈 🔴→🟢→🔵:再加更難的測試,逼出完整的表。

def test_big_numbers():
    assert to_roman(40) == "XL"
    assert to_roman(90) == "XC"
    assert to_roman(1994) == "MCMXCIV"

程式結構已經對了,只要把對照表補完整——這一步既是讓測試變綠,也是重構(藍燈):結構不變,只擴充資料。

def to_roman(n):
    table = [
        (1000, "M"), (900, "CM"), (500, "D"), (400, "CD"),
        (100, "C"), (90, "XC"), (50, "L"), (40, "XL"),
        (10, "X"), (9, "IX"), (5, "V"), (4, "IV"), (1, "I"),
    ]
    result = []
    for value, symbol in table:
        while n >= value:
            result.append(symbol)
            n -= value
    return "".join(result)

全部測試通過。注意這趟旅程帶來的三個禮物:

🔑
TDD 給你的,遠不只是測試

一、它逼你先想清楚規格:寫測試前,你必須先回答「怎樣才算對」——這把模糊的需求變成精確的例子。 二、它讓你永遠有安全網:每一步都在綠燈保護下進行,重構時一旦改壞,立刻紅燈。 三、你最後免費得到一整套測試:它們不是事後補的功課,而是開發的副產品。 就算你不嚴格遵守 TDD,「動手前先想一個具體的測試案例」這個習慣本身,就價值連城。

進階測試法:比手寫案例更強的三招

到目前為止,我們都在「手動挑案例」。但人腦想得到的案例,恰恰是 bug 最不會躲的地方(因為你想得到,你就會處理它)。 bug 真正躲的,是你沒想到的角落。下面三招,都是用來攻擊「你沒想到」的。

性質測試(property-based testing):驗證「性質」而非「例子」

手寫測試是「給定輸入 X,輸出應該是 Y」;性質測試則更高一層:「無論輸入是什麼,輸出都應該滿足某個性質」。 然後讓工具自動生成成千上百個隨機輸入去攻擊這個性質。Python 的代表工具是 Hypothesis。

經典例子:測試排序。你不必想「[3,1,2] 應該變 [1,2,3]」這種具體案例,而是宣告排序必須滿足的性質: 

from hypothesis import given, strategies as st
from collections import Counter

def my_sort(xs):
    return sorted(xs)

@given(st.lists(st.integers()))      # Hypothesis 會自動生成各種整數串列
def test_length_preserved(xs):
    assert len(my_sort(xs)) == len(xs)             # 性質一:長度不變

@given(st.lists(st.integers()))
def test_is_ordered(xs):
    result = my_sort(xs)
    for i in range(len(result) - 1):
        assert result[i] <= result[i + 1]          # 性質二:相鄰元素遞增

@given(st.lists(st.integers()))
def test_is_permutation(xs):
    assert Counter(my_sort(xs)) == Counter(xs)     # 性質三:就是原本那堆數(沒憑空多/少)

這三個性質合起來,幾乎就是「正確排序」的定義。Hypothesis 會丟空串列、單元素、巨大串列、全相同、含負數、 含重複……各種你懶得手寫的刁鑽案例。一旦找到反例,它還會自動把反例縮小(shrink)到最簡形式—— 這正是前面 delta debugging 的精神!來看它抓 bug 的威力: 

def lower_first(s):
    return s[0].lower() + s[1:]     # 把字串首字母轉小寫……有個沒想到的洞

@given(st.text())                    # 自動生成各種字串
def test_lower_first_no_crash(s):
    lower_first(s)                   # 性質:對任何字串都不該崩潰

# Hypothesis 跑沒幾下就回報:
# Falsifying example: test_lower_first_no_crash(s='')
# —— 空字串!s[0] 會 IndexError。而且它「縮小」到了最簡單的反例:''

你大概永遠不會手動想到去測「空字串」——但 Hypothesis 會,而且每次都會。這就是性質測試的價值: 它幫你探索你想不到的輸入空間。

模糊測試(fuzzing):用混亂的輸入找崩潰

模糊測試(fuzzing)是性質測試的近親,但目標更純粹、更暴力:持續餵給程式大量隨機或變異(mutated)的輸入, 看它會不會崩潰、當掉、或洩漏記憶體。它通常只驗證一個最低限度的性質——「不可以崩潰」。

現代的模糊器(如 AFL++、libFuzzer,Python 有 Atheris)更聰明:它們是 覆蓋率導向(coverage-guided)的——會觀察哪些輸入觸發了新的程式路徑,然後優先在那些輸入上做變異, 像演化一樣逐步鑽進程式更深的角落。這讓 fuzzing 在挖掘記憶體安全漏洞(緩衝區溢位、越界讀寫)上效果驚人。

🛡️
fuzzing 在資安裡的角色

我們在第 36 章(安全程式設計)看過:fuzzing 的精神,正是「永遠假設輸入是惡意的」這句資安鐵律的自動化版本。 無數 C/C++ 函式庫的重大漏洞(包括著名的 Heartbleed 類問題)都是 fuzzing 挖出來的;Google 的 OSS-Fuzz 專案 至今已為開源軟體找出數萬個 bug。當你的程式會接觸不可信的外部輸入(網路封包、檔案格式、使用者上傳),fuzzing 幾乎是必備的防線。

變異測試(mutation testing):測試你的測試

前面我們一直在用測試檢查程式。但有個更深的問題:誰來檢查你的測試?一套覆蓋率 100% 的測試, 可能根本沒在認真驗證(還記得那個一個 assert 都沒有的反例嗎)。變異測試(mutation testing)就是來回答這個問題的。

它的點子既邪惡又天才:故意把你的程式改壞,製造出一堆「變異體(mutant)」——把 > 改成 >=、 把 + 改成 -、把 return x 改成 return None……然後對每一個變異體跑你的測試套件。 邏輯是:如果你的測試夠好,它應該要因為這個故意製造的錯而失敗(我們說這個變異體被「殺死(killed)」了)。 如果改壞了程式、測試卻還是全部通過(變異體「存活」了),那代表——你的測試有盲區,沒能抓到這種錯。 

# 原始程式
def is_adult(age):
    return age >= 18

# 變異體:把 >= 改成 >
def is_adult_mutant(age):
    return age > 18

# 如果你的測試只有 assert is_adult(20) == True、assert is_adult(10) == False,
# 那這個變異體會「存活」——因為你沒測到 age == 18 這個邊界!
# 補上 assert is_adult(18) == True,就能「殺死」這個變異體。

變異分數(mutation score)= 被殺死的變異體 / 變異體總數,它衡量的是測試的「強度」,遠比覆蓋率誠實。 Python 有 mutmut、cosmic-ray 等工具。注意上面那個例子——存活的變異體精準地指出「你漏測了 18 這個邊界值」, 這跟我們前面講的邊界值分析完美呼應:變異測試會自動幫你發現你漏掉的邊界。

測試替身:當程式有「外部相依」

單元測試要「快、穩、隔離」,但真實程式常常相依於外部世界:它會寄 email、打 API、讀資料庫、看現在幾點。 這些東西慢(網路要等)、不穩(API 可能掛)、有副作用(你不想每跑一次測試就真的寄一封信)。 解法是用測試替身(test double)——一個假的、可控的東西,在測試時頂替真貨。常見三種:

替身它是什麼典型用途
stub(樁)回傳事先寫死的答案讓被測程式拿到一個可預測的回應,例如「假裝 API 回傳了這筆資料」
mock(模擬)會記錄自己被怎麼呼叫的假物件驗證「互動」:被測程式有沒有正確地呼叫它(呼叫幾次、傳什麼參數)
fake(偽件)有真的、但簡化的實作例如用「記憶體裡的字典」當作假資料庫,行為像真的但不落地

用 Python 內建的 unittest.mock 看一個 mock 的例子。假設 notify_user 會透過一個「寄信器」寄信, 我們不想在測試裡真的寄信,只想驗證「它有沒有用正確的地址與內容去呼叫寄信器」: 

from unittest.mock import Mock

def notify_user(user_id, mailer):
    address = f"user{user_id}@example.com"
    mailer.send(address, "歡迎加入!")        # 真實情況下這會寄出一封信
    return True

def test_notify_user_sends_correct_email():
    fake_mailer = Mock()                       # 一個假的寄信器,不會真的寄信
    notify_user(42, fake_mailer)
    # 驗證「互動」:send 是否被呼叫了一次,且參數正確?
    fake_mailer.send.assert_called_once_with("user42@example.com", "歡迎加入!")
⚠️
別 mock 上癮

測試替身很方便,但用過頭會把測試變成「測試你對實作的假設」而不是「測試行為」。如果你 mock 了一切, 測試可能全綠,程式卻在真實環境整個壞掉(因為你 mock 的行為跟真貨不一樣)。原則: mock 那些慢的、不穩的、有副作用的邊界(網路、資料庫、時間),核心邏輯則用真貨直接測。 需要大量 mock 才測得動,常常是一個訊號——你的程式相依糾纏太深,該重新設計了。這把我們帶到最後一個主題。

心態:bug 是老師,可測試性是好設計的影子

技術講完了,但決定你能走多遠的,是心態。最後送你三個觀念,它們會貫穿你整個工程生涯。

第一,bug 是學習機會,不是羞恥。第 5 章說過:連最頂尖的工程師,寫程式的時間有一大半花在除錯上。 每一個你親手抓到的 bug,都在你腦中刻下一條「下次別再這樣」的神經迴路。會寫程式的人和很會寫程式的人, 差別不在「誰寫的 bug 少」(都很多),而在「誰從每個 bug 學到更多、誰抓 bug 的系統更強」。 把除錯當成查案的樂趣,而不是失敗的懲罰。

第二,防禦性程式設計(defensive programming)。第 5、36 章埋下的種子在這裡長成習慣: 在函式入口驗證輸入、用斷言守住內部不變量、對「不該發生」的情況明確報錯而不是默默吞掉。 防禦性程式設計的精髓,是讓 bug 盡早、盡可能大聲地失效——一個在感染當下就崩潰的程式, 遠比一個帶著錯誤狀態默默跑下去、最後給你一個無法解釋的錯誤結果的程式,好除錯一百倍。

🔑
可測試性,是好設計的副產品

這是整章最值得帶走的一句話。當你發現某段程式很難寫測試——要 mock 一大堆東西、要先建好複雜的環境、 一個函式做了太多事不知從何測起——這幾乎總是在告訴你:這段程式的設計有問題。 它可能耦合太緊、職責太雜、相依太多。反過來,當你帶著「這要怎麼測?」的問題去寫程式,你會自然地寫出 職責單一、相依清楚、輸入輸出分明的函式——也就是好設計。所以「為了測試而設計」不是額外負擔, 它是一面鏡子,照出你設計的好壞。難測的程式,通常就是壞的程式。

動手練習

看懂和會做是兩回事。下面六題由淺入深,涵蓋本章的核心技能。請務必先自己動手,卡住了再看解答。

✏️ 練習 46.1 除錯

下面這個 count_vowels 想數出字串裡的母音數量,但它少數了一些。 請用科學除錯法(觀察 → 假設 → 預測 → 實驗 → 修正)定位並修好它。寫下你的假設與驗證它的實驗。

def count_vowels(s):
    vowels = "aeiou"
    count = 0
    for ch in s:
        if ch in vowels:
            count += 1
    return count

print(count_vowels("Hello"))      # 預期 2,實際 1
print(count_vowels("AEIOU"))      # 預期 5,實際 0
看解答

觀察:"Hello" 應有 e、o 兩個母音卻只數到 1;"AEIOU" 全是母音卻數到 0。

假設:兩個失敗案例的共通點是「大寫母音沒被算到」。我猜 vowels 只放了小寫, 而 "Hello" 的 H…等比對只認小寫,大寫的 E(在 AEIOU 裡)完全沒被認出。

預測:若假設成立,則「全小寫母音」的輸入應該正確。預測 count_vowels("aeiou") == 5(正確), 而任何含大寫母音的都會少算。

實驗:跑 count_vowels("aeiou") → 得到 5,正確!對照 "AEIOU" → 0,錯誤。假設成立。

修正:比對前先把字元轉小寫(或把大寫也放進 vowels)。

def count_vowels(s):
    vowels = "aeiou"
    count = 0
    for ch in s.lower():       # 統一轉小寫再比對
        if ch in vowels:
            count += 1
    return count

assert count_vowels("Hello") == 2
assert count_vowels("AEIOU") == 5
assert count_vowels("xyz") == 0      # 別忘了補一個邊界:沒有母音
print("通過 ✅")

重點不在這幾行修正,而在你先形成「是大小寫問題」的假設、再用一個實驗確認的過程——而不是亂猜亂改。

✏️ 練習 46.2 測試設計

有一個 ticket_price(age) 票價函式,規則如下:0–2 歲免費(0 元)、3–12 歲兒童票(80 元)、 13–64 歲全票(150 元)、65 歲以上敬老票(60 元);年齡為負數或非整數應視為不合法。 請不看實作,只憑這段規格,用等價分割 + 邊界值分析設計一組測試案例。你能列出哪些等價類?哪些邊界?

看解答

等價類(各取一代表):不合法負數(如 −1)、嬰兒 [0,2](如 1)、兒童 [3,12](如 8)、 全票 [13,64](如 30)、敬老 [65,∞)(如 70)。

邊界值(每個交界的兩側,最容易 off-by-one):−1/0、2/3、12/13、64/65。 這些正是「到底 12 算兒童還是全票」這種最容易寫錯的地方。

import pytest
from tickets import ticket_price

@pytest.mark.parametrize("age, price", [
    (0,   0),     # ★ 邊界:免費區下端
    (1,   0),     # 嬰兒區代表
    (2,   0),     # ★ 邊界:免費區上端
    (3,   80),    # ★ 邊界:跳到兒童票
    (8,   80),    # 兒童區代表
    (12,  80),    # ★ 邊界:兒童票上端
    (13,  150),   # ★ 邊界:跳到全票
    (30,  150),   # 全票區代表
    (64,  150),   # ★ 邊界:全票上端
    (65,  60),    # ★ 邊界:跳到敬老票
    (70,  60),    # 敬老區代表
])
def test_ticket_price(age, price):
    assert ticket_price(age) == price

@pytest.mark.parametrize("bad_age", [-1, -100, 3.5, "abc"])
def test_ticket_price_invalid(bad_age):
    with pytest.raises((ValueError, TypeError)):
        ticket_price(bad_age)

注意:11 個正常案例裡有 8 個是邊界——這正確反映了「bug 最愛躲在邊界」。能列出這份清單, 代表你已經把規格徹底想透了,這比實作本身更有價值。

✏️ 練習 46.3 實作

def is_palindrome(s):
    s = s.lower()
    return s == s[::-1]    # s[::-1] 是反轉字串
看解答

from palindrome import is_palindrome

@pytest.mark.parametrize("text, expected", [

✏️ 練習 46.4 TDD

用 TDD(先寫測試,再寫程式)開發一個 run_length_encode(s):把字串做「行程長度編碼」, 例如 "aaabbc" → "a3b2c1"、"x" → "x1"、"" → ""。 請先寫出會失敗的測試(紅),再寫出剛好能通過的實作(綠)。把你的紅綠步驟寫下來。

看解答

🔴 紅:先把「對」定義成測試。此時函式還不存在,全部失敗。

# test_rle.py
import pytest
from rle import run_length_encode

@pytest.mark.parametrize("text, expected", [
    ("",       ""),          # 邊界:空字串
    ("x",      "x1"),        # 邊界:單一字元
    ("aaabbc", "a3b2c1"),    # 一般情況
    ("abc",    "a1b1c1"),    # 沒有重複
    ("aaaa",   "a4"),        # 全相同
])
def test_run_length_encode(text, expected):
    assert run_length_encode(text) == expected

🟢 綠:寫剛好能讓全部通過的實作。逐字掃描,累計連續相同字元的長度。

# rle.py
def run_length_encode(s):
    if not s:                    # 邊界:空字串直接回傳空
        return ""
    result = []
    prev = s[0]
    count = 1
    for ch in s[1:]:
        if ch == prev:
            count += 1           # 還是同一個字元,計數加一
        else:
            result.append(f"{prev}{count}")   # 換字元了,把上一段寫出來
            prev = ch
            count = 1
    result.append(f"{prev}{count}")           # 別忘了最後一段!
    return "".join(result)

🔵 重構:這個版本已經乾淨,可不動。重點是體會:因為測試先寫好, 你寫實作時心裡清楚知道「最後一段別漏掉」「空字串要處理」——這兩個正是此題最容易出的 bug, 而你的測試早就替你守住了。這就是 TDD 的威力。

✏️ 練習 46.5 挑戰

用 Hypothesis 性質測試驗證 Python 的 list(reversed(...))。請寫出至少兩個「性質」: (1) 反轉兩次會回到原本;(2) 反轉後長度不變。然後思考:你能否設計一個性質,專門抓出一個「故意寫錯」的反轉實作?

看解答 
from hypothesis import given, strategies as st

def my_reverse(xs):
    return list(reversed(xs))

@given(st.lists(st.integers()))
def test_reverse_twice_is_identity(xs):
    assert my_reverse(my_reverse(xs)) == xs       # 性質一:反轉兩次 = 原本

@given(st.lists(st.integers()))
def test_reverse_preserves_length(xs):
    assert len(my_reverse(xs)) == len(xs)          # 性質二:長度不變

@given(st.lists(st.integers()))
def test_reverse_first_becomes_last(xs):
    if xs:                                          # 非空時
        assert my_reverse(xs)[0] == xs[-1]          # 性質三:原本最後一個變成第一個

抓錯的能力:假設有人把實作寫成 return xs(根本沒反轉)。性質一、二都還是會通過 (反轉兩次=原本 ✓、長度不變 ✓)——這告訴我們「性質不夠強」!但性質三會立刻抓到它: 對 [1, 2],錯誤實作的 [0] 是 1,而 xs[-1] 是 2,不相等,測試失敗。

教訓:性質測試的威力,取決於你的性質寫得夠不夠刻畫本質。 這也呼應變異測試的精神——好的測試,要能殺死「偷懶的錯誤實作」。 最強的性質往往是「跟一個已知正確的參考實作比對」:assert my_reverse(xs) == xs[::-1]。

✏️ 練習 46.6 挑戰

實作一個簡化版的 delta debugging。給定一個「失敗判定函式」fails(s) (回傳 True 代表這個輸入會觸發 bug),寫一個 minimize(s, fails), 把一個會失敗的字串縮小成「再刪任何一塊就不再失敗」的最小輸入。用「含有子字串 "bug" 就失敗」來測試你的縮小器。

看解答

核心策略(圖 46-3):把字串切成 n 塊,試著刪掉每一塊;若刪掉後還是失敗,就採用這個更短的版本。 刪不動時,把粒度切得更細(n 加倍),直到無法再縮。

def minimize(s, fails):
    """把會讓 fails(s)==True 的輸入縮到最小"""
    assert fails(s), "初始輸入就必須是會失敗的"
    n = 2
    while len(s) >= 2:
        chunk = len(s) // n
        reduced = False
        # 試著刪掉每一塊
        for i in range(0, len(s), chunk):
            candidate = s[:i] + s[i + chunk:]      # 刪掉第 i 塊
            if candidate and fails(candidate):
                s = candidate                      # 刪了還是失敗 → 採用更短的
                n = max(n - 1, 2)
                reduced = True
                break
        if not reduced:
            if n >= len(s):                         # 已經細到逐字元都刪不動
                break
            n = min(n * 2, len(s))                  # 切更細再試
    return s

# 測試:含 "bug" 就算失敗
def fails(s):
    return "bug" in s

original = "the quick brown bug jumps over"
print(minimize(original, fails))   # 縮成最小,結果是 "bug"

跑起來,一段長句子會被自動縮成核心的 "bug" 三個字元——再刪任何一個都不再含 "bug",也就不再失敗。 這正是 Hypothesis「shrinking」與 git bisect 背後的同一個思想:用機械化的二分式刪減, 自動逼出「觸發 bug 的最小條件」。你親手體會了一次,以後再看到這個概念就不陌生了。

把這一章串起來

我們從一隻 1947 年的飛蛾出發,把「除蟲」變成了一門有理論、有方法、可累積的科學。回顧這趟旅程的路標:

  • bug 有三層:缺陷(寫錯的程式碼)→ 感染(錯誤的狀態)→ 失效(看得見的錯誤)。關鍵領悟:失效點往往不是缺陷點,所以要往回走,不要盯著現場。
  • 科學除錯法:觀察 → 假設 → 預測 → 實驗 → 修正。用假設驅動、記錄你試過什麼、一次只改一件事。
  • 重現是第一步:最小可重現案例與 delta debugging,把巨大的失敗自動縮成最小核心。
  • 觀察與定位:print/logging、斷言、除錯器看狀態;二分定位(含 git bisect)與向後推理找源頭。
  • 測試的理論:輸入空間爆炸 → 測不完;Dijkstra——測試只能證明 bug 存在,不能證明不存在;覆蓋率是地圖不是 KPI,高覆蓋 ≠ 無 bug。
  • 測試設計:等價分割挑代表、邊界值分析攻交界、決策表覆蓋組合。
  • 自動化:用 pytest 的 parametrize 與 fixtures 寫專業測試;TDD 紅綠重構讓測試先行、設計變好。
  • 進階三招:性質測試(Hypothesis)驗證性質、模糊測試暴力找崩潰、變異測試「測試你的測試」。
  • 測試替身:stub/mock/fake 隔離外部相依——但別 mock 上癮。
  • 心態:bug 是老師不是羞恥;防禦性程式設計讓 bug 盡早大聲失效;可測試性是好設計的影子——難測的程式通常就是壞的設計。

第 5 章我們把除錯與測試當「入門技能」教了一遍;這一章,我們把它們升級成「進階理論」。 從現在起,當你寫下任何一個函式,腦中應該同時響起兩個問題:「它會在哪裡壞?」與「我怎麼證明它真的會動?」—— 能持續、嚴肅地回答這兩個問題的人,才是真正可靠的工程師。
