計算機圖學

從一個遊戲畫面到一部皮克斯電影,螢幕上每一個亮起的像素,背後都是線性代數與幾何在飛速運算。這一章,我們把「把 3D 世界變成 2D 影像」這件魔法,一層一層拆給你看——而且最後你會親手寫出一個能渲染出一顆發亮球體的光線追蹤器。

一切都是把 3D 算成 2D

你正盯著的這塊螢幕,本質上是一面由幾百萬顆小燈組成的網格。它本身完全不知道什麼叫「3D 世界」、什麼叫「一隻在森林裡奔跑的龍」。 它只會做一件事:聽從指令,讓每一顆小燈亮成某個顏色。所以,當你在遊戲裡轉動視角、看見遠山近樹隨之移動、 陽光在水面閃爍——這一切的「立體感」,其實是一場精心計算的錯覺: 程式在每一個瞬間,把一個用數字描述的三維場景,算成一張二維的顏色網格,送去點亮那些小燈。

計算機圖學(computer graphics)研究的,正是這場錯覺背後的數學與演算法: 如何用向量描述空間中的點、用矩陣搬動與旋轉它們、用幾何決定哪個物體擋住哪個物體、 用光照模型算出表面該有多亮——最後,把這一切壓扁成一張圖。一句話: 圖學就是「把幾何與光,轉換成像素」的科學。

要完成這個轉換,業界長期演化出兩條主要路線,它們的世界觀幾乎完全相反,值得你一開始就分清楚。

兩大路線:光柵化 vs 光線追蹤

光柵化(rasterization)的思路是「從幾何出發」:我手上有一堆三角形(構成龍的身體), 我把每個三角形投影到螢幕上,算出它蓋住了哪些像素,再幫那些像素塗上顏色。它的提問是: 「這個三角形,落在哪些像素裡?」這條路極快,而且現代 GPU 為它量身打造, 所以它是所有即時(real-time)應用——尤其是遊戲——的主力。

光線追蹤(ray tracing)的思路剛好倒過來,是「從眼睛出發」:對螢幕上的每一個像素, 我從攝影機射出一條光線,問:「這條光線,飛出去會打到場景裡的什麼東西?」打到什麼,那個像素就顯示什麼顏色。 因為它模擬的是光真實的傳播,反射、折射、陰影、柔和的間接光都能自然地浮現,畫面逼真到能以假亂真。 代價是慢——所以它長年是離線(offline)渲染(電影特效、動畫)的王者, 直到近年 GPU 硬體加速,才開始走進即時領域。

🔑
一句話分清兩條路

光柵化問的是「三角形落在哪些像素」(物體 → 像素),光線追蹤問的是「這個像素看到什麼」(像素 → 物體)。 前者快、適合即時互動(遊戲);後者真實、適合離線追求畫質(電影)。 這個「正向投影 vs 反向追光」的對立,是理解整個圖學版圖最重要的一條分界線。

影像是什麼?像素、色彩與影格緩衝

在談 3D 之前,我們得先弄懂最終產物——影像本身。一張數位影像,說穿了就是一個二維的顏色網格。 網格裡的每一個小格子,叫像素(pixel,picture element 的縮寫)。 所謂解析度(resolution),就是這個網格有多少行、多少列:1920 × 1080(俗稱 1080p) 代表寬 1920 顆、高 1080 顆像素,總共約 200 萬顆。4K 螢幕則有約 830 萬顆。 圖學的最終任務,就是替這幾百萬顆像素,各自決定一個顏色。

色彩:用三個數字描述一道光

那「顏色」又怎麼用數字表示?最常見的是 RGB:用紅(Red)、綠(Green)、藍(Blue) 三道光的強度去混合出任何顏色——這是模仿人眼視網膜裡三種錐狀細胞的設計。 每個分量通常用 8 個位元(一個 byte)表示,範圍 0–255。所以一個像素 = 三個 0–255 的數字: (255, 0, 0) 是純紅、(255, 255, 255) 是白、(0, 0, 0) 是黑、 (255, 200, 0) 是橘黃。每分量 8 位元、共 24 位元,叫 24-bit 色深(color depth), 能表現 $2^{24}\approx 1670$ 萬種顏色——這就是「全彩」的由來。

有時還會加上第四個分量 alpha,表示不透明度:alpha = 255 完全不透明,alpha = 0 完全透明。 它讓我們能把多張圖層「疊」起來(例如把半透明的玻璃、煙霧、UI 疊在場景上), 這個疊合過程叫 alpha 混合(alpha blending):$C = \alpha\,C_{\text{前}} + (1-\alpha)\,C_{\text{後}}$。

動手:產生你的第一張圖

說了這麼多,不如親手生一張圖。最樸素的圖檔格式之一是 PPM,它幾乎沒有任何壓縮或花俏結構—— 就是一個文字檔,開頭寫尺寸,接著把每個像素的 R G B 一個個列出來。 因為它太簡單,所以是學圖學時「親手寫出一張圖」的最佳起點。我們來畫一張橫紅、縱綠的漸層: 

# 用純 Python 產生一張 PPM 漸層圖,不需任何函式庫
W, H = 256, 256
with open("gradient.ppm", "w") as f:
    f.write(f"P3\n{W} {H}\n255\n")   # P3 = 純文字 RGB;接著寬高;最大值 255
    for y in range(H):
        for x in range(W):
            r = x          # 左到右:紅 0 → 255
            g = y          # 上到下:綠 0 → 255
            b = 64         # 藍固定一個低值
            f.write(f"{r} {g} {b}\n")
# 用任何看圖軟體(或線上 PPM viewer)打開 gradient.ppm,就能看到一片彩色漸層

如果你裝了 Pillow(PIL),可以更方便地產生 PNG,並逐像素設定顏色。 注意內層那兩個 for 迴圈——它正是「對每個像素決定一個顏色」這個圖學核心動作最赤裸的樣子: 

from PIL import Image

img = Image.new("RGB", (256, 256))
px = img.load()
for y in range(256):
    for x in range(256):
        px[x, y] = (x, y, 64)        # 同樣的漸層
img.save("gradient.png")

恭喜——你已經寫了一個最迷你的「渲染器」。本章最後那個光線追蹤器,結構其實和這兩個迴圈一模一樣, 只是「決定顏色」那一行,會從「(x, y, 64)」升級成「射一條光線、求交、算光照」。

2D 圖形與座標:向量是基本語言

要描述「空間裡的東西在哪、往哪」,圖學用的語言是向量(vector)。 這是整個第 54 章線性代數(linear algebra)的主角,這裡我們先建立夠用的幾何直覺 (之後讀第 54 章會更紮實)。一個點(point)就是空間裡的一個位置,例如 2D 平面上的 $(3, 2)$; 一個向量則是一個「位移」——一個有大小、有方向的箭頭,例如「往右 3、往上 2」,寫成 $\mathbf{v} = (3, 2)$。 點與向量數值長得一樣,但意義不同:點是「位置」,向量是「從哪到哪的箭頭」。

向量的四個基本操作

圖學裡幾乎所有計算,都建立在四個基本向量操作上。我們逐一用幾何直覺理解:

  • 加法 / 減法:$\mathbf{a} + \mathbf{b}$ 是把兩個箭頭「首尾相接」走完的總位移; $\mathbf{b} - \mathbf{a}$ 則是「從 $\mathbf{a}$ 指向 $\mathbf{b}$」的箭頭(終點減起點)。
  • 長度(length / magnitude):$\lVert\mathbf{a}\rVert = \sqrt{a_x^2 + a_y^2}$,就是畢氏定理量箭頭多長。
  • 正規化(normalize):把向量除以自己的長度,$\hat{\mathbf{a}} = \mathbf{a} / \lVert\mathbf{a}\rVert$, 得到一個長度為 1、只保留方向的單位向量(unit vector)。「方向」在圖學裡無所不在(光的方向、表面朝向),通常都要先正規化。
  • 內積(dot product):$\mathbf{a}\cdot\mathbf{b} = a_x b_x + a_y b_y = \lVert\mathbf{a}\rVert\,\lVert\mathbf{b}\rVert\cos\theta$。 它把兩個向量「吃進去、吐出一個數字」,這個數字告訴你它們方向有多一致。

內積是這四者裡最該細品的一個,因為它是光照計算的心臟。從公式 $\mathbf{a}\cdot\mathbf{b} = \lVert\mathbf{a}\rVert\lVert\mathbf{b}\rVert\cos\theta$ 看,當兩個都是單位向量時, 內積就直接等於 $\cos\theta$:方向完全相同 → 內積 = 1;互相垂直 → 內積 = 0;完全相反 → 內積 = $-1$。 等一下你會看到,「一個表面被照得多亮」幾乎就等於「表面朝向」與「光源方向」這兩個向量的內積。

幾何變換:用矩陣搬動世界

場景裡的物體需要被搬動、放大、旋轉:角色要走路(平移)、寶箱要由小變大出現(縮放)、 星球要自轉(旋轉)。這些操作統稱幾何變換(geometric transformation)。 我們先看每一種怎麼用數學表達,再看一個漂亮的統一觀點。

  • 平移(translation):把每個點都加上同一個位移向量。$(x, y) \to (x + t_x,\ y + t_y)$。
  • 縮放(scaling):把座標各乘上一個倍率。$(x, y) \to (s_x\,x,\ s_y\,y)$。
  • 旋轉(rotation):繞原點轉 $\theta$ 角。$(x, y) \to (x\cos\theta - y\sin\theta,\ x\sin\theta + y\cos\theta)$。

為什麼用矩陣?

縮放與旋轉,都可以寫成一個 2 × 2 矩陣乘上一個點(把點看成欄向量)。例如旋轉:

$$\begin{bmatrix} x' \\ y' \end{bmatrix} = \begin{bmatrix} \cos\theta & -\sin\theta \\ \sin\theta & \cos\theta \end{bmatrix} \begin{bmatrix} x \\ y \end{bmatrix} = \begin{bmatrix} x\cos\theta - y\sin\theta \\ x\sin\theta + y\cos\theta \end{bmatrix}$$

用矩陣有兩個巨大的好處。第一,形式統一:不管是縮放、旋轉、推移(shear),都是「一個矩陣乘一個點」, 硬體可以用同一套電路處理。第二——也是更深刻的——變換可以「組合」成一個: 如果你想「先旋轉、再縮放」,只要把兩個矩陣相乘成一個矩陣 $M = S \cdot R$, 之後對成千上萬個頂點,每個都只要做一次矩陣乘法,而不是兩次。場景裡幾百萬個頂點時,這個省下來的量很可觀。

⚠️
矩陣乘法不可交換,順序很重要

$A \cdot B \neq B \cdot A$!「先旋轉再平移」和「先平移再旋轉」會得到完全不同的結果 (你可以想像:先把人轉個身再往前走,跟先往前走再轉身,落點當然不一樣)。 組合變換時,作用在點上的順序是從右到左讀:$M = T \cdot R$ 套到點 $\mathbf{p}$ 上是 $T(R(\mathbf{p}))$——先旋轉,後平移。這是初學者最常踩的坑。

齊次座標:讓平移也能塞進矩陣

但這裡有個惱人的不對稱:縮放和旋轉都能寫成 2 × 2 矩陣乘法,唯獨平移不行—— 平移是「加上一個向量」,而矩陣乘法只會做「乘加的線性組合」,沒辦法憑空生出一個常數加項。 這意味著一連串變換裡只要夾了平移,就沒法全部融成一個矩陣,前面說的好處就破功了。

圖學界的解法優雅得近乎魔法:升一個維度。把 2D 的點 $(x, y)$ 寫成 3 個數字 $(x, y, 1)$, 多補一個固定為 1 的分量;對應地,變換矩陣升成 3 × 3。這套座標系統就叫齊次座標(homogeneous coordinates)。 奇蹟發生在:平移現在可以寫成矩陣乘法了——因為那個固定的 1,乘上矩陣最後一欄的 $t_x, t_y$, 剛好就把平移量「加」了進來:

$$\begin{bmatrix} x' \\ y' \\ 1 \end{bmatrix} = \begin{bmatrix} 1 & 0 & t_x \\ 0 & 1 & t_y \\ 0 & 0 & 1 \end{bmatrix} \begin{bmatrix} x \\ y \\ 1 \end{bmatrix} = \begin{bmatrix} x + t_x \\ y + t_y \\ 1 \end{bmatrix}$$

看出來了嗎?第一列算出來是 $1\cdot x + 0\cdot y + t_x \cdot 1 = x + t_x$——平移量被那個「1」帶了進來。 於是平移、旋轉、縮放現在全都是 3 × 3 矩陣乘法,可以自由相乘、組合成單一矩陣。 這就是為什麼你在圖學 API 裡看到的變換矩陣,在 2D 是 3 × 3、在 3D 是 4 × 4—— 那「多出來的一維」,正是為了收編平移。

🔑
齊次座標解決了什麼

用矩陣表達變換的好處是「能組合成一個」;但平移天生不是線性的,塞不進普通矩陣乘法。 齊次座標把點多補一維(設為 1),讓平移也能用矩陣乘法表示,於是所有變換統一成矩陣相乘—— 這是現代 GPU 管線能把一長串變換「壓成一個矩陣、餵給每個頂點」的根本前提。(它還有個額外紅利:能優雅地表達透視投影,稍後會見到。)

動手:用 numpy 旋轉一組點

來把理論寫成程式。我們用 numpy 對一個正方形的四個頂點做旋轉,並驗證矩陣組合的威力:

import numpy as np

def rotation(theta):
    """繞原點旋轉 theta 弧度的 2x2 矩陣"""
    c, s = np.cos(theta), np.sin(theta)
    return np.array([[c, -s],
                     [s,  c]])

# 正方形的四個頂點(每一「列」是一個點)
square = np.array([[0, 0],
                   [1, 0],
                   [1, 1],
                   [0, 1]], dtype=float)

R = rotation(np.pi / 4)          # 旋轉 45 度
# 點是列向量,所以右乘 R 的轉置;等價於對每個點做 R @ p
rotated = square @ R.T
print(rotated.round(3))
# [[ 0.     0.   ]
#  [ 0.707  0.707]
#  [ 0.     1.414]
#  [-0.707  0.707]]   ← 整個正方形被轉了 45 度

下面是齊次座標版本,展示「先旋轉、再平移」如何組合成單一 3 × 3 矩陣:

import numpy as np

def translation(tx, ty):
    return np.array([[1, 0, tx],
                     [0, 1, ty],
                     [0, 0, 1]], dtype=float)

def rotation_h(theta):
    c, s = np.cos(theta), np.sin(theta)
    return np.array([[c, -s, 0],
                     [s,  c, 0],
                     [0,  0, 1]], dtype=float)

p = np.array([1, 0, 1])                  # 點 (1, 0),補上齊次分量 1
M = translation(3, 2) @ rotation_h(np.pi / 2)   # 先旋轉 90°,再平移 (3,2)
print((M @ p).round(3))                   # [3. 3. 1.] -> 點 (3, 3)
# 驗證:(1,0) 轉 90° -> (0,1),再 +(3,2) -> (3,3)。順序「從右到左」!

3D 繪圖管線:從模型到像素的流水線

現在把場景升到 3D。一個 3D 物體(比如一隻茶壺)要變成螢幕上的像素,得經過一條固定的流水線, 叫繪圖管線(graphics pipeline)。它和你在第 11 章看過的管線(pipeline)架構是同一種思想: 把一個複雜任務拆成一連串階段,資料像在輸送帶上一站站被加工。每個頂點依序經過:

每一次「換座標系」,本質上就是乘上一個 4 × 4 齊次矩陣。物體誕生在自己的模型座標(以茶壺中心為原點); 擺進場景變成世界座標;再用「攝影機在哪、朝哪看」把整個世界轉成從鏡頭看出去的視圖座標; 然後最關鍵的一步——投影(projection),把 3D 壓扁成 2D。

透視投影:為什麼遠的東西看起來小

你一定注意過:鐵軌往遠方延伸會「收斂」,同樣大的人站遠了看起來就小。這個現象叫透視(perspective), 是我們的眼睛和相機共有的特性。它的數學原理意外地簡單:把座標除以深度。 想像攝影機在原點,眼前有一面影像平面(螢幕)。一個在深度 $z$ 處、高度為 $y$ 的點, 投影到平面上的高度是 $y' = \dfrac{y}{-z}$(取負號是因為攝影機看向 $-z$ 方向)。 $z$ 越大(越遠),分母越大,$y'$ 就越小——這就是「遠則小」的全部祕密。

有趣的是,這個「除以深度」的動作,正好能用齊次座標漂亮地表達:投影矩陣會把深度資訊塞進那個原本固定為 1 的齊次分量 $w$, 最後再讓所有座標除以 $w$(這步叫透視除法)。這也是齊次座標除了收編平移之外的第二個大紅利。

光柵化:把三角形變成像素

頂點被投影成 2D 後,接力棒交給光柵化:把每個三角形「填」成一格格像素。 你可能會問:為什麼基本圖元是三角形(triangle),而不是正方形或圓?幾個務實的理由: 三角形是「最簡單能圍出一塊面」的多邊形(三個點一定共面、一定是平的,不會像四邊形那樣翹曲); 任何複雜曲面都能用一堆小三角形拼出來(這叫網格 mesh);而且三角形的內部判定與插值在數學上特別乾淨。 所以無論是恐龍、車子還是人臉,在 GPU 眼裡都是幾百萬個三角形。

哪些像素在三角形內?重心座標

光柵化的核心問題是:「一個像素的中心,落在三角形裡還是外?」 判斷方法有好幾種,圖學最愛用的是重心座標(barycentric coordinates): 把三角形內任一點 $\mathbf{p}$ 表示成三個頂點的加權平均 $\mathbf{p} = \alpha A + \beta B + \gamma C$, 其中權重 $\alpha + \beta + \gamma = 1$。當且僅當三個權重都 ≥ 0 時,點在三角形內。 這套權重還有個附加超能力:它能把頂點上的屬性(顏色、深度、紋理座標)平滑地內插到三角形內部每個像素—— 後面的著色幾乎都靠它。

深度緩衝:誰擋住誰?

場景裡的三角形會互相遮擋:近的擋住遠的。光柵化一個個畫三角形時,怎麼確保「後面的不會蓋掉前面的」? 最笨的辦法是「先排序,從遠畫到近」(畫家演算法),但遇到互相穿插的三角形就會出錯。 現代做法簡單又穩健:深度緩衝(z-buffer,又叫 depth buffer)。

它的點子是:除了存顏色的影格緩衝,再額外開一塊一樣大的緩衝,專門記錄每個像素「目前畫上去的東西有多近」(深度 $z$)。 每當要在某像素畫一個新片元,就比一下它的深度:比現有的更近,才覆蓋顏色並更新深度;否則丟棄。 這樣不管三角形以什麼順序送進來,最後留在每個像素的,永遠是最靠近攝影機的那個。

著色與光照:讓表面看起來有立體感

到目前為止,我們只會把三角形填成「一塊顏色」,看起來扁平死板。要有立體感,得模擬光怎麼打在表面上。 這需要一個關鍵資訊:法向量(normal)——每個表面點「朝外」的方向,一個垂直於該表面的單位向量,記作 $\mathbf{N}$。 法向量就像皮膚上的細毛指向外面,它決定了表面「面對」著哪個方向,也就決定了它能接到多少光。

Phong 光照模型:三種光的疊加

最經典、直覺也最好的近似,是 1975 年提出的 Phong 光照模型(Phong reflection model)。 它把一個點看到的光,拆成三個成分相加:

  • 環境光(ambient):模擬「四面八方亂反射、無所不在」的微弱底光,讓背光面不至於全黑。它是一個常數 $k_a I_a$,跟方向無關。
  • 漫反射(diffuse):粗糙表面(如石膏、紙)把光均勻散開,亮度只取決於「表面有多正對光源」。 這正是內積登場的地方:亮度 $\propto \max(0,\ \mathbf{N}\cdot\mathbf{L})$,其中 $\mathbf{L}$ 是指向光源的單位向量。 正對光($\mathbf{N}\cdot\mathbf{L}=1$)最亮,側對($=0$)就暗,背對($<0$,取 0)全暗。這條規則叫 Lambert 餘弦定律。
  • 鏡面反射(specular):光滑表面(如金屬、塑膠、濕潤的眼睛)會在特定角度反射出一個刺眼的高光點(highlight)。 它取決於「反射光方向 $\mathbf{R}$ 與視線方向 $\mathbf{V}$ 有多接近」:$\propto (\max(0,\ \mathbf{R}\cdot\mathbf{V}))^{\alpha}$, 那個指數 $\alpha$(光澤度 shininess)越大,高光越小越銳利。

把三者加起來,就是 Phong 模型完整的式子:

$$I = \underbrace{k_a I_a}_{\text{環境}} + \underbrace{k_d (\mathbf{N}\cdot\mathbf{L})\, I_d}_{\text{漫反射}} + \underbrace{k_s (\mathbf{R}\cdot\mathbf{V})^{\alpha}\, I_s}_{\text{鏡面}}$$

著色器:在 GPU 上跑的小程式

上面這套光照計算,在現代 GPU 上不是寫死的,而是由你提供的一段小程式來做——這段程式就叫著色器(shader)。 最常見的兩種:頂點著色器(vertex shader)對每個頂點跑一次(負責那些矩陣變換), 片元著色器(fragment / pixel shader)對每個像素跑一次(負責算最終顏色,例如上面的 Phong 公式)。 GPU 會平行地對幾百萬個頂點、像素同時跑同一段著色器——這是它快的祕密,稍後再談。

順帶一提兩種容易混淆的「著色」:Gouraud shading 只在三角形的三個頂點算光照, 再把顏色內插到內部(快,但高光容易糊掉);Phong shading 則把法向量內插到每個像素、 在每個像素各算一次光照(慢一點,但高光漂亮銳利)。注意「Phong 光照模型」和「Phong shading」是兩件不同的事: 前者是算什麼(那條公式),後者是在哪算(逐像素)。

光線追蹤:反向追光,真實感的王道

現在進入本章最迷人的部分。光柵化是「把三角形畫到螢幕」,而光線追蹤反其道而行—— 它模擬光真實的物理:對螢幕上每個像素,從攝影機射出一條光線(ray), 問它「飛出去第一個撞到什麼物體」,然後在撞擊點計算顏色。一條光線用一個起點 $\mathbf{o}$ 與方向 $\mathbf{d}$ 描述, 線上的點就是 $\mathbf{p}(t) = \mathbf{o} + t\,\mathbf{d}$($t > 0$ 是沿著方向走的距離)。

光線追蹤之所以能產生那麼逼真的畫面,是因為一旦你會「射一條光線、找最近交點」, 所有真實光學現象都能自然地、用同一招長出來:

  • 陰影(shadow):在撞擊點,朝光源再射一條「陰影光線」。如果它在抵達光源前就撞到別的物體,代表被擋住了——這個點在陰影裡。
  • 反射(reflection):在鏡面撞擊點,算出反射方向,再射一條光線出去,看它撞到什麼,把那個顏色帶回來。鏡子、金屬、水面的倒影就是這樣來的。
  • 折射(refraction):對玻璃、水這類透明物,光線會穿透並轉彎(依司乃耳定律),再射一條折射光線追下去。

注意「反射光線又射出一條光線、再撞到鏡子又再反射……」這個結構——它是天然的遞迴(recursion) (還記得第 3 章嗎?)。一條主光線會「生出」反射與折射的子光線,子光線又可能生出孫光線, 直到打到非反射表面、或達到設定的遞迴深度上限才停。這就是它名字裡「追蹤(tracing)」的由來:追著光一路彈下去。

核心數學:光線與球的交點

光線追蹤的引擎,就是「光線與物體求交」。我們以最簡單的物體——球為例,因為它的交點有漂亮的封閉解。 球面上的點 $\mathbf{p}$ 滿足「到球心 $\mathbf{c}$ 的距離等於半徑 $r$」,即 $\lVert\mathbf{p} - \mathbf{c}\rVert^2 = r^2$。 把光線 $\mathbf{p}(t) = \mathbf{o} + t\mathbf{d}$ 代進去,展開後會得到一個對 $t$ 的一元二次方程式:

$$\underbrace{(\mathbf{d}\cdot\mathbf{d})}_{a}\,t^2 + \underbrace{2\,\mathbf{d}\cdot(\mathbf{o}-\mathbf{c})}_{b}\,t + \underbrace{(\mathbf{o}-\mathbf{c})\cdot(\mathbf{o}-\mathbf{c}) - r^2}_{c} = 0$$

剩下的就是國中數學:看判別式 $\Delta = b^2 - 4ac$。$\Delta < 0$:無實根,光線沒打中球; $\Delta \ge 0$:有交點,取較近的那個正根 $t = \dfrac{-b - \sqrt{\Delta}}{2a}$ 作為命中距離。一切就這麼簡單。

動手:寫一個能跑的極簡光線追蹤器

把上面的東西串起來,我們就能寫出一個真正能跑、會輸出影像的迷你光線追蹤器。 它對每個像素射一條光線、和一顆球求交、在命中點用漫反射(Lambert)上色,最後輸出一張 PPM 圖。 程式刻意只用標準函式庫,讓你看清每一步的數學: 

import math

# ---------- 一點點向量工具(用 tuple 表示 3D 向量)----------
def sub(a, b): return (a[0]-b[0], a[1]-b[1], a[2]-b[2])
def add(a, b): return (a[0]+b[0], a[1]+b[1], a[2]+b[2])
def mul(a, s): return (a[0]*s, a[1]*s, a[2]*s)
def dot(a, b): return a[0]*b[0] + a[1]*b[1] + a[2]*b[2]
def normalize(a):
    L = math.sqrt(dot(a, a))
    return (a[0]/L, a[1]/L, a[2]/L)

# ---------- 光線與球求交,回傳最近的正交點距離 t(沒中回傳 None)----------
def ray_sphere(o, d, center, r):
    oc = sub(o, center)
    a = dot(d, d)
    b = 2 * dot(oc, d)
    c = dot(oc, oc) - r*r
    disc = b*b - 4*a*c              # 判別式
    if disc < 0:
        return None                # 無實根 → 沒打中
    t = (-b - math.sqrt(disc)) / (2*a)   # 取較近的交點
    return t if t > 0 else None

# ---------- 場景設定 ----------
W, H = 240, 180                    # 影像解析度
eye      = (0.0, 0.0, 0.0)         # 攝影機在原點
sphere_c = (0.0, 0.0, -5.0)        # 球心在前方 5 單位
sphere_r = 1.5
light    = (5.0, 5.0, 0.0)         # 點光源位置
aspect   = W / H

# ---------- 主迴圈:對每個像素射一條光線 ----------
pixels = []
for j in range(H):
    for i in range(W):
        # 把像素 (i, j) 映射到 z = -1 的影像平面上,範圍約 [-1, 1]
        x = (2*(i + 0.5)/W - 1) * aspect
        y = 1 - 2*(j + 0.5)/H
        d = normalize((x, y, -1.0))         # 這條光線的方向
        t = ray_sphere(eye, d, sphere_c, sphere_r)
        if t is None:
            pixels.append((25, 28, 40))     # 背景:深藍灰
        else:
            p = add(eye, mul(d, t))         # 命中點
            n = normalize(sub(p, sphere_c)) # 球面法向量
            l = normalize(sub(light, p))    # 指向光源
            diff = max(0.0, dot(n, l))      # 漫反射強度 = max(0, N·L)
            shade = 0.1 + 0.9 * diff        # 環境光 0.1 + 漫反射
            pixels.append((int(230*shade), int(80*shade), int(70*shade)))

# ---------- 輸出 PPM ----------
with open("sphere.ppm", "w") as f:
    f.write(f"P3\n{W} {H}\n255\n")
    for (r, g, b) in pixels:
        f.write(f"{r} {g} {b}\n")
print("已輸出 sphere.ppm —— 打開它,你會看到一顆有光照的紅球!")

打開 sphere.ppm,你會看到一顆紅球:正對光源的左上方明亮、邊緣與背光處漸暗—— 那個從亮到暗的漸層,完全來自那一行 max(0.0, dot(n, l)),也就是 Lambert 餘弦定律。 這短短幾十行,已經包含了光線追蹤的全部骨架:射線 → 求交 → 法向量 → 光照。 本章末的挑戰題,會帶你把它擴充成多球、加陰影,甚至遞迴反射。

💡
為什麼這個迴圈「天生平行」

注意上面那個雙重迴圈:每個像素的計算彼此完全獨立——算第 (3, 7) 個像素時,根本不需要知道別的像素算出什麼。 這種「大量互不相干的相同計算」正是 GPU 的拿手好戲,也是為什麼真實世界的光線追蹤器會丟到 GPU 上跑、快上千百倍。 這個「資料平行」的主題,我們會在第十部平行程式設計裡深入。

紋理貼圖與抗鋸齒

紋理貼圖:把照片貼到表面上

如果每個三角形只能是單一顏色或平滑漸層,牆磚、木紋、人臉的細節就得用天文數字般多的三角形才畫得出來,太不划算。 聰明的做法是紋理貼圖(texture mapping):準備一張 2D 圖片(紋理), 在三角形的每個頂點標上它對應到紋理上的座標(稱為 UV 座標,範圍通常是 0 到 1), 再用前面說的重心座標,把 UV 平滑內插到三角形內每個像素,於是每個像素都能去紋理圖上「取樣」對應的顏色。 一張磚牆照片,就這樣被「糊」到一個簡單的四方形上,瞬間有了豐富細節——而幾何依然只是兩個三角形。

抗鋸齒:撫平階梯狀的邊緣

因為像素是方格,斜線與曲線邊緣會呈現難看的「階梯狀」鋸齒(aliasing)——這是把連續的幾何硬塞進離散網格的必然代價。 抗鋸齒(anti-aliasing)的核心想法是:邊緣上的像素,不要非黑即白地硬切,而是依它「被圖形蓋住多少比例」 塗上一個中間的灰階。常見做法是超取樣(supersampling):在每個像素內取多個樣本點、算完再平均, 讓邊緣由「硬切」變成「柔和過渡」,人眼看起來就平滑多了。

OpenGL、GPU,與即時 vs 離線的取捨

你不必每次都從頭手刻管線。OpenGL(以及 Direct3D、Vulkan、Metal、WebGPU 等)是業界的繪圖 API: 它把「設定矩陣、送頂點、跑著色器、寫進影格緩衝」這套流程,標準化成一組函式,讓你直接驅動 GPU。

為什麼非 GPU 不可?回想本章所有的核心動作:對幾百萬個頂點各乘一次矩陣、對幾百萬個像素各跑一次著色器、 對每個像素各射一條光線——這些都是「大量、互不相干、做著相同運算」的工作。 CPU 有幾個強大但數量少的核心,適合複雜的循序邏輯;GPU 則有成千上萬個簡單核心,專為這種大規模資料平行而生。 把同一段著色器同時餵給上千個像素並行計算,正是即時 3D 得以流暢的根本原因(這個平行運算的主題,留待第十部詳談)。

⚖️
即時 vs 離線:同一個權衡的兩端

即時渲染(real-time)(遊戲、VR)必須在 16 毫秒內畫完一張(才有 60 FPS), 所以選用快的光柵化,並用各種「夠像就好」的近似(陰影貼圖、環境光遮蔽、螢幕空間反射)來偽造真實感。 離線渲染(offline)(電影、動畫)一張圖可以算上好幾小時,於是放心用光線追蹤(及更進階的路徑追蹤) 去逼近真正的物理光學,換取以假亂真的畫質。近年硬體光線追蹤與 AI 降噪,正在快速模糊這條界線—— 遊戲已開始用上即時光線追蹤了。

圖學能拿來做什麼?

計算機圖學早已滲透進你生活的每個角落,遠不只是遊戲:

  • 電玩遊戲:即時光柵化的主戰場,追求在毫秒內畫出可互動的世界。
  • 電影與動畫:Pixar、工業光魔等用離線光線/路徑追蹤,渲染出真假難辨的特效與全 CG 電影。
  • CAD 與工程:建築、機械、晶片設計都靠 3D 建模與精確的幾何渲染來設計與審視。
  • 科學與資料視覺化(visualization):把醫學斷層掃描、流體模擬、分子結構、龐大資料變成看得懂的圖像。
  • VR / AR:虛擬實境與擴增實境需要極低延遲、雙眼立體的即時渲染,把圖學的即時性逼到極限。

動手做:練習題

✏️ 練習 39.1 實作

給兩個 2D 向量 $\mathbf{a} = (3, 4)$ 與 $\mathbf{b} = (4, 3)$,手算它們的內積與夾角, 再用 numpy 驗證。(提示:$\cos\theta = \dfrac{\mathbf{a}\cdot\mathbf{b}}{\lVert\mathbf{a}\rVert\,\lVert\mathbf{b}\rVert}$。)

看解答

內積 $\mathbf{a}\cdot\mathbf{b} = 3\times 4 + 4\times 3 = 24$。兩者長度都是 $\sqrt{3^2+4^2}=5$。 所以 $\cos\theta = \dfrac{24}{5\times 5} = \dfrac{24}{25} = 0.96$,夾角 $\theta = \arccos(0.96) \approx 16.26°$。

import numpy as np

a = np.array([3, 4])
b = np.array([4, 3])

d = np.dot(a, b)                                  # 24
cos_t = d / (np.linalg.norm(a) * np.linalg.norm(b))   # 0.96
angle = np.degrees(np.arccos(cos_t))             # 16.26 度
print(f"內積 = {d},  夾角 ≈ {angle:.2f}°")
# 內積 = 24,  夾角 ≈ 16.26°

這正是漫反射光照的計算核心:把 $\mathbf{a}$、$\mathbf{b}$ 換成「法向量」與「指向光源的向量」, 它們的(正規化後)內積就直接是表面亮度。

✏️ 練習 39.2 實作

用旋轉矩陣把點 $(1, 0)$ 繞原點旋轉 $90°$,手算預測結果,再用 numpy 驗證。 接著想想:旋轉 $180°$ 會得到什麼點?

看解答

$90°$ 時 $\cos 90° = 0,\ \sin 90° = 1$,旋轉矩陣是 $\begin{bmatrix}0 & -1\\ 1 & 0\end{bmatrix}$。 套到 $(1,0)$:$x' = 0\cdot 1 - 1\cdot 0 = 0$,$y' = 1\cdot 1 + 0\cdot 0 = 1$,得到 $(0, 1)$—— 原本指向右的點轉到了正上方,合理。旋轉 $180°$ 則得到 $(-1, 0)$(轉到正左方)。

import numpy as np

def rotate(p, deg):
    t = np.radians(deg)
    R = np.array([[np.cos(t), -np.sin(t)],
                  [np.sin(t),  np.cos(t)]])
    return R @ p

print(rotate(np.array([1, 0]), 90).round(3))    # [0. 1.]
print(rotate(np.array([1, 0]), 180).round(3))   # [-1.  0.]
✏️ 練習 39.3 思考

用你自己的話解釋:為什麼普通的 2 × 2 矩陣無法表示平移,而齊次座標(把點寫成 $(x, y, 1)$、矩陣升成 3 × 3)就可以? 動手把平移矩陣乘上 $(x, y, 1)$,看看那個「1」在裡面扮演什麼角色。

看解答

2 × 2 矩陣乘向量,算出來的每個分量都是 $x$、$y$ 的線性組合(形如 $ax + by$), 永遠沒有「不乘任何座標的常數項」。但平移 $(x, y)\to(x+t_x,\ y+t_y)$ 需要的正是這種常數加項,所以辦不到。

齊次座標的巧妙在於:多補的那個固定為 1 的分量,讓矩陣可以「乘到一個必定是 1 的東西」,於是常數項就有了著力點:

$$\begin{bmatrix} 1 & 0 & t_x \\ 0 & 1 & t_y \\ 0 & 0 & 1 \end{bmatrix}\begin{bmatrix} x \\ y \\ 1 \end{bmatrix} = \begin{bmatrix} 1\cdot x + 0\cdot y + t_x\cdot 1 \\ 0\cdot x + 1\cdot y + t_y\cdot 1 \\ 1 \end{bmatrix} = \begin{bmatrix} x + t_x \\ y + t_y \\ 1 \end{bmatrix}$$

關鍵就是第三欄的 $t_x, t_y$ 乘上那個 1。因為平移現在也是矩陣乘法,就能和旋轉、縮放自由相乘、合成單一矩陣—— 這是 GPU 把一長串變換壓成「一個矩陣餵給每個頂點」的前提。

✏️ 練習 39.4 實作

實作光線與球的交點函式 ray_sphere(o, d, center, r),回傳最近的正交點距離 $t$(沒打中回傳 None)。 用它測試:從原點往 $-z$ 方向射出的光線,是否打中球心在 $(0,0,-5)$、半徑 2 的球?交點在哪?

看解答

把光線代入球面方程式,整理成 $at^2+bt+c=0$,其中 $a=\mathbf{d}\cdot\mathbf{d}$、$b=2\mathbf{d}\cdot(\mathbf{o}-\mathbf{c})$、 $c=(\mathbf{o}-\mathbf{c})\cdot(\mathbf{o}-\mathbf{c})-r^2$,再看判別式:

import math

def dot(a, b): return a[0]*b[0] + a[1]*b[1] + a[2]*b[2]
def sub(a, b): return (a[0]-b[0], a[1]-b[1], a[2]-b[2])

def ray_sphere(o, d, center, r):
    oc = sub(o, center)
    a = dot(d, d)
    b = 2 * dot(oc, d)
    c = dot(oc, oc) - r*r
    disc = b*b - 4*a*c
    if disc < 0:
        return None                    # 沒打中
    t = (-b - math.sqrt(disc)) / (2*a) # 較近的根
    return t if t > 0 else None

o = (0, 0, 0)
d = (0, 0, -1)                         # 往 -z 射
t = ray_sphere(o, d, (0, 0, -5), 2)
print("t =", t)                        # t = 3.0
if t:
    hit = (o[0]+t*d[0], o[1]+t*d[1], o[2]+t*d[2])
    print("交點 =", hit)               # 交點 = (0, 0, -3)

球心在 $z=-5$、半徑 2,所以最近的表面在 $z=-3$,光線飛了 $t=3$ 個單位打中它——和直覺完全吻合。 這個函式就是上面那個完整光線追蹤器的引擎。

✏️ 練習 39.5 實作

實作 重心座標 / 邊測試來判斷一個點是否在三角形內:寫 in_triangle(p, a, b, c)。 (提示:用「邊的符號」——點對三角形每條邊算一個叉積符號,若三個符號一致,點就在內部。) 測試點 $(0.3, 0.3)$ 是否在頂點為 $(0,0)$、$(1,0)$、$(0,1)$ 的三角形內。

看解答

對每條有向邊,用 2D 叉積算出待測點落在邊的哪一側(正/負)。若點在三角形內,它對三條邊會落在同一側,三個符號同號:

def edge_sign(p, a, b):
    # 2D 叉積:(b-a) x (p-a),正負代表 p 在有向邊 a->b 的哪一側
    return (b[0]-a[0])*(p[1]-a[1]) - (b[1]-a[1])*(p[0]-a[0])

def in_triangle(p, a, b, c):
    d1 = edge_sign(p, a, b)
    d2 = edge_sign(p, b, c)
    d3 = edge_sign(p, c, a)
    has_neg = (d1 < 0) or (d2 < 0) or (d3 < 0)
    has_pos = (d1 > 0) or (d2 > 0) or (d3 > 0)
    return not (has_neg and has_pos)    # 沒有同時出現正負 → 同側 → 在內部

A, B, C = (0, 0), (1, 0), (0, 1)
print(in_triangle((0.3, 0.3), A, B, C))   # True(在內部)
print(in_triangle((0.8, 0.8), A, B, C))   # False(在斜邊外)

$(0.3, 0.3)$ 的座標和 $0.3+0.3=0.6 < 1$,在斜邊內側,結果 True; $(0.8, 0.8)$ 的和 $1.6 > 1$,跑到斜邊外,結果 False。這正是光柵化「逐像素判斷是否在三角形內」的核心。

✏️ 練習 39.6 挑戰

擴充本章的極簡光線追蹤器,任選(或全做)以下強化: (a) 在場景裡再加一顆球(不同位置/顏色),正確處理「取最近交點」; (b) 加上陰影:命中點朝光源射一條陰影光線,若被任何球擋住就只留環境光。

看解答

關鍵改動有兩處:用一個「球清單」並對每條光線找所有球裡最近的命中;以及在著色前,從命中點朝光源射一條陰影光線做遮擋測試。 注意陰影光線要從命中點稍微「往外推一點點」(沿法向量加一個小 epsilon),否則會因浮點誤差打到自己而產生黑斑(self-shadowing):

import math

def sub(a,b): return (a[0]-b[0], a[1]-b[1], a[2]-b[2])
def add(a,b): return (a[0]+b[0], a[1]+b[1], a[2]+b[2])
def mul(a,s): return (a[0]*s, a[1]*s, a[2]*s)
def dot(a,b): return a[0]*b[0]+a[1]*b[1]+a[2]*b[2]
def normalize(a):
    L = math.sqrt(dot(a,a)); return (a[0]/L, a[1]/L, a[2]/L)

def ray_sphere(o, d, c, r):
    oc = sub(o, c); a = dot(d,d); b = 2*dot(oc,d); cc = dot(oc,oc)-r*r
    disc = b*b - 4*a*cc
    if disc < 0: return None
    t = (-b - math.sqrt(disc)) / (2*a)
    return t if t > 1e-4 else None

# 場景:多顆球,每顆有顏色
spheres = [
    {"c": (-1.1, 0.0, -5.0), "r": 1.0, "color": (230, 80, 70)},   # 紅
    {"c": ( 1.3, -0.3, -6.0), "r": 1.3, "color": (70, 130, 230)}, # 藍
]
light = (5.0, 5.0, 0.0)

def nearest_hit(o, d):
    best_t, best = None, None
    for s in spheres:
        t = ray_sphere(o, d, s["c"], s["r"])
        if t is not None and (best_t is None or t < best_t):
            best_t, best = t, s
    return best_t, best

def in_shadow(p, n):
    o = add(p, mul(n, 1e-3))          # 沿法向量推開,避免自我陰影
    l = normalize(sub(light, p))
    t, hit = nearest_hit(o, l)
    return hit is not None             # 路上撞到任何球 → 在陰影中

W, H = 240, 180; eye = (0,0,0); aspect = W/H
pixels = []
for j in range(H):
    for i in range(W):
        x = (2*(i+0.5)/W - 1)*aspect
        y = 1 - 2*(j+0.5)/H
        d = normalize((x, y, -1.0))
        t, s = nearest_hit(eye, d)
        if s is None:
            pixels.append((25, 28, 40))
        else:
            p = add(eye, mul(d, t))
            n = normalize(sub(p, s["c"]))
            if in_shadow(p, n):
                shade = 0.1                       # 只剩環境光
            else:
                l = normalize(sub(light, p))
                shade = 0.1 + 0.9*max(0.0, dot(n, l))
            pixels.append(tuple(int(ch*shade) for ch in s["color"]))

with open("scene.ppm", "w") as f:
    f.write(f"P3\n{W} {H}\n255\n")
    for (r,g,b) in pixels: f.write(f"{r} {g} {b}\n")
print("已輸出 scene.ppm:兩顆球 + 陰影!")

跑完打開 scene.ppm,你會看到一紅一藍兩顆球,而且某顆球可能在另一顆上投下陰影。 想再進一步?把「陰影」那段邏輯改成遞迴地沿反射方向再射一條光線、把回傳顏色混進來,你就做出了鏡面反射—— 那正是光線追蹤遞迴本質(第 3 章)的展現。

把這一章串起來

我們從「螢幕只是一面會發光的網格」出發,一路走到能親手渲染一顆發亮的球。回顧這趟旅程的路標:

  • 兩大路線:光柵化(物體→像素,快,遊戲)與光線追蹤(像素→物體,真實,電影),是理解整個圖學版圖的主軸。
  • 影像基礎:像素、解析度、RGB 色深、alpha,最終都寫進影格緩衝那一長串數字。
  • 向量是空間的語言;內積衡量方向一致程度,是光照的心臟。
  • 矩陣統一表達變換並能組合;齊次座標把平移也收編進矩陣乘法,還順手搞定透視。
  • 繪圖管線:模型→世界→視圖→投影→裁剪→光柵化→像素,前半幾何、後半像素;透視就是「除以深度」。
  • 光柵化用重心座標判斷像素歸屬,用 z-buffer 解決遮擋。
  • Phong 光照把光拆成環境+漫反射+鏡面;著色器是 GPU 上平行跑的小程式。
  • 光線追蹤射線求交、遞迴追光,讓陰影、反射、折射自然湧現——你已親手寫出它的骨架。
  • 紋理提供細節、抗鋸齒撫平邊緣、GPU 用大規模平行讓這一切跑得即時。

最重要的體悟也許是:那些令人屏息的畫面,並不是什麼遙不可及的魔法,而是一層層樸實的數學疊出來的—— 向量描述方向、矩陣搬動世界、內積決定明暗、二次方程式找出交點。你手上那幾十行光線追蹤器, 和好萊塢的渲染農場,差別只在規模與精緻,核心思想是同一套。下一章我們轉向軟體工程, 談的不再是「怎麼算出漂亮的圖」,而是「怎麼讓一群人把龐大的軟體又快又穩地造出來」。
