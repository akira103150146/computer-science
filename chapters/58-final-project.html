那份你當初讀不完的連結清單

還記得第 0 章嗎?我們是這樣開始的——

面對一份列著三十幾門課、上百個連結的清單,大多數人撐不過第三週。

那時候,OSSU 的課程地圖在你眼前攤開,像一座望不到頂的山。你可能半信半疑地翻開了第一章, 讀了「計算是什麼」,寫了第一行 print("Hello")。然後是遞迴、是資料結構、是演算法; 是親手用 NAND 閘刻出一台電腦,是看懂作業系統怎麼排程、網路封包怎麼旅行;是密碼學、是機器學習、 是編譯器、是那些你一度以為「這輩子都不會懂」的數學。

而現在,你讀到了最後一章。

停下來,認真感受一下這件事的重量:那份你當初讀不完的清單,你走完了。 不是跳著看、不是看個熱鬧,而是一章一章、一個練習一個練習,把它啃了下來。光是這份耐心, 就已經把你和九成中途放棄的人區隔開來。

但這一章不是用來慶功的——至少不只是。它是用來畢業的。而畢業,需要一場畢業考。

🔑
學習的最後一哩,是「做」

讀懂和會做,是兩件事;會做和「做完一個真東西」,又是兩件事。 知識在你腦中是零散的拼圖,直到你親手把它們拼成一個能運作的系統,你才真正擁有它。 這一章要做的,就是帶你規劃並完成一個畢業專案(final project)—— 一個整合你所學、解決一個真實問題、可以驕傲地展示給別人看的作品。

這正是 OSSU 課程地圖的最後一站,也是本書的最後一站。前面所有的章節,都是為了這一刻在做準備。 所以,讓我們認真把它做好。

為什麼專案才是真正的畢業考

一場期末考能驗證你「記得」什麼;一個專案,才能驗證你「會」什麼。差別在於——專案逼你面對 課本永遠不會給你的東西:真實世界的混亂。以下是專案之所以無可取代的幾個理由。

它逼你把零散的知識整合起來

在課本裡,演算法歸演算法、資料庫歸資料庫、網路歸網路,每個主題都被乾淨地切開、分章講解。 但真實的軟體不是這樣運作的:一個按鈕被按下的那一瞬間,可能同時牽動了前端、API、資料庫查詢、 一段排序演算法、一次身份驗證、和一筆日誌寫入。專案是第一次,你必須把這些知識同時用上、 讓它們彼此咬合。那種「啊,原來這些東西是這樣連在一起的」的頓悟,只會發生在動手做的時候。

它讓你面對「沒有標準答案」的世界

課本的練習題,總有一個「正解」躲在最後一頁。但真實問題沒有解答篇:該用哪個框架?資料表怎麼設計? 這個效能瓶頸要先解嗎?需求改了怎麼辦?這些問題沒有人會告訴你答案,你得自己權衡、自己決定、 自己承擔後果。這種在不確定中做決策的能力,正是業界最看重、也最難從課堂學到的東西。

它幫你建立作品集

當你去求職、接案、或申請任何東西,「我讀完了一整套 CS 課程」是一句蒼白的話; 「這是我做的東西,你可以現在就點進去用」則是一記重拳。一個作品集(portfolio)裡的真實專案, 勝過十張結業證書。它讓對方看見你能做什麼,而不是聽你說你會什麼。

它教你「完成」一件事的紀律

這也許是最珍貴的一課。開始一個專案很容易,興奮個三天;但把它做完——撐過中段那片「不有趣、 但必須做」的沼澤,處理掉那些煩人的邊界情況,寫好 README,真正讓它上線——是一種稀有的能力。 世界上「做了一半的專案」多如牛毛,「真正完成的作品」鳳毛麟角。學會完成,你就贏過了絕大多數人。

ℹ️
而且,它就是最好的學習方式

還記得第 0 章說的嗎——「CS 是一門動詞,不是名詞。」你不會因為讀過排序演算法就學會它, 你會因為「親手寫出來、寫錯、除錯、再寫對」而學會。專案把這個道理放大到極致: 當你需要某個知識來解決眼前真實的問題時,你學它的速度與深度,會遠遠超過為了考試而讀它的時候。 做專案的過程,本身就是你學得最快、記得最牢的一段時間。

如何選一個好題目

選題是整個專案最重要、也最容易被輕忽的一步。選錯題,你會在三週後失去動力;選對題, 它會像磁鐵一樣把你拉著往前走。好題目通常從同一個地方長出來:你自己。

從「你自己的痛點或興趣」出發

最好的專案點子,往往是「我自己很想要、但市面上沒有(或做得很爛)的東西」。 因為當你是自己的第一個使用者,你天然就知道它該長什麼樣、哪裡好用哪裡卡——你不需要去「猜」需求。 而且當你卡關、想放棄時,「我真的想用到這個東西」這份動機,會把你拉回鍵盤前。

所以,先問自己幾個問題:我每天/每週重複做哪些煩人的事,能不能用程式自動化? 我有沒有哪個興趣(音樂、運動、遊戲、投資、語言學習)缺一個趁手的小工具? 我用過的哪個 App,有個讓我抓狂的缺陷,我能不能做一個更好的?

好專案的三個特徵

  • 解決一個真問題:有一個真實的人(哪怕只是你自己)會因為它而過得更好。不是為了做而做。
  • 技術上有挑戰,但可行:它應該逼你學一點新東西、用上你學過的硬知識——但又不至於難到讓你卡死。 甜蜜點是「我大概知道怎麼做,但有幾塊得邊做邊查」。
  • 你會想用它:如果連你自己做完都不想用,別人更不會。這是檢驗「是不是真問題」最誠實的試紙。
⚠️
最大的陷阱:野心過大

初學者選題最常見的死法,是範圍爆炸。「我要做一個比 Facebook 更好的社群網站」「我要做一個能打贏人類的遊戲 AI 平台」—— 這類題目聽起來很酷,但你會在第二週淹死在無窮無盡的細節裡,然後永遠放棄。
請記住:一個做完的小專案,價值遠勝過十個做不完的大專案。 寧可選一個「小到你確定能做完」的題目,做完之後再往上加功能。範圍能完成,是好題目的必要條件—— 其他一切都建立在「你真的會把它做完」之上。

💡
別小看「開源貢獻」這條路

如果你想不到要從零做什麼,替一個既有的開源專案貢獻是極好的畢業專案——而且更貼近真實工作。 你會練到讀懂別人(而且是大量)的程式碼、用 git 走完整套協作流程(fork → branch → PR → code review)、 以及和維護者溝通。從一個標著「good first issue」的小修正開始,你的名字就會出現在一個被全世界使用的專案裡。

從點子到計畫:寫一頁提案

有了點子,先別急著開 IDE 寫程式。花一個小時,把它寫成一頁專案提案(project proposal)。 這一頁不是繁文縟節,它是逼你「在動手前先想清楚」的工具——你會在寫的過程中, 發現自己其實還沒想明白的地方(這正是它的價值)。一頁提案該回答這幾個問題: 

專案提案 — 一頁版

1. 問題    我想解決什麼問題?誰有這個問題?
2. 使用者  誰會用?在什麼情境下用?(最重要的使用者是「我自己」也可以)
3. MVP    最小、但真的能用的版本,要有哪些功能?(只列 3~5 條核心)
4. 不做什麼 為了能完成,哪些誘人的功能「先砍掉」、留到以後?
5. 技術選型 語言、框架、資料庫、部署平台 —— 以及「為什麼選它」
6. 里程碑  M1 / M2 / M3…,每個里程碑都對應一個「能展示的成果」
7. 風險    我最不確定、最可能卡住的地方是什麼?要先去探一探嗎?

注意第 4 點「不做什麼」常被遺漏,卻和「要做什麼」一樣重要。 一份只有「要做」的清單會無限膨脹;明確寫下「這次先不做」,才是你守住範圍的護欄。

MVP:先做能動的最小版本

提案的核心是 MVP(Minimum Viable Product,最小可行產品)—— 這個觀念你在第 40 章學敏捷時見過。它的精神是:不要花三個月做一個「完美的大東西」才第一次拿出來; 而是先做出一個雖然陽春、但真的能從頭到尾跑一遍的最小版本,讓它能用,然後再一輪一輪地加東西。

舉個經典的比喻:如果你的目標是「一台車」,MVP 不是「先做四個輪子」(那階段什麼都不能載你), 而是「先做一台滑板車」——它很簡陋,但它現在就能載著你移動,而且你能立刻從使用中學到東西。 再來是腳踏車、機車,最後才是車。每一步都是「一個能用的完整東西」,只是越來越好。

🔑
先讓它能動,再讓它變好

程式設計界有一句名言:「Make it work, make it right, make it fast.」—— 先讓它能動,再讓它對(乾淨、可維護),最後才讓它快。 順序千萬別顛倒。新手最常見的錯誤,是還沒做出能動的東西,就先陷在「哪個架構最優雅」「效能夠不夠」的完美主義裡—— 結果什麼都沒生出來。先求有,再求好。一個能動的醜東西,永遠贏過一個完美的空想。

把整本書用上

現在,到了這趟旅程的高潮。

你或許曾在讀某一章時想:「這東西我這輩子用得到嗎?」——布林邏輯、離散數學、TCP 三次握手、 紅黑樹、雜湊、公鑰密碼……它們當時看起來像是各自獨立的孤島。但一個夠完整的專案, 會像一根線,把這些孤島串成一條項鍊。讓我們看看,當你做一個「有後端、有資料、有使用者」的 真實 Web 應用時,你會如何同時用上這本書的每一部:

  • 第四部 · CS 工具——你用 git 管理每一次改動、用 shell 跑指令、在你慣用的編輯器裡高效工作。專案的每一天都活在這套工具上。
  • 第二部 · 程式設計核心 + 第十部 · 進階程式設計——你用設計、OOP、架構把程式拆成乾淨的模組,用測試與除錯守住品質,讓它半年後還改得動。
  • 第六部 · 演算法與資料結構——當資料變多、回應變慢,你用對的資料結構和演算法讓它在合理的時間複雜度內跑完,而不是天真的雙重迴圈。
  • 第五部 · 計算機系統 + 第八部 · 應用——你用網路與作業系統的知識搭起後端伺服器,用資料庫與 API 儲存與交換資料。
  • 第七部 · 資訊安全 + 第十三部 · 進階資安——你用密碼學雜湊使用者密碼、用 HTTPS 加密傳輸,並擋下 SQL injection、XSS 等 Web 攻擊,保護你的使用者。
  • 第八部 · 應用(ML / 圖學)——你用機器學習加一個推薦或分類功能,或用圖學把資料畫成漂亮的互動視覺化,讓作品更出色。
  • 第三部 · 計算數學 + 第十四部 · 進階數學——背後那些演算法、那個 ML 模型、那個機率估計,全都站在微積分、離散數學、線性代數與機率的肩膀上。
  • 第九部 · 倫理——你誠實地對待使用者的資料與隱私,不做暗黑模式、不濫用資料,確保你做的是一個負責任的產品。

看見了嗎?沒有一部是白讀的。下面這張圖,把它畫成一句話:你的畢業專案,是全書十五部的匯流點。

ℹ️
不必每部都「用好用滿」

當然,沒有哪個專案會把十五部都用到極致——一個純前端小工具可能幾乎用不到資料庫, 一個資料分析專案可能完全不碰圖學。這張圖的重點不是「你必須全用上」, 而是「當你需要時,你已經有了」。你的工具箱裡有十五格抽屜,做專案時,你會驚訝於自己 一個個把它們拉開來用的樣子——那一刻,你會真正感覺到自己「畢業了」。

開發的務實流程

計畫寫好了,該動手了。下面是一套對個人專案行之有效的務實流程——它其實就是把第 40 章的軟體工程, 縮小到「一個人也能執行」的尺度。整個流程是一個循環,而不是一條直線:

版本控制與小步提交

從第一天就 git init(第 18 章)。不要等「做得差不多了」才開始用版本控制——那時你早就弄丟了一堆能救命的歷史。 養成小步提交的習慣:每完成一個小而完整的改動,就 commit 一次,並寫一句說得清楚的訊息 (「加入使用者登入」而不是「fix」「update」「123」)。這些 commit 訊息,就是你專案的日記。

寫 README 與文件

README 是你專案的門面,別人(包括三個月後的你)第一眼看的就是它。 一個好的 README 不需要長,但要讓人三分鐘內搞懂「這是什麼、怎麼跑起來」。基本骨架: 

# 專案名稱

一句話說明這個專案在做什麼、解決誰的什麼問題。

## Demo
（一張截圖 / 一段 GIF / 一個線上連結 —— 這最有說服力)

## 功能
- 核心功能一
- 核心功能二

## 安裝與執行
    git clone https://github.com/你的帳號/專案.git
    cd 專案
    （安裝相依、啟動的指令）

## 使用的技術
前端、後端、資料庫、部署平台,以及為什麼這樣選

## 我學到了什麼
（這段是給「未來的雇主/你自己」看的:這個專案讓你成長最多的地方)

## 授權
MIT

測試、CI 與部署

  • 測試(第 46 章):至少替最核心、最容易出錯的邏輯寫單元測試。它是你「敢大膽改程式」的安全網,而不是額外的雜事。
  • 持續整合 CI(第 40 章):用 GitHub Actions 之類的工具,設定「每次 push 自動跑測試」。綠燈才安心,紅燈馬上知道。這幾乎是免費的、值得學一次的投資。
  • 部署(deployment):這是個人專案最常被略過、卻最關鍵的一步——讓別人能真的點進去用。 一個躺在你硬碟裡的專案,和一個有公開網址的專案,在說服力上是天壤之別。現在有很多免費或便宜的平台 (靜態網站、容器、無伺服器函式)能讓部署變得很簡單,挑一個,把它推上去。

收集回饋,然後迭代

上線之後,把它拿給真實的人用——朋友、同好社群、Reddit、Discord。你會立刻發現一堆「自己永遠想不到」的問題與點子。 把這些回饋收進你的待辦清單,排好優先順序,進入下一輪迭代。記住圖 58-2:這個迴路轉得越多圈,你的作品越成熟, 你也學得越多。

完成與發表:讓世界看見

做完不等於結束。一個沒被任何人看見的好作品,就像一棵在無人森林裡倒下的樹。 花一點力氣把它「發表」出去,你的努力才會產生複利。

  • 把它放上 GitHub:配一份用心的 README、加上截圖、選一個開源授權(如 MIT)。一個整理乾淨的 repo,本身就是一份履歷。
  • 寫一篇介紹文或 demo:用一篇部落格、一支短影片、或一則貼文,講清楚三件事——你做了什麼、怎麼做的、學到了什麼。 尤其是「踩過哪些坑、怎麼爬出來」,那往往是最有價值、最能展現你思考過程的部分。
  • 接受同儕評審:OSSU 的畢業專案精神,正是「由全世界的同儕來檢視」。把作品丟進 OSSU 的 Discord 或相關社群, 請別人給意見。被看、被問、被挑戰,是讓作品(和你)變強的最快方式——這也呼應了第 40 章的 code review 文化。
  • 把它加進履歷與作品集:這個專案現在是你最有力的「證據」。在履歷上,一行「我做了 X,用了 Y 技術,解決了 Z 問題,線上連結在此」, 勝過十行空泛的形容詞。

三個範例專案的規劃草圖

理論講夠了,來看三個具體的例子。它們難度與方向各異,但都符合「能完成、用得上書中知識、做完想用」的標準。 每個都附上:題目、會動用書中哪些部、MVP、以及做完之後可以怎麼長大。

範例 A:個人理財追蹤 Web App

一個讓你記錄收支、自動分類、用圖表看見「錢花到哪去」的網頁應用。痛點很真實——大家都想理財,但現成 App 不是太複雜就是要收費。

  • 動用書中:第二部(模組設計)、第八部(資料庫存交易、REST API)、第十部(測試金額計算這種出錯成本高的邏輯)、第十三部(登入與資料保護)、第三部(基本統計做月報表)。
  • MVP:能新增一筆「金額 + 分類 + 日期」、列出清單、顯示本月總支出。就這樣,先讓它能用。
  • 可擴充:圓餅圖視覺化(圖學)、預算提醒、匯入銀行 CSV、用簡單 ML 自動猜分類、多裝置同步。

範例 B:小型程式語言直譯器

替一個你自己設計的迷你語言(或一個現有語言的子集)寫一個直譯器(interpreter)。 這是一個「硬核但無比有成就感」的題目,直接呼應第 42 章(編譯器)——當你親手讓自己寫的程式碼「跑起來」, 你對「程式語言到底是什麼」的理解會徹底升級。

  • 動用書中:第 42 章(詞法分析、語法分析、求值)、第六部(用樹這個資料結構表示 AST、遞迴走訪)、第一/二部(扎實的程式設計)、第十部(用測試確保每個語言特性都正確)。
  • MVP:能算整數的四則運算式(2 + 3 * 4)——也就是一個「會尊重運算優先序」的計算機。麻雀雖小,五臟俱全。
  • 可擴充:加入變數與賦值、if 與迴圈、函式、再到自訂資料型別;甚至往前一步,把直譯改成編譯到第 22 章的組合語言或一個虛擬機。

範例 C:資料分析 / ML 小專案

挑一個你真心好奇的問題,找一份公開資料,把它分析出一個有趣的結論,最後(進階)訓練一個簡單的模型來預測。 例如:「哪些因素最影響某城市的房價?」或「我能不能用留言文字判斷它是正評還是負評?」

  • 動用書中:第八部(機器學習)、第三/十四部(統計、線性代數、機率——模型背後的數學)、第六部(資料清理與處理的演算法)、第八部(用資料庫管理資料集)。
  • MVP:把資料載入、清理乾淨,畫出幾張能說故事的圖表,寫下一個你自己都覺得「喔,有意思」的觀察。模型可以之後再加。
  • 可擴充:訓練一個分類/迴歸模型並誠實評估它的準確度、把它包成一個能輸入新資料就吐出預測的小網頁、做成互動式儀表板。

🚀 行動 58.2 規劃

替你圈選的那個點子,寫出完整的一頁提案(用本章那七個欄位:問題、使用者、MVP、不做什麼、技術選型、里程碑、風險)。 特別認真寫「不做什麼」那一欄——它是你守住範圍的護欄。

看範例(記帳小網站)
  • 問題:我用 Excel 記帳,但很難一眼看出錢花到哪、本月超支沒。
  • 使用者:我自己,以及任何想「無痛記帳並看見趨勢」的人。手機與電腦都會用。
  • MVP:① 新增一筆收支(金額/分類/日期)② 列出全部紀錄 ③ 顯示本月總支出。
  • 不做什麼(先砍):多人帳本、銀行自動匯入、預算提醒、外幣、App 上架——通通留到以後。
  • 技術選型:Python + Flask(我熟)、SQLite(夠用、零設定)、部署到免費平台。選它們是因為「能最快做出能動的東西」。
  • 里程碑:M1 後端能存/讀一筆交易;M2 前端表單 + 清單頁;M3 月總計 + 部署上線;M4 圓餅圖。
  • 風險:我沒部署過 Flask App——先花半天做個「Hello World 部署」探路,排除最大的未知。
🚀 行動 58.3 拆解

把你的專案拆成 3~5 個里程碑。鐵律:每一個里程碑結束時,都要有一個「能展示給別人看的成果」 (哪怕很陽春)。避免「先把所有後端寫完、再來寫前端」這種「中途什麼都看不到」的切法。

看範例
  • M1(能動的最小版):一個命令列程式,能輸入一筆交易、印出全部交易與總額。→ 可展示:在終端機跑給人看。
  • M2(上網頁):做成網頁,有表單能新增、有頁面能列出。→ 可展示:在瀏覽器操作。
  • M3(上線):加月總計,部署到公開網址。→ 可展示:傳一個連結給朋友,他現在就能用。
  • M4(變好看):加圓餅圖、分類篩選。→ 可展示:截圖發一篇介紹文。

注意:M1 就已經是「一個完整、能用的東西」了,只是很小——這就是 MVP 的滑板車。

🚀 行動 58.4 串起全書

對著圖 58-1,列出你的專案會具體用到本書哪些章。每一項寫一句「用它來做什麼」。 這一步會讓你發現:原來你已經擁有的工具,比你以為的多得多。

看範例(記帳小網站)
  • 第 18 章(git):第一天就 init,每個里程碑開一條分支、小步提交。
  • 第 7、8 章(設計/OOP):把「交易」設計成一個乾淨的類別/模組。
  • 第 37 章(資料庫):用 SQLite 設計 transactions 資料表、寫 SQL 查月總計。
  • 第 40 章(軟工)+ 第 46 章(測試):替「總額計算」寫單元測試(金額算錯很嚴重),設定 CI。
  • 第 34、35 章(資安):登入密碼用雜湊存、用 HTTPS,別把別人的財務資料裸奔。
  • 第 39 章(圖學)+ 第 13 章(統計):把分類支出畫成圓餅圖。
  • 第 41 章(倫理):財務資料極度敏感——只收必要的、講清楚怎麼用、能讓使用者刪除。
🚀 行動 58.5 挑戰 · 真的動手

進階,但這才是重點:實際開一個 GitHub repo,寫下你的 README(用本章的骨架,至少填好「這是什麼/功能/我學到什麼」), 並做出你的第一個 commit。當你按下 push 的那一刻,你的畢業專案就正式存在於世界上了。

看參考指令 
mkdir my-project && cd my-project
git init

# 用編輯器寫好 README.md(這是你專案的門面)
# 至少寫清楚:這是什麼、要有哪些功能、你想學到什麼

git add README.md
git commit -m "初始提交:專案緣起、目標與 MVP 規劃"

# 在 GitHub 網站上開一個空的 repo,然後:
git branch -M main
git remote add origin https://github.com/你的帳號/my-project.git
git push -u origin main

恭喜——你的旅程,從「讀」正式跨進了「做」。剩下的,就是一個 commit、一個 commit 地把它養大。

把這一章串起來

這一章,我們把整本書收束成一個動作:做一個專案。

  • 專案是真正的畢業考:它逼你整合知識、面對沒有標準答案的混亂、建立作品集,並學會「完成」的紀律——而它本身就是最好的學習方式。
  • 選題從你自己出發:好題目解決真問題、有挑戰但可行、你自己會想用;而最大的陷阱是野心過大——範圍能完成是一切的前提。
  • 先寫一頁提案,再用 MVP 思維迭代:先讓它能動,再讓它變好;讓「開發→測試→上線→回饋」的迴路轉得越快越好。
  • 一個專案,動用全書十五部(圖 58-1):git 管開發、設計與測試保品質、演算法求效率、系統與資料庫撐後端、資安護使用者、ML 與圖學添亮點、數學打底、倫理把關。
  • 完成不等於結束:寫好 README、放上 GitHub、寫一篇介紹、接受同儕評審、加進作品集——讓世界看見你做了什麼。
