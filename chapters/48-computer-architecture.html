一台「能跑」的 CPU,為什麼還不夠

在第 21 章,我們把 ALU、暫存器、控制單元、程式計數器組裝起來,讓它跑那個永不停止的 取指 → 解碼 → 執行 → 回寫週期。那台機器是「對」的:你給它一段機器碼,它會老老實實算出正確答案。 但如果你拿它去跑現代遊戲,你會等到天荒地老。它慢。

慢在哪裡?問題就出在那個樸實的指令週期。我們上一章的設計,本質上是「一條指令做完,才開始下一條」: 取指、解碼、跑 ALU、讀寫記憶體、寫回……整套走完,才回頭抓下一條。這就像一家洗衣店, 一次只洗一袋衣服:洗完、烘完、摺完、收進櫃子,才肯收下一位客人的髒衣服。中間大部分機器,大部分時間都在發呆。

過去半個世紀,計算機結構(computer architecture)這門學問,絕大部分的精力都花在同一件事上: 在「指令集不變、結果不變」的前提下,把同樣一段程式跑得更快。它靠的不是把時脈無限拉高 (那條路早就撞牆了),而是一連串極聰明的把戲——管線、快取、亂序執行、平行……

這些把戲絕大多數對你「不可見」:你寫的 C 或 Python 完全不知道它們存在。但有兩個——快取與分支預測—— 會從幕後伸出手,讓「邏輯上一模一樣」的兩段程式,實測速度差上好幾倍。這一章的終極目的,就是讓你看懂這些幕後機制, 從此能寫出「對硬體友善」的程式。先從一件最基本卻最常被搞錯的事開始:怎麼衡量「快」。

衡量效能:別再只盯著 GHz

街頭巷尾比 CPU,大家張口就是「我這顆 4 GHz,比你那顆 3 GHz 快」。這句話幾乎總是錯的。 要看懂為什麼,我們需要一條計算機結構裡最重要的公式。一個程式真正花掉的 CPU 時間,由三個因子相乘決定:

🔑
鐵打的效能公式(CPU performance equation)

$$\text{CPU 時間} = \underbrace{\text{指令數}}_{\text{IC}} \times \underbrace{\text{CPI}}_{\text{每指令週期數}} \times \underbrace{\text{時脈週期}}_{1/\text{頻率}}$$ 也就是:跑完一個程式要花多久,取決於(1)要執行幾條指令、(2)平均每條指令花幾個時脈週期(CPI,Cycles Per Instruction)、(3)每個時脈週期多長(時脈頻率的倒數)。GHz 只影響第三項。

看清楚了嗎?GHz(頻率)只決定三個因子裡的一個。一顆 4 GHz 的 CPU 如果 CPI 很差(每條指令拖很多週期), 或者它的指令集需要更多指令才能做完同一件事,照樣可能輸給 3 GHz 的對手。把這三項拆開,你會發現它們分別被誰決定:

因子主要被誰影響怎麼改善
指令數(IC)指令集架構(ISA)、編譯器、你寫的演算法更好的演算法 / 編譯器最佳化 / 更強的指令
CPICPU 微架構(管線、快取、亂序…)本章的主角:管線、快取、ILP
時脈週期製程、電路設計提高頻率(受功耗 / 散熱限制)

那要怎麼公平地比兩顆 CPU?業界的答案是基準測試(benchmark):跑一組標準化、有代表性的真實程式 (例如 SPEC CPU),量它們的實際完成時間,再綜合成分數。一句話總結: 能信的不是規格表上的 GHz,而是「跑你真正在乎的工作,要花多久」。

Amdahl 定律:該優化哪裡?

假設你量了基準測試,發現程式很慢,於是準備動手優化。第一個該問的問題不是「怎麼優化」,而是「優化哪裡」。 這裡有一條冷酷的定律會幫你算清楚——Amdahl 定律(Amdahl's Law)(我們在第 43 章談平行運算時會再深入,這裡先用它的通用形式):

如果某個部分占了總時間的比例 $p$,而你把這個部分加速了 $s$ 倍,那麼整個程式的總加速比是:

$$S_{\text{總}} = \frac{1}{(1 - p) + \dfrac{p}{s}}$$

這條式子有個讓人清醒的推論:無論你把那 $p$ 的部分加速到多誇張($s \to \infty$),總加速比也頂多只能到 $\dfrac{1}{1-p}$。 舉例:某函式占了程式 90% 的時間($p = 0.9$),你把它加速 10 倍($s = 10$):

$$S_{\text{總}} = \frac{1}{(1 - 0.9) + \dfrac{0.9}{10}} = \frac{1}{0.1 + 0.09} = \frac{1}{0.19} \approx 5.3 \text{ 倍}$$

但如果你跑去優化那只占 10% 的部分,就算把它變成瞬間完成,總加速也只有 $\frac{1}{0.9} \approx 1.11$ 倍。 這就是計算機結構裡反覆出現的鐵律——

🔑
優化常見情況(make the common case fast)

把力氣花在「執行得最頻繁、占時間最多」的那條路上,回報才大。這正是 CPU 設計者的中心思想: 快取為什麼有效?因為「絕大多數記憶體存取」其實集中在一小塊資料上。分支預測為什麼有效? 因為「絕大多數分支」其實方向很固定。整顆現代 CPU,就是一台「把常見情況優化到極致」的機器。

指令集架構:RISC 與 CISC 的真實取捨

第 21 章我們提過:指令集架構(ISA)是硬體與軟體之間的契約——它規定「CPU 聽得懂哪些指令、有哪些暫存器、怎麼定址」。 當時我們用一句話帶過 RISC 與 CISC。現在我們有了效能公式,可以看懂這兩派背後的取捨到底在賭什麼。

回看公式 $\text{CPU 時間} = \text{IC} \times \text{CPI} \times \text{時脈週期}$。兩派其實是在這三個因子之間做不同的押注:

CISC(複雜指令集)RISC(精簡指令集)
哲學一條指令做很多事每條指令只做一件簡單的事
指令數 IC少(一條抵多條)多(同樣的事要拆成好幾條)
指令長度不定長(1~15 bytes)固定長(通常 4 bytes)
記憶體存取運算可直接吃記憶體運算元只有 load/store 碰記憶體,運算只碰暫存器
解碼難度難(變長、複雜)易(整齊劃一,好做管線)
代表x86 / x86-64ARM、RISC-V、MIPS

CISC 賭的是「降低 IC」:在記憶體又小又貴、編譯器又笨的 1970 年代,讓一條指令幹很多事是合理的——程式更小、人寫組語更省力。 RISC 賭的是「降低 CPI、提高頻率」:指令簡單又定長,解碼超快,而且整齊到非常好「管線化」(下一節的主角)。 1980 年代之後記憶體變便宜、編譯器變聰明,RISC 的賭注勝出,成了今天的主流思想。

ℹ️
一個你會喜歡的祕密:現代 x86 其實「內心是 RISC」

x86 在外表上仍是 CISC(為了相容三十年的軟體),但現代 Intel/AMD 晶片在解碼階段,會把每條複雜的 x86 指令 拆成數條內部的、類 RISC 的微指令(micro-ops,μops),然後讓後端管線去跑這些簡單的微指令。 所以「CISC vs RISC」之爭,某種程度上 RISC 是贏家——連 CISC 都在肚子裡偷偷變成 RISC。 至於 RISC-V,它是近年崛起的開放指令集:任何人都能免費使用、自由實作,不必付授權金,正在學術界與嵌入式領域快速擴張。

管線:讓指令重疊起來(重頭戲一)

回到開頭那家「一次只服務一位客人」的洗衣店。它的問題不是機器太慢,而是機器沒同時用: 烘衣機在轉的時候,洗衣機卻閒著。一個聰明的老闆會怎麼做?

當第一袋衣服從洗衣機進到烘衣機,就立刻把第二袋衣服丟進洗衣機。

於是洗、烘、摺三道工序重疊進行:任一時刻,三台機器可能都在忙不同的衣服。每袋衣服走完全程的時間沒變 (這叫延遲 latency),但「平均多久能交出一袋乾淨衣服」大幅縮短了(這叫吞吐量 throughput)。 這就是管線(pipelining)的全部精神。

CPU 把指令週期切成幾個「工序」,讓多條指令在不同工序裡同時前進。經典的 RISC 設計切成 5 級:

階段英文做什麼洗衣店類比
IFInstruction Fetch從記憶體抓指令把髒衣服拿進來
IDInstruction Decode解碼、讀暫存器分類、看洗標
EXExecuteALU 運算、算位址洗
MEMMemory access讀寫資料記憶體烘
WBWrite Back把結果寫回暫存器摺好收櫃

🔑
吞吐量 ≠ 延遲

管線不會讓單一條指令更快走完(延遲不變,甚至因為多了階段暫存器而略增)。它讓整體吞吐量暴增—— 單位時間內「完工」的指令變多了。這是兩個常被混為一談、卻截然不同的效能指標。對使用者而言,真正有感的是吞吐量。

管線到底快多少?

假設管線有 $k$ 級、要跑 $n$ 條指令。沒有管線時(每條指令乖乖花 $k$ 個週期才換下一條),總共要 $n \times k$ 個週期。 有了管線,第一條指令花 $k$ 個週期灌滿管線,之後每個週期吐一條,總共只要 $k + (n - 1)$ 個週期。加速比:

$$\text{加速比} = \frac{n \times k}{k + (n - 1)} \xrightarrow{\;n \to \infty\;} k$$

當指令很多($n$ 很大),加速比逼近 $k$——級數越多,理論上越快。那為什麼現代 CPU 不切成 100 級? 因為(1)每多一級就多一層暫存器與成本;(2)更致命的是——上面這個漂亮的公式,假設管線永遠順暢、永不卡住。 現實沒這麼好。管線會卡,而它卡住的原因,叫做危障。

管線危障與解法:現實的反擊

危障(hazard)指的是「某種狀況,讓下一條指令不能在預定的時脈順利進入它的階段」。它有三類,我們逐一擊破。

① 結構危障:兩條指令搶同一個零件

結構危障(structural hazard)最單純:兩條指令在同一個時脈,想用同一個硬體資源。 例如,如果指令記憶體和資料記憶體共用同一個埠,那麼當「指令 4 想 IF(讀指令)」碰上「指令 1 想 MEM(讀資料)」,兩者撞車。 解法通常是「多給硬體」:把指令快取和資料快取分開(這叫哈佛式切法),或讓暫存器檔在前半週期寫、後半週期讀。 結構危障靠砸硬體就能消除,所以在設計良好的 CPU 裡很少見。真正棘手的是後兩種。

② 資料危障:我要用的數,上一條還沒算好

資料危障(data hazard)是管線最常見的麻煩。看這兩條指令(用類 RISC 組語): 

add  r3, r1, r2     # r3 = r1 + r2
sub  r5, r3, r4     # r5 = r3 - r4   ← 要用上一條算出的 r3!

問題來了:add 要到第 5 級(WB)才把結果寫回 r3;但 sub 在第 3 個時脈就要進 ID 去讀 r3—— 那時 r3 根本還是舊值!這種「讀取一個還沒被寫回的值」叫 RAW(Read After Write)資料危障。

最笨的解法是停頓(stall):讓 sub 在原地等,插入幾個空週期(俗稱泡泡 bubble), 等 r3 真的寫回了再放行。但這浪費了寶貴的週期。聰明得多的解法是前饋 / 旁路(forwarding / bypassing):

🔑
前饋(forwarding):結果不必等寫回

注意一個關鍵事實:add 的結果,其實在它的 EX 階段結束時就算出來了(只是還沒「正式」寫回暫存器)。 那何不加一條捷徑,把 ALU 的輸出直接送到下一條指令的 ALU 輸入,繞過「寫回再讀出」的繞路? 這就是前饋。有了它,上面那個 add/sub 完全不用停頓。

但前饋並非萬能。有一種情況連它都救不了——load-use 危障:當你從記憶體載入一個值, 然後立刻就要用它: 

lw   r1, 0(r2)      # 從記憶體載入到 r1(資料要到 MEM 階段才拿到)
add  r3, r1, r4     # 立刻用 r1 → 即使有前饋,還是得停 1 拍

因為 lw 的資料要到第 4 級(MEM)結束才到手,但 add 的 EX 在第 4 級就需要它,時間上差一步, 無論如何都得插一個泡泡。聰明的編譯器會試著「重排指令」,在 lw 和用到它的指令之間塞一條無關的指令來填這個泡泡—— 這就是編譯器與硬體在背後為你做的優化之一。

③ 控制危障:下一條指令到底在哪?

最後、也最昂貴的是控制危障(control hazard),又叫分支危障。問題出在:管線需要持續餵指令, 每個時脈都要 IF 抓下一條。可是當它碰到一個條件分支(if、迴圈判斷)時, 「下一條指令在哪」要等分支算完(可能到 EX 階段)才知道。那在算出來之前的這幾個時脈,管線該抓什麼?

總不能讓管線空轉等待(那等於每個分支都付出好幾個泡泡的代價,而程式裡分支多得是)。現代 CPU 的賭徒式解法是 分支預測(branch prediction):猜一個方向,然後大膽地照猜測繼續抓指令、繼續執行。

  • 猜對了:賺到!管線從沒卡過,零代價。
  • 猜錯了:糟糕。已經抓進來、跑了一半的「錯路」指令全部得作廢清空(flush),從正確位址重新抓。 這個浪費叫分支錯誤預測代價(misprediction penalty),在深管線的現代 CPU 上可能高達 15~20 個週期。

現代分支預測器準到嚇人(命中率常 > 95%),靠的是記錄「這個分支歷史上往哪走」,用歷史模式下注。 對「迴圈尾巴那個幾乎總是跳回去」的分支,它幾乎不會猜錯。但這也埋下一個對程式設計師可見的陷阱: 如果你的分支方向「飄忽不定、毫無規律」,預測器就會頻頻猜錯,你的迴圈會莫名其妙地慢下來。

對程式設計師可見:已排序 vs 未排序的詭異速度差

這是計算機結構裡最有名的「真實案例」。下面這段 C,對一個裝著 0~255 隨機值的大陣列,把所有 ≥ 128 的元素加總,重複很多次: 

#include <stdio.h>
#include <stdlib.h>
#include <time.h>

static int cmp(const void *a, const void *b) {
    return (*(const int *)a) - (*(const int *)b);
}

int main(void) {
    const int N = 32768;
    int *data = malloc(N * sizeof(int));
    for (int i = 0; i < N; i++)
        data[i] = rand() % 256;          /* 0..255 的隨機值 */

    /* qsort(data, N, sizeof(int), cmp);   ← 把這行的註解拿掉,陣列就變成「已排序」 */

    clock_t start = clock();
    long long sum = 0;
    for (int r = 0; r < 100000; r++)     /* 重複很多次,放大差異 */
        for (int i = 0; i < N; i++)
            if (data[i] >= 128)          /* ← 關鍵分支:猜得準不準,全看這裡 */
                sum += data[i];

    double secs = (double)(clock() - start) / CLOCKS_PER_SEC;
    printf("sum=%lld, time=%.2f s\n", sum, secs);
    free(data);
    return 0;
}

實測結果讓無數人嚇一跳:把 qsort 那行打開(讓陣列先排序),整段程式會快上好幾倍 ——而它做的計算工作量一模一樣(同樣 N 個元素、同樣的比較與加法)。怎麼會這樣?

🔑
為什麼排序後會變快?關鍵在那個 if

未排序時,data[i] >= 128 的結果在「真/ 假」之間隨機跳動,分支預測器約有一半機率猜錯, 每次猜錯就是一次管線清空——成千上萬次的錯誤預測累積成巨大開銷。排序後,前半段全部 < 128(分支「總是假」), 後半段全部 ≥ 128(分支「總是真」),只有正中間切換一次。預測器面對這種「長條規律」,幾乎全部猜中, 錯誤預測趨近於零。同樣的計算,只因資料的「可預測性」不同,速度天差地別。這就是硬體偷偷影響你程式的鐵證。

順帶一提:這類「靠分支」的瓶頸,有時能用無分支(branchless)技巧改寫掉。例如 sum += (data[i] >= 128) * data[i]; 或用位元遮罩,讓 CPU 不必賭方向。 但別急著到處這樣寫——可讀性的代價往往不值得;真正的教訓是:當你發現一個熱迴圈裡有難以預測的分支,它可能就是你的隱形瓶頸。

記憶體階層:速度與容量的金字塔(重頭戲二)

管線解決了「CPU 內部」的效率。但還有一個更大的瓶頸在外面:記憶體很慢。慢到什麼程度? 如果把 CPU 存取暫存器的時間想成「1 秒」,那麼跑去主記憶體(DRAM)拿一筆資料,大約等於「幾分鐘」。 CPU 每等一次記憶體,就是幾百個時脈週期的乾等——管線跑得再順都救不了。

為什麼不乾脆把記憶體做得跟暫存器一樣快?因為又快又大又便宜的記憶體不存在。這是物理與經濟的鐵則: 快的(SRAM)貴又占空間,所以只能做小;大的(DRAM、磁碟)便宜但慢。工程師的妥協,是把不同速度/容量的記憶體 分層疊起來,形成記憶體階層(memory hierarchy):

但這座金字塔憑什麼有用?如果程式的記憶體存取真的「完全隨機、毫無規律」,那再多快取也救不了——CPU 想要的東西總在最慢的那層。 它之所以有效,是因為真實程式有一個美妙的統計傾向,叫局部性原理(principle of locality):

🔑
兩種局部性(locality)

時間局部性(temporal locality):剛存取過的資料,很可能很快又會被存取。 (例:迴圈裡的計數器、累加變數,每一輪都用到。)
空間局部性(spatial locality):存取了某個位址,它附近的位址很可能也快被存取。 (例:走訪陣列時,一個接一個,位址連續。)
快取的整個設計,就是賭程式有這兩種局部性——而絕大多數程式都有。這又是一次「優化常見情況」。

快取:它到底怎麼運作

快取(cache)是夾在 CPU 與主記憶體之間的一塊小而快的記憶體,自動保存「最近用過、以及它附近」的資料。 它的運作完全自動、對程式透明——你不會在 C 裡寫「請把這個放進 L1」,硬體替你打理一切。我們把它拆開來看。

快取行:搬資料以「整塊」為單位

為了利用空間局部性,快取不以「一個位元組」為單位搬資料,而是以一整塊連續的位元組—— 快取行(cache line / block),在現代 x86 上通常是 64 bytes。 當你存取某個位址,快取會把「包含它的那整條 64 位元組」一次拉上來。所以你讀了 a[0], 其實 a[1] 到 a[15](假設是 4-byte int)也順便進了快取——下次讀它們,直接命中。

命中與失誤

  • 命中(hit):要的資料正好在快取裡,飛快拿到。
  • 失誤(miss):不在快取裡,得往下一層(L2、L3、DRAM)拿,慢。失誤分三類(著名的 3C): 強制失誤(compulsory,第一次碰當然沒有)、容量失誤(capacity,快取裝不下整個工作集)、 衝突失誤(conflict,好幾塊資料搶同一個位置)。

對映方式:一塊資料能放在快取的哪裡?

主記憶體很大,快取很小,所以多塊記憶體必然要共用快取裡的位置。「一塊記憶體資料可以放進快取的哪些格子」, 就是對映方式(mapping / associativity)。為了查找,CPU 把位址切成三段:

三種對映方式,其實是「組內有幾路」這個光譜的兩端與中間:

方式一塊資料能放哪優點缺點
直接對映(direct-mapped)只能放唯一一格(1 路)查找最快、最便宜容易衝突失誤(兩塊搶同一格就互踢)
組相聯(set-associative)同一組裡的 N 格任選(N 路)衝突少很多,折衷之選要平行比對 N 個 tag
全相聯(fully-associative)整個快取任何一格衝突失誤最少要比對全部 tag,昂貴,只用於極小的快取(如 TLB)

實務上,L1/L2/L3 幾乎都用組相聯(常見 4~16 路),取「彈性」與「成本」的平衡。

替換與寫策略

當一組已經滿了、又來一塊新資料,得踢掉一塊。踢哪塊?這就是替換策略(replacement policy)。 最理想的近似是 LRU(Least Recently Used,最久沒用的先踢)——它正是賭時間局部性: 最久沒碰的,最不可能馬上再用。(LRU 你在第 29 章資料結構裡見過,它在快取硬體裡也是同一個思想;真實硬體常用「偽 LRU」來近似,因為精確 LRU 太貴。)

另一個問題:寫入資料時,該怎麼跟下層記憶體同步?這是寫策略(write policy):

  • 寫穿(write-through):寫快取的同時也寫主記憶體。簡單、資料永遠一致,但每次寫都得碰慢記憶體。
  • 寫回(write-back):只先寫快取,把該行標記為「髒(dirty)」;等它要被替換出去時才一次寫回記憶體。 快得多(多次寫只在最後同步一次),代價是複雜度與「資料暫時不一致」(在多核系統裡會引出快取一致性問題)。

現代 CPU 的資料快取幾乎都用寫回,因為它對「同一塊資料反覆寫」這種常見情況快得多。

AMAT:把這一切收進一條公式

快取好不好,最終要看它讓「平均一次記憶體存取要花多久」變成多少。這個指標叫 AMAT(Average Memory Access Time,平均記憶體存取時間):

$$\text{AMAT} = \text{命中時間} + \text{失誤率} \times \text{失誤代價}$$

舉例:L1 命中要 1 個週期,失誤率 5%,失誤後要跑去記憶體拿要花 100 個週期(失誤代價)。則 $\text{AMAT} = 1 + 0.05 \times 100 = 6$ 個週期。注意:失誤率雖然只有 5%,但因為失誤代價極高, 它對平均時間的貢獻($0.05 \times 100 = 5$)遠超過命中本身的 1——這就是為什麼「降低失誤率」這麼值得。

多層快取時,公式會套疊:某層失誤,就遞迴地去問下一層。例如加一層 L2(存取 10 週期、區域失誤率 20%):

$$\text{AMAT} = 1 + 0.05 \times \big(10 + 0.20 \times 100\big) = 1 + 0.05 \times 30 = 2.5 \text{ 週期}$$

多花一層 L2,就把平均存取時間從 6 個週期壓到 2.5 個——這就是階層式快取的威力。

快取對程式效能的真實影響

前面講的都是硬體在背後做的事。現在輪到最重要的問題:身為程式設計師,這跟我有什麼關係? 關係大了。同一個演算法、同樣的計算量,只因為你「走訪記憶體的順序」不同,就可能差好幾倍——因為一種順序對快取友善,另一種把快取打得稀爛。

列優先 vs 行優先:一個改動,數倍差距

C 的二維陣列是列優先(row-major)存放的:a[i][j] 在記憶體裡,是「第 0 列整列、接著第 1 列整列……」連續排列。 所以同一列裡相鄰的 a[i][0], a[i][1], a[i][2]… 在記憶體上緊挨著;但同一行裡的 a[0][j], a[1][j]… 之間,卻隔了「一整列」那麼遠。

下面這段 C 把兩種順序各跑一次,加總同一個大陣列。計算量完全相同,只有迴圈內外層對調:

#include <stdio.h>
#include <time.h>

#define N 4096
static int a[N][N];          /* 64 MB,遠大於快取,逼出快取效應 */

int main(void) {
    /* 列優先:內層跑 j,沿記憶體連續走 → 快取友善 */
    clock_t t1 = clock();
    long long s1 = 0;
    for (int i = 0; i < N; i++)
        for (int j = 0; j < N; j++)
            s1 += a[i][j];
    double row = (double)(clock() - t1) / CLOCKS_PER_SEC;

    /* 行優先:內層跑 i,每步跳一整列 → 快取不友善 */
    clock_t t2 = clock();
    long long s2 = 0;
    for (int j = 0; j < N; j++)
        for (int i = 0; i < N; i++)
            s2 += a[i][j];
    double col = (double)(clock() - t2) / CLOCKS_PER_SEC;

    printf("列優先 row-major: %.3f s\n行優先 col-major: %.3f s\n", row, col);
    return 0;
}
🔑
快取友善(cache-friendly)的程式,可以快好幾倍

實測上,行優先版本通常比列優先版本慢 3~10 倍(視 N、快取行大小、機器而定),儘管兩者做的加法次數一模一樣。 差別純粹來自記憶體存取的順序:列優先吃飽每條快取行,行優先則不斷浪費剛載入的行。 這給你一條極實用的準則:盡量讓你的記憶體存取「沿著記憶體連續前進」——這往往比任何花俏的微優化都有效。

矩陣相乘:快取效應的經典舞台

最經典的例子是矩陣相乘 $C = A \times B$。教科書的天真寫法用 i, j, k 的順序: 

/* 天真版:i-j-k 順序 */
for (int i = 0; i < N; i++)
    for (int j = 0; j < N; j++)
        for (int k = 0; k < N; k++)
            C[i][j] += A[i][k] * B[k][j];   /* B[k][j]:k 變動時跳一整列 → 不友善 */

問題在 B[k][j]:最內層 k 變動時,它在記憶體上每步跳一整列,對 B 的存取極不友善。 只要把迴圈重排成 i-k-j 順序,B 與 C 的最內層存取都沿著列連續走,速度往往快上數倍——而結果完全相同: 

/* 快取友善版:i-k-j 順序(只是換了迴圈巢狀次序!) */
for (int i = 0; i < N; i++)
    for (int k = 0; k < N; k++)
        for (int j = 0; j < N; j++)
            C[i][j] += A[i][k] * B[k][j];   /* 現在 B[k][j]、C[i][j] 都沿列連續 */

高效能函式庫(BLAS)更進一步用分塊(blocking / tiling):把大矩陣切成小到能整塊塞進快取的子塊, 在子塊內反覆運算榨乾時間局部性,再換下一塊。光是「對快取友善」這一招,就能讓矩陣相乘快上一個數量級—— 這也是為什麼 NumPy、PyTorch 底層從不自己寫天真三重迴圈,而是呼叫高度調校過的 BLAS。

指令級平行(ILP):一個時脈做更多事

管線讓多條指令「重疊」,但它的天花板是「每個時脈完成一條」。能不能每個時脈完成好幾條? 能——只要不同指令之間沒有相依,就可以真的同時跑。挖掘程式裡這種「可平行的潛力」, 叫指令級平行(ILP,Instruction-Level Parallelism)。現代 CPU 用三招把 ILP 榨到極致:

  • 超純量(superscalar):CPU 內建多套執行單元(多個 ALU、多個載入/儲存埠), 每個時脈一次發射(issue)好幾條指令。今天的桌機核心常是 4~8 路超純量。
  • 亂序執行(out-of-order execution, OoO):不死守程式碼的先後順序。如果第 3 條指令在等記憶體(卡住), 而第 5 條的資料都齊了,CPU 就先跑第 5 條,別讓執行單元閒著。它靠「保留站」與「重排序緩衝(ROB)」追蹤相依關係, 對外仍維持「看起來像照順序完成」的假象。這需要暫存器重新命名來消除假相依。
  • 推測執行(speculative execution):配合分支預測,CPU 不等分支結果就樂觀地往前衝, 先把猜測路徑上的指令算了;猜對賺到,猜錯就丟棄。它是現代 CPU 高效能的命脈。
⚠️
推測執行的黑暗面:Spectre 與 Meltdown

2018 年,研究者發現推測執行藏著一個世代級的安全漏洞:CPU 在「猜錯後丟棄」那些指令時,雖然會回滾暫存器與記憶體狀態, 卻沒清掉它們在快取裡留下的痕跡。攻擊者於是能透過「測量某個位址是否被快取」這種旁路(side channel), 反推出本不該被讀到的機密(例如別的行程的記憶體)。這就是 Spectre / Meltdown。 它震撼之處在於:漏洞不在任何軟體 bug,而在硬體為了快而做的根本性優化。我們在第七部(資訊安全)談過旁路攻擊的思維—— 這是「效能」與「安全」在最底層直接對撞的著名一役。

還有一條不同的路:VLIW(超長指令字)。它把「找出可平行指令」的工作從硬體丟回給編譯器, 由編譯器事先把多個能並行的運算打包進一條超長指令。Intel 的 Itanium 是著名嘗試,但因為編譯器難以靜態預測動態行為,商業上不算成功—— 這反而印證了亂序執行那種「動態、在硬體裡見機行事」的路線之強大。

更大的平行:執行緒級與資料級

ILP 終究受限於「單一指令流裡能找到多少獨立指令」。要再快,就得跳到更高層次的平行。

  • 同時多執行緒(SMT,即 Intel 的超執行緒 Hyper-Threading):讓一個實體核心 同時跑兩個(或更多)硬體執行緒,共用同一套執行單元。當一個執行緒卡在記憶體等待,另一個正好填補空檔, 把昂貴的執行單元餵飽。對作業系統而言,它看起來像多了一顆「邏輯核心」。
  • 多核(multicore):乾脆在一塊晶片上放好幾顆完整的核心,真正同時跑多個程式/執行緒。 這是過去二十年提升效能的主軸(因為單核頻率撞到功耗牆了)。但多核能帶來多少加速, 又回到了 Amdahl 定律的轄區——程式裡無法平行的那部分,會限制總加速比。這正是第 43 章「平行與並行」的核心戰場。
  • SIMD / 向量指令(Single Instruction, Multiple Data):一條指令同時對一整排資料做同樣的運算。 例如一條 AVX 指令可以一次加 8 對浮點數。這是資料級平行,對影像、訊號、科學運算、機器學習特別有效(x86 的 SSE/AVX、ARM 的 NEON)。
  • GPU:把資料級平行推到極致——數千個小核心,同時對海量資料跑同一段程式。 它正是為「同一招套在幾百萬個像素 / 矩陣元素上」而生,所以圖形(第 39 章)、深度學習都靠它。一句話:CPU 是少數幾個全能的大力士,GPU 是千軍萬馬的小兵團。

從架構看虛擬記憶體與 TLB

我們在第 24 章(作業系統)從「軟體」角度看過虛擬記憶體:每個行程都以為自己獨佔一整片連續的記憶體, 作業系統用分頁表(page table)把虛擬位址翻成實體位址,換來隔離與保護。現在從硬體架構角度補上關鍵一塊。

位址轉譯如果純靠軟體查表會慢死,所以 CPU 內建一個專責硬體:MMU(記憶體管理單元)。 但這裡有個「雞生蛋」的效率陷阱:分頁表自己也住在記憶體裡,所以每存取一筆資料,豈不是要先多一次記憶體存取去查表? 那等於把記憶體存取的成本翻倍。解法你已經很熟了——加一層快取:

🔑
TLB:專門快取「位址翻譯」的快取

TLB(Translation Lookaside Buffer)是 MMU 裡一塊極小、極快、通常全相聯的快取, 專門記住最近用過的「虛擬頁 → 實體頁框」對照。每次存取記憶體,CPU 先問 TLB: 命中就立刻拿到實體位址(幾乎零成本);失誤才去走完整的分頁表(page table walk)再把結果存進 TLB。 因為程式對「分頁」也有強烈的局部性(同一批頁反覆使用),TLB 命中率極高——這又是一次「優化常見情況」與「記憶體階層」思想的完美再現。

看出整章的母題了嗎?遇到慢的東西,就在它前面加一層又快又小的快取,並賭局部性—— 資料用快取、指令用指令快取、頁翻譯用 TLB。同一個思想,在 CPU 的每一個角落反覆上演。

✏️ 練習 48.1 思考

有兩款 CPU 要跑同一個程式(指令數相同)。設計 A:時脈 3 GHz、平均 CPI = 2。設計 B:時脈 4 GHz、平均 CPI = 3。 只看 GHz 的人會說 B 比較快。用效能公式算算看,到底誰快?快多少?這題想讓你體會什麼?

看解答

用 $\text{CPU 時間} = \dfrac{\text{IC} \times \text{CPI}}{\text{頻率}}$。設指令數為 IC(兩邊一樣,可直接比係數):

$$T_A = \frac{\text{IC} \times 2}{3\times10^9} = \text{IC} \times 0.667\,\text{ns}, \qquad T_B = \frac{\text{IC} \times 3}{4\times10^9} = \text{IC} \times 0.75\,\text{ns}$$

所以 $\dfrac{T_B}{T_A} = \dfrac{0.75}{0.667} \approx 1.125$:設計 A 反而快約 12.5%,儘管它的 GHz 比較低。 教訓:頻率只是三個因子之一。B 雖然時脈快,但它的 CPI 太差(也許管線設計較弱、快取較小), 整體反而輸。這就是為什麼「只比 GHz」是外行人的比法。

✏️ 練習 48.2 計算

(a) 一個 5 級管線,要跑 1000 條指令,理想情況(完全無危障)下,相對於「非管線、每條指令花 5 個週期」的版本,加速比是多少? (b) 為什麼這個加速比達不到理論上限 5?(c) 再用 Amdahl 的精神想:如果把管線加深到 10 級,加速比會自動翻倍嗎?

看解答

(a) 非管線:$1000 \times 5 = 5000$ 週期。管線:$k + (n-1) = 5 + 999 = 1004$ 週期。 加速比 $= \dfrac{5000}{1004} \approx 4.98$ 倍。

(b) 因為一開始要花 $k-1=4$ 個週期「灌滿」管線(此時還沒有指令完工),這段啟動開銷讓加速比略低於 5。 指令數越多,這點開銷被攤得越薄,加速比就越逼近 5——但永遠到不了。

(c) 不會自動翻倍。理論上限變成 10,但現實中危障會反咬:管線越深, 分支錯誤預測要清空的指令越多(代價越大),load-use 的相對成本也升高。所以加深管線有報酬遞減, 甚至過深會因危障而變慢——這也是現代 CPU 管線深度停在十幾級、而非上百級的原因。

✏️ 練習 48.3 分析

找出下面這段組語裡的資料危障:標出哪幾對指令有 RAW 相依、哪些靠前饋就能解決、 哪一個是即使有前饋也必須停頓的(提示:load-use)。

1:  lw   r1, 0(r6)      # r1 = 記憶體[r6]
2:  add  r2, r1, r7     # r2 = r1 + r7
3:  sub  r4, r2, r1     # r4 = r2 - r1
4:  lw   r5, 4(r6)      # r5 = 記憶體[r6+4]
5:  or   r8, r9, r10    # r8 = r9 | r10
看解答

逐對找「上一條寫的暫存器,下一條馬上讀」:

  • 1 → 2(都用 r1):這是 load-use 危障!指令 1 是 lw, 它的資料要到 MEM 階段才到手,而指令 2 緊接著就要在 EX 用它。即使有前饋,也得停頓 1 拍(插一個泡泡)。
  • 2 → 3(都用 r2):一般 RAW。指令 2 是 ALU 運算,結果在它 EX 末端就有了, 靠前饋即可解決,不必停頓。(指令 3 也用到 r1,但 r1 早已就緒,沒問題。)
  • 4、5:指令 4 的 r5 沒被緊接的指令使用,指令 5 用的 r9, r10 與前面無關——無危障。

小optimization:編譯器可以把無關的指令 5(or)搬到指令 1 和 2 之間, 正好填掉那個 load-use 泡泡,讓管線一拍都不浪費。這就是「指令排程(instruction scheduling)」。

✏️ 練習 48.4 思考

用你自己的話解釋:為什麼把一個陣列先排序,會讓「加總所有 ≥ 128 的元素」這個迴圈快上好幾倍? (明明計算量一樣。)接著想一個進階問題:如果改成「加總所有元素」(沒有那個 if),排序還會帶來速度差異嗎?為什麼?

看解答

排序為何變快:瓶頸在迴圈裡的條件分支 if (data[i] >= 128)。未排序時,資料隨機, 這個分支的「真/假」也跟著亂跳,分支預測器約一半機率猜錯;每猜錯一次就得清空管線(misprediction penalty)。 排序後,前半段全 < 128、後半段全 ≥ 128,分支變成「先連續假、再連續真」的規律模式,預測器幾乎全猜中, 錯誤預測趨近零,於是省下了海量的管線清空。同樣的計算,因資料可預測性不同而快慢迥異。

進階:沒有 if 的話,排序不會再帶來明顯差異。因為「加總全部元素」沒有任何依資料而定的分支 ——每個元素都照加不誤,沒有東西需要預測。分支預測的勝負舞台消失了,排序與否就無關緊要(走訪順序、快取行為都一樣)。 這恰恰反證了:前面的速度差,真的來自那個分支,而不是別的。

✏️ 練習 48.5 計算

(a) 某迴圈循序走訪一個 int 陣列(每個 int 4 bytes),快取行為 64 bytes,陣列遠大於快取。 假設沒有預取,理論上的失誤率大約是多少?(b) 已知 L1 命中時間 = 1 週期、失誤代價 = 100 週期,用這個失誤率算 AMAT。 (c) 如果改成行優先走訪(每次跳一整列、超過一個快取行),失誤率與 AMAT 又大約變怎樣?

看解答

(a) 一條 64-byte 快取行能裝 $64 / 4 = 16$ 個 int。循序走訪時,每 16 次存取只有第一次失誤 (把整行拉上來),後面 15 次都命中。所以失誤率 $\approx \dfrac{1}{16} = 6.25\%$。這就是空間局部性的紅利。

(b) $\text{AMAT} = 1 + 0.0625 \times 100 = 1 + 6.25 = \mathbf{7.25}$ 週期。

(c) 行優先時,每次存取都跳到不同的快取行,等於每次都失誤(剛載入的行只用了 1 個元素就被棄置), 失誤率趨近 100%。則 $\text{AMAT} \approx 1 + 1.0 \times 100 = \mathbf{101}$ 週期。 對比 (b) 的 7.25,慢了約 14 倍!這就是「列優先 vs 行優先」速度差的數量級來源(實機因為有硬體預取器與多層快取,差距會比這個理論值溫和一些,但方向完全一致)。

✏️ 練習 48.6 實作

親手量一次快取效應。寫一支 C 程式,對一個夠大的二維陣列(例如 4096 × 4096 的 int)分別用 列優先(內層跑 j)與行優先(內層跑 i)各加總一次,用 clock() 計時,印出兩者耗時。 回報你機器上的倍數差距,並解釋為什麼。

看解答

直接用本章「列優先 vs 行優先」那段程式即可。編譯與執行(建議關掉激進最佳化以免迴圈被改寫,並用 -O1 觀察):

gcc -O1 cache_demo.c -o cache_demo
./cache_demo

典型輸出(會因機器而異):

列優先 row-major: 0.042 s
行優先 col-major: 0.310 s

這台機器上行優先慢了約 7 倍。原因:列優先沿記憶體連續走,每條 64-byte 快取行的 16 個 int 都被用到, 失誤率約 1/16;行優先每步跳一整列(4096 個 int = 16 KB),遠超一條快取行,幾乎每次存取都失誤,還可能連 TLB 都頻頻失誤。 計算量完全相同,差距純粹來自記憶體存取順序。

⚠️
量測陷阱

若用 -O2/-O3,編譯器可能發現你只是在加總、把整個迴圈優化掉或自動向量化,讓你量到假數據。 對策:把結果 printf 出來(逼編譯器真的算)、用 volatile、或降低最佳化等級。「正確地量測效能」本身就是一門學問。

✏️ 練習 48.7 計算

一個程式,90% 的執行時間花在一個矩陣運算函式上,其餘 10% 是無法平行也無法加速的雜務。 (a) 如果你把那個矩陣函式靠快取優化 + SIMD 加速了 8 倍,整個程式的總加速比是多少? (b) 如果你能把那個函式加速到「無限快」,總加速比的上限是多少?(c) 這告訴你接下來該把力氣花在哪?

看解答

用 Amdahl 定律 $S = \dfrac{1}{(1-p) + p/s}$,這裡 $p = 0.9$。

(a) $s = 8$:$S = \dfrac{1}{0.1 + 0.9/8} = \dfrac{1}{0.1 + 0.1125} = \dfrac{1}{0.2125} \approx \mathbf{4.7}$ 倍。 注意:把熱點加速 8 倍,整體只快 4.7 倍——因為那 10% 的雜務開始變成新瓶頸。

(b) $s \to \infty$:$S \to \dfrac{1}{1-0.9} = \dfrac{1}{0.1} = \mathbf{10}$ 倍。這是天花板,再怎麼努力都突破不了。

(c) 你已經拿到 4.7 倍、逼近 10 倍上限的一半以上;繼續死磕這個函式,報酬遞減。 此刻更該回頭看那原本只占 10%、現在卻成了新瓶頸的雜務(或思考演算法層面的改變)。 這就是 Amdahl 定律的實務智慧:持續重新評估「現在的瓶頸在哪」,別把力氣浪費在已經不是瓶頸的地方。

✏️ 練習 48.8 挑戰

天真版矩陣相乘用 i-j-k 順序,把它重排成 i-k-j 順序後會明顯變快。 (a) 解釋為什麼:在 i-j-k 裡,哪個陣列的存取對快取最不友善?(b) 換成 i-k-j 後, 為什麼三個陣列的最內層存取都變友善了?(c) 進一步,「分塊(blocking)」又是靠哪一種局部性再榨出效能的?

看解答

回顧 C[i][j] += A[i][k] * B[k][j](列優先儲存)。

(a) 在 i-j-k 裡,最內層是 k。看三個存取:A[i][k] 隨 k 沿列連續(友善); C[i][j] 在最內層不變(超友善);但 B[k][j] 隨 k 變動時,每步跳一整列 (位址 +N 個元素),對 B 的存取最不友善,是主要瓶頸。

(b) 換成 i-k-j 後,最內層變 j:B[k][j] 隨 j 沿列連續(友善); C[i][j] 也隨 j 沿列連續(友善);A[i][k] 在最內層不變(超友善)。 三者全部對快取友善,失誤率大降——而且這只是「換了迴圈巢狀的次序」,結果完全相同。

(c) 分塊(把矩陣切成能整塊塞進快取的小子塊,在子塊內反覆運算後才換下一塊)主要榨的是時間局部性: 讓一塊載入快取的資料,在被踢出去之前被反覆重用到最大化,而不是用一次就丟。 這能把矩陣相乘再加速一個數量級,也是 BLAS 等高效能函式庫的核心招數之一。

把這一章串起來

第 21 章我們造了一台「能跑」的 CPU;這一章,我們看清了工程師如何讓它「跑得飛快」。回顧這趟旅程的路標:

  • 效能公式 $\text{CPU 時間} = \text{IC} \times \text{CPI} \times \text{時脈週期}$ 是一切的尺。GHz 只管其中一格,「只看 GHz」會騙你。Amdahl 定律則告訴你「優化常見情況」。
  • RISC vs CISC 是在 IC、CPI、頻率之間下不同的賭注。RISC 的「簡單定長、好管線」思想勝出——連現代 x86 內心都偷偷是 RISC。
  • 管線讓指令重疊執行,把吞吐量推向「每週期一條」(加速比逼近級數 $k$);但它會被三種危障卡住:結構、資料(靠前饋/停頓解決)、控制(靠分支預測賭方向)。
  • 分支預測對程式設計師可見:難預測的分支(如未排序資料上的 if)會讓迴圈莫名變慢。
  • 記憶體階層與局部性原理是現代效能的另一根支柱:在慢記憶體前面層層疊上又快又小的快取。
  • 快取的所有機制——快取行、對映、LRU 替換、寫回、AMAT——都在賭時間/空間局部性。而它對你真實可見:列優先 vs 行優先、矩陣相乘的迴圈重排,輕鬆差好幾倍。
  • 更上層的平行:ILP(超純量、亂序、推測執行——以及它的黑暗面 Spectre)、執行緒/資料級(SMT、多核、SIMD、GPU)。
  • TLB 是「為頁翻譯量身打造的快取」——再一次驗證了整章的母題:遇到慢,就加一層快取,並賭局部性。

最深的一課也許是這個:現代 CPU 是數十年最佳化的層層結晶,而其中有些結晶(快取、分支預測)會悄悄決定你程式的快慢。 你不必去寫組合語言,但只要心裡裝著「局部性」與「可預測性」這兩把尺,你就能寫出比別人快好幾倍的程式—— 而且常常,只是把迴圈的次序對調一下而已。這正是「懂硬體」對一個軟體工程師的真正價值。
