一個關於「畫得出來」與「算得快」的故事

打開任何一款 3D 遊戲,你的螢幕每秒被重畫六十次,上百萬個三角形被丟到顯卡裡光柵化成像素。 「把這些點和線畫出來」對電腦來說,快得像呼吸。但現在換個問題:遊戲裡那把劍揮過去, 到底有沒有砍到敵人?角色往前走一步,會不會穿牆? 這些問題不是「畫」,而是「算」——而它們,常常比畫一張圖難上許多。

這就是計算幾何(computational geometry)的領地:研究「幾何問題的演算法」。 它關心的不是「這個圖形長什麼樣」,而是「用電腦解一個幾何問題,要花多少步驟?有沒有更聰明的算法?」 一旦你開始用「複雜度」的眼光看待點、線、面,會發現幾何不再只是國中課本裡的圓規直尺, 而是一門和排序、搜尋一樣紮實的演算法學問。

它的應用幾乎無所不在,而且你每天都在用:

  • 電玩與物理引擎:碰撞偵測(collision detection)、視線判斷、可見區域——劍有沒有砍到人,就是線段與多邊形的相交問題。
  • GIS 地圖與導航:你在哪個行政區裡?(點在多邊形內)最近的加油站是哪間?(最近鄰搜尋)兩條道路有沒有交叉?(線段相交)
  • 機器人路徑規劃:在佈滿障礙的空間裡,替機器手臂或自駕車找一條不撞牆的路。
  • CAD 與 3D 列印:零件邊界、布林運算、把模型切片成一層層列印路徑。
  • 電腦視覺與圖學:把一堆掃描點包成輪廓、做網格生成——這裡會直接用到第 39 章的向量運算。

這一章,我們會從最小的磚塊——點與向量——蓋起,一路蓋到凸包、最近點對這些漂亮的演算法。 你會發現,幾乎所有計算幾何的把戲,都站在同一個樸素到不可思議的運算上:外積。

🔑
計算幾何在問什麼

不是「這個幾何圖形是什麼」,而是「解這個幾何問題的演算法,正不正確、快不快」。 「畫得出來」靠的是顯示;「算得快」靠的是演算法設計——分治、掃描線、空間資料結構。這一章, 就是把第 27 章的分治、第 39 章的向量、第 29 章的資料結構,全都拉到幾何的舞台上會合。

幾何的瑞士刀:點、向量與外積

計算幾何的世界裡,最基本的元素是點(point)。在平面上,一個點就是一對座標 $(x, y)$, 在 Python 裡我們乾脆用 tuple 表示:(3, 4)。兩個點相減,得到一個向量(vector)—— 它有方向、有長度,描述「從 A 走到 B 要往哪、走多遠」。線段(segment)則由兩個端點界定。 這些你在第 39 章圖學裡都見過,我們直接拿來用。

內積:量「方向有多一致」

兩個向量 $\vec{u} = (u_x, u_y)$ 與 $\vec{v} = (v_x, v_y)$ 的內積(dot product)定義為:

$$ \vec{u} \cdot \vec{v} = u_x v_x + u_y v_y = |\vec{u}|\,|\vec{v}|\cos\theta. $$

它回傳一個純量,而那個 $\cos\theta$ 告訴你兩向量「方向有多一致」:內積為正,夾角是銳角(同向); 為零,兩向量垂直;為負,夾角是鈍角(逆向)。在幾何裡,內積最常拿來算投影、算夾角、判斷垂直。 但今天真正的主角,是它的雙胞胎兄弟——外積。

外積:一個正負號,定左右

在三維裡,外積(cross product)會吐出一個垂直於兩向量的新向量。但在平面上,我們只關心它的 $z$ 分量—— 一個純量:

$$ \vec{u} \times \vec{v} = u_x v_y - u_y v_x. $$

這個其貌不揚的式子,是整個計算幾何的瑞士刀。它的幾何意義有兩層,層層致命:

  • 大小:$|\vec{u} \times \vec{v}|$ 等於以 $\vec{u}, \vec{v}$ 為兩邊張出的平行四邊形面積(等一下算多邊形面積會用到)。
  • 正負號:它告訴你 $\vec{v}$ 在 $\vec{u}$ 的左邊還是右邊。這就是大名鼎鼎的方向測試(orientation test)。

具體來說,給三個點 $A, B, C$,我們算 $\overrightarrow{AB} \times \overrightarrow{AC}$。把它想成: 你站在 $A$,面朝 $B$ 走過去,然後問「$C$ 在我的左手邊還是右手邊?」——也就是「從 $AB$ 到 $AC$ 是左轉還是右轉」:

$$ \text{cross}(A,B,C) = (B_x - A_x)(C_y - A_y) - (B_y - A_y)(C_x - A_x). $$

  • 結果 > 0:左轉(逆時針, counter-clockwise),$C$ 在 $AB$ 左側。
  • 結果 < 0:右轉(順時針, clockwise),$C$ 在 $AB$ 右側。
  • 結果 = 0:三點共線(collinear)。

翻成 Python,簡短到不可思議——而你接下來的每一個演算法,幾乎都是它的變奏:

def cross(o, a, b):
    """向量 OA 與 OB 的外積(z 分量)。
       > 0:從 OA 到 OB 左轉(逆時針);< 0:右轉(順時針);= 0:三點共線。"""
    return (a[0] - o[0]) * (b[1] - o[1]) - (a[1] - o[1]) * (b[0] - o[0])

def orientation(o, a, b):
    c = cross(o, a, b)
    if c > 0:
        return "左轉 (CCW)"
    elif c < 0:
        return "右轉 (CW)"
    else:
        return "共線"

print(orientation((0, 0), (4, 0), (2, 3)))    # 左轉 (CCW)
print(orientation((0, 0), (4, 0), (2, -3)))   # 右轉 (CW)
print(orientation((0, 0), (4, 0), (8, 0)))    # 共線
🔑
外積是計算幾何的瑞士刀

一個 $(b_x-a_x)(c_y-a_y) - (b_y-a_y)(c_x-a_x)$,只要看正負號,就能回答「左轉還右轉/在哪一側/共不共線」。 線段相交、凸包、點在多邊形內、多邊形面積——本章幾乎每個演算法都靠它。把這把刀磨利,後面全是順水推舟。 它還有個大優點:全程只用加減乘,不用除法、不用開根號,所以用整數座標時可以做到完全精確、零誤差。

核心就是外積取正負號:

def turn(a, b, c):
    v = (b[0] - a[0]) * (c[1] - a[1]) - (b[1] - a[1]) * (c[0] - a[0])
    if v > 0:
        return "LEFT"
    elif v < 0:
        return "RIGHT"
    return "COLLINEAR"

square = [(0, 0), (1, 0), (1, 1), (0, 1)]
n = len(square)
for i in range(n):
    a = square[i]
    b = square[(i + 1) % n]
    c = square[(i + 2) % n]
    print(a, b, c, "→", turn(a, b, c))

「逐角都左轉(或都右轉)」其實是判斷一個多邊形是否為凸多邊形的方法——只要轉向不一致,就有凹角。 而「全部左轉」代表頂點是逆時針排列,這個資訊待會兒在凸包與鞋帶公式裡都會派上用場。

線段相交:兩刀有沒有交叉

有了方向測試,我們來解第一個實用問題:給兩條線段,它們有沒有相交? 這在 GIS(兩條路有沒有交叉口)、電玩(雷射有沒有打到牆)裡天天上演。

直覺是這樣:線段 $\overline{p_1p_2}$ 和 $\overline{p_3p_4}$ 相交,當且僅當—— $p_1$ 和 $p_2$ 分別落在直線 $p_3p_4$ 的兩側,而且 $p_3$ 和 $p_4$ 也分別落在直線 $p_1p_2$ 的兩側。 「落在兩側」正好就是兩次方向測試符號相反。一張圖就懂:

但魔鬼藏在細節:如果某次方向測試是 0(共線),代表一個端點恰好落在另一條線段所在的直線上。 這時還得確認它是不是真的落在「線段範圍內」(而不是延長線上)。這就需要一個輔助判斷 on_segment。 把一般情況和特例都顧到,就得到一個穩健的實作: 

def sign(x):
    """回傳 x 的正負號:1、-1 或 0。"""
    return (x > 0) - (x < 0)

def cross(o, a, b):
    return (a[0] - o[0]) * (b[1] - o[1]) - (a[1] - o[1]) * (b[0] - o[0])

def on_segment(a, b, p):
    """已知 a, b, p 三點共線,判斷 p 是否落在線段 ab 上(含端點)。"""
    return (min(a[0], b[0]) <= p[0] <= max(a[0], b[0]) and
            min(a[1], b[1]) <= p[1] <= max(a[1], b[1]))

def segments_intersect(p1, p2, p3, p4):
    """判斷線段 p1p2 與 p3p4 是否相交(含端點接觸、部分重疊)。"""
    d1 = sign(cross(p3, p4, p1))   # p1 相對於直線 p3p4 在哪一側
    d2 = sign(cross(p3, p4, p2))   # p2 ...
    d3 = sign(cross(p1, p2, p3))   # p3 相對於直線 p1p2 在哪一側
    d4 = sign(cross(p1, p2, p4))   # p4 ...

    # 一般情況:兩端點分別落在對方線段的兩側(符號相反)
    if d1 != d2 and d3 != d4:
        return True

    # 特例:某端點共線且恰好落在對方線段上
    if d1 == 0 and on_segment(p3, p4, p1): return True
    if d2 == 0 and on_segment(p3, p4, p2): return True
    if d3 == 0 and on_segment(p1, p2, p3): return True
    if d4 == 0 and on_segment(p1, p2, p4): return True

    return False

print(segments_intersect((0, 0), (4, 4), (0, 4), (4, 0)))   # True(交叉成 X)
print(segments_intersect((0, 0), (1, 1), (2, 2), (3, 3)))   # False(共線但不重疊)
print(segments_intersect((0, 0), (2, 2), (1, 1), (3, 3)))   # True(共線且重疊)
⚠️
幾何演算法的真正難處,是「特例」

一般情況的程式碼往往三五行就寫完;真正讓人焦頭爛額的,是退化情況(degenerate cases): 端點剛好碰到、線段共線重疊、三點共線、重複的點……上面那段 if 之所以又臭又長,九成都在處理特例。 這是計算幾何反覆出現的主題:美麗的主幹,加上一圈惱人的邊界。寫測試時,務必把這些刁鑽情況都餵進去。

多條線段呢?掃描線登場

如果不是兩條,而是 $n$ 條線段,要找出所有交點呢?最笨的方法是兩兩比對, $\binom{n}{2}$ 對,$O(n^2)$。但聰明的做法叫掃描線(sweep line):想像一條垂直線由左往右掃過整個平面, 它只在「線段的端點」和「交點」處需要停下來處理事件。我們維護一個「目前被掃描線切到的線段」的有序集合, 關鍵洞見是:兩條線段要相交,它們必定曾經在掃描線上『上下相鄰』過。 於是我們只需檢查相鄰線段,不必兩兩都比。這個 Bentley–Ottmann 演算法能在 $O((n + k)\log n)$ 內找出全部 $k$ 個交點。

掃描線是計算幾何的另一把招牌利器(地位等同第 27 章的分治),它把「平面上的二維問題」降維成「一維事件序列」來處理。 我們在凸包、最近點對裡會再嗅到它的氣味。

本章重頭戲:凸包(convex hull)

想像桌上釘了一堆圖釘,你拿一條橡皮筋把它們整個圈起來,然後放手——橡皮筋會「啪」地收緊, 貼上最外圈的那些釘子,圍出一個沒有凹陷的多邊形。這個多邊形,就是這堆點的凸包(convex hull): 包住所有點的最小凸多邊形。

「凸(convex)」的意思是:多邊形裡任取兩點,連線都不會跑到多邊形外面——沒有任何凹角。 凸包是計算幾何的「Hello World」:它是碰撞偵測的粗略包圍盒、是形狀分析的骨架、是許多更難演算法的第一步。 值得我們好好把它寫出來。

Andrew's monotone chain:乾淨俐落的 $O(n\log n)$

建凸包的演算法不少,我們選一個又快又好寫的:Andrew's monotone chain(單調鏈)。它的點子是:

  1. 把所有點依 $x$ 座標排序(同 $x$ 再依 $y$)。光這一步就鎖死了 $O(n\log n)$。
  2. 從左到右掃一遍,建出凸包的下半部(lower hull)。
  3. 從右到左再掃一遍,建出上半部(upper hull)。
  4. 把上下兩條鏈接起來,就是完整的凸包。

每一步掃描時,我們維護一個「目前的鏈」。每加入一個新點,就回頭檢查: 最後那三個點構成的轉彎,是不是「右轉或共線」?如果是,代表上一個點造成了凹陷,把它彈掉, 繼續檢查,直到剩下的都是漂亮的左轉為止。這個「加入、回頭、彈掉」的動作, 像極了第 22、23 章那種用堆疊(stack)處理的手法——而它正是 Graham scan 演算法的靈魂。 

def convex_hull(points):
    """回傳點集的凸包頂點,逆時針排列(不含共線的多餘點)。"""
    # 1. 依 x 再依 y 排序,並去除重複點
    pts = sorted(set(points))
    if len(pts) <= 2:
        return pts                      # 0、1、2 個點:凸包就是它們自己

    def cross(o, a, b):
        return (a[0]-o[0]) * (b[1]-o[1]) - (a[1]-o[1]) * (b[0]-o[0])

    # 2. 下半部:由左掃到右
    lower = []
    for p in pts:
        # 只要最後兩點到 p 不是「嚴格左轉」,就彈掉造成凹陷的點
        while len(lower) >= 2 and cross(lower[-2], lower[-1], p) <= 0:
            lower.pop()
        lower.append(p)

    # 3. 上半部:由右掃回左
    upper = []
    for p in reversed(pts):
        while len(upper) >= 2 and cross(upper[-2], upper[-1], p) <= 0:
            upper.pop()
        upper.append(p)

    # 4. 接起來(各去掉最後一點,因為它是另一條鏈的起點,會重複)
    return lower[:-1] + upper[:-1]

pts = [(0,0),(1,1),(2,2),(2,0),(0,2),(1,2),(3,1),(1,0),(1,3)]
print(convex_hull(pts))
# [(0, 0), (2, 0), (3, 1), (1, 3), (0, 2)]

注意那個 <= 0:用 <= 會把共線的點也彈掉(凸包邊上不留多餘的點); 如果你希望保留共線點,改成 < 0 即可。這種「等號要不要算」的抉擇,又是一個典型的退化情況決策點。

ℹ️
為什麼掃描只花 $O(n)$?

每趟掃描裡,每個點只會被「push 進堆疊一次」,而且「最多被 pop 出來一次」——一旦被彈出就再也不回來。 所以兩個動作合計最多 $2n$ 次,是漂亮的 $O(n)$(這叫攤還分析 amortized analysis,呼應第 26、29 章)。 整個演算法的瓶頸,其實是最前面那一步排序。排序 $O(n\log n)$ + 掃描 $O(n)$ = $O(n\log n)$。

Jarvis march:像「包禮物」一樣,而且會看輸出

還有一個更直觀的凸包演算法,叫Jarvis march,又名禮物包裝法(gift wrapping)。 想像你拿包裝紙裹禮物:從最左邊那個一定在凸包上的點出發,「貼著」最外圈一路繞—— 每一步都找「相對目前點最靠右(轉最少)」的下一個點,直到繞回起點。

它的複雜度很特別:$O(nh)$,其中 $h$ 是凸包上的頂點數。如果這堆點剛好擠成一團、凸包只有少少幾個頂點($h$ 很小), Jarvis march 反而比 monotone chain 還快!這種「執行時間取決於輸出大小」的演算法,叫做 輸出敏感(output-sensitive)。但最壞情況下(所有點都在凸包上,$h = n$),它退化成 $O(n^2)$,比單調鏈慢。

💡
沒有「最好」的凸包演算法,只有「最適合」的

點很多、凸包頂點也多 → 用 monotone chain / Graham scan($O(n\log n)$,穩)。 點很多、但凸包頂點極少 → Jarvis march($O(nh)$)可能更快。 還有 Chan's algorithm 把兩者優點結合成 $O(n\log h)$。選演算法,要看你的資料長什麼樣——這是貫穿整本書的工程素養。

import random

def cross(o, a, b):
    return (a[0]-o[0])*(b[1]-o[1]) - (a[1]-o[1])*(b[0]-o[0])

def verify_hull(points, hull):
    n = len(hull)
    # 1. 凸性:逆時針走,每個轉角都不右轉
    for i in range(n):
        if cross(hull[i], hull[(i+1) % n], hull[(i+2) % n]) < 0:
            return False
    # 2. 所有點都在凸包內或邊上:對每條邊,點都不在右側
    for p in points:
        for i in range(n):
            if cross(hull[i], hull[(i+1) % n], p) < 0:
                return False
    return True

pts = [(random.randint(0, 1000), random.randint(0, 1000)) for _ in range(200)]

隨機點均勻散布在正方形裡時,凸包頂點數的期望大約是 $O(\log n)$ 級別——所以 200 個點往往只有十幾個頂點。 這也說明了為什麼輸出敏感的演算法在實務上常常很划算。

點在多邊形內嗎?射線法

下一個經典問題,GIS 與遊戲都離不開:給一個多邊形和一個點,這個點在多邊形裡面還是外面? (你點在地圖上,系統判斷你在哪個行政區;子彈打中,判斷有沒有落在敵人輪廓內。)

有個漂亮到像魔術的解法:射線法(ray casting)。從待測點往任意方向(通常往右)射出一條射線, 數它穿過多邊形邊界幾次:

  • 穿過奇數次 → 點在裡面。
  • 穿過偶數次(含 0 次) → 點在外面。

直覺是這樣:你從很遠的外面沿射線走回待測點,每穿過一次邊界,你的「裡外身分」就翻轉一次。 從外面出發,翻奇數次就會落在裡面。這就是著名的奇偶規則(even-odd rule),也叫喬丹曲線定理的計算版。

實作只要對每條邊判斷:它有沒有「跨越」通過待測點的水平線?若有,算出交點的 $x$,看它在待測點的右邊嗎?

def point_in_polygon(pt, poly):
    """射線法判斷點 pt 是否在多邊形 poly(頂點依序排列)內部。"""
    x, y = pt
    n = len(poly)
    inside = False
    j = n - 1                              # j 從最後一點開始,(j, i) 構成一條邊
    for i in range(n):
        xi, yi = poly[i]
        xj, yj = poly[j]
        # 這條邊是否「上下橫跨」待測點所在的水平高度 y?
        if (yi > y) != (yj > y):
            # 算出邊與水平線 y 的交點 x 座標
            x_cross = xi + (y - yi) * (xj - xi) / (yj - yi)
            if x < x_cross:                # 交點在 pt 右邊 → 射線穿過這條邊
                inside = not inside        # 翻轉裡外身分
        j = i
    return inside

square = [(0, 0), (4, 0), (4, 4), (0, 4)]
print(point_in_polygon((2, 2), square))   # True
print(point_in_polygon((5, 2), square))   # False
print(point_in_polygon((2, 5), square))   # False

那句 (yi > y) != (yj > y) 很精巧:它一口氣處理了「邊是往上還往下」並排除了水平邊, 還順手化解了「射線剛好穿過某個頂點」這個惱人特例(因為兩端用了嚴格不等與非嚴格的不對稱比較)。 計算幾何就是這樣,把退化情況悄悄藏進一個巧妙的比較裡。

多邊形面積:鞋帶公式

要算一個任意多邊形的面積——可能凹凹凸凸、奇形怪狀——有個優雅到不可思議的公式,叫鞋帶公式(shoelace formula), 因為它計算時兩兩交叉相乘的樣子,像在綁鞋帶。它的根源,正是我們的老朋友:外積等於平行四邊形面積。

想法是:把多邊形拆成許多以原點為頂點的三角形(有些「正向」、有些「負向」),用外積把它們的有號面積加起來, 正負會自動抵消,只留下多邊形本身。對頂點依序 $(x_0,y_0), (x_1,y_1), \dots, (x_{n-1},y_{n-1})$:

$$ A = \frac{1}{2}\left| \sum_{i=0}^{n-1} \left( x_i\,y_{i+1} - x_{i+1}\,y_i \right) \right|, \qquad (x_n, y_n) \equiv (x_0, y_0). $$

括號裡每一項 $x_i y_{i+1} - x_{i+1} y_i$,正是相鄰兩頂點向量的外積——也就是原點與這條邊張出的三角形面積的兩倍(有號)。 加總、取絕對值、除以 2,就是面積。神奇的是:它對凹多邊形一樣成立,而且不必先把多邊形三角化。 

def polygon_area(poly):
    """鞋帶公式:回傳多邊形(頂點依序排列)的面積。"""
    n = len(poly)
    s = 0
    for i in range(n):
        x1, y1 = poly[i]
        x2, y2 = poly[(i + 1) % n]      # 下一個頂點,最後一個接回第一個
        s += x1 * y2 - x2 * y1          # 累加外積(有號)
    return abs(s) / 2

def signed_area(poly):
    """有號面積:正 = 逆時針排列,負 = 順時針。"""
    n = len(poly)
    s = sum(poly[i][0] * poly[(i+1) % n][1] - poly[(i+1) % n][0] * poly[i][1]
            for i in range(n))
    return s / 2

square = [(0, 0), (4, 0), (4, 4), (0, 4)]
print(polygon_area(square))      # 16.0

triangle = [(0, 0), (4, 0), (0, 3)]
print(polygon_area(triangle))    # 6.0

print(signed_area(square))       # 8.0 → 正 → 逆時針
print(signed_area(square[::-1])) # -8.0 → 負 → 順時針

附帶一個漂亮的紅利:有號面積(不取絕對值)的正負號,直接告訴你頂點是逆時針(正)還是順時針(負)排列的—— 這在判斷多邊形朝向、法向量方向時非常好用,呼應第 39 章圖學裡的面朝向問題。

最近點對:分治法的再次勝利

平面上撒了 $n$ 個點,要找出距離最近的那一對。這在天文(找雙星)、航管(防撞)、 群聚分析裡都是核心。最笨的辦法,當然是每兩點都量一次距離,$\binom{n}{2}$ 對,$O(n^2)$: 

import math

def dist(a, b):
    return math.hypot(a[0] - b[0], a[1] - b[1])   # √((Δx)² + (Δy)²)

def closest_pair_brute(points):
    best, pair = float('inf'), None
    n = len(points)
    for i in range(n):
        for j in range(i + 1, n):
            d = dist(points[i], points[j])
            if d < best:
                best, pair = d, (points[i], points[j])
    return best, pair

$n$ 一萬,就是上億次比較。能更快嗎?能——而且用的正是第 27 章那把利劍:分治法。

分治的三步:分、解、合(那條中線帶狀區是靈魂)

  1. 分解:依 $x$ 座標排序後,用一條垂直中線把點切成左半與右半。
  2. 解決:遞迴求左半的最近點對距離 $d_L$、右半的最近點對距離 $d_R$。令 $d = \min(d_L, d_R)$。
  3. 合併:最近的一對,可能橫跨中線(一點在左、一點在右)。這一步是全部關鍵。

合併看似又要兩兩比對橫跨的點,豈不又是 $O(n^2)$?關鍵的幾何洞見來了: 橫跨的那一對如果比 $d$ 還近,它們必定都落在中線左右各 $d$ 寬的一條帶狀區(strip)裡。 更妙的是,把帶狀區裡的點依 $y$ 排序後,每個點只需要和它後面至多 7 個點比較就夠了—— 因為在一個 $d \times 2d$ 的小框裡,塞不下太多彼此距離都 $\ge d$ 的點(鴿籠原理)。於是合併只花 $O(n)$。

把這套邏輯寫成程式。注意我們事先準備好「依 $x$ 排序」與「依 $y$ 排序」兩份點,以維持每層 $O(n)$:

import math

def dist(a, b):
    return math.hypot(a[0] - b[0], a[1] - b[1])

def closest_pair(points):
    """分治法找最近點對,O(n log n)。假設點皆相異。"""
    px = sorted(points)                          # 依 x 排序
    py = sorted(points, key=lambda p: p[1])      # 依 y 排序

    def rec(px, py):
        n = len(px)
        if n <= 3:                               # 基底:點少,直接暴力
            best, pair = float('inf'), None
            for i in range(n):
                for j in range(i + 1, n):
                    d = dist(px[i], px[j])
                    if d < best:
                        best, pair = d, (px[i], px[j])
            return best, pair

        mid = n // 2
        midx = px[mid][0]
        Lx, Rx = px[:mid], px[mid:]              # 依 x 切成左右兩半
        Lset = set(Lx)
        Ly = [p for p in py if p in Lset]        # 維持依 y 排序的左右子集
        Ry = [p for p in py if p not in Lset]

        dl, pl = rec(Lx, Ly)                     # 遞迴左半
        dr, pr = rec(Rx, Ry)                     # 遞迴右半
        d, pair = (dl, pl) if dl < dr else (dr, pr)

        # 合併:檢查橫跨中線、且離中線 < d 的點(已依 y 排序)
        strip = [p for p in py if abs(p[0] - midx) < d]
        for i in range(len(strip)):
            j = i + 1
            # 關鍵:y 差距已 >= d 就不必再比,所以內層只跑常數次
            while j < len(strip) and strip[j][1] - strip[i][1] < d:
                dd = dist(strip[i], strip[j])
                if dd < d:
                    d, pair = dd, (strip[i], strip[j])
                j += 1
        return d, pair

    return rec(px, py)

pts = [(2, 3), (12, 30), (40, 50), (5, 1), (12, 10), (3, 4)]
d, pair = closest_pair(pts)
print(f"最近距離 = {d:.4f},點對 = {pair}")   # 最近距離 ≈ 1.4142,點對 ((2, 3), (3, 4))

它的遞迴關係,正是第 27 章那條招牌:$T(n) = 2\,T(n/2) + O(n)$,套主定理 Case 2,得 $T(n) = O(n\log n)$。 和合併排序一模一樣的形狀!從 $O(n^2)$ 砍到 $O(n\log n)$,在 $n$ 上萬時就是「跑不動」與「秒回」的天壤之別。

🔑
分治的威力,來自「合併能不能變便宜」

最近點對最初看似無解——合併好像非得兩兩比不可。是那個「帶狀區裡每點只需比常數個鄰居」的幾何觀察, 才把合併壓到 $O(n)$,讓整體掉到 $O(n\log n)$。這呼應第 27 章的鐵律:分治演算法快不快,十之八九取決於合併那一步設計得多巧。 幾何的洞見,在這裡直接變成了複雜度的勝利。

把點「組織」起來:空間資料結構

前面我們解的都是「一次性」的問題。但實務上常常是:點集固定不動,然後你要反覆查詢—— 「離這裡最近的餐廳是哪間?」「這個矩形框裡有哪些點?」如果每次都掃過全部 $n$ 個點,太慢。 這時要先花一點力氣把點組織成空間資料結構,之後每次查詢就快得多。這正好把第 29 章的樹, 升級到二維、三維的世界。

  • k-d 樹(k-d tree):把第 29 章的二元搜尋樹推廣到 $k$ 維。它輪流用不同維度切割空間: 第一層用 $x$ 切、第二層用 $y$ 切、第三層又回到 $x$……每個節點代表一塊矩形區域。 建好後,範圍查詢(range query)與最近鄰搜尋(nearest neighbor search)平均都能做到 $O(\log n)$ 級別。
  • 四元樹(quadtree):另一種思路——把一個正方形區域,只要裡頭點太多就切成四個小正方形, 遞迴下去。它特別適合分布不均的點(地圖上有的地方密、有的地方空),也廣泛用於影像壓縮、碰撞偵測的空間分割。 三維版本叫八元樹(octree)。

這些結構不只是幾何玩具:第 38 章的 k 最近鄰(k-NN)分類器,要替每個查詢點找最近的 $k$ 個訓練樣本, 若用 k-d 樹,就能避免每次都掃過全部資料,在維度不太高時大幅加速。空間資料結構,是「幾何 × 資料結構」的甜蜜交會點。

Voronoi 圖與 Delaunay 三角化:一瞥

最後,留兩個計算幾何最美的概念給你當甜點。給平面上一堆「站點(sites)」, Voronoi 圖(Voronoi diagram)把整個平面切成一塊塊區域,讓每塊區域裡的所有位置,都離它對應的那個站點最近。 想知道「離我最近的郵局是哪間」?你站的地方落在哪塊 Voronoi 區域,答案就是那塊的站點。它能用掃描線在 $O(n\log n)$ 內建好。

而它的對偶(dual)——把「共享邊界的相鄰站點」連起來——就得到Delaunay 三角化(Delaunay triangulation)。 Delaunay 有個迷人性質:它盡量避免又瘦又長的三角形(最大化最小角),所以是地形建模、有限元素網格生成、 3D 重建的基石。這對「孿生兄弟」是計算幾何皇冠上的寶石,值得你日後專門深究。

當幾何遇上浮點:穩健性的惡夢

到這裡,演算法都很漂亮。但有個幽靈一直潛伏在底下,現在該揭開了:浮點誤差。 還記得第 2 章說過,電腦用浮點數表示小數時,$0.1 + 0.2$ 不會剛好等於 $0.3$ 嗎?在計算幾何裡, 這種微小誤差會釀成邏輯災難。

想想我們的瑞士刀——方向測試。它靠 cross(...) 的正負號做判斷。如果三點幾乎共線, 理論上 cross 應該是 0,但浮點算出來可能是 $-10^{-16}$ 這種「假的負數」。於是程式「以為」是右轉, 做出錯誤決策。在凸包裡,這可能讓某個本該留下的點被誤刪、或反過來——輕則結果略歪, 重則演算法邏輯打結、無限迴圈、或崩潰。

⚠️
幾何演算法最隱蔽的 bug,來自浮點與退化情況

「在紙上百分百正確」的演算法,到了浮點世界可能在退化情況(三點共線、四點共圓、重複點、垂直線)上悄悄出錯。 這就是穩健性(robustness)問題。常見的對策有三種:
① 用整數座標:外積只用加減乘,整數運算完全精確、零誤差——能用整數就別用浮點(回想本章一開始就強調過這點)。
② 加容差(epsilon):把 x == 0 改成 abs(x) < 1e-9。簡單,但 epsilon 多大很難拿捏,治標不治本。
③ 精確幾何運算:用任意精度有理數或專門的「精確謂詞」函式庫(如 CGAL)。最可靠,但較慢、較複雜。
這正是第 2 章浮點、與第 55 章數值方法在幾何裡的迴響:真實世界的數,不是數學課本裡那種無限精確的數。

所以,當你日後寫(或除錯)幾何程式,腦中要常駐一個警鈴:「這裡會不會踩到退化情況?浮點誤差會不會讓我的正負號判斷翻車?」 計算幾何的高手,有一半的功力,花在馴服這頭浮點巨獸上。

把這一章串起來

我們從「畫得出來不等於算得快」出發,走完了計算幾何的一條主幹道。回顧這趟旅程的路標:

  • 外積是整章的瑞士刀:一個正負號,定左轉、右轉、共線;它的大小,又是平行四邊形面積。本章幾乎每個演算法都靠它。
  • 線段相交用四次方向測試判斷「互相跨越」;多線段交點則交給掃描線降維處理。
  • 凸包(橡皮筋直覺)用 Andrew's monotone chain 在 $O(n\log n)$ 建好;Jarvis march 則是輸出敏感的 $O(nh)$。
  • 射線法用奇偶穿越數判斷點在多邊形內;鞋帶公式用外積累加算任意多邊形(含凹)的面積,還順手告訴你頂點朝向。
  • 最近點對用分治(呼應第 27 章)把 $O(n^2)$ 砍成 $O(n\log n)$,靈魂在「中線帶狀區只需比常數個鄰居」。
  • k-d 樹、四元樹把點組織起來(呼應第 29 章),讓範圍查詢與最近鄰搜尋(呼應第 38 章 k-NN)變快。
  • Voronoi 圖與 Delaunay 三角化是這門學問的皇冠寶石,應用遍及最近設施、網格生成、地形建模。
  • 浮點穩健性是潛伏的惡夢:能用整數就別用浮點,小心每一個退化情況。

看見了嗎?計算幾何不是一門孤立的學問,它是一個會合點:幾何的直覺(外積、向量)、 演算法的範式(分治、掃描線)、資料結構的組織力(樹),在這裡全部交織成一張網。 當你能把一個「看得見」的幾何問題,翻譯成「算得快」的演算法,你就真正跨進了這門優雅學問的大門。
