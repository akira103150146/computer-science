什麼是計算、程式與演算法

電腦科學(Computer Science)其實不是關於電腦的科學,正如天文學不是關於望遠鏡的科學。那它到底是關於什麼?

  • 命令式知識 vs 宣告式知識,以及電腦只懂哪一種

一個關於食譜的故事

想像你阿嬤要把她的招牌滷肉飯食譜傳給你。她有兩種做法。

第一種,她說:「滷肉飯嘛,就是那個香香、油亮油亮、配飯很下飯的東西。」 這描述了滷肉飯是什麼——但你照這句話,一輩子也做不出來。

第二種,她給你一張紙:「五花肉切丁,熱油爆香紅蔥頭,下肉拌炒到變色, 加醬油兩大匙、米酒一匙、冰糖少許、水蓋過肉,小火燉一小時……」 這描述了怎麼做——照著做,你就能複製出那碗滷肉飯。

這兩種知識,正是理解整個計算機科學的起點。

🔑
兩種知識

宣告式知識(declarative knowledge):陳述「事實」與「是什麼」。例如:「2 的平方根,是一個平方後等於 2 的正數。」
命令式知識(imperative knowledge):描述「如何做」的步驟。例如:「要算 2 的平方根,先猜一個數,然後不斷用某個公式修正這個猜測……」

電腦,說穿了,是一台只懂命令式知識的機器。你沒辦法告訴電腦「請給我平方根的定義」就期待它算出答案; 你必須給它一連串它能一步步執行的明確指令。而這串「如何做」的精確步驟,就叫做——

演算法:把「怎麼做」說清楚

演算法(algorithm)是為了解決某個問題,而設計的一組有限、明確、可執行的步驟。 它就像阿嬤那張食譜紙。一個夠格的演算法,必須滿足幾個條件:

  • 明確(definite):每一步都清楚無歧義。不能寫「適量」、「差不多」。
  • 有限(finite):必須在有限步驟內結束,不能跑到天荒地老。
  • 有效(effective):每一步都簡單到「機器能照做」。
  • 有輸入與輸出:吃進資料,吐出結果。
📜
「演算法」這個字怎麼來的

它源自九世紀波斯數學家 al-Khwārizmī(花拉子米)的名字。他寫的數學著作傳入歐洲後, 他的拉丁化名字 Algoritmi 就慢慢演變成今天的 algorithm。同一本書的書名 al-jabr, 則變成了「代數(algebra)」。一個人,貢獻了兩個 CS/數學的核心字彙。

演算法不是程式

這裡有個關鍵區別,很多人一輩子搞混:

演算法是「想法」,程式是「想法的某種具體寫法」。

「把一疊考卷按分數由高到低排好」是一個演算法概念;你可以用中文描述它、可以畫流程圖、 可以用 Python 寫、也可以用 C 寫。同一個演算法,可以有無數種程式實作。 程式(program),就是用某種程式語言(programming language) 把演算法寫成電腦看得懂的形式。

來看一個最簡單的演算法:找出一串數字裡的最大值。先用「人話」寫:

演算法:找最大值
輸入:一串數字
1. 假設第一個數字就是目前最大的(先記住它)
2. 從第二個數字開始,逐一檢查每個數字:
      如果這個數字比「目前最大」還大,就更新「目前最大」為它
3. 全部看完後,「目前最大」就是答案
輸出:最大值

同一個演算法,用 Python 寫出來:

def find_max(numbers):
    largest = numbers[0]          # 步驟 1:先假設第一個最大
    for n in numbers[1:]:         # 步驟 2:從第二個開始逐一檢查
        if n > largest:
            largest = n           # 發現更大的,就更新
    return largest                # 步驟 3:回傳結果

print(find_max([3, 9, 2, 7, 5]))  # 輸出 9

同一個演算法,再用 C 寫一次。注意「想法」完全一樣,只是語言的「腔調」不同:

#include <stdio.h>

int find_max(int numbers[], int n) {
    int largest = numbers[0];          /* 步驟 1 */
    for (int i = 1; i < n; i++) {      /* 步驟 2 */
        if (numbers[i] > largest) {
            largest = numbers[i];      /* 更新 */
        }
    }
    return largest;                    /* 步驟 3 */
}

int main(void) {
    int data[] = {3, 9, 2, 7, 5};
    printf("%d\n", find_max(data, 5)); /* 輸出 9 */
    return 0;
}
🔍
觀察:同一個靈魂,兩副軀殼

Python 版簡潔,因為它幫你隱藏了很多細節(例如不用宣告型別、不用告訴它陣列多長)。 C 版囉嗦,因為它要求你把一切講清楚(int 型別、陣列長度 n)。 但兩者背後的演算法一模一樣。學會分辨「演算法」與「語言細節」,你就跨過了初學者最大的一道坎。

✏️ 練習 1.1 思考

上面的 find_max 演算法,如果輸入是一個空的串列(裡面一個數字都沒有),會發生什麼事? 這算不算違反了演算法「明確」的要求?你會怎麼修正?

看解答

會出錯!numbers[0] 會試圖取「第一個元素」,但空串列沒有第一個元素, Python 會丟出 IndexError,C 則是讀到未定義的記憶體(更危險,可能不報錯但給你垃圾值)。

這正是「邊界情況(edge case)」——演算法設計時最容易被忽略、實務上最常出 bug 的地方。一個嚴謹的版本應該先處理它:

def find_max(numbers):
    if len(numbers) == 0:
        raise ValueError("不能對空串列找最大值")
    largest = numbers[0]
    for n in numbers[1:]:
        if n > largest:
            largest = n
    return largest

「永遠先想邊界情況」會是貫穿本書的一句忠告。

那……「計算」到底是什麼?

現在我們可以回答開頭的大哉問了。當電腦執行一個程式,它做的事情,本質上是: 依照一連串明確的規則,把資訊從一種狀態,一步步轉換成另一種狀態。 這個轉換的過程,就是計算(computation)。

「找最大值」是把「一串數字」這個狀態,轉換成「一個數字」這個狀態。 把照片變黑白、計算薪資、預測天氣、讓遊戲角色移動……表面千差萬別, 骨子裡全都是「依規則轉換資訊」。這就是為什麼一台機器能做這麼多看似無關的事—— 因為它們在最底層,都是同一件事:計算。

🔑
電腦科學不是關於電腦

計算這個概念,不依賴任何特定機器。你可以用算盤計算、用紙筆計算、 用矽晶片計算,甚至(理論上)用骨牌計算。電腦只是目前「最快、最便宜」的計算工具。 電腦科學真正研究的,是「計算」本身:什麼問題可以被計算?要花多少步驟? 有沒有更聰明的方法?這也是為什麼有人說:「Computer science is no more about computers than astronomy is about telescopes.」(這句話常被歸給 Dijkstra。)

計算的極限:有些事,電腦永遠算不出來

這聽起來很科幻,但它是被嚴格證明的數學事實。1936 年, 艾倫・圖靈(Alan Turing)提出一個極簡的抽象機器(後人稱 圖靈機 Turing machine),用它精確定義了「什麼叫做可計算」。 然後他證明了:存在一些問題,沒有任何演算法能解決它們—— 最著名的就是「停機問題(halting problem)」:你無法寫出一個萬能程式, 去判斷「任意給定的程式會不會無限跑下去」。

現在你只需要記住這個震撼的結論:計算有它的疆界。 這個疆界長什麼樣子,我們會在第十二部「計算理論」裡仔細描繪。 現在,先讓我們回到疆界之內,學會駕馭計算的力量。

抽象:CS 最強大的武器

當你開車時,你需要懂引擎內部的燃燒原理嗎?不用。你只要會用方向盤、油門、煞車—— 這三個介面(interface)。引擎多複雜,都被「藏」在引擎蓋底下了。 這種「隱藏複雜細節、只露出簡單介面」的手法,就是抽象(abstraction), 而它是人類對付複雜度的終極武器,也是整個 CS 的靈魂。

回頭看 find_max。一旦寫好它,你以後要找最大值,只要寫 find_max(data), 完全不用再去想裡面那個迴圈是怎麼跑的。你把「找最大值」這個複雜過程, 抽象成了一個簡單的名字。這就是函式(function)的威力——它是程式設計中最基本的抽象工具。

抽象會一層一層往上疊。電晶體被抽象成邏輯閘,邏輯閘被抽象成 CPU, 機器碼被抽象成組合語言,組合語言被抽象成 C,C 被抽象成 Python…… 每一層都「站在前一層的肩膀上」,讓你不必每次都從電子的運動開始想。 這整座抽象的高塔,就是接下來這本書要帶你逐層攀爬的。

💡
學習時善用抽象

當你學一個新東西卻被細節淹沒時,問自己:「我現在這一層,真的需要懂下一層的細節嗎?」 通常答案是「暫時不用」。先把這一層當黑盒子用熟,建立直覺,之後再掀開蓋子。 這種「分層理解」的能力,是高效自學的關鍵。

讓程式「做決定」:邏輯與分支

到目前為止,我們的程式都是「直線」執行的。但真正有趣的計算,需要程式能根據情況做出不同反應。 這就要靠條件判斷(conditional)。來寫一個會「思考」的小程式:判斷一個數是奇數還是偶數。 

def odd_or_even(n):
    if n % 2 == 0:        # % 是「取餘數」。除以 2 餘 0 就是偶數
        return "偶數"
    else:
        return "奇數"

print(odd_or_even(10))   # 偶數
print(odd_or_even(7))    # 奇數

% 這個運算子叫模數(modulo),回傳除法的餘數。 「除以 2 餘 0」是判斷偶數最常用的技巧,你會在無數程式裡看到它。 if / else 則讓程式產生了「岔路」:條件成立走一條路,不成立走另一條。 有了分支與迴圈(我們在 find_max 用過的 for), 程式就有了表達任何演算法的能力。

✏️ 練習 1.2 實作

寫一個 Python 函式 fizzbuzz(n),印出 1 到 n 的每個數字,但是:

  • 是 3 的倍數,印 Fizz
  • 是 5 的倍數,印 Buzz
  • 同時是 3 和 5 的倍數,印 FizzBuzz
  • 其他情況,印那個數字本身

這題叫「FizzBuzz」,是業界惡名昭彰的面試篩選題——簡單,卻能篩掉不少其實不會寫程式的人。

看解答

關鍵在判斷順序:必須先檢查「同時是 3 和 5 的倍數」,否則它會先被 Fizz 攔截。

def fizzbuzz(n):
    for i in range(1, n + 1):     # range(1, n+1) 產生 1, 2, ..., n
        if i % 3 == 0 and i % 5 == 0:
            print("FizzBuzz")
        elif i % 3 == 0:
            print("Fizz")
        elif i % 5 == 0:
            print("Buzz")
        else:
            print(i)

fizzbuzz(15)

進階思考:「同時是 3 和 5 的倍數」等價於「是 15 的倍數」。你能用這個觀察讓程式更簡潔嗎? 另外,有沒有辦法完全不用巢狀的 if/elif?(提示:用字串拼接。) 這些「同一題的不同寫法」,正是培養品味的好練習。

把這一章串起來

我們從一碗滷肉飯出發,抵達了計算的疆界。回顧這趟旅程的幾個路標:

  • 知識有兩種:宣告式(是什麼)與命令式(怎麼做)。電腦只懂後者。
  • 演算法是解決問題的明確步驟(那張食譜),程式是它用某種語言寫成的具體形式(照食譜煮出的那碗飯)。同一個演算法可以有無數種程式實作。
  • 計算的本質,是「依規則一步步轉換資訊」。它不依賴特定機器,而且——有些問題,計算永遠解不了。
  • 抽象是 CS 對付複雜度的終極武器。函式是你的第一個抽象工具,而整個電腦世界是一座層層相疊的抽象高塔。
  • 有了條件分支與迴圈,程式就能表達任何演算法。

下一章,我們要正式坐進 Python 的駕駛艙,從變數、型別、運算子開始, 把「命令式知識」一條條教給電腦。系好安全帶——這就是你成為計算機科學家的真正起點。
