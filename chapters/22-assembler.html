從「讀懂」到「自動翻譯」

第 21 章的結尾,你已經是個能與機器對話的人了。你能盯著 1110000010010000 這串 0 與 1, 把它拆成 comp / dest / jump,讀出它的意思是「D=D+A」;反過來,你也能把心裡的想法寫成 @i、D=M、0;JMP 這樣的組合語言。

但你大概也已經受夠了。手寫組合語言的迴圈,還得自己數位址——「跳回去」要跳到第幾條指令? 用到一個變數 i,它該擺在記憶體第幾格?每改一行,後面所有位址都得重算。 更別說最後還要逐條把助記符對照成 16 個位元,抄錯一位,整個程式就壞了。

這正是 1950 年代第一批程式設計師的真實處境。他們很快就想到一個念頭: 這種機械、無腦、照規則替換的苦工,不正是電腦最擅長的事嗎? 既然第 21 章我們悟到了「指令也只是數字」,那「把組合語言翻成機器碼」就只是一個 「讀進一堆文字、吐出一堆數字」的普通程式而已。這個程式,就叫做組譯器(assembler)。

🔑
第一個語言翻譯器

組譯器是人類寫出的第一種、也是最簡單的「語言翻譯器(language translator)」—— 一種「吃進一種語言、吐出另一種語言」的程式。它把組合語言(assembly language)(給人看的) 翻成機器語言(machine language)(給 CPU 看的)。 往上,還有更高階的翻譯器:把 C 翻成組語的編譯器(compiler)、把位元組碼翻成機器碼的 VM 翻譯器……它們全都建立在同一個洞見上,只是層級更高、活更難。 學會這一章,你就掌握了所有語言處理器的共通骨架。

助記符與符號:給位元取個好名字

組合語言到底「友善」在哪?說穿了,就兩件事:助記符與符號。

助記符(mnemonic)是好記的英文縮寫,代替難記的位元組合。 與其寫 1110000010010000,我們寫 D=D+A;與其背 jump 欄位的 001, 我們寫 JGT(Jump if Greater Than,大於就跳)。D、M、+、JMP ——這些都是人腦能瞬間理解、機器碼欄位的「別名」。

符號(symbol)更進一步,是替「數字位址」取的名字。比起 @24576, 寫 @KBD(鍵盤)一眼就懂用途;比起記住「迴圈起點在第 6 條指令、所以寫 @6」, 寫 @LOOP 既好讀、又不會因為你在前面插了一行而失準。

回憶一下第 21 章那兩種 Hack 指令,它們的組語寫法就是助記符與符號的組合:

  • A 指令:@值 或 @符號。例如 @21(載入數值 21)、@LOOP(載入 LOOP 所代表的位址)。
  • C 指令:完整格式是 dest = comp ; jump,其中 dest= 與 ;jump 都可省略。例如 D=D+A、M=M+1、D;JGT、0;JMP。

組譯器的工作,就是把這些好記的文字,一行一行變回 CPU 唯一聽得懂的 16 位元數字。 而且因為組語和機器碼是一對一的(一行組語剛好對應一條機器指令),這個翻譯出奇地直接—— 我們先來看它最單純的那一面。

組譯的本質:其實大多是「查表替換」

先把符號放一邊(假裝 @ 後面永遠是數字、程式裡沒有任何標籤)。在這個簡化世界裡, 組譯幾乎沒有任何「智慧」可言,它只是三件笨事:

  1. 看一行是 A 指令還是 C 指令(看開頭是不是 @)。
  2. A 指令:把 @ 後面的數字轉成 15 位元二進位,前面補一個 0。
  3. C 指令:把它拆成 dest、comp、jump 三段,各自查表換成位元,前面接上 111。

那三張「表」,正是第 21 章你已經見過的 comp / dest / jump 對照表。組譯器把它們直接寫成 Python 字典—— 這就是整個組譯器的「code(轉碼)模組」: 

# comp 欄位對照表:鍵是助記符,值是「a 位 + 6 個 ALU 控制位元」共 7 位
# (這 6 個位元,字面上就是第 20 章那顆 ALU 的 zx,nx,zy,ny,f,no)
COMP = {
    "0":  "0101010", "1":  "0111111", "-1": "0111010",
    "D":  "0001100", "A":  "0110000", "M":  "1110000",
    "!D": "0001101", "!A": "0110001", "!M": "1110001",
    "-D": "0001111", "-A": "0110011", "-M": "1110011",
    "D+1":"0011111", "A+1":"0110111", "M+1":"1110111",
    "D-1":"0001110", "A-1":"0110010", "M-1":"1110010",
    "D+A":"0000010", "D+M":"1000010",
    "D-A":"0010011", "D-M":"1010011",
    "A-D":"0000111", "M-D":"1000111",
    "D&A":"0000000", "D&M":"1000000",
    "D|A":"0010101", "D|M":"1010101",
}

# dest 欄位:三個獨立開關 d1 d2 d3 → 要不要存到 A、D、M
DEST = {None:"000", "M":"001", "D":"010", "MD":"011",
        "A":"100", "AM":"101", "AD":"110", "AMD":"111"}

# jump 欄位:三個位元 j1 j2 j3 → 拿 comp 的結果跟 0 比,符合就跳
JUMP = {None:"000", "JGT":"001", "JEQ":"010", "JGE":"011",
        "JLT":"100", "JNE":"101", "JLE":"110", "JMP":"111"}

有了這三張表,翻譯一條 C 指令就只是「拆字串 → 查字典 → 拼起來」。我們先寫一個只懂「數字版」A 指令的迷你組譯器, 感受一下那種「毫無智慧」的直接: 

def translate(line):
    """把『單獨一條』指令翻成 16 位元(假設沒有符號、@ 後面都是數字)。"""
    if line.startswith("@"):
        # A 指令:0 + 15 位元的數值
        value = int(line[1:])
        return "0" + format(value, "015b")    # format(.., "015b") = 補滿 15 位的二進位
    else:
        # C 指令:dest=comp;jump,dest= 和 ;jump 都可能省略
        dest = comp = jump = None
        rest = line
        if "=" in rest:
            dest, rest = rest.split("=", 1)     # 切出 dest
        if ";" in rest:
            comp, jump = rest.split(";", 1)     # 切出 jump
        else:
            comp = rest
        return "111" + COMP[comp] + DEST[dest] + JUMP[jump]

for asm in ["@2", "D=A", "@3", "D=D+A", "@0", "M=D"]:
    print(asm, "->", translate(asm))

拿 D=D+A 對照:dest=D 查表得 010、comp=D+A 查表得 0000010、 jump 省略得 000,接上開頭的 111,合起來 111 + 0000010 + 010 + 000 = 1110000010010000。 和第 21 章圖 21-4 解剖的那條一模一樣。組譯器沒做別的,就是把這個對照過程自動化而已。

如果世界上所有組合語言程式都長得像「RAM[0] = 2 + 3」這麼乖——沒有迴圈、沒有變數、 @ 後面永遠是數字——那組譯器到這裡就寫完了。可惜真實程式不長這樣。一碰到符號, 真正的好戲才開始。

符號:組譯器真正的難關

翻開任何一段像樣的組語(例如第 21 章那個累加迴圈),你會看到三種「不是數字」的符號: 

    @R0          // R0 是什麼?
    D=M
    @i           // i 是什麼?
    M=1
(LOOP)           // (LOOP) 又是什麼?
    @i
    ...
    @LOOP        // 這裡的 LOOP 要變成哪個數字?
    0;JMP

組譯器必須把每個符號解析成一個確切的數字,才能繼續查表。而符號其實分三種, 各有各的處理方式——搞懂這三種,你就抓住了組譯器的精華。

一、預定義符號(predefined symbols)

有些符號是 Hack 平台內建、固定的,組譯器一開始就該知道它們等於多少。 它們只是「常用位址的好記別名」:

符號數值(位址)用途
R0 ~ R150 ~ 15最前面 16 格 RAM 的別名,當「虛擬暫存器」用
SP / LCL / ARG / THIS / THAT0 / 1 / 2 / 3 / 4之後的 VM 與編譯器會用到的指標(和 R0~R4 是同一批位址的別名)
SCREEN16384螢幕記憶體區的起點(記憶體映射 I/O)
KBD24576鍵盤那一格

處理方式最簡單:組譯器啟動時,先把這些放進一張符號表(symbol table)—— 說穿了就是一個「符號 → 數字」的字典。

二、標籤符號(label symbols)

程式碼裡用括號寫的 (LOOP)、(END) 叫標籤(label)。 它不是一條真正的指令(不會佔記憶體、不會產生機器碼),而是一個「書籤」: 它代表「緊接在它下面那條指令的位址」。這樣 @LOOP 才知道該跳到第幾條。

關鍵在於:標籤代表的是指令位址(ROM 位址),也就是「這是程式裡的第幾條指令」。 因為標籤行本身不算指令,所以為它編位址時要跳過它自己。

三、變數符號(variable symbols)

既不是預定義、也不是標籤的符號(例如 @i、@sum、@counter), 就是變數(variable)——程式想用、卻沒明講擺哪的一格記憶體。組譯器的責任, 是替每個新變數自動配置一格 RAM。Hack 的慣例是:變數從位址 16 開始, 遇到一個沒見過的新變數就給它下一個位址(16、17、18……)。

為什麼從 16?因為 0~15 已經被 R0~R15 佔走了。所以第一個變數拿到 16, 第二個拿到 17,以此類推。

🔑
標籤 vs 變數:同樣是符號,意義天差地遠

這是最容易搞混的一點:標籤(label)代表「指令在 ROM 裡的位址」,給 jump 當跳躍目標; 變數(variable)代表「資料在 RAM 裡的位址」,給你存取數值。 一個指向「程式」、一個指向「資料」。組譯器靠一條簡單規則分辨它們: 有沒有用 (...) 定義過?有,就是標籤;沒有(又不是預定義),就當成變數,配一格 RAM 給它。

為什麼非得「兩遍」不可?

現在我們要面對組譯器最有名的設計決定:它通常要把原始碼掃過兩遍(two-pass)。 為什麼不能一遍搞定?問題出在標籤。看這一行: 

    @END        // 這裡用到 END
    D;JGT
    ...
(END)           // 但 END 到這裡才定義!

當組譯器從上往下讀,讀到 @END 時,它還不知道 END 是第幾條指令—— 因為 (END) 出現在後面。這叫向前參照(forward reference): 「用到一個之後才定義的名字」。迴圈裡往前跳、if 往後跳出去,都會製造向前參照, 它幾乎無所不在。

⚠️
一遍為什麼會卡住

如果硬要一遍翻完,讀到 @END 時你只能猜或留白, 等之後看到 (END) 再回頭補。「回頭補」是可行的(這叫 backpatching),但很麻煩、容易出錯。 更簡單、更清楚的辦法是:先專心把所有標籤的位址找齊,再來翻譯。於是就有了兩遍。

兩遍的分工乾淨俐落:

  • 第一遍(Pass 1):只建符號表。從頭掃一次,不產生任何機器碼。 維護一個「指令位址計數器」,每遇到一條真指令就 +1;每遇到一個標籤 (XXX), 就把 XXX → 目前的位址 記進符號表(注意:標籤行不讓計數器 +1)。掃完,所有標籤的位址都齊了。
  • 第二遍(Pass 2):才真正翻譯。再掃一次,逐條產生機器碼。這次遇到 @符號, 符號保證查得到:若是預定義或標籤,直接查表;若還是查不到,那它就是個新變數, 當場配一格 RAM(從 16 起)並記進表裡。C 指令照舊查 comp/dest/jump。

走一遍兩個 pass(用第 21 章的累加迴圈)

抽象講完,我們拿一段真程式走一遍。這就是第 21 章那個「R1 = 1 + 2 + … + R0」的累加迴圈, 裡頭三種符號齊全:預定義 R0/R1、標籤 LOOP/END、變數 i/sum。 右邊標出了第一遍數出來的指令位址(注意標籤行不佔位址): 

    @R0          // 位址 0
    D=M          // 1     D = n
    @i           // 2
    M=1          // 3     i = 1
    @sum         // 4
    M=0          // 5     sum = 0
(LOOP)           //       ← 標籤!不佔位址,LOOP = 6
    @i           // 6
    D=M          // 7
    @R0          // 8
    D=D-M        // 9     D = i - n
    @END         // 10
    D;JGT        // 11    i > n 就跳到 END
    @sum         // 12
    D=M          // 13
    @i           // 14
    D=D+M        // 15    D = sum + i
    @sum         // 16
    M=D          // 17    sum = sum + i
    @i           // 18
    M=M+1        // 19    i = i + 1
    @LOOP        // 20
    0;JMP        // 21    跳回 LOOP
(END)            //       ← 標籤!END = 22
    @END         // 22
    0;JMP        // 23

第一遍結束,符號表在預定義符號之外,多了兩個標籤: LOOP → 6、END → 22。注意 (LOOP) 寫在位址 5 的指令之後、 位址 6 的指令之前,所以它等於 6。

第二遍從頭再掃,逐條翻譯。一路上第一次遇到沒見過的變數就配 RAM: @i(在位址 2 那行第一次出現)→ 配到 16;接著 @sum → 配到 17。 (R0、R1 是預定義,早就在表裡,不用配。)於是最終符號表新增了:

符號種類值在哪一遍決定的
LOOP標籤6第一遍
END標籤22第一遍
i變數16第二遍(第一個新變數)
sum變數17第二遍(第二個新變數)

看懂這張表,你就看懂了組譯器的靈魂。剩下的,是把它寫成程式。

從零打造一個 Hack 組譯器

我們現在把所有零件組裝成一個完整、可直接執行的 Hack 組譯器。它由四個小模組構成, 每個都對應前面講過的一個概念:

  • Parser(剖析):去掉空白與註解,留下乾淨的指令行。
  • Symbol table(符號表):那張「符號 → 數字」的字典,先塞滿預定義符號。
  • Code(轉碼):comp / dest / jump 三張表(上面已經寫好)。
  • 兩遍流程:把它們串起來的主程式。

Parser:洗掉雜訊

原始碼裡有縮排、空行、行內註解(// 之後到行尾)。這些對人類友善,對組譯器卻是雜訊。 Parser 第一件事就是把它們清掉,只留下「真正有意義」的行: 

def clean(lines):
    """去掉註解與所有空白,丟掉空行,回傳乾淨的指令/標籤行串列。"""
    result = []
    for line in lines:
        pos = line.find("//")          # 找行內註解
        if pos != -1:
            line = line[:pos]          # 砍掉 // 到行尾
        line = "".join(line.split())   # 去掉所有空白(含前後縮排與中間空格、tab)
        if line:                       # 跳過空行
            result.append(line)
    return result
💡
為什麼用 "".join(line.split()) 去空白?

line.split() 會依任意空白(空格、tab、換行)把字串切成單字串列, "".join(...) 再無縫接回去——一口氣清掉前後縮排和中間所有空白。 這樣即使有人把 D = M + 1 寫得鬆鬆散散,也能正確變成 D=M+1。 對 Hack 來說這安全,因為它的指令裡本來就沒有「有意義的空白」。

Symbol table:先塞滿預定義符號

def initial_symbols():
    """建立符號表,先放入所有 Hack 預定義符號。"""
    table = {
        "SP": 0, "LCL": 1, "ARG": 2, "THIS": 3, "THAT": 4,
        "SCREEN": 16384, "KBD": 24576,
    }
    for i in range(16):
        table["R" + str(i)] = i      # R0..R15 → 0..15
    return table

第一遍:建立標籤位址

def first_pass(lines, table):
    """掃一遍,把每個 (LABEL) 的『指令位址』記進符號表。不產生任何輸出。"""
    address = 0                       # 下一條『真指令』的 ROM 位址
    for line in lines:
        if line.startswith("(") and line.endswith(")"):
            label = line[1:-1]       # 去掉括號,取出標籤名
            table[label] = address   # 標籤 = 緊接其後那條指令的位址
            # 注意:標籤行不是指令,所以 address 不 +1
        else:
            address += 1             # 只有真指令才佔一個位址

第二遍:翻譯,順便配置變數

def second_pass(lines, table):
    """再掃一遍,逐條翻成 16 位元;遇到新變數就從位址 16 起配置 RAM。"""
    output = []
    next_var = 16                    # 下一個變數要配的 RAM 位址
    for line in lines:
        if line.startswith("("):
            continue                 # 標籤不產生機器碼,跳過

        if line.startswith("@"):     # ---- A 指令 ----
            symbol = line[1:]
            if symbol.isdigit():
                value = int(symbol)              # @數字:直接用
            else:
                if symbol not in table:          # 沒見過 → 是新變數
                    table[symbol] = next_var     # 配一格 RAM
                    next_var += 1
                value = table[symbol]            # 查表(預定義/標籤/剛配的變數)
            output.append("0" + format(value, "015b"))

        else:                        # ---- C 指令 ----
            dest = comp = jump = None
            rest = line
            if "=" in rest:
                dest, rest = rest.split("=", 1)
            if ";" in rest:
                comp, jump = rest.split(";", 1)
            else:
                comp = rest
            output.append("111" + COMP[comp] + DEST[dest] + JUMP[jump])
    return output

主程式:把四個模組串起來

def assemble(source_text):
    """完整的 Hack 組譯器:吃進組語字串,吐出機器碼字串(每行 16 位元)。"""
    lines = clean(source_text.splitlines())   # Parser
    table = initial_symbols()                 # Symbol table(預定義)
    first_pass(lines, table)                  # 第一遍:標籤位址
    machine = second_pass(lines, table)       # 第二遍:翻譯 + 配變數
    return "\n".join(machine)

🔑
組譯器其實「不懂」程式

注意整個過程裡,組譯器完全不理解這段程式在算什麼——它不知道這是個迴圈、不知道在做累加。 它只是忠實地「切字串、查表、配位址、拼位元」。這正是翻譯器的本質: 它做的是形式轉換(syntactic translation),而不是「理解語意」。 這也是為什麼組譯這麼快、這麼可靠——因為它笨,但它笨得精確。

動手練習

✏️ 練習 22.1 手算

當一回組譯器,把下面四行 Hack 組語手動翻成 16 位元機器碼。 (用本章的 comp / dest / jump 三張表;C 指令格式是 111 a-cccccc ddd jjj。)

(a)  @5
(b)  D=A
(c)  MD=M+1
(d)  D;JGT
看解答

(a) @5:A 指令,開頭 0,後面 15 位是 5 的二進位 000000000000101 → 0000000000000101。

(b) D=A:C 指令。comp=A 查表得 0110000、dest=D 得 010、無 jump 得 000。 接上 111 → 111 0110000 010 000 = 1110110000010000。

(c) MD=M+1:comp=M+1 得 1110111(a=1)、dest=MD 得 011、無 jump 得 000 → 111 1110111 011 000 = 1111110111011000。

(d) D;JGT:comp=D 得 0001100、無 dest 得 000、jump=JGT 得 001 → 111 0001100 000 001 = 1110001100000001。

不放心的話,把它們丟進本章的 assemble() 跑一遍核對——這正是「先手算、再用程式驗證」的好習慣。

✏️ 練習 22.2 思考

用你自己的話解釋:為什麼組譯器通常需要兩遍?如果硬要只掃一遍,具體會在哪種指令上卡住? 舉一個會出問題的最小例子。

看解答

核心原因是向前參照(forward reference):程式可以「用到一個在後面才定義的標籤」。 組譯器從上往下讀,讀到 @END 時,如果 (END) 還在下面、尚未掃到,它就不知道 END 等於幾, 無法把這條 A 指令翻成正確的數字。

最小例子:

@SKIP      // 用到 SKIP
0;JMP
@5
(SKIP)     // SKIP 到這裡才定義
M=1

一遍流程讀到第一行 @SKIP 時,SKIP 還沒進符號表,只能猜或留空。 兩遍則乾淨地解決它:第一遍先把所有標籤位址找齊(這裡 SKIP = 3), 第二遍翻譯時 @SKIP 保證查得到。 (附帶一提:變數不會有這個困擾,因為變數沒有「先用後定義」的問題——第一次用到就現場配位址即可。 真正逼出兩遍的,是標籤。)

對下面這段程式,寫出組譯器跑完兩遍後,符號表裡新增的所有條目 (預定義符號不用列),並標明每個是標籤還是變數、值是多少、在第幾遍決定。

    @count
    M=0
(BACK)
    @count
    M=M+1
    @x
    D=M
    @BACK
    D;JLT
    @done
    M=1
(STOP)
    @STOP
    0;JMP
看解答

先數第一遍的指令位址(標籤行不佔位址):

@count    0
M=0       1
(BACK)    → BACK = 2
@count    2
M=M+1     3
@x        4
D=M       5
@BACK     6
D;JLT     7
@done     8
M=1       9
(STOP)    → STOP = 10

0;JMP     11

第一遍建立兩個標籤:BACK = 2、STOP = 10。

第二遍依「第一次出現的順序」配置變數,從 16 起: 先遇到 @count → 16,再遇到 @x → 17,再遇到 @done → 18。 (@BACK、@STOP 是標籤,早在表裡,不算新變數。)

符號種類值哪一遍
BACK標籤2第一遍
STOP標籤10第一遍
count變數16第二遍
x變數17第二遍
done變數18第二遍

易錯點:變數順序看的是第一次出現的位置,不是字母順序;標籤位址看的是「下一條真指令」,別把標籤行也數進去。

✏️ 練習 22.4 實作

本章的 clean() 只處理 // 行內註解。請擴充它,讓它也能處理跨多行的 區塊註解 /* ... */(可能整段橫跨好幾行,像 C 那樣)。寫出新版函式,並想一個能測出它正確的例子。

看解答

區塊註解可能跨行,所以不能一行一行獨立處理——要用一個狀態旗標記住「現在是不是正處於註解中」。 最簡單的做法:先把整份原始碼當成一個大字串,用一個小迴圈掃過,在「正常」與「註解中」兩個狀態間切換:

def strip_block_comments(text):
    """移除所有 /* ... */ 區塊註解(可跨行)。"""
    result = []
    i = 0
    in_comment = False
    while i < len(text):
        if not in_comment and text[i:i+2] == "/*":
            in_comment = True
            i += 2
        elif in_comment and text[i:i+2] == "*/":
            in_comment = False
            i += 2
        elif in_comment:
            i += 1                      # 註解中:整個字元丟掉
        else:
            result.append(text[i])      # 正常:保留
            i += 1
    return "".join(result)

def clean(lines_or_text):
    """先去區塊註解,再去 // 行內註解與空白。"""
    text = lines_or_text if isinstance(lines_or_text, str) else "\n".join(lines_or_text)
    text = strip_block_comments(text)   # 先處理跨行的 /* ... */
    result = []
    for line in text.splitlines():
        pos = line.find("//")
        if pos != -1:
            line = line[:pos]
        line = "".join(line.split())
        if line:
            result.append(line)
    return result

測試例子(註解橫跨三行,還跟程式擠在同一行):

src = """@2  /* 把 2
載入 A，這段註解
跨了好幾行 */  D=A
@3"""
print(clean(src))   # ['@2', 'D=A', '@3']

關鍵測點:(1) 跨行註解被整段吃掉;(2) 註解後面同一行的 D=A 要保留; (3) 一般 // 與空白照樣處理。進階者可以再想:如果字串裡同時有 // 和 /* 該誰先? (Hack 沒有字串字面值,所以不用煩惱「註解符號出現在字串裡」這種更棘手的情況。)

✏️ 練習 22.5 實作

別再用查表了——請用計算實作 dest 模組。寫一個函式 dest_bits(dest), 吃進像 "AMD"、"MD"、"D" 或 None 的 dest 助記符, 回傳 3 位元字串 d1 d2 d3。(提示:d1=有沒有 A、d2=有沒有 D、d3=有沒有 M。) 並驗證它對所有合法 dest 都和本章的 DEST 表一致。

看解答

dest 的三個位元,各自就是「結果要不要存進 A / D / M」這三個獨立開關。 所以只要檢查字串裡有沒有出現對應字母即可,根本不需要查表:

def dest_bits(dest):
    if dest is None:
        return "000"
    d1 = "1" if "A" in dest else "0"   # 要不要存 A
    d2 = "1" if "D" in dest else "0"   # 要不要存 D
    d3 = "1" if "M" in dest else "0"   # 要不要存 M
    return d1 + d2 + d3

# 對照本章的 DEST 表,逐一驗證
DEST = {None:"000","M":"001","D":"010","MD":"011",
        "A":"100","AM":"101","AD":"110","AMD":"111"}
for key, expected in DEST.items():
    assert dest_bits(key) == expected, (key, dest_bits(key), expected)
print("全部一致！")

這題的價值在於:它讓你看清楚 dest 欄位的物理意義——三個位元不是隨便編號的, 它們就是「A 載入開關、D 載入開關、M 載入開關」三條獨立的控制線(呼應第 21 章說的「控制單元只是佈線」)。 查表只是把這個結構「背下來」,用算的才真正懂它。 想再進一步?試著用同樣的「拆位元」精神,寫一個把 16 位元機器碼反組譯回組語的 disassemble()。

✏️ 練習 22.6 挑戰

把本章的組譯器補成一支真正能用的命令列工具:讀進一個 .asm 檔(例如 Sum.asm), 產生同名的 .hack 檔(Sum.hack),每行一條 16 位元機器碼。 (這正是 Nand2Tetris Project 6 要交的東西。)

看解答

我們已經有了 assemble(source_text) 這個核心,剩下的只是「讀檔、改副檔名、寫檔」的外殼:

import sys

def main():
    in_path = sys.argv[1]                       # 例如 Sum.asm
    with open(in_path, encoding="utf-8") as f:
        source = f.read()

    machine = assemble(source)                  # 重用本章的核心組譯器

    out_path = in_path.rsplit(".", 1)[0] + ".hack"   # Sum.asm → Sum.hack
    with open(out_path, "w", encoding="utf-8") as f:
        f.write(machine + "\n")

    print("已輸出:", out_path)

if __name__ == "__main__":
    main()

把這一章串起來

這一章,我們把第 21 章「能讀懂、能手寫」的能力,升級成「讓程式自動翻譯」。回顧路標:

  • 組譯器是人類第一個、也最簡單的語言翻譯器:把組合語言(給人看)變成機器碼(給 CPU 看)。它存在的前提,正是第 21 章那句「指令也只是數字」。
  • 組譯的本質,大半只是查表替換:A 指令轉二進位;C 指令切成 comp/dest/jump,各自查表,接上 111。三張表直接接回第 21 章的 ALU 控制位元。
  • 真正的精華在符號。三種:預定義(R0~R15、SP、SCREEN、KBD…)、標籤(代表指令位址,給 jump 用)、變數(代表 RAM 位址,從 16 起自動配置)。
  • 因為有向前參照,組譯器要掃兩遍:第一遍只建標籤位址,第二遍才翻譯、順便配變數。符號表是兩遍之間的橋。
  • 我們用不到 70 行 Python,寫出一個完整可運作的 Hack 組譯器,親眼看 @i、@LOOP 變成正確的位址數字。
  • 組譯器是翻譯鏈的第一環。往上,編譯器與 VM 翻譯器用的是同一套思維(剖析、符號表、生成),只是層級更高。

我們已經打通了從「組合語言」到「機器碼」的最後一哩路。但人類也不會直接用組合語言寫大程式——太累、太貼近硬體了。 我們真正想用的是有變數、有函式、有物件的高階語言。那從高階語言到組合語言這一大段落差,又是誰來填補? 下一章,我們要往抽象高塔上爬一層,認識編譯器(compiler)與 VM 翻譯器—— 同樣是「翻譯」,但這次,輸入和輸出不再一對一,難度也將真正升級。你在組譯器這裡建立的每一個直覺,都會帶著你走下去。
