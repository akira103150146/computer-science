從理論到奪旗:歡迎來到 CTF

前面幾章,我們把資安的世界觀(第 34 章)、密碼學(第 35 章)、防禦性程式設計(第 36 章)、 Web 安全(第 52 章)、數位鑑識(第 53 章)都認真讀過了。你「知道」了很多——知道 SQL 注入怎麼發生、 知道封包一層層套著、知道檔案有「魔術位元組」。但「知道」和「做得出來」之間,隔著一條看不見的河。 這一章,就是帶你親手划過去。

划船的工具,叫做 CTF(Capture The Flag,奪旗賽):一種把資安技能遊戲化的競賽。 最常見的「解題型(Jeopardy)」CTF 長這樣——主辦方準備一堆刻意藏著漏洞或祕密的題目,分成 Web、Pwn(二進位攻擊)、 Crypto(密碼)、Forensics(鑑識)、Reversing(逆向)、Misc(雜項)等類別;你的任務是想辦法把每題藏起來的 「旗標(flag)」找出來。旗標通常是一串有固定格式的字串,像 FLAG{...}、flag{...}、 picoCTF{...},把它貼回計分板就得分。一面旗,代表你「真的做到了」這件事——它無從造假,因為你拿不到旗,就是沒攻下。

CTF 之所以是學資安最好的練法,正是因為這個「無從造假」:它逼你從「讀懂」走到「做出」。本章我們聚焦三類最適合入門、 又最能對應前面理論的題型,而且全部用 Python 為主、實作導向地走一遍:

開打前的鐵律:只打授權靶機

在寫任何一行攻擊程式之前,我們得先把最重要的事情講清楚——比任何技術都重要。這一章會教你怎麼讓登入繞過、怎麼越權讀別人的資料、 怎麼從別人的連線封包還原網址。這些能力,只有在對的地方使用,才是「白帽」;用錯地方,就是犯罪。

⛔
這條線,從頭到尾不鬆手

本章所有的「攻擊」操作,只能對你自己擁有、或獲得明確書面授權的環境進行——具體來說,就是你 在自己電腦上跑起來的本機靶機(如 vuln_server.py、DVWA、OWASP Juice Shop、WebGoat), 或像 PortSwigger Web Security Academy、picoCTF 這種專門設計給你合法練習的平台。 絕對不要對任何不屬於你的真實系統(別人的網站、公司的伺服器、學校的服務、公共 Wi-Fi 上別人的流量)做這些事—— 即使「只是看看」、即使你沒造成損害,在絕大多數國家(包含台灣的《刑法》妨害電腦使用罪章)都是犯罪。 封包分析也一樣:你只能分析你有權擷取的流量(你自己的機器、實驗室授權的抓包檔),竊聽他人通訊同樣違法。 這條倫理與法律的界線,我們在第 52、53 章反覆強調過,並會在第 41 章「計算倫理」正式深入。 技術是中性的;讓它成為盾,而不是矛,是工程師不可退讓的職業底線。

為什麼要練攻擊?因為你必須先看懂攻擊怎麼成立,才能真正把它擋住。一個沒親手繞過過登入的人, 寫起參數化查詢總是半信半疑;一個親手用 ' OR '1'='1 把登入打穿過一次的人,從此再也不會把使用者輸入直接拼進 SQL。 本章的每一個攻擊示範,後面都緊跟著一段「那要怎麼防」——攻是為了學防,這是貫穿全章的精神。

Web 攻防(一):HTTP 實戰與偵察

Web 攻防的第一步,永遠是偵察(reconnaissance,recon):在動手之前,先把目標看清楚。 而看清楚一個網站,靠的就是第 25 章那套 HTTP 請求-回應模型。我們先用兩個工具把這套地基喚醒: 瀏覽器的開發者工具(DevTools),以及 Python 的 requests 套件。

先在本機把靶機跑起來(假設靶機監聽 http://127.0.0.1:8000)。打開瀏覽器按 F12 開啟 DevTools, 切到「Network(網路)」分頁重新整理頁面,你會看到每一個請求的方法、狀態碼、回應標頭、cookie—— 這就是 Web 的「X 光機」。但在 CTF 與自動化測試裡,我們更愛用程式來看,因為它可以重複、可以批次、可以寫成腳本: 

import requests

BASE = "http://127.0.0.1:8000"          # ← 本機靶機,絕不是真實網站

r = requests.get(BASE + "/")            # 發一個 GET 請求到首頁
print("狀態碼 :", r.status_code)         # 200 / 301 / 403 / 404 ... (第 25 章)
print("伺服器 :", r.headers.get("Server"))
print("Cookie :", r.cookies.get_dict()) # 伺服器有沒有發 Set-Cookie?
print("------ 回應內容(前 300 字)------")
print(r.text[:300])

幾行程式,就把瀏覽器在背景做的事攤在你眼前:你發出請求、伺服器回你一個狀態碼 (200 成功、301/302 轉址、403 禁止、404 找不到)、一份回應標頭、 可能還有 cookie,以及最重要的——回應內容(HTML)。偵察的精髓,就是在這些東西裡找「不該被你看到、卻被看到了」的線索。

最經典的偵察:看原始碼與 robots.txt

新手最容易忽略、卻最常中獎的兩個地方:網頁原始碼裡的註解,和 robots.txt。 開發者常把測試帳密、隱藏路徑、甚至旗標,寫在 HTML 註解裡以為「使用者看不到」——但任何人按右鍵「檢視原始碼」就一覽無遺。 而 robots.txt 本意是告訴搜尋引擎「哪些路徑別爬」,結果反而像一張「這裡有東西別看喔」的藏寶圖, 把後台路徑通通列出來: 

import re, requests
BASE = "http://127.0.0.1:8000"

def find_flag(text):
    """從一段文字裡撈出 FLAG{...} 格式的旗標"""
    m = re.search(r"FLAG\{[^}]*\}", text)
    return m.group(0) if m else "(沒找到)"

# 偵察點 1:首頁 HTML 原始碼裡的註解,例如 <!-- FLAG{...} -->
html = requests.get(BASE + "/").text
print("原始碼註解的旗 :", find_flag(html))

# 偵察點 2:robots.txt 常洩漏不該公開的路徑
print("------ robots.txt ------")
print(requests.get(BASE + "/robots.txt").text)
💡
偵察就是「把每個角落都翻一遍」

除了原始碼註解與 robots.txt,常見的偵察點還有:/sitemap.xml、.git/ 目錄外洩、 備份檔(index.php.bak、config.old)、回應標頭裡的版本資訊、錯誤頁面噴出的堆疊追蹤、 以及 JavaScript 檔案裡寫死的 API 端點與金鑰。偵察不需要高深技巧,需要的是耐心與「什麼都點開看一下」的好奇心。 在防禦面,這也提醒你:別把任何祕密留在前端或公開檔案裡——使用者「看不到」不等於「拿不到」(這正是第 52 章「把按鈕藏起來不是安全」的同一個道理)。

Web 攻防(二):SQL 注入繞過登入

偵察完,進入第一個真正的攻擊。第 36、52 章都講過 SQL 注入(SQL injection)的成因: 當程式把使用者輸入直接拼進 SQL 查詢字串,使用者就能讓自己的輸入「越界」變成 SQL 語法的一部分。 現在我們親手做一次最經典的——繞過登入。

假設靶機的登入後端,寫了這樣一段會出事的程式(這是反面教材): 

# ⛔ 靶機裡刻意留的漏洞:把帳密「直接拼進」SQL
def check_login(username, password):
    query = ("SELECT * FROM users "
             "WHERE username = '" + username + "' "
             "AND password = '" + password + "'")
    return db.execute(query).fetchone()    # 有撈到一列 → 視為登入成功

正常情況下,你輸入 alice / secret,查詢變成 ... WHERE username = 'alice' AND password = 'secret',只有帳密都對才撈得到那一列。 但如果我們在帳號欄位填入 ' OR '1'='1' -- 呢?(注意最後有個空格,讓 -- 真正生效。)拼接後的查詢變成: 

SELECT * FROM users WHERE username = '' OR '1'='1' -- ' AND password = '...'

三件事同時發生,登入就此被攻破:

  • 那個 '(單引號)提前關閉了 username 的字串,讓後面的東西被當成 SQL 語法,而不是資料。
  • OR '1'='1' 是一個恆真的條件——只要有 OR 接上一個永遠成立的判斷,整個 WHERE 就對所有列都成立。
  • -- 是 SQL 的行註解,它把後面的 AND password = '...' 整段「註解掉」,密碼檢查直接消失。

結果:查詢撈回了使用者表的第一列(通常就是管理員),伺服器以為帳密正確,放你進去。我們用 requests 自動化這個攻擊: 

import re, requests
BASE = "http://127.0.0.1:8000"          # 本機授權靶機

def find_flag(text):
    m = re.search(r"FLAG\{[^}]*\}", text)
    return m.group(0) if m else "(沒找到)"

# 經典的登入繞過 payload:' OR '1'='1' --
payload = {"username": "' OR '1'='1' -- ", "password": "whatever"}
r = requests.post(BASE + "/login", data=payload)

print("狀態碼 :", r.status_code)
print("繞過登入拿到的旗 :", find_flag(r.text))

那要怎麼防?參數化查詢

你親手打穿了登入,現在你會對防禦有完全不同的體感。解法第 36、52 章都講過,這裡再釘一次: 參數化查詢(prepared statement / parameterized query)。它讓「資料」和「指令」走兩條完全分開的通道, 資料庫永遠把參數當「純資料」,無論裡面有多少單引號、OR 或 --,都不可能變成 SQL 語法: 

# ⛔ 危險:字串拼接(f-string 跟字串相加一樣致命)
def check_login(u, p):
    return db.execute(
        f"SELECT * FROM users WHERE username='{u}' AND password='{p}'"
    ).fetchone()

# ✅ 安全:參數化查詢,資料走獨立通道,? 裡的東西永遠是「純資料」
def check_login(u, p):
    return db.execute(
        "SELECT * FROM users WHERE username = ? AND password = ?", (u, p)
    ).fetchone()

當然,真實系統還要再加一層:密碼絕不明文比對,而是存「加鹽的慢雜湊」(bcrypt/scrypt/Argon2,呼應第 35 章), 登入時比對雜湊。但擋住 SQL 注入這個洞的根本,就是參數化——把資料和指令徹底分開。

✏️ 練習 71.2 實作

在本機靶機上,寫一支 Python 程式,用 requests 自動對 /login 發出 SQL 注入 payload 繞過登入、拿到旗標。 進階:除了 ' OR '1'='1' -- ,再多試一個不同寫法的 payload(例如用 # 當註解、或 ' OR 1=1 -- ), 並說明為什麼它們效果相同。最後寫出「正確的後端」應該長什麼樣。

看解答 
import re, requests
BASE = "http://127.0.0.1:8000"          # 本機授權靶機

def find_flag(text):
    m = re.search(r"FLAG\{[^}]*\}", text)
    return m.group(0) if m else "(沒找到)"

# 試多個等效 payload,逐一測試哪個能繞過
payloads = [
    "' OR '1'='1' -- ",     # 單引號跳出 + 恆真 + 行註解
    "' OR 1=1 -- ",         # 1=1 同樣恆真
    "admin' -- ",           # 已知帳號 + 直接註解掉密碼檢查
]
for p in payloads:
    r = requests.post(BASE + "/login",
                      data={"username": p, "password": "x"})
    flag = find_flag(r.text)
    print(f"payload={p!r:24} → {flag}")

為什麼等效:三者都做到同一件事——用 ' 跳出 username 字串、 用一個恆真條件('1'='1' 或 1=1)或一個已知帳號(admin')讓查詢撈到一列, 再用註解符號(-- 或某些資料庫的 #)把後面的密碼檢查整段消掉。差別只是「讓 WHERE 成立」的手法不同。

正確的後端:用參數化查詢(... WHERE username = ? AND password = ?), 並改成「先用帳號查出該使用者、再用 bcrypt 等比對密碼雜湊」,而不是把密碼也拼進 SQL。這樣任何 payload 都只會被當成「一個剛好叫這個怪名字的帳號」,查無此人而已。

Web 攻防(三):IDOR 越權存取

第三個 Web 攻擊,是 OWASP 常年第一名的洞:不安全的直接物件參考(Insecure Direct Object Reference,IDOR)。 它的本質是授權(authorization)缺失——系統檢查了「你有沒有登入」(認證),卻忘了檢查「這筆資料是不是你的」(授權)。

想像靶機有個「看公告」的端點 /note?id=1。你登入後,看到自己的公告網址是 ?id=1。 於是你做一件再自然不過的事:把 1 改成 2、3……結果你看到了別人的、甚至管理員的私人公告。 這就是 IDOR——靠竄改一個可預測的 id,就存取了不屬於你的資源: 

import re, requests
BASE = "http://127.0.0.1:8000"          # 本機授權靶機

def find_flag(text):
    m = re.search(r"FLAG\{[^}]*\}", text)
    return m.group(0) if m else "(沒找到)"

# 逐一嘗試不同的 id,看看哪一個翻出了「不該給我看」的資料
for note_id in range(1, 6):
    r = requests.get(BASE + "/note", params={"id": note_id})
    flag = find_flag(r.text)
    mark = "  ← 越權看到別人的!" if flag != "(沒找到)" else ""
    print(f"/note?id={note_id} → {flag}{mark}")

看出 IDOR 為什麼可怕了嗎?它不需要任何高深技巧——一個迴圈、一個會自己加一的數字,就能把整個資料庫的資料「枚舉(enumerate)」出來。 現實中無數的資料外洩事件,起點就是這麼樸素的一個漏洞。

那要怎麼防?伺服器端的存取控制

修法第 52 章講得很清楚:在伺服器端,對每一個敏感操作,都明確檢查「這筆資源,是不是屬於當前這個使用者」: 

# ⛔ 只檢查「有沒有登入」,沒檢查「這筆是不是你的」
@app.route("/note")
def get_note():
    note = db.get_note(request.args.get("id"))
    return render(note)                    # 改 id 就能看別人的

# ✅ 在伺服器端比對「擁有者是否為當前使用者」
@app.route("/note")
def get_note():
    note = db.get_note(request.args.get("id"))
    if note is None or note.owner_id != current_user.id:
        abort(404)                         # 不是你的 → 當作不存在
    return render(note)
🔑
認證 ≠ 授權

IDOR 的教訓只有一句話:「你是誰」(認證)和「你能碰什麼」(授權)是兩個獨立的檢查,缺一不可。 系統常把力氣全花在登入機制上,卻在每個讀資料、改資料的端點忘了再問一句「這個已登入的人,有權碰這一筆嗎?」。 補強的方向有兩個:一是每個敏感操作都做伺服器端的擁有者/權限檢查(治本);二是別用可預測的流水號當對外的 id (改用隨機的 UUID,讓攻擊者連「下一個 id 是什麼」都猜不到——這是縱深防禦的第二層,但不能取代授權檢查)。

換跑道。Web 攻防是站在「應用層」打,接下來我們要潛到更底層——封包(packet)。 這一整段,我們不裝任何套件(不用 scapy、不用 Wireshark),純粹用 Python 從一堆十六進位裡, 親手把封包一層層拆開。這正是 CTF 鑑識/網路題最需要的底層功力,也是把第 25 章網路分層從「圖」變成「手感」的最好方法。

封包分析(二):手刻解析 MAC / IP / TCP 旗標

切好了層,現在把每一層裡的欄位讀出來。先寫兩個小工具——把連續的位元組變成人看得懂的 MAC 與 IP: 

def mac(off):                    # 從 off 開始取 6 個 byte,拼成 aa:bb:cc:dd:ee:ff
    return ":".join(f"{x:02x}" for x in data[off:off + 6])

def ipv4(off):                   # 從 off 開始取 4 個 byte,拼成 192.168.0.1
    return ".".join(str(x) for x in data[off:off + 4])

第一層:乙太網路。目的 MAC 在 0、來源 MAC 在 6、型別在 12。型別是兩個 byte 拼成的 16 位元數, 要把高位 byte「左移 8 位」再跟低位 byte「合併」——0x0800 代表上層是 IPv4: 

dst_mac = mac(0)                 # 6c:4c:36:11:22:33 ← 這一段線路的收件網卡(常是閘道)
src_mac = mac(6)                 # aa:bb:cc:dd:ee:ff ← 寄件網卡(我的電腦)
ethertype = (data[12] << 8) | data[13]    # 兩個 byte 合成 16 位元 → 0x0800 = IPv4
print(f"目的 MAC : {dst_mac}")
print(f"來源 MAC : {src_mac}")
print(f"型別     : {ethertype:#06x}  ({'IPv4' if ethertype == 0x0800 else '其他'})")

第二層:IP。這裡有個經典陷阱:IP 標頭長度不一定是 20!第一個 byte(0x45)的低 4 個 bits 是 IHL(Internet Header Length),單位是「4 bytes」。0x45 & 0x0f = 5,所以標頭長 5 × 4 = 20 bytes。 懂得用 IHL 算長度,是你能正確找到「TCP 從哪裡開始」的關鍵。TTL 在 IP 起點 +8、協定在 +9(6=TCP)、來源/目的 IP 在 +12/+16: 

IP = 14                                    # IP 標頭從第 14 byte 開始
ihl = (data[IP] & 0x0f) * 4                 # 0x45 → 低4位=5 → 5*4 = 20 bytes
ttl = data[IP + 8]                          # 存活時間 TTL
proto = data[IP + 9]                        # 6 = TCP, 17 = UDP, 1 = ICMP
src_ip = ipv4(IP + 12)                      # 來源 IP
dst_ip = ipv4(IP + 16)                      # 目的 IP
print(f"來源 IP : {src_ip}   目的 IP : {dst_ip}")
print(f"TTL     : {ttl}      協定 : {proto} ({'TCP' if proto == 6 else '?'})")
print(f"IP 標頭長度 : {ihl} bytes")

第三層:TCP 與旗標。TCP 接在 IP 後面,起點是 14 + ihl。來源埠、目的埠各是兩個 byte (0x01bb = 443,正是 HTTPS)。最精彩的是第 13 個 byte 的 旗標(flags):它是一個位元遮罩(bitmask), 每個 bit 代表一個旗標——SYN=0x02、ACK=0x10、FIN=0x01、RST=0x04、PSH=0x08。 要判斷某個旗標有沒有設,就用位元 AND(&)去「遮」它: 

TCP = IP + ihl                             # TCP 標頭從這裡開始(= 34)
src_port = (data[TCP] << 8) | data[TCP + 1]
dst_port = (data[TCP + 2] << 8) | data[TCP + 3]
flags = data[TCP + 13]                      # 旗標 byte = 0x02

names = []
for bit, name in [(0x02, "SYN"), (0x10, "ACK"), (0x01, "FIN"),
                  (0x04, "RST"), (0x08, "PSH")]:
    if flags & bit:                         # 用位元 AND 檢查這個旗標有沒有設
        names.append(name)

print(f"{src_ip}:{src_port} → {dst_ip}:{dst_port}")
print(f"旗標 : {'+'.join(names)}")          # → SYN(這是連線的第一棒)

封包分析(三):從 TLS ClientHello 挖出 SNI 網址

現在來破除一個常見的誤解:「網站都上 HTTPS 了,封包都加密了,旁邊的人應該什麼都看不到吧?」——不全對。 HTTPS(第 25、35、52 章)確實把連線內容加密了,但在加密真正建立之前,客戶端要先送一個 ClientHello 跟伺服器打招呼,而這個招呼裡有一個欄位是明文的: SNI(Server Name Indication,伺服器名稱指示)——也就是你要連去哪個網域。

為什麼需要 SNI?因為一台伺服器常同時架了好幾個網站(共用同一個 IP),客戶端必須在握手時先講明「我要找的是 www.example.com」,伺服器才知道該拿哪張憑證出來。代價就是:你連線的內容被保護了,但你「造訪了哪個網站」這件事,仍然會曝光。 這對隱私、對 CTF 鑑識題,都是關鍵線索。

ClientHello 藏在 TCP 的資料(payload)裡,結構一層套一層,但只要照著「長度欄位」一段一段跳過去,就能定位到 SNI。 我們手刻一個簡化版的解析器(假設封包不含 IP 選項): 

def u16(b, off):                 # 兩個 byte 拼成一個 16 位元整數(大端序)
    return (b[off] << 8) | b[off + 1]

def tcp_payload(data):           # 切掉乙太網路+IP+TCP 標頭,取出真正的資料
    ihl = (data[14] & 0x0f) * 4
    t = 14 + ihl
    tcp_hdr = ((data[t + 12] >> 4) & 0x0f) * 4   # TCP 標頭長度也要算
    return data[t + tcp_hdr:]

def find_sni(payload):
    # TLS record: 0x16=Handshake, 之後 payload[5] 應為 0x01=ClientHello
    if len(payload) < 6 or payload[0] != 0x16 or payload[5] != 0x01:
        return None
    p = 5 + 4 + 2 + 32           # 跳過 record 標頭+握手型別/長度+版本+32 位元組亂數
    p += 1 + payload[p]          # 跳過 session id(1 byte 長度 + 內容)
    p += 2 + u16(payload, p)     # 跳過 cipher suites(2 byte 長度 + 內容)
    p += 1 + payload[p]          # 跳過壓縮方法(1 byte 長度 + 內容)
    ext_total = u16(payload, p); p += 2          # 擴充欄位(extensions)總長
    end = p + ext_total
    while p + 4 <= end:          # 逐一掃過每個擴充欄位
        etype = u16(payload, p)
        elen = u16(payload, p + 2)
        p += 4
        if etype == 0x0000:                      # 0x0000 = server_name(SNI)
            name_len = u16(payload, p + 3)
            return payload[p + 5: p + 5 + name_len].decode("ascii")
        p += elen
    return None

sni = find_sni(tcp_payload(data))
print("這條連線要造訪的網址(SNI) :", sni)   # 例如 www.cyivs.cy.edu.tw

加密保護「內容」,但不一定保護「去向」

這一關最該帶走的觀念,不是那串解析程式碼,而是:「加密」和「匿名」是兩件事。 TLS 讓中間人讀不到你傳輸的內容(密碼、訊息),但傳統的 SNI 仍以明文洩漏「你造訪了哪個網站」; DNS 查詢(若沒加密)也會洩漏同樣的資訊。這也是為什麼近年有 Encrypted ClientHello(ECH) 與 DNS over HTTPS(DoH)等技術,試圖把這最後一塊「去向」也藏起來。理解這條界線,你才知道 HTTPS 到底保護了什麼、沒保護什麼。

封包分析(壓軸):揪出 ARP 投毒的中間人

現在是封包分析的壓軸,也是把第 34 章中間人攻擊(MITM)從理論變成「親手抓到」的時刻。 我們要當一次偵探,從一份抓包檔裡,揪出一個躲在區域網路裡的中間人。

原理:ARP 投毒怎麼成立

先理解攻擊。在區域網路裡,電腦要傳資料給「閘道器(gateway,通往外網的門)」,它得先知道閘道的 MAC 位址。 它怎麼知道?靠 ARP(Address Resolution Protocol):大聲廣播問「誰是 192.168.0.1?把你的 MAC 告訴我」。 問題在於——ARP 完全沒有驗證機制,誰回答它就信誰。

ARP 投毒(ARP spoofing / poisoning)就鑽這個洞:攻擊者不斷發送偽造的 ARP 回應, 對受害電腦謊稱「我才是閘道(192.168.0.1 的 MAC 是我的)」,同時對閘道謊稱「我才是那台受害電腦」。 於是受害者所有「要送往外網」的封包,都先送到了攻擊者的網卡,攻擊者看完(甚至竄改)再轉給真正的閘道。 受害者完全無感——這就是中間人:他坐在你和閘道之間,你的流量全部過他的手。

怎麼從封包抓到他:兩條鐵證

當偵探,我們有兩條互相印證的線索:

  • 線索一:同一個 IP,對應到兩個 MAC。電腦「送出去」的封包,目的 MAC 都交給閘道(6c:4c:...); 但「收回來」的封包,來源 MAC 卻有很高比例是一個陌生的 MAC(12:ef:...)。同一個對外角色,MAC 卻不一致,非常可疑。
  • 線索二:那個陌生 MAC 是「軟體偽造」的。MAC 位址的第一個 byte 裡,有一個 U/L 位元(本地管理位元,locally administered bit,值 0x02): 原廠燒進網卡的 MAC,這個位元通常是 0;而軟體自己捏造的 MAC,常會把這個位元設成 1。 用 第一個byte & 0x02 一驗就知道。
from collections import Counter

CLIENT_IP = "192.168.0.136"      # 受害電腦
GATEWAY_OR_SERVER_IP = "203.68.92.170"   # 對外的另一端

def mac(data, off):
    return ":".join(f"{x:02x}" for x in data[off:off + 6])

def ipv4(data, off):
    return ".".join(str(x) for x in data[off:off + 4])

def is_spoofed(mac_str):         # 本地管理位元(0x02)被設 → 軟體偽造
    first_byte = int(mac_str.split(":")[0], 16)
    return (first_byte & 0x02) != 0

outbound = Counter()             # 出向:電腦把封包「交給」哪些 MAC
inbound = Counter()              # 入向:回來的封包「來自」哪些 MAC

for ts, data in packets:         # packets 是你從抓包檔解析出的所有封包
    src_ip, dst_ip = ipv4(data, 26), ipv4(data, 30)   # 14+12, 14+16
    dst_mac, src_mac = mac(data, 0), mac(data, 6)
    if src_ip == CLIENT_IP:
        outbound[dst_mac] += 1
    elif src_ip == GATEWAY_OR_SERVER_IP:
        inbound[src_mac] += 1

print("【出向】電腦把封包交給的 MAC :", outbound.most_common())
print("【入向】回來的封包來自的 MAC :")
total = sum(inbound.values()) or 1
for m, c in inbound.most_common():
    tag = " ⚠️ 軟體偽造!" if is_spoofed(m) else " (原廠網卡)"
    print(f"   {m}  ×{c}  ({c * 100 // total}%){tag}")

# 自動下結論
suspects = [m for m in inbound if is_spoofed(m)]
if suspects and outbound:
    print("\n🚨 高度懷疑 ARP 投毒中間人:出向都交給閘道,")
    print(f"   但 {sum(inbound[m] for m in suspects)} 個回程封包來自偽造 MAC {suspects}")

偵測到異常 ≠ 一定是攻擊

抓到「同一 IP 兩個 MAC + 偽造 MAC」是強烈的可疑訊號,但下結論前要保持鑑識的嚴謹(第 53 章): 也可能是合法的網路設備——例如某些企業/校園的合法側錄或流量管理設備、負載平衡、 或網路架構造成的正常現象。正確的態度是:把它當成「需要進一步查清的高度可疑事件」, 去確認那台裝置到底是什麼、是否獲授權,而不是一看到就斷言「被駭了」。鑑識講求證據與謹慎,不是腦補。

那要怎麼防 ARP 投毒?根本問題是 ARP 沒有驗證。實務上的防線有:在交換器上開啟 Dynamic ARP Inspection(DAI) 與 DHCP Snooping、對關鍵主機設定靜態 ARP 表、 以及最重要的一層——全程使用 HTTPS/TLS:即使攻擊者真的當了中間人,沒有有效憑證,他也解不開、也無法神不知鬼不覺地竄改你的加密流量(這正是第 52 章說的,TLS 是擋 MITM 的關鍵)。

最後,工具一句話:以上我們全部手刻,是為了練底層功;在真實工作裡,你會用 Wireshark—— 業界標準的封包分析工具,它能把抓包檔(.pcap)用圖形介面漂亮地分層展開,還能用過濾器(如 arp、eth.src==12:ef:...) 瞬間篩出你要的封包。但能手刻的人,用 Wireshark 會更清楚自己在看什麼。

檔案鑑識(一):魔術位元組與檔案修復

最後一條路線:檔案鑑識(forensics),呼應第 53 章。CTF 的鑑識題,常常給你一個「看起來普通」的檔案, 要你看穿它的真面目、挖出藏在裡面的祕密。我們從最基本、也最重要的觀念開始:檔案不靠副檔名決定身分。

副檔名(.jpg、.pdf)只是給人看的標籤,可以隨便改。一個檔案真正是什麼, 看的是它開頭那幾個位元組——魔術位元組(magic bytes),也叫檔案簽章(file signature)。 每種格式都有固定的開頭:

檔案類型魔術位元組(十六進位)可見字元
PNG 圖片89 50 4E 47 0D 0A 1A 0A.PNG....
JPEG 圖片FF D8 FF(不可見)
GIF 圖片47 49 46 38GIF8
PDF 文件25 50 44 46%PDF
ZIP / docx / jar50 4B 03 04PK..
ELF 執行檔(Linux)7F 45 4C 46.ELF

在終端機裡,三個工具是鑑識的瑞士刀:file(靠魔術位元組判斷真實類型,不理會副檔名)、 strings(把檔案裡所有「可見文字」撈出來——旗標常常就這樣裸奔在裡面)、 xxd(以十六進位 + ASCII 並排顯示,讓你看清每個位元組): 

# file:看穿真實類型(就算副檔名造假也騙不過它)
file mystery.jpg
# mystery.jpg: PNG image data, 800 x 600   ← 副檔名是 jpg,其實是 PNG!

# strings:把檔案裡的可見字串撈出來,直接 grep 找旗標
strings firmware.bin | grep FLAG

# xxd:看檔頭的十六進位(確認魔術位元組)
xxd mystery.jpg | head -n 2

用 Python 做同樣的事(純標準庫): 

SIGNATURES = {
    b"\x89PNG\r\n\x1a\n": "PNG 圖片",
    b"\xff\xd8\xff":      "JPEG 圖片",
    b"GIF87a":            "GIF 圖片",
    b"GIF89a":            "GIF 圖片",
    b"%PDF":              "PDF 文件",
    b"PK\x03\x04":        "ZIP / docx / jar ...",
    b"\x7fELF":           "ELF 執行檔",
}

def identify(path):
    head = open(path, "rb").read(16)
    for sig, name in SIGNATURES.items():
        if head.startswith(sig):
            return name
    return "未知(可能副檔名造假或檔頭損壞)"

print(identify("mystery.jpg"))     # → PNG 圖片(副檔名騙不了魔術位元組)

# 撈出檔案裡的旗標(strings 的 Python 版)
import re
data = open("firmware.bin", "rb").read()
m = re.search(rb"FLAG\{[^}]*\}", data)    # bytes 模式的正規表示式
print("夾帶文字裡的旗 :", m.group(0).decode() if m else "(沒找到)")

檔案修復:把壞掉的檔頭補回去

懂了魔術位元組,你就能做一件很「CTF」的事:修復檔案。題目常給你一個「打不開」的圖片—— 往往就是檔頭的魔術位元組被故意改壞了。你只要把正確的簽章補回去,檔案就活了: 

# 一個「打不開」的 PNG:前 8 個 byte 被改壞了
PNG_MAGIC = b"\x89PNG\r\n\x1a\n"           # PNG 正確的 8 byte 簽章
data = open("broken.png", "rb").read()
print("壞掉的檔頭 :", data[:8].hex(" "))

fixed = PNG_MAGIC + data[8:]               # 把前 8 byte 換回正確簽章
open("fixed.png", "wb").write(fixed)
print("修好了!現在 fixed.png 可以正常打開")

這背後的觀念很深刻:檔案的「結構」是有規則的——有固定的開頭、固定的區塊格式、固定的結尾標記。 鑑識的本事,就是熟悉這些規則,於是你能看出「哪裡不對勁、哪裡被動過手腳、哪裡藏了東西」。

檔案鑑識(二):metadata 與 EXIF

檔案除了「內容」,還帶著一層你常常忽略的東西:中繼資料(metadata)——關於這個檔案本身的資料。 其中最有名、也最常洩漏隱私的,是照片裡的 EXIF(Exchangeable Image File Format)。

你用手機拍的每一張照片,相機/手機都會默默把一堆資訊塞進檔案的 EXIF 區段:拍攝時間、相機型號、 鏡頭參數,甚至——GPS 經緯度座標。也就是說,一張隨手分享的照片,可能精準地洩漏了「你在什麼時間、什麼地點拍的」。 鑑識調查靠它定位嫌疑人;但反過來,一般人不小心分享原圖,也可能因此暴露住家位置。這是隱私的一大破口(呼應第 41 章)。

讀 EXIF 最方便的工具是 exiftool,一行就把所有 metadata 攤開: 

# exiftool:讀出照片的全部 metadata(時間、相機、GPS...)
exiftool photo.jpg

# 只看有沒有 GPS 座標(洩漏拍攝地點)
exiftool photo.jpg | grep -i gps

EXIF 本身格式繁複,手刻解析較費工(實務直接用 exiftool 或 Python 的 Pillow / exifread 套件), 但你可以先用最樸素的方法「聞到」它的存在——EXIF 區段裡常有可讀的字串(相機廠牌、軟體名),用前面 strings 的技巧就能瞥見: 

# 用 Pillow 讀 EXIF(若已安裝);這裡示範取出 GPS 與拍攝時間
from PIL import Image
from PIL.ExifTags import TAGS

img = Image.open("photo.jpg")
exif = img._getexif() or {}
for tag_id, value in exif.items():
    name = TAGS.get(tag_id, tag_id)
    if name in ("DateTimeOriginal", "Make", "Model", "GPSInfo"):
        print(f"{name:18}: {value}")
💡
兩面刃:鑑識的線索,也是隱私的破口

metadata 是典型的雙面刃。對鑑識而言,EXIF 的時間與 GPS 是還原「何時、何地」的黃金證據; 對隱私而言,它是不小心就會洩漏行蹤的地雷。實務上:多數社群平台上傳照片時會自動清掉 EXIF, 但你用原檔(email 附件、雲端連結、檔案傳輸)分享時通常不會。 想保護隱私,分享前可用 exiftool -all= photo.jpg 把 metadata 清乾淨。

檔案鑑識(三):carving 與 LSB 隱寫

最後兩招,是檔案鑑識最有「魔術感」的部分:把「藏在檔案裡的另一個檔案/訊息」挖出來。

檔案 carving:從一個檔案裡「雕」出另一個

檔案 carving(雕取)的概念是:很多檔案格式有明確的「結束標記」, 例如 PNG 一定以 IEND 區塊結尾、JPEG 以 FF D9 結尾。如果一個 PNG 在 IEND 之後還有資料, 那後面那串東西,十之八九是偷偷夾帶進去的(因為正常的圖片檢視器讀到 IEND 就停了,根本不會理會後面)。 我們就「順著結構,把該結束的地方之後的東西雕出來」: 

# 從 PNG 的 IEND 結束標記「之後」雕出被夾帶的資料
png = open("photo.png", "rb").read()
i = png.index(b"IEND")              # 找到 PNG 應該結束的地方
appended = png[i + 8:]             # 跳過 'IEND'(4 byte) + CRC(4 byte)
print("IEND 之後夾帶的資料(前 64 byte):", appended[:64])

import re
m = re.search(rb"FLAG\{[^}]*\}", appended)
print("雕出來的旗 :", m.group(0).decode() if m else "(沒找到)")

這正是業界工具 binwalk 在做的事(只是它更強):它會掃過整個檔案, 辨認出裡面所有嵌入的檔案簽章(一個 firmware 韌體裡常包著好幾個壓縮檔、檔案系統、圖片),然後幫你自動把它們全部分離出來: 

binwalk firmware.bin        # 掃描:列出裡面藏了哪些檔案
binwalk -e firmware.bin     # extract:把藏在裡面的檔案通通雕出來

LSB 隱寫術:把訊息藏進圖片的最低位元

carving 是「把檔案接在檔案後面」,藏得還算明顯。隱寫術(steganography)則高明得多—— 它把祕密藏進看起來毫無異樣的圖片「像素裡」,肉眼完全看不出來。最經典的手法叫 LSB(Least Significant Bit,最低有效位元)隱寫。

原理優雅得令人會心一笑:一個像素的顏色值是 0–255 的數字(8 個 bit)。把它的最低位的那一個 bit 改掉,顏色頂多變動 1(例如 200 → 201)——人眼根本分辨不出這點差異。於是,我們就把祕密訊息的每一個 bit, 偷偷塞進一個個像素的最低位元裡。一張圖有幾十萬個像素,藏個幾百字綽綽有餘,而圖片看起來毫髮無傷:

既然藏進去是「取每個像素的最低位、湊成訊息」,那解碼就是反過來:把每個像素的最低位元取出來, 每 8 個拼成一個 byte,串成原本的訊息。我們用最單純的 BMP(像素資料從第 54 byte 開始,沒有壓縮)來示範,純標準庫: 

def lsb_decode(bmp):
    pixels = bmp[54:]            # BMP 的像素資料從第 54 byte 開始
    out = bytearray()
    cur = 0
    for i, p in enumerate(pixels):
        cur = (cur << 1) | (p & 1)    # 取這個 byte 的最低位元,MSB first 拼回
        if i % 8 == 7:                 # 每湊滿 8 個 bit = 1 個 byte
            if cur == 0:               # 用 0 當訊息結束符
                break
            out.append(cur)
            cur = 0
    return out.decode("ascii", "replace")

bmp = open("secret.bmp", "rb").read()
print("從像素最低位元解出的訊息 :", lsb_decode(bmp))   # 例如 FLAG{...}

看懂這段程式,你就掌握了隱寫術的鑰匙。(cur << 1) | (p & 1) 這一行是精華: p & 1 取出像素的最低位元,cur << 1 把已收集的 bits 往左挪一格騰出位置,再用 | 把新 bit 補進來—— 這正是「一個 bit 一個 bit 把 byte 組回來」的標準寫法。

✏️ 練習 71.7 挑戰

(a) 用上面的 lsb_decode(),把藏在 secret.bmp 像素最低位元裡的旗標解出來。 (b) 想一想:如果隱寫者不是用「最低 1 個 bit」,而是用「最低 2 個 bits」來藏訊息,你的解碼程式要怎麼改? (c) 為什麼 LSB 隱寫不適合直接用在 JPEG 這種「有失真壓縮」的格式上?

看解答 
# (a) 解出旗標
def lsb_decode(bmp, nbits=1):       # (b) 順手支援「每像素取 nbits 個位元」
    pixels = bmp[54:]
    out = bytearray()
    cur, got = 0, 0
    mask = (1 << nbits) - 1         # nbits=1→0b1, nbits=2→0b11
    for p in pixels:
        cur = (cur << nbits) | (p & mask)   # 一次取低 nbits 位
        got += nbits
        while got >= 8:            # 每湊滿 8 位元就吐一個 byte
            got -= 8
            byte = (cur >> got) & 0xff
            if byte == 0:
                return out.decode("ascii", "replace")
            out.append(byte)
    return out.decode("ascii", "replace")

bmp = open("secret.bmp", "rb").read()
print("旗標 :", lsb_decode(bmp, nbits=1))

(b) 把「每像素取 1 個 bit」改成「取 nbits 個 bit」:用遮罩 mask = (1 << nbits) - 1 取低位, 並改成「累積到滿 8 位元就吐一個 byte」的通用作法(如上)。用 2 個 bit 藏,容量加倍,但對畫質的影響也變大(顏色最多差 3)。

(c) 因為 JPEG 是失真壓縮:它會為了縮小檔案,把人眼不敏感的細節丟掉並重算像素值。 而 LSB 隱寫的訊息恰恰藏在「人眼不敏感的最低位元」——JPEG 壓縮一過,這些最低位元就被改得面目全非,藏的訊息直接被毀掉。 所以 LSB 隱寫適合用在 BMP、PNG 這類無損(lossless)格式上。(針對 JPEG 有專門的隱寫法,藏在 DCT 係數而非原始像素,屬於進階主題。)
