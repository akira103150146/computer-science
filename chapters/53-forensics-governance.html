數位鑑識、治理與安全開發

寫程式的人問「怎麼讓它跑起來」,攻擊者問「怎麼讓它跑出我要的結果」。而當壞事真的發生了,還有第三種人會問:「到底發生了什麼、是誰做的、傷得多重——下次又該怎麼從一開始就擋住?」這一章,屬於那第三種人。

週五傍晚六點的那通電話

值班工程師小林正準備下班,手機突然震個不停。監控系統的警報寫著:凌晨三點十七分, 公司一台對外的伺服器,對一個從來沒見過的境外 IP,送出了將近 4GB 的資料。 三點十七分——那時候根本沒人在加班。

十分鐘後,主管打來,連珠炮似地問了一串問題:「到底發生了什麼?是誰做的? 他進來多久了、還碰了哪些機器?客戶的個資有沒有外洩?我們依法要不要通報主管機關、通知客戶? 還有——這個洞,當初為什麼會在?下次怎麼從根本上避免?」

你發現了嗎?這一連串問題,其實分屬三個不同的世界,而它們正好就是這一章的三個主角:

  • 「發生了什麼、誰做的、碰了哪些東西」——這是數位鑑識與事件回應:事發之後,如何科學地還原真相、止血善後。
  • 「依法要不要通報、誰該負責、我們的政策怎麼說」——這是治理、風險與合規(GRC):技術之外,組織層級的承諾、決策與規範。
  • 「這個洞當初為什麼會在、下次怎麼避免」——這是安全軟體開發(secure SDLC):把安全織進開發流程的每一針一線,讓洞從一開始就更難出現。

在第 34 章,我們學會了「像攻擊者一樣思考」,建立了 CIA、風險、威脅模型、安全設計原則這套世界觀; 第 35、36 章分別深入了密碼學與防禦性程式設計;第 40 章則談了軟體工程與事後檢討。 這一章(以及本部稍早的第 52 章)把鏡頭從「單一程式、單一漏洞」拉遠, 對準整個組織:當事情發生時怎麼查、平時怎麼管、開發時怎麼從源頭防。歡迎來到資安的成人世界。

🔑
資安的三個時態

把資安想成三個時態,這一章就有了骨架:事後(past)——出事了怎麼查、怎麼救,是鑑識與事件回應; 當下(present)——我們現在的風險有多大、政策與法規怎麼規定,是治理與合規; 事前(future)——怎麼讓下次更不容易出事,是把安全左移進開發流程。三個時態合起來, 才是一個組織完整的「安全姿態(security posture)」。

事發之後:什麼是數位鑑識

數位鑑識(digital forensics),是用科學、可重現、經得起檢驗的方法, 從數位裝置與資料中找出證據,以還原「發生了什麼事」的學問。它脫胎自傳統的刑事鑑識 (指紋、彈道、DNA),只是把「現場」換成了硬碟、記憶體、網路封包與雲端日誌。

關鍵字是「科學」與「經得起檢驗」。資安事件發生後,你當然可以憑直覺亂翻一通、 東點點西看看——但那樣得到的「結論」,既不可靠、也可能在過程中毀掉真正的證據, 更別說拿上法庭或交給稽核時會被一句「你怎麼證明這沒被你動過?」打回票。鑑識的核心精神, 就是用嚴謹的程序,讓你的發現可信、可重現、可追溯。

它的應用場景比你想的廣:

  • 資安事件調查:就像開頭的故事——查清入侵的範圍、手法、起點與影響,好決定怎麼止血、要不要通報。
  • 犯罪偵查:詐欺、智慧財產竊取、兒少保護、商業間諜……數位證據早已是現代司法的主角。
  • 合規與內部調查:員工違規、資料外洩的責任歸屬、稽核要求的舉證,都需要可信的數位證據。
⚠️
最容易犯的第一個錯:破壞現場

發現可疑機器時,新手的本能反應往往是「趕快登入看看」「重開機試試」「把可疑檔案刪掉」—— 這幾乎是鑑識上的災難。每一次你操作那台機器,都在改變它的狀態:覆蓋記憶體、更新檔案存取時間、 甚至觸發攻擊者預設的「自毀」程式。鑑識的鐵則是:盡可能不改變原始證據,所有分析都在「複本」上做。 這就是為什麼接下來的流程,第一步永遠是「保全」,而不是「調查」。

鑑識流程:從識別到報告

雖然各家機構的用詞略有出入,數位鑑識大致都遵循一條由保全到報告的流水線。 記住它的順序,比背它的名字重要——因為順序本身就藏著智慧:

  • 識別(Identification):確定哪些裝置、系統、帳號、日誌可能與事件有關,劃定調查範圍。哪台機器是現場?資料可能流到哪?
  • 保全(Preservation):把證據「凍結」起來,防止它被改變、覆蓋或自然消失。這一步往往要爭分奪秒(尤其是易失的記憶體),也是整條流程裡最容易搞砸、卻最關鍵的一步。
  • 蒐集(Collection):用鑑識上可靠的方法取得資料——例如對硬碟製作「位元對位元」的完整映像(image),而不是普通的複製檔案。對原始證據盡量唯讀、不寫入。
  • 檢驗(Examination):在複本上做技術處理:還原已刪除的檔案、解開壓縮、過濾出相關時間區間的紀錄、抽取出待分析的資料。
  • 分析(Analysis):把零散的證據拼成故事——攻擊者何時進來、走了哪條路、碰了什麼、帶走了什麼。建立時間軸(timeline)是這一步的核心。
  • 報告(Reporting):用清楚、誠實、不誇大的方式寫下發現、方法與結論,讓非技術的主管、律師、法官或稽核員也能讀懂並信任。

監管鏈:讓證據「站得住腳」

想像一個謀殺案,凶刀從現場被撿起來後,中間「不知道被誰拿去哪裡放了三天」才送進實驗室。 辯護律師只要問一句:「你怎麼證明這把刀沒被掉包、沒被動過手腳?」這個證據就廢了。 數位證據一模一樣,而且更脆弱——因為複製、修改數位檔案,比動真刀容易太多了。

監管鏈(chain of custody),就是一份從證據被取得的那一刻起, 鉅細靡遺記錄「誰、在何時、對它做了什麼、交給了誰」的不間斷紀錄。 它要回答的問題只有一個:這份證據,從現場到法庭(或到稽核報告)的整段旅程, 每一秒都有人負責、都能交代,從來沒有一個「沒人看著、可能被動手腳」的空窗。 

監管鏈紀錄(示意)
─────────────────────────────────────────────────────────────────
時間              經手人      動作                        證物雜湊(SHA-256,前 8 碼)
2026-06-29 18:40  林工程師    伺服器拔網路線、製作磁碟映像   a3f51e9c
2026-06-29 19:05  林工程師    封入防靜電袋、貼封條、簽名     a3f51e9c
2026-06-29 19:30  陳鑑識員    自林工程師處接收、入庫         a3f51e9c  ← 重算一致 ✓
2026-06-30 09:15  陳鑑識員    取出、於唯讀工作站開始檢驗     a3f51e9c  ← 重算一致 ✓
─────────────────────────────────────────────────────────────────
每次交接、每次取用,都記錄人、時、事,並重新比對雜湊。

為什麼用雜湊驗證證據未被竄改

上面那串 a3f51e9c 是什麼?它是整份磁碟映像的雜湊值(hash)—— 還記得第 35 章密碼學嗎?雜湊函式(像 SHA-256)會把任意大小的資料, 壓成一個固定長度的「指紋」。它有兩個鑑識上無可取代的性質:

  • 極度敏感:原始資料只要改動一個位元,算出來的雜湊值就會天翻地覆地不同(雪崩效應)。
  • 實務上不可偽造:你幾乎不可能刻意找出「另一份不同的資料,卻有相同的雜湊」(抗碰撞性)。

所以做法是:在取得證據的當下,立刻算一次雜湊並記錄下來。 之後任何時間、任何人,都能對手上的映像「重算一次」,只要結果和當初那串一致, 就在數學上幾乎鐵證如山地證明了——這份證據,一個位元都沒被動過。 

# 取得證據的當下,先算一次雜湊(指紋),並把它存起來
sha256sum disk_image.dd > disk_image.dd.sha256

# 之後任何時候,都能重算並比對,證明映像未被竄改
sha256sum -c disk_image.dd.sha256
# disk_image.dd: OK        ← 一致,證據完整

# 製作映像時也常見「邊複製邊算雜湊」的鑑識工具,
# 確保「來源」與「複本」的雜湊在取證當下就一致。
🔑
證據完整性 = 監管鏈 + 雜湊

這兩者是一組搭檔,缺一不可:雜湊用數學證明「內容沒被改」; 監管鏈用紀錄證明「整段過程沒有可被動手腳的空窗,而且改動的話也有人負責」。 有了它們,你的鑑識結論才從「我覺得是這樣」升級成「我能證明是這樣」。這也呼應了第 34 章的 完整性(Integrity)與不可否認性——只是這次,我們是把它們用在證據本身上。

✏️ 練習 53.1 思考

請用自己的話解釋:為什麼鑑識人員要在取得證據的當下就算雜湊,而不是「等分析完再算」? 另外,若一個案子完全沒有維護監管鏈,但磁碟映像的雜湊「現在算起來」和半年前報告裡寫的一致, 這樣的證據可信嗎?為什麼?

看解答

為什麼當下就算:雜湊證明的是「從算的那一刻起,內容沒變」。如果等分析完才算, 那麼「取得到分析」這段期間的任何竄改(無意的覆寫或惡意的動手腳)都已經被「洗白」進你記錄的那串雜湊裡了—— 你只能證明「分析後到現在」沒變,卻無法證明「拿到的就是現場原貌」。所以必須在最源頭、改變最少的時刻就定錨。

沒有監管鏈、只有雜湊一致:可信度大打折扣。雜湊一致只證明「這份檔案的內容,和半年前那份檔案一樣」, 但它無法證明:當初那份檔案是不是真的來自現場、中間有沒有被掉包成「另一份內容自洽但被竄改過、再重新記錄雜湊」的版本、 半年來這份證據放在哪、誰碰得到。雜湊保「內容」,監管鏈保「來歷與過程」。法庭與稽核兩者都要。

證據藏在哪裡:來源與蒐集順序

證據不只在硬碟裡。一台被入侵的機器,證據散落在好幾個地方,而它們的「壽命」天差地遠—— 這就引出鑑識最反直覺、也最重要的一個原則。先看證據都藏在哪:

  • 磁碟(disk):檔案、日誌、瀏覽紀錄。更妙的是,「刪除」一個檔案通常只是把它標記為可覆寫, 資料其實還躺在那裡,直到被新資料蓋掉——所以已刪除檔案常常救得回來。 還有所謂的 slack space(間隙空間):檔案不會剛好填滿一個磁碟區塊,尾端那截「沒用到的空間」裡, 可能殘留著前一個被刪檔案的碎片。攻擊者藏東西、或不小心留下痕跡,都常在這些角落。
  • 記憶體(memory / RAM):這是易失性(volatile)證據——機器一斷電,內容瞬間消失。 但 RAM 裡藏著最珍貴的東西:正在執行的惡意程式、解密後的金鑰與密碼、網路連線、剪貼簿、 甚至「只在記憶體裡跑、從不落地到硬碟」的無檔案(fileless)惡意程式。關機,就等於把它毀屍滅跡。
  • 網路日誌(network logs):防火牆、路由器、IDS/IPS、Proxy、DNS 的紀錄, 能告訴你資料流向哪個外部 IP、流了多少、什麼時候——這正是開頭故事裡警報的來源。它常存在別台機器上,相對持久但也會輪替覆寫。
  • 行動裝置(mobile devices):手機是另一個世界(加密、雲端同步、各家系統不同), 但藏著通訊紀錄、定位、App 資料,是現代調查的重點戰場,通常需要專門的工具與授權。

易失性順序:先收最快消失的

既然不同證據的壽命不同,蒐集時就有一條鐵律:越容易消失的,越要先收。 這叫易失性順序(order of volatility)。一個經典的反面教材是:技術人員一發現機器被入侵, 立刻「拔電源、保全硬碟」——結果硬碟是保住了,但 RAM 裡那個正在運行、握著解密金鑰的惡意程式, 連同所有網路連線狀態,在斷電的瞬間全部灰飛煙滅。最值錢的證據,被「保全」這個動作親手毀掉了。

⚠️
「該不該拔電源?」沒有標準答案,但有原則

這是事件回應現場真實的兩難:不拔,攻擊者可能持續竊取資料或觸發自毀;拔了,記憶體證據全沒、 全碟加密的機器可能再也開不了(金鑰只在 RAM 裡)。原則是:在情況允許下,先做記憶體傾印與蒐集易失證據, 再決定隔離方式;若必須立刻止血,則「網路隔離(拔網路線、隔離網段)」通常比「直接斷電」破壞更小。 這正是鑑識(要保全證據)與事件回應(要趕快止血)之間,需要事先在計畫裡權衡清楚的地方。

✏️ 練習 53.2 分析

你接手一台疑似被入侵、目前仍開機運轉的伺服器。下列證據來源,請依照正確的蒐集順序 (先收哪個、後收哪個)排出來,並說明你的理由:

  1. 路由器上的對外連線日誌(存在另一台網管設備)
  2. 這台機器的記憶體(RAM)內容
  3. 硬碟上的檔案系統(含可能的已刪除檔)
  4. 硬碟的交換空間(swap)
看解答

正確順序:b → d → c → a(由最易失到最持久)。

  1. (b) 記憶體最先收:它一斷電或一重開機就永久消失,而且藏著執行中的惡意程式、網路連線、解密金鑰等無可取代的證據。要趁機器還活著時做「記憶體傾印」。
  2. (d) 交換空間次之:它在磁碟上,但屬於「暫存」性質,重開機常被清空,比一般檔案易失。
  3. (c) 硬碟檔案系統再來:相對持久,可以製作位元對位元的映像來保全(並立刻算雜湊)。已刪除檔與 slack space 都靠這份完整映像才救得回。
  4. (a) 路由器日誌最後:它存在另一台設備上,不會因為你這台伺服器斷電而消失,相對最持久(但仍要留意日誌輪替覆寫的時限,別拖太久)。

核心理由就是易失性順序:先搶救最快消失的,把「不可逆的損失」風險降到最低。 全程當然都要維護監管鏈、對每份取得的證據算雜湊。

事件回應:鑑識上場的舞台

鑑識回答「發生了什麼」,但一場資安事件需要的不只是查清楚——還要趕快止血、修復、並從中學習。 管理整起事件從爆發到收尾的這套流程,叫事件回應(incident response, IR)。 還記得第 34 章說「資安是一種文化與流程」嗎?事件回應就是這句話最具體的展現: 它是一個有準備、有劇本、會演練的生命週期,而不是出事後手忙腳亂地臨場發揮。

  • ① 準備(Preparation):事發之前就要做的功課——寫好事件回應計畫、組好應變團隊、備妥工具與聯絡名單、做好備份、平時就演練。準備是唯一一個在「平時」做的階段,卻決定了出事時的成敗。
  • ② 偵測與分析(Detection & Analysis):從監控、日誌、告警(或使用者通報)中發現「有事不對勁」,並判斷這是不是真的事件、有多嚴重、影響範圍多大。開頭那則 4GB 外傳的警報,就是這一步。
  • ③ 圍堵(Containment):阻止災情擴大——隔離受害機器、停用被盜帳號、封鎖惡意 IP。這裡和鑑識最容易打架:圍堵要快,但別在慌亂中毀掉易失證據(回想上一節的「先傾印記憶體再隔離」)。
  • ④ 根除(Eradication):把攻擊者與其留下的東西徹底清乾淨——移除惡意程式、關掉被利用的漏洞、重置被竊的憑證。沒根除乾淨,攻擊者會再從同一個後門回來。
  • ⑤ 復原(Recovery):把系統安全地恢復上線——從乾淨的備份還原、加強監控、確認攻擊者真的走了。要謹慎,別把藏有後門的「髒備份」又救回來。
  • ⑥ 事後檢討(Lessons Learned):事件落幕後,團隊坐下來誠實檢討:怎麼進來的?我們哪裡慢了、哪裡做對了?要改什麼才不會重演?——這正是第 40 章講的事後檢討(postmortem),而且應該是對事不對人(blameless)的,目的是改進系統與流程,不是找戰犯。
🔗
鑑識與事件回應的關係

兩者像是同一場戰役裡的兩種角色,常常並肩作戰但目標微妙不同。事件回應的首要目標是 趕快止血、恢復營運;數位鑑識的首要目標是保全證據、查清真相。 它們會互相拉扯(救火 vs 保留現場),所以成熟的組織會在「準備」階段就把規矩講好: 什麼情況下優先保全證據、什麼情況下優先止血、誰有權拍板。事後檢討產出的改進, 很多最後會變成下一節「治理」裡的政策,以及最後幾節「安全開發」裡的修補。

✏️ 練習 53.3 設計

回到開頭的故事:週一早上,你確認了那台對外伺服器在凌晨被入侵,且目前仍在對外傳送資料。 請依照事件回應的六個階段,寫出你會採取的具體步驟(至少涵蓋偵測確認、圍堵、根除、復原、檢討), 並特別說明:你會怎麼處理易失證據與止血之間的衝突?

看解答

(參考答案,合理即可。一份好答案會同時兼顧「止血」與「保全證據」。)

  1. 準備(已在平時就緒):啟動事件回應計畫、召集應變小組、開一條安全的溝通管道、開始記錄時間線(這份紀錄本身日後也是監管鏈的一部分)。
  2. 偵測與分析:確認警報為真——比對防火牆/網路日誌,確定外傳的目的 IP、流量、起始時間;初步判斷影響範圍(這台碰得到哪些系統、哪些資料)。
  3. 圍堵(與鑑識並行):先對這台機器做記憶體傾印(搶救正在運行的惡意程式、連線與金鑰),接著用網路隔離(拔網路線或移到隔離網段)止住外傳,而不是直接斷電——這樣既止了血,又保住了易失證據。同時製作磁碟映像、立刻算雜湊、啟動監管鏈。停用任何疑似被盜的帳號、輪替相關憑證。
  4. 根除:在複本上分析出攻擊者的進入點與留下的後門/惡意程式,確認並修補被利用的漏洞,清除所有植入物,重置所有可能外洩的密碼與金鑰。
  5. 復原:從確認乾淨的備份重建系統(別救回髒備份),上線前加強監控、確認攻擊者真的離開,再逐步恢復對外服務。
  6. 事後檢討:對事不對人地檢討——攻擊怎麼進來的、我們偵測花了多久、哪裡可以更快;產出具體改進(補哪個流程、改哪段程式、加哪個監控),並評估法遵義務(是否需依法通報主管機關/通知當事人,見下一節)。

易失證據 vs 止血的衝突:關鍵在「先傾印記憶體、再以網路隔離取代直接斷電」—— 這個順序讓你不必在「保住證據」和「停止外傳」之間二選一。能事先在計畫裡寫好這條,正是「準備」階段的價值。

治理:技術之外,需要管理層點頭

把場景從事件現場拉開。你可能會想:資安不就是工程師的事嗎——把程式寫好、把系統設好不就行了? 殘酷的現實是:再強的技術,也擋不住一個「老闆覺得資安不重要、不肯給時間給預算」的組織。 資安要成功,必須是一件由上而下、被當成正事來管的事。這就是安全治理(security governance): 由組織的領導層,為資安設定方向、分配資源、明確責任、並確保它真的被執行的整套管理機制。

治理具體落地,通常是一套由抽象到具體的文件層級:

  • 政策(policy):最高層次的「我們的立場與意志」。例如:「本公司所有客戶個資一律加密儲存」「所有對外系統都必須開啟 MFA」。它通常短、原則性,且由管理層簽署背書。
  • 標準(standard):把政策變成可衡量的硬性要求。例如:「加密一律使用 AES-256」「密碼至少 12 碼」。
  • 程序 / 指引(procedure / guideline):手把手的步驟說明。例如:「申請新權限的流程是……」「發生事件時的通報步驟是……」(沒錯,上一節的事件回應計畫,就是一份程序)。
🔑
為什麼技術之外需要「管理層承諾」?

因為資安的每一個決定,本質上都是取捨(trade-off)(這正是第 34 章的核心): 多一道驗證就多一點摩擦,修一個技術債就少做一個新功能,做威脅模型就慢一點上線。 這些取捨跨越團隊、牽動預算、影響營收——工程師無權、也無法獨自拍板。 唯有當管理層真心把資安當回事,願意為它分配時間、人力、預算,並在「趕快上線」與「做安全」衝突時站在後者這邊, 底下的技術努力才有意義。否則,安全永遠會是那個「等有空再說」、結果永遠沒空的項目。

安全文化:寫在紙上的,不等於做在身上的

但治理最深的一層,不是文件,而是文化(culture)——呼應第 34 章的結尾: 資安是一種文化與流程。再完美的政策,如果員工心裡覺得「這只是 IT 在找麻煩」「能繞就繞」,那它就是一堆廢紙。 好的安全文化是:大家發自內心覺得安全是「自己的事」——看到可疑信會回報而不是默默點開、 發現同事在分享密碼會善意提醒、做新功能時會主動想「這樣安全嗎」。 這需要持續的教育、合理(別逼人造假)的規則,以及——最重要的——領導層以身作則。 當老闆自己也乖乖開 MFA、自己也尊重安全流程,文化才立得起來。

風險管理:把不確定,變成可以決策的東西

治理要分配資源,但資源永遠有限——你不可能把每個風險都堵到滴水不漏。所以治理的核心引擎,是 風險管理(risk management):有系統地辨識風險、衡量它有多嚴重、再決定怎麼處理的過程。 它讓「我們夠安全了嗎」這個模糊的焦慮,變成一張可以排序、可以決策、可以交代的清單。

風險評估:它有多嚴重?

第一步是風險評估(risk assessment)。還記得第 34 章那條公式嗎? $$\text{風險} \approx \text{威脅} \times \text{弱點} \times \text{衝擊}$$ 評估就是把這條公式,套到你每一項資產上,逐一拷問:

  • 資產(asset):這值多少?(客戶個資、原始碼、營運系統……)
  • 威脅(threat):誰/什麼可能來傷害它?可能性多高?
  • 弱點(vulnerability):它有哪些可被利用的缺口?
  • 衝擊(impact):萬一真出事,損失多大?(金錢、商譽、法律責任、人命……)

把「可能性」與「衝擊」各分高/中/低,畫成一張風險矩陣(risk matrix),你就能一眼看出 哪些風險是「紅色、要優先處理」,哪些是「綠色、可以先放著」。風險管理不是消滅所有風險(那不可能), 而是把有限的資源,投到最該投的地方。

風險處置:那……我們要怎麼辦?

評估出風險後,面對每一項,你有四種(也只有四種)基本選擇,合稱風險處置(risk treatment):

  • 接受(Accept):評估後認為「不值得花成本去處理」,就有意識地承擔它。注意是有意識地——是經過評估後的決定,不是「沒注意到」。
  • 降低 / 減緩(Reduce / Mitigate):採取措施把可能性或衝擊壓下來。這是最常見的一種——打補丁、加 MFA、做備份、縮小攻擊面,都是降低。
  • 轉移(Transfer):把風險(的一部分)轉嫁給別人。最典型的是網路保險(cyber insurance),或把某項服務外包給更專業、更有能力扛的供應商。
  • 避免(Avoid):乾脆不做那件帶來風險的事。例如:評估後決定「不自己存信用卡卡號了,直接串接專業的金流商」,風險的源頭就消失了。
🔑
殘餘風險:沒有「零風險」這回事

不管你怎麼處置,做完之後總會剩下一點風險,這叫殘餘風險(residual risk)。 你裝了防火牆、打了補丁、買了保險,但天底下沒有絕對安全——殘餘風險永遠 > 0。 風險管理的終點不是「歸零」,而是把殘餘風險降到組織「可接受的程度(risk appetite)」, 並由有權的人簽字認可、承擔它。這份「明知還有風險、但我決定接受」的白紙黑字, 正是治理與工程的交會點:工程把風險降低,治理決定降到哪、剩下的誰來扛。

✏️ 練習 53.4 分析

一間小型電商有以下狀況:把客戶的信用卡卡號明文存在自架的資料庫裡, 而資料庫的備份是工程師每週手動複製到辦公室一台沒上鎖的電腦。 請對它做一個迷你風險評估與處置建議:

  1. 指出至少 2 個風險(各自的資產、威脅、弱點、衝擊)。
  2. 對每個風險,從「接受/降低/轉移/避免」中選一個處置方式並說明理由。
看解答

(參考答案,合理即可。)

風險一:卡號明文外洩

  • 資產:客戶信用卡卡號(極敏感、受法規規範)。威脅:外部入侵者、惡意內部人員。弱點:明文儲存、無加密。衝擊:極高——盜刷損失、巨額罰款、商譽崩盤、可能違反 PCI DSS(支付卡產業資料安全標準)。
  • 處置:避免為上策——根本不要自己存卡號,改串接合規的第三方金流商(把存卡的責任也一併轉移出去)。若業務上真的必須存,退而求其次是降低:加密 + 嚴格存取控制 + 符合 PCI DSS。

風險二:備份電腦失竊或被存取

  • 資產:整份資料庫備份(等於全部客戶資料)。威脅:小偷、清潔人員、任何能進辦公室的人。弱點:電腦沒上鎖、備份未加密、實體安全薄弱。衝擊:高——一次失竊就等於全資料庫外洩。
  • 處置:降低——備份檔加密、存放在上鎖且權限受控的地方、改用自動化且加密的異地備份、落實最小權限。

注意這題怎麼把第 34 章的「資產/威脅/弱點/衝擊」語言,接上了「四種處置 + 合規(PCI DSS)」——這正是把風險「翻譯成決策」的樣子。

合規與框架:為什麼需要一張別人畫好的地圖

你可能會問:既然我們會自己做風險管理,為什麼還要管那一堆英文縮寫的「框架」和「法規」? 答案有三個層次,一個比一個現實:

  • 法律強制:某些資料、某些產業,法律白紙黑字要求你必須達到一定的保護標準。不遵守,就是違法,會被罰、會被告。
  • 客戶要求:大公司在跟你簽約前,往往會問「你有沒有 ISO 27001 / SOC 2?」沒有,生意可能就談不成。合規成了商業的入場券。
  • 信任的證明:你說「我們很安全」沒人信;但若有獨立第三方稽核幫你背書,這份信任就有了憑據。

框架(framework)就是前人整理好的「資安該做哪些事」的清單與地圖, 讓你不必從零摸索。你不需要記住所有細節,但該認得這幾個最常被提起的名字,各用一句話記住它:

名稱性質一句話
ISO/IEC 27001國際標準建立「資訊安全管理系統(ISMS)」的國際標準,可被認證——證明你「有一套在管」。
NIST CSF美國框架美國 NIST 的網路安全框架,用「治理・辨識・保護・偵測・回應・復原」幾個功能,當作組織盤點資安的共通語言。
SOC 2稽核報告針對服務型/雲端公司,由會計師事務所稽核其安全、可用性等控制是否落實——B2B 軟體業的常見門檻。
PCI DSS產業標準只要你碰信用卡資料,就得遵守的支付卡產業資料安全標準。

除了框架,還有法規(regulation)——那是有法律強制力的硬要求。兩個你一定要聽過的, 都和第 41 章的「隱私」直接相關:

  • GDPR(歐盟一般資料保護規則):全世界最有影響力的個資保護法。它給了使用者「被遺忘權」「可攜權」等權利,要求企業合法、最小化地處理個資,並在資料外洩時限期通報(通常是知悉後 72 小時內)——還記得開頭主管問「要不要通報」嗎?在 GDPR 適用範圍內,這往往是法律義務,不是選擇題。違者最高可罰全球營收的 4%。
  • HIPAA(美國健康保險可攜與責任法案):規範美國醫療相關資料(病歷等)的保護,對機密性與存取控制有嚴格要求。
  • (台灣讀者可對應到個人資料保護法(個資法)。)

稽核:用「證據」證明你說到做到

框架與法規寫了一堆要求,但組織到底有沒有真的做到?這就靠稽核(audit): 由內部或獨立的第三方,依據某個標準,蒐集證據、逐項檢查你的控制措施是否存在、是否有效。 稽核的精神和本章開頭的鑑識其實一脈相承——都是「用可信的證據說話,而不是憑嘴上保證」。 這也是為什麼前面講的日誌、紀錄、監管鏈如此重要:沒有紀錄,你就無法證明自己有在做。

💡
合規 ≠ 安全(但它是好起點)

一個常見的陷阱是把「通過稽核、拿到證書」當成終點,變成打勾文化(checkbox compliance)—— 表單填好填滿,實際卻漏洞百出。記住:合規是「最低標準」,不是「足夠安全」。 框架是別人替你想好的基本盤,幫你不漏掉常識性的東西;但真正的安全,還是得回到你自己的 風險管理——因為只有你最了解自己的資產與威脅。把合規當地基,而不是天花板。

把安全「左移」:在洞出現之前就堵住

現在來回答主管最後那個問題:「這個洞當初為什麼會在?下次怎麼避免?」答案,藏在軟體是怎麼被做出來的。 傳統上,很多團隊把安全放在開發流程的最後——程式都寫完、快上線了,才丟給資安團隊「掃一掃」。 這種「事後才補」的模式,正是第 34 章警告過的「安全當成事後外掛」,而它在經濟上糟糕透頂。

有一個被引用了無數次的觀察:一個安全缺陷,你越晚發現,修起來越貴——而且不是貴一點,是貴幾十、上百倍。 在「需求/設計」階段發現,可能只是改一句話、調一張圖;但若拖到「上線後」才爆出來,你要面對的是緊急修補、 資料外洩、停機、賠償、商譽損失……

🔑
左移(Shift Left)

「左移」就是把安全活動,從開發流程的尾端,往前挪到需求與設計階段。 它的精神和第 34 章「安全要設計進去(security by design)」完全一致,只是換個角度看: 與其在終點站攔截漏洞,不如在它出生之前就不讓它存在。早一點想、早一點防, 不只更便宜,做出來的安全也更紮實——因為它長在系統的骨子裡,而不是事後黏上去的補丁。

各階段的安全活動:secure SDLC

把「左移」落實,就得到了安全軟體開發生命週期(secure SDLC): 在軟體開發的每一個階段,都嵌入對應的安全活動,而不是只在某一站做。 這也呼應了第 40 章講的軟體工程流程——只是我們替每個階段都裝上了一副「安全的眼睛」:

階段安全活動它在防什麼 / 呼應
需求
Requirements		 明確寫下安全需求(如「密碼須加鹽雜湊」);撰寫濫用案例(abuse case):不只想「使用者會怎麼用」,還想「攻擊者會怎麼濫用」。 從源頭就把安全當成「功能」來規格化,而不是漏掉。
設計
Design		 威脅建模(threat modeling)(第 34 章的 STRIDE);套用安全設計原則(最小權限、失效安全預設……,第 34 章);攻擊面分析(把對外的入口縮到最小)。 在最便宜的階段就找出設計層級的洞——這類洞事後最難補。
實作
Implementation		 安全編碼(輸入驗證、輸出編碼等,第 36 章);程式碼審查(code review);相依套件管理(只用可信、有在維護、無已知漏洞的套件)。 擋掉 SQL injection、XSS 等實作層漏洞(第 36 章)。
驗證
Verification		 SAST(靜態分析,讀原始碼找洞)、DAST(動態分析,對跑起來的系統測)、fuzzing(模糊測試)(餵亂七八糟的輸入找崩潰,第 46 章)、滲透測試(penetration testing)。 上線前用多種角度交叉檢驗(第 36、46 章)。
部署與維運
Deploy & Operate		 設定強化(hardening)(關掉預設帳號、不必要的服務)、持續修補(patching)、日誌與監控。 第 34 章縱深防禦的落地;讓你能及早偵測下一次事件(接回本章開頭)。

舉個「濫用案例」的例子,你就懂它和一般需求差在哪——它逼你在寫規格時,就先替攻擊者想好劇本: 

使用案例(use case):
  使用者輸入帳號密碼 → 系統驗證 → 登入成功,進入首頁

濫用案例(abuse case):
  攻擊者拿外洩的「帳號+密碼」清單,用機器人大量嘗試登入(憑證填充)
    → 對策:登入限流、帳號鎖定、MFA、偵測異常登入
  攻擊者在密碼欄位塞入 SQL 片段,想騙過驗證(SQL injection)
    → 對策:參數化查詢、輸入驗證(第 36 章)

驗證階段那幾個縮寫常被搞混,用一句話分清楚:SAST 像「不執行程式、直接讀原始碼挑錯」的校稿(白箱); DAST 像「不看原始碼、直接戳跑起來的系統」的黑箱測試;fuzzing 是丟海量的隨機/畸形輸入, 看程式會不會崩潰或出怪事(第 46 章);滲透測試則是請(獲得授權的)專家, 實際模擬攻擊者,綜合手法試著攻破系統。它們互補,合起來才是縱深的驗證。

✏️ 練習 53.5 設計

你的團隊要開發一個「線上預約掛號系統」:民眾註冊登入後可預約門診、查看自己的就診紀錄; 診所人員可管理時段與看到當日名單。請替它規劃一份 secure SDLC,在需求、設計、實作、驗證、部署維運 五個階段,各列出至少一項具體的安全活動,並簡短說明它防什麼。

看解答

(參考答案,合理且能對應階段即可。)

  • 需求:寫下安全需求——「就診紀錄屬敏感個資,須加密儲存、僅本人與授權醫護可讀」;寫濫用案例——「攻擊者改網址裡的 patient_id 看別人的病歷(IDOR)」「機器人灌爆熱門時段」。防的是後續漏掉關鍵保護與授權檢查。
  • 設計:對「預約」「查紀錄」做威脅建模(STRIDE),標出信任邊界;套用最小權限(診所人員只看得到自己診所、當日名單);攻擊面分析(對外只開必要的 API)。防設計層級的越權與洩漏。
  • 實作:安全編碼(所有資料庫查詢參數化、輸入驗證);每個敏感操作做伺服器端授權檢查(防 IDOR);相依套件只選有維護、無已知漏洞者。防 SQL injection、XSS、越權。
  • 驗證:CI 中跑 SAST 與相依套件漏洞掃描;對登入、預約、查紀錄做 DAST 與授權測試(故意拿 A 帳號去存取 B 的紀錄);上線前做一次授權滲透測試。防上線帶著已知洞。
  • 部署維運:設定強化(關掉預設帳號、最小化開放埠)、全程 HTTPS、定期修補、記錄並監控登入與資料存取(可歸責 + 及早偵測異常)。防上線後被慢慢滲透而無人察覺。

注意這份規劃怎麼把第 34 章(威脅模型、最小權限)、第 36 章(安全編碼、SAST/DAST)、第 46 章(測試)、 本章(監控、可歸責)全部串成一條線——這就是 secure SDLC 的全貌。

軟體供應鏈安全:你沒寫的那 90% 程式碼

這裡有個會讓初學者吃驚的事實:你的應用程式裡,你親手寫的程式碼,可能只佔一小部分。 其餘大量是你「拿來用」的——開源套件、函式庫、框架、容器映像、建置工具。這些「別人寫的、你依賴的東西」, 合稱軟體供應鏈(software supply chain)。而它正是近年資安最熱、也最痛的戰場。

道理很簡單:你的安全,被你最弱的那個相依套件決定。就算你自己的程式碼固若金湯, 只要你用的某個第三方套件有漏洞、或被植入了惡意程式,整個系統就破了一個大洞——而且這個洞, 可能順著「依賴的依賴的依賴」深埋在第七層,你連它存在都不知道(呼應第 36 章講的相依管理)。

📜
兩記警鐘:SolarWinds 與 Log4Shell

SolarWinds(2020):攻擊者沒有直接攻擊那些大型機構,而是入侵了一家叫 SolarWinds 的網管軟體公司, 在它的軟體更新裡植入後門。於是,當全球上萬個客戶(包括許多政府與大企業)「乖乖更新」時, 等於親手把後門裝進了自己家。這是供應鏈攻擊最教科書的形態:攻破一個上游,毒害所有下游。

Log4Shell(2021):一個極其普及的 Java 日誌套件 Log4j,被發現有個嚴重到離譜的漏洞(CVE-2021-44228), 攻擊者只要讓系統「記一行特製的日誌」就能遠端執行任意程式碼。問題是 Log4j 藏在無數軟體的深層依賴裡, 很多公司花了好幾天才搞清楚「我到底有沒有用到它、用在哪」。它血淋淋地證明了一件事: 如果你不知道自己用了什麼,出事時你連從哪裡開始救都不知道。

Log4Shell 那句「不知道自己用了什麼」的教訓,直接催熟了三個今天的關鍵實務:

  • SBOM(軟體物料清單,Software Bill of Materials):像食品的「成分標示」一樣, 列出你的軟體用了哪些套件、什麼版本、什麼授權。有了它,下次再爆出「某套件有漏洞」,你幾秒鐘就能查出自己有沒有中、中在哪。常見格式有 SPDX、CycloneDX。
  • 相依套件掃描(SCA, Software Composition Analysis):自動比對你的 SBOM 與已知漏洞資料庫(CVE), 一發現你用的版本有已知漏洞就示警——最好直接接進 CI,擋在合併之前。
  • 簽章與驗證(signing):用數位簽章(第 35 章)確保你拿到的套件/更新,真的來自宣稱的作者、 且沒被掉包——這正是 SolarWinds 那種「更新被動手腳」最需要的防線。
SBOM 條目(概念示意)
─────────────────────────────────────────────
元件:   log4j-core
版本:   2.14.1
授權:   Apache-2.0
雜湊:   sha256:9f5a...  ← 驗證「拿到的就是這一份」(完整性)
來源:   Maven Central(已驗證簽章)
狀態:   ⚠ 已知漏洞 CVE-2021-44228(Log4Shell,Critical)→ 需升級至 2.17.x
─────────────────────────────────────────────
💡
最樸實的供應鏈防線

不必一開始就上很潮的工具,先把這幾件基本功做好,就能擋掉一大半問題: ① 鎖定版本(lockfile)並驗證雜湊,讓每次安裝拿到的都是同一份、可驗證的東西; ② 把相依漏洞掃描接進 CI,有高風險 CVE 就讓建置失敗; ③ 減少不必要的依賴(回到第 34 章「機制經濟性」與「最小化攻擊面」——你沒用的套件,不會有漏洞); ④ 對新增的依賴有把關流程,而不是任何人想 import 什麼就 import 什麼。

✏️ 練習 53.6 挑戰

請替一個團隊設計一套相依套件安全管理流程。至少涵蓋:(a) 新套件要進專案前的把關、 (b) 已在用的套件如何持續監控漏洞、(c) 爆出嚴重漏洞(像 Log4Shell)時如何快速反應。 盡量把本章與第 36 章的觀念用上。

看解答

(參考流程,合理即可。一份好答案會橫跨「引入前 / 使用中 / 出事時」三個時態。)

(a) 引入前把關:

  • 評估必要性:真的需要嗎?能不能用標準函式庫做掉?(最小化依賴,呼應機制經濟性)
  • 盡職調查:這套件有沒有人維護、社群活躍度、最近更新、是否有已知未修漏洞、授權是否相容。
  • 來源可信:從官方倉庫取得並驗證簽章/雜湊;固定版本(pin)+ lockfile。
  • 需經審查核准,不是任何人隨手就加(納入治理:有政策、有流程)。

(b) 使用中持續監控:

  • 維護一份SBOM,清楚知道「我用了什麼、什麼版本、在哪」。
  • 把相依漏洞掃描(SCA)接進 CI,每次提交自動比對 CVE;設定「出現高風險漏洞就讓建置失敗」的關卡。
  • 訂閱安全公告;定期(例如每月)做依賴更新,別讓版本爛掉到難以升級。

(c) 重大漏洞應變(把它當成一次事件回應):

  • 用 SBOM 幾分鐘內盤點「我們有沒有中、中在哪些系統」——這正是有 SBOM 與沒有 SBOM 的天壤之別。
  • 依風險決定處置:能升級就緊急升級+測試+部署;短期無法升級就先用緩解措施(降低)或暫時下線受影響功能(避免)。
  • 監控是否已被利用(查日誌),必要時啟動完整事件回應流程。
  • 事後檢討:當初怎麼會用到它、能不能更快盤點與修補,把學到的回饋進(a)(b)。

安全成熟度與持續改進:把安全變成「呼吸」

把上面所有東西串起來,你會發現一個共同點:它們都不是「做一次就好」的任務,而是需要持續運轉的流程。 一個組織的資安強不強,與其說看它「買了什麼工具」,不如說看它「把安全內化成流程的程度」有多深—— 這就是安全成熟度(security maturity)的概念。

最不成熟的組織,是「出事才救火、平時不管」;成熟一點的,是「有政策、會定期檢查」; 最成熟的,則是把安全自動化、織進日常的每一次提交與部署,讓「做安全」變得像呼吸一樣自然、 甚至感覺不到它的存在。這個境界,有個現在很紅的名字:DevSecOps。

還記得第 40 章的 CI/CD(持續整合/持續交付)嗎?DevSecOps 的精神,就是把「安全(Sec)」這個字, 無縫塞進 DevOps 的自動化流水線裡:讓安全檢查跟著每一次程式提交自動執行,而不是等到上線前才人工跑一輪。 這正是「左移」+「自動化」+「持續」三者的合體: 

# CI/CD 流水線中的安全關卡(概念示意,呼應第 40 章)
stages:
  - build
  - security:                 # 每一次提交都自動跑,擋在合併之前
      - secret-scan           # 掃出不小心寫進程式碼的密碼/金鑰
      - sca                   # 相依套件漏洞掃描(比對 SBOM 與 CVE)
      - sast                  # 靜態分析,讀原始碼找洞
  - test
  - deploy:
      - dast                  # 對部署好的測試環境做動態掃描
      - config-check          # 檢查部署設定有沒有強化(hardening)
🔑
DevSecOps:安全是「大家的事」,而且「一直在做」

傳統模式裡,安全是「資安團隊」在最後一關把守的責任。DevSecOps 翻轉了這件事: 安全變成開發、維運、資安「共同的、貫穿全程的」責任,並透過自動化讓它在每一次提交時都低成本地發生。 這呼應了第 34 章那句「資安是文化與流程」——當安全檢查像跑單元測試一樣理所當然、像 CI 失敗一樣即時回饋, 安全才真正從「額外的負擔」變成「開發的一部分」。持續改進,沒有終點。

⛔
立場重申:防禦與合規,不教破壞

這一章教你怎麼調查事件、管理風險、合規地處理資料、 安全地開發軟體——全都站在防禦與合規這一側。請牢記幾條界線: 數位鑑識必須在你擁有或獲得明確授權的系統與資料上進行,並嚴守監管鏈與當地法律 (證據的取得方式不合法,再有力也可能不被採用,甚至讓你自己違法); 處理個資要遵循 GDPR / 個資法等規範; 滲透測試一定要有書面授權。和第 34、41 章一樣:技術力越強,越要守住這條線。

把這一章串起來

我們從週五傍晚那通電話出發,走過了資安的三個時態。回顧這趟旅程的路標:

  • 數位鑑識是「事發之後,科學地查清真相」:流程是識別→保全→蒐集→檢驗→分析→報告, 而監管鏈 + 雜湊讓證據「站得住腳」(內容沒被改、過程沒空窗)。
  • 證據散落在磁碟(含已刪除檔、slack space)、記憶體、網路、行動裝置;易失性順序要你「先收最快消失的」——記憶體優先。
  • 事件回應是六階段的循環:準備→偵測→圍堵→根除→復原→事後檢討(第 40 章 postmortem);它與鑑識並肩,但要在「止血」與「保全證據」間權衡。
  • 治理(GRC):資安需要管理層承諾與安全文化;靠風險管理(評估資產/威脅/弱點/衝擊 → 接受/降低/轉移/避免 → 殘餘風險)把不確定變成決策。
  • 合規與框架(ISO 27001、NIST CSF、SOC 2、PCI DSS)是別人畫好的地圖,法規(GDPR、HIPAA,呼應第 41 章)是硬要求,稽核用證據證明你說到做到——但記得:合規是地基,不是天花板。
  • secure SDLC把安全左移(越早修越便宜),每階段都有安全活動:需求(濫用案例)、設計(威脅建模)、實作(安全編碼,第 36 章)、驗證(SAST/DAST/fuzzing/滲透,第 36、46 章)、部署維運(強化/修補/監控)。
  • 軟體供應鏈安全(SBOM、SCA、簽章)——SolarWinds 與 Log4Shell 的教訓;以及用 DevSecOps 把安全自動化、持續化,追求成熟度。

所以,回到主管那一連串問題:查清楚發生了什麼,是鑑識;判斷該不該通報、誰負責,是治理與合規; 而要讓「下次不再發生」,則要回到安全開發的源頭。這三件事,沒有一件是「一個人把程式寫對」就能解決的—— 它們都是組織層級的流程、文化與紀律。這,正是這一章最想留給你的一句話:

資安,從來不只是把程式碼寫對;它是一整個組織,把「在乎安全」這件事,織進每一天、每一次決定裡的長期修行。
